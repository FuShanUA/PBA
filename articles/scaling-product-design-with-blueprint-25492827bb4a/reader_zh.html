借助 Blueprint 扩展产品设计

Blueprint 是一个全新开源的设计系统，以一组可组合的 React 组件形式实现，并针对桌面应用进行了优化。它提供了一种系统化的方式，在众多不同团队间应用统一且一致的设计，从而改变了 Palantir 扩展设计和前端工程的方式。Blueprint 经过我们内部工程团队和设计师的实战检验，极易上手，文档详尽，维护积极，随时可用！请查阅完整文档。这篇博文简要介绍了 Blueprint 在 Palantir 的发展历程、指导其设计的原则，以及一些实现细节。

背景

就在几年前，Palantir 核心产品的用户界面还是用 Java 构建的。那是那种你需要下载并安装的传统软件。当时的 Web 技术还不够成熟，无法构建高性能且易于维护的分析软件，但随着浏览器的进化和新 Web 技术（如 Node、Sass、ES6）的发展，情况开始改变。突然之间，我们可以直接在浏览器中交付完整深度的数据分析软件。

我们很快意识到缺了一些重要的东西。我们的工作存在孤岛效应：平台团队没有相互借鉴彼此的成果，前线部署工程师构建了一次性的解决方案，而我们的设计师和产品工程师经常陷入重复造轮子的境地。结果就是，我们的应用生态系统缺乏连贯的用户体验，效率低下拖慢了我们的步伐，工作质量也大受影响。

正是在这种背景下，我们开始考虑构建一个设计系统。Blueprint 最初只是一名单个设计师的业余项目。随着项目的发展及其潜力逐渐明朗，显然我们需要配备一支由设计师和工程师组成的团队，他们的职责是创建新工具，作为公司其他部门的力量倍增器。经过多次迭代，你所熟知的 Blueprint 诞生了。我们的最终目标是构建一个设计系统，使我们能够：

  • 专注于真正重要的事情。 通过为视觉和交互提供易于使用且符合高标准可靠默认值，我们将设计师和工程师从一遍又一遍重复创建相同的工作中解放出来，从而允许他们利用自己的时间和才干去解决关键问题。
  • 弥合设计与工程之间的鸿沟。 设计师花费了大量时间制作模型和原型，这些都是对软件外观和行为的理想化表达，他们拼命试图向开发者和产品利益相关者传达自己的想法。在整个过程中，由于工具的低效，许多细节在转化中流失了。更好的工具能够弥合沟通鸿沟，让设计师和工程师使用同一种语言，最大化彼此的影响力。
  • 使其运作和感觉都像 Palantir。 Palantir 平台技术栈是根据特定客户工作流配置的，这意味着根据分析目标的不同，我们可能在不同用例中部署不同的平台能力集。通过利用相同的设计系统来构建每一项能力，我们确保了跨平台一致、高质量的用户体验，为我们的用户设定了明确的期望——这有助于使我们的生态系统变得可预测且值得信赖。

指导原则

Blueprint 的架构受一套原则指导，我们相信这些原则对于有效的设计系统至关重要。

组合

Blueprint 旨在通过促进组合和可重用性来保持事物的灵活性。想象一套乐高积木，少数几个基础件就能拼装出复杂的结构。你绝不会要求乐高给你发一个拼装好的模型——魔力在于拼装过程，在于积木块的组合方式。同样地，Blueprint 组件可以组合起来创建更高层次的结构。

无障碍访问

很多时候，无障碍访问被视为事后才考虑的问题。我们决定对无障碍访问坚持高标准质量，因为我们相信无障碍访问是优秀设计的一项基本原则。《Web 内容无障碍访问指南 2.0》帮助我们定义了功能的基础：

  • 可区分的内容和颜色感知。 Blueprint 的调色板及其用法始终保持足够的对比度。所选颜色确保了色盲等感官障碍人群也能无障碍访问。
  • 键盘导航。 各浏览器间的键盘交互既不一致也不可靠。Blueprint 确保了跨操作系统和浏览器一致的键盘导航行为。

这些功能是无障碍访问四大原则中两项的示例：内容需要是可感知的和可操作的。另外两项——可理解和健壮性——是单靠 Blueprint 无法强制执行的方面。通过计时和导航等策略来解决这些问题的责任，落在各个开发者身上。

开发者体验

Palantir 的开发团队在定义其产品的技术栈方面享有极大的自由，我们始终致力于为手头的工作选择最佳工具。Blueprint 早期押注于 TypeScript；事实证明，这一选择对于提供优于 JavaScript 的卓越开发者体验至关重要，并最终推动了 Blueprint 在 Palantir 的普及。更具体地说，我们观察到了以下优势：

  1. TypeScript 迫使开发者对假设更加明确，而不是在脑海中隐式地构建 API 契约并在事后补充文档，它还使得保持高质量、最新的文档变得轻而易举（稍后详述）。
  2. 尽管 TypeScript 并非能捕获所有运行时错误的万无一失的系统，但它确实通过工具集成大幅提升了生产力。现代 IDE 提供了对 React props 的验证和内联文档；这极大地加速了开发和重构过程。

另一个重要的方面是我们如何设计 CSS API。与其他 CSS 框架相比，Blueprint 尝试采取一种最小侵入的方法。全局行为和样式仅限于排版和跨浏览器标准化；其他所有内容都需要通过 CSS 类或显式使用交互式组件来选择性启用。例如，有时你确实只需要在应用中使用一个自定义的 <button>，而不是 .pt-button。作为用户，你不必为了实现这一点而与框架作斗争。

长期以来，React 一直是 Palantir 首选的框架。其极简的 API 极易上手，且社区活跃、知识渊博。它的速度和对组合的强调使其成为构建复杂、高度交互应用程序的轻松之选。Blueprint 为 React 应用提供了大量基础构建块，从几乎微不足道的 Button 到明显更复杂的 Popover 或 DateRangePicker。

机器人让我们更硬核、更优秀、更迅速、更强大

我们在 GitHub 仓库中使用 CircleCI 进行持续集成构建。如果所有测试通过，我们的 CI 机器人会在相应的提交下发布包含实时预览链接的评论。这极大减少了开发新功能时的摩擦，因为审查者极少需要检出、构建并在本地运行项目以提供反馈——他们只需点击链接即可与最新编译的代码进行交互。

回馈社区

Blueprint 被 Palantir 数百名前线部署工程师、产品工程师和设计师使用。它已成为我们产品设计基础设施的基石。工程师和设计师选择 Blueprint 是因为它具有：

  • 一个由文档详尽且支持良好的开源项目组成的技术栈，
  • 一个全面的组件库，
  • 从一开始就融入的无障碍设计，
  • 优秀的文档。

回首 Blueprint 在 Palantir 内部取得的成就，我们为自己的工作以及迄今为止所完成的一切感到自豪，即使这仅仅是旅程的开端。

我们相信，无论你是个人、小型初创公司还是大型组织，Blueprint 都为启动和扩展产品设计提供了一种新方法。这就是为什么我们非常激动能向开源社区回馈力量。我们期待看到社区用 Blueprint 创造出什么成果。
