gödel 的架构与 Gradle 类似，包含一个引导可执行文件 (godelw) 和一个配置目录 (godel)。将 gödel 添加到项目后，典型的开发周期如下：

  • ./godelw idea gogland 生成 Gogland 项目
  • ./godelw verify 运行检查和测试
  • ./godelw publish 构建并发布二进制文件

gödel 使用 Go 编写，并由其自身构建。它委托标准的 Go 命令和工具来执行各项任务。gödel 简化了构建系统的设置，确保了连贯且可预测的开发体验，并通过将构建逻辑与构建配置分离，使组织和社区能够标准化其构建工作流。

动机

当 Palantir 开始在一些项目中使用 Go 时，我们很难为开发、测试、构建和发布提供一致的体验。开发人员需要在代码上手动运行一系列静态检查和测试，然后将代码提交至运行持续集成的版本控制系统。随后，CI 系统会运行相同的一组静态检查和测试，为各种目标平台构建可执行文件，创建分发包并上传以供分发。保持不同开发环境与 CI 系统之间的一致性是一项挑战，尤其是随着团队规模的扩大。我们开始采用执行静态检查的工具（如 errcheck），但由于它们是通过 go get 安装的（该命令会安装运行时的最新版本），因此结果会因开发人员配置环境的时间不同而有所差异。

随着项目复杂度的增加，我们引入了脚本来编排验证和发布工作流——紧接着便陷入了脚本带来的典型困境：脚本逻辑变得越来越复杂，项目级别的配置也与构建逻辑本身纠缠不清。当团队启动新项目时，他们会从其他项目复制脚本并做些微调。这个过程容易出错，且日后很难进行全局更新。

这些问题影响了我们的开发速度，因此我们决定从问题的本质入手。我们希望构建系统具备以下特性：

  • 稳定：每个项目都会声明在项目级别运行的确切检查和操作集，并与项目源代码一同进行版本控制
  • 模块化：项目维护的是构建系统配置，而非构建逻辑
  • 可维护：构建逻辑由中央团队维护，并可轻松跨项目更新
  • 一致：不同项目的开发环境和构建行为保持一致

我们查看了各种开源的 Go 代码库，但未找到令人满意的解决方案。许多较小的项目使用 CI 脚本执行 go get 检查，并将 go get 作为其分发机制。较大的项目通常使用 makefiles 和脚本。Gradle 是 JVM 生态系统中出色的构建系统，但看起来我们需要为 Go 支持开发多个插件，而且为了构建我们的 Go 项目而要求安装 JVM 也并不合适。

作为声明式构建工具的 gödel

基于上述列表的指引，我们在 2016 年 Palantir 的半年一度黑客周期间启动了 gödel 项目。自那时起，它已成熟并成为 Palantir 所有 Go 项目的标准构建系统。其设计借鉴了 Gradle：gödel 二进制文件封装了执行一系列构建任务（如语法检查、编译、测试执行、打包、发布等）的逻辑，而启用了 gödel 的项目则指定所需的 gödel 版本以及针对该构建的项目特定配置。gödel 保证在项目的给定 Git commit 上执行 ./godel verify 时，无论该命令是在开发人员的笔记本电脑上还是在 CI 系统上运行，都会执行完全相同的验证任务集。

通过向项目添加 godelw 脚本和 godel 配置目录，即可为项目启用 gödel。调用 ./godelw 脚本会触发在本地下载并安装正确版本的 godel 二进制文件。一旦将 gödel 添加到项目中，典型的开发周期如下：

  • 运行 ./godelw idea gogland 生成 Gogland 项目（配置为在保存时运行 ./godelw format）
  • 编写代码
  • 运行 ./godelw verify 验证项目是否通过所有检查（静态代码分析、测试），并强制确保许可证头和 go generate 任务等处于正确状态
  • 配置 CI 系统运行 ./godelw publish 构建并发布项目的分发物

《gödel 教程》提供了 gödel 的完整指南，包括设置项目以及利用 gödel 的所有功能。从我们迄今为止完成的项目来看，我们观察到了以下主要收益：

  • 提升新项目的开发速度。 开发人员可以更快速地设置能够构建和发布制品的新项目。过去，他们不得不从其他项目手动复制和编辑脚本，这是一个容易出错且耗时的过程。
  • 更快、更广泛地采用构建逻辑更新。 现在，只需查看项目的 godel/godel.properties 文件中的版本，即可确定项目所使用的构建逻辑。更棒的是，我们可以自动将构建系统升级推送到所有项目。在引入 gödel 之前，跨项目推送构建逻辑更新是一个痛苦且耗时的过程，因此许多脚本都过期了。
  • 随着时间推移，在 CI 中重新运行构建时更加一致。 由于我们的构建脚本使用 go get 安装检查逻辑，CI 构建在一段时间内并不稳定。使用 gödel 锁定正在运行的检查和逻辑的版本，提高了我们构建的长期可复现性。
  • 促进项目间开发者的协作。 由于我们的项目已将 gödel 标准化，开发人员能够更轻松地快速签出并为不同项目做出贡献，从而增加了跨产品协作以及直接的介入贡献和拉取请求。
  • 知识与价值的复利效应。 借助 gödel，构建系统的改进可以轻松地与我们的 Go 社区共享。例如，当我们想创建一个允许项目构建其产品 Docker 镜像的任务时，开发人员可以轻松地将此功能添加到 gödel 中，因为它已经具备了产品及其分发物的概念。所有特定于构建的配置都集中在 godel/config 目录中，这种方式也让团队能够轻松查看彼此的配置，以确定如何执行特定操作。过去，此类配置通常与脚本逻辑纠缠在一起，难以阅读且难以应用于其他项目。

结论

使用 gödel 极大地改善了 Palantir 的 Go 开发体验。它简化了新项目的引导流程，并显著降低了管理大型项目的复杂度。gödel 在开发人员和 CI 系统之间提供了一致的体验。它还在低门槛与高上限之间取得了平衡：gödel 的默认配置足以构建、分发和发布一个标准项目，同时 gödel 也具有足够的灵活性，能够支持构建大型的、多产品类型的项目，这些项目会创建多个分发物并进行广泛的自定义。

我们相信每个人都能从 gödel 所支持的基本原则中获益，并希望 Go 社区中的其他人也能觉得它有用。我们确信 gödel 在当前形态下就能提供显著价值，并已将其作为 1.0.0 版本发布。然而，gödel 目前的架构使其无法达到应有的灵活性：由于所有任务都必须编译到 gödel 自身中，因此无法动态自定义可用的 lint 检查，也无法在不修改核心代码库的情况下向 gödel 添加自定义任务。gödel 发展的下一个重要步骤是使架构更具可插拔性，从而实现动态添加或自定义任务。

我们期待您的反馈并欢迎贡献！

附注：gödel 的名字是对 Kurt Gödel 的致敬。与许多 Go 工具一样，它是涉及“go”的双关语。它也是对哥德尔不完备定理的双关，其理念在于：标准的 Go 工具为 Go 项目提供了一个自洽的系统，但存在一些仅靠标准 Go 工具本身无法证明为真的真理。gödel 则充当此系统之外的工具。构建工具 gödel 与编程语言 Gödel 并无关联。
