# 满天云彩 Robert Fink 14 分钟阅读 2020 年 4 月 27 日 [Palantir](https://www.palantir.com/) 的客户使用我们的 [Foundry](https://www.palantir.com/palantir-foundry/) 和 [Gotham](https://www.palantir.com/palantir-gotham/) 软件平台来解决他们最棘手的数据问题。我们的许多客户在高度受监管的环境中运营，这给我们的软件运行方式带来了额外限制。本文介绍了在异构环境中运行 SaaS 平台的复杂性，并概述了为管理整个集群中数百个专属安装而开发的解决方案。 ![image](https://miro.medium.com/v2/resize:fit:700/1*CBxC4lo0gUXaKKcwNOzA.png) 关于我们软件的一个最常见的误解是“Palantir 将所有数据都吸入其数据库中”，暗示我们维护了某种全知全能的数据吸尘器，把所有客户的数据都吞噬到一个中心节点。但事实恰恰相反：**我们将软件带给客户**，而不是**将他们的数据带给我们的软件**。每个客户都在 Foundry 和 Gotham 环境中管理**他们自己的**数据；这些环境可以在物理上隔离——例如政府机构的断网安装——或者通过网络、加密或访问控制边界划分为虚拟环境。当然，我们的客户也始终保留对其数据的完整合法所有权。 因此，我们的软件工程师必须在各种不同的部署环境中维护大量的 Foundry 或 Gotham 安装。每个安装都包含数百个微服务、前端应用和数据库系统，由全球数十个工程团队开发。我们常说我们维护着**满天云彩**，意思是每个安装本身就是一个复杂的云原生应用，而我们维护着数百个这样的应用。这些“云”形态各异：我们在多种商业云提供商上运行软件，我们的一些政府客户有法律限制，要求在没有互联网连接的涉密数据中心进行安装。 虽然 Foundry 和 Gotham 都采用软件即服务 模式，但为了满足客户的独特需求，我们开发了几种超越典型 SaaS 提供商的选项。其中一种配置是我们的**专属 Foundry** 和**专属 Gotham** 产品，允许 Palantir 满足客户特定的数据驻留和隔离要求。虽然仍是我们全托管云平台的一部分，但专属 Foundry 和专属 Gotham 代表了我们 SaaS 平台向额外安装的分区。 为了应对这种复杂性，我们开发了一系列部署基础设施工具。本文接下来的部分将描述我们支持的不同部署场景、我们的持续交付系统（从万米高空俯瞰的视角），最后是我们提供给开发人员的用于监控、调试和调优应用的工具。 ### 部署场景 我们与世界各地的客户合作，他们有各种合规性、业务和安全的考量。数据本地化法律可能规定客户数据必须存储在特定地理区域，例如某个特定国家或欧盟。其他客户可能出于架构或业务原因，对某些云提供商有强烈的偏好或抵触。我们的一些政府客户在涉密环境中使用我们的软件管理数据，通常在物理隔离的数据中心和没有互联网连接的网络中。 大型组织的生产系统通常需要流程和变更管理认证，这些认证将开发、维护和记录软件系统的最佳实践编纂成文。例如，大多数美国政府机构要求软件必须托管在 FedRAMP 认证的环境中。我们的大多数客户都期望获得 SOC 或 ISO 27001 认证。 我们自 2014 年起就通过公有云提供商提供 Foundry 和 Gotham 的 SaaS 模式，如今绝大多数安装都托管在 AWS 或 Azure 上。我们最近获得了 FedRAMP Moderate 和 DoD Impact Level 5 认证，以支持我们为美国政府客户提供的基于云的 SaaS 产品，并正在其他司法管辖区寻求类似认证，例如澳大利亚针对 PROTECTED 数据的 IRAP。 ### Skylab：管理单个环境 为了满足客户的需求和期望，我们必须投资于能够在上述复杂场景中运行的部署基础设施。为了管理单个 Foundry 或 Gotham 安装中运行的数百个服务、应用和数据库系统，我们首先构建了一个名为 **Skylab** 的系统（以美国的第一个空间站命名）。Skylab 负责每个应用的生命周期（如安装、启动、停止、升级）及其配置（RPC 和服务发现、加密密钥及其他运行时配置）。 ![image](https://miro.medium.com/v2/resize:fit:700/1*fwIY9Km63j5QxnurWKxPjQ.png) *Skylab 是 Foundry 或 Gotham 安装的主机和应用的控制平面。管理员通过 Skylab Web UI 和一系列命令行实用程序来配置和管理应用的生命周期。* 管理员通过 Web UI 配置应用，Skylab 负责对受影响的机器和应用执行必要的配置或生命周期变更。 在内部，Skylab 遵循类似 Kubernetes 的声明式模型：管理员设置期望的最终状态，Skylab 执行将应用从当前状态切换到期望状态所需的步骤。环境中的每台机器都运行一个所谓的**部署代理**，执行从中央 Skylab 控制平面中继的生命周期命令。 ![image](https://miro.medium.com/v2/resize:fit:700/1*tI7pUtQ32XoAarcqqXjTWQ.png) *Skylab Web UI 的截图，展示了在开发安装中 Postgres 数据库的配置。请注意，大多数配置条目来源于产品默认值，而 PGDATA 条目已被 Skylab 覆盖。* 由 Skylab 管理的应用必须遵循我们的**服务布局规范** (SLS)，该规范定义了如何将应用代码、生命周期管理脚本（启动/停止/重启）、硬件需求（如内存、磁盘、CPU）和默认配置打包成一个独立的 tarball。例如，为了升级一个应用，Skylab 会从制品库下载新的应用 SLS 包，将其复制到目标机器，然后要求部署代理停止旧的应用进程，对默认配置应用配置覆盖，并启动新版本的应用。 总结来说，Skylab 是单个安装**内部**所有 Foundry 或 Gotham 组件的控制平面。 ### Apollo：管理多个环境 虽然 Skylab 简化了单个 Foundry 或 Gotham 安装中数百个组件的维护工作，但很快我们发现，仅靠 Skylab 来维护数百个 Foundry 和 Gotham 安装仍然效率低下： - 首先，Skylab 要求管理员执行手动任务来升级或配置特定应用；这既耗时又容易出错。 - 其次，我们的安装的应用版本和配置很快出现了分歧，使得开发人员的调试和性能分析变得更加困难。 我们接下来构建了 **Apollo**，一个用于 Foundry 和 Gotham 的持续交付系统，主要目标有两个： - 首先，跨安装协调并自动化安装管理。 - 其次，作为**安全**软件交付的推动者，并大幅改善开发人员的反馈回路。 Apollo 充当不同安装中 Skylab 实例的幕后操纵者：我们现在不再要求管理员在每个环境中执行相同的升级或配置任务，而是在 Apollo 的配置存储库中集中声明软件版本和配置目标，并让 Apollo 将必要的变更传播到相关的 Skylab 实例。 ![image](https://miro.medium.com/v2/resize:fit:700/1*lHD1ybESBOETJm94NE9Zrg.png) *Apollo 是跨安装环境管理多个 Palantir 安装的控制平面。管理员集中配置应用，Apollo 通过委托给每个环境的 Skylab 实例来应用配置。控制流是这样的：Skylab 实例向 Apollo 查询升级任务，而不是 Apollo 直接深入安装环境；Apollo 本身不需要与 Skylab 实例建立直接连接。* 在概念上，Apollo 非常简单：每个 Skylab 实例不断提交其环境的当前状态，例如正在运行哪些应用及其版本、每个应用的健康状况、当前配置等。Apollo 将观察到的当前状态与声明的目标状态进行比较，计算出每个环境需要应用的变更任务队列；Skylab 实例从队列中获取变更任务，并通过各自的部署代理执行变更。与 Skylab 一样，Apollo 也有一个 Web UI，管理员和开发人员可以通过它查看哪个组件安装在哪个环境中、观察升级进度或召回发布（见下文）。 ![image](https://miro.medium.com/v2/resize:fit:700/1*hKvcsxK4pBuqxhcaGDqAgw.png) *Apollo Web UI 的截图，展示了与“notifications”组件尝试蓝绿升级相关的一些 Apollo 任务。* Apollo 提供的自动化允许我们做一些很酷的事情，例如： **持续交付。** 我们的构建系统将 Foundry 和 Gotham 组件发布到与 Apollo 连接的中央制品库。Apollo 会自动了解哪些组件的哪些版本可用，并根据特定规则应用升级；例如，主要升级可能需要更长的浸泡时间，而补丁或错误修复可以更快地推出。 **安全交付。** Apollo 是我们软件质量和测试方法的关键组件：我们使用合并前的单元测试以及合成集成或端到端测试（在我们的持续集成环境中执行），然后用新版本激进地升级金丝雀环境。 许多组件直接从 develop 分支发布到金丝雀环境，从而大大缩短了软件开发人员的反馈回路：现在通常的做法是，在几分钟内就能在金丝雀安装（例如，我们运行 Palantir 自身数据分析和工作流的内部 Foundry 安装）中观察到新功能或配置变更。 Apollo 通过观察性能指标和错误状态来自动**裁决**发布，然后逐步将通过裁决的发布推向整个集群，从内部安装开始，最终到达更保守的、关键任务环境。服务和环境可以表明它们的风险偏好，从而在“快速但有风险”或“缓慢但经过测试”的升级之间选择合适的权衡。 你可以在最近的一篇 Quora 帖子中阅读更多关于我们的开发实践和技术栈的信息。 ## 在收件箱中获取 Robert Fink 的文章 加入 Medium 了解哪些发布需要数据库迁移，这些迁移构成了升级或降级障碍，并选择与这些约束兼容的升级路径；例如，Apollo 会通过一系列强制数据库迁移来逐步升级特定服务，而不是简单地直接跳到最新可用的版本，那会使数据库处于不兼容状态。 这意味着即使是复杂的升级和迁移，对开发人员来说也是无需干预的：他们只需在发布元数据中指明存在数据库迁移，Apollo 就会负责以安全的方式推出该版本。 **召回。** 如果应用的某个发布被确认有错误、不稳定或运行缓慢，我们可以召回该发布，并强制在所有环境中升级或降级到已知良好的版本。由于召回事件引起的此类变更任务优先于常规的持续交付升级，以便关键错误修复能快速推出。 召回可以由管理员和开发人员完成，也可以由 Apollo 自己完成：Apollo 裁决系统随时间推移监控每个组件的健康状况（例如通过观察错误日志和性能指标），并自动召回看起来不健康的发布。 **蓝绿部署。** 这本身就可以写一篇博客文章了，但简而言之，蓝绿部署的思想是通过安装应用的第二个实例并慢慢将流量从旧实例转移到新实例来升级应用。 我们所有的应用和服务都可以部署在支持零停机升级的高可用、多节点配置中。在这样的高可用设置中，我们可以简单地逐一添加和删除节点，以便从旧版本升级到新版本。 Apollo 的发布裁决机制（见上文）会在第一个节点升级完成后立即开始收集性能和错误信号，从而为发布质量提供早期指标。从这个意义上说，蓝绿部署不仅是一种零停机推出机制，更是一个额外的安全网，允许我们进一步加快开发人员的反馈回路。 **浸泡测试**。Apollo 检查每个发布的结构化变更日志以评估其风险；例如，微小的外观更改或错误修复可被视为较低风险，而 API 更改、新功能或数据库迁移通常表明风险较高的发布。Apollo 根据这些风险启发式调整浸泡时间，即在更广泛地推出发布之前，用于观察和裁决发布的时间量。 **变更管理策略。** Apollo 的声明式状态（例如，应安装哪些应用及使用何种配置）存储在 Git 存储库中。我们使用一套丰富的合并和代码审查策略系统来控制谁可以对安装执行更改。例如，我们可以通过要求不同团队的开发人员在代码合并前分别进行审批来强制实现职责分离。 总结来说，Apollo 是一个持续交付系统，管理着 Palantir 的所有 Foundry 和 Gotham 安装集群。它通过将配置变更和软件升级任务转发到相应的每个环境的 Skylab 实例来工作。 ### 异步交付 虽然上述的 Apollo 对于在云环境中运行的安装效果很好，但对于必须在断网环境中运营的政府客户来说，它并没有解决问题。这些环境具有单向、高延迟的通信特点：将新的软件版本发布到这些环境中通常涉及刻录 DVD、病毒扫描和人工将二进制文件上传到服务器。 *题外话：此配置仅作为我们涉密产品的一部分提供给政府客户。这一决定背后的原因很简单：物理上断开的环境是双输的，对客户来说成本要高得多，管理起来更困难，也更难保持最新。事实上，对于网络未进行物理隔离的组织来说，本地部署环境可能构成更大的安全风险。例如，对于涉密政府网络上的断网安装而言，关键安全补丁延迟一周可能是经过计算的、可接受的风险（因为没有连接到互联网的线缆，且安装与其对手之间有物理边界），但对于我们互联网连接系统的商业客户来说，这种延迟是不可接受的。Apollo 和持续交付允许我们以最小的延迟在所有联网的安装中推出此类修复。* 我们最初实现持续异步交付的方法是由负责管理断网安装的前线部署工程师 团队（即直接与相应客户合作的软件工程师）推动的。受到 Apollo 为联网云环境带来的阶跃式改进的启发，他们构建了一个精简版的 Apollo，名为 SLIM，可以在每个环境中运行。 对于这些断网环境，我们在内部运行了一个环境副本，可以在其中暂存每个组件的升级。我们首先验证想要升级的版本集，然后对应用版本和配置进行快照，最后将配置快照连同相关的发布二进制文件一起运送到客户环境。当前线部署工程师 团队将快照导入环境后，Skylab 和 SLIM 将协同执行快照所描述的变更。 ![image](https://miro.medium.com/v2/resize:fit:700/1*SZ8VRJA7K6g87uG5MjDXhg.png) *断网版 Apollo 的工作方式与集中式 Apollo 一样，只是 Apollo 服务安装在政府网络内部。此设置支持不允许客户环境与我们的中央 Apollo 服务之间进行双向网络流量的气隙网络。* 这种方法起初对我们很有用，但随着 Apollo 开发出更高级的功能，如蓝绿部署，我们发现我们必须在 SLIM 上投入大量精力以保持与 Apollo 的功能对等，并且我们面临一些关于如何处理 Apollo 某些功能（如自动发布召回）的艰难决定，这些功能在断网环境中运行没有意义，因为它们依赖于来自多个安装的发布反馈。 为了简化流程，我们决定重构 Apollo，以便我们可以在所有断网环境中部署独立的 Apollo 安装。（此时我们还决定 Apollo 应该能够自行持续升级。） 巧合的是，我们的 FedRAMP 架构依赖于一个专用的 Apollo 实例来管理 FedRAMP 云环境中的 Foundry 和 Gotham 安装；因此，重构工作同时实现了断网和 FedRAMP 两种工作流。如今，我们在断网和云环境中使用相同的推出和召回策略，以确保我们的软件在所有安装中都具有可预测的、稳定的配置。 ### 收集信号 众所周知，软件总是会有令人兴奋和意想不到的行为，因此 Foundry 和 Gotham 的构建方式必须允许开发人员理解和调试他们的组件，即使他们无法访问已安装的软件或机器。为此，我们依赖两个主要的调试信号来源：性能指标以及系统和软件日志。 棘手的地方在于决定可以将哪些指标和日志收集并提供给开发人员，而不会有泄露敏感客户信息的风险。基本规则是，任何客户数据和任何用户生成的内容都不能离开环境。保守地说（我们宁愿防患于未然），这意味着开发人员只能依赖编译时静态数据（例如代码常量）或随机标识符来进行调试工作。为了将敏感信息与我们所谓的“安全”（即“可安全记录”之意）信息区分开来，我们投资建设了日志记录基础设施和库，允许开发人员将某些信息标记为“安全”，例如： ``` log.error("Failed to read dataset", SafeArg.of("datasetId", datasetId), UnsafeArg.of("userId", userId)); ``` 日志基础设施能够区分安全和不安全参数，仅将安全信息提供给开发人员。在这种情况下，开发人员可以访问错误消息和随机的数据集标识符，但无法访问可能敏感的用户标识符。我们使用编译时代码分析来验证开发人员不会意外泄露不安全信息，例如通过： ``` log.error("Failed to read dataset, user=" + userId); ``` 此代码在我们的持续集成基础设施上无法编译，因此无法进入发布版本。将安全日志信息与不安全信息分离开来，为我们的工程师提供了最大的信息量，同时确保我们绝不会无意中将客户数据暴露给开发人员。 ![image](https://miro.medium.com/v2/resize:fit:700/1*i9EiXAiOB3E6zGvWN4Qn-g.png) *所谓的“安全”日志和指标信息不包含客户数据，并被发送到中央日志索引基础设施，以支持开发人员调试和性能分析工作流。对于断网安装，我们会在环境中与 Foundry 或 Gotham 一起安装 Apollo 的日志和指标数据库及分析工具。* 我们所有的服务都以结构化格式发出性能和健康指标，Apollo 可以对其进行检查，例如为了在蓝绿升级期间比较两个软件版本的性能。 除了服务日志、RPC 请求日志和性能指标外，我们的部署基础设施还支持自动收集事件、分布式堆栈跟踪（类似于 Zipkin 和 OpenTracing）、线程转储以及诊断日志（如垃圾回收统计）。我们每天收集超过 30TB 的日志，并将日志保留 14 天以供调试工作流使用。我们的端点指标每天涵盖大约 8000 万个独立时间序列和 1000 亿个数据点。
