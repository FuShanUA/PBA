现今的现代 Web 应用程序通常由数十到数千个独立服务组成，每个服务提供应用程序功能的一部分。这种微服务架构具有明显的优势（关注点分离、独立可扩展性、更快的迭代和部署、软件与团队架构的一致性等），但也是有代价的。要理解应用程序的行为——包括资源利用率、性能、错误等——需要调查复杂的分布式系统。本地开发工具（如 IDE 或调试器）不足以分析和理解运行在异构基础设施上、以多种不同语言编写的、未同步进程的行为。遥测技术提供了一种解决方案。

遥测——相当于本地开发工作流中 IDE 和调试器的分布式系统版本——允许开发人员和站点可靠性工程师（SRE）了解应用程序的性能、健康状况和使用模式。标准的调试工具被分布式指标和日志收集及聚合基础设施所取代，包括 Datadog、OpenTracing/Zipkin、Grafana、Influx 和 Prometheus 等工具。这套工具使得实现系统健康和日志信息的端到端遥测成为可能，这是任何分布式大规模应用程序的关键基础设施。

但这种端到端的理解并不是免费获得的。为了利用各种日志流，必须构建索引、搜索和分析基础设施。例如，开发人员可能想要调查针对某个服务的哪些 API 调用触发了某些已知的错误状态，这可以通过一条错误日志条目来证明。假设应用程序的每次部署的每个服务都会发出一连串的日志行，我们必须收集、索引并针对全部日志回答搜索查询。

在这篇文章中，我们描述了如何通过开发一个基于预计算 Lucene 索引并结合冷热存储的系统来提供针对这些索引的搜索服务，从而构建一个稳定、可扩展且高性能的遥测基础设施。

遥测管道的组件

一个典型的端到端日志管道包含以下几个阶段：

日志生成
应用程序的每个服务负责添加日志语句，传达有关其状态、进度、成功和失败条件的关键信息。我们在所有服务和应用程序中规范和标准化了日志生成的三个方面：

  • 首先，每个日志级别（WARN、INFO、ERROR 等）都与语义和告警阈值相关联。例如，ERROR 级别的日志通常会触发受影响团队的电话值班告警。
  • 其次，我们维护了一些指南，说明哪些类型的信息可以包含在日志中。例如，身份验证令牌或密码绝不能出现在日志中。
  • 最后，我们为结构化日志定义了 JSON 格式，包括源主机、日期时间、消息、日志级别、日志类型等字段。常用语言（Java、Go、Python）的库将标准日志框架（例如 Java 服务的 SLF4J）发出的消息透明地编码为 JSON 格式。

收集
根据环境的不同，JSON 格式的日志被发送到文件或标准输出。每个服务的日志收集器提取所有日志，并将它们推送到全局流中（例如 Kafka 或 AWS Kinesis）。

过滤和标准化
在从流中索引日志之前，我们通过白名单和黑名单过滤日志：只允许定义的环境集合提交日志，并且日志必须符合语法和内容限制。由于日志模式可能会随着时间的推移而演变，我们通过将不同模式版本的日志映射到最新支持的模式来协调它们。

索引
最后，我们对过滤和标准化后的日志进行索引。索引是考虑到典型的搜索工作负载而进行的：我们对全文日志消息和所有结构化字段进行索引，包括日期时间、日志类型、错误级别、源主机和环境等。

搜索
开发人员和 SRE 通过自定义 UI 或 API 与索引日志进行交互。例如，用户查询来自某个服务或环境的所有日志的实时流、包含特定令牌或字符串的日志、对应于调用追踪 ID 的日志等。

Elasticsearch 遥测管道

现在让我们仔细看看日志管道的索引和搜索阶段。随着对 Palantir Foundry 集中式日志基础设施的需求变得明显，我们使用现成的组件构建了第一个版本，并选择了 Elasticsearch 用于索引和搜索阶段。


基于 Elasticsearch 的索引和查询架构。

该架构很简单：许多索引节点从流中读取日志行块（见上文），并将每个结构化日志行索引到 Elasticsearch 集群中。一旦 Elasticsearch 写入成功，流读取就会被确认。用户和 UI 使用 Elasticsearch API 对 Elasticsearch 集群执行搜索。

基于 Elasticsearch 的解决方案的优势包括：

  • 上市时间：通过组装现有组件，我们能够在几周内交付日志基础设施的第一个版本。
  • 有据可查且稳定的 API：索引和搜索工作流使用稳定、经过实战检验的 API，所有参与的开发人员和团队都可以轻松访问。
  • 系统管理工具：Elasticsearch 提供了系统指标，使我们能够了解日志管道的健康状况，以及用于调整集群行为的工具和配置。

达到 Elasticsearch 的极限

随着服务和应用程序部署数量的增长，摄入到 Elasticsearch 中的日志行数量也在增加。到 2017 年，我们运行了一个由 100 多台机器组成的生产集群，以处理每天数 TB 的日志量。此时很明显，Elasticsearch 已不再是日志管道正确的后端选择，这不是因为 Elasticsearch 本身存在根本性缺陷，而是因为 Elasticsearch 的设计目标与日志索引的要求之间存在阻抗失配：

  • 我们花费了大量时间来管理 Elasticsearch 集群。诸如节点重启和集群及节点调整大小等日常集群管理活动需要大量的人工干预，而且我们经历了频繁的停机。
  • 保留策略难以管理。调试和监控工作流只需要有限数量的日志数据；例如，过去 30 天的数据。在 Elasticsearch 中实现保留策略比较困难。
  • 所有数据都处于“热”存储中。Elasticsearch 将所有数据存储在本地硬盘上，通常带有冗余。这与标准访问模式形成对比：近期日志的查询频率远高于旧日志，导致成本/性能比不佳。
  • 索引和搜索基础设施紧密耦合。索引和搜索工作负载共享相同的 Elasticsearch 基础设施，因此无法独立扩展。然而，索引和搜索工作负载通常具有非常不同的负载特征：索引负载大致是恒定的，因为它是日志生成应用程序规模的函数。相反，搜索负载取决于并发用户的数量，因此会随着一天中的时间和一周中的某天而激增。此外，索引管道的停机意味着搜索能力的停机，反之亦然。
  • 索引和搜索吞吐量无法动态扩展。例如，假设集群的规模设定为在最大吞吐量的 75% 下处理稳态索引工作负载。那么上游日志基础设施计划内或计划外的 x 分钟停机就需要 3x 分钟让索引器追赶上来。通过添加额外的临时容量来提高索引器的吞吐量是不容易实现的。

我们一开始使用通用数据库快速起步，但当我们的需求向着 Elasticsearch 未曾设计的方向发展时，即对大规模结构化、不可变流进行相对简单的查询，我们达到了它的极限。

回归第一性原理：索引和搜索

现在让我们重新审视索引和搜索基础设施的需求，然后根据这些需求勾勒出一种替代架构：

  • 日志数据表现为一连串小的（字节到低千字节量级）、不可变、结构化的记录。
  • 日志数据通常具有稳定但持续增长的规模。索引器应该能够处理偶尔的突发情况；例如，由于上游日志生产者的停机或延迟造成的突发。
  • 近期日志比旧日志更有价值，查询也更频繁。搜索系统对近期日志提供更好的搜索性能，对旧日志提供较差的性能是可以接受的。日志可以在可配置的保留阈值（例如 30 天）后被删除。
  • 搜索是针对结构化记录字段的的关键字查询。这些可以是精确匹配查询、子字符串匹配查询、范围查询或全文查询。

这些需求与 Elasticsearch 数据模型之间的一个关键区别在于索引记录的不可变性：我们绝不会在日志行生成或索引后对其进行更新。在分布式系统设计方面，这意味着 Elasticsearch 中协调和同步复杂性的主要驱动因素消失了，我们可以在保持相同的最终一致性保证的同时，简化索引和搜索组件之间的交互点。特别是，可以预计算搜索索引，将其存储在廉价的冷存储（如 S3）中，然后将其加载到搜索节点中以回答查询。

第二个区别在于查询复杂性：为了支持连接和聚合等非本地操作，分布式数据库需要大量的集群协调。这通常会导致索引性能下降，并需要复杂的集群维护协议。

基于预计算 Lucene 索引的修订版索引和查询架构。

简而言之，我们将索引和搜索解耦为两个不同的阶段：索引器消费日志流，生成时间和空间受限的 Lucene 索引，并将它们推送到冷存储。搜索节点将相关索引分片的子集提取到热存储（本地磁盘和内存）中，以便为查询提供服务。让我们详细看看这些组件。

索引器
假设是一个分片的日志流（例如 AWS Kinesis 或 Kafka 模型），每个索引器消费日志流的一个按时间排序的子集（例如 Kinesis 的“分片”），并生成一个由本地存储支持的、时间和空间受限（例如 1 小时和 10GB 的最小值）的 Lucene 索引。由于日志记录的模式是已知的，我们可以使用从字段到索引配置的静态映射，例如如何分词、使用什么数据类型等。当达到时间或空间限制时，索引器将压缩后的 Lucene 索引推送到冷存储，并在索引目录中注册相应的元数据。然后它提交流中的位置，并从一个新的空索引重复该过程。

索引目录
目录存储生成的索引分片列表，作为指向冷存储位置和索引元数据（如日志类型和索引开始/结束日期）的指针。目录存储应该是持久的；我们通常使用 Cassandra 来实现此目的。

搜索数据节点
每个搜索数据节点负责提供预计算 Lucene 分片的一部分。它从协调器（见下文）获取相关分片的分配，然后从冷存储获取相应的索引，并暴露一个 RPC API 用于搜索所服务的索引。

搜索协调器
协调器跟踪可用的搜索节点，并管理相关索引分片到节点的分配。我们可以在这里调整许多参数，特别是分配的冗余级别。由于所有数据都是不可变的，多个搜索节点提供相同的索引分片是可以接受的，而无需任何复杂的同步逻辑。索引分片分配遵循每个索引的保留设置；例如，它可能会通告某个日志类型过去 2 天的数据分片，或者另一个日志类型过去 7 天的分片。

搜索聚合器节点
聚合器暴露一个 RPC 查询 API（例如 HTTP/JSON 或 protobuf），并将传入的查询请求转发给数据节点。它从协调器了解到哪些数据节点提供的分片与关于日志类型和查询时间窗口的查询相关。聚合器实现了查询语言：它转发过滤查询，对聚合查询结果应用限制，并对结果进行去重（见下文）。

去重
由于日志是不可变的，通过它们的内部 ID（例如日志记录的哈希值）对搜索结果进行去重是很简单的。去重的必要性可能不是很明显，但考虑以下边缘情况作为例子：一个索引器成功地将一个索引分片推送到冷存储并在目录中注册，但流提交失败。然后另一个索引器节点可能会从流中提取相同的日志记录，并将它们作为额外的、部分重复的索引分片推送。这种模式并非日志索引工作流或所呈现的架构所独有：分布式系统中的“恰好一次”处理需要带有提交协议的协调事务，或者下游幂等处理。由于我们的记录是不可变的，后一种选择实现起来要简单得多。

低延迟索引
所述的数据流具有受最大索引分片年龄（例如 1 小时）限制的最坏情况下的端到端延迟。这对于调试和日志追踪工作流来说当然是不够的。为了降低延迟，我们添加了一组额外的低延迟索引，它们消费日志流的单独分片副本，并为最近几个小时的日志数据生成本地的、短生命的索引。这些同置的索引和搜索节点乍一看可能非常类似于 Elasticsearch 模型，但有两个关键区别：首先，它们只保留几个小时的日志；其次，它们不必处理任何协调的复杂性。

保留策略和可调整的成本与性能

所述系统在三个地方保存索引数据：首先，在内存或磁盘上的低延迟索引中；其次，在搜索数据节点的磁盘上；第三，在冷存储（S3）中。这种存储层次结构允许我们根据按时间统计的日志访问频率，针对基础设施成本调整搜索性能：绝大多数查询是针对近期日志的，通常是过去一小时的，有时是当天的；只有极少数情况下，用户对几天前的数据感兴趣。因此，我们利用在内存或快速 SSD 磁盘上保存过去一小时数据的低延迟索引来处理大量查询。接下来，我们利用搜索数据节点磁盘上的索引来处理针对当天数据的查询。最后，针对旧数据的查询由搜索数据节点按需准备的索引来处理：当这样的查询进入系统时，协调器将相关索引分片添加到分配中，搜索 UI 向用户指示正在准备相关数据，然后将查询分派到后端。相关分片由数据节点缓存，因此针对相同时间范围的额外查询可以立即分派。我们发现大多数此类按需查询可以在几分钟内得到回答。

存储层次结构提供了三个不同的保留窗口，可以进行调整以达到可接受的成本/性能权衡：低延迟索引中的数据最大年龄、搜索数据节点的默认分配中的数据最大年龄以及冷存储中的数据最大年龄。前两个参数仅影响性能，而后者决定了可供用户搜索的数据的最大年龄。

按需工作流显然受益于一种允许我们只加载少量包含给定查询相关数据子集的索引分片的索引策略。这里要调整的明显参数是时间范围。此外，由于日志可以归因于生成它的源服务或组件以及环境，我们按（时间范围、组件、环境）对索引进行分片，并要求按需工作流的用户在其查询中指定这些参数。我们发现这提供了一个可接受的可用性与延迟的权衡。

经验教训

我们正在从 Elasticsearch 后端切换到所描述的 Lucene 后端。到目前为止，重新设计的架构看起来很有前景：它更稳定，并且在我们当前的保留设置下，所需的计算资源量显著减少了约 50%。更重要的是，这次重新设计是实现日志基础设施面向未来的关键一步，因为我们预计明年的日志量将增长 2 到 3 倍，而当前规模下 Elasticsearch 后端的可行性极限已经触顶。

回想起来，最初选择 Elasticsearch 是正确的：我们依托稳定的数据基础快速起步，并有时间在已经能够向我们的开发团队提供针对服务日志的查询能力的同时，对基础设施的其他部分（日志格式和库、白名单/黑名单、UI 等）进行迭代。

随着我们需求的变化和数据规模的增长，架构重新设计是顺理成章的下一步。它也是实现细粒度访问控制、自动扩缩容和速率限制等功能的起点。见证一个小型团队的两三名工程师如何在几个月内重新思考和重建我们数据基础设施的重要组成部分，真是一件乐事。不要害怕着手执行这样的任务，即使这意味着要用 Lucene 和自定义协调逻辑等更底层的组件来替换像 Elasticsearch 这样成熟的系统。当然，站在巨人的肩膀上总是有益的，但有时候也要三思，以挑选合适的泰坦家族。
