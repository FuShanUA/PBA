代码审查最佳实践

动机

我们进行代码审查（CR）是为了提升代码质量，并得益于其对团队和公司文化的积极影响。例如：

  • 提交者的动力：想到会有一组审查者来检查变更请求，提交者往往会清理遗留问题、整合 TODO，并从整体上完善提交。通过同行认可其编码专业能力，是许多程序员的骄傲之源。
  • 知识共享：这能在多个方面帮助开发团队：
  • 代码审查能将新增、修改或移除的功能明确传达给团队成员，便于他们在此基础上推进工作。
  • 提交者使用的某种技术或算法可能会让审查者有所启发。更广泛地说，代码审查有助于提升整个组织的质量标准。
  • 审查者可能掌握某些编程技术或代码库的知识，能帮助改进或巩固变更；例如，其他人可能正在并行开发类似的功能或修复。
  • 积极的互动与沟通能增强团队成员间的社会联系。
  • 一致性：代码库的一致性使代码更易读、易懂，有助于预防 Bug，并促进常驻开发者和流动性开发者之间的协作。
  • 可读性：对于代码段的作者来说，由于这是他们的心血结晶，很难评判其可读性；而对于没有完整上下文的审查者来说则容易得多。可读性高的代码更具复用性、更少 Bug 且更具前瞻性。
  • 意外错误（如拼写错误）以及结构性错误（如死代码、逻辑或算法 Bug、性能或架构隐患），往往更容易被具有外部视角的挑剔审查者发现。研究表明，即使是简短的非正式代码审查，也能对代码质量和 Bug 发生率产生显著影响。
  • 合规性与监管环境通常要求进行审查。代码审查是避免常见安全陷阱的绝佳方式。如果你的功能或环境有严格的安全要求，它将受益于（且可能必须经过）本地安全专家的审查（OWASP 的指南是该流程的一个很好的范例）。

审查什么

这个问题没有绝对正确的答案，每个开发团队都应商定自己的方法。有些团队倾向于审查合入主干分支的每一处变更，而其他团队则设定一个“琐碎性”阈值，低于该阈值则无需审查。这需要在有效利用工程师（提交者和审查者）的时间与维护代码质量之间进行权衡。在某些受监管的环境中，即使是琐碎的变更也可能需要进行代码审查。

代码审查不分等级：身为团队中最资深的员工，并不意味着你的代码就不需要审查。即使在极少数情况下代码完美无瑕，审查也提供了指导和协作的机会，并且至少能分散代码库中的代码理解权。

何时审查

代码审查应在自动化检查（测试、风格检查及其他 CI 流程）成功通过后，但在代码合入仓库主干分支之前进行。我们通常不会对自上次发布以来的汇总变更进行正式的代码审查。

对于需要作为单个单元合入主干分支但太大而无法放入一次合理代码审查的复杂变更，可以考虑堆叠式 CR 模型：创建一个主分支 feature/big-feature 和若干辅助分支（例如 feature/big-feature-api、feature/big-feature-testing 等），每个辅助分支封装一部分功能，并针对 feature/big-feature 分支单独进行代码审查。一旦所有辅助分支合入 feature/big-feature 后，再创建一个 CR，将后者合入主干分支。

准备待审查的代码

提交易于审查的 CR 是作者的责任，以免浪费审查者的时间和动力：

  • 范围与规模。变更应具有狭窄、定义明确且自包含的范围，并在此范围内详尽覆盖。例如，一个变更可以实现新功能或修复 Bug。较短的变更优于较长的变更。如果一次 CR 对超过约 5 个文件进行了实质性更改，或者编写耗时超过 1-2 天，或者审查时间超过 20 分钟，请考虑将其拆分为多个自包含的 CR。例如，开发人员可以提交一个变更，通过接口和文档定义新功能的 API，然后再提交第二个变更，为这些接口添加实现。
  • 只提交完整的、（通过 diff）自审的以及自测的 CR。为了节省审查者的时间，在指定审查者之前，请测试提交的变更（即运行测试套件），并确保它们在本地和 CI 服务器上都能通过所有构建、测试和代码质量检查。
  • 重构变更不应改变行为；反之，改变行为的变更应避免重构和代码格式化。这样做有几个充分的理由：
  • 重构变更通常涉及大量代码行和文件，因此受关注的程度较低。无意的行为改变可能会在无人察觉的情况下混入代码库。
  • 大规模重构变更会破坏 cherry-picking、rebase 等源代码控制魔法。撤销作为全仓库重构提交的一部分而引入的行为变更是非常麻烦的。
  • 宝贵的人工审查时间应花在程序逻辑上，而不是风格、语法或格式争论上。我们倾向于使用 Checkstyle、TSLint、Baseline、Prettier 等自动化工具来解决这些问题。

提交信息

以下是一个遵循广泛引用的标准的好提交信息示例：

首字母大写、简短（不超过 80 个字符）的摘要

如有必要，提供更详细的解释性文本。将其换行至约 120 个字符左右。在某些语境下，第一行被视为邮件主题，其余文本视为正文。将摘要与正文分开的空行至关重要（除非你完全省略正文）；如果将两者连在一起，rebase 等工具可能会感到困惑。

寻找审查者

由提交者提议一两位熟悉代码库的审查者是惯例。通常，其中一位审查者是项目负责人或高级工程师。项目数据主人应考虑订阅其项目，以便在有新的 CR 时收到通知。超过三方的代码审查通常效率低下甚至适得其反，因为不同的审查者可能会提出相互矛盾的修改意见。这可能意味着对正确的实现存在根本性分歧，应在代码审查之外的高带宽论坛中解决，例如面对面或与所有相关方召开视频会议。

执行代码审查

代码审查是不同团队成员之间的同步点，因此有可能阻碍进度。因此，代码审查需要及时（以小时计，而非天计），团队成员和负责人需意识到这一时间投入，并据此安排审查时间的优先级。如果你认为自己无法按时完成审查，请立即告知提交者，以便他们另寻他人。

审查应足够彻底，使得审查者能够向另一位开发者合理详细地解释该变更。这确保了代码库的细节为不止一人所知。

作为审查者，你有责任执行编码标准并保持高质量。代码审查与其说是一门科学，不如说是一门艺术。学习的唯一方法就是去实践；有经验的审查者应考虑让经验较少的审查者参与他们的变更，并让他们先进行审查。假设作者遵循了上述指南（特别是关于自审和确保代码运行方面），以下是审查者在代码审查中应关注的事项清单：

目的

  • 这段代码是否实现了作者的目的？ 每次变更都应有特定原因（新功能、重构、Bug 修复等）。提交的代码是否真正实现了这一目的？
  • 提出问题。 函数和类的存在必有原因。当审查者不清楚原因时，这可能表明代码需要重写，或者需要通过注释或测试来提供支持。

实现

  • 思考你会如何解决这个问题。 如果方法不同，为什么？你的代码是否处理了更多（边界）情况？是否更短/更易/更简洁/更快/更安全且功能等效？你是否发现了当前代码未捕捉到的某种底层模式？
  • 你是否看到了有用抽象的潜力？ 部分重复的代码通常表明可以提取出更抽象或通用的功能片段，然后在不同上下文中复用。
  • 像对手一样思考，但态度要友好。 试着通过构思有问题的配置/输入数据来破坏作者的代码，从而“抓住”他们走捷径或遗漏案例。
  • 思考库或现有的产品代码。 当有人重新实现已有功能时，往往只是因为他们不知道该功能已经存在。有时，代码或功能是故意重复的，例如为了规避依赖。在这种情况下，代码注释可以澄清意图。引入的功能是否已由现有的库提供？
  • 变更是否遵循了标准模式？ 既有的代码库通常在命名约定、程序逻辑分解、数据类型定义等方面展现出模式。通常希望变更的实现能与现有模式保持一致。
  • 变更是否增加了编译时或运行时依赖（尤其是子项目之间）？ 我们希望保持产品松散耦合，依赖尽可能少。对依赖项和构建系统的变更应受到严格审查。

可读性与风格

  • 思考你的阅读体验。 你是否在合理的时间内掌握了概念？流程是否合理，变量和方法名是否易于理解？你是否能在多个文件或函数之间保持思路连贯？不一致的命名是否让你感到困扰？
  • 代码是否遵循了编码指南和代码风格？ 代码在风格、API 约定等方面是否与项目保持一致？如上所述，我们倾向于使用自动化工具来解决风格争论。
  • 这段代码有 TODO 吗？ TODO 只会在代码中堆积，并随时间推移变得陈旧。请作者在 GitHub Issues 或 JIRA 中提交一个工单，并将问题编号附在 TODO 后。提议的代码变更不应包含被注释掉的代码。

可维护性

  • 阅读测试。 如果没有测试而本应该有，请要求作者编写。真正不可测试的功能很少，而不幸的是，未经测试的功能实现却很常见。检查测试本身：它们是否覆盖了有趣的用例？它们可读吗？这次 CR 是否降低了整体测试覆盖率？想想这段代码可能会怎样出问题。测试的风格标准通常与核心代码不同，但同样重要。
  • 此 CR 是否引入了破坏测试代码、预发环境或集成测试的风险？ 这些通常不作为提交前/合入前检查的一部分，但它们出问题对所有人都是痛苦的。需要特别注意的具体事项包括：移除测试工具或模式、配置变更以及构件布局/结构变更。
  • 此变更是否破坏了向后兼容性？ 如果是，此时合入变更是否合适，还是应推迟到后续版本中？破坏可能包括数据库或模式变更、公共 API 变更、用户工作流程变更等。
  • 这段代码需要集成测试吗？ 有时候，仅靠单元测试无法充分测试代码，尤其是当代码与外部系统或配置交互时。
  • 对代码级文档、注释和提交信息留下反馈。 冗余的注释会使代码杂乱，而简略的提交信息会让未来的贡献者感到困惑。这并不总是适用，但高质量的注释和提交信息在未来会物超所值。（想想你曾见过的极佳或极差的提交信息或注释。）
  • 外部文档是否已更新？ 如果你的项目维护了 README、CHANGELOG 或其他文档，它们是否已更新以反映这些变更？过时的文档可能比没有文档更令人困惑，且未来修复的成本将高于现在更新的成本。

不要忘记赞扬简洁/可读/高效/优雅的代码。相反，驳回或不批准 CR 并非无礼之举。如果变更冗余或无关，请附上说明予以驳回。如果由于一个或多个致命缺陷你认为其不可接受，请予以不批准，同样需附上说明。有时，CR 的正确结果是“我们换个完全不同的方式来做”，甚至是“我们根本不要做这个”。

尊重被审查者。虽然对抗性思维很管用，但这不是你的功能，你也不能做所有决定。如果你无法在代码现状下与被审查者达成一致，请切换到实时沟通或寻求第三方意见。

安全

验证 API 端点是否执行了与代码库其余部分一致的适当授权和身份验证。检查其他常见弱点，例如配置薄弱、恶意用户输入、缺失日志事件等。如有疑问，请将 CR 提交给应用安全专家。

注释：简洁、友好、可执行

审查应简洁并使用中性的语言。批评代码，而不是作者。当某些事情不清楚时，要求澄清而不是假设对方无知。避免使用物主代词，特别是在与评估结合时：“在你的变更之前，我的代码是好的”、“你的方法有 Bug”等。避免绝对判断：“这绝不可能工作”、“结果总是错的”。

尝试区分建议（例如，“建议：提取方法以提升可读性”）、必需的修改（例如，“添加 @Override”）以及需要讨论或澄清的要点（例如，“这真的是正确的行为吗？如果是，请添加注释解释逻辑。”）。考虑提供链接或指针，以深入解释某个问题。

当你完成代码审查后，向作者表明你希望在多大程度上让其对你的评论作出回应，以及在实施变更后是否希望再次审查 CR（例如，“回复几个小建议后即可合入”对比“请考虑我的建议，并告知我何时可以再看一次。”）。

回应审查

代码审查的部分目的在于改进作者的变更请求；因此，不要对审查者的建议感到冒犯，即使你不同意也要认真对待。回复每一条评论，即使只是简单的“确认”或“完成”。解释你为何做出某些决定，为何存在某个函数等。如果你无法与审查者达成一致，请切换到实时沟通或寻求外部意见。

修复应推送到同一分支，但放在单独的提交中。在审查过程中压缩提交会让审查者难以跟进变更。

不同团队有不同的合入策略：有些团队只允许项目数据主人合入，而其他团队则允许贡献者在获得积极的代码审查后合入。

面对面代码审查

对于大多数代码审查，基于异步 diff 的工具（如 Reviewable、Gerrit 或 GitHub）是绝佳选择。对于复杂的变更，或者在专业知识或经验差异很大的各方之间进行审查，面对面进行可能更高效，可以是在同一屏幕或投影仪前，也可以是通过视频会议或屏幕共享工具远程进行。

  private int uniqueUsersCount;

  /** 如果无法提取 s，则返回 {@link Optional#empty}。 */

  /** 如果 {@code s} 无法重写，则返回 null。 */
  //R: 应统一返回值：此处也应使用 Optional<>
  public String rewriteString(String s);
