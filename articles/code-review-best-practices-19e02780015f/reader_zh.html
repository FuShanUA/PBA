动机

我们进行代码审查 (CR) 旨在提升代码质量，并借此对团队和公司文化产生积极影响。例如：

  • 提交者会因此受到激励，因为他们知道会有一组审查者来审视变更请求：提交者往往会清理遗留问题，整合 TODO 项，并从整体上完善提交。通过同行认可编程专长，对许多程序员而言是自豪感的源泉。
  • 知识共享通过多种方式赋能开发团队：
  • 代码审查会向团队成员明确传达新增、修改或移除的功能，后续成员便可在此基础上推进工作。
  • 提交者采用的技术或算法可能值得审查者借鉴。更广泛地说，代码审查有助于提升整个组织的质量标准。
  • 审查者可能掌握某些编程技术或对代码库更熟悉，从而帮助改进或整合变更；例如，其他人可能正并行开发类似的功能或修复方案。
  • 积极的互动与沟通能增强团队成员间的社交纽带。
  • 一致性能让代码库更易于阅读和理解，有助于预防缺陷，并促进常规开发人员与流动性开发人员之间的协同工作。
  • 可读性对于孕育代码的作者而言往往难以评判，但对于缺乏完整上下文的审查者来说却显而易见。可读性强的代码更具复用性、更少缺陷，且更经得起时间考验。
  • 偶然性错误（如拼写错误）和结构性错误（如死代码、逻辑或算法缺陷、性能或架构隐患）往往更容易被具备外部视角的严苛审查者发现。研究表明，即使是短暂且非正式的代码审查，也能对代码质量和缺陷频率产生显著影响。
  • 合规性与监管环境通常要求进行审查。代码审查是规避常见安全陷阱的绝佳手段。如果你的功能或环境有严格的安全要求，它将受益于（甚至必须经过）安全专家的审查（OWASP 的指南是该流程的绝佳范例）。

审查内容

这个问题没有永恒的标准答案，各开发团队应自行商定方法。有些团队倾向于审查所有合并至主分支的变更，而另一些团队则设定“微不足道”的阈值，低于此阈值则无需审查。这需要在有效利用工程师（包括作者和审查者）的时间与维持代码质量之间取得平衡。在特定的监管环境下，即便是微小变更也可能需要代码审查。

代码审查不分层级：身为团队中最资深的成员，并不意味着你的代码就不需要审查。即便在极少数情况下代码完美无瑕，审查过程也提供了指导和协作的机会，且至少能让代码库中的代码认知更加多元化。

审查时机

代码审查应在自动化检查（测试、风格检查、其他持续集成）成功通过后，但在代码合并至仓库主分支之前进行。我们通常不对自上次发布以来的累积变更进行正式的代码审查。

对于需要作为整体单元合并至主分支的复杂变更，若规模过大无法一次性纳入合理的代码审查，可考虑采用堆叠式审查模型：创建主分支 feature/big-feature 以及若干次级分支（如 feature/big-feature-api、feature/big-feature-testing 等），每个次级分支封装一部分功能，并单独针对 feature/big-feature 分支进行代码审查。待所有次级分支合并至 feature/big-feature 后，再创建将其合并至主分支的代码审查。

代码审查前的准备

提交易于审查的代码审查请求是作者的责任，以免浪费审查者的时间和精力：

  • 范围与规模。变更应具备聚焦、明确且自包含的范围，并完整覆盖该范围。例如，一次变更可以实现新功能或修复缺陷。短篇变更优于长篇。如果一次代码审查对超过约 5 个文件进行了实质性修改，或编写耗时超过 1-2 天，或审查需超过 20 分钟，请考虑将其拆分为多个自包含的代码审查。例如，开发者可以提交一次变更，通过接口和文档定义新功能的 API，再提交第二次变更以添加这些接口的实现。
  • 仅提交完整的、已自我审查（通过 diff）且已自我测试的代码审查。为节省审查者时间，在指定审查者之前，请测试提交的变更（即运行测试套件），确保它们在本地和 CI 服务器上均能通过所有构建、测试及代码质量检查。
  • 重构变更不应改变行为；反之，行为变更应避免包含重构和代码格式化。原因如下：
  • 重构变更通常涉及大量代码行和文件，审查者往往会不够关注。无意的行为变更可能在无人察觉的情况下混入代码库。
  • 大规模重构变更会破坏 cherry-picking、rebase 等源代码控制操作。要撤销作为全仓库重构提交的一部分而引入的行为变更，过程将极其繁杂。
  • 宝贵的人工审查时间应投入到程序逻辑上，而非风格、语法或格式争论。我们更倾向于使用 Checkstyle、TSLint、Baseline、Prettier 等自动化工具来解决这些问题。

寻找审查者

由提交者提议一两位熟悉代码库的审查者是常规做法。通常，其中一位审查者为项目负责人或资深工程师。项目数据主人应考虑订阅他们的项目，以便在新代码审查提出时收到通知。超过三方参与的代码审查往往效率低下甚至适得其反，因为不同审查者可能提出相悖的修改建议。这可能意味着对正确的实现存在根本性分歧，应在代码审查之外的高带宽沟通渠道中解决，例如当面对面会议或涉及所有相关方的视频会议。

执行代码审查

代码审查是团队成员间的同步节点，因此有可能阻碍进度。因此，代码审查必须及时（以小时计，而非天计），团队成员和负责人需意识到这一时间投入，并据此优先安排审查时间。若认为自己无法按时完成审查，请立即告知提交者，以便其另寻他人。

审查应足够详尽，使审查者能以合理的细节向其他开发者解释该变更。这确保代码库的细节不只掌握在单个人手中。

作为审查者，你有责任贯彻编码标准并保持高质量。代码审查与其说是一门科学，不如说是一门艺术。唯一的学习途径就是实践；经验丰富的审查者应考虑让经验不足的审查者先审查自己的变更。假设作者已遵循上述指南（尤其是自我审查和确保代码运行），以下列出审查者在代码审查中应关注的要点：

目的

  • 这段代码是否达成了作者的目的？ 每次变更都应有明确的原因（新功能、重构、修复缺陷等）。提交的代码是否真正实现了该目的？
  • 提出问题。 函数和类的存在必有原因。当审查者不明白原因时，可能意味着代码需要重写，或者需要通过注释或测试来提供支撑。

实现

  • 思考你会如何解决该问题。 如果方法不同，原因何在？你的代码是否处理了更多（边缘）情况？是否更短、更简单、更清晰、更快速或更安全，且功能等效？你是否发现了当前代码未捕捉到的某些潜在模式？
  • 是否看到了有用的抽象潜力？ 部分重复的代码往往暗示可以提取出更抽象或通用的功能模块，并在不同场景中复用。
  • 像对手一样思考，但态度要友善。 试图通过构造有问题的配置或输入数据来“抓出”作者走捷径或遗漏的用例，以此破坏其代码。
  • 思考可用的库或现有产品代码。 当有人重新实现现有功能时，往往只是因为他们不知道该功能已经存在。有时，代码或功能的重复是有意为之，例如为了避免依赖。在这种情况下，代码注释可以澄清意图。引入的功能是否已由现有库提供？
  • 变更是否遵循了标准模式？ 成熟的代码库通常在命名约定、程序逻辑分解、数据类型定义等方面展现出特定模式。通常期望变更能遵循现有模式来实现。
  • 变更是否引入了编译时或运行时依赖（尤其是子项目之间）？ 我们希望保持产品松散耦合，依赖越少越好。对依赖项和构建系统的变更应受到严格审查。

可读性与风格

  • 思考你的阅读体验。 你是否在合理时间内理解了概念？流程是否合理，变量和方法命名是否易于理解？你能否在多个文件或函数间保持思路连贯？是否存在因不一致的命名而让你感到困扰的情况？
  • 代码是否遵循了编码规范和代码风格？ 代码在风格、API 约定等方面是否与项目保持一致？如前所述，我们倾向于通过自动化工具解决风格争论。
  • 这段代码是否包含 TODO？ TODO 在代码中只会堆积，并随着时间推移变得过时。请让作者在 GitHub Issues 或 JIRA 中提交工单，并将问题编号附在 TODO 后。提议的代码变更不应包含被注释掉的代码。

可维护性

  • 阅读测试。 如果没有测试但本应有，请要求作者编写。真正无法测试的功能极少，遗憾的是，未经测试的功能实现却屡见不鲜。检查测试本身：它们是否覆盖了有意义的用例？是否可读？该代码审查是否降低了整体测试覆盖率？思考这段代码可能崩溃的方式。测试的规范标准通常与核心代码不同，但同样重要。
  • 此次代码审查是否带来了破坏测试代码、预发布环境或集成测试的风险？ 这些通常不在提交前或合并前检查范围内，但一旦停摆，大家都会很痛苦。需要特别注意的具体方面包括：测试工具或模式的移除、配置变更，以及产物布局或结构的变更。
  • 此变更是否破坏了向后兼容性？ 如果是，此时合并是否合适，还是应推迟到后续版本？破坏可能包括数据库或模式变更、公共 API 变更、用户工作流变更等。
  • 此代码是否需要集成测试？ 有时代码仅靠单元测试无法充分测试，尤其是与外部系统或配置交互的代码。
  • 对代码级文档、注释和提交信息提出反馈。 冗余的注释会使代码杂乱，而简短的提交信息会让未来的贡献者感到困惑。这并非总是适用，但高质量的注释和提交信息在未来定会物超所值。（回想一下你见过的出色或极糟的提交信息或注释。）
  • 外部文档是否已更新？ 如果你的项目维护了 README、CHANGELOG 或其他文档，是否已更新以反映这些变更？过时的文档可能比没有文档更令人困惑，且未来修复的成本远高于现在及时更新。

不要忘记对简洁、可读、高效或优雅的代码给予赞赏。反之，拒绝或否决代码审查并不粗鲁。如果变更是冗余或无关的，请附带说明予以拒绝。如果由于存在致命缺陷而认为其不可接受，请同样附带说明予以否决。有时代码审查的正确结果是“让我们用完全不同的方式来做”，甚至是“让我们根本不要做这个”。

尊重被审查者。尽管对抗性思维很有用，但这并非你的功能，你也不能做所有决定。如果无法与被审查者就当前代码达成一致，请切换到实时沟通或寻求第三方意见。

安全

验证 API 端点是否执行了与代码库其余部分一致的授权和身份验证。检查其他常见弱点，例如配置薄弱、恶意用户输入、缺失日志事件等。如有疑问，请将代码审查转交给应用安全专家。

评论：简明、友善、可操作

审查意见应简明扼要，并使用中立语言。批评代码，而非作者。遇到不清楚的地方，应要求澄清，而非臆断对方无知。避免使用物主代词，尤其是在评价时：“我的代码在你的变更之前是正常的”、“你的方法有缺陷”等。避免使用绝对化判断：“这绝不可行”、“结果总是错的”。

尽量区分建议（例如，“建议：提取方法以提升可读性”）、必须修改项（例如，“添加 @Override”）以及需要讨论或澄清的要点（例如，“这真的是正确的行为吗？如果是，请添加注释解释逻辑。”）。考虑提供指向问题深入解释的链接或指引。

完成代码审查后，应表明你希望作者在何种程度上回应你的评论，以及在实现变更后是否希望再次审查（例如，“回应几个小建议后即可合并”与“请考虑我的建议并告知何时可以再审”）。

回应审查

代码审查的部分目的在于改进作者的变更请求；因此，不要对审查者的建议感到被冒犯，即使不认同也要认真对待。回应每一条评论，哪怕只是简单的“确认”或“已处理”。解释你做出某些决定的原因、某个函数存在的理由等。如果无法与审查者达成共识，请切换到实时沟通或寻求外部意见。

修复内容应推送到同一分支，但需放在独立的提交中。在审查过程中压缩提交会使审查者难以跟进变更。

不同团队有不同的合并策略：有些团队仅允许项目数据主人执行合并，而有些团队则在代码审查通过后允许贡献者自行合并。

面对面代码审查

对于大多数代码审查，诸如 Reviewable、Gerrit 或 GitHub 等基于 diff 的异步工具是绝佳选择。但对于复杂变更，或参与者专业背景和经验差异较大的审查，面对面进行会更高效，可以在同一屏幕或投影仪前，或通过视频会议或屏幕共享工具远程进行。

命名不一致

  private int uniqueUsersCount;

  /** 如果无法重写 {@code s}，则返回 null。 */
  //R: 应统一返回值：此处也应使用 Optional<>
  public String rewriteString(String s);
