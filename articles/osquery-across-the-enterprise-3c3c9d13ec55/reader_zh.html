任何高效的事件响应团队都需要具备一种能力：能够向整个机群中的单个或多个主机“提出问题”，并及时获得准确答复。要在成千上万台主机上检测事件并做出响应，就必须深入理解用户、应用程序和设备层面的各类操作与行为。尽管端点检测与防护工具能开箱即用地提供一定助力，但对安全相关事件进行深度洞察与分析，对于检测高级威胁至关重要。目前，Palantir 在 Windows、Mac 和 Linux 系统中维护了一套 osquery 部署体系，专门用于解答这些问题。

osquery 最初由 Facebook 开发，是一款开源工具，它将操作系统配置数据以关系型数据库表的形式暴露出来。通过对这些表执行类似 SQL 的查询，用户可以收集有关系统当前状态及其随时间推移所发生变化的有价值数据。

本篇博文有两个目标：第一，为多平台 osquery 部署提供配置指导；第二，介绍我们开源的 osquery 配置集：https://github.com/palantir/osquery-configuration 。该 GitHub 项目提供了必要的基础构件，可作为组织快速评估并将 osquery 部署到生产环境的有用参考。我们的配置代表了一项基线安全标准，若与集中式日志平台结合使用，可在检测与响应方面立即产生安全成效。

2017 年发生了几起备受瞩目的公共安全事件，这些事件要求我们对端点和应用程序配置进行快速响应和深度自查：

  • 公开披露了允许技术不高的攻击者利用 Windows 系统未知漏洞的工具
  • 数个 Chrome 扩展程序遭劫持以传播恶意代码
  • 某个版本的 CCleaner.exe 被植入恶意后门
  • 在 dnsmasq 这款广泛使用的应用中发现了多个远程代码执行（RCE）漏洞

上述每起事件都要求我们具备向整个机群提出一系列“问题”的能力，以便识别受波及的系统。

案例研究：dnsmasq 漏洞

在 Google 安全团队发布博文详细介绍 dnsmasq 的众多漏洞后，我们的信息安全团队迅速采取行动，移除所有安装了 dnsmasq 的主机，并在仍需使用该组件的地方将其升级至已修复版本。让情况变得更加复杂的是，一些开发团队将 dnsmasq Docker 镜像作为其开发工作流的一部分，因此识别 dnsmasq 安装情况绝非简单搜索已安装程序那么容易。为了确保能彻底清点所有安装了任何形式 dnsmasq 的主机，我们使用了四个独立的查询：

  • 通过枚举相关的 launchd plist 来查找经由 Homebrew 或 MacPorts 安装的 dnsmasq：
    SELECT * FROM launchd WHERE name LIKE '%dnsmasq%';
  • 查找名称中包含 dnsmasq 的正在运行的 Docker 容器：
    SELECT name FROM docker_containers WHERE name LIKE '%dnsmasq%';
  • 发现在 localhost 53 端口上监听 dnsmasq 的主机：
    SELECT DISTINCT(processes.name), process_open_sockets.local_port FROM processes JOIN process_open_sockets USING (pid) WHERE local_port=53 AND processes.name='dnsmasq';
  • 查找通过 Homebrew 安装 dnsmasq 的用户：
    SELECT * FROM homebrew_packages WHERE name='dnsmasq';

这种方法比简单枚举主机上已安装的软件包要全面得多。在开始之前，我们已有其中两个查询就位，增加另外两个针对 dnsmasq 的特定查询只花了几分钟时间。不到 24 小时，我们就获得了一份安装了 dnsmasq 的主机完整清单，并可以针对性地进行移除或升级。

osquery 基础

表

osquery 将系统的日志和状态信息收集并聚合到一组预定义的表中。用户可以通过对这些表执行 SQL 查询来探查系统状态。查询可以通过 osqueryi（一个交互式 SQL 环境）或 osqueryd（一个用于执行重复性计划查询的长期运行守护进程）来发起。

以下是一个查询给定系统主机名和 CPU 类型的示例：

$ osqueryi
osquery> SELECT hostname, cpu_type from system_info;
+---------------+----------+
| hostname      | cpu_type |
+---------------+----------+
| mycomputer    | x86_64   |
+---------------+----------+

可使用 .tables 命令来探索可用表的列表：

osquery> .tables
  => acpi_tables
  => ad_config
  => alf
  => alf_exceptions
  => alf_explicit_auths
  <snip>

每个表都有指定的 https://osquery.io/schema/ 架构，其中标明了可用列、列类型以及描述性细节。可使用 .schema 命令查看表的架构：

osquery> .schema os_version;
CREATE TABLE os_version(
`name` TEXT,
 `version` TEXT,
`major` INTEGER,
`minor` INTEGER,
`patch` INTEGER,
`build` TEXT,
`platform` TEXT,
`platform_like` TEXT,
`codename` TEXT
);

除了简单的单表查询外，osquery 还能跨多个表聚合和关联信息，例如使用 SQL JOIN 命令：

osquery> SELECT hash.md5, path, file.filename, file.uid, file.gid 
         FROM hash 
         JOIN file USING (path) 
         WHERE path='/bin/ls';
+---------------------------------+---------+----------+-----+-----+
| md5                             | path    | filename | uid | gid |
+---------------------------------+---------+----------+-----+-----+
| 528722ae3e3e6087b453560e8d025f76| /bin/ls | ls       | 0   | 0   |
+---------------------------------+---------+----------+-----+-----+

基于事件的表

上述标准表的内容是在对表执行查询时填充的。这种模型使得持续监控系统属性变得困难。例如，要随时间推移维护运行进程列表，就需要用户以极短的间隔调度执行 SELECT * FROM processes 形式的查询。即便如此，短暂运行的进程仍可能成为漏网之鱼。

基于事件的表通过近实时收集和存储事件来弥补这一缺陷。这些表确保在设定的查询间隔期间发生的事件也能被收集到表中，并根据用户自定义的过期选项进行清理。任何以 _events 结尾的 osquery 表都是基于事件的表，例如 file_events、hardware_events 和 user_events。osquery 文档中描述了事件过期机制。

查询与日志记录

osqueryd 是 osqueryi 的守护进程版本，用于运行计划查询。在标准配置下，你需向 osqueryd 提供一个配置文件，其中包含一系列查询及相应的计划安排。产生的事件随后将被记录到文件系统中。以下是一个配置文件示例：

根据所需功能，osquery 支持三种不同的事件记录方式。这些日志类型分别为：

  • 差分：缓存初始结果，后续查询将仅报告自上次查询以来的变化。
  • 差分（忽略移除）：与差分方式相同，但仅报告表中新增的内容。
  • 快照：不缓存查询结果，每次查询均报告查询时刻的当前状态。

差分查询

osquery 的默认日志记录方法是差分，这些查询的结果会被写入 osqueryd.results.log。差分查询非常适合用于了解事件何时发生以及发生了什么变化。例如，以下计划查询用于确定当前安装的操作系统的版本：

示例配置查询：
"os_version": {
  "query": "SELECT name, version FROM os_version;",
  "interval": 86400,
  "description": "Record the version of the OS when an upgrade occurs"
  }

示例查询结果：
{    
   action: removed   
   calendarTime: Wed Jan 1 12:30:00 2017 UTC    
   columns:    {       
     name: Microsoft Windows 7 Enterprise       
     version: 6.1.7601    
  }    
   hostIdentifier: sample-hostname    
   name: os_version       
}
----------------------------------------------
{    
   action: added    
   calendarTime: Wed Jan 1 12:30:00 2017 UTC    
   columns:    {       
     name: Microsoft Windows 10 Enterprise       
     version: 10.0.14393    
  }    
   hostIdentifier: sample-hostname    
   name: os_version       
}

该查询首次运行时，由于没有 os_versions 表中的先前数据可供比对，它会记录当前的操作系统版本。尽管该查询计划每 86400 秒（24 小时）运行一次，但只有在操作系统名称或版本发生变化时才会生成事件。发生升级时，将生成两个日志事件：旧操作系统版本的日志操作字段会被标记为“removed”，而新版本的该字段则会被标记为“added”。

这种差分查询有助于确定特定主机何时升级了操作系统以及升级到了哪个版本，但在确定给定时间点的操作系统版本方面则用处不大。如果你的 osquery 日志是集中管理的，可能需要向前追溯很长时间的日志，才能找到每台主机最近的操作系统版本升级事件，进而确定各主机当前运行的操作系统版本。

快照查询

快照查询用于描述特定时间点的表状态。这些查询的结果会被写入一个单独的日志文件：osqueryd.snapshots.log。对查询启用快照功能后，每次运行查询都会返回表的完整内容，无论结果在这段时间内是否发生了变化。例如，以下计划查询用于确定当前安装的操作系统的版本：

示例配置查询：
"os_version_snapshot": {
  "query": "SELECT name, version FROM os_version;",
  "interval": 86400,
  "description": "Record the currently installed OS version",
  "snapshot": true
  }

示例查询结果：
{    
   action: snapshot    
   calendarTime: Wed Jan 1 12:30:00 2017 UTC     
   hostIdentifier: sample-hostname    
   name: os_version_snapshot    
   snapshot:[       
    {       
       name: Microsoft Windows 10 Enterprise    
       version: 10.0.14393    
    }    
   ]       
}

使用快照后，操作系统名称和版本每 86400 秒（24 小时）就会被记录一次。由于该查询结果每天记录一次，因此要搜索主机当前的操作系统版本，只需检索过去 24 小时的快照日志即可。在任何时刻，你都可以在集中式日志平台中查询过去 24 小时内此查询的快照日志，从而查看包含你各主机操作系统名称和版本的最新数据集。

查询包

查询包是预定义查询的集合，这些查询通常同属一个常见类别。其中一些查询包（如 osx-attacks 查询包）会与 osquery 项目的查询包一起打包提供。作为本博文的一部分，我们发布了过去一年中开发的新查询包。像 unwanted-chrome-extensions 这样的查询包已被合并到 osquery 核心项目中，并且我们还向 windows-attack 查询包贡献了额外的查询。目前，我们的 GitHub 存储库包含以下查询包：

  • 安全工具检查：当发现主机上未运行端点安全工具时生成事件
  • Windows 应用程序安全：记录 Windows 上敏感注册表项的值，这些项一旦被修改可能会禁用安全控制
  • Windows 合规性：记录与错误报告、日志记录和更新等操作系统核心功能相关的注册表项的值
  • Windows 注册表监控：确保敏感注册表项存在，并验证其当前值是否与预期值匹配。这些查询触发的任何事件都需要跟进调查。

其中一些查询包中的查询是针对我们的环境量身定制的，但我们希望大家在构建自己的配置时能将其作为参考。

审计进程与套接字事件

osquery 较为强大的功能之一，是支持使用内核审计框架对 Linux 上的所有进程和套接字事件进行原生审计。一种常见的误解是，osquery 审计需要安装 auditd。然而，文档指出：

审计不需要任何 audit 配置或 auditd；实际上，如果使用 osquery 的进程审计功能，就不应运行 auditd。这引起了一些混淆，因为 audit、auditd 和 libaudit 这几个词容易产生歧义——osquery 仅使用内核中的审计功能。

从宏观上看，审计的工作原理如下：

  • 通过 —nodisable_audit 和 -audit_allow_config=true 标志启用 osquery 审计模式
  • 审计规则由 osquery 自动配置，可使用 auditctl 实用程序查看
  • osqueryd 通过审计 netlink 套接字使用内核审计子系统生成的事件
  • 传入的事件被填充到基于事件的表中以供使用

截至撰稿时，osquery 只需启用 3 条不同的系统调用审计规则即可同时进行进程和网络审计：用于进程事件的 execve，以及用于套接字事件的 bind 和 connect。启用进程和套接字审计后，结果将分别填充到 process_events 和 socket_events 表中。以下是在 osqueryi 中查看这些事件大致样貌的简要概述。（注意：在 osqueryi 中启用审计模式时，请确保 osqueryd 未运行。）

# osqueryctl stop
# osqueryi --nodisable_audit --nodisable_events --audit_allow_config=true --audit_persist=true --audit_allow_sockets --logger_plugin=filesystem --events_expiry=1
osquery> select pid, path, cmdline, auid from process_events;
+-------+-----------------+-------------------------+------------+
| pid   | path            | cmdline                 | auid       | 
+-------+-----------------+-------------------------+------------+
| 23548 | /usr/bin/id     | id                      | 500        |
| 23557 | /usr/bin/whoami | whoami                  | 500        |
| 23586 | /usr/bin/curl   | curl https://google.com | 500        |
+-------+-----------------+-------------------------+------------+
osquery> select action, pid, path, remote_address, remote_port from socket_events;
+---------+-------+---------------+----------------+-------------+
| action  | pid   | path          | remote_address | remote_port |      
+---------+-------+---------------+----------------+-------------+
| connect | 23586 | /usr/bin/curl | 8.8.8.8        | 53          |
| connect | 23586 | /usr/bin/curl | 172.217.6.78   | 443         |
+---------+-------+---------------+----------------+-------------+

减少日志量

考虑使用以下方法来减少 osquery 在审计模式下生成的整体日志量：

  • 精选所需的进程和套接字事件列。通过排除某些列来缩小单个进程和套接字事件的体积，有助于在不牺牲实用性的前提下减少记录的数据总量。
  • 过滤掉由其他监控工具产生的大量进程事件。
  • 通过为缺乏安全或取证价值的条目（如 date、tr、head）创建 path 和 cmdline 过滤器，滤除高频出现的进程事件。
  • 过滤掉 remote_address 列包含 localhost、RFC3927 和环回地址的套接字事件。

有关示例，请参阅我们的服务器配置。

审计性能考量

在 Palantir，我们生产环境每天处理大约 16 亿条来自 osquery 的事件。必须认识到，osquery 审计可能会导致系统性能下降。在早期测试中，我们遇到了几种不同的情况：处于高负载状态的系统开始锁死，实际上陷入了停滞。经过大量的调试和分析，我们确定了大部分问题，其原因可归结为以下三者之一：

  • 8.26 之前版本的 Rsyslog 在某些情况下可能会完全锁死
  • 旧内核版本缺少防止审计锁死的补丁
  • Linux 内核审计代码并非总是健壮的

多亏 osquery 开发团队迅速响应并实施了修复，在 2.5.0 到 2.8.0 版本之间实现了显著的性能提升。拥有来自敏感主机的进程和网络审计日志，为事件响应提供了宝贵的信息来源，也为编写基于主机的检测规则奠定了极好的基础。

部署概述

在规划向公司 Linux 服务器部署时，我们决定使用 Git 存储库来管理 osquery 配置数据。CircleCI 关联到该存储库，以确保配置有效且格式良好。初步检查通过后，我们会将新更改推送到一部分非关键的暂存服务器。在验证配置更改且确保未发生性能倒退后，我们再使用配置管理工具将更新后的配置推送到生产服务器。

使用 Fleet 管理 osquery 端点

在规划向员工工作站部署 osquery 时，我们认识到必须解决的主要挑战之一是：如何将 osquery 日志从主机的文件系统卸载并传输到我们的集中式日志平台。就在我们寻找此问题解决方案的同一时间，Kolide 开始提供一款名为 Fleet 的免费开源产品。Fleet 是一款服务器，允许用户管理 osquery 代理机群，这些代理可通过 osquery 内置的 TLS 客户端功能原生连接至该服务器。无需在每个端点上管理配置文件和转发日志，通过 TLS 连接到 Fleet 的 osquery 守护进程可以配置为从 Fleet 服务器接收配置，进而转发生成的 osquery 日志。这种模式的优势在于：

  • Fleet 提供了一个统一的位置，可查看当前已注册的所有 osquery 代理
  • 我们可以随时对整个机群运行临时查询
  • 查询一经添加到查询包中便会立即生效

这绝非 Fleet 功能的完整列表，但这些领域为我们的特定部署场景提供了最大的助力和收益。

Windows 上的 osquery

Windows 上的 osquery 为注册表、WMI 及许多以往难以用单一工具监控的领域提供了强大的深度洞察。它并非试图取代现有的端点安全工具，而是作为完美的轻量级伴侣，为其他解决方案经常忽视的操作系统区域提供可见性。

注册表监控

尽管现有许多工具可以在注册表项被修改时生成事件，但鲜少有工具能让你自定义要监控的项，更重要的是，还能检查这些项在主机上是否存在。在配置监控时，我们很快意识到，仅报告项的更改是不够的；还要确保你试图监控的项一开始就是存在的。在监控注册表时，我们基本上需要留意两种不同的情况：

  1. 某个注册表项被我们的组织设定为特定值，我们希望知道该值何时发生变化。
    - 例如：HKLM\Software\Microsoft\Windows\CurrentVersion\Policies\System\Audit\ProcessCreationIncludeCmdLine_Enabled 注册表项应始终设置为 1 以启用命令行审计。我们希望知道该项的值何时变为非 1。
    - 然而，仅监控该项的更改是不够的。我们还希望知道是否存在某个时刻主机上根本没有该项的情况。这将表明该项从未被创建（配置错误）或在某个时刻被删除了（可能的攻击者活动）。
    - 示例：HKLM\System\CurrentControlSet\Control\CrashControl\SendAlert 项默认情况下不存在。但是，如果创建了该项并将其设置为零，它可以阻止崩溃通知进入 Windows 系统事件日志。攻击者或恶意软件有时会创建此项以降低被发现的概率。
    - 我们可以创建一个查询，以便在发现主机上存在或创建了此项时通知我们

osquery 的注册表表允许我们通过快照查询随时查看整个机群的注册表值。这些注册表快照让异常值分析变得极其简单。此外，注册表表还足够灵活，用户能够在尝试检查项的值之前，先检查该项是否存在或不存在。

自启动项——“恶意软件藏得住，但必须运行。”

微软将这些项目称为自启动扩展点。这些是 Windows 操作系统中无需用户有意启动即可自动运行的项目，包括驱动程序、编解码器、注册表项以及操作系统的许多其他区域，是恶意软件在主机上建立持久性的极其常见的方式。autoexec 表包含了注册表项、服务、计划任务和其他构件的集合，可帮助你识别和编录系统上的所有自启动扩展点。遗憾的是，Windows 中目前存在一些 osquery 尚无法查询的自启动扩展点，如编解码器、外壳扩展等。autoexec 表是一个极好的起点，但在枚举所有 Windows 自启动扩展点方面还不够全面。

如果你正在寻找一种以编程方式编目和记录自启动扩展点的方法，且希望使用 Sysinternals Autoruns 工具，不妨花点时间了解一下 AutorunsToWinEventLog——这是我们专门为此目的编写的一个小脚本。

实现威胁狩猎

一个成熟的信息安全组织的特征之一，是能够主动搜索网络和配置数据，旨在识别表明存在恶意活动的事件或配置错误。这个过程通常被称为“威胁狩猎”，而进行高质量“狩猎”的前提，通常是对网络和端点具有极高的可见性和深度自查能力。osquery 在生成某些数据集方面极具价值，这些数据集能让你回答以往由于难以获取所需数据而难以回答的问题。试想一下，如果没有 osquery，要在数百或数千台主机上收集以下数据集该有多么困难：

  • 由 Apple 以外的颁发机构签名的活动 MacOS 内核扩展的名称和 SHA256 哈希值
  • 每台 MacOS 和基于 Linux 的系统上每个用户已安装的所有 crontab 条目
  • 在 launchd 中加载的、标记为活动状态且启动位于 MacOS 主机 /Users/ 目录下的应用程序的所有 plist
  • 在某个时间点曾插入过具有特定序列号的 USB 设备的所有主机的主机名

另一种有效策略是使用 osquery 进行基本的异常值分析。例如，在极少进行手动配置的生产环境中，预期主机是相当同质的，并具有相同的 crontab 条目集和已安装的内核模块。通过从生产主机收集这些数据，可以轻松使用集中式日志平台过滤出出现在大多数主机上的 crontab 条目和内核模块。剩下的异常值则可以加以检查，以确定它们为何会存在于原本同质的环境中。为了简洁起见，这个例子有些过于简化，但基本的异常值分析在跨不同操作系统识别恶意持久化项方面可能极为有效。

当有一种工具能在如此程度上实现并简化数据收集时，想象力就成了搜寻恶意行为时的限制因素。如需更多想法和狩猎灵感，可使用 Mitre ATT&CK 矩阵来帮助界定狩猎标准。以识别配置错误而非恶意活动作为终结的狩猎不应被视为失败： remediate（补救）这些配置错误可能会防止或限制未来的违规事件！
