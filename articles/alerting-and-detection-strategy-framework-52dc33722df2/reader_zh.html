# 告警与检测策略框架 ![](https://miro.medium.com/v2/resize:fit:700/1*FnPDYeZVrGTbuE7Lj7JhgQ.png) 告警与检测策略（Alerting and Detection Strategies, 简称 ADS）是高效事件响应（Incident Response, 简称 IR）和检测团队的关键需求之一。运转良好的 IR 团队会投入大量时间和精力，针对告警提出新的假设。处理得当，新的 ADS 能为发现异常及潜在恶意活动提供极具价值的信号；处理不当，IR 团队则会被大量低质量告警淹没，引发告警疲劳和士气低落，最终对其检测和消除威胁的使命产生适得其反的效果。 这篇博文详细介绍了 Palantir 的 IR 团队如何开发 ADS、我们使用的流程框架，以及在早期告警开发中踩过的坑。此外，我们还在 GitHub 仓库中发布了 ADS 示例。该项目提供了必要的基础组件，供希望提升检测策略效能的组织采纳此框架。我们也希望分享部分内部 ADS 能够激发更多讨论，并促进更广泛的告警共享。 ## 框架的必要性 ADS 框架是一套关于 ADS 设计、实施和推行的文档模板、流程和规范。在就此框架达成共识之前，我们在实施告警策略时面临重大挑战：缺乏严谨性、文档记录、同行评审以及整体的质量标准，导致低质量告警被直接部署到生产系统。这常常引发团队对告警的漠视，或者需要投入额外的工程时间和资源去修复和更新告警。 临时性或不成熟的告警开发可能导致以下潜在问题： *告警缺乏充分的文档记录。* - 待命工程师可能对相关软件或目标操作系统不熟悉。 - 可能缺乏关于何种情况构成误报的明确指引。 - 发生真阳性命中时，可能没有相应的响应预案。 *告警未经过持久性验证。* - 告警可能基于过于狭窄的条件，从而产生漏报。 - 告警可能基于过于宽泛的条件，从而产生误报。 - 告警可能依赖于错误的假设、不一致的数据源或技术。 *告警在投入生产前未经评审。* - 可能未要求进行变更控制。 - 告警的优先级可能不一致或定义不明确。 - 告警在实施过程中可能未将真阳性测试纳入其中。 ## ADS 框架 我们的 ADS 框架旨在缓解上述问题。它帮助我们对新 ADS 的假设生成、测试和管理进行体系化构建。该框架包含以下部分： - 目标 - 分类 - 策略摘要 - 技术背景 - 盲点与假设 - 误报 - 验证 - 优先级 - 响应 - 附加资源 在部署新 ADS 之前，必须完成并评审上述各个部分。这能确保任何给定告警都具备充分的文档记录、经过持久性验证，并在生产部署前经过评审。我们所有处于生产环境或草稿状态的告警均按照此模板进行记录，文档存储在持久、版本控制且集中的位置（例如 Wiki、GitHub 条目等）。 对 ADS 的后续变更（如修改白名单、调整策略或标记额外的盲点）应在修改时同步纳入 ADS 文档。在内部，我们采用类似 GitHub 的模型：任何功能请求或缺陷都会作为相应 ADS 的 Issue（议题）被提交。一位待命工程师会对该缺陷或功能请求进行分流处理，完成修改，并在关闭 Issue 前更新 ADS 文档。这一流程确保了 ADS 文档的时效性，同时留下了完整的变更审计轨迹。 接下来，我们将详细剖析各个部分。 **目标** 以简短的纯文本描述 ADS 旨在检测的行为类型。 **分类** 将 ADS 映射到 MITRE 对抗战术、技术和常识（ATT&CK）框架中的相关条目。ATT&CK 为对手可能使用的各种后渗透技术与策略提供了一套通用语言。 映射到 ATT&CK 框架有助于对技术进行进一步调查，提供了 ADS 将应用于杀伤链哪些环节的参考，并能进一步洞察告警覆盖的盲区并提供度量指标。在我们的环境中，我们建立了一个知识库，将所有 ADS 映射到 MITRE ATT&CK 框架的各个组件。在为新告警生成假设时，工程师只需查阅我们在特定 ATT&CK 技术上的强项或弱项即可。 **策略摘要** 是对 ADS 运作机制的高层次概述。它描述了告警的监测对象、使用的技术数据源、执行的任何富化操作，以及任何旨在减少误报的步骤。 **技术背景** 提供了响应者理解告警所有组件所需的详细信息和背景。这部分应适当链接到任何平台或工具知识，并包含与告警直接相关的信息。技术背景部分的目标是，为响应者提供一个自包含的参考依据，使其即便对 ADS 本身没有直接的专业知识，也能对任何潜在告警做出判断。 **盲点与假设** 是已识别出的问题、假设以及 ADS 可能无法触发的领域。没有完美的 ADS，识别假设和盲点有助于其他工程师理解 ADS 在何种情况下可能失效或被对手规避。 **误报** 是指由于配置错误、环境特殊性或其他非恶意场景导致 ADS 错误触发的已知情况。这些误报告警应在安全信息与事件管理工具（SIEM）中被抑制，以防止在发生已知误报事件时生成告警。 **验证** 列出了触发该告警所需的代表性真阳性事件的生成步骤。这类似于单元测试，描述了工程师如何促使 ADS 触发。这可以是用于生成告警的步骤说明、用于触发 ADS 的脚本（例如 Red Canary 的 Atomic Red Team 测试），或是在告警测试与编排平台中使用的场景。 **优先级** 描述了 ADS 可能被标记的各种告警级别。虽然告警本身在通过 SIEM 配置触发时应反映其优先级（例如高、中、低），但此部分详细说明了特定优先级的判定标准。 **响应** 是在该告警触发时的通用响应步骤。这些步骤指导下一名响应者如何对告警进行分流和调查。 **附加资源** 是有助于理解该 ADS 的任何其他内部、外部或技术参考资料。 ## 案例研究：针对 Python Empyre 的告警 在介绍完 ADS 框架后，我们将描述 IR 工程师如何构建并部署一条生产告警。在此示例中，我们将着眼于开发一个 ADS，旨在识别受感染工作站上的 Python Empyre（一种 MacOS 植入物）。在此场景中，会发生以下步骤： 1. 某个契机（例如推文、文章、观察到的攻击者活动）激发了 IR 工程师对告警的构想。在此案例中，工程师阅读了一篇关于 Python Empyre 的文章，鉴于公司拥有大量 Mac 设备，认定这是一项值得 pursued 的检测策略。 2. IR 工程师开始开发告警。他们在分析环境中下载 Python Empyre，并审查遥测数据源。这些数据源包括基于主机的日志和工具、网络日志以及其他取证工件。 3. IR 工程师发现了 Python Empyre 的一个有趣特征。Python Empyre 会执行一项特殊检查，以识别主机上是否运行着 Little Snitch（一款 MacOS 应用防火墙）。在默认状态下，一旦发现 Little Snitch 存在，Python Empyre 便会退出。 4. IR 工程师决定基于对 Little Snitch 检查行为的检测来编写 ADS。他们认为，由于这是默认功能，该检查足以捕获未经修改的 Python Empyre，且很可能不会产生大量误报。 5. IR 工程师在告警的 GitHub 仓库中新建了一条 ADS 记录。 IR 工程师查看分析环境中的进程执行事件，并识别出 Python Empyre 用于查找 Little Snitch 的具体命令： `/bin/sh -c ps -ef | grep Little\ Snitch | grep -v grep` 深入挖掘 Python Empyre 的源码，可以找到执行此检查的显式代码： ```python try: if safeChecks.lower() == 'true': launcherBase += "import re, subprocess;" launcherBase += "cmd = \"ps -ef | grep Little\ Snitch | grep -v grep\"\n" launcherBase += "ps = subprocess.Popen(cmd, shell=True, stdout=subprocess.PIPE)\n" launcherBase += "out = ps.stdout.read()\n" launcherBase += "ps.stdout.close()\n" launcherBase += "if re.search(\"Little Snitch\", out):\n" launcherBase += " sys.exit()\n" except Exception as e: p = "[!] Error setting LittleSnitch in stager: " + str(e) print helpers.color(p, color='red') ``` IR 工程师决定细化告警条件，将范围扩大到由 grep 二进制程序调用的、针对 Little Snitch 的更广泛查询： `grep Little\ Snitch` IR 工程师记录下此行为，并在 SIEM 中进行历史搜索，查找与此特征完全匹配的记录。回顾过去 90 天的进程执行活动，仅命中了其测试活动的记录，以及部分系统管理或更新软件的命中记录。这表明该告警具有高保真度，即环境中似乎不存在对同一命令的任何合法调用。 IR 工程师记录下白名单条目，并为 SIEM 起草告警（例如计划查询、转换等），专门查找该命令调用。ADS 模板中填入了各部分的初稿。 此时，IR 工程师到达了 ADS 评审检查点。评审人员对告警进行检查，验证 ADS 从源端到 SIEM 的功能是否正常，包括真阳性测试。该 ADS 被分配一个用于内部跟踪的规则 ID（RID），IR 工程师开始推进生产部署。尽管这比单纯写一条告警的工作量更大，但它为 ADS 的质量带来了几项益处： - ADS 必须完全填写完毕，这保证了文档的基线标准。 - 必须进行本地化测试、研究、误报最小化和真阳性验证，才能完成 ADS 的所有部分。这副作用是还能在安全工具中填充与告警相关的数据以供分析。 - ADS 必须由另一名工程师评审，从而发现可改进之处、优化策略以及减少误报的方法。 Little Snitch 案例研究完整的 ADS 如下： **目标：** 通过查找进程和命令行特征，检测潜在恶意软件试图发现主机上是否存在 Little Snitch 的行为。 **分类：** 这些尝试被归类为发现 / 安全软件发现。 **策略摘要：** 该策略将按以下方式运作： - 使用终端检测工具记录 MacOS 主机的进程及进程命令行信息。 - 查找任何对 Little Snitch 的显式进程或命令行引用。 - 抑制已知的正常进程和命令行参数 - Little Snitch 更新程序 - Little Snitch 安装程序 - Little Snitch 健康检查 - 对任何其他进程或命令行活动触发告警。 **技术背景：** Little Snitch 是一款 MacOS 应用防火墙，允许用户围绕应用程序在网络上的通信方式生成规则集。 在极其严格的模式下，一旦应用程序偏离规则集，Little Snitch 就会弹出通知。例如，以下事件可能会触发交互式告警： - 观察到新进程试图在网络上进行通信。 - 进程正在与规则集不同的新 IP 地址或端口通信。 由于 Little Snitch 弹窗具有侵入性，多款 MacOS 植入物会显式检查进程、内核扩展（kext）及其他组件。这通常表现为通过显式调用进程或目录命令，并针对 Little Snitch 进行子过滤。 例如，植入物可能会查找以下组件： - 运行中的 Little Snitch 进程 - Little Snitch 内核扩展 - Little Snitch 属性列表 - Little Snitch 规则 **盲点与假设：** 此策略依赖于以下假设： - 终端检测工具在系统上运行且功能正常。 - 进程执行事件正在被记录。 - 终端检测工具的日志已上报至服务器。 - 终端检测工具已正确将日志转发至 SIEM。 - SIEM 已成功索引终端检测工具的日志。 - 攻击者工具包会执行搜索，以识别是否安装或运行了 Little Snitch。 如果上述任何假设被打破，就会出现盲点。例如，以下情况不会触发告警： - 终端检测工具被篡改或禁用。 - 攻击者植入物未以生成子进程的方式搜索 Little Snitch。 - 在搜索 Little Snitch 时进行了混淆，从而规避了我们的正则表达式。 **误报：** 此 ADS 可能在以下几种情况下出现误报： - 用户显式地对 Little Snitch 安装情况进行查询，例如 grep 查找进程、搜索文件。 - Little Snitch 执行更新、安装或卸载操作，例如我们遗漏了对已知正常进程的白名单处理。 - 管理工具对 Little Snitch 执行操作，例如我们遗漏了对已知正常进程的白名单处理。 已知误报包括： - Little Snitch 软件更新程序 大多数误报可归因于查看 Little Snitch 当前状态的脚本或用户行为。这些要么是受信任的二进制文件（例如我们的管理工具），要么是明确无害的用户行为（例如执行查询的进程是用户 Shell 进程的子进程）。 **优先级：** 在所有情况下，优先级均设为中。 **验证：** 可通过在 MacOS 主机上执行以下命令来对此 ADS 进行验证： `/bin/sh -c ps -ef | grep Little\\ Snitch | grep -v grep` **响应：** 如果此告警触发，建议采取以下响应程序： - 查看管理工具，识别主机上是否安装了 Little Snitch。 - 如果主机上未安装 Little Snitch，则情况可能更为可疑。 - 查看触发此告警的进程。梳理进程链。 - 哪个进程触发了此告警？ - 进程以哪个用户身份运行？ - 父进程是什么？ - 此链中是否存在任何异常差异？ - 查看触发此告警的进程。检查二进制文件。 - 这是 Shell 进程吗？ - 进程是否经过数字签名？ - 父进程是否经过数字签名？ - 此二进制文件的普遍程度如何？ - 这看起来是否属于用户主动发起的性质？ - 是否在长时间运行的 Shell 中执行？ - 是否有其他指标表明这是用户手动输入的？ - 如果该活动可能由用户发起，通过我们的聊天客户端联系用户，要求其说明行为。 - 如果用户对此行为不知情，则升级为安全事件。 - 如果进程行为看起来异常，或者未安装 Little Snitch，则升级为安全事件。 此 ADS 记录也可在我们的公开 GitHub 仓库中找到。 ## 更多 ADS 示例 ADS 框架可在我们的公开 GitHub 仓库中找到，我们还纳入了自行开发的几项内部 ADS 作为参考示例。 我们鼓励其他组织开发各自独特的 ADS，并找到一种机制与更广泛的信息安全社区共享。尽管公开承认和记录内部告警存在运营安全方面的考量，但我们希望这些示例能促进更大的共享与合作，激发其他防御者的检测增强思路，并最终提高攻击者的运营成本。 ## 延伸阅读与致谢 我们要感谢以下各方对信息安全社区的贡献，或对开发 ADS 框架提供的协助： -
