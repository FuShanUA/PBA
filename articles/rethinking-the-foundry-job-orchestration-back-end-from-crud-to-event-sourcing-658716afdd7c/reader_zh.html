重新思考 Foundry 作业编排后端：从 CRUD 到事件溯源

在 2019 年 Reversim 大会上，Tomer Gabel 在他的演讲《微服务：回顾》中，全面总结了基于事件的软件设计相比于传统 CRUD 的优势。Tomer 建议在此背景下研究 CAP 定理：当然，它同样适用于事件溯源设计，那些棘手的分布式系统问题并不会凭空消失。

受他演讲的启发，我撰写了这篇博文，回顾了将 Palantir Foundry 作业编排系统的 CRUD 数据存储重写为事件溯源和内存映像的经验。我将首先介绍编排系统在 Foundry 架构中的作用，解释为什么我们遇到了 CRUD 存储后端的瓶颈，然后通过一个具体的 Java 最小端到端示例，说明基于事件溯源的重新实现。该示例的重点在于维护图数据结构时遇到的一致性问题。请注意，这篇文章不会过多介绍事件溯源或 CQRS 的背景知识，但你可以轻松在网上找到大量相关资料。

警告：示例实现未作任何优化。它纯粹作为教学材料，旨在突出 CRUD 和事件溯源实现在概念上的异同。

Foundry 中的作业编排

我们的客户将 Palantir Foundry 作为数据驱动决策的基石；然而，仅就这篇博文而言，让我们将重点放在 Foundry 作为编写和执行数据转换的平台上。例如，下面的代码清单展示了一个 PySpark 数据转换，它对温度传感器读数和对应气象站位置的数据集进行连接操作，从而生成一个名为 temperature_by_region 的新数据集。

PySpark 数据转换示例

更正式地说，每个数据转换都是一组输入数据集和输出数据集之间的函数。执行转换即意味着将转换函数应用于输入数据集，从而生成输出数据集。我们将这种转换的规范称为作业规范（简称作业）；它包含输入和输出数据集的名称（例如 S3 中的 Parquet 文件）以及转换的源代码（例如 PySpark 程序）。compute_temperature_by_region 有两个输入数据集（stations 和 temperatures）和一个输出数据集（temperature_by_region）。

如果作业 JOB1 声明的输入是另一个作业 JOB0 的输出，我们就说 JOB1 依赖于 JOB0：JOB0 必须在 JOB1 之前执行，以产生 JOB1 的输入。所有作业规范的集合构成了一个二分图，其中的节点是作业规范（青色）和数据集（紫色），边是作业的输入和输出。左图展示了此类图的一个示例；很明显，我们需要先用 JOB0 计算出 temperature by region 数据集，然后再运行 JOB1。

作业编排问题如下：给定一个目标输出数据集，计算生成该数据集所需的作业偏序集。我们将这个集合称为编排计划。请注意，只有当相关子图无环时，计划才存在。

寻找编排计划有一种直截了当的算法：从目标输出数据集出发，遍历图以收集所有传递性的作业依赖关系；在这些路径上遇到的边定义了偏序关系。我们发现，如果对作业图施加两个额外的约束，用户就能更容易地理解作业编排的语义：

完整的作业规范图应该是无环的。这意味着所有子图都是无环的，因此每个数据集都存在编排计划。

每个输出数据集最多只能由一个作业生成。此约束意味着计划是确定性的。

无环性和确定性共同意味着，每个数据集都恰好存在一个编排计划。简而言之，我们可以利用作业依赖关系，推导出生成给定目标输出数据集的数据转换的执行顺序。（……这并非什么高深莫测的学问……）

CRUD 编排服务

Foundry 作业编排系统的简化架构。

Foundry 用户通常通过在受 Git 支持的 Web IDE 中编写数据转换来与 Foundry 的编排系统交互；见左图。预提交钩子为每个转换创建作业规范，并通过简单的 CRUD API 将其存储在编排服务中。

预提交钩子提取输入/输出注释中的数据集名称（见上面的代码清单），以确定不同作业之间的依赖关系。编排服务存储所有作业规范，并提供用于按计划或用户请求执行输出数据集编排计划的 API。

编排服务的 CRUD 实现通过邻接表将图存储在类似 Cassandra 的数据库中。

我们对编排服务的首次实现，采用了从其 CRUD API 到 CRUD 数据库的简单直通：我们将作业规范图作为邻接表存储在 Cassandra 中。左图说明了这个模型。虽然这个模型对于编排服务的基本 CRUD 部分来说是一个简单有效的解决方案，但它有两个主要缺陷：

为了找到目标数据集的计划，我们需要遍历作业图以寻找所有传递性依赖。由于图是通过邻接表存储的，这个过程需要 O(N) 次数据库查找，其中 N 是目标依赖链的长度。当一些客户的数据集和转换数量达到十万个时，编排服务再也无法提供交互式的性能。

类似地，无环性检查需要遍历图的大部分。随着作业更新数量的增加，我们最终关闭了主动无环性检查（从而允许图中存在环），只在计算和执行计划时才检查子图是否无环。（作业执行的次数远少于作业修改的次数。）这个模型并不理想，因为我们更希望快速失败，在用户编写转换时就能告知他们该转换是非法的，而不是等到他们执行时才告知。

事件溯源编排服务

编排服务的事件溯源实现维护了一个有序的图变异事件日志，并针对内存映像执行查询。

为了解决这两个缺陷，我们采用事件溯源和作业规范图的内存映像重写了编排服务的存储层；见左图。我们不再在数据库中维护始终最新的图邻接表，而是存储一个线性化的图变异事件序列。

为了处理针对图的读取请求，我们维护了最终一致的图内存副本；当然，针对内存图进行图遍历比针对数据库支持的图要快得多。（你会注意到事件溯源方案在这里有一个有趣的局限性：如果图无法再装入内存怎么办？）

编排服务在将变异事件追加到事件日志之前，会检查图的无环性和确定性。这个问题的核心在于一致性，无论是 CRUD 还是事件溯源实现都是如此。在 CRUD 模型中，我们通过使用 AtlasDb 作为 MVCC 数据库来保证原子更新（换句话说，我们避免了幻读）：我们可以在事务中检查约束并执行图更新。在事件溯源模型中，我们通过变异事件的密集线性顺序来保证原子更新：我们在版本 N 的图上检查约束，并在版本 N+1 处插入变异事件。以下各节将更详细地描述此模型。

示例实现

现在让我们来看看 CRUD 和事件溯源作业编排服务的具体示例实现。编排服务的核心是一个简单的图数据库 API：

一个简单的图数据库 API

用户可以通过 GraphDb API 创建图，添加节点和边。API 提到了一些实现必须维护的约束：用户不能添加已存在的节点或冗余的边，并且被管理的图必须是无环的。

虽然这个虚拟实现运行在单个 JVM 中，但设想的场景是一个分布式系统（例如 Web 服务），具有多个并发用户分别进行读和写的代码路径。在 Foundry 的案例中，CRUD 和事件溯源实现都支持在多个节点上以高可用配置运行作业编排服务。特别是，当多个用户并发访问图时，我们需要提供一致性保证。

CRUD 实现

CRUD 实现使用 GraphStore 来持久化图节点和边，并使用 DistributedLock 来协调服务不同节点间的变异。存储 API 的设计使其可以直接在类似 Cassandra 的数据模型（或扩展到任何关系型数据库）之上实现。注意：为了简化本示例的展示，存储层不提供任何事务或 MVCC 保证，并发问题是通过全局分布式锁来实现的。

图数据库 API 的 CRUD 实现

CrudGraphDb 实现很简单：在存储变异（即添加节点或边）之前，我们检查该变异是否满足相关的 API 约束。由于 GraphStore 数据库不支持事务，我们简单地将每个读/写代码路径包装在分布式读写锁中（类似 Zookeeper）。这保证了并发请求情况下的正确性。

CRUD 方法的一个标准缺陷是图的重水合（即从数据库读取图）非常昂贵：由于我们无法增量维护完整的图，我们检查无环约束的唯一希望是从数据库读取图的完整副本，然后检查它是否无环；接着丢弃该图，在下一次添加边时再次读取。当然，我们可以尝试在这里更聪明一些：例如，如果底层数据库支持版本控制，我们可以检查自上次读取后图是否被更新过，如果没有，则使用缓存的图。（所有这些都必须在锁的保护下进行。）

人们可能会想，我们是否可以通过将约束检查推送到事务性底层存储中来避免分布式锁。答案（通常是）“不，我们做不到”：典型数据库后端（包括键值存储、类 Cassandra 模型，甚至 SQL 数据库）的表达能力不足以检查诸如图连通性或无环性的非局部约束。SQL 数据库确实可以维护诸如非空、主键或连接完整性等局部约束，但我怀疑大多数现实用例中的约束要比这复杂得多。

事件溯源：一次失败的尝试

事件溯源实现的首次尝试使用了 EventStore 和 MemoryImage（又称投影或聚合，另见 Fowler:MemoryImage），但没有使用分布式锁。其理念是，我们将诸如 NodeAdded 和 EdgeAdded 之类的变异事件追加到事件存储中，内存映像订阅事件流并将当前图维护在内存中，例如作为链接的 Java 对象。由于变异事件是不可变的，我们永远不必回溯过去。相比 CRUD 方法，最大的好处是我们只需支付增量维护成本，而不必为每次变异从存储中重新水合整个图。

事件溯源图实现的一次失败尝试；它无法在并发访问下保证无环性约束。

实现看起来很简单：我们针对当前内存中的图检查约束（与 CrudGraphDb 实现中相同），如果满足约束，我们就将相应的事件发送到事件存储。MemoryImage 本身只是订阅新事件，并在收到新事件通知时更新图：

事件溯源图的一个简单内存映像。

遗憾的是，这个实现是有问题的，因为当我们检查约束时，内存映像图可能会在我们底下发生变化。用数据库术语来说，BrokenEventGraphDb 遭受了幻读。这尤其令人失望，因为我们已经试图非常聪明：事件存储中的事件按序列号排序，存储会验证我们是否按正确的顺序插入事件：0、1、2、3 等。

修复竞态条件

修复竞态条件的一种方法是使用乐观并发控制：在 EventGraphDb 中，我们针对已知固定版本的图检查约束，然后以确切的下一个序列号发出变异事件。如果事件存储拒绝了该事件，我们就知道其他人通过插入具有冲突序列号的事件在我们底下更改了图。我们可以选择放弃，或者尝试使用希望是更新的内存映像再次尝试：

一致地提交图变异事件：我们首先在某个版本 N 检查图约束，然后仅当变异事件成为紧接着的下一个事件 N+1 时才发出它。

这个简单的示例展示了 CRUD 和事件溯源方法在并发方面的结构性相似之处：在这两种情况下，我们都需要原子地执行约束检查和状态变异，例如通过线性化状态变异。在 CRUD 实现中，我们选择使用分布式锁来线性化变异；在事件溯源实现中，我们选择使用密集、有序的事件序列来线性化变异。请注意，这种差异是出于选择，并不是 CRUD 和事件溯源风格之间的原则性差异：例如，我们可以在事件溯源实现中使用悲观锁来线性化变异。

此处选择的乐观并发方法在 N 个用户执行并发变异时表现出典型的性能惩罚：N 次并发变异尝试中有 N-1 次因序列号不匹配而失败；值得注意的是，这 N-1 次尝试已经检查了图约束，从而浪费了大量的计算资源。更好的策略可能是线性化变异：通过同步队列在本地进行，或者在分布式系统中通过领导者选举进行……就像在 CRUD 情况中一样。这里的教训是，将范式从 CRUD 更改为事件溯源并不能神奇地解决分布式系统中根本的协调和一致性问题。

和之前一样，我们能否通过将约束推送到底层存储来完全避免应用层的并发控制？不幸的是，在事件溯源世界中，“不”这个答案同样坚定，因为底层数据库（MySQL、Kafka、Kinesis 等）无法表达诸如图连通性或无环性的复杂约束。

快照

MemoryImage 类支持快照：它会定期将当前图的快照存储到一个简单的键值底层存储中。为了重新水合内存映像（例如启动新节点或清理缓存以释放内存时），我们可以加载最新的快照，并仅应用比快照更新的（按序列号）那一部分事件。快照方法用时间换取内存：我们存储的快照越多，消耗的内存就越多，但我们重新水合与事件保持同步的内存映像的速度就越快（理想情况下……）。根据我们的经验，在持续部署的系统中保持较低的启动时间至关重要；例如，当节点需要数小时的启动时间才能可用时，蓝绿部署策略就变得行不通了。

保留期

到目前为止，我们假设事件存储具有无限的保留期，即它从不删除事件，并且快照存储仅仅是一种优化。在病态情况下，这意味着与 CRUD 世界相比，存在线性的内存开销；例如，序列 addNode(1), removeNode(1), addNode(1), removeNode(1), ... 产生了一个大小恒定的图，但事件存储却呈线性增长。

我知道有两类解决方案可以解决这个问题：首先，如果所有消费者（即 MemoryImage 及其相关组件）都持有大于 N 的持久快照，我们就可以从事件存储中修剪早于 N 的事件。其次，我们可以通过特定领域的逻辑检查事件并删除冗余或幂等事件：例如，如果后续事件添加和删除了同一个节点，我们可以修剪冗余的“较早”事件，只保留最新的事件。

当然，这两种方法都以较小的内存占用为代价，使“纯”事件溯源方法的某些不变量和好处失效：我们不再拥有完整的审计跟踪，无法回溯时间，而且——也许最重要的是——我们将业务逻辑中的错误固化到了快照中，并删除了我们可能需要用来修复状态的事件信息。

结论

这篇关于 Foundry 作业编排以及 CRUD 和事件溯源在 Foundry 架构中相关性的博文到此结束。主要的收获是，许多棘手的分布式系统问题，特别是一致性问题，在事件溯源模型中并不会神奇地消失。就像计算机科学和软件工程中的大多数事物一样，在 CRUD 和事件溯源模型之间进行选择是一个权衡问题。
