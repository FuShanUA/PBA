# 使用 osquery 进行审计：第二部分 — 配置与实现 ![](https://miro.medium.com/v2/resize:fit:700/1*VdbOuGDB6Qo0tPiCRIiaoQ.png) 在《本系列的第一部分》中，我们介绍了 Linux 审计框架的基础知识。在这篇博文中，我们将重点讨论 osquery 审计的实现细节。除了配置审计外，我们还将为读者提供一些策略，以减少大多数审计配置伴随而来的性能影响和日志量。 ## 使用 osquery 启用审计 命令行标志是更改 osquery 核心行为的主要方法。基础配置中与审计相关的标志列表如下： - **--audit_allow_sockets=true** 告诉 osquery 记录网络连接（bind() 和 connect() 系统调用） - **--audit_allow_process_events=true** 告诉 osquery 记录进程执行（execve() 系统调用） - **--audit_allow_config=true** 告诉 osquery 允许更改审计配置选项。如果你使用自定义审计规则，可能需要禁用此选项。 - **--audit_persist=true** 告诉 osquery 在失去连接时重新获取对 netlink 套接字的访问权限 - **--disable_audit=false** 告诉 osquery 启用审计 - **--events_expiry=1** 告诉 osquery 在单次选择后使事件过期 - **--events_max=500000** 告诉 osquery 在查询间隔定义的 SELECT 之间缓冲 500,000 个事件 - **--logger_plugin=filesystem** 告诉 osquery 将日志记录到文件系统 - **--watchdog_memory_limit=350** 告诉 osquery 将内存限制提升至 350mb 这些命令行标志通常保存在 `/etc/osquery/osquery.flags` 中。 除了命令行标志之外，你还需要确保在 `osquery.conf` 配置文件中查询 `process_events` 和 `socket_events` 表。 Palantir 已经在这里开源了我们的 osquery 配置： - osquery-configuration - osquery.conf - osquery.flags ## 日志记录注意事项 ### 选择要从每个表记录的特定列 虽然在查询中简单地使用通配符很诱人（例如 `SELECT * FROM process_events` 和 `SELECT * FROM socket_events`），但请考虑一下，当这些数据扩展到 10 台、1000 台甚至 10,000 台主机时，会消耗多少空间。如果你能通过只选择某些列将平均进程和套接字事件大小减少 25%，那么你整体的 osquery 审计日志量就会缩减 25%。在收集数据时要有所选择，以尽量减少 osquery 生成的日志量。 ### 使用 SQL 过滤无关事件 考虑一下你是否需要记录系统上每个二进制文件的进程执行情况。有许多工具使用非常频繁，但提供的安全或取证价值很小，例如 `awk`、`sed`、`tr`、`cut`。 简单地创建如下查询对 osquery 的性能影响微乎其微，却可以节省大量的日志量： `SELECT * FROM process_events WHERE path NOT IN ('/usr/bin/awk', '/usr/bin/sed', '/usr/bin/tr');` 注意：正如本博文第一部分所述，以这种方式排除事件并不会降低性能开销。osquery 仍然必须处理通过 SQL 过滤掉的事件，只是不记录它们而已。 ### 仅记录成功的套接字事件 `socket_events` 表包含一个名为“success”的列，指示连接是否成功。根据你的日志量要求，确定成功与不成功的套接字事件各占多少百分比，并可能过滤掉所有不成功的连接，这可能值得一试。显然，这是在可见性和日志量之间的一种权衡。 ### 按计数监控前 10 个进程 在 Palantir，我们维护着一个审计仪表板，用于密切关注生成日志最多的进程。如果我们发现特定进程在系统上生成的事件比其他进程多几个数量级，我们就能采取行动，通过一行修改来过滤它们。 ## 审计故障排除 如果你在让 osquery 记录进程和审计事件时遇到问题，请检查清单上的每一项： - 确保 osquery 以 root 身份运行 - osquery.flags 中存在（本文档中记录的）正确标志 - osquery.conf 中存在针对进程和/或套接字事件表的查询 - osquery.flags 和 osquery.conf 位于正确的位置（Linux 上为 `/etc/osquery`） - osquery 已安装并正在运行 - `auditctl -s` 输出中列出的 pid 映射到 osqueryd 进程并且不会改变 - 没有其他审计消费者（auditd/go-audit/auditbeat）同时运行 - `auditctl -l` 的输出中至少列出了一条系统调用规则 - 如果使用 syslog，请确保其配置为允许大消息且不进行速率限制 如果你已经验证了所有这些项目，我们有两种主要的故障排除策略： 1. 停止任何正在运行的 osquery 进程，并使用 `--verbose` 标志将 osqueryd 作为前台进程运行，检查输出中是否有错误： (a) `$ osqueryctl stop; osqueryd --flagfile=/etc/osquery/osquery.flags --verbose` 2. 停止任何正在运行的 osquery 进程，并使用隐藏的 `--audit_debug=true` 标志运行 osqueryi： (a) `$ osqueryctl stop; osqueryi --flagfile=/etc/osquery/osquery.flags --audit_debug=true --verbose` (b) osqueryi 将让你进入一个交互式 shell，类似于 sqlite 呈现的 shell (c) `--audit_debug` 标志将输出 osqueryd 从 netlink 套接字接收到的原始审计事件，然后再将其处理成内部表。如果没有出现审计事件，很可能是某些配置有误。 ## 了解性能影响 使用 osquery 进行审计主要以两种方式引入性能开销： 1. 当在内核中启用审计并且存在审计规则时，内核必须执行额外的操作，将每个系统调用与审计规则列表进行比较，并生成实际的审计事件。 2. 审计消费者（在本例中为 osquery）从内核 netlink 套接字提取事件，将事件数据解析为其内部表格式，并将该事件存储在 RocksDB 中。一旦事件被查询选择，该事件就会被写入日志文件或通过日志插件发送。 正如我们之前提到的，根据启用的审计选项，osquery 仅监控 1-3 个特定的系统调用。因此，每次内核中发生这些系统调用之一时，都会生成一个审计事件。审计事件越多，内核生成它们的工作就越多，osquery 解析它们并将其存储在数据库中的工作也就越多。这对底层发生的事情进行了极度简化的说明，但理解这一点至关重要。 ### osquery 压力测试 osquery 的官方 git 仓库包含一个用 Python 编写的初级压力测试。这个压力测试只是创建用户定义数量的 shell，并将数据发送到 UDP 套接字。通过将此压力测试与 `perf` 工具结合使用，你可以观察运行审计和 osquery 产生的计算开销。 注意：如果你在虚拟机上使用 perf 工具，请务必先启用虚拟机上的性能计数器/代码分析。 **审计与 osquery 禁用：** ``` # perf stat python system_stress.py -i lo -n 10 Expecting 10240 (default shell) processes Executed 10240 (default shell) processes Elapsed: 41.857626915 Performance counter stats for 'python system_stress.py -i lo -n 10': 83219.491922 task-clock (msec) # 1.986 CPUs utilized 41.897018727 seconds time elapsed ``` **审计与 osquery 启用：** ``` # perf stat python system_stress.py -i lo -n 10 Expecting 10240 (default shell) processes Executed 10240 (default shell) processes Elapsed: 47.5021560192 Performance counter stats for 'python system_stress.py -i lo -n 10': 84672.567230 task-clock (msec) # 1.781 CPUs utilized 47.535512135 seconds time elapsed ``` 如你所见，启用 osquery 审计后，压力测试完成的时间多了约 6 秒。虽然这是一种非常粗略且不精确的计算计算开销的方法，但它确实非常清楚地说明了问题。当审计和 osquery 未运行时，压力测试能够更快地完成其操作。 ### 测量进程生成的系统调用 要基本了解哪些进程正在生成哪些系统调用，你可以结合使用 auditd、ausearch 和 aureport。Auditd 将审计事件记录到日志文件中，而 ausearch 结合 aureport 对 auditd 日志文件进行一些方便的解析，以显示每个进程生成了多少特定类型的系统调用。 ``` # Stop osquery so it doesn’t conflict with auditd sudo osqueryctl stop # Clear out any existing audit rules sudo auditctl -D # Insert audit rules used by osquery sudo auditctl -a always,exit -S execve sudo auditctl -a always,exit -S bind sudo auditctl -a always,exit -S connect # List audit rules sudo auditctl -l # Remove old audit logs sudo rm /var/log/audit/*.log # Start auditd and let it run for 30 seconds sudo service auditd start; sleep 30; sudo service auditd stop # Delete audit rules sudo auditctl -D # Create a report using aureport for each syscall that we have a # rule for for SYSCALL in connect bind execve do echo -e "\nReport for $SYSCALL syscall" ausearch --syscall $SYSCALL --line-buffered --raw | aureport --executable --summary | head -15 done ``` 运行 `system_stress.py` 后生成的报告片段如下所示： ``` Report for connect syscall Executable Summary Report ================================= total file ================================= 10240 /usr/bin/python2.7 532 /usr/sbin/aureport 14 /usr/sbin/asearch ``` 运行此报告的主要目的是了解特定系统上生成了多少每种系统调用，并确定是否有任何异常进程生成的系统调用比其他进程多得多。这些异常进程最有可能引起性能问题。如果收集这些进程的日志并不重要，请考虑通过审计规则过滤它们。 ## 使用审计规则排除进程 在某些情况下，我怀疑用户曾试图在系统上使用 osquery 启用审计，却发现 osquery 消耗了不可接受的 CPU 周期。由于没有明显的解决方案，他们认定它不是合适的工具，并放弃了将其用于审计目的的想法。 虽然承认审计框架在某些方面存在不足，但我们还是希望提供一些选项，以减少在系统上启用它带来的性能影响。 一种选项是为那些你不感兴趣监控的进程创建自定义审计规则列表。考虑一下上面压力测试生成的报告。`/usr/bin/python2.7` 负责生成 10240 次 connect() 系统调用。我们可以将此二进制文件作为创建排除审计规则时的示例。 这些“过滤”审计规则的格式如下： `-a never,exit -F exe=/path/to/bin -S [all|syscall_name]` 我们告诉审计对于位于 `/path/to/bin` 的二进制文件，**永远不要**为**所有**系统调用或特定系统调用生成审计事件。 ### 使用审计规则过滤嘈杂的进程 1. 停止审计和 osquery 2. 编辑审计规则文件（有时位于 `/etc/audit/rules.d/audit.rules`） 3. 以上面提到的格式添加“过滤”审计规则 4. 添加 osquery 所需的审计规则 5. 生成的 audit.rules 文件应如下所示： ``` # Delete any pre-existing rules -D # Don't generate audit events for python2.7 -a never,exit -F exe=/usr/bin/python2.7 -S all # Increase the buffers to survive stress events. # Make this bigger for busy systems -b 1024 # Add rules for osquery auditing -a always,exit -S execve -a always,exit -S bind -a always,exit -S connect # Enable audit -e 1 ``` 6. 在 `osquery.flags` 文件中将 `--audit_allow_config` 标志设置为 `false`。我们不再希望 osquery 管理我们的审计配置，因为我们现在使用的是自定义规则。 7. 启动 osquery 8. 运行 `augenrules --load` 以合并规则并将其加载到内核中。你可能会看到有关指定架构的消息——它们可以被忽略。 9. 运行 `auditctl -l` 列出活动的审计规则，并确认你添加的审计规则已存在 遗憾的是，审计规则在 exe 字段中不支持通配符，因此你必须逐个显式地将二进制文件加入白名单。除了通过路径将二进制文件加入白名单外，审计还支持通过 PID、UID 和 SELinux 上下文进行过滤。 我们认识到这个过程有些棘手，因此我们提交了以下 osquery 功能请求以简化此工作流程：https://github.com/facebook/osquery/issues/5308 ## 结论 这篇分为两部分的博文系列介绍了对审计框架的基本理解，为在审计模式下配置 osquery 提供了指导，并提出了减少审计和审计消费者在系统上的性能占用空间的策略。展望未来，很明显审计正迅速成为一种过时的解决方案，而 eBPF 等较新的技术可能会成为性能强得多的替代品。在那成为现实之前，我们将专注于为当前可用的工具提供最佳指导。 ## 参考 - https://slack.engineering/syscall-auditing-at-scale-e6a3ca8ac1b8 -
