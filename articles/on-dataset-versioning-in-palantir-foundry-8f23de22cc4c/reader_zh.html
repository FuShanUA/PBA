浅谈 Palantir Foundry 中的数据集版本控制


Robert Fink

7 分钟阅读量 · 2019年11月14日

如果您最近访问过 Palantir 的网站，就会发现我们的平台能够让组织“像软件工程师管理代码一样管理数据”。Git 等协作工具在软件工程工具链中扮演着核心角色：Git 提交确立代码版本，而 Git 分支则将开发人员和功能特性隔离到各自的沙箱中。在这篇博文中，我们将探讨为什么数据集版本控制和沙箱化对协作式数据工程至关重要，这完全类比了 Git 工作流对于软件工程师的重要性。

我们讨论的起点是一个经常从客户那里听到的简单需求：“我们能否将数据集从 Palantir Foundry 导出到现有的数据湖或 S3 存储桶中？”虽然这当然是可以实现的，但必须明白，导出后的数据集恰恰缺失了那些让 Foundry 成为卓越协作式数据工程工具的版本控制和沙箱化特性。通过 Git 的类比我们将看到，如果（反事实地）缺失这些特性，在软件工程场景下很容易导致软件缺陷，而在数据工程场景下则会引发数据不一致。

协作式软件工程

如今，任何重要的软件代码库都托管在源码管理系统中，例如 Git。简而言之，Git 为开发者提供了一个具备虚拟沙箱的版本化文件系统：不同的 Git 提交对应代码库（即源代码文件）的不同版本，而 Git 分支允许开发者在不干扰他人工作的情况下实现自己的功能特性或修复缺陷。

现在，类比前文提到的“将 Foundry 数据集导出到 S3”的需求，让我们设想一下，如果将 Git 仓库导出到一个不具备 Git 功能的标准文件系统中会意味着什么。例如，我们可以运行一个定期脚本（可能由持续集成工作流或 Git 钩子触发），检出 master 分支上的最新提交，然后将其源代码文件复制到其他目录，比如 /home/rfink/git-export/。

显而易见，这样导出的 Git 仓库无法支持分支工作流，原因很简单，标准文件系统不支持分支。更微妙的是，如果多个用户尝试并发修改文件，导出的仓库会遭遇写/写冲突。例如，如果导出脚本的两个实例针对两个不同的提交执行两次并发导出，那么导出目录很可能呈现出两个代码版本的非确定性混合状态。因此，就写入工作流而言，导出的仓库严格来说是一个单用户环境。

也许最令人惊讶的是，即便是对于只读工作流，导出的代码目录甚至也无法如实地呈现源代码：如果用户在导出脚本运行期间（也就是在更新源代码的导出快照时）尝试编译源代码，编译器看到的很可能是多个代码版本的不一致混合体。换言之，该导出操作不是原子的。在最好的情况下，编译器会报错（可能是语法错误）；但在更糟的情况下，编译器可能成功通过，却产生了一个不正确的程序（比如带有导致资金蒸发的竞态条件），这就是编译语义上不兼容的源代码文件所带来的后果。

为了安全地使用导出的目录，我们需要通过读写锁来同步所有访问操作，特别是要防止一个写入者与一个或多个读取者/写入者并发访问。（只要没有并发写入者，多个并发读取者是没有问题的。）文件系统通常无法跨文件和目录提供此类锁定机制，因此与 Git 仓库相比，导出的代码目录实际上并没有太大用处。那么，标准的代码导出机制（例如导出到构件仓库如 JCenter 的 Java 源码 JAR 文件）存储的是不可变的构件，这也就不足为奇了：它们被一次性导出和写入，通过提交哈希进行唯一命名，此后便不再更改。这其实就是实现读写锁的一种方式……

协作式数据工程

现在，让我们将注意力转向数据工程工作流，并应用上述经验。正如 Git 对代码进行版本控制一样，Foundry 对数据也进行了版本控制：Foundry 数据集（以及其中包含的所有文件）是版本化的，并支持将分支作为沙箱机制。这意味着 Foundry 数据集支持安全的并发访问：由于每个数据集版本都是不可变的（就像 Git 提交是不可变的一样），不同的用户可以在某个特定的固定版本上读取数据集，同时其他用户可以更新同一个数据集；该用户执行的更改（例如添加或删除数据、更改模式等）会作为新版本¹ 进行存储，因此不会干扰其他用户的读取工作流。如果不同的用户分别写入各自的分支，他们甚至可以并发写入同一个数据集。

这种版本控制和分支功能对于数据管道尤为重要。数据管道是一系列数据转换步骤，将某些输入数据集转化为一个或多个经过细化/过滤/充实的派生数据集。由于输出数据集是版本化的，我们可以安全地调度此类转换作业，以跨数据平台更新数据集，而无需担心数据损坏的风险。

那么，“将 Foundry 数据集导出到 S3”意味着什么呢？就像 Git 仓库可以导出到普通文件系统一样，Foundry 数据集当然也可以导出到 S3 存储桶。（事实上，在幕后，Foundry 数据集通常已经存储在 S3 存储桶中了，就像 Git 仓库通常已经存储在文件系统中一样。）然而，通过将数据集从版本化文件系统导出到非版本化文件系统（S3²），导出的数据集会出现我们在前文 Git 案例中描述过的相同异常情况：

  • 我们必须确保同一时间只有一个导出作业在运行，否则不同的导出作业可能会互相覆盖破坏数据。（幸运的是，Foundry 的构建系统开箱即用地强制执行了这一约束。）
  • 导出数据集的读取者需要确保他们读取的是一致的导出快照。如果他们在导出作业正在更新数据集时读取数据，就会面临观察到不一致数据的风险。就像 Git 和编译器的案例一样，在最好的情况下，观察到的快照在语法上是无效的（读取者能够察觉到）；但在最坏的情况下，数据集在语法上是有效的，但在语义上却是不一致的。这种情况下，读取者甚至根本无法察觉到自己读取了错误的数据。

就像源代码的情况一样，我们可以通过导出数据集的不可变快照（例如导出到 S3 存桶中带有时间戳的目录）来解决并发问题。或者，我们可以写入一个原子的 LOCK 文件，通过它的存在来告知消费方当前目录不宜读取。这两种方法都要求读取者理解并遵守该机制：在第一种情况下，读取者需要不断将其源文件夹引用更新为最新可用的时间戳（也就是说，这不再是一个简单的“扁平”导出）；在第二种情况下，读取者在读取时必须关注 LOCK 文件³。

一种更明智的方法是使用 Foundry 的 Hadoop FileSystem 实现来替代标准的 Hadoop 实现，然后直接与 S3 或由 Hadoop 支撑的 Foundry 数据集进行交互。我们的 FileSystem 实现是标准 Hadoop JAR 的直接替代品，它提供了透明的版本控制和分支支持⁴，并且避免了维护数据集导出副本所带来的延迟和存储成本。请注意，这正是 Azure 和 AWS 在各自的 Hadoop FileSystem 实现中所采用的方法，它们将存储在块存储后端中的数据作为“虚拟”的 Hadoop 文件和文件夹供人访问；换言之，从编程和 API 的角度来看，通过 Foundry FileSystem 实现读写 Foundry 数据集，等同于通过 S3 FileSystem 实现读写由 S3 支撑的数据集。

结论

Foundry 的数据管理层通过将代码版本控制和沙箱化等软件工程最佳实践扩展到数据和代码，从而实现了协作式数据工程工作流；我们将此概念称为数据和代码的协同版本控制。像 Skywise 这样基于 Foundry 构建的大规模数据平台，为超过 10,000 名数据工程师和分析师在数据资产、数据模型、业务逻辑和数据驱动应用之间的协作提供了强大支撑。如果您想了解更多信息或为我们的旅程贡献力量，请与我们联系！

脚注

¹ 为了避免过高的存储成本，Foundry 存储了快照和差异的混合体，就像 Git 一样。这对于包含不可变记录的仅追加数据集（例如系统日志或传感器读数）尤为有用，这些数据集通常是我们客户使用的最大（且增长最快）的数据集。我们的版本控制方案对于此类数据集没有存储开销：在幕后，我们实际上将每个差异存储在底层文件系统中的单独文件夹中（例如，datasetA/diff1，datasetA/diff2，...），这样整个数据集就简单地由 datasetA/* 表示。（实际情况比这更复杂，因为用户可以有选择地从这些差异中删除文件，但您可以从中看出在这种情况下零存储开销的理念。）
² 虽然 S3 支持对象级版本控制，但一个数据集通常包含多个文件（例如 Parquet 文件）。S3 不支持对整个由 S3 支撑的 Hadoop 子目录进行原子版本控制。
³ 然而，细节决定成败……读取者该如何在整个读取过程中持续监控 LOCK 文件的不存在状态呢？
⁴ 除了版本控制和分支之外，我们的实现还提供了客户端/端到端加密。
