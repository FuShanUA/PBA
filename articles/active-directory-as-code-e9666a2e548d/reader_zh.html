聆听

Windows 自动化过去常常困难重重，至少也算不上直截了当，主要表现为依赖疯狂右键的部署方式，而基于 API 的管理往往是被事后才想起来的补丁。但时代正在变！随着 DevOps 的兴起、Windows Server 2016 的发布以及 PowerShell 生态的壮大，重新设计传统 Windows 基础设施的契机已经开启。

活动目录（AD）是较为普遍的基于 Windows 的系统之一，也是大多数企业环境的基石。对许多组织而言，它一直是在本地部署的。在 Palantir，我们 ❤️ 基础设施即代码（参见《堆栈溢出 Terraforming》和《门卫》），因此当受命在 AWS 中部署一套隔离、高可用且安全的 AD 基础设施时，我们开始探索如何将基础设施即代码实践应用于 AD。目标在于实现 AD 部署的自动化、可重复以及代码化配置。此外，我们希望将所有与补丁和配置管理相关的更新集成到 CI/CD 流水线中。

本文将逐步介绍我们解决此问题的方法，概述整个部署过程，包括使用 Packer 构建 AD AMI、使用 Terraform 配置 AD 基础设施，以及在 Vault 中存储配置密钥。

活动目录的 Packer 化

我们基础设施即代码的方法涉及通过更新和部署分层的、不可变镜像来管理配置。根据我们的经验，这种方式降低了系统的熵，将配置代码化，并且更契合 CI/CD 工作流，从而实现更快的迭代。

我们的 AD 镜像是建立在标准 Windows 镜像之上的下游层，使用 Packer、Jenkins 和 AWS CodeBuild 构建的自定义流水线完成。基础镜像包含自定义的期望状态配置（DSC）模块，用于管理 Windows 的各类组件、自动扩缩组（ASG）生命周期钩子，最重要的是管理安全工具。通过在基础镜像中完成这些配置，无论镜像最终被如何使用，我们都能强制推行安全和最佳实践。

创建标准化的 AD 镜像

AD 镜像可以按实例生命周期的逻辑组件划分：初始镜像创建、实例引导启动以及下线清理。

镜像创建

通常的最佳实践是在初始构建阶段前置尽可能多的逻辑，因为此过程只需执行一次，而引导脚本则在每个实例上运行。但对于 AD 镜像而言，这点相关性不大，因为 AD 镜像通常比较轻量，包依赖极少。

期望状态配置（DSC）模块

AD 配置向来是非常依赖 GUI 的工作流，极难实现自动化。近年来，PowerShell 已成为提升工程师工作效率的可靠选择，但管理配置偏移始终是个挑战。这时候 DSC 模块就派上用场了🎉。

DSC 模块是配置并保持 Windows 环境配置状态的绝佳方式，只需极少的用户干预。DSC 配置会在主机上定期运行，不仅能报告偏移，还能强化期望状态（类似于第三方配置工具）。

其中之一便是微软的 AD DSC 模块。为了说明 DSC 如何发挥倍增器的作用，这里有一个创建组的简单示例。对于单个组来说，这种写法可能显得兴师动众，但真正的好处在于，你可以用同样的精力遍历一组资源，如下所示。Groups 的初始内容可以在 Packer 构建中指定（静态 CSV），也可以通过外部查找动态生成。

DSC 配置示例

<#
    .SYNOPSIS
        示例展示如何摄取包含 N 个 AD 组的列表，
        并使用单个代码块创建各自的资源

        AD 组可以直接固化在 AMI 中，也可以从
        外部源获取
#>

$ConfigData = @{
    AllNodes = @(
        @{
            NodeName = '*'
            Groups = (Get-Content "C:\dsc\groups.csv")
        }
    )
}

Configuration NodeConfiguration 
{
    Import-DSCResource -ModuleName xActiveDirectory

    Node $AllNodes.NodeName {
        foreach ($group in $node.Groups) {
            xADGroup $group
            {
                GroupName = $group
                Ensure = "Present"
                # 其他参数
            }
        }
    }
}

NodeConfiguration -ConfigurationData $ConfigData

我们更进一步，构建了额外的模块来实现从零开始搭建集群。这些模块处理从配置核心 Windows 功能到部署新域控的一切事务。通过将这些任务实现为模块，我们免费获得了 DSC 固有的优势，例如重启韧性和配置偏移的缓解。

引导脚本

密钥。 处理静态凭证等配置密钥的问题，在 AD 这样敏感的环境中需要格外审慎。在磁盘上存储加密密钥、在引导时手动输入，或者两者结合，都不是理想的解决方案。我们寻找的方案需要能够：

  • 基于 API 驱动，以便接入我们的自动化体系
  • 解决安全引入问题，确保只有受信任的实例才能获取访问权限
  • 强制执行基于角色的访问控制，确保管理员（创建密钥者）与实例（使用密钥者）之间的职责分离
  • 强制执行可配置的访问窗口，限制实例只能在规定时段内访问所需密钥

基于上述标准，我们最终选择使用 Vault 来存储大多数自动化过程中的密钥。我们进一步打造了一套自动化管理角色和策略的生态系统，使我们能够大规模扩展的同时将管理开销降至最低。通过将 Vault 与 AWS 的 IAM 服务集成，我们可以轻松地为密钥赋予权限，并控制什么对象可以访问它们以及访问期限。加上适当的审计和控制，这给了我们两全其美的结果：自动化与安全的密钥管理。

除了确保配置实例的逻辑正确之外，同样重要的是进行验证，以减少将实例投入服务时的误报。AWS 提供了一个名为生命周期钩子的工具来实现这一点。由于生命周期钩子的完成操作是在引导脚本中手动调用的，因此脚本可以包含额外的逻辑，在声明实例投入服务之前验证设置和服务。

实例清理

生命周期中最后需要处理的部分是实例下线。在云中启动实例为我们带来了极大的灵活性，但我们也必须为节点不可避免的故障或用户主动发起的替换做好准备。发生这种情况时，我们会尝试尽可能优雅地终止实例。例如，我们可能需要转移灵活单主机操作（FSMO）角色并清理 DNS 条目。

我们选择使用简单的计划任务来实现生命周期钩子，以检查实例在 ASG 中的状态。当状态被设置为 Terminating:Wait 时，我们运行清理逻辑并显式完成终止钩子。我们知道生命周期钩子并不能保证完成或触发（例如，当实例遭遇硬件故障时），因此如果您对一致性有要求，应该考虑实现外部的清理服务或在引导脚本中添加额外逻辑。

集大成：Terraforming 活动目录

引导基础架构

现在 Packer 配置已经完成，是时候使用 Terraform 配置我们的 AD 基础设施并部署 AMI 了。我们通过创建并调用一个 Terraform 模块来实现这一点，该模块会自动引导我们的新林。引导新林包括部署一个主域控制器（DC）作为 FSMO 角色持有者，然后更新 VPC 的 DHCP 选项集，以便实例能够解析 AD DNS。

我们选择用于自动化 AD 林引导的设计模式是将流程分为两个不同的状态，只需在 Terraform 模块中更新所需变量（lifecycle, configure_dhcp_os）并应用它，即可在两者之间切换。

让我们来看看这两个状态下的模块调用，首先是引导状态，在此状态下我们将主 DC 部署到 VPC：

# Bootstrap Forest
module "ad" {
    source = "git@private-github:ad/terraform.git"

    env      = "staging"
    mod_name = "MyADForest"

    key_pair_name = "myawesomekeypair"
    vpc_id        = "vpc-12345"
    subnet_ids    = ["subnet-54321", "subnet-64533"]

    trusted_cidrs = ["15.0.0.0/8"]
    need_trusted_cidrs = "true"

    domain_name       = "ad.forest"
    domain_short_name = "ad"
    base_fqdn         = "DC=ad,DC=forest"
    vault_addr        = "https://vault.secret.place"

    need_fsmo   = "true"

    # Add me for step 1 and swap me out for step 2
    lifecycle = "bootstrap"

    # Set me to true when lifecyle = "steady"
    configure_dhcp_os = "false"
}

引导状态完成后，我们切换到稳定状态，在此状态下部署第二个 DC 并更新 DHCP 选项集。除了对 lifecycle 和 configure_dhcp_os 变量进行修改外，模块调用完全相同：

# Apply Steady State
module "ad" {
    source = "git@private-github:ad/terraform.git"

    env      = "staging"
    mod_name = "MyADForest"

    key_pair_name = "myawesomekeypair"
    vpc_id        = "vpc-12345"
    subnet_ids    = ["subnet-54321", "subnet-64533"]

    trusted_cidrs = ["15.0.0.0/8"]
    need_trusted_cidrs = "true"

    domain_name       = "ad.forest"
    domain_short_name = "ad"
    base_fqdn         = "DC=ad,DC=forest"
    vault_addr        = "https://vault.secret.place"

    need_fsmo   = "true"

    # Add me for step 1 and swap me out for step 2
    lifecycle = "steady"

    # Set me to true when lifecyle = "steady"
    configure_dhcp_os = "true"
}

使用这种设计模式，我们能够自动化整个部署过程，并根据需要在两个状态之间手动切换。通过使用 Terraform 中的 count 原语和插值函数，相关资源在这两个状态中被有条件地预置。

管理稳定状态

一旦 AD 基础设施进入稳定状态，我们会使用《门卫》通过使用更新后的 AMI 替换实例来更新配置和打补丁。我们以串行模式运行《门卫》，以便优雅地下线一个 DC，并通过启动带有新镜像的 DC 来替换它，正如上文“实例清理”部分所述。一旦第一个 DC 被替换，《门卫》将继续轮换下一个 DC。

结论

通过上述方法，我们成功创建了一个隔离、高可用的 AD 环境，并完全使用代码进行管理。它让安全之举成为了轻松之举，因为所有配置都存在于源代码控制中，我们能够使用基于 Git 的工作流，结合 2-FA，来把关并批准变更。此外，我们发现这种将补丁管理流程与 CI/CD 流水线绑定的方法，显著减少了阻力，从而大幅提升了补丁合规性。

除了在安全方面的斩获，我们还通过缓解配置偏移以及将代码作为文档源改善了运维体验。我们针对该林的灾难恢复策略，说白了也就是在不同区域重新部署代码，这一点也颇有助益。此外，像变更跟踪和同行评审这种通常专属于软件开发的好处，现在也被应用到了我们的 AD 运维流程中。
