合并应用更改

多名协作者共同开发同一个 Slate 应用时，可能需要将各自的更改合并在一起以实现版本控制。当两名或更多用户同时编辑同一版本的应用并尝试保存更改时，版本控制就显得尤为必要。由于第二次保存操作所基于的版本已被第一次保存操作更新为过期状态，如果不做处理直接保存，必然会导致其中一名用户的更改丢失。为解决此问题，第二名用户必须从以下选项中做出选择：

  • 放弃自己的更改。
  • 覆盖第一名用户的更改。
  • 使用其他名称保存应用。
  • 将自己的更改与第一名用户的更改合并。

以下部分将介绍合并流程。

合并冲突

当多名用户修改或删除同一个元素时，就会产生合并冲突。例如，一名用户删除了某个组件，而另一名用户却对其进行了修改，便会引发冲突。另一个引发冲突的例子是两名用户同时修改了同一个查询。

当原本可合并的组件 ID 发生碰撞时，也会出现冲突，这迫使用户只能保留其中一个版本。如果应用开发者没有遵循使用语义化组件名称的最佳实践，而是依赖默认组件名称，这种冲突会更为常见。

无冲突

如果保存时未检测到冲突，Slate 会自动合并应用 JSON，并显示合并后应用的预览。如果对更改满意，请选择接受并保存。如需修正，请选择进行更改查看并编辑 JSON。

存在冲突时合并

保存时若检测到合并冲突，系统会呈现更改的 JSON 视图。合并视图中包含三个独立的面板，分别展示另一名用户的更改、合并结果以及您本人的更改。一旦识别出合并冲突，Slate 不会尝试在单个元素内部合并更改，而是默认保留另一名用户的更改。

对于每一个冲突：

  1. 选择带有 Conflict 标签的元素以定位到冲突处。
  2. 修改生成的 JSON，使其保留您本人和另一名用户更改中希望留存的内容。
  3. 对合并结果满意后，选择标记为已解决。

解决所有冲突且对 JSON 更改满意后，请选择预览最终结果。随后系统会展示合并后应用的可视化预览，您可以选择保存该应用。
