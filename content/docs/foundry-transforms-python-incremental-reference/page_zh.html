增量使用指南

增量装饰器

本指南涉及增量构建与非增量构建。假设所有情况下都使用了 incremental() 装饰器。因此，这种术语只是指转换是否实际上被增量执行。

incremental() 装饰器可以用来将转换的计算函数包装上启用增量计算的逻辑：

Copied!
1import polars as pl
2from transforms.api import transform, incremental, Input, Output
3
4@incremental()
5@transform.using(
6    students=Input('/examples/students_hair_eye_color'),
7    processed=Output('/examples/hair_eye_color_processed')
8)
9def filter_hair_color(students, processed):
10    students_df = students.polars(mode='added')  # Or just students.polars() since 'added' is default
11    processed.write_table(students_df.filter(pl.col("hair") == 'Brown'))

使用 incremental() 装饰器可以包装任何使用 transform()、transform.using()、transform_df() 或 transform_pandas() 装饰器的现有转换。请注意，您的转换的 compute 函数 必须 支持既以增量方式又以非增量方式运行。incremental() 装饰器做两件关键的事情：

  • 它允许转换查找其上一次构建的信息。利用这些信息，incremental() 装饰器然后根据以下 描述的要求 决定是否可以根据要求进行增量计算。
  • 它将输入、输出和上下文对象转换为提供额外功能的增量子类。具体来说，TransformInput 变为 IncrementalTransformInput，TransformOutput 变为 IncrementalTransformOutput，而 TransformContext 变为 IncrementalTransformContext。这些增量对象随后被传递给由装饰器包装的转换。

增量装饰器接受六个参数：

Copied!
1transforms.api.incremental(
2    require_incremental=False,
3    semantic_version=1,
4    snapshot_inputs=None,
5    allow_retention=False,
6    strict_append=False,
7    v2_semantics=False
8)

将 require_incremental 参数设置为 True 会使转换在无法以增量方式运行时失败。在以下两种情况下，即使 require_incremental=True，转换也可以作为快照运行：

  1. 其中一个输出以前从未构建过。
  2. 语义版本已更改，意味着明确请求了快照。

要调试转换无法进行增量运行的原因，请在驱动程序日志中查找警告 transforms.api._incremental: Not running incrementally。

在 :func:~transforms.api.incremental 装饰器上的 semantic_version 参数允许您强制下一次转换运行为非增量。

  • 如果当前运行的语义版本与上一次运行的语义版本不同，转换将非增量运行。
  • 如果没有指定，语义版本将被设置为 1。
  • 如果上一次运行的语义版本不存在（例如，当将现有转换转换为增量转换时），则假定值 1。这允许转换开始进行增量运行，而无需新的快照。
  • 要强制转换的后续运行为非增量，您可以增加 semantic_version 参数上的 @incremental() 装饰器。

snapshot_inputs 参数允许您将一些输入定义为快照输入，与非快照输入不同，快照输入支持更新和删除操作。有关更多信息，请参见 快照输入。

将 allow_retention 参数设置为 True，可以在保持转换的增量性的同时，通过保留策略删除输入和输出数据集中的文件。

如果 strict_append 参数设置为 True 且输入数据集是增量的，则底层 Foundry 事务 类型将被设置为 APPEND，并且将使用 APPEND 事务进行增量写入。尝试覆盖现有文件会导致异常。 如果输入数据集不是增量的，strict_append 将以 SNAPSHOT 方式运行。您应该使用 require_incremental=True 来确保代码以 APPEND 方式运行。尝试覆盖现有文件将成功。 请注意，写入操作可能不会覆盖任何文件，甚至包括辅助文件，如 Parquet 摘要元数据或 Hadoop SUCCESS 文件。所有 Foundry 格式的增量写入都应该支持 strict_append 模式。

如果 v2_semantics 参数设置为 True，将使用 v2 增量语义。v2 和 v1 增量语义之间的行为不应该有任何差异，我们建议所有用户将其设置为 True。非目录的输入和输出资源只有在使用 v2 语义时才能以增量方式读取/写入。

重要信息

如上所述，您的转换（transform）的计算函数被incremental()装饰器包裹，必须支持以增量和非增量方式运行。默认读取和写入模式（在本指南其余部分有更详细的说明）可以协助满足这一双重逻辑需求，但可能仍需要根据is_incremental属性的计算上下文进行分支处理。

另一个关键点是，仅使用incremental()装饰器与transform_df()或transform_pandas()时，只能访问默认的读取和写入模式。如果您有一个转换，其新增的输出行仅取决于新增的输入行（参见追加示例）则此方法足够。 ）。然而，如果您的转换执行了更复杂的逻辑（例如连接、聚合或去重），需要您设置输入读取模式或输出写入模式，那么您应该使用incremental()装饰器与transform()。使用增量装饰器与transform()可以设置读取和写入模式。

警告

请注意，代码仓库预览功能始终会在非增量模式下运行转换。即使将require_incremental=True传递给incremental()装饰器也是如此。这与VS Code预览功能不同，后者会尊重增量性。

输入和输出的增量模式

增量转换是一个强大的功能。然而，第一次使用时API可能不太直观。在VS Code预览中，增量转换的行为与构建期间相同。我们建议使用 VS Code预览 当试验API并实际理解它时。

增量转换输入

transforms.api.IncrementalTransformInput 对象扩展了数据读取方法，以接受一个可选的读取模式参数。

如果您使用增量装饰器定义了一个转换，那么读取模式的行为会根据您的转换是按增量方式运行还是非增量方式运行而有所不同：

读取模式增量行为非增量行为
added *返回一个 DataFrame ↗，其中包含自上次运行转换以来添加到输入的新行。返回一个 DataFrame ↗，其中包含整个数据集，因为所有行都被视为 未见过的。
previous返回一个 DataFrame ↗，其中包含上一次运行转换时给出的完整输入。返回一个空的 DataFrame ↗。
current 返回一个包含当前运行整个输入数据集的 DataFrame ↗。返回一个包含当前运行整个输入数据集的 DataFrame ↗。这将与 added 相同。

默认的读取模式是 added。

在某些情况下，即使转换被标记为 incremental()，也不希望输入以增量方式处理。有关更多信息以及这些类型输入的读取模式行为如何不同的信息，请参阅 快照输入 部分。

请注意，默认的输出读取模式是 current，可用的输出读取模式包括 added、current 和 previous。有关输出读取模式的更多信息，请参阅下面的章节。

增量转换的性质意味着我们会从上一个 SNAPSHOT 事务开始加载输入数据集的所有过去事务，以构建输入视图。如果您开始注意到增量转换逐渐变慢，我们建议对您的增量输入数据集运行一个快照构建。

增量转换输出

transforms.api.IncrementalTransformOutput 对象提供了对输出数据集的读取和写入模式的访问。编写与增量构建和非增量构建都兼容的逻辑的关键是 modify 的默认写入模式。有两种写入模式：

  • modify：这种模式会用构建期间写入的数据修改现有的输出。例如，当输出处于 modify 模式时，调用 write_dataframe() 或 write_table() 会将写入的 DataFrame 追加到现有输出中。
  • replace：这种模式会用构建期间写入的数据完全替换输出。

当一个转换以增量方式运行时，输出的默认写入模式被设置为 modify。当一个转换以非增量方式运行时，输出的默认写入模式被设置为 replace。

请记住输入 DataFrame 的默认读取模式是 added。由于输入的默认读取模式是 added 和输出的默认写入模式是 modify，因此编写与增量和非增量构建兼容的逻辑变得更容易：

11

13    # When incremental, we read only new rows, and append them to the output.
14    processed.write_table(
15        new_students_df.filter(pl.col("hair") == 'Brown')

当需要计算正确的写入模式并手动设置时，增量计算有更复杂的使用场景。这可以通过在增量输出上使用 set_mode() 方法来实现。

当使用 transform() 或 transform.using() 装饰器时，可以手动设置输出写入模式。使用这些装饰器，你可以在显式调用 write 方法保存输出之前使用 set_mode()。然而，transform_df() 和 transform_pandas() 装饰器会调用 write_dataframe() 和 write_pandas() 来保存 DataFrame 输出。这意味着所使用的写入模式将由 incremental() 装饰器决定。

警告

使用 set_mode() 时，要确保在增量运行或非增量运行时都是有效行为。如果不是这样，你应该使用 is_incremental 属性。

除了写入模式外，transforms.api.IncrementalTransformOutput 还可以读取输出数据集中的 DataFrame。可以使用 data reading 方法如 pandas()、polars() 或 dataframe() 来完成，这些方法接受一个可选的读取模式参数。当使用 transform.using() 时，将模式作为关键字参数指定：output.pandas(mode='previous')。默认读取模式设置为 current，其他可用的输出读取模式是 added 和 previous。读取模式的行为会根据数据集的写入模式而有所不同。

尽管默认读取模式是 current，但在大多数情况下你应该使用 previous。其他读取模式应该用于在写入数据集后读取数据集。

增量转换上下文

与 TransformContext 对象相比，IncrementalTransformContext 对象提供了一个额外的属性：is_incremental。如果转换是按增量方式运行的，则此属性被设置为 True，这意味着：

  • 默认输出写入模式设置为 modify，并且
  • 输入的默认读取模式设置为 added。

增量模式摘要

增量装饰器允许您通过指定读取模式 "previous" 来访问转换的先前输入和输出。这样，您可以基于历史上下文进行当前构建。如果在快照模式下运行转换，则 "previous" 数据帧将为空，因为这是第一次运行，或者由于逻辑或数据发生了重大变化，需要重新计算。

然而，最常见的情况是对于输入使用 "added" 模式，对于输出使用 "modify" 模式。这些模式是默认使用的。它们可以让您检索输入数据集中新添加的行，处理它们，并将它们追加到输出数据集。

除了将行追加到输出外，您可能还想修改已经存在于输出数据集中的某些现有行。为此，请如常见场景示例中所示，使用 "replace" 模式。

假设 /examples/students_hair_eye_color 输入数据集被完全替换为一组新的学生。如我们所见，将新学生集追加到之前的输出结果中会导致错误的输出数据集。这是 incremental() 装饰器决定 不 以增量方式运行转换的一个示例。

要以增量方式运行一个转换，必须满足以下要求：

所有非快照（non_snapshot）输入数据集 自上次运行以来仅添加了文件 或 有从保留中产生的删除。

  • 如果某个输入的最后事务是修改现有文件的 UPDATE 或者不是来自保留的删除，则该输入只有在标记为 快照输入 时才会以增量方式运行。
  • 转换的 输入列表 可能会被修改
    • 不损害构建的增量性。
  • 对于多输出转换，所有 输出都是通过相同的转换最后构建的。

如果一个转换具有 incremental() 装饰器，但上述任何要求未满足，则该转换将自动非增量地运行。这意味着默认的输出写入模式将被设置为 replace 而不是 modify，输入将以非增量方式呈现。这也意味着在转换中读取输出将返回一个空的数据框，因为无法访问之前的记录。同样，输入也将以非增量方式呈现。如果我们设置 require_incremental=True，则转换会失败而不是非增量运行。

通常希望允许某些输入被完全重写而不影响转换的增量运行能力。有关更多信息，请参见 快照输入。

可以通过将 require_incremental=True 参数传递给 incremental 装饰器，强制转换仅以增量方式运行（除非它以前从未运行过或语义版本已提升）。如果转换无法以增量方式运行，它会故意失败而不是尝试非增量运行。

仅追加输入更改

如果自上次运行以来，其所有增量输入仅添加了文件（通过 APPEND 或 UPDATE 事务），则可以以增量方式运行转换。

相反，如果以下任何一项为真，则不能以增量方式运行转换：

  • 增量输入被完全重写（例如，有 SNAPSHOT 事务）。
  • 增量输入通过 UPDATE 或 DELETE 事务更新或删除了文件。

例如，如果 students_hair_eye_color 中的学生列表完全改变，则过滤后的学生之前的输出无效，必须替换。

从保留中删除的输入

如果上游数据集无限增长，而您希望能够删除旧行（使用 Foundry 中的保留）而不影响下游计算的增量性，那么依赖于该数据集的增量转换必须显式设置为允许保留的输入。这可以通过使用 allow_retention 装饰器的 transforms.api.incremental 参数来实现。

  • 如果此字段设置为 True，则在评估输入是否保持增量性时，将忽略来自保留策略的所有删除操作。这意味着来自保留的 removed 输入不会影响增量性，而且如果唯一非 added 的输入是具有保留行的输入，则转换仍将按增量方式运行。
  • 如果字段为 False（默认值），输入数据集中的任何 removed 类型的变化都会导致转换执行快照。

警告

快照输入

在某些情况下，输入可以被完全重写而不影响转换的增量性。例如，假设你有一个简单的参考数据集，它将电话号码国家代码映射到国家，并且会定期被重写。对该数据集的更改不一定会影响任何先前计算的结果，因此不应阻止转换的增量执行。

默认情况下，如上所述，如果自上次运行转换以来任何输入已被完全重写，则不能增量运行该转换。快照输入不参与此检查，它们的起始事务可以在不同运行之间有所不同。

可以通过在 snapshot_inputs 装饰器上使用 incremental() 参数来配置快照输入。

如果你的增量转换使用了 Cipher 资源 和 bellaso_python_lib，这些加密器、解密器和哈希器需要被列为快照输入。

快照输入的行为在转换以增量方式或非增量方式运行时是相同的。因此，added 和 current 读取模式将始终返回整个数据集。所有其他读取模式将返回空数据集。

由于对之前看到的快照输入版本没有任何限制，可以在保留以增量方式运行转换的能力的同时添加或删除快照输入。请记住，如果输入的修改从根本上改变了转换的语义，则值得审查是否需要更新 semantic_version 参数在 incremental() 装饰器上的设置。

输入的更改

现有输入列表可以被修改。在以下情况之一为真时，将保留增量性：

  • 添加了新的输入或新的快照输入。
  • 移除了现有的输入或现有的快照输入。

请注意，增量转换必须至少有一个输入。

每个非快照输入数据集的起始事务必须与前一次运行所使用的事务一致。

最后由同一转换生成的输出

对于 多输出增量转换，每个之前生成的输出的最后提交事务必须由相同的转换生成。

没有先前构建历史的输出将不受上述条件限制，并且不会阻止构建进行增量构建。

转换的输入和输出数据集必须不同

数据转换会接收一个输入数据集，执行操作，并生成一个输出数据集。转换的输入数据集和输出数据集必须不同。如果输入和输出都是同一个数据集，会导致循环（圆形）依赖，使得转换无法执行。

增量计算的要求摘要

只有当其所有增量输入仅添加了文件时，才能进行增量运行；如果删除了文件，则这些文件只能通过保留使用allow_retention=True进行删除。快照输入不在此检查范围内。
