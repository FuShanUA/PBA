增量使用指南

增量装饰器

本指南探讨的是增量构建与非增量构建的区别。这里假定在所有场景下均使用了 incremental() 装饰器。因此，相关术语仅用于表示数据转换是否实际以增量方式执行。

可以使用 incremental() 装饰器对数据转换的计算函数进行封装，从而实现增量计算逻辑：

`incremental()` 装饰器可用于包装任何使用了 `transform()`、`transform.using()`、`transform_df()` 或 `transform_pandas()` 装饰器的现有转换。请注意，转换的计算函数*必须*同时支持以增量方式和非增量方式运行。`incremental()` 装饰器主要完成两项关键工作： - 允许转换查询其前一次构建的相关信息。利用这些信息，`incremental()` 装饰器会根据下文所述的要求，判断该转换是否能够以增量方式运行。 - 将输入、输出和上下文对象转换为提供额外功能的增量子类。具体而言，`TransformInput` 会变为 `IncrementalTransformInput`，`TransformOutput` 会变为 `IncrementalTransformOutput`，而 `TransformContext` 会变为 `IncrementalTransformContext`。随后，这些增量对象将被传递给由该装饰器包装的转换。 增量装饰器接受六个参数：
已复制！
1transforms.api.incremental( 2 require_incremental=False, 3semantic_version=1, 4snapshot_inputs=None, 5allow_retention=False, 6strict_append=False, 7v2_semantics=False 8)

将 require_incremental 参数设置为 True 后，若转换无法以增量方式运行，则会直接失败。即便设置了 require_incremental=True，在以下两种情况下，转换仍允许以快照方式运行：

  1. 某个输出此前从未构建过。
  2. 语义版本发生变更，即明确请求了快照。

若要排查转换无法增量运行的原因，请在驱动日志中查找 transforms.api._incremental: Not running incrementally 警告信息。

:func:~transforms.api.incremental 装饰器上的 semantic_version 参数可强制下一次转换以非增量方式运行。

  • 若当前运行的语义版本与上一次不同，转换将以非增量方式执行。
  • 若未显式指定，语义版本默认为 1。
  • 若上一次运行的语义版本不存在（例如将现有非增量转换转为增量转换时），系统将假定其值为 1。这样转换即可直接开始增量运行，无需重新生成快照。
  • 要强制后续转换以非增量方式运行，只需提升 @incremental() 装饰器上的 semantic_version 参数值即可。

snapshot_inputs 参数用于将部分输入定义为快照输入，与非快照输入不同，此类输入支持更新和删除操作。详情请参阅 快照输入。

将 allow_retention 参数设置为 True 后，保留策略即可删除输入和输出数据集中的文件，同时不影响转换的增量特性。

若 strict_append 参数设置为 True 且输入数据集为增量类型，底层 Foundry 事务 类型将设为 APPEND，并在增量写入时使用 APPEND 事务。若尝试覆盖已有文件，将引发异常。 若输入数据集并非增量类型，strict_append 将以 SNAPSHOT 方式运行。此时应使用 require_incremental=True 确保代码以 APPEND 方式增量执行。此时尝试覆盖已有文件将会成功。 需注意，写入操作不得覆盖任何文件，即便是 Parquet 摘要元数据或 Hadoop SUCCESS 文件等辅助文件也不例外。所有 Foundry 格式的增量写入均应支持 strict_append 模式。

如果将 v2_semantics 参数设置为 True，将会使用 v2 增量语义。v2 与 v1 增量语义在行为上应无差异，我们建议所有用户都将此参数设置为 True。只有使用 v2 语义，才能以增量方式读取或写入非数据目录的输入和输出资源。

重要信息

如前所述，使用 incremental() 装饰器封装的转换计算函数必须同时支持增量与非增量两种运行方式。默认的读写模式（本指南后续部分将作详述）有助于应对这种双重逻辑需求，但在某些情况下，可能仍需根据计算上下文的 is_incremental 属性进行逻辑分支处理。

另一个关键点是，将 incremental() 装饰器与 transform_df() 或 transform_pandas() 搭配使用时，仅能使用默认的读写模式。如果转换操作中新增的输出行仅取决于新增的输入行（参见追加示例），这种默认模式便已足够。然而，如果转换操作涉及更复杂的逻辑（例如连接、聚合或去重），需要你自行设置输入读取模式或输出写入模式，则应将 incremental() 装饰器与 transform() 搭配使用。将增量装饰器与 transform() 结合，允许你自行设置读写模式。

警告

请注意，Code Repositories 预览功能会始终以非增量模式运行转换。即使向 incremental() 装饰器传入 require_incremental=True，情况也是如此。这一点与 VS Code 预览功能不同，后者会遵循增量特性。

输入与输出的增量模式

增量转换是一项强大的功能。然而，初次使用该 API 时可能会觉得不够直观。在VS Code 预览中，增量转换的行为与其在实际构建期间的表现完全一致。我们建议在摸索 API 用法和理解其实践机制时，使用VS Code 预览。

IncrementalTransformInput

transforms.api.IncrementalTransformInput 对象扩展了数据读取方法，增加了一个可选的读取模式参数。

如果使用增量装饰器定义转换，读取模式的行为会因转换是增量运行还是非增量运行而有所不同：

在某些情况下，尽管转换被标记为 incremental()，但不希望以增量方式处理输入。有关更多信息以及此类输入的读取模式行为有何不同，请参阅快照输入部分。

增量转换的特性意味着我们需要从上一个 SNAPSHOT 事务开始加载输入数据集上的所有过去事务，以构建输入视图。如果您开始发现增量转换变得越来越慢，我们建议您对增量输入数据集执行 SNAPSHOT 构建。

IncrementalTransformOutput

transforms.api.IncrementalTransformOutput 对象提供了对输出数据集读写模式的访问权限。编写兼容增量与非增量构建逻辑的关键在于默认写入模式 modify。共有两种写入模式：

  • modify：此模式使用构建期间写入的数据修改现有输出。例如，当输出处于 modify 模式时，调用 write_dataframe() 或 write_table() 会将写入的 DataFrame 追加到现有输出中。
  • replace：此模式将使用构建期间写入的数据完全替换输出。

以增量方式运行转换时，输出的默认写入模式设为 modify。以非增量方式运行转换时，输出的默认写入模式则设为 replace。

回顾可知，输入 DataFrame 的默认读取模式为 added。正是由于默认的输入读取模式为 added，且默认的输出写入模式为 modify，编写兼容增量与非增量构建的写入逻辑因此变得轻松得多：

在增量计算场景中，存在一些更复杂的用例，此时可能需要计算正确的写入模式并手动设置。可以通过增量输出上的 set_mode() 方法来完成此操作。

在使用 transform() 或 transform.using() 装饰器时，可以手动设置输出写入模式。通过这些装饰器，可以在显式调用写入方法保存输出之前，使用 set_mode()。然而，transform_df() 和 transform_pandas() 装饰器会调用 write_dataframe() 和 write_pandas() 来保存 DataFrame 输出。这意味着所使用的写入模式将由 incremental() 装饰器决定。

警告

使用 set_mode() 时，请确保无论转换是增量运行还是非增量运行，此行为都是有效的。如果不是这种情况，则应使用 is_incremental 属性。

除了写入模式外，transforms.api.IncrementalTransformOutput 还允许从输出数据集中读取 DataFrame。可以使用 pandas()、polars() 或 dataframe() 等数据读取方法来完成此操作，这些方法接受一个可选的读取模式参数。使用 transform.using() 时，将模式指定为关键字参数：output.pandas(mode='previous')。默认读取模式设置为 current，其他可用的输出读取模式为 added 和 previous。根据数据集设置的写入模式不同，读取模式的行为也会有所不同。

尽管默认的读取模式是 current，但在大多数情况下应使用 previous。其他读取模式应用于在向数据集写入数据后读取数据集。

IncrementalTransformContext

与 TransformContext 对象相比，IncrementalTransformContext 对象提供了一个额外的属性：is_incremental。如果转换以增量方式运行，该属性会被设为 True，这意味着：

  • 默认的输出写入模式设为 modify，并且
  • 默认的输入读取模式设为 added。

增量模式摘要

增量装饰器允许你通过指定读取模式为 "previous" 来访问转换之前的输入和输出。这样，你就可以基于历史上下文来进行当前构建。如果转换以快照模式运行，"previous" 数据框将为空，因为这是首次运行，或者是因为逻辑或数据发生了重大变化而需要重新计算。

然而，最常见的情况是对输入使用 "added" 模式，对输出使用 "modify" 模式。这些模式是默认使用的。它们让你能够从输入数据集中提取新增的行，进行处理，然后将其追加到输出数据集中。

相比于在输出中追加行记录，您可能需要修改输出数据集中已存在的某些行记录。针对此类需求，请使用“替换”模式，具体用法请参考常见场景示例。

假设 /examples/students_hair_eye_color 输入数据集被一组全新的学生数据完全替换。如我们所见，将这组新学生数据追加到之前的输出中会导致输出数据集出错。这种情况就展示了 incremental() 装饰器决定不以增量方式运行转换的一个场景。

要让转换以增量方式运行，必须满足以下要求：

转换的所有非快照输入数据集，要么自上次运行以来仅添加了文件，要么发生了由数据留存策略触发的删除操作。

  • 如果某个输入的最后一次事务是修改了现有文件的 UPDATE 操作，或者是非数据留存策略触发的删除操作，那么只有将其标记为快照输入时，该输入才会进行增量运行。
  • 可以在不影响构建增量性的前提下，对转换的输入列表进行修改。
  • 对于多输出转换，所有输出上次均须通过同一转换构建。

如果转换虽然使用了 incremental() 装饰器，但未能满足上述任何要求，系统将自动以非增量方式运行该转换。这意味着默认的输出写入模式将被设为 replace 而非 modify，且输入数据将以非增量形式呈现。这同时意味着，在转换中读取输出将返回一个空的数据帧，因为无法访问之前的历史记录。同理，输入也将以非增量形式呈现。如果设置 require_incremental=True，转换将直接报错失败，而不会以非增量方式运行。

通常情况下，我们希望允许某些输入被完全重写，同时不影响转换进行增量运行的能力。详情请参阅快照输入。

通过向 incremental 装饰器传入 require_incremental=True 参数，可以强制转换仅以增量方式运行（除非它从未运行过或语义化版本号发生了变更）。如果转换无法以增量方式运行，系统将故意使其失败，而不会尝试以非增量方式运行。

仅追加的输入变更

如果某个转换的所有增量输入自上次运行以来仅添加了文件（通过 APPEND 或 UPDATE 事务），则该转换可以增量运行。

反之，如果其增量输入存在以下任一情况，则该转换无法增量运行：

  • 增量输入被完全重写（例如，进行了 SNAPSHOT 事务）。
  • 增量输入通过 UPDATE 或 DELETE 事务更新或删除了文件。

例如，如果 students_hair_eye_color 中的学生列表完全改变，则之前输出的筛选后的学生数据将失效且必须被替换。

因数据保留策略而产生删除操作的输入

如果上游数据集无限增长，且您希望在不影响下游计算增量性的前提下删除旧记录（使用 Foundry 中的数据保留策略），则必须显式地将依赖于该数据集的增量转换设置为允许保留输入。这可以通过 transforms.api.incremental 装饰器的 allow_retention 参数来实现。

  • 如果此字段设置为 True，则在评估输入是否保持增量性时，将忽略所有来自数据保留策略的删除操作。这意味着来自数据保留策略的 removed 输入不会破坏增量性，并且如果仅有的非 added 输入是带有保留记录的输入，该转换仍将增量运行。
  • 如果该字段为 False（默认值），输入数据集中的任何 removed 类型更改都将导致该转换进行快照运行。

快照输入

在某些情况下，输入可以被完全重写而不会使转换的增量特性失效。例如，假设您有一个简单的参考数据集，用于将电话号码国家代码映射到国家，并且该数据集会定期重写。对该数据集的变更不一定会使先前任何计算的结果失效，因此不应阻止转换以增量方式运行。

如上所述，默认情况下，如果自上次运行转换以来有任何输入被完全重写，则转换无法以增量方式运行。快照输入将排除在此检查之外，并且允许它们在不同运行中的起始事务不同。

可以通过在 incremental() 装饰器上使用 snapshot_inputs 参数来配置快照输入。

如果您的增量转换使用了 Cipher 资源和 bellaso_python_lib，则需要将这些加密器、解密器和哈希器列为快照输入。

无论转换以增量还是非增量方式运行，快照输入的行为都是一致的。因此，added 和 current 读取模式将始终返回整个数据集。所有其他读取模式将返回空数据集。

鉴于对先前已见的快照输入版本没有任何约束，因此可以在添加或删除快照输入的同时，保留增量运行转换的能力。请记住，如果对输入的修改从根本上改变了转换的语义，建议审查是否应更新 incremental() 装饰器上的 semantic_version 参数。

输入的更改

可以修改现有输入的列表。在满足以下任一条件时，将保留增量特性：

  • 添加了新输入或新快照输入。
  • 删除了现有输入或现有快照输入。

请注意，增量转换必须至少具有一个输入。

每个非快照输入数据集的起始事务必须与上一次运行中使用的事务保持一致。

由同一转换最后构建的输出

对于多输出增量转换，每个先前构建的输出的最后提交事务必须由同一转换生成。

没有先前构建历史的输出将不受上述条件限制，并且不会阻止构建以增量方式运行。

转换的输入和输出数据集必须不同

数据转换接收一个输入数据集，执行操作，并生成一个输出数据集。转换的输入数据集和输出数据集必须不同。将同一数据集同时作为输入和输出会导致循环（环形）依赖，使转换无法执行。

增量计算要求摘要

当且仅当某转换的所有增量输入仅追加了文件时，该转换才可增量运行；若存在文件被删除，这些文件必须仅通过设置 allow_retention=True 的保留机制进行删除。快照输入不在此检查范围内。
