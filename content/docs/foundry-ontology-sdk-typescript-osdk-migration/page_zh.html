TypeScript OSDK 迁移指南（1.x 至 2.0）

本指南旨在帮助您将使用 TypeScript OSDK 1.x 的应用迁移至 TypeScript OSDK 2.0。以下章节将阐述两个版本之间的差异，并重点介绍语法和结构上的相关变更。

TypeScript 文档也可在开发者控制台的 /workspace/developer-console/ 中查阅。使用版本选择器选择 2.0 即可查看 TypeScript OSDK 2.0 的文档。

为什么从 TypeScript OSDK 1.x 升级到 2.0？

性能与可扩展性

TypeScript OSDK 1.x 允许用户在 Palantir 平台之外与本体进行交互。然而，该初始版本的可扩展性存在局限：

  • TypeScript OSDK 1.x 会为任何对象、Action 或其他本体项生成完整的代码实现。这意味着 OSDK 1.x 的规模与整个本体的大小成正比，而本体可能会非常庞大。
  • TypeScript OSDK 1.x 将客户端与生成的本体紧密耦合。

TypeScript OSDK 2.0 对 SDK 的生成方式和本体的访问机制进行了全面重构，从而在性能和易用性上带来了多项提升。

  • TypeScript OSDK 2.0 的规模仅与本体的形状和元数据呈线性关系，而不再取决于实际的本体。由于 OSDK 不再与本体内的所有项绑定，其性能得到了显著提升。TypeScript OSDK 2.0 还支持延迟加载，这意味着应用程序只会在需要时才加载所使用的内容。
  • TypeScript OSDK 2.0 将客户端与生成的代码分离，通过支持更快捷的库依赖更新，使得部署快速热修复更加容易，且无需重新生成 SDK。此外，TypeScript OSDK 2.0 还以指数级提升了 OSDK 代码库的代码复用能力。
  • TypeScript OSDK 2.0 提供了精简的代码生成机制，在面对大型本体进行构建时，可大幅提升迭代速度。

有关两个版本之间主要设计差异及其背后原因的更多详细信息，请查阅开源 TypeScript OSDK 代码库中的《开发者文档 ↗》。

支持与功能

全新 TypeScript 版本的 OSDK 增强了与本体（Ontology）的交互能力。因此，接口和媒体集等新功能的发布将需要 TypeScript OSDK 2.0 版本。

Palantir 将在 TypeScript OSDK 2.0 发布后，至少一年内继续对 1.x 版本的 TypeScript OSDK 提供维护支持。

如何从 TypeScript OSDK 1.x 升级到 2.0？

你可以在应用程序的 SDK 版本 选项卡中，使用 TypeScript OSDK 2.0 生成器为现有的开发者控制台应用生成新的 SDK 版本。现有应用程序还必须具备所需的依赖项才能使用新生成的 SDK。在开发者控制台中创建的新应用将默认使用 TypeScript OSDK 2.0 生成器生成 SDK，开发者可以按照 开始开发 选项卡中的说明快速上手。

从 TypeScript OSDK 1.x 迁移到 2.0

  1. 确保你的项目中已安装所有新所需的依赖项。
  2. 实例化全新的 TypeScript OSDK 2.0 客户端。
  3. 修改项目的查询调用，重点更新调用语法并根据需要调整返回类型的处理逻辑。
  4. 修改项目的操作调用。除了语法之外，主要变更是更新可能涉及验证或操作编辑响应的代码。
  5. 迁移对象类型。
    1. 全面审查代码中 OSDK 对象类型的使用方式。
      1. 确定代码库中依赖 OSDK 1.x 版本生成对象类型的位置，并了解它们与 2.0 版本所用类型的差异。
      2. 为您的代码库决定一条最优解决路径。
    2. 接下来，检查 OSDK 的使用情况，并修改对象加载方式以使用新客户端。请按照对象语法转换章节中的步骤操作，并留意属性类型的变更。
  6. 查看两个 OSDK 版本之间的其他差异。

创建

在 TypeScript OSDK 2.0 中，客户端通过 createClient 函数创建。该函数需要一个名为 ontologyRid 的附加参数。ontologyRid 参数可作为字符串传入，也可从环境变量读取，或者从 SDK 包中生成的新 $ontologyRid 变量中动态读取。

TypeScript OSDK 1.x（旧版）



已复制！
1import { FoundryClient, PublicClientAuth } from "@your-generated-sdk";
2
3//公开 OAuth
4const auth = new PublicClientAuth({
5        clientId: FOUNDRY_CLIENT_ID,
6        url: FOUNDRY_API_URL,
7        redirectUrl: APPLICATION_REDIRECT_URL,
8    });
9
10const legacyClient = new FoundryClient({
11    url: API_PROXY_TARGET_URL,
12    auth: auth,
13});



TypeScript OSDK 2.0




已复制！
1import { createClient } from "@osdk/client";
2import {
3    PublicOauthClient,
4    createConfidentialOauthClient,
5    createPublicOauthClient
6} from "@osdk/oauth";
7import { $ontologyRid } from "@your-generated-sdk";
8
9// 机密 OAuth
10const auth = createConfidentialOauthClient(
11    FOUNDRY_CLIENT_ID,
12    FOUNDRY_CLIENT_SECRET,
13    FOUNDRY_API_URL,
14);
15
16// 或者
17
18// 公共 OAuth
19const auth = createPublicOauthClient(
20    FOUNDRY_CLIENT_ID,
21    FOUNDRY_API_URL,
22    APPLICATION_REDIRECT_URL
23);
24
25const client = createClient(
26    FOUNDRY_API_URL,
27    $ontologyRid, // 见说明

如果你通过 CLI 安装本体（Ontology）的 OSDK，你的 $ontologyRid 将从 SDK 包的根目录导出，你可以按上述方式进行访问。否则，你需要将已知的 ontologyRid 传递到构造客户端的位置。你可以从 Foundry 中的 Ontology Manager 获取此信息。选择所需的本体，导航至 Ontology configuration 标签页，并从 Ontology metadata 部分复制你的本体 RID。


用法与通用语法


TypeScript OSDK 1.x（旧版）


在 TypeScript OSDK 1.x 中，你可以通过嵌套对象结构访问客户端上的方法。例如：


 legacyClient.objects.legacyObject.fetchPage();



TypeScript OSDK 2.0


在 TypeScript OSDK 2.0 中，客户端现在可以直接调用。这意味着你可以与客户端进行交互，并传入你要操作的对象或操作（Actions）。新的用法模式通常遵循以下语法：


client(myObject).fetchPage()
client($Objects.myObject).fetchPage()



对象


TypeScript 1.x 版本与 2.0 版本中对象类型的主要区别在于，访问对象数据所使用的类型不同。在 1.x 版本中，你可以通过 myObject 类型来访问对象的属性。然而在 2.0 版本中，旧版对象类型的类似功能被封装在了 Osdk.Instance<myObject> 类型中。


这体现在整个 SDK 的方法返回类型中。例如，TypeScript OSDK 1.x 的页面结果类型为 Page<myObject>，而 TypeScript OSDK 2.0 的页面结果类型为 PageResult<Osdk.Instance<myObject>>。


在转换代码时，这会引发你可能面临的若干兼容性问题。


问题：Osdk.Instance<myObject> 不等同于 legacyMyObject


这两个对象内部的属性发生了重大变化。


    


  • TypeScript OSDK 1.x 使用 GeoShape, GeoPoint，而 TypeScript OSDK 2.0 使用 GeoJSON
    • 

  • TypeScript OSDK 1.x 使用 Timestamp, LocalDate，而 TypeScript OSDK 2.0 将这些类型视作 strings
    • 



假设你想用新客户端替换原有的对象加载调用，但又不想改动那些需要旧版 TypeScript OSDK 1.x 对象类型的辅助函数。假设你仍然从旧版 TypeScript OSDK 1.x 导入对象类型，在这种情况下，由于这两种类型互不兼容，你会遇到报错。




已复制！
1const objectResult: Result<legacyMyObject, GetObjectError> =
2    await legacyClient.ontology.
3objects.legacyMyObject.fetchOne("<primaryKey");
4
5const object = client(myObject).fetchOne("<primaryKey>");
6   -> 返回 Osdk.Instance<myObjectV2>
7
8const locationName = getObjectLocation(object)
9
10function getLocation(obj: legacyMyObject){...} : 未修改



这是因为 Osdk.Instance<myObjectV2> 与 legacyMyObject 不兼容。


问题：ObjectType ≠ Osdk.Instance


在 TypeScript OSDK 2.0 中，使用 Osdk.Instance<> 包装对象类型对于访问各种属性至关重要。Osdk 包装器会将这些属性提升至该类型的顶层，使其更类似于 TypeScript OSDK 1.x 中的遗留类型。


然而，如果将对象类型的导入修改为来自 TypeScript OSDK 2.0，就会出现问题。若没有 Osdk 包装器，将无法在顶层访问对象属性，并且所导入的内容也会缺失上述列出的其他字段。




已复制！
1const object = client(myObjectV2).fetchOne("<primaryKey>");
2    -> 返回 Osdk.Instance<myObjectV2>
3const locationName = getObjectLocation(object)
4
5function getLocation(obj: myObjectV2){ ... }



这是因为 Osdk.Instance<myObjectV2> 与 myObjectV2 不兼容。


解决方案路径


在修改对象检索逻辑时，可考虑通过以下方式修改代码，以便在 TypeScript OSDK 2.0 中使用新的 OSDK 对象类型：


    

  1. 直接替换：在修改对象检索逻辑时，将代码库中旧版对象类型的所有实例替换为对应的 TypeScript OSDK 2.0 对象类型。此方案适用于简单的使用场景，但对于大量使用 TypeScript OSDK 1.x 对象类型的代码库而言，实施难度较大。
    2. 

  2. 自定义前端类型：在将对象迁移至 TypeScript OSDK 2.0 之前，先重构代码，改用自定义前端类型替代内置的旧版 TypeScript OSDK 1.x 类型。随后，编写一个转换函数，将旧版 TypeScript OSDK 1.x 类型映射到自定义类型。开始迁移至 TypeScript OSDK 2.0 时，只需更新该转换函数即可，代码库其余部分不受影响。
    3.

过滤


WHERE 子句


TypeScript OSDK 1.x (legacy)**


legacyClient.ontology.objects.legacyObject
    .where(query => query.legacyProperty.startsWith("foo"))...



TypeScript OSDK 2.0



client(myObject).where({
        myObjectProperty: { $startsWith: "foo" }
    })...



orderBy 子句


现在，orderBy 子句需在传递给 fetchPage 的对象中指定，而不再作为单独的过滤子句。


TypeScript OSDK 1.x (旧版)


client.ontology.objects.legacyObject
    .orderBy(sortBy => sortBy.OntologyProperty.asc())
    .fetchPageWithErrors({ pageSize: 30 });



TypeScript OSDK 2.0


client(myObject).fetchPage({
        $orderBy: { "OntologyProperty": "asc" }
        $pageSize: 30
    });



聚合


TypeScript OSDK 2.0 语法支持传入参数来设定聚合结果的排序方式。由此，在指定 groupBy 聚合时，你可以控制返回结果的排列顺序。可用的选项包括 unordered（无序）、asc（升序）和 desc（降序）。

操作


简单操作


TypeScript OSDK 2.0 中应用简单操作的语法与旧版 TypeScript OSDK 1.x 语法非常相似：两者都需要调用 applyAction 函数。


TypeScript OSDK 1.x (旧版)


await legacyClient.ontology.actions.createObject({ ...params });



TypeScript OSDK 2.0


await client(createObject).applyAction({ ...params });



批量操作


批量操作也需要像简单操作那样进行类似的语法更改。


TypeScript OSDK 1.x (旧版)


await legacyClient.ontology.batchActions.createObject({ ...params });



TypeScript OSDK 2.0


await client(createObject).batchApplyAction({ ...params });



操作校验与编辑


如果之前在 OSDK 1.x 中校验输入或接收 ActionEdits，现在可以通过在 Options 对象中指定 $returnEdits 和 $validateOnly 属性来定义这些行为。无法同时返回编辑结果和仅进行校验。


TypeScript OSDK 1.x (旧版)


await legacyClient.ontology.actions.`createObject`({ ...params },
    { mode: ActionExcecutionMode.VALIDATE_AND_EXECUTE,
    returnEdits: ReturnEditsMode.ALL });

// If you only want to validate
await legacyClient.ontology.actions.createObject({ ...params },
    { mode: ActionExcecutionMode.VALIDATE_ONLY });



TypeScript OSDK 2.0


// This call will throw on validation errors
await client(createObject).applyAction({ ...params }, { $returnEdits: true });

await client(createObject).applyAction({ ...params }, { $validateOnly: true });




查询


TypeScript OSDK 1.x 与 2.0 查询语法的差异，与 Actions 语法的变动十分相似。在 TypeScript OSDK 2.0 中，传入要执行的查询后，即可在客户端调用 executeFunction 对象。


TypeScript OSDK 1.x（旧版）


await legacyClient.queries.legacyQuery({ ...params });



TypeScript OSDK 2.0


await client(exampleQuery).executeFunction({ ...params });



其他差异


DateTime 处理


TypeScript OSDK 1.x 版本曾为处理日期和时间提供过自定义类型，例如 LocalDate 和 Timestamp。TypeScript OSDK 2.0 不再为 DateTime 提供具体类型。取而代之的是，对于 OSDK 中返回和接收 DateTime 值的操作，我们直接采用 ISO 8601 ↗ 国际标准格式的原始字符串。


使用 DateTime 属性时，OSDK 期望接收的是格式正确的 ISO 8601 字符串。例如，在使用 where 子句并基于 DateTime property 筛选对象集合时，必须提供格式正确的字符串：


client(myObject).where({
    "dateTimeProperty": {
        "$gt": "2010-10-01T00:00:00Z",
    }
});



各类 OSDK 方法返回的 DateTime 值也会采用这种格式。例如，加载对象时，对象上的 DateTime 属性将返回日期时间字符串。
