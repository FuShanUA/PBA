TypeScript OSDK 升级指南（1.x 到 2.0）

本指南旨在帮助您将使用 TypeScript OSDK 1.x 的应用程序升级到 TypeScript OSDK 2.0。以下章节解释了两个版本之间的差异，并强调了语法和结构上的相关更改。

TypeScript 文档也可在 /workspace/developer-console/ 的开发者控制台中平台内获取。使用版本选择器选择 2.0 以查看关于 TypeScript OSDK 2.0 的文档。

为什么从 TypeScript OSDK 1.x 升级到 2.0？

性能和可扩展性

TypeScript OSDK 1.x 允许用户在 Palantir 平台之外与本体论（Ontology）进行交互。然而，此初始版本的可扩展性有限：

  • TypeScript OSDK 1.x 会为任何对象（object）、操作（action）或其他本体论（ontology）项生成完整的代码实现。这意味着 OSDK 1.x 的规模会随着整个本体论（Ontology）的大小而变化，这可能非常庞大。
  • TypeScript OSDK 1.x 会紧密地将客户端与生成的本体论（Ontology）耦合。

TypeScript OSDK 2.0 通过彻底改进 SDK 的生成方式以及如何访问本体论（Ontology），提供了多项性能和可用性改进。

  • TypeScript OSDK 2.0 的扩展性是线性的，它基于本体论（Ontology）的结构和元数据，而不是实际的本体论（Ontology）。由于 OSDK 不再与您本体论（Ontology）中的所有项目相关联，这显著提升了 OSDK 的性能。TypeScript OSDK 2.0 还支持延迟加载（lazy loading），这意味着应用程序仅在使用时才需要其所用的部分。
  • TypeScript OSDK 2.0 将客户端与生成的代码分离，这使得通过启用更快的库依赖更新而不必重新生成 SDK 来部署快速热修复变得更加容易。TypeScript OSDK 2.0 还通过 OSDK 代码库显著提高了代码重用能力。
  • TypeScript OSDK 2.0 提供了简化的代码生成，当构建大型本体论时，可以显著提高迭代速度。

有关两个版本之间主要设计差异以及背后原因的更多详细信息，请查看开源 TypeScript OSDK 仓库中的 开发者文档 ↗。

支持与功能

OSDK 的新 TypeScript 版本增加了与本体论的交互能力。因此，诸如 接口 和 媒体集 等新功能发布将需要 TypeScript OSDK 2.0。

Palantir将在TypeScript OSDK 2.0发布后至少一年内继续支持OSDK的TypeScript版本1.x。

如何从TypeScript OSDK 1.x升级到2.0？

您可以使用应用程序中的SDK版本选项卡中的TypeScript OSDK 2.0生成器，为现有的开发者控制台应用程序生成新的SDK版本。现有应用程序还必须有所需的依赖项以使用新生成的SDK。在开发者控制台中创建的新应用程序默认会使用TypeScript OSDK 2.0生成器生成SDK，并且开发人员可以通过遵循开始开发选项卡中的说明快速入门。

初始设置

从npm安装@osdk/client包的最新版本，或者相应地更新项目的package-lock.json。

Copied!
1npm install @osdk/client
2npm install @osdk/oauth

客户端/认证

创建

TypeScript OSDK 2.0中的客户端使用createClient函数创建。此函数需要一个名为ontologyRid的附加参数。ontologyRid参数可以作为字符串传递，从环境变量中读取，或从新生成的$ontologyRid变量中动态读取，该变量在您的SDK包中生成。

TypeScript OSDK 1.x（遗留版本）

Copied!
1import { FoundryClient, PublicClientAuth } from "@your-generated-sdk";
2
3//Public OAuth
4const auth = new PublicClientAuth({
5        clientId: FOUNDRY_CLIENT_ID,
6        url: FOUNDRY_API_URL,
7        redirectUrl: APPLICATION_REDIRECT_URL,
8    });
9
10const legacyClient = new FoundryClient({
11    url: API_PROXY_TARGET_URL,
12    auth: auth,
13});

TypeScript OSDK 2.0

Copied!
1import { createClient } from "@osdk/client";
2import {
3    PublicOauthClient,
4    createConfidentialOauthClient,
5    createPublicOauthClient
6} from "@osdk/oauth";
7import { $ontologyRid } from "@your-generated-sdk";
8
9// Confidential OAuth
10const auth = createConfidentialOauthClient(
11    FOUNDRY_CLIENT_ID,
12    FOUNDRY_CLIENT_SECRET,
13    FOUNDRY_API_URL,
14);
15
16// Or
17
18// Public OAuth
19const auth = createPublicOauthClient(
20    FOUNDRY_CLIENT_ID,
21    FOUNDRY_API_URL,
22    APPLICATION_REDIRECT_URL
23);
24
25const client = createClient(
26    FOUNDRY_API_URL,
27    $ontologyRid, // See note
28    auth
29);

如果您通过 CLI 安装您的本体论的 OSDK，您的 $ontologyRid 将从 SDK 包的根目录导出，您可以如上所述访问。否则，您需要将您已知的 ontologyRid 传递到您构建客户端的位置。您可以通过 Foundry 中的本体论管理器获取此信息。选择您所需的本体论，导航到 本体论配置 选项卡，并从 本体论元数据 部分复制您的本体论 RID。

使用方法和通用语法

TypeScript OSDK 1.x（遗留版本）

在 TypeScript OSDK 1.x 中，您可以通过嵌套对象结构访问客户端上的方法。例如：

 legacyClient.objects.legacyObject.fetchPage();

TypeScript OSDK 2.0

在 TypeScript OSDK 2.0 中，客户端现在可以直接调用。这意味着您可以与客户端进行交互，并传入您想要操作的对象或操作。新的使用模式通常遵循以下语法：

client(myObject).fetchPage()
client($Objects.myObject).fetchPage()

对象

TypeScript 1.x 版本和 2.0 版本中对象类型的最主要区别在于用于访问对象数据的不同类型。在 1.x 版本中，你可以通过与 myObject 类型进行交互来访问对象的属性。然而，在 2.0 版本中，类似旧版对象类型的功能存在于包装类型中：Osdk.Instance<myObject>.

这在 SDK 中的方法返回类型中有所体现。例如，TypeScript OSDK 1.x 页面结果类型是 Page<myObject>，而 TypeScript OSDK 2.0 页面结果类型是 PageResult<Osdk.Instance<myObject>>。

这会在转换代码时带来一些不同的兼容性问题。

问题: Osdk.Instance<myObject> 不等同于 legacyMyObject

两个对象中的属性发生了重大变化。

  • TypeScript OSDK 1.x 使用 GeoShape, GeoPoint，而 TypeScript OSDK 2.0 使用 GeoJSON
  • TypeScript OSDK 1.x 使用 Timestamp, LocalDate，而 TypeScript OSDK 2.0 将这些类型视为 strings

想象你想要用新的客户端替换你的对象加载调用，但又不希望更改任何会从旧版 TypeScript OSDK 1.x 中期望对象类型的辅助函数。假设你仍然从旧版 TypeScript OSDK 1.x 中导入对象类型；在这种情况下，由于两种类型不兼容，你将遇到错误。

Copied!
1const objectResult: Result<legacyMyObject, GetObjectError> =
2    await legacyClient.ontology.
3    objects.legacyMyObject.fetchOne("<primaryKey");
4
5const object = client(myObject).fetchOne("<primaryKey>");
6   -> Returns Osdk.Instance<myObjectV2>
7
8const locationName = getObjectLocation(object)
9
10function getLocation(obj: legacyMyObject){...} : Unmodified

这是因为在 Osdk.Instance<myObjectV2> 和 legacyMyObject 之间不兼容。

问题：ObjectType ≠ Osdk.Instance

在 TypeScript OSDK 2.0 中，将对象类型包裹在 Osdk.Instance<> 中对于访问各种属性至关重要。Osdk 包装器会将这些属性提升到类型的最顶层，使其更类似于 TypeScript OSDK 1.x 中的旧类型。

然而，如果你修改对象类型导入以来自 TypeScript OSDK 2.0，就会出现一个问题。如果没有 Osdk 包装器，你就无法访问对象属性的最顶层，并且导入的字段将缺少上面列出的其他字段。

Copied!
1const object = client(myObjectV2).fetchOne("<primaryKey>");
2    -> returns Osdk.Instance<myObjectV2>
3const locationName = getObjectLocation(object)
4
5function getLocation(obj: myObjectV2){ ... }

这是因为 Osdk.Instance<myObjectV2> 与 myObjectV2 不兼容。

解决方案路径

考虑以下方式，您可以编辑代码以在TypeScript OSDK 2.0中使用新的OSDK对象类型：

  1. 直接替换： 在修改对象检索时，将代码库中所有旧对象类型的实例替换为相应的TypeScript OSDK 2.0对象类型。这对于简单的用例是最优的，但对于大量使用TypeScript OSDK 1.x对象类型的代码库来说具有挑战性。
  2. 自定义前端类型： 在将对象迁移到TypeScript OSDK 2.0之前，重构您的代码以使用自定义前端类型而不是内置的旧TypeScript OSDK 1.x类型。然后，开发一个转换函数，将旧的TypeScript OSDK 1.x类型映射到您的自定义类型。当您开始迁移到TypeScript OSDK 2.0时，只需更新转换函数，其余代码库保持不变。

语法翻译

本节包含一些简单的示例，说明如何在TypeScript OSDK 1.x和2.0客户端之间进行映射。有关更复杂的TypeScript OSDK 2.0语法示例，请参考 Palantir的公共GitHub仓库上的OSDK示例 ↗。

加载单个对象

TypeScript OSDK 1.x（旧版）

Copied!
1const objectResult: Result<legacyObject, GetObjectError> =
2    await legacyClient.ontology.objects.legacyObject.fetchOneWithErrors("<primaryKey>");

TypeScript OSDK 2.0

Copied!
1const objectResult: Result<Osdk.Instance<myObject>> =
2    await client(myObject).fetchOneWithErrors("<primaryKey>");

您也可以使用 fetchOne 方法来返回不带结果包装的对象。

Load objects with paging

TypeScript OSDK 1.x (legacy)

Copied!
1const objectResult =
2    await legacyClient.ontology.objects.legacyObject.fetchPage({ pageSize: 30 });

TypeScript OSDK 2.0

Copied!
1const objectResult = await client(myObject).fetchPage({ $pageSize: 30 });
2
3const object = objectResult.data[0];

Load all objects

TypeScript OSDK 1.x (legacy)

Copied!
1const objects: legacyObject[]= [];
2
3for await (const obj of client.ontology.objects.legacyObject.asyncIter()) {
4    objects.push(obj);
5}
6const object = objects.value[0];

TypeScript OSDK 2.0

Copied!
1const objects: myObject[]= [];
2
3for await(const obj of client(myObject).asyncIter()) {
4    objects.push(obj);
5}
6const object = objects.value[0];

Links

TypeScript OSDK 1.x (legacy)

Copied!
1const object = ...load ontology object with legacy client
2const link = object.legacyObjectProperty.fetchPage({ pageSize: 100 });

TypeScript OSDK 2.0

Copied!
1const object = ...load ontology object with new client
2const link = object.$link.myObjectProperty.fetchPage({ $pageSize: 100 });

Filtering

WHERE clauses

TypeScript OSDK 1.x (legacy)**
legacyClient.ontology.objects.legacyObject
    .where(query => query.legacyProperty.startsWith("foo"))...
TypeScript OSDK 2.0
client(myObject).where({
        myObjectProperty: { $startsWith: "foo" }
    })...

orderBy 子句

排序子句现在在传递给 fetchPage 的对象中指定，而不是单独的过滤子句。

TypeScript OSDK 1.x（旧版）
client.ontology.objects.legacyObject
    .orderBy(sortBy => sortBy.OntologyProperty.asc())
    .fetchPageWithErrors({ pageSize: 30 });
TypeScript OSDK 2.0
client(myObject).fetchPage({
        $orderBy: { "OntologyProperty": "asc" }
        $pageSize: 30
    });

聚合

TypeScript OSDK 2.0 语法允许您传入一个参数，用于指定聚合的顺序。这使您能够控制在指定 groupBy 聚合时结果返回的顺序。可用选项包括 unordered、asc（升序）和 desc（降序）。

返回对象数量

TypeScript OSDK 1.x（旧版）
Copied!
1const aggResults = legacyClient.ontology.objects.legacyObject.where(...).count().compute();
2if (isOk(aggResults)) {
3    const count: number = aggResults.value;
4}
TypeScript OSDK 2.0
Copied!
1const aggResults = client(myObject).where(...).aggregate({
2    select: { $count: "unordered" }
3    })
4const count: number = aggResults.$count;

按 groupBy 分组

TypeScript OSDK 1.x（旧版） 
legacyClient.ontology.objects.legacyObject.where(...)
    .groupBy(legacyObject => legacyObject.imageId.exact)
    .groupBy(legacyObject => legacyObject.objectLabel.exact)
    .count().
    .compute();
TypeScript OSDK 2.0
client(myObject).where(...).aggregate({ $select: { $count: "unordered" },
    $groupBy: { imageId: "exact", objectLabel: "exact" }});

Actions

Simple Actions

TypeScript OSDK 2.0's syntax for applying simple Actions is very similar to the legacy TypeScript OSDK 1.x syntax: in both, you call the applyAction function.

TypeScript OSDK 1.x (legacy)

await legacyClient.ontology.actions.createObject({ ...params });

TypeScript OSDK 2.0

await client(createObject).applyAction({ ...params });

Batch Actions

Batch Actions will require a similar syntax change as with simple Actions.

TypeScript OSDK 1.x (legacy)

await legacyClient.ontology.batchActions.createObject({ ...params });

TypeScript OSDK 2.0

await client(createObject).batchApplyAction({ ...params });

Action validations and edits

If you were validating inputs or receiving ActionEdits in OSDK 1.x, you can now define these by specifying $returnEdits and $validateOnly properties in the Options object. It is not possible to return edits and validateOnly at the same time.

TypeScript OSDK 1.x (legacy)

await legacyClient.ontology.actions.`createObject`({ ...params },
    { mode: ActionExcecutionMode.VALIDATE_AND_EXECUTE,
    returnEdits: ReturnEditsMode.ALL });

// If you only want to validate
await legacyClient.ontology.actions.createObject({ ...params },
    { mode: ActionExcecutionMode.VALIDATE_ONLY });

TypeScript OSDK 2.0

// This call will throw on validation errors
await client(createObject).applyAction({ ...params }, { $returnEdits: true });

await client(createObject).applyAction({ ...params }, { $validateOnly: true });

Queries

The change between TypeScript OSDK 1.x and 2.0 query syntax is very similar to the change for Actions. In TypeScript OSDK 2.0, you can call the executeFunction object on your client after you pass in the query that you want to execute.

TypeScript OSDK 1.x (legacy)

await legacyClient.queries.legacyQuery({ ...params });

TypeScript OSDK 2.0

await client(exampleQuery).executeFunction({ ...params });

Other differences

DateTime handling

The 1.x version of the TypeScript OSDK provided custom types for handling dates and times, such as LocalDate and Timestamp. TypeScript OSDK 2.0 does not provide concrete types for DateTime. Instead, for operations in the OSDK that return and accept DateTime values we instead use raw strings in the international standard of ISO 8601 ↗.

The OSDK expects properly formatted ISO 8601 strings when using DateTime properties. For example, when filtering an object set using a where clause using a DateTime property, you must provide a properly formatted string:

client(myObject).where({
    "dateTimeProperty": {
        "$gt": "2010-10-01T00:00:00Z",
    }
});

DateTime values from various OSDK methods will also be returned in this format. For example, when loading objects a DateTime property on an object will be returned with a datetime string.

Common utilities

OntologyObject

The OntologyObject type is not present in the TypeScript OSDK V2. A contextually equivalent type is the OsdkBase type found in the @osdk/api package.

IsOntologyObject

IsOntologyObject is not present in TypeScript OSDK V2.
