优化Postgres中的查询

本页专注于优化Postgres中的查询以实现最佳性能。我们通常建议在Ontology功能之上构建Slate应用程序，使用诸如 对象集（object sets） 和 操作（Actions） 等功能来读取和写入数据。

设计一个响应式应用程序需要仔细的规划和考虑。数据模型、查询结构和依赖关系图在应用性能和可用性中都起着重要作用。本指南专注于可以在Slate中编写的PostgreSQL查询的性能优化方面，并包括性能启发式方法、识别优化机会的方法以及提高性能的建议。

查询性能优化是一个迭代过程。有多种不同的方式可以编写相同的请求以产生“正确”的答案。

性能启发式

PostgreSQL查询的目标执行时间应为<= 500ms。对于数据集小于10m记录的大多数使用场景，这应该是可行的，但需要在设计数据模型时进行仔细规划和考虑。

在评估查询性能时，特别是使用新数据集时，在收集和分析统计数据之前，请确保多次运行查询。这将有助于确保数据集被正确缓存，并且性能更符合预期的使用情况。如果您想了解更多关于PostgreSQL缓存的信息，请参阅以下链接：

EXPLAIN... 解释

EXPLAIN 是 PostgreSQL 提供的一个非常有用的命令，用于返回查询执行计划。每当 PostgreSQL 接收到一个请求时，都会创建一个查询计划，该计划使用查询结构和数据属性来确定服务请求的最快方式。我们将从快速回顾 EXPLAIN 命令开始，因为这将在整个指南中被引用。

EXPLAIN

NODES

你可以将节点视为一个逻辑工作单元，或者查询评估中的一个步骤。节点以倒置图的形式返回，这意味着响应的第一行是最后执行的工作单元。每个节点前面都有->。

-> Index Scan using event_type_idx on event (...)

为什么这很重要 : 节点将用于识别查询执行计划中的低效操作，并帮助优先进行性能调优。

成本

第一个数字是启动成本（获取第一条记录的时间）。第二个数字是从启动到完成处理整个节点的成本。

(**cost=86.83..4577.07** rows=2368 width=10)

成本是一个< strong >估计值< /strong >，Postgres 查询规划器根据对象（通常是表）的< strong >统计信息< /strong >生成。虽然这个数字不代表实际运行时间，但应与实际执行直接相关。

成本是多个工作组件的组合：顺序获取、非顺序（随机）获取、行处理、处理操作符（函数）和索引条目处理。成本表示 I/O 和 CPU 活动；数字越大，Postgres 认为完成该任务需要做的工作就越多。需要注意的是，Postgres 查询优化器基于成本来决定使用哪种执行计划。

行数

此计划节点将输出的行数估计。

(cost=86.83..4577.07 **rows=2368** width=10)

为何重要: 行数可用于识别返回大量数据和/或行为不符合预期的节点。

宽度

该节点输出的行的平均大小（以字节为单位）的估计值。

(cost=86.83..4577.07 rows=2368 **width=10**)

为何重要: 宽度可用于识别输出具有非常大的属性或大量列的行的节点。

EXPLAIN ANALYZE

节点

参见上文。

实际时间

与成本类似，第一个数字是启动所需的实际时间（毫秒 ms）。第二个数字是处理整个节点从启动到完成的实际时间。

(**actual time=10.313..12.530** rows=4857 loops=1)

顾名思义，实际时间是通过执行语句来捕获的。关键字 ANALYZE 告诉 Postgres 执行查询并显示执行计划。如果您在查询超时时遇到困难，移除 ANALYZE 将只返回查询计划，这应该比执行查询快得多。

为何重要: 这是表明哪个节点或操作导致性能问题的最清晰指标。

行数

节点输出的估计行数。

(actual time=10.313..12.530 **rows=4857** loops=1)

为什么这很重要：ROWS 可以帮助提供上下文，说明为什么某个操作可能比预期的耗时更长。

循环

报告节点的执行总数。显示的实际时间和行数是每次执行的平均值。将 LOOPS 值乘以实际时间即可得到在该节点上花费的总时间。

(actual time=10.313..12.530 rows=150 **loops=10**)

理解操作（计划节点）

扫描

  • sequential scans (顺序扫描): 顺序扫描操作会扫描磁盘上存储的整个关系（表）（类似于 TABLE ACCESS FULL）。可以在任何关系上执行顺序扫描；无论关系的模式、大小、约束和索引是否存在。
    • 顺序扫描的特点如下：
      • 启动速度快（顺序 I/O 比随机访问快得多）。
      • 每个块只读一次。
      • 输出无序。
  • index scans: 索引扫描执行 B-tree 遍历，遍历叶节点以找到所有匹配条目，并获取相应的表数据。它类似于 INDEX RANGE SCAN 然后是 TABLE ACCESS BY INDEX ROWID 操作
    • 索引扫描的特点如下：
      • 随机访问比顺序 I/O 慢得多。
      • 需要额外的 I/O 来访问索引。
      • 可能会多次读取相同的块。
      • 输出有序。
  • bitmap index/heap scans: 一个普通的索引扫描一次从索引中获取一个元组指针，并立即访问表中的该元组。位图扫描一次性从索引中获取所有元组指针，使用内存中的“位图”数据结构对它们进行排序，然后按照物理元组位置顺序访问表中的元组。
    • 以下是一个位图索引/堆扫描的特征：
      • 具有索引选择性的顺序 I/O。
      • 启动较慢，因为需要读取并排序所有索引元组。
      • 通常用于 IN 和 =ANY(array) 运算符，以及低选择性索引扫描。
      • 可以结合多个索引。
      • 输出无序。
  • index only: 索引仅扫描执行 B-tree 遍历，并遍历叶节点以查找所有匹配项。由于索引包含满足查询的所有列（MVCC 可见性信息除外），因此不需要表访问。

连接

连接操作通常一次只处理两个表。当查询涉及连接多个表时，连接是按顺序执行的：首先连接两个表，然后将中间结果与下一个表连接，依此类推。在连接的上下文中，术语“表”也可以指“中间结果”。

  • nested loop: 通过从一个表获取结果，并对第一个表中的每一行查询另一个表来连接两个表。
    • 通常是性能最差的连接形式。
    • 生成第一条记录的速度很快。
    • 如果第二个子节点较慢，可能会导致性能下降。
    • 唯一能够执行 CROSS JOIN 的连接方式。
    • 唯一能够执行不等式连接条件的连接方式。
  • merge join: (排序) 合并连接像拉链一样合并两个已排序的列表。连接的两边都必须预先排序。
    • 只能用于等值连接条件。
    • 对于大数据集来说，通常性能最好。
    • 需要有序输入 - 这可能需要缓慢的排序或索引扫描。
    • 启动速度较慢，因为所有索引元组都会被读取和排序。
  • hash joins: 哈希连接将连接一侧的候选记录加载到哈希表中（计划中用 Hash 标记），然后对连接另一侧的每条记录进行探测。
    • 只能用于等值连接条件。
    • 通常在将大表与小表连接时性能最佳。
    • 仅适用于可哈希的数据类型。
    • 由于需要对较小的表进行哈希处理，开始时速度较慢。
    • 如果表统计信息过时且不正确，性能会受到负面影响。

聚合

  • GroupAggregate: 按照分组子句对已排序的集合进行聚合。此操作不会缓冲大量数据。
  • HashAggregate: 使用临时哈希表来对记录进行分组。HashAggregate 操作不需要预先排序的数据集；相反，它使用大量内存来生成中间结果（非管道化）。输出没有任何有意义的顺序。
  • Unique
  • WindowAgg：表示使用窗口函数。

其他

  • Sort
    • 与 ORDER BY、DISTINCT、GROUP BY、UNION 和合并连接一起使用。
    • 启动时间较长。
    • 如果排序可以放入 work_mem，则可以使用快速排序。
    • 如果排序不能放入内存，则会溢出到磁盘并使用临时文件，这可能非常昂贵。
  • Limit
    • 处理 LIMIT 和 OFFSET。
    • 如果没有 WHERE 子句，可以用于 min() 和 max()。
    • 对于 OFFSET 被跳过的记录仍然会被生成/实例化，但在返回结果之前被丢弃。
    • 子扫描的成本仍然是全部成本。
    • 与限制结合的排序可以使用优化的排序形式。

查询调优最佳实践

查询优化最佳实践

SELECT

  • 仅包括显示所需的属性，这将有助于限制获取的页面以支持请求。

DISTINCT

  • DISTINCT 来自 PostgreSQL 代码库的较旧部分，并使用了效率较低的方法来识别唯一记录。
  • 如果可能的话，避免使用 DISTINCT，可以使用 GROUP BY 或子查询。

JOIN

  • When joining tables, try to use a simple equality statement in the ON clause (such as a.id = b.person_id). Doing so allows more efficient join techniques to be used (such as Hash Join rather than Nested Loop Join).
  • Convert subqueries to JOIN statements when possible as this usually allows the optimizer to understand the intent and possibly chose a better plan.
  • Use JOINs properly: Are you using GROUP BY or DISTINCT just because you are getting duplicate results? This usually indicates improper JOIN usage and may result in a higher costs.
  • If the execution plan is using a Hash Join it can be very slow if table size estimates are wrong. Therefore, make sure your table statistics are accurate by reviewing your vacuuming strategy.
  • Avoid correlated subqueries where possible; they can significantly increase query cost.
  • Use EXISTS when checking for existence of rows based on criterion because it “short-circuits” (stops processing when it finds at least one match).

WHERE clause

  • Avoid LIKE if possible.
  • Avoid passing large lists into an IN() statements - instead, consider using a JOIN condition or adjusting logic to be exclusion-based.
  • Avoid function calls in the WHERE clause.

UNION vs UNION ALL

UNION will eliminate duplicate records; this requires sorting the tables in question.

We strongly suggest avoiding the use of UNION given the cost associated with the request. There should be very few instances where this is required and, if so, it should be done in the Transforms layer before querying from Slate.

UNION ALL will not eliminate duplicates and will efficiently append the rows of one table to another.

See Common Mistakes: UNION VS. UNION ALL ↗.

Indexes

Considering adding indexes under the following conditions:

  • Eliminate sequential scans (seq scan), unless it's a small table and/or the query is fetching more than 5% of the rows.
  • If using a multi-column index, pay attention to the order in which you define the included columns.
  • Use indexes that are highly selective on frequently-used columns.

Identify missing indexes

Copied!
1SELECT
2  relname,
3  seq_scan - idx_scan AS too_much_seq,
4  CASE
5    WHEN seq_scan - coalesce(idx_scan, 0) > 0 THEN 'missing index?'
6    ELSE 'OK'
7  END,
8  pg_relation_size(relname::regclass) AS rel_size, seq_scan, idx_scan
9FROM pg_stat_all_tablesWHERE schemaname = 'public'AND pg_relation_size(relname::regclass) > 80000
10ORDER BY too_much_seq DESC;

Identify unused indexes

Copied!
1SELECT
2  indexrelid::regclass as index,
3  relid::regclass as table,
4  'DROP INDEX ' || indexrelid::regclass || ';' as drop_statement
5FROM pg_stat_user_indexes
6JOIN pg_index USING (indexrelid)
7WHERE idx_scan = 0AND indisunique is false;
