优化 Postgres 中的查询

本页重点介绍如何对 Postgres 中的查询进行调优以获取最佳性能。我们通常建议在 Ontology 能力的基础上构建 Slate 应用程序，并使用对象集和操作等功能来进行数据的读写。

设计响应迅速的应用程序需要周密的规划与考量。数据模型、查询结构以及依赖图各自在应用程序的性能和可用性方面发挥着重要作用。本指南重点探讨在 Slate 中编写 PostgreSQL 查询时的性能调优问题，涵盖性能启发式规则、识别调优机会的方法以及提升性能的建议。

查询性能调优是一个迭代过程。为了获得“正确”的结果，同一请求可以有多种不同的编写方式。

性能启发式规则

PostgreSQL 查询的目标执行时间设定为小于等于 500 毫秒。对于涉及数据集小于 1000 万条记录的大多数用例而言，这应当是可以实现的，尽管在设计数据模型时仍需进行周密的规划与考量。

在评估查询性能时，尤其是面对全新数据集，请务必在收集和分析统计信息前多次运行查询。这有助于确保数据集已被妥善缓存，且性能表现更贴合预期使用情况。如果您想深入了解 PostgreSQL 缓存，请参阅以下链接：

  • 《PostgreSQL 物理存储简介》
  • 《深入探究物理存储》
  • 《Postgres 官方文档》

EXPLAIN 命令详解

EXPLAIN 是 Postgres 提供的一个极为实用的命令，用于返回查询执行计划。Postgres 收到的每个请求都会生成一份查询计划，该计划利用查询结构和数据属性来确定满足该请求的最快方式。我们将首先简要回顾 EXPLAIN 命令，因为本指南全文都会引用此命令。

解释

节点

你可以将节点视为一个逻辑工作单元，或查询求值中的一个步骤。这些节点以倒序图的形式返回，这意味着响应的第一行是最后执行的工作单元。每个节点前面都带有 ->。

-> Index Scan using event_type_idx on event (...)

为什么这很重要：这些节点将用于识别查询执行计划中的低效操作，并帮助确定性能调优工作的优先级。

成本

第一个数字是启动成本（检索第一条记录的时间）。第二个数字是从启动到完成整个节点的处理成本。

(**cost=86.83..4577.07** rows=2368 width=10)

成本是 Postgres 查询规划器基于对象（通常是表）统计信息生成的估计值。虽然这个数字并不代表实际运行时间，但它应与实际执行情况直接相关。

成本是多个工作组成部分的集合：顺序获取、非顺序（随机）获取、行处理、处理操作符（函数）以及处理索引项。成本反映了 I/O 和 CPU 活动；数字越大，Postgres 认为完成该任务所需的工作量就越大。需要注意的是，Postgres 查询优化器会根据成本来决定使用哪种执行计划。

行数

该计划节点预计输出的行数。

(cost=86.83..4577.07 **rows=2368** width=10)

为什么这很重要：行数可用于识别返回大量数据且/或表现异常的节点。

宽度

节点输出行的预计平均大小（以字节为单位）。

(cost=86.83..4577.07 rows=2368 **width=10**)

重要性说明：WIDTH 可用于识别输出具有超大属性或大量列的数据行所在的节点。

EXPLAIN ANALYZE

节点

见上文。

实际时间

与成本类似，第一个数值表示启动所需的实际时间（毫秒，ms）。第二个数值表示从启动到完成整个节点处理所需的实际时间。

(**actual time=10.313..12.530** rows=4857 loops=1)

顾名思义，实际时间是通过执行语句捕获的。关键字 ANALYZE 指示 Postgres 在展示执行计划的同时执行查询。如果查询超时令人困扰，移除 ANALYZE 将仅返回查询计划，其速度应显著快于执行查询本身。

重要性说明：这是表明哪些节点或操作引发了性能问题的最直观指标。

行数

该节点输出的预估行数。

(actual time=10.313..12.530 **rows=4857** loops=1)

重要性说明：ROWS 可帮助提供上下文，说明为何某项特定操作所耗时间超出预期。

循环次数

报告该节点的总执行次数。展示的实际时间和行数值为每次执行的平均值。将 LOOPS 值乘以实际时间，即可得出在该节点耗费的总时间。

(actual time=10.313..12.530 rows=150 **loops=10**)

理解操作（计划节点）

扫描

    - `sequential scans` (顺序扫描)：Seq Scan 操作扫描存储在磁盘上的整个关系（表）（类似于 TABLE ACCESS FULL）。无论关系模式、大小、约束以及是否存在索引，始终可以对关系执行顺序扫描。 - 顺序扫描的特征如下： - 启动快（顺序 I/O 比随机访问快得多）。 - 每个数据块只读取一次。 - 产生无序输出。 - `index scans`：Index Scan 执行 B 树遍历，走过叶子节点查找所有匹配项，并获取相应的表数据。类似于 INDEX RANGE SCAN 后接 TABLE ACCESS BY INDEX ROWID 操作。 - Index Scan 的特征如下： - 随机访问比顺序 I/O 慢得多。 - 需要额外的 I/O 访问索引。 - 可能多次读取同一个数据块。 - 产生有序输出。 - `bitmap index/heap scans`：普通的 Index Scan 每次从索引中获取一个元组指针，并立即访问表中的该元组。位图扫描一次性从索引中获取所有元组指针，使用内存中的“位图”数据结构进行排序，然后按物理元组位置的顺序访问表元组。 - 位图索引/堆扫描的特征如下： - 具备索引选择性的顺序 I/O。 - 启动慢，因为需要读取并排序所有索引元组。 - 常用于 IN 和 =ANY(array) 操作符，以及低选择性的索引扫描。 - 可以组合多个索引。 - 产生无序输出。 - `index only`：Index Only Scan 执行 B 树遍历并走过叶子节点查找所有匹配项。不需要访问表，因为索引包含了满足查询所需的所有列（MVCC 可见性信息除外）。 ### 连接 连接操作通常一次只处理两个表。当查询涉及连接两个以上的表时，这些连接会按顺序执行：先连接两个表，然后将中间结果与下一个表连接，依此类推。在连接的语境中，“表”一词因此也可以指“中间结果”。

    • nested loop：通过从一个表获取结果，并针对该表的每一行查询另一个表，来实现两个表的连接。
      • 通常是性能最差的连接方式。
      • 生成第一条记录的速度快。
      • 如果第二个子节点响应缓慢，则可能出现负面的性能影响。
      • 是唯一能够执行交叉连接（CROSS JOIN）的连接方式。
      • 是唯一能够处理不等值连接条件的连接方式。
    • merge join：（排序）合并连接像拉链一样合并两个已排序的列表。连接的两端都必须预先排序。
      • 只能用于等值连接条件。
      • 通常是处理大型数据集时性能最佳的连接方式。
      • 要求输入有序——这可能需要缓慢的排序或索引扫描。
      • 启动缓慢，因为需要读取并排序所有索引元组。
    • hash joins：哈希连接将连接一端的候选记录加载到哈希表（在执行计划中标记为 Hash）中，随后针对连接另一端的每条记录探测该哈希表。
      • 只能用于等值连接条件。
      • 通常是连接大表与小表时性能最佳的方式。
      • 仅适用于可哈希的数据类型。
      • 由于需要对较小的表进行哈希处理，启动较为缓慢。
      • 如果表统计信息过期或不准确，会对性能产生负面影响。

    聚合

    • GroupAggregate：根据 group by 子句对预先排序的数据集进行聚合。此操作不会缓存大量数据。
    • HashAggregate：使用临时哈希表对记录进行分组。HashAggregate 操作不需要预先排序的数据集；相反，它会消耗大量内存来物化中间结果（非流水线方式）。其输出没有任何有意义的顺序。
    • Unique
    • WindowAgg：表示使用了窗口函数。

    杂项

    • Sort
      • 发生在 ORDER BY、DISTINCT、GROUP BY、UNION 和合并连接（merge joins）中。
      • 启动时间较长。
      • 如果排序操作能放入 work_mem，则可以使用快速排序（quicksort）。
      • 如果排序操作无法完全放入内存，将溢出到磁盘并使用临时文件，这会带来极大的开销。
    • Limit
      • 同时处理 LIMIT 和 OFFSET。
      • 如果没有 WHERE 子句，可用于实现 min() 和 max()。
      • 针对 OFFSET 跳过的记录仍会被生成或物化，只是在返回结果前被丢弃。
      • 子扫描的成本仍然是全量成本。
      • 排序结合 limit 可以使用优化形式的排序。

    查询调优最佳实践

    SELECT

    • 仅包含需要显示的属性，这有助于减少为支持请求而获取的数据页数量。

    DISTINCT

    • DISTINCT 源于 PostgreSQL 代码库中较早的部分，在识别去重记录时使用的方法效率较低。
    • 如果可能，请使用 GROUP BY 或子查询来避免使用 DISTINCT。

    JOIN

    • 在进行表连接时，尽量在 ON 子句中使用简单的等值条件（例如 a.id = b.person_id）。这样能够启用更高效的连接技术（例如使用哈希连接而非嵌套循环连接）。
    • 尽可能将子查询转换为 JOIN 语句，因为这通常能让优化器更好地理解查询意图，从而可能选择更优的执行计划。
    • 正确使用 JOIN：你是否仅仅因为得到了重复结果才去使用 GROUP BY 或 DISTINCT？这通常意味着 JOIN 使用不当，并可能导致更高的开销。
    • 如果执行计划采用哈希连接（Hash Join），一旦表大小估算出现偏差，极易导致严重的性能劣化。因此，务必审查并优化数据清理策略，确保表统计信息的准确性。
    • 尽量避免使用关联子查询，这会大幅推高查询成本。
    • 检查行是否存在时，推荐使用 EXISTS。其具备“短路”特性（一旦匹配到首条记录即刻终止后续处理）。

    WHERE 子句

    • 尽可能避免使用 LIKE。
    • 避免向 IN() 语句中传入庞大的列表，可改用 JOIN 条件或将逻辑调整为基于排除的处理方式。
    • 杜绝在 WHERE 子句中进行函数调用。

    GROUP BY 与 GROUPING SET

    参见《PostgreSQL GROUPING SETS 官方文档 ↗》。

    UNION 与 UNION ALL

    UNION 会清除重复记录，这要求对相关表执行排序操作。

    鉴于其高昂的执行成本，强烈建议避免使用 UNION。实际业务中鲜有必须使用它的场景；若确有必要，应在从 Slate 发起查询之前，于 Transforms 层中完成此操作。

    UNION ALL 不去重，能够高效地将一张表的数据行追加至另一张表。

    参见《常见误区：UNION 与 UNION ALL ↗》。

    索引

    在以下情况下，可考虑添加索引：

    • 旨在消除顺序扫描，除非处理的是小表，或者查询获取的数据行占比超过 5%。
    • 采用多列索引时，需谨慎设定包含列的先后顺序。
    • 在频繁调用的列上，应建立高选择性的索引。
