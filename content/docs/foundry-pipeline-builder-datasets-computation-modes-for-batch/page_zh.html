批处理输入数据集的计算模式

根据具体的使用场景，你可以选择以快照或增量的方式读取输入数据集。

快照计算

快照计算会对整个输入数据集执行转换操作，而不仅仅是处理新增数据。每次构建时，输出数据集都会被最新的管道输出完全覆盖。

最适用于以下场景：

  • 输入数据集未通过 APPEND 事务进行更新。
    • 当输入数据使用 SNAPSHOT 事务写入时，无法以增量方式读取该输入。
  • 输出数据集无法通过 APPEND 事务进行更新。
    • 示例：每次运行时整个输出数据集都会发生改变，因此需要采用快照输出。
  • 输入数据集较小。
    • 在这种情况下，快照计算与增量计算的效率相当。

增量计算

增量计算仅对自上次构建以来追加到所选输入中的新数据执行转换操作。这种方式能够减少计算资源消耗，但存在一些重要的限制。

只有当选定的输入数据集通过不修改现有文件的 APPEND 或 UPDATE 事务发生更改时，管道才会以增量计算方式运行。将快照输入标记为增量输入不会产生任何效果。

最适用于以下场景：

  • 输入数据集通过 APPEND 事务或附加性 UPDATE 事务进行更改。
    • 这意味着在添加新数据时，先前的输出保持不变。增量计算能够减少每次构建时处理的数据量。
  • 不需要引用先前的输出。
    • Pipeline Builder 目前不允许使用读取模式：previous。不过，某些输出写入模式支持该读取模式的常见用例。详细了解 Pipeline Builder 中的输出写入模式。
  • 输入数据集较大且经常添加新数据。
    • 增量构建可以节省计算资源和时间，并带来性能上的提升。

增量计算的限制

本部分概述了可能适用于你工作流程的限制条件。在进行增量计算设置前请仔细阅读，以确保正确实现。

  • 连接（Joins）：在涉及增量数据集的连接操作中，增量数据集必须位于连接的左侧，而快照数据集位于右侧。系统也支持两个增量数据集之间的连接。
    • 连接中的快照输入：如果快照输入接收到新事务，那么任何涉及增量数据集的下游连接仍将以增量方式运行。Pipeline Builder 不支持利用连接右侧快照输入的变更来强制重新运行管道。
  • 合并（Unions）：所有参与合并的输入必须使用相同的计算模式（全部为快照或全部为增量）。
  • 转换（Transforms）：可能会改变先前输出的转换操作仅限于当前事务。窗口函数、聚合和透视操作仅应用于当前事务的数据，而不应用于先前的输出。
  • 重放（Replays）：如果管道逻辑发生变更，且你希望将新逻辑应用于已处理过的输入事务，可以选择在部署时进行重放。系统仅支持对整个输入进行重放。

有关更多信息，请参阅Pipeline Builder 中的增量计算示例。
