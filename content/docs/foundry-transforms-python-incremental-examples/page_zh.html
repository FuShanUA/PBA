Append

An append-only incremental computation is one where the added output rows are a function only of the added input rows. This means that to compute its output, the transform does the following:

  • Looks at any newly added input data,
  • Computes any new output rows–which are a function only of these added input rows, and
  • Appends the new output to the existing output.

更改列类型、将日期格式化为字符串以及执行过滤操作，均属于仅追加计算的范畴。在这些场景中，每新增一行输入数据，都会经过转换或删除处理，进而生成对应的输出行。

不难发现，将仅追加转换改造为增量转换，唯一的区别仅在于添加了 incremental() 装饰器。

在增量运行模式下，added 这一默认读取模式意味着转换操作只会读取新增的学生数据；而 modify 这一默认写入模式则表示，转换操作仅会将过滤后的新增学生数据追加至输出端。

在非增量运行模式下，added 这一默认读取模式意味着转换操作会读取完整的输入数据；而 replace 这一默认写入模式则表示，转换操作会使用全量过滤后的学生数据直接替换原有输出。

1from transforms.api import transform, incremental, Input, Output
2
3@incremental()
4@transform(
5    students=Input('/examples/students_hair_eye_color'),
6    processed=Output('/examples/hair_eye_color_processed')
7)
8def incremental_filter(students, processed):
9    new_students_df = students.dataframe()
10    processed.write_dataframe(
11        new_students_df.filter(new_students_df.hair == 'Brown')
12    )

合并与追加

有时，转换操作需要引用其先前的输出，以便增量计算出更新结果。`distinct()` ↗ 方法就是一个典型的例子。 若要在转换中删除重复行（假设当前输出是正确的），转换操作必须先对输入中的任何新行进行去重，然后检查这些行是否已存在于输出中。



已复制！
1@incremental()
2@transform(
3    students=Input('/examples/students_hair_eye_color'),
4    processed=Output('/examples/hair_eye_color_processed')
5)
6def incremental_distinct(students, processed):
7    new_students_df = students.dataframe()
8    processed.write_dataframe(
9        new_students_df.distinct().subtract(
10            processed.dataframe('previous', schema=new_students_df.schema)
11        )
12    )

这里我们在输出数据集上使用了 previous 读取模式。这会返回上一次构建期间输出的 DataFrame ↗。由于可能不存在 previous 输出，我们必须为 dataframe('previous') 调用提供一个模式（schema），以便能够正确构建一个空的 DataFrame。

合并与替换

有些转换操作总是会替换其全部输出。然而，这些转换通常依然能从增量计算中获益。聚合统计就是一个这样的例子。例如，统计某一列中每个不同值出现的次数。



已复制！
1from pyspark.sql import functions as F
2
3@incremental()
4@transform(
5    students=Input('/examples/students_hair_eye_color'),
6    processed=Output('/examples/hair_eye_color_processed')
7)
8def incremental_group_by(students, processed):
9    # 仅计算新增学生的发色数量。
10    new_hair_counts_df = students.dataframe().groupBy('hair').count()
11
12    # 与历史计数值进行合并
13    out_schema = new_hair_counts_df.schema
14    all_counts_df = new_hair_counts_df.union(
15        processed.dataframe('previous', schema=out_schema)
16    )
17
18    # 按发色分组，对两组计数值进行求和。
19    totals_df = all_counts_df.groupBy('hair').agg(F.sum('count').alias('count'))
20
21    # 为完全替换输出结果，我们始终将输出模式设为 'replace'。
22    processed.set_mode('replace')
23    processed.write_dataframe(totals_df.select(out_schema.fieldNames()))

同样地，由于可能不存在 previous 输出，我们必须为 dataframe('previous') 调用提供一个模式，以便正确构建一个空的 DataFrame。

合并与追加及模式变更处理

有时，增量转换需要从添加到无模式输入数据集中的文件创建 Spark DataFrame，然后将该 DataFrame 的内容追加到输出中。对于此类转换，有两种实现模式。

  • 静态指定预期的模式，并强制要求生成的 DataFrame 具有该模式（通过忽略输入数据中的多余字段、为输入数据中缺失的字段填充 null 值等方式）。

  • 动态捕获输入数据中存在的任何字段。

对于动态捕获实现模式，必须确保追加到输出的 DataFrame 具有与现有输出兼容的模式。要使模式与现有输出兼容，必须满足以下条件：

  • 新数据中与现有输出列名相同的列，其数据类型也必须相同。

  • 现有输出中的所有列都必须存在于新数据中。

  • 新数据中不能存在与现有数据列名仅大小写不同的列；例如，如果现有数据中已存在列 value，则新数据中不能存在列 Value。

为确保满足这些条件，需要动态检查现有数据的模式，这意味着在不指定模式的情况下调用 dataframe('previous')。只要转换以增量方式运行，就支持此操作。

以下代码综合运用了这些原则。



Copied!
1from functools import reduce
2
3from pyspark.sql import functions as F
4from transforms.api import transform, incremental, Input, Output
5from transforms.verbs.dataframes import sanitize_schema_for_parquet
6
7
8@incremental()
9@transform(
10    csvs=Input('/examples/dataset_of_csvs'),
11    parsed=Output('/examples/parsed_csvs')
12)
13def incremental_read_csv(ctx, csvs, parsed):
14    input_fs = csvs.filesystem()
15
16    def process_file(f):
17        df = (
18            ctx.spark_session.read
19            # Set inferSchema to False so that every column is a string
20            # This prevents issues due to inconsistent inference results between files,
21            # both within and across incremental builds.
22            .option("inferSchema", False)
23            .option("header", True)
24            .csv(input_fs.hadoop_path + "/" + f)
25        )
26        sanitized = sanitize_schema_for_parquet(df)
27
28        # Make all columns lowercase to prevent issues due to inconsistent casing
29        # between files, both within and across incremental builds.
30        # Note that this logic does not handle the situation
31        # where a single file contains columns differing only in case.
32        return sanitized.select(*(F.col(a).alias(a.lower()) for a in sanitized.columns))
33
34    dfs = [process_file(f.path) for f in input_fs.ls()]
35
36    if len(dfs) == 0:
37        parsed.abort()
38        return
39
40    df = reduce(
41        lambda a, b: a.unionByName(b, allowMissingColumns=True),
42        dfs[1:],
43        dfs[0],
44    )
45
46    if not ctx.is_incremental:
47        parsed.write_dataframe(df)
48    else:
49        existing_columns = parsed.dataframe("previous").columns
50        columns_in_new_data = set(df.columns)
51        resolved_schema_df = df.select(
52            *df.columns,
53            *(
54                F.lit(None).cast("string").alias(col)
55                for col in existing_columns
56                if col not in columns_in_new_data
57            )
58        )
59        parsed.write_dataframe(resolved_schema_df)

利用增量转换关联大型数据集

假设有两张表——客户提交的 Orders 表和已完成配送的 Deliveries 表，我们希望计算出一张 DeliveryDuration 表，记录物品配送所需的时间。尽管 Orders 和 Deliveries 两张表只会追加新行，现有行永不修改，但直接对这两个增量数据集进行简单的关联操作是行不通的。例如，Orders 表中可能包含尚未出现在 Deliveries 表中的 orderIds。

Orders:                               Deliveries:
+---------+---------------+           +---------+--------------+           +---------+------------------+
| orderId | submittedDate |           | orderId | deliveryDate |           | orderId | deliveryDuration |
+---------+---------------+           +---------+--------------+   ---->   +---------+------------------+
| 1001    | 2019-08-21    |  join on  | 1001    | 2019-08-23   |           | 1001    | 2                |
+---------+---------------+  orderId  +---------+--------------+           +---------+------------------+
| 1002    | 2019-08-22    |
+---------+---------------+
| 1003    | 2019-08-23    |
+---------+---------------+

假设 Orders 和 Deliveries 两张表中的 orderId 都是严格递增的，我们可以检查上一次计算 deliveryDuration 时的最后一个 orderId（maxComputedOrderId），然后仅从 Orders 和 Deliveries 表中提取 orderId 大于 maxComputedOrderId 的数据行：

1from transforms.api import transform, Input, Output, incremental
2from pyspark.sql import types as T
3from pyspark.sql import functions as F
4
5@incremental(snapshot_inputs=['orders', 'deliveries'])
6@transform(
7    orders=Input('/example/Orders'),
8    deliveries=Input('/example/Deliveries'),
9    delivery_duration=Output('/example/New_Delivery_Date')
10)
11def compute_delivery_duration(orders, deliveries, delivery_duration):
12    def to_fields(datatype, names, nullable=True):
13        return [T.StructField(n, datatype, nullable) for n in names]
14
15    # Generate a schema to pass for deliveryDuration
16    fields = to_fields(T.IntegerType(), ['orderId', 'deliveryDuration'])
17
18    # Explicitly define the schema as you can't refer to the previous version schema
19    maxComputedOrderId = (
20        delivery_duration
21        .dataframe('previous', schema=T.StructType(fields))
22        .groupby()
23        .max('orderId')
24        .collect()[0][0]
25    )
26
27    # At first iteration, maxComputedOrderId is empty because delivery_duration dataset doesn't exist yet
28    if maxComputedOrderId == None:
29        maxComputedOrderId = 0
30
31    ordersNotProcessed = orders.dataframe().filter(F.col('orderId') > maxComputedOrderId)
32    deliveriesNotProcessed = deliveries.dataframe().filter(F.col('orderId') > maxComputedOrderId)
33
34    newDurations = (
35        ordersNotProcessed
36        .join(deliveriesNotProcessed, 'orderId', how='left')
37        .withColumn('deliveryDuration', F.datediff(F.col('deliveryDate'), F.col('submittedDate')))
38        .drop(*['submittedDate', 'deliveryDate'])
39    )
40
41    delivery_duration.write_dataframe(newDurations)

处理模式或逻辑变更

假设我们希望从现在起为增量数据集增加一列。在输出中增加新列并不会使 is_incremental 标识失效，因此下一次运行将计算新行并写入带有新列的数据，而在之前写入的所有行中，该列将为 null。

不过，我们可能也希望为之前的行填充该列。提升转换的 semantic_version 会使其执行一次非增量运行；如果您使用的是“added”读取模式，输入将包含所有数据，从而允许您重新计算并添加新列。

如果您的转换此前一直在基于快照输入创建历史数据集，那么情况会稍微复杂一些，因为之前的数据位于输入端的一堆快照事务中。在这种情况下，请联系您的 Palantir 代表。

本示例讨论的是增加新列，但上述逻辑同样适用于各类逻辑变更。

在分支上开发增量代码

创建一个新分支并在其上运行构建，将以增量方式执行构建。您创建分支时所基于的原始分支上最后提交的最后一个事务，将被视为新分支首次构建时的前一个事务。

示例总结

我们了解了如何通过以下方式增量处理数据：

  • 获取新增行，对其进行处理并追加到输出中；
  • 获取新增行，根据输出中已有的行对其进行过滤，并追加到输出中；
  • 获取新增行，基于新行和输出中已有的行计算聚合，并用新的聚合统计结果替换输出；
  • 利用增量转换对大型数据集进行连接。

我们还探讨了如何：

  • 处理增量转换的模式或逻辑变更
  • 基于输入的完整内容在分支上开发增量代码，而无需重新构建

增量 Python 错误

要理解增量错误，阅读事务和数据集视图的概念会更容易，有时甚至是必要的。

数据目录事务错误

有用的上下文

当作业以增量方式运行时，其增量输入数据集仅包含未处理的事务范围，而不是完整的数据集视图。

假设一个数据集具有以下事务历史：

SNAPSHOT (1) --> UPDATE (2) --> UPDATE (3) --> UPDATE (4) --> UPDATE (5)
                                   |
                       上次处理的事务

上一次构建数据集时，最新的事务是 (3)。此后，事务 (4) 和 (5) 已提交，因此未处理的事务范围是 (4) — (5)。

数据集视图是事务范围 (1) — (5)。视图“顶部”（即最近）的事务有时被称为分支的 HEAD（同样这是与 git 的类比）。与 git 中一样，分支是指向事务的指针，因此我们说该分支指向事务 (5)。多个分支可以指向多个事务，并且分支之间可能共享事务历史：

SNAPSHOT (1) ─> UPDATE (2) ─> UPDATE (3) ─> UPDATE (4) ─> UPDATE (5)     [develop]
                                 |
                                 └─> UPDATE                              [feature-branch]

错误：Catalog:TransactionsNotInView

为了使作业以增量方式运行，会在作业开始时执行一系列检查。 其中一项检查会验证未处理的事务范围是否严格属于增量（即仅追加的文件更改，请参阅增量计算的要求）。该检查通过对比未处理事务范围和已处理事务范围中的文件来实现。

然而，如果分支的 HEAD 已被移动，增量任务便会处于不一致状态：此时再比较两个范围已毫无意义，因此会抛出 Catalog:TransactionNotInView 错误。

有关此错误的发生机制，请参见下图：

SNAPSHOT (1) ─> UPDATE (2) ─> UPDATE (3)      ─> UPDATE (4) ─> UPDATE (5)
                   |          (上次处理的                  (分支先前的
                   |            事务)                     HEAD，现已孤立)
                   |
                   └─> UPDATE (6) --> UPDATE (7, 分支当前的 HEAD)

在此图中，已处理的事务范围为 (1) — (5)，当前分支的 HEAD 指向 (7)，而当前视图由事务 (1)、(2)、(6) 和 (7) 构成。

这是一种不一致状态，因为并非所有已处理事务都位于分支 HEAD 的上游：实际上 (3) 就不是。换言之，先前的 HEAD (3) 已不再是当前视图的一部分，这就是抛出 Catalog:TransactionNotInView 的原因。

错误：Catalog:InconsistentBranchesOrder

另一个可能抛出的数据目录错误是 Catalog:InconsistentBranchesOrder，当上次处理的事务（prevTransaction）大于分支 HEAD（endTransaction）时便会触发。如果数据集的 HEAD 被移至先前事务之前的某个事务处，就会发生这种情况。

有关此错误的发生机制，请参见下图：

SNAPSHOT (1) --> UPDATE (2) --> UPDATE (3) --> UPDATE (4) --> UPDATE (5)
                                   |                             |
                               当前 HEAD            上次处理的事务

错误修复

分支的 HEAD 发生变更通常出于以下两种原因：

  • 用户通过数据目录端点有意更新了分支的 HEAD。
  • 某些事务未通过转换任务提交。例如，在《代码工作簿》中合并分支时，数据集也会被“合并”。
  • 然而，《代码工作簿》数据集上的事务始终是 SNAPSHOT 类型，因此它们不会导致不一致状态。

要解决此问题，您需要采取以下操作之一：

  • 以快照方式运行转换；例如，通过提升语义版本号。这会启动一个新的数据集视图，从而重置上文提到的增量检查。
  • 手动将分支的 HEAD 更新为指向某个事务，该事务必须处于已处理范围的下游。此操作必须使用 updateBranch2 端点，并将最近处理的事务作为 parentRef。请注意，我们仅建议有经验的用户使用此端点。
