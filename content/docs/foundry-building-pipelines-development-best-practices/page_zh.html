开发最佳实践

本指南旨在为正在开发数据转换逻辑的流水线开发者提供指导。其中通用建议侧重于构建整洁流水线的高阶原则，对项目经理或平台管理员同样具有参考价值。

通用最佳实践

流水线开发即软件开发

通用软件开发领域的诸多最佳实践，同样适用于定义构成数据流水线的各项数据转换逻辑。以下列举了几项体现这一理念的常见实践：

  • 降低认知负担： 尽可能减少不必要的思考过程。若需使用晦涩复杂的函数或 API 调用，务必提供清晰的文档说明。
  • 善待未来的自己： 流水线是数据企业的基石。在规划与实施阶段，应秉持长远眼光与大局观。
  • 不要重复自己 (DRY)： 重复的代码或概念不仅增加维护成本，还易引发隐蔽错误。应从各个层面进行高频重构，包括构建并发布自有库或包以供跨仓库调用，从而确保重复度降至最低。
  • 规避技术债： 这是长远思维的必然推论。面对特定项目需求的截止日期压力，技术债极易滋生，但终将反噬。
  • 规范至上： 先立规矩后坚守；这能有效降低认知负担，并提升代码仓库间的可读性。例如，列名和数据集名称通常约定使用 snake_case 命名法——遵循该规范意味着，其他任何想要引用你那出色数据集的人都会清楚它叫 awesome_dataset，而不是 AwesomeDataset 或 Awesome_Dataset。
  • 少即是多： 相比于包含少数庞大单元的系统，由众多细小单元构成的系统更易于维护与逻辑推演；因此，应倾向于以下做法：
    • 保持 Foundry 项目作用域紧凑
    • 将较小的数据转换串联使用
    • 编写简短的函数——并在转换中善用辅助函数

反模式

  • 覆盖优先于删除： 与其他文件系统不同，Foundry 资源和数据集不仅与文件本身关联，还与其他工件相连。因此，若你在迭代流水线或数据，仅当你从根本上希望彻底清除该类数据时，才应执行删除操作。例如，若向数据集写入了包含错误数据的事务，切勿删除该数据集；而应写入新的 SNAPSHOT 事务来覆盖前序事务。试图删除数据集往往会比在同路径下创建新数据集引发更多麻烦。
  • 避免引入循环依赖： 若你打算将某数据转换的输出数据集作为其他转换的输入，务必确保代码中未引入任何循环依赖。Foundry 的构建编排层会尝试阻止你在当前开发分支上配置任何环路；若检测到环路，你的代码仓库分支将无法通过检查。

Foundry 会在当前开发分支上检查循环依赖，但不会在编写代码时跨所有分支执行此检查，例如可能存在仅存于主分支的本体回写数据集的情况。然而，在尝试将特性分支与包含循环依赖的分支合并时，若 Foundry 在其他分支上检测到循环依赖，依然会导致检查失败。

项目文件夹结构

有关通过多个项目组织数据整体流转的宏观模型描述，请参阅《推荐的项目结构》。

最佳实践

  • 在项目级别管理权限： 若预判需要在项目内进一步细分权限，应考虑是否将该项目进一步拆分为更小的单元。《了解有关应用和管理权限的概念与实践的更多信息》。
  • 保持项目作用域紧凑： 避免“功能蔓延”，不要在项目中添加关联性不强的资源或特定用例的逻辑。
  • 使用有意义的文件夹名称： 切记，你所设计的项目不仅要供团队日常迭代和开发使用，还要供团队外部人员消费。至少应该包含 /Documentation 文件夹和 /Output 文件夹。不同的用例或工作流项目会有更具体的命名需求，但始终要牢记，项目的结构和命名方案就是为访问者提供的路标。

反模式

  • 在个人主文件夹中工作： 个人文件夹不适合用于开发——其访问控制极为严格，且难以共享。建议在项目中创建一个 /scratch 文件夹来存放实验内容或临时性工作，而不是在个人主文件夹中进行构建。

命名

最佳实践

  • 遵循约定： 在命名列、数据集、代码库、文件及其他资源时，应遵循组织内的通用模式或 Foundry 实例的既定规范。
  • 选择具有描述性的名称： 多花点时间构思两三个词的名称，让读者能直观了解所命名资源的用途或内容。建议将名称中最具辨识度的部分放在最前面；这有助于处理长名称的情况，避免在下拉菜单中被截断，同时在浏览代码库或文件列表时也能快速区分不同的引用。避免使用缩写。

反模式

  • 晦涩难懂的名称： 如果仅使用递增数字（如 dataset1、dataset2 等）或单个字母来命名，将会增加代码阅读和重构的难度，也会让新接手的开发者更难上手你的工作。
  • 仅靠路径区分的名称： 某些情况下这是可以接受的，例如当你有 /raw/my_important_dataset 随后又有 /clean/my_important_dataset 时；但在许多情况下，这种命名模式会在只突出显示数据集名称的视图中造成混淆。请记住，系统会跟踪并清晰地展示数据血缘关系，因此你无需将这种“状态”信息硬编码到数据集名称中。

数据类型

最佳实践

显式转换列类型： 如果你正在数据源项目中工作，请务必在 raw → clean 的转换过程中显式转换列类型，即使数据连接的 Schema 推断已选择了正确的值。一旦源系统的列类型发生更改，或者在同步过程中出现无效值导致推断错误，这种做法也能帮你及时发现破坏性变更。

模式

对“仅时间”数据类型使用时间戳： 对于值为 “10:59:00” 这类的字段，Spark 并没有纯时间的数据类型。为了能够利用 Spark 时间戳类型自带的时间函数，可以将值转换为秒，然后加上 -2208988800，再将其转换为时间戳，从而将其年份置于公元 0 年。或者，也可以将其保留为字符串，让用户根据需要自行解析。

反模式

  • 将所有数字字段都转换为数值类型： 设想一个包含飞行器 ID 的列，如 545、972、314。人们很容易将其转换为整数列（毕竟它们看起来就像整数，在源系统中甚至可能就是整数）。然而，这种做法存在重大缺陷：

    • 如果前导零是有意义的，例如 123 和 0123 都是合法的 ID 且需要区分，这种做法就会导致错误。
    • 这些 ID 在用户界面中的显示格式可能不符合预期（例如显示为 545.0、1,234，或者右对齐）。
    • 如果 ID 数值过长，可能会触发 MAX_INT 溢出问题。
    • 数值函数永远不会应用于这些值（例如，你绝不会将两个飞机 ID 相加）。基本原则是，只有当对字段进行算术运算是合理的情况下，才将其转换为数值类型；否则，应将其作为字符串处理。

  • 在不同时区存储时间戳： — Spark 时间戳与时区无关。它们在内部以 UTC（即 Zulu、GMT）存储；时区的显示应由前端负责处理。

    • 如果出于显示目的需要特定时区，请使用 PySpark 的 from_unixtime() 函数 ↗ 将其存储为相应时区的字符串。
    • 如果给定数据源中的时间戳不在 UTC 时区，请使用 PySpark 的 to_utc_timestamp() 函数 ↗ 将其标准化为 UTC。

代码注释

反模式

  • 在提交中注释掉代码： 在进行修改时，很容易倾向于将旧代码注释掉并保留下来以供参考，或者以备日后需要回滚。在迭代过程中可以使用注释，但切勿提交带有被注释掉语句的代码。这样做会累积冗余代码并降低可读性。在之前的提交记录中很容易找到旧代码。

  • 带有作者信息和日期的注释： 这些信息会在提交/Git 仓库中自动跟踪。手动添加注释往往容易疏于更新，从而产生冗余信息。

  • 过度冗长的注释： 注释应阐明决策背后的理由，而非解释逻辑本身。努力编写“自文档化”的代码；如果一组语句难以理解，这明确表明需要进行重构和简化。

仓库实践

最佳实践

  • 保护 master 分支： 如果你在团队中协同开发，或者只是在开发一个长期维护的个人项目，请保护好 master 分支，并实践 GitFlow ↗ 或你偏好的开发工作流。核心概念就是要确保合入 master 的代码都经过了审查和测试。

  • 编写提交信息： 提交信息是仓库中所有活动的日志。请花时间为你的变更写一段有用的描述：

    • 代码审查人员可以查看你的提交记录，从而了解你的工作流以及做了哪些修改。
    • 如果你想回滚某个变更，如果能一眼找到你需要的提交记录，操作会容易得多。
    • 请注意，点击构建按钮会自动生成带有时间戳的提交信息；请避免使用这种方式。请先点击提交，编写提交信息，然后再点击构建。

  • 清理你的分支： 在长期存在的仓库中，分支可能会不断堆积。如果某个分支的开发已被放弃，尤其是当它已被合并到另一个分支时，请通过删除该分支来保持整洁。这有助于清晰辨别哪些分支正处于活跃开发状态。

  • 升级你的仓库： 收到提示时，请按照步骤升级仓库中的语言包。此过程将向包含升级内容的活跃分支发起一个拉取请求。你可以随意在升级分支上运行一次管道构建，以确保任何版本提升都不会影响你的代码。然而，随时保持这些升级的最新状态，通常能确保你不会遇到在其他地方已出现且已在升级版本中修复的边缘情况。

  • 实践代码审查： 在与队友协作开发数据转换逻辑时，请在拉取请求过程中落实一些代码审查实践。我们已经分享了关于《代码审查最佳实践》 ↗ 的观点，其中许多概念同样适用于审查数据转换代码。

模式

在仓库之间共享代码： Foundry 中的仓库基于项目级别运行，这有着多方面的原因，但往往存在一些可以在管道间复用的逻辑，这具有以下几个优势：

  • 遵循 DRY 原则实现通用代码复用。

  • 避免在数据基础底座的不同区域出现分叉或不一致的逻辑。

  • 有些管道理想情况下会使用基础管道，但它们有着更严格的 SLA 或性能要求。此时，常见的解决方案是共享逻辑而非共享转换或数据集，以便关键管道能够依赖预过滤的数据集，减少转换次数，并采用不同的构建调度等。

  • 代码仓库是实现这一目标的极佳方式。例如，在使用 Python 转换时，库可以将其自身发布到 Conda 频道，并允许其他仓库调用。请参阅《共享 Python 库》的相关文档。

  • 共享仓库的另一个优势在于语义化版本控制。共享仓库可以使用版本号（例如 1.0.0、1.0.1、2.0.0）对其提交进行标记，而调用方库则可以自行决定如何获取新版本。例如，某个仓库可能会选择获取最新版本（如上文的 2.0.0），或者仅获取特定版本（如 1.0.1），并推迟获取新版本，直到他们手动决定更新。对于关键管道而言，如果管道数据主人希望有机会选择加入或批准对共享仓库的更改，后一种情况尤为有用。

  • 发布：同理，如果团队希望为管道制定明确的发布计划，一种可选方案（可避免使用暂存实例或长期开发分支）是将逻辑提取为共享仓库中的函数，并利用语义版本号将调用方仓库保持在主要发布版本，如 1.0.0、2.0.0，甚至 .0.0。这样一来，开发者可以继续对逻辑进行迭代，并标记中间版本，而不会导致其在主分支上生效。此外，在调用方仓库的分支上，只要开发者不在发布日期前将这些中间版本合并到主分支，他们就可以随时获取这些中间版本。

单元测试

单元测试是提升并维护代码质量的一种常用方法。在单元测试 ↗中，软件中细小且离散的组件（“单元”）会以独立、分离且自动化的方式进行测试。

  • Python 单元测试：您可以按照《Python 单元测试指南》，将 pytest 单元测试启用为 Python 转换仓库持续集成检查的一部分。

  • Java 单元测试：有关配置 Java 转换单元测试的步骤，请参阅《Java 单元测试文档》。

最佳实践

单元测试应当：

  • 一次仅测试一段逻辑；
  • 保持轻量（参阅特定语言的说明，了解如何减少内存使用）；
  • 不依赖额外输入即可运行；并且
  • 不相互依赖或相互调用。

健康检查

最佳实践

检查您的数据健康度：通常情况下，一旦管道的某部分完成构建，很容易设定好调度便置之不理。然而，即便您的逻辑严谨无误，传入数据也可能发生改变，进而影响构建过程，导致性能下降、数据规模剧增甚至构建彻底失败。对数据集大小和构建时间配置基础检查，即使您并未配置告警，也能为您提供这些关键指标随时间变化的视图，以便您观察，例如，数据集大小的增长速率或该数据集的平均构建时间。阅读更多关于可用的特定健康检查及其配置方法的内容。

模式

扩展健康检查：在大多数情况下，默认的健康检查配置便已足够。但如果您需要更高的灵活性，可以考虑在管道中添加一个或多个派生的 health check 数据集。该数据集的转换可以执行任意逻辑，以判定其输入数据集（即您正在校验的数据集）的有效性，随后输出特定格式的数据，以便简单的健康检查（如 Allowed Value）能够报告该数据集是否有效。

将该数据集的调度设置为输入数据集每次更新时即进行构建，您便拥有了额外的一套全面健康检查机制。
