事务

Foundry 的 Iceberg 目录在标准 Iceberg 基础上增加了 全部或无事务语义。当作业通过 Foundry 的构建系统执行多个写入时，所有写入要么全部成功，要么被完全丢弃，这与 Foundry 目录数据集的现有保证相匹配。此页面解释了 Foundry 的 Iceberg 事务语义的工作原理，以及它与默认 Iceberg 行为的比较，以及它对编写管道意味着什么。

本页描述的事务保证仅适用于通过 Foundry 的构建系统运行作业的情况。当从外部客户端写入 Foundry 的 Iceberg 目录时，您将获得标准的 Iceberg 事务行为。有关在此上下文中事务如何工作的详细信息，请参阅 PyIceberg API 文档 ↗。

Foundry Iceberg 事务语义

当在Foundry中运行构建时，Foundry提供全部或无更新。Foundry会将所有Iceberg表的读取和写入自动包装在一个事务中。用户无需采取任何操作来配置此功能；这是使用Foundry的构建系统向Iceberg表写入的默认行为。

相比之下，标准的Iceberg缺乏多更新事务保证。相反，它提供原子更新：每个更新都会单独应用。这是一个有意设计的选择，使乐观并发成为可能，意味着多个写入者可以同时对同一张表进行操作。虽然这种模型对于单次写入事务表现良好，但对于执行单个事务中多次写入的管道来说，可能会出现正确性问题，无论这些更新是应用于一张表还是多张表。Foundry的全部或无事务模型通过协调作业中所有写入的提交来解决这一问题，因此永远不会显示部分更新。

特别是，Foundry的事务模型提供了以下保证：

  • 全部或无提交: 作业中的所有表更新都会一起提交。如果作业在任何时刻失败，则下游消费者或其他作业都不会看到部分写入。这与目录数据集事务的行为一致。
  • 可重复读取: 如果在作业中多次读取同一张表，两次都会返回相同的数据，即使在两次读取之间该表被外部更新了。在作业的整个过程中，您的管道会看到其输入的稳定、一致的视图。
  • 作业可以看到自己的写入: 在作业中进行的写入操作在同作业中的后续读取中立即可见。这些写入对其他作业和外部读者来说，在事务提交之前是不可见的。
  • 多表快照隔离: Foundry 在事务开始时捕获所有输入表的一致性快照。这确保了在作业过程中无法部分读取输入的外部部分更新，并且数据溯源——哪个版本的输入产生了哪个版本的输出——会被正确记录。

使用标准的Iceberg事务语义：第一个写入操作会被提交。此时order_summary表将反映新的数据批次，而customer_metrics表不会。这两个输出处于不一致的状态。当任务重新运行时，它会重新读取相同的输入行，并再次将它们写入order_summary，从而导致前一次部分运行的数据重复。

使用Foundry的Iceberg事务语义：第一个写入操作不会被提交。此时order_summary表将< strong >不会反映新的数据批次。两个表都保持不变，处于一致的状态。当任务重新运行时，它会正确地将新数据写入两个表，而不会出现重复。

作业队列和乐观并发

Foundry -> Foundry Apollo -> Apollo Gotham -> Gotham AIP -> AIP Ontology -> 本体论 ontology -> 本体论 Pipeline -> 管道 pipeline -> 管道 Transform -> 转换 transform -> 转换 Workshop -> Workshop workshop -> Workshop Code Repositories -> 代码仓库 Code Workspaces -> 代码工作区 Data Integration -> 数据集成 Model Integration -> 模型集成 Object -> 对象 object -> 对象 Dataset -> 数据集 dataset -> 数据集 Tenant -> 租户 tenant -> 租户 Marking -> 标记 marking -> 标记 Permission -> 权限 permission -> 权限 Role -> 角色 role -> 角色 Function -> 函数 function -> 函数 Action -> 操作 action -> 操作 Property -> 属性 property -> 属性 Interface -> 接口 interface -> 接口 内容： Foundry 的构建系统确保在任何时间点，针对给定输出最多只有一个作业在运行。写入同一输出的作业将被排队并按顺序运行，而不是同时运行。这意味着实际上，常规管道作业不会出现写入冲突，但管道作业和 Iceberg 维护作业之间可能会出现冲突。

相比之下，标准的 Iceberg 使用乐观并发，允许多个写入者同时对同一张表进行操作，并在提交时检测和解决冲突。Foundry 的事务模型为了更严格的更新和正确性，放弃了乐观并发。如果事务遇到冲突的并发更新，提交将失败，构建系统会重试该作业。正确性始终得到保证，代价是发生冲突更新时需要重新计算该作业。

冲突的一个常见原因是 维护任务，例如压缩，目前这些任务与常规管道作业同时运行。如果压缩任务在作业运行期间更新了表，该作业的事务将无法提交，构建系统会重试该作业。对于频繁进行压缩的表，这可能导致偶尔的重试。

Iceberg 快照类型和 Foundry 数据集事务

如果您熟悉 Foundry 数据集，您所知道的事务类型与 Iceberg 快照类型之间的对应关系如下：

Iceberg 快照类型Foundry 数据集事务描述
追加快照APPEND添加新的数据文件；现有文件保持不变。
覆盖快照UPDATE, SNAPSHOT添加和删除数据文件，改变逻辑记录集（包括部分和完全覆盖）。
删除快照DELETE删除数据文件或添加删除文件以逻辑上删除行。
替换快照(无对应项)重写数据文件而不改变逻辑内容（例如，压缩）。
Why does a first write appear as an append snapshot?

A common point of confusion: the first write to a new Iceberg table is recorded as an append snapshot, even when the intent is to fully replace the table contents. This is expected. When a table is empty, appending data and replacing data are logically equivalent, as there is nothing to overwrite. Iceberg records the operation as an "append" because files were only added and none were removed. The practical implication is that if you see an append snapshot in a table's history for what you expected to be a full load, check whether it was the table's first write. On subsequent full rewrites to a non-empty table, the operation will correctly appear as an overwrite snapshot.
