响应类型

CEL策略表达式使用两类函数：

  • 结果函数：返回一个策略结果
  • 谓词函数：返回一个布尔值，用于在不同结果之间进行分支

每个策略表达式必须在每条代码路径上返回一个结果函数。

结果函数

approve(reason)

自动批准更改，而无需任何审查者。原因会记录在审计日志中。

approve("Trusted environment import from source-of-truth.")

reject(reason)

拒绝更改。被拒绝的更改在编辑并重新提交之前无法应用。

reject("Replica count cannot exceed 10.")

no_op()

表示该策略对更改没有意见。结果将被丢弃，其他启用的策略将继续正常评估。

在仅打算应用于特定更改类型或实体类型的策略中，使用 no_op() 作为默认分支。请参阅 示例 以了解使用 no_op() 作为回退的模式。

require_role(rid, roleId, reason)

要求持有给定资源上特定角色的用户批准更改。它需要以下参数：

  • rid：资源的标识符。从 referencedResources.key.rid 获取。
  • roleId：角色标识符，例如 "entity:operator" 或 "environment:admin"。
  • reason: 说明为何需要此批准的信息。
require_role(referencedResources.installation.rid, "entity:operator", "Entity config changes require operator approval.")

require_operation(rid, operation, reason)

需要在给定资源上拥有特定操作权限（根据策略，可能是编辑器、审批者、管理员或操作员）的用户来批准更改。它接受以下参数：

  • rid: 资源的标识符。
  • operation: 操作字符串，例如 "entity:approve-config-update"。
  • reason: 说明为何需要此批准的信息。
require_operation(referencedResources.installation.rid, "entity:approve-config-update", "Entity operators must approve config updates.")

require_team(teamName, reason)

需要指定的Apollo团队进行审查。它接受以下参数：

  • teamName: Apollo团队的名称。
  • reason: 说明为何需要该团队批准的信息。
require_team("platform-security", "Security team must approve network configuration changes.")

require_any(requirements, reason)

当提供的任一要求被满足时，就会返回成功。这允许在一组审核者要求中使用OR语义。requirements中的每个元素都必须是对require_role、require_operation或require_team的调用。它接受以下参数：

  • requirements：一个审核者要求列表。
  • reason：描述组合要求的信息。
require_any([
  require_role(referencedResources.installation.rid, "entity:operator", "Entity operators may approve."),
  require_team("platform-oncall", "The on-call team may approve.")
], "Either an entity operator or the on-call team must approve.")

谓词函数

谓词函数返回布尔值，并用于三元表达式中的条件。它们本身不会产生策略结果。

should_auto_approve_service_user(type, author, onBehalfOf, editors)

返回 true，如果更改是由受信任的第三方 Apollo 服务账户进行的，并且该更改类型已自动批准，而且自更改请求打开以来没有其他人编辑过该请求。

此函数对受信任的 Apollo 服务账户列表以及每个账户被批准的更改类型进行编码。它旨在用于默认策略中，以确保自动化的 Apollo 工作流不需要手动审批。

默认策略为每种支持的更改类型调用 should_auto_approve_service_user。删除或修改此调用可能会破坏依赖于自动审批的自动化 Apollo 工作流。

should_auto_approve_service_user(type, author, onBehalfOf, editors)
  ? approve("Authored by a trusted first-party service user.")
  : require_role(referencedResources.installation.rid, "entity:operator", "Operator approval required.")

is_author_service_user(username, author, onBehalfOf, editors)

返回 true，如果给定的用户名与更改请求作者或 onBehalfOf 用户匹配，匹配的用户是服务账户，并且自更改请求打开以来没有其他用户编辑过该请求。它除了标准输入字段外，还需要一个额外参数：

  • username：要检查的服务账户用户名。

当您希望自动批准特定命名的服务账户所做的更改时，请使用此函数。对于检查受信任的第三方 Apollo 服务账户，请优先使用 should_auto_approve_service_user。

is_author_service_user("svc-deploy", author, onBehalfOf, editors)
  ? approve("Trusted deployment service account.")
  : require_role(referencedResources.installation.rid, "entity:operator", "Operator approval required.")

is_trusted_module_request(author, metadata, editors)

如果更改请求是由 Apollo 模块部署服务打开的，包含模块安装元数据，并且自打开以来未被任何其他用户编辑过，则返回 true。

is_trusted_module_request(author, metadata, editors)
  ? approve("Trusted module request.")
  : no_op()

is_trusted_import_request(author, metadata, editors)

返回 true，如果该变更请求是由 Apollo 作为环境导入的一部分打开的，并且自打开以来未被任何其他用户编辑。

环境导入表示从真实来源环境的同步，被视为可信的更改。

is_trusted_import_request(author, metadata, editors)
  ? approve("Trusted environment import from source-of-truth.")
  : no_op()
