管道类型

在 Foundry 中，您可以创建三种主要类型的管道，每种类型在以下几个维度上各有取舍：

  • 延迟。端到端运行管道需要多长时间？
  • 复杂性。编写和长期维护管道的难度有多大？
  • 计算成本。就计算时长而言，管道的实际开销是多少？
  • 应对数据规模变化的弹性。随着时间推移，将有多少新增数据流入您的管道？

这三种管道类型分别是：

  • 批处理管道在每次运行时，会重新计算所有发生变更的数据集。
  • 增量管道仅处理自上次运行以来发生变更的新数据。
  • 流式管道持续运行，并在新数据到达平台时进行处理。

下文将讨论每种管道类型、其权衡取舍，以及如何开始编写该类型的管道。为方便查阅，下表根据上述权衡维度对管道类型进行了总结。

评估维度批处理增量流式
延迟高低极低
复杂性低中高
计算成本中低高
应对数据规模变化的弹性低高高

Foundry 还提供了批处理和增量管道的高速版本，能够提升性能，代价是不支持标准管道中可用的所有表达式和转换。

批处理

在批处理管道中，只要上游数据发生变化，管道中的所有数据集都会重新计算。由于所有内容均重新计算，管道的端到端性能随时间推移保持高度一致，且代码编写和维护的复杂性降至最低。为了让更多用户能够参与批处理管道的构建，平台提供了广泛的语言和工具支持，包括 SQL。

根据上述维度对批处理管道进行分析：

  • 批处理管道的延迟可能很高，因为每当数据落入平台时都必须处理所有数据，即使自上次同步以来数据并未发生变化。然而，如果整体数据规模较小，延迟可能较低。
  • 批处理管道的复杂性非常低。尽管通常仍需了解过滤、连接和聚合等表操作概念，但无需掌握更多额外知识。
  • 批处理管道的计算成本可能较高，因为每次构建管道时都会发生大量重复计算。同样，如果整体数据规模较小，这一因素可以忽略不计。
  • 应对数据规模变化的弹性较低。如果有大量新数据流入管道，例如当输入数据集代表海量事件时，批处理管道将变得成本高昂且难以驾驭，运行时间也会过长。

在大多数情况下，您在 Foundry 中开发管道时应从创建批处理管道开始，随着管道用例得到验证，再将其扩展以支持增量计算。在许多情况下，您可以无限期地使用批处理管道，特别是当数据规模较低时（例如，少于数千万行）。

如果您预计未来需要将管道改为增量模式，我们建议使用 Python 或 Java 进行批处理管道开发，因为这些语言支持增量计算。

高速

高速管道此前被称为轻量级管道，“轻量级”一词是指与处理的数据规模相比，执行这些管道所需的时间和计算资源更少。更名是为了体现，即使对于大规模数据集，高速管道也能同时缩短执行时间并降低计算资源消耗。

Pipeline Builder 现已支持高速管道，能够在特定条件下提供更快的执行速度。它采用由 DataFusion ↗ 驱动的后端，这是一个用 Rust ↗ 编写的开源查询引擎。与传统的基于 Spark 的管道相比，高速管道可以显著加快中小型数据集的计算过程。

您可以从零开始创建高速管道，也可以将现有的批处理或增量管道转换为高速管道。如需了解更多信息，请参阅《Pipeline Builder 中的高速管道》

增量管道

在增量管道中，系统仅会计算自上次构建以来发生变更的数据行或数据文件。这种方式适合处理事件数据，以及海量数据会随时间不断变化的其他数据集。除了能减少整体计算量外，与批处理管道相比，增量管道还能显著降低端到端延迟。目前仅 Python 和 Java API 支持增量计算。

依据上述标准，我们来考量增量管道的各项表现：

  • 增量管道的延迟可以控制在极低水平——通常在分钟级别。根据我们的经验，这种级别的延迟足以满足绝大多数组织的需求。
  • 增量管道的复杂度处于中高水平。编写和维护增量管道不仅需要处理表格型数据集这一高层概念，还需要深入理解数据是如何流入 Foundry 数据集事务的，掌握如何应对输入数据集以非增量方式更新的情况，以及如何长期维持系统的高性能表现。
  • 在处理大规模数据集时，增量管道的计算成本低于批处理管道，因为系统可以将实际发生的计算量压缩到最小。
  • 对数据规模变更的抗性极高。由于仅处理新数据，当大部分存量数据保持不变时，增量管道能够有效避免在海量数据集上重复执行计算。

流式管道

在流式管道中，代码会持续运行，处理任何流入 Foundry 的新数据，从而实现最低延迟，但这也带来了最高的复杂度和计算成本。通常情况下，与其将流式管道视作普通的计算作业，不如将其更接近于微服务管理——想要成功运行流式管道，必须对系统正常运行时间、容错能力以及有状态操作进行深思熟虑的规划。

依据上述标准，我们来考量流式管道的各项表现：

  • 流式管道的延迟可以控制在极低水平。如果您的业务场景要求在不到一分钟内获取可用数据，流式管道将是理想之选。
  • 流式管道的复杂度极高。编写这类管道需要规避大量潜在模式，例如有状态操作，这类操作极易在未来引发不可预见的系统不稳定。此外，与批处理或增量管道相比，流式管道对故障的容忍度更低，因为一旦发生宕机，那些高度依赖数据持续新鲜的业务场景会将任何中断视作服务中断。
  • 流式管道的计算成本极高，因为其核心机制要求计算资源必须随时待命以处理新的输入数据。
  • 对数据规模变更的抗性极高。流式管道专为支持高吞吐量而设计，通常能够比批处理甚至增量管道更好地应对数据规模的大幅波动。

在大多数情况下，除非业务场景对延迟有极苛刻的要求，否则最好避免创建流式管道。增量管道通常能够将端到端延迟优化至分钟级别，在满足大部分需求的同时，避免了流式管道带来的额外复杂度和计算成本。
