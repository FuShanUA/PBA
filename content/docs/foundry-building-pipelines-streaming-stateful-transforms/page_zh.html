流式有状态转换

Foundry 数据管道支持有状态的流式数据转换，能够以极速流处理效率实现复杂的数据转换逻辑。

什么是状态？

在流式数据处理中，如果每一行输出数据都依赖于之前已处理行中所包含的信息，那么这种数据转换就是有状态的。在不同数据行之间持久化留存的信息就称为状态。每行数据都可以对该状态进行读取和修改。

求和聚合转换就是有状态转换的一个典型示例。参考下面的示例表，我们可以使用求和聚合转换来计算每天传感器读数值为 hot 的次数。在表中，越靠上的行表示越晚流入的数据。

有状态转换可以存储任何可序列化类型的状态，包括整行数据，这使其能够实现复杂的行为逻辑。Pipeline Builder 中的状态转换均为预构建组件，能够自动处理状态类型及其演进过程。

用户并非总能直接访问状态内容。状态也可用于支撑后端处理逻辑。例如，在数据管道 Flink 作业中读取流式输入源时，系统会为每个分区存储最新偏移量的状态；这样一旦发生故障或重启，作业即可从最后成功的位点恢复运行。

为什么状态在流式数据处理中如此强大？

Foundry 流式数据处理采用 Flink 架构来提供低延迟数据管道能力；每行数据处理完毕后，会立即向下传递给下一个操作。与批处理转换不同，流式转换只能在无法纵览数据全貌的情况下，逐行确定下一次的输出结果。

对于无状态（也称为无状态）的流式转换而言，这种架构意味着转换逻辑一次只能依赖于单行数据。例如，无状态转换可能只是简单地对整型列的数值加 5。

相比之下，有状态的流式转换不仅能够访问之前数据行的持久化信息，还能继续保持逐行、即时地处理到达数据的能力。

Foundry 有状态流处理提供了精确一次保证选项，这也是管道的默认配置。启用该选项后，导致状态变更的数据行将确保仅变更状态一次，并按键按序执行。从而实现了精准且复杂的流式数据处理行为。

举例来说，如果你要执行求和操作，即使流处理重启或发生故障，求和结果也始终准确。如果执行排序操作，即使你在作业已产生部分非实时排序结果的中途重启作业，每次输入也始终会精确产生一次输出。

键控状态

Pipeline Builder 中的所有有状态流式转换均采用键控状态，并要求用户指定分区键列。针对键列值不同的数据行，有状态转换会分别独立处理。这使得后端能够并行处理数据，从而应对海量数据规模。

可以看到，Day 值为 Monday 的行与值为 Tuesday 的行，其状态是独立计算的。键值为 Tuesday 的数据行出现，不会影响 Monday 所存储的状态；同样地，如果后续到达的数据行在 Day 列上是不同的值（比如 Wednesday），这些键的状态也不会受到任何影响。

务必谨慎选择键，因为如果键导致数据记录分布不均，将会人为增加系统负载并限制吞吐量。详情请参阅流式键最佳实践。

事件时间与水位线

有状态流式转换通常依赖时间信息。由于数据流是持续不断的，并且随时可能接收到新的实时数据行，因此将时间相近的数据行分组在一起通常是有意义的。例如，《外部缓存连接转换》 仅当两个输入流中的数据行在连接列上具有相同的值，并且其时间戳在过期时限内时，才将它们连接起来。Pipeline Builder 流式处理使用 Flink 事件时间以接近确定性的方式实现有状态转换，从而在重放时产生相同或非常相似的输出。

Pipeline Builder 中执行基于时间操作的有状态转换，要求在其上游配置“分配时间戳和水位线”转换；如果流水线图中缺少该转换，将会产生验证错误。“分配时间戳和水位线”会为每行数据分配一个“事件时间”，通常是该行中包含的时间戳列。水位线体现了每个转换操作对“当前时间”的近乎确定性的感知，它是一个单调递增的值，紧随该算子在任何输入行中所见到的最大事件时间值。例如，当“外部缓存连接”判断缓存中的条目是否过期时，它会检查水位线是否大于或等于过期时间；只有当该连接接收到至少具有该事件时间的输入行时，此条件才成立。

对于具有单个流输入的转换算子，其水位线是每个并行实例所见最大事件时间中的最小值。对于具有多个流输入的转换，其水位线是其各输入流水位线中的最小值。

重放会产生相似但有时略有不同的输出。这是因为在重放时，不同的分区键可能会被分配给不同的 Flink 并行实例，并且具有多个输入的算子在上游的处理速率可能有所不同。不支持也不建议使用 Flink 处理时间，因为在重放时它可能会产生截然不同且可能违反直觉的结果。

如果某个并行实例未接收到记录，它将不会产生水位线。这将阻碍转换算子的整体水位线推进，这通常是键分布不均的症状，对状态过期和窗口具有重要影响。要解决此问题，请在 Pipeline Builder 中配置流空闲。

如果某个并行实例在配置的处理时间内未接收到记录，那么它的水位线将不再被纳入转换算子整体水位线的计算中。虽然此配置可以解决水位线停滞的问题，但将此超时时间设置得过短，可能会导致较慢的实例被错误地标记为空闲。随后，较快的实例将推进转换算子的整体水位线，这可能会导致更多的记录被丢弃。

状态过期

存储庞大的状态可能会导致性能瓶颈，对吞吐量和延迟产生负面影响，因此 Pipeline Builder 要求用户限制状态大小。

通常，状态通过用户提供的缓存时间过期机制进行限制。对于需要缓存时间参数的有状态转换，状态通常存储在每个键的状态 cache 中，直到水位线超过该键所见到的最后一个事件时间加上过期时间。

在水位线停滞的情况下，有状态转换将无法及时清除状态。这可能导致意外的输出和无限制的状态增长。

窗口与触发器

“窗口聚合转换”允许用户设置 窗口（即将数据行及其状态分组在一起的策略），以及 触发器（即决定何时应产生聚合输出的策略）。

窗口

目前支持的窗口包括：

  • 滚动事件时间：将时间划分为固定长度、不重叠的连续区间。具有相同键且事件时间落在同一区间内的数据行将被分在一组。例如，您可以将同一天中事件时间处于同一小时内的所有数据行分在一组。
  • 计数：给定用户指定的计数 n，对于每个键，将该键最新的 n 行数据分在一组。
  • 会话：将属于同一会话的行进行分组。如果行共享同一个键，且在事件时间中该键对应的行未出现超过用户指定会话间隔的断档，则这些行属于同一会话。例如，在包含流媒体平台用户行为数据的数据集中，可以将用户一次工作流中的所有行归为一组，直到用户休息为止。

依赖时间的窗口（例如滚动事件时间窗口和会话窗口）最终会在水位线推进到足够远时关闭。

  • 如果未设置允许延迟或设置为零，窗口将保持开启状态，直到水位线越过窗口的结束时间。此时窗口会关闭，可能会产生输出，并删除其状态。
  • 如果指定了允许延迟，窗口将保持开启状态，直到水位线越过窗口结束时间加上允许延迟的时间。这样即使水位线已过窗口结束时间，晚到或乱序的记录仍能纳入该窗口中。
  • 当水位线越过窗口结束时间加上允许延迟的时间后到达的行将始终被丢弃，因为窗口已经关闭且其状态已被删除。

依赖时间的窗口还允许指定自定义触发器。

在水位线停滞的情况下，窗口会长时间保持开启。这可能导致行无法输出、状态无限增长或触发器无法触发。

触发器

目前支持的触发器如下：

  • 水位线后触发：当水位线越过窗口结束时间时触发窗口输出。允许针对水位线未到窗口结束时间之前，以及水位线越过窗口结束时间之后（即窗口因允许延迟仍存活时）这两种情况，分别指定其他自定义触发器。例如，用户可能希望在窗口关闭前不产生任何输出，但希望在允许延迟期间，针对晚到的记录看到每一次输出。
  • 计数触发：给定用户指定的计数 n，在收到 n 的倍数行后，对每个键触发一次窗口输出。
  • 窗口关闭触发：仅在窗口关闭并删除状态时触发输出。每个窗口仅且仅在窗口结束时触发一次。

有状态流处理最佳实践

庞大的状态会对性能产生负面影响，因此在设计有状态流水线时，建议采用尽可能“严格”的状态过期策略。这通常意味着不要将缓存时间过期设置得过长，也不要将计数窗口的计数值设置得过大。

对于需要大状态的流水线，其性能（包括吞吐量、检查点持续时间和延迟）会随 Flink 作业的并行度而扩展。可以在流式流水线设置中调整并行度，更大的并行度能够提升数据处理能力，并加快状态的读写速度。

应选择合适的键进行有状态转换，因为键列的值过多或行分布不均，都会导致瓶颈或扩展问题。
