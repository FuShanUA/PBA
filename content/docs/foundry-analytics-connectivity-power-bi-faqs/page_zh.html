数据传输量是否存在大小限制？

存在。查阅关于《执行引擎》的文档，了解使用《Spark 引擎》时 Foundry 对最终结果大小施加的限制，以及如何通过使用《直接读取引擎》来规避这些限制。

如何在导入模式和 DirectQuery 模式下优化连接器的使用？

该连接器提供两种连接模式选项，可在设置数据连接时指定。

在导入模式下，选定的表和列将导入至 Power BI®。在创建可视化对象或与之交互时，Power BI® 会使用这些导入的数据。若要查看自初次导入或最近一次刷新以来底层数据的变动，必须手动刷新数据或通过计划安排刷新。

  • 使用指南：若仪表板需在中小型 Foundry 数据集基础上实现快速响应和高度交互，推荐采用导入模式。除非处理的数据量极为庞大，且无法在仪表板消费层之前的 Foundry 环节进行预过滤，否则在多数情况下导入模式通常是更优的选择。
  • 使用限制：由于通过连接器传输数据存在大小限制阈值，可能无法导入大型数据集。详情请参阅《数据传输量是否存在大小限制？》。

在DirectQuery 模式下，数据不会被导入或复制到 Power BI® 中。在创建可视化对象或与之交互时，Power BI® 会将 Foundry 作为底层数据源进行查询，因此所查看的始终是最新数据，同时可以将数据转换和筛选操作推回至 Foundry 层执行。

  • 使用指南：若需在极大数据集基础上构建交互性较低的简单仪表板，推荐采用 DirectQuery 模式，因为可以利用 Foundry 的计算引擎动态查询并提取结果。然而，由于需要将计算任务推回至 Foundry 的分布式计算引擎，其性能相较于导入模式更慢。出于此原因，建议将 DirectQuery 仪表板的查询数量保持在相对较少的水平。
  • 使用限制：与导入模式类似，用户向 Power BI® 返回的结果大小上限相同。不过，只要返回的结果在限制范围内，用户依然可以在 Foundry 中处理规模大得多的数据集。

更广泛的使用建议

建议尽可能采用导入模式进行操作。如果数据量较大，可考虑在导入前于 Foundry 中缩减其体积，例如过滤掉多余的行、剔除不必要的列，或尽可能进行预聚合。这类 Foundry 预处理工作可以通过 Foundry 转换或 Contour 完成，并配置为自动构建。另外，也可考虑将表拆分为若干符合大小限制阈值的较小分块进行导入，例如按日历年度分别导入数据集。

在 DirectQuery 模式下工作时，我们建议将仪表板保持在相对较少的查询和简单的交互状态。这种方法一方面平衡了 DirectQuery 处理超大规模数据集的能力，另一方面兼顾了终端用户对性能的期望。

最后，在处理超大数据量时，可考虑使用复合模型 ↗，以实现“两全其美”的效果。通过复合模型，报表可以同时包含导入（Import）和 DirectQuery 数据连接。例如，您可以配置一个导入连接，用于处理预先聚合或筛选后的概览和关键统计数据，从而在用户打开报表时提供快速的加载速度。在同一报表中，您还可以配置 DirectQuery 连接，以便按需进行更深入的分析。

在 Foundry 和 Power BI® 中访问控制如何运作？

使用 Foundry Power BI® 集成开发的报表会自动遵循报表开发者在 Foundry 中配置的访问权限。例如，在 Power BI® Desktop 中构建报表的开发者，只能访问其在 Foundry 中通常有权访问的那些数据集或受限视图。

如果已发布的报表使用数据刷新功能，将使用 Power BI® 网关上配置的数据源凭据，而不会继承报表使用者的任何凭据。

有关在网关上管理数据源的更多信息，请参阅《Microsoft Power BI® 文档中的管理数据源 - 导入/计划刷新》 ↗。

如何将 Foundry 集成与 Power BI® 服务结合使用？

Palantir 的许多客户使用 Power BI® 服务 ↗在其组织内部更广泛地发布和分发报表。

为了在这些报表中利用 Foundry 集成，请使用 Power BI® 网关 ↗，并安装 Foundry 连接器和 ODBC 驱动程序。在网关上使用 Foundry 连接类型创建连接时，将身份验证方法设置为密钥，并提供为 Foundry 中的服务帐户生成的令牌。随后，该网关将建立与 Foundry 的连接，并可供托管在 Power BI® 服务中的报表使用。

当通过安装了 Foundry 连接器的 Power BI® 网关访问 Foundry 时，令牌身份验证是唯一支持的验证方式。目前尚不支持在 Power BI® 服务中使用 OAuth 身份验证，因为第三方开发的 Power BI® 连接器无法使用此功能。如果您的用例需要 OAuth 身份验证，请考虑在 Power BI® 建议 ↗站点上提交反馈，并向您的 Palantir 和 Microsoft 代表反馈意见。

Foundry Power BI® 集成在 Power BI® 网关中的运作方式与您在 Power BI® Desktop 中所习惯的方式大致相同。请与您的 Power BI® 管理员合作，确保网关上已完成安装。

如何使用 Foundry 集成在 Power BI® 服务中创建数据流？

Foundry Power BI® 集成兼容 Power BI® 数据流，并可使用 Power Query Online 编辑器。

在开始之前，请确保您有权访问已配置好 Foundry 现有连接的本地网关。由于 Power BI® 服务存在尚未解决的问题，目前不支持在数据流设置过程中创建新的网关连接。

要创建从 Foundry 提取数据的数据流，请按照以下步骤操作：

  1. 在您的 Power BI® 工作区中，选择新建，然后从下拉菜单中选择数据流。
  2. 选择添加新表。
  3. 在选择数据源页面上，从连接器列表中选择Palantir Foundry。
  4. 在连接到数据源页面上，指定与本地数据网关上已配置的连接相匹配的基础 URL。例如，https://<subdomain>.palantirfoundry.com/。或者，您也可以提供数据集 RID和分支。
  5. 输入基础 URL后，确保连接下拉菜单中显示了您的本地网关名称。如果未出现匹配的网关，请检查基础 URL字段是否与网关连接中配置的 URL 完全匹配（包括结尾的斜杠）。
  6. 选择下一步继续。
  7. 在选择数据页面上，选择所需的表，然后选择转换数据。
  8. 继续使用 Power Query 编辑器创建并保存您的数据流。

为数据流创建新网关连接时出现“意外错误”

在数据流的连接到数据源页面上，尝试创建新的本地网关连接时，您可能会收到以下错误消息：

意外错误 (会话 ID: <UUID>, 区域: <REGION>)

目前存在一个活跃的 Power BI® 服务问题，导致在数据流设置过程中无法创建网关连接。要解决此问题，请在创建数据流之前在网关上设置 Foundry 连接。在 Power BI® 服务中，您可以在设置 > 管理连接和网关 > 连接 > 新建中管理网关连接。

在网关上配置好可用的连接后，当您提供匹配的基础 URL时，数据流的连接到数据源页面将自动选择您的网关。

如何通过 Power BI® 连接器配置额外的 ODBC 驱动程序设置？

您可以配置未在 Power BI® 连接器界面中显示的 ODBC 驱动程序选项，例如代理设置。

安装 ODBC 驱动程序时，它会自动创建一个名为 FoundrySql 的 Windows 系统 DSN。在此 DSN 上配置的任何额外设置都将应用于 Power BI® 连接器（不包括服务器名称或身份验证选项，这些已由 Power BI® 连接器指定）。可以在 Windows ODBC 数据源管理器应用程序中按常规方式配置此 DSN。

什么是令牌，为什么在基于令牌的身份验证中需要使用它？

您可以将身份验证令牌视为您的私人密码，授权 Power BI® 代表您访问 Foundry 数据。您可以按照《生成令牌的文档》中的说明，在“账户 → 设置 → 令牌”页面下管理现有令牌并创建新令牌。

为什么 OAuth 身份验证不起作用？

如果在尝试通过 OAuth 进行身份验证时遇到错误消息，可能是尚未启用此功能。您看到的错误消息可能与下面的截图类似。要使用此功能，必须将 Power BI® 启用为第三方应用程序。《管理第三方应用程序》安全文档详细说明了 Foundry 管理员如何启用此集成。

如果尚未启用 OAuth 身份验证，您可以继续通过备用的《基于令牌的身份验证选项》进行连接。

如何从基于令牌的身份验证切换到基于 OAuth 的身份验证？

如果您之前已经从 Power BI® 身份验证到 Foundry，现在想要主动切换到使用基于令牌的身份验证，首先需要从 Power BI® 清除现有凭据。

您可以按照《无法连接》下的说明清除现有凭据并重新进行身份验证。

无法连接

如果您收到“无法连接”消息，这可能是您的身份验证凭据已过期。请按照以下说明清除现有凭据并配置新的数据源。

  1. 在 Power BI® 中，导航至“文件”>“选项和设置”>“数据源设置”
  2. 在 Palantir Foundry 连接器上，选择“清除权限”。（如果您有多个 Foundry 连接，请全部清除。）
  3. 《配置新数据源》并《重新通过 Foundry 进行身份验证》。请注意，如果您使用的是基于令牌的身份验证，则需要生成新令牌。

从 Power BI® Server 下载后报告为空白

这可能是身份验证凭据存在问题，或者 Power BI® 需要请求您授权运行查询。遇到这种情况时，Power BI® 可能会静默失败，并显示空白的报告页面，且不提供“应用更改”选项。此时，您应点击“编辑查询”。该操作会为您提供继续后续操作的选项。如果仍未生效，请联系您的 Palantir 代表并反馈错误信息。

为什么在表导航器中看不到我的表？

通过“获取数据” → “Palantir Foundry”选择数据集时，您可能会发现某些在 Foundry 中有权限访问的表并未显示在文件夹导航器中。出现这种情况，是因为您拥有对表的访问权限，却没有对应父文件夹结构的访问权限。

只要您在 Foundry 中确实拥有该表的访问权限，就仍然可以将其导入 Power BI® 进行可视化。您可以使用上一页的可选对话框来指定“数据集 RID”。RID 是数据集的唯一标识符，即使移动数据集，该标识符也保持稳定。您可以在 Foundry 中导航至目标数据集的“关于”页面，点击“查看更多”并复制 RID 值，以找到正确的 RID。

有关查找 RID 的更多详细说明，请参阅《指南：在 Foundry 中识别数据集的 RID 或文件路径》

因表大小导致导入缓慢或导入失败

通常情况下，表越大，导入处理所需的时间就越长。根据您的 Power BI® 订阅级别，Power BI® 中的“导入模式”可能被限制为 1GB。适用于 Power BI® 的 Palantir Foundry 连接器也设定了导入大小限制——详情请参阅“我传输的数据量有大小限制吗？”。

如果因表大小导致表导入失败或进程缓慢，请查阅《如何优化跨导入和 DirectQuery 模式的连接器使用？》，了解将连接器与大型数据集配合使用的技巧。

数据集的数据刷新功能失效

如果您在 Power BI® 中通过表导航器选择数据集，您的数据源将依赖于存放在该位置的数据集。如果数据集在 Foundry 中被重命名或移动，可能会导致数据刷新出现问题，因为 Power BI® 将无法再定位该数据集。

如果预计数据集会发生移动，建议使用数据集 RID 来配置数据源。有关通过 RID 配置数据源的更多细节，请参阅“为什么在表导航器中看不到我的表？”。
