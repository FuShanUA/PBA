项目结构

默认引导的仓库

以下是引导的 Python 转换仓库的标准结构：

transforms-python
├── conda_recipe
│   └── meta.yaml
└── src
    ├── myproject
    │   ├── __init__.py
    │   ├── datasets
    │   │   ├── __init__.py
    │   │   └── examples.py
    │   └── pipeline.py
    ├── setup.cfg
    └── setup.py

在仓库中还有其他文件，可以通过进入 代码仓库 中的 文件资源管理器 选项卡中的 设置 齿轮图标并选择 显示隐藏文件和文件夹 来查看。在几乎所有情况下，这些隐藏文件不应被编辑；Palantir 不为对这些隐藏文件进行自定义更改的仓库提供支持。

您可以在下面了解更多有关以下文件的信息：

  • pipeline.py
  • setup.py
  • examples.py
  • meta.yaml

确保在继续阅读之前，您已经阅读了 入门指南。此外，本页假设您正在使用默认项目结构，该结构包含在引导的 Python 转换仓库中。

仓库升级文件更改

当您第一次创建仓库时，它会使用当时最新Python转换模板版本的默认内容进行初始化。在后续的 仓库升级 过程中，仓库中的文件将被升级以与最新的Python转换模板版本的内容保持一致。这些文件的自定义用户更改可能在新的升级模板中被覆盖，以确保一致性。我们不支持对这些文件的自定义更改，因为这可能导致意外的行为。

以下文件将 不会 在仓库升级过程中被覆盖：

  • conda_recipe 和 src 文件夹中的默认文件
  • 内部和外部 build.gradle 文件

以下文件将在仓库升级期间与最新的 Python 模板文件合并。如果存在任何共同的键，将选择 Python 模板中的版本：

  • gradle.properties
  • versions.properties

其余文件将被升级覆盖，以匹配最新 Python 模板版本的文件。

pipeline.py

在此文件中，您定义了项目的 Pipeline（管道），这是与您的数据转换相关的 Transform（转换）对象的注册表。以下是默认的 src/myproject/pipeline.py 文件：

Copied!
1from transforms.api import Pipeline
2from myproject import datasets
3my_pipeline = Pipeline()
4my_pipeline.discover_transforms(datasets)

请注意，默认的 pipeline.py 文件使用 自动注册 将您的 Transform（转换）对象添加到项目的 Pipeline（管道）。自动注册会发现项目中 datasets 包中的所有 Transform（转换）对象。因此，如果您重新组织项目，使得您的转换逻辑不在 datasets 文件夹中，请确保适当更新您的 src/myproject/pipeline.py 文件。

或者，您可以使用 手动注册 显式地将每个 Transform（转换）对象添加到项目的 Pipeline（管道）中。

手动注册。除非您的工作流需要您显式地将每个转换对象添加到您的管道中，否则建议使用自动注册。有关管道对象的更多信息，请参阅 描述管道的部分。

setup.py

在此文件中，您定义了一个名为 transforms.pipeline 的 root 入口点，该入口点与您的项目的 Pipeline 相关联——这允许 Python 转换发现您的项目的 Pipeline。这是默认的 src/setup.py 文件：

Copied!
1import os
2from setuptools import find_packages, setup
3
4setup(
5    name=os.environ['PKG_NAME'],
6    version=os.environ['PKG_VERSION'],
7
8    description='Python data transformation project',
9
10    # Modify the author for this project
11    author='{{REPOSITORY_ORG_NAME}}',
12
13    packages=find_packages(exclude=['contrib', 'docs', 'test']),
14
15    # Instead, specify your dependencies in conda_recipe/meta.yml
16    install_requires=[],
17
18    entry_points={
19        'transforms.pipelines': [
20            'root = myproject.pipeline:my_pipeline'
21        ]
22    }
23)

如果您修改了默认项目结构，可能需要修改 src/setup.py 文件中的内容。更多信息，请参阅 描述 transforms.pipeline 入口点的部分。

examples.py

此文件包含您的数据转换代码。这是默认的 src/myproject/datasets/examples.py 文件的未注释版本：

Copied!
1from transforms.api import Input, Output, transform, LightweightInput, LightweightOutput
2
3
4@transform.using(
5    output_dataset=Output("/path/to/output/dataset"),
6    input_dataset=Input("/path/to/input/dataset"),
7)
8def compute(input_dataset: LightweightInput, output_dataset: LightweightOutput) -> None:
9    output_dataset.write_table(input_dataset.polars(lazy=True))

取消注释示例代码后，您可以分别将 /path/to/input/dataset 和 /path/to/out/dataset 替换为输入和输出数据集的完整路径。如果您的数据转换依赖于多个数据集，您可以提供额外的输入数据集。您可以更新 compute 函数以包含将您的输入数据集转换为输出数据集的代码。此外，请注意，一个 Python 文件支持创建多个输出数据集。

请注意，示例代码使用了 Polars 计算引擎。您也可以修改代码以使用其他计算选项。有关计算选项的更多信息，请参阅 计算引擎，以及 入门 以获取所有引擎的代码示例。

有关创建 Transform（转换）对象的更多信息，这些对象描述了您的输入和输出数据集以及转换逻辑，请参阅 描述 Transforms 的部分。

meta.yaml

一个 conda 构建配方是一个包含构建 conda ↗ 包所需的所有元数据和脚本的目录。构建配方中的一个文件是 meta.yaml —— 该文件包含所有元数据。有关此文件结构的更多信息，请参阅 关于 meta.yaml 文件的 conda 文档 ↗。 这里是默认的 conda_recipe/meta.yaml 文件：

Copied!
1# If you need to modify the runtime requirements for your package,
2# update the 'requirements.run' section in this file
3
4package:
5  name: "{{ PACKAGE_NAME }}"
6  version: "{{ PACKAGE_VERSION }}"
7
8source:
9  path: ../src
10
11requirements:
12  # Tools required to build the package. These packages are run on the build system and include
13  # things such as revision control systems (Git, SVN) make tools (GNU make, Autotool, CMake) and
14  # compilers (real cross, pseudo-cross, or native when not cross-compiling), and any source pre-processors.
15  # https://docs.conda.io/projects/conda-build/en/latest/resources/define-metadata.html#build
16  build:
17    - python 3.9.*
18    - setuptools
19
20  # Packages required to run the package. These are the dependencies that are installed automatically
21  # whenever the package is installed.
22  # https://docs.conda.io/projects/conda-build/en/latest/resources/define-metadata.html#run
23  run:
24    - python 3.9.*
25    - transforms {{ PYTHON_TRANSFORMS_VERSION }}
26    - transforms-expectations
27    - transforms-verbs
28
29build:
30  script: python setup.py install --single-version-externally-managed --record=record.txt

如果您需要的 Python 转换项目有额外的构建依赖项，您可以使用 包标签页 来发现可用的包，并如 分享 Python 库的文档 中所述，自动将这些包添加到您的 meta.yml 文件中。此步骤会自动检测生成您尝试导入的包的渠道，并将其作为 支撑仓库 添加。

也可以手动更新此文件中的“requirements”部分。但是，强烈建议不要手动进行此操作，因为您可能会请求到不可用的包和版本，这随后会导致您的仓库上的 Checks 失败。对于您添加的任何依赖项，请确保 您的依赖项所需的包是可用的。

请注意，您可能不需要修改“requirements”以外的其他部分。

支持的 Python 3 版本

Palantir 支持 Python 的活跃版本，遵循 Python 软件基金会的生命周期计划。有关更多详细信息，请参阅 Python 版本支持页面。

示例用法：

Copied!
1requirements:
2  build:
3    - python 3.9.*
4    - setuptools
5
6  # Any extra packages required to run your package.
7  run:
8    - python 3.9.*
9    - transforms {{ PYTHON_TRANSFORMS_VERSION }}
  • 请确保构建和运行部分中的 Python 依赖项是相同的。Python 依赖项之间的不匹配可能导致意外的结果和失败。
  • 像python >=3.9或python >3.9,<=3.10.11这样的范围不支持用于Python版本。

固定运行时版本

如果您需要特定的库版本进行固定，而您希望手动添加而不是使用推荐的包标签，您可以在requirements块中与库名称一起显式指定。以下是固定版本的示例：

Copied!
1requirements:
2  run:
3    # The below pins an explicit version
4    - mylibrary 1.0.1
5
6    # The below specifies a maximum version (version equal or lower):
7    - scipy <=1.4.0

注意：

  • 操作符后不允许有空格。例如，scipy <= 1.4.0将无法通过CI检查。
  • 操作符>=对于版本在Foundry中尚未得到支持。

使用pip管理的依赖项

如果您需要一个特定的库，该库不能通过Conda获得，但可以通过安装使用pip ↗获得，那么您可以这样做。 ，您可以在附加的 pip 部分中声明这些内容。依赖项将安装在您的 Conda 环境之上。以下是添加 pip 依赖项的示例：

Copied!
1requirements:
2  build:
3    - python 3.9.*
4    - setuptools
5
6  run:
7    - python 3.9.*
8    - transforms {{ PYTHON_TRANSFORMS_VERSION }}
9
10  pip:
11    - pypicloud

注意：

  • 添加到 pip 部分的依赖项将安装在从 run 部分中的包派生的 Conda 环境之上。因此，删除 run 或 build 将导致失败。
  • pip 部分只能在 Python 转换仓库中使用，不能在 Python 库中使用。
