项目结构

默认引导仓库

以下是引导创建的 Python 转换仓库的标准结构：

transforms-python
├── conda_recipe
│   └── meta.yaml
└── src
    ├── myproject
    │   ├── __init__.py
    │   ├── datasets
    │   │   ├── __init__.py
    │   │   └── examples.py
    │   └── pipeline.py
    ├── setup.cfg
    └── setup.py

仓库内还有其他文件，可以通过代码仓库的文件资源管理器选项卡中的设置齿轮图标，选择显示隐藏文件和文件夹来查看。在绝大多数情况下，不应修改这些隐藏文件；对于对这些隐藏文件进行了自定义更改的仓库，Palantir 不提供支持。

您可以在下方了解更多关于以下文件的信息：

  • pipeline.py
  • setup.py
  • examples.py
  • meta.yaml

在继续阅读之前，请确保您已阅读《入门指南》。此外，本页假设您使用的是引导创建的 Python 转换仓库中包含的默认项目结构。

仓库升级文件变更

首次创建仓库时，会使用当时最新的 Python 转换模板版本的默认内容进行引导初始化。在后续的仓库升级过程中，仓库中的文件会进行升级，以与最新 Python 转换模板版本的内容保持一致。为了确保一致性，在应用新升级模板期间，用户对这些文件的自定义更改可能会被覆盖。我们不支持对这些文件进行自定义更改，因为这可能导致意外行为。

以下文件在仓库升级期间不会被覆盖：

  • conda_recipe 和 src 文件夹中的默认文件
  • 内部和外部的 build.gradle 文件

以下文件在仓库升级期间将与最新的 Python 模板文件进行合并。如果存在相同的键，将采用 Python 模板的版本：

  • gradle.properties
  • versions.properties

剩余的文件将在升级过程中被覆盖，以匹配最新版本的 Python 模板文件。

pipeline.py

在该文件中，你可以定义项目的 Pipeline，它是一个用于注册与数据转换相关 Transform 对象的集合。以下是默认的 src/myproject/pipeline.py 文件：

Copied!
1from transforms.api import Pipeline
2from myproject import datasets
3my_pipeline = Pipeline()
4my_pipeline.discover_transforms(datasets)

请注意，默认的 pipeline.py 文件使用自动注册将您的 Transform 对象添加到项目的 Pipeline 中。自动注册会发现项目 datasets 包中的所有 Transform 对象。因此，如果您重构了项目，导致转换逻辑不再包含在 datasets 文件夹中，请确保相应地更新 src/myproject/pipeline.py 文件。

或者，您也可以使用手动注册，将每个 Transform 对象显式添加到项目的 Pipeline 中。除非您的工作流要求您将每个 Transform 对象显式添加到 Pipeline 中，否则建议使用自动注册。有关 Pipeline 对象的更多信息，请参阅描述 Pipelines 的章节。

setup.py

在此文件中，您需要定义一个名为 root 的 transforms.pipeline 入口点，并将其与项目的 Pipeline 关联——这使得 Python 转换能够发现您项目的 Pipeline。以下是默认的 src/setup.py 文件：


```python
import os
from setuptools import find_packages, setup

setup(
    name=os.environ['PKG_NAME'],
    version=os.environ['PKG_VERSION'],

    description='Python 数据转换项目',

    # 修改此项目的作者
    author='{{REPOSITORY_ORG_NAME}}',

    packages=find_packages(exclude=['contrib', 'docs', 'test']),

    # 相反，请在 conda_recipe/meta.yml 中指定您的依赖项
    install_requires=[],

    entry_points={
        'transforms.pipelines': [
            'root = myproject.pipeline:my_pipeline'
        ]
    }
)
```


如果修改默认项目结构，可能需要修改 src/setup.py 文件中的内容。更多信息请参阅描述 transforms.pipeline 入口点的章节。


examples.py


此文件包含数据转换代码。以下是默认 src/myproject/datasets/examples.py 文件的无注释版本：



```python
from transforms.api import Input, Output, transform, LightweightInput, LightweightOutput


@transform.using(
    output_dataset=Output("/path/to/output/dataset"),
    input_dataset=Input("/path/to/input/dataset"),
)
def compute(input_dataset: LightweightInput, output_dataset: LightweightOutput) -> None:
    output_dataset.write_table(input_dataset.polars(lazy=True))
```


取消注释示例代码后，你可以将 /path/to/input/dataset 和 /path/to/out/dataset 分别替换为输入和输出数据集的完整路径。如果数据转换依赖多个数据集，你可以提供额外的输入数据集。你可以更新 compute 函数，在其中编写将输入数据集转换为输出数据集的代码。此外，请注意，单个 Python 文件支持创建多个输出数据集。


需要注意的是，示例代码使用了 Polars 计算引擎。你也可以修改代码，选择其他计算选项。有关计算选项的更多详细信息，请参阅《计算引擎》；有关所有引擎的代码示例，请参阅《入门指南》。


有关创建转换对象的更多信息——转换对象用于描述你的输入和输出数据集以及转换逻辑——请参阅《描述转换的章节》。


meta.yaml


conda 构建配方是一个目录，其中包含构建 conda ↗ 包所需的所有元数据和脚本。构建配方中的一个文件是 meta.yaml——该文件包含了所有的元数据。有关此文件结构的更多信息，请参阅《conda 关于 meta.yaml 文件的文档 ↗》。
以下是默认的 conda_recipe/meta.yaml 文件：



```yaml
# 如果需要修改包的运行时依赖要求，
# 请更新此文件中的 'requirements.run' 部分

package:
  name: "{{ PACKAGE_NAME }}"
  version: "{{ PACKAGE_VERSION }}"

source:
  path: ../src

requirements:
  # 构建包所需的工具。这些包在构建系统上运行，包括
  # 版本控制系统（Git、SVN）、构建工具（GNU make、Autotool、CMake）以及
  # 编译器（真正的交叉编译器、伪交叉编译器，或非交叉编译时的本机编译器），以及任何源代码预处理器。
  # https://docs.conda.io/projects/conda-build/en/latest/resources/define-metadata.html#build
  build:
    - python 3.9.*
    - setuptools

  # 运行包所需的依赖。这些是安装包时
  # 自动安装的依赖项。
  # https://docs.conda.io/projects/conda-build/en/latest/resources/define-metadata.html#run
  run:
    - python 3.9.*
    - transforms {{ PYTHON_TRANSFORMS_VERSION }}
    - transforms-expectations
    - transforms-verbs

build:
  script: python setup.py install --single-version-externally-managed --record=record.txt
```

如果您的 Python 转换项目需要任何额外的构建依赖项，您可以使用 package 选项卡来发现可用的包，并按照《共享 Python 库指南》中的说明，自动将它们添加到您的 `meta.yml` 文件中。此步骤会自动检测生成您尝试导入的包的通道，并将其添加为支持存储库。

您也可以手动更新此文件中的“requirements”部分。但是，强烈建议不要手动执行此操作，因为您可能会请求不可用的包和版本，从而导致您的存储库检查失败。对于您添加的任何依赖项，请确保依赖项所需的包可用。

请注意，您不太可能需要修改“requirements”以外的部分。

### 支持的 Python 3 版本

Palantir 支持 Python 的活跃版本，并遵循 Python 软件基金会的生命周期结束时间表。有关更多详细信息，请参阅《Python 版本支持页面》。

使用示例：


已复制！
1requirements:
2  build:
3    - python 3.9.*
4    - setuptools
5
6  # 运行包所需的任何额外依赖包。
7  run:
8    - python 3.9.*
9    - transforms {{ PYTHON_TRANSFORMS_VERSION }}



    

  • 请确保构建和运行部分中的 Python 依赖项保持一致。Python 依赖项不匹配可能会导致意外结果和失败。
    • 

  • Python 版本不支持诸如 python >=3.9 或 python >3.9,<=3.10.11 的范围指定。
    • 



锁定运行时版本


如果你的转换操作需要锁定特定的库版本，并且你希望手动添加此版本而不是使用推荐的软件包选项卡，则可以在 requirements 块中将其与库名称一起显式指定。以下是锁定版本的示例：




Copied!
1requirements:
2  run:
3    # 下面指定了一个明确的版本
4    - mylibrary 1.0.1
5
6    # 下面指定了最高版本（等于或低于该版本）：
7    - scipy <=1.4.0


注意：


    

  • 运算符后不允许有空格。例如 scipy <= 1.4.0 会导致 CI 检查失败。
    • 

  • Foundry 尚不支持用于版本的 >= 运算符。
    • 



使用由 pip 管理的依赖项




如果你的转换操作需要某个特定的库，且该库无法通过 Conda 获取，但可以通过 pip ↗ 安装，你可以将这些库声明在额外的 pip 区块中。该依赖项将叠加安装在现有的 Conda 环境之上。以下为添加 pip 依赖项的示例：




已复制！
1requirements:
2  build:
3    - python 3.9.*
4    - setuptools
5
6  run:
7    - python 3.9.*
8    - transforms {{ PYTHON_TRANSFORMS_VERSION }}
9
10  pip:
11    - pypicloud



注意：


    

  • 添加到 pip 部分的依赖项会安装在由 run 部分中的包派生出的 Conda 环境之上。因此，移除 run 或 build 会导致失败。
    • 

  • pip 部分只能在 Python 转换仓库中使用，不能用于 Python 库中。
