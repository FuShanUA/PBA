导出

Data Connection 支持将 数据集（dataset） 和 流（stream） 从 Foundry 导出到外部系统。这可以用于各种目的：

  • 在 Foundry 中使用数据管道（pipeline）清理和转换的数据可以同步到数据仓库或数据湖等系统。此模式在 "How Palantir Foundry Fuels Your Data Platform" ↗ 中有更详细的描述。
  • 通过 批量部署（batch deployments） 创建的机器学习模型（model）的推理结果可以导出到其他系统，以实现在整个组织中操作化（operationalize）机器学习项目。
  • 在 本体论（ontology） 中从终端用户捕获的操作数据可以写入其他系统，在 Foundry 外部进行分析。

Data Connection 的导出功能目前还不支持所有源类型。请查看 源类型概览文档以查看可用性。如果导出功能可用，每个源页面将列出文件导出、流式导出、表导出或旧版导出任务支持。例如，Amazon S3源类型支持文件导出，而BigQuery源则需要导出任务。
一些目前尚不支持文件、流式或表导出功能的源可能支持旧版导出任务配置。对于已实现更新导出功能的源类型，不再推荐使用导出任务。

更新的导出行为

在2025年6月之前，如果构建过程中没有新的文件或行需要导出，则导出将被标记为failed。从2025年6月开始，如果没有新的文件或行需要导出，导出将被标记为success。此更改可能会影响依赖于原始失败行为的工作流；用户需要相应地调整受影响的工作流。

导出类型

导出的行为将取决于目标系统可以接受的数据类型。

导出类型摘要
文件导出用于导出到不支持模式的系统。数据集中的原始文件会按计划复制并写入目标系统。
流式导出用于连续导出到支持事件流的系统。Foundry 流中的记录会持续推送到目标系统。
表格导出用于导出到具有模式的系统。根据所选的导出模式，数据集的行将按计划插入到目标系统中。

文件导出

一个支持文件导出的系统示例是S3。

文件导出将选定的Foundry数据集中的文件写入配置的目标位置。默认情况下，只有自上游数据集上次成功导出事务以来修改过的文件才会被写入。这意味着如果某文件在给定事务中未被更新，则在任何下游导出的下一次计划或手动运行时不会重新导出。

如果必须重新导出整个数据集，请在相同的数据源上设置一个新的导出，或确保您的上游转换覆盖所有需要导出的文件。

默认情况下，如果目标位置已存在文件，导出任务将用导出的数据覆盖该文件。此行为可能因源类型而异。为了避免意外覆盖目标系统中的数据，我们建议在Foundry中创建一个专用的子文件夹，用于存放从Foundry导出的数据。

流式导出

流式目标的导出会持续将记录传输到目标位置，只要导出任务在Foundry中运行。如果任务被停止并重新启动，它将从上次停止的位置继续传输记录。

以下源类型支持流式导出：

  • Amazon Kinesis
  • Amazon SQS
  • Google Pub/Sub
  • Kafka
  • PostgreSQL
  • Solace

流式导出重放行为

配置到流式目标的导出时，您必须指定在Foundry中重新播放流时的期望行为。通常，在对处理逻辑进行重大更改后，流会被重新播放。在这种情况下，必须通过以下方式之一使用新逻辑重新处理先前处理的记录：

  • 重新导出重新播放的记录：当流被重新播放时，重新导出所有记录。请注意，之前导出的任何记录都将再次被导出，外部系统必须配置为处理重复项。

  • 不重新导出重新播放的记录：暂停导出，直到重新播放的偏移量与导出作业中的最新偏移量匹配。由于偏移量不能保证在重新播放的流之间匹配，此选项通常会导致一些无法再被导出的记录丢失。

表导出

截至2024年6月，表导出功能通常已可用，但可能不适用于所有源类型。当可用时，建议优先使用表导出而非导出任务。要检查源类型是否支持表导出，请访问数据连接中的新源页面并查找Table exports标签，或访问相关源类型的文档页面。

表导出允许将数据导出到包含具有已知模式的表格数据的系统中，其中外部目标模式与您希望导出的Foundry数据集的模式匹配。表导出将根据下文所述的所选表导出模式，执行系统特定的INSERT语句或等效API调用，以将行写入目标表。

表导出模式

所选的导出模式指定了在表导出过程中数据将如何被导出。以下导出模式适用于所有表导出，并且在数据连接的导出设置页面上也有详细说明。

表导出模式描述
高效地将数据集镜像到外部表（推荐）外部表将始终与您在Foundry数据集中看到的内容匹配。

这是通过始终增量地导出当前数据集视图中任何未导出的事务，并在Foundry数据集上有SNAPSHOT事务时截断外部表来实现的。

这种模式不支持在正在导出的数据集中有UPDATE和DELETE事务。
不截断的完整数据集始终导出Foundry数据集的整个视图的快照，而不会先截断外部表。

注意：此选项几乎总是会导致外部表中出现重复项。当外部系统在每次运行后消费并删除行，或者您希望重新导出已导出的行时，此选项可能有用。
带截断的完整数据集此选项将截断（删除）目标表，然后导出完整的当前数据集视图的快照。

这将导致外部表始终与 Foundry 数据集保持一致，但比“高效地镜像到外部表”的增量选项效率更低。

如果在导出之间对外部表进行了编辑，并且希望始终覆盖它们，则此选项可能有用。
按增量导出仅从当前视图中导出未导出的交易，而不截断目标表。

此模式不支持正在导出的数据集中的UPDATE和DELETE交易。此选项仅镜像APPEND交易的数据集，以避免再次导出相同的数据。如果上游数据集包含SNAPSHOT交易，则此模式可能会在目标表中产生重复记录。
截断后按增量导出此选项会截断（删除）目标表，然后仅导出当前视图中尚未以前导出的交易。

当将目标表视为消息队列时（例如，当其他任务从表中提取行并将其写入其他位置时），此模式很有用。

注意：此模式不支持上游数据集中的UPDATE和DELETE交易。
按增量导出，如果没有APPEND则失败仅从当前视图中导出未导出的事务，在存在 SNAPSHOT、UPDATE 或 DELETE 事务时（首次运行后）会失败。

只要该数据集只包含 APPEND 事务，此选项就会镜像 Foundry 数据集。这可以防止在 Foundry 数据集中存在快照时重新导出所有数据，并保证永远不会将重复数据导出到目标表。

导出模式不适用于旧版导出任务。

表导出的特殊注意事项

  • 从Foundry导出的数据集必须与外部源保持1:1匹配，包括精确的列名（区分大小写）和数据类型。例如，将COLUMN_ABC类型的LONG导出到COLUMN_ABC类型为STRING的目标位置时，将在运行时失败。
  • 在设置表格导出时，如果源要求，需填写DATABASE、SCHEMA和TABLE字段。缺少其中任何一个字段会导致导出在运行时失败。同样，包含不必要的字段也可能导致失败。为了获得最高的可靠性，请使用探索源和自动填充来在源中选择目标。
  • 当使用带有表格导出的代理工作程序时，代理必须在Linux主机上运行。不支持在Windows代理上运行表格导出。
  • 要导出到表格目标的数据集的基础文件必须是Parquet格式。CSV和其他支持模式的文件类型将无法导出。
  • 目标表格必须已经在源系统中存在；Foundry不会自动创建它。
  • 在使用具有截断功能的导出模式时，输入到Foundry源配置中的凭据必须具有外部系统中截断表的权限。
  • Array、Map和Struct类型不支持导出。如果要导出的数据集包含Array、Map或Struct类型的列，则导出将失败。

表导出中的SQL方言

目前仅支持自定义JDBC源类型进行表导出。尽管Foundry为特定系统（如Oracle）提供了专用连接器，但除非通过自定义JDBC连接建立连接，否则不会出现表导出源功能选项。

INSERT 语句示例

以下示例提供了有关表导出所使用的语法的更多信息。如果您的源系统不支持此格式的 INSERT 语句，则在运行时 INSERT 语句将失败。

假设我们的源系统中存在一个 target_table，其模式与 Foundry 中的相应数据集相同。外部源和 Foundry 之间的列名和数据类型必须匹配。以下是 Foundry 中数据集的一个模拟示例：

column1column2column3
value1value2value3
value4value5value6
value7value8value9

单行的 INSERT 语句遵循以下一般格式：

INSERT INTO target_table (column1, column2, column3) VALUES ('value1', 'value2', 'value3');

根据源类型，将对一行或多行数据应用插入语句。例如，Snowflake支持batch inserts，因此允许一次写入多行。

INSERT INTO target_table (column1, column2, column3) VALUES ('value1', 'value2', 'value3'), ('value4', 'value5', 'value6'), ('value7', 'value8', 'value9');

相比之下，像Apache Hive这样的源对batch inserts的支持有限，并将通过三个后续的插入语句进行导出。

INSERT INTO target_table (column1, column2, column3) VALUES ('value1', 'value2', 'value3');
INSERT INTO target_table (column1, column2, column3) VALUES ('value4', 'value5', 'value6');
INSERT INTO target_table (column1, column2, column3) VALUES ('value7', 'value8', 'value9');
TRUNCATE TABLE 语句示例

像Full dataset with truncation这样的导出选项会先在外部源上截断目标表，然后运行一个插入语句。实现这一点的实际代码如下所示：

TRUNCATE TABLE table_name;
INSERT INTO target_table (column1, column2, column3) VALUES ('value1', 'value2', 'value3');

如果需要更具体的方言，我们建议使用Foundry的基于代码的连接选项，例如 外部转换。

多线程表导出

一些源类型支持多线程表导出，这可以提高导出性能。在使用多线程导出时，Foundry会在目标系统中创建一个临时的暂存表以保持原子性。数据会并行写入此暂存表，当导出完成时，再将其复制到最终的目标表中。导出完成后，暂存表会自动删除。

在使用多线程导出时，凭据必须在目标系统的表上具有CREATE、READ、UPDATE和DELETE权限。

准备导出数据

通常，导出不支持作为导出作业一部分执行的数据转换。这意味着您要导出的数据集或流应该已经处于您所需的格式，包括过滤器、重命名或重新分区的文件以及数据的任何其他转换。

管道构建器 和 代码仓库 是Foundry用于构建数据转换管道的工具；这两个应用程序都应该提供您需要的完整工具来准备您的数据以进行导出，包括任务可扩展性、监控、版本控制以及根据需要编写任意逻辑的灵活性。

控制输出文件结构

包含结构化数据的Foundry数据集默认将它们的基础文件存储在spark/目录下。如果您需要在导出时将文件放在特定的目录结构中 - 例如按日期分区组织 - 您可以使用一个 Python 转换 在导出之前按照所需的布局写入数据集。

以下 Python 转换使用 文件系统 API 将文件直接复制到基于日期的路径：

Copied!
1from transforms.api import transform, Input, Output
2import datetime
3import shutil
4
5@transform(
6    output_dataset=Output("/path/to/output/dataset"),
7    input_dataset=Input("/path/to/input/dataset"),
8)
9def compute(input_dataset, output_dataset):
10    dt = datetime.date.today().strftime("%Y%m%d")
11    for file_status in input_dataset.filesystem().ls():
12        relative_path = file_status.path[len("spark/"):] if file_status.path.startswith("spark/") else file_status.path
13        with input_dataset.filesystem().open(file_status.path, "rb") as in_f:
14            with output_dataset.filesystem().open(f"data_dt={dt}/{relative_path}", "wb") as out_f:
15                shutil.copyfileobj(in_f, out_f)

通过此转换，导出的数据集将在路径如 data_dt=20240101/part-00000-xxx.parquet 下包含文件，而不是 spark/part-00000-xxx.parquet。配置文件导出以使用此转换后的数据集作为要导出的数据集。首次构建后，选择 应用模式 在数据集上进行操作，以便在Foundry中以表格数据形式查看。

在将数据导出到Kafka时，可以允许对流记录进行Base64解码。有关Kafka导出的更多信息，请参阅完整的Kafka连接器文档。

设置导出

为源启用导出

要导出数据，您必须在< strong>连接设置< /strong>部分中启用源的导出功能。支持导出的源类型将在屏幕左侧显示一个< strong>导出配置< /strong>选项卡。具有Information Security Officer角色的Foundry用户应导航到此选项卡并打开< strong>启用对此源的导出< /strong>选项。

Information Security Officer 是 Foundry 中的默认角色；用户可以在 控制面板 的 注册权限 下被授予 Information Security Officer 角色。

启用导出后，您必须提供可以从此源导出的一组 标记。如果在导出配置中没有添加标记和组织，则带有这些标记或组织的数据将无法导出到此源。要添加可导出的标记，用户必须同时是 Information Security Officer 并且在他们希望允许导出到此源的标记或组织上拥有取消标记权限。

例如，您可能有一个带有 Sensitive 标记的 Palantir 组织中的数据集。要导出此数据集，您必须将 Sensitive 标记和 Palantir 组织添加到此源的 可导出标记 集合中。

可导出标记还控制哪些资源元数据可以出现在 监控视图的 Slack 通知 中。

在数据连接源配置中输入的凭据必须具有对外部系统中表的写入权限。例如，S3 需要 "s3:PutObject" 权限。其他 文件导出 如果目标路径不存在，可能需要特定的权限来创建目录。表导出 在使用删除模式进行导出时可能需要额外的权限 当使用一种会截断的导出模式时。

请查看 源类型概览 文档中列出的各个源页面，以了解具体的注意事项。

incrementalType: <snapshot|incremental>		受支持这些选项一起使用类似于上述文档中的 表导出模式。并非所有组合都适用于导出任务，且这些组合已被 规范化 ↗ 为提供的六种表导出模式。
exporterThreads: <integer>不受支持如果您需要对导出器线程进行细粒度控制，请使用 外部转换。
quoteIdentifiers: <boolean>不受支持此选项仅在目标系统中创建表时相关，而新导出不支持此功能。在配置导出之前，表必须已存在。
exportTransactionIsolation: READ_UNCOMMITTED不可配置 与transactionIsolation: READ_COMMITTED相同，如果可用且已配置，则事务隔离将来自源配置。
