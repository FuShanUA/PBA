导出

数据连接支持将数据集和流从 Foundry 导出到外部系统。这在多种场景下非常实用：

  • 通过数据管道在 Foundry 中清洗和转换后的数据，可以同步至数据仓库或数据湖等系统。这一模式在《Palantir Foundry 如何为您的数据平台赋能》中进行了详细阐述。
  • 利用批量部署创建的机器学习模型所输出的推理结果，可以导出到其他系统，从而在整个组织范围内推动机器学习项目的落地应用。
  • 在本体中从最终用户处采集的运营数据，可以写入其他系统，以便在 Foundry 之外进行分析。

数据连接导出功能目前尚未支持所有源类型。请查阅源类型概述文档中列出的各个源类型页面，以确认其可用性。如果具备导出能力，每个源页面都会标明支持文件导出、流式导出、表导出还是旧版导出任务。例如，Amazon S3 源类型支持文件导出，而 BigQuery 源则需要使用导出任务。
部分暂不支持文件、流式或表导出功能的源，可能仍支持旧版导出任务配置。对于已实现最新导出功能的源类型，不再推荐使用导出任务。

更新的导出行为

在 2025 年 6 月之前，如果构建期间没有新的文件或行需要导出，导出状态会被标记为failed。自 2025 年 6 月起，若无新的文件或行需要导出，导出状态将标记为success。此项变更可能会影响依赖原有失败行为的工作流；用户需要相应地调整受影响的工作流。

导出类型

导出行为取决于目标系统所能接受的数据类型。

导出类型摘要
文件导出用于向无模式的系统导出。数据集中的原始文件将按计划复制并写入目标系统。
流式导出用于持续向支持事件流的系统导出。来自 Foundry 流的记录将被持续推送到目标系统。
表导出用于向具备模式的系统导出。数据集的行将按计划并根据所选的导出模式插入到目标系统中。

文件导出

支持文件导出的系统示例之一是 S3。

文件导出会将所选 Foundry 数据集中的文件写入配置好的目标位置。默认情况下，系统只会写入上游数据集中自上次成功导出事务以来发生过修改的文件。这意味着，如果某个文件在特定事务中未更新，那么在后续任何计划内或手动触发的下游导出中，该文件都不会被重新导出。

如果必须重新导出整个数据集，请在同一源上新建一个导出任务，或者确保上游转换逻辑会覆盖所有需要导出的文件。

默认情况下，如果目标位置已存在同名文件，导出作业会用导出的数据覆盖该文件。此行为可能因源类型而异。为了避免意外覆盖目标系统中的已有数据，建议您创建一个专用子文件夹，专门用于存放从 Foundry 导出的数据。

流式导出

只要导出作业在 Foundry 中持续运行，向流式目标位置的导出操作就会不断将记录流式传输至目标端。如果作业停止并重新启动，它将从上次中断的位置继续流式传输记录。

以下源类型支持流式导出：

  • Amazon Kinesis
  • Amazon SQS
  • Google Pub/Sub
  • Kafka
  • PostgreSQL
  • Solace

流式导出重放行为

在配置导出到流式传输目标端时，必须指定在 Foundry 中重放数据流时的预期行为。通常，在修改了破坏性处理逻辑后，系统会重放数据流。在这种情况下，之前处理过的记录必须使用新逻辑，通过以下方式之一重新处理：

  • 导出重放的记录：重放数据流时重新导出所有记录。请注意，之前导出的任何记录都将再次被导出，因此必须将外部系统配置为能够处理重复数据。

  • 不导出重放的记录：暂停导出操作，直到重放的偏移量与导出作业中的最新偏移量匹配为止。由于重放的数据流之间无法保证偏移量一致，此选项通常会导致部分记录丢失，永远不会被导出。

表导出

截至 2024 年 6 月，表导出功能已正式发布，但可能并非适用于所有源类型。如果可用，建议使用表导出功能而非导出任务。要检查源类型是否支持表导出，请访问数据连接中的新源页面并查找 表导出 标签，或访问相关源类型的文档页面。

表导出允许将数据导出到包含已知模式表数据的系统中，且外部目标模式与希望导出的 Foundry 数据集模式相匹配。表导出将执行特定于系统的 INSERT 语句或等效的 API 调用，按照下文所述的选定表导出模式将行写入目标表。

表导出模式

所选的导出模式指定了表导出期间数据导出的方式。以下导出模式适用于所有表导出，并且在数据连接的导出设置页面上也有相关说明。

表导出模式描述
高效地将数据集镜像到外部表（推荐）外部表将始终与 Foundry 数据集中所见内容保持一致。

实现方式是始终从当前数据集视图增量导出所有未导出的事务，并在 Foundry 数据集上执行 SNAPSHOT 事务时清空外部表。

此模式不支持正在导出的数据集中的 UPDATE 和 DELETE 事务。
导出完整数据集而不清空始终导出 Foundry 数据集完整视图的快照，而不先清空外部表。

注意：此选项几乎总是会导致外部表中出现重复数据。如果外部系统在每次运行后会读取并删除行，或者希望重新导出已导出的行，此选项将非常有用。
导出完整数据集并清空此选项将截断（删除）目标表，然后导出当前完整数据集视图的快照。

这将使外部表始终镜像 Foundry 数据集，但效率低于“高效地镜像到外部表”这一增量选项。

如果导出期间对外部表进行了编辑，并且希望始终覆盖这些编辑，此选项可能会非常有用。
增量导出仅从当前视图中导出未导出的事务，而不清空目标表。

此模式不支持正在导出的数据集中的 UPDATE 和 DELETE 事务。此选项仅通过 APPEND 事务镜像数据集，以避免再次导出相同数据。如果上游数据集存在 SNAPSHOT 事务，此模式可能会在目标表中产生重复记录。
增量导出并清空此选项会截断（删除）目标表，然后仅从当前视图中导出之前未导出的事务。

当将目标表视为消息队列时（例如，当另一个任务从表中提取行并将其写入其他位置时），此模式非常有用。

注意：此模式不支持上游数据集中的 UPDATE 和 DELETE 事务。
增量导出，非 APPEND 则失败仅从当前视图中导出未导出的事务，如果存在 SNAPSHOT、UPDATE 或 DELETE 事务（在首次运行之后），则导出失败。

只要 Foundry 数据集仅包含 APPEND 事务，此选项就会对其进行镜像。它可以防止在 Foundry 数据集出现快照的情况下重新导出所有数据，并确保导出到目标表中的数据永远不会出现重复。

传统导出任务不支持导出模式。

表导出的特殊注意事项

  • 从 Foundry 导出的数据集必须与外部源保持 1:1 的匹配关系，包括准确的列名（区分大小写）和数据类型。例如，将 LONG 类型的 COLUMN_ABC 导出到目标端中 COLUMN_ABC 为 STRING 类型的位置，会在运行时失败。
  • 设置表导出时，如果源端有要求，必须填写 DATABASE、SCHEMA 和 TABLE 字段。遗漏其中任何字段都会导致运行时导出失败。同理，包含不必要的字段也可能导致失败。为获得最高可靠性，建议使用“探索源”和“自动填充”功能来选择源端的目标位置。
  • 在表导出中使用 agent worker 时，智能体必须运行在 Linux 主机上。不支持使用 Windows 智能体运行表导出。
  • 导出到表格目标端的数据集，其底层文件必须为 Parquet 格式。CSV 和其他支持 schema 的文件类型将无法导出。
  • 目标表必须已存在于源系统中；Foundry 不会自动创建该表。
  • 使用包含截断操作的导出模式时，在 Foundry 源配置中输入的凭据必须具有在外部系统中截断该表的权限。
  • 不支持导出 Array、Map 和 Struct 类型。如果要导出的数据集包含 Array、Map 或 Struct 类型的列，导出将会失败。

表导出中的 SQL 方言

目前，自定义 JDBC 源类型支持表导出功能。尽管 Foundry 为 Oracle 等特定系统提供了专用连接器，但除非通过 自定义 JDBC 连接 建立连接，否则表导出源功能不会作为选项出现。

在向自定义 JDBC 源执行表导出时，如果你提供了自己的驱动 JAR 文件，必须确认 Palantir 用于导出数据的 INSERT 语句语法与你系统所接受的 SQL 方言兼容。

此外，你还需要确保数据库中使用的类型与下方列出的支持类型兼容。

INSERT 语句示例

以下示例提供了关于表导出所用语法的更多信息。如果你的源系统不支持此格式的 INSERT 语句，这些 INSERT 语句在运行时将会失败。

假设源系统中存在一个 target_table，其 schema 与 Foundry 中的对应数据集相同。外部源和 Foundry 之间的列名和数据类型必须匹配。以下是 Foundry 中数据集的模拟示例：

column1column2column3
value1value2value3
value4value5value6
value7value8value9

单行的 INSERT 语句遵循以下通用格式：

INSERT INTO target_table (column1, column2, column3) VALUES ('value1', 'value2', 'value3');

根据源类型的不同，insert 语句将应用于一行或多行数据。例如，Snowflake 平台 支持 batch inserts，因此允许一次写入多行。

INSERT INTO target_table (column1, column2, column3) VALUES ('value1', 'value2', 'value3'), ('value4', 'value5', 'value6'), ('value7', 'value8', 'value9');

相比之下，Apache Hive 等源对 batch inserts 的支持有限，将通过连续三条 insert 语句完成导出。


INSERT INTO target_table (column1, column2, column3) VALUES ('value1', 'value2', 'value3');
INSERT INTO target_table (column1, column2, column3) VALUES ('value4', 'value5', 'value6');
INSERT INTO target_table (column1, column2, column3) VALUES ('value7', 'value8', 'value9');
TRUNCATE TABLE 语句示例

选择 Full dataset with truncation（截断后导出全量数据集）这类导出选项时，系统会先在外部源端截断目标表，然后再执行插入语句。实现该操作的实际代码如下例所示：

TRUNCATE TABLE table_name;
INSERT INTO target_table (column1, column2, column3) VALUES ('value1', 'value2', 'value3');

如需使用更特定的方言，我们推荐使用 Foundry 的《基于代码的连接选项》之一，例如《外部转换》。

多线程表导出

部分源类型支持多线程表导出，从而提升导出性能。采用多线程导出时，Foundry 会在目标系统中创建一个临时暂存表以保障原子性。数据会并行写入该暂存表，待导出完成后再复制到最终目标表。导出结束后，系统会自动删除此暂存表。

使用多线程导出时，凭据必须对目标系统中的表拥有 CREATE、READ、UPDATE 和 DELETE 权限。

准备导出数据

通常情况下，导出操作不支持将数据转换作为导出作业的一部分来执行。这意味着，选定的导出数据集或流应已处于目标格式，包括已完成过滤、文件重命名或重新分区以及其他任何数据转换操作。

《Pipeline Builder》和《Code Repositories》是 Foundry 用于构建数据转换流水线的工具；这两款应用均可提供准备导出数据所需的完整工具链，涵盖作业可扩展性、监控、版本控制，以及按需编写任意逻辑的灵活性。

控制输出文件结构

默认情况下，包含结构化数据的 Foundry 数据集会将其底层文件存储在 spark/ 目录下。如果要求导出的文件落入特定的目录结构中（例如按日期分区组织），可以在导出前使用《Python 转换》以所需布局写入数据集。

以下 Python 转换示例利用《文件系统 API》将文件直接复制到基于日期的路径下：


```python
from transforms.api import transform, Input, Output
import datetime
import shutil

@transform(
    output_dataset=Output("/path/to/output/dataset"),
    input_dataset=Input("/path/to/input/dataset"),
)
def compute(input_dataset, output_dataset):
    dt = datetime.date.today().strftime("%Y%m%d")
    for file_status in input_dataset.filesystem().ls():
        relative_path = file_status.path[len("spark/"):] if file_status.path.startswith("spark/") else file_status.path
        with input_dataset.filesystem().open(file_status.path, "rb") as in_f:
            with output_dataset.filesystem().open(f"data_dt={dt}/{relative_path}", "wb") as out_f:
                shutil.copyfileobj(in_f, out_f)
```


经过此转换，导出的数据集将包含路径如 data_dt=20240101/part-00000-xxx.parquet 的文件，而非 spark/part-00000-xxx.parquet。将文件导出配置为使用该转换后的数据集作为导出对象。首次构建后，在该数据集上选择 应用表结构，即可在 Foundry 中将其作为表格数据查看。


在导出至 Kafka 时，可以允许对流记录进行 Base64 解码。有关 Kafka 导出的更多信息，请查阅完整的 《Kafka 连接器文档》。


设置导出


为数据源启用导出


要导出数据，必须在目标数据源的 连接设置 部分启用导出功能。支持导出功能的源类型将在屏幕左侧显示 导出配置 选项卡。具有 信息安全官 角色的 Foundry 用户应导航至该选项卡，并开启 启用导出至此数据源 选项。


信息安全官 是 Foundry 中的默认角色；可以在 控制面板 的 注册权限 下为用户授予该角色。


启用导出后，必须提供可导出至此数据源的一组 标识。如果未在导出配置中添加标识和组织，带有这些标识或组织的数据将无法导出至此数据源。要添加可导出标识，用户必须同时具备 信息安全官 角色，并对希望允许导出至此数据源的标识或组织拥有去除标识的权限。


举例来说，你可能有一个在 Palantir 组织下带有 敏感 标识的数据集。要导出此数据集，必须将 敏感 标识和 Palantir 组织同时添加到该数据源的 可导出标识 集合中。


可导出标识还控制哪些资源元数据可以出现在 来自监控视图的 Slack 通知 中。




在数据连接源配置中输入的凭据，必须具有对外部系统中表的写入权限。例如，S3 需要 "s3:PutObject" 权限。如果目标路径尚不存在，其他 文件导出 可能需要特定权限来创建目录。表导出 在 使用执行截断操作的导出模式时 可能需要额外权限。
请查阅 源类型概述 文档中列出的各个数据源页面，了解针对特定数据源的注意事项。
