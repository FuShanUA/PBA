构建生产流水线

生产工作流需要可靠的流水线作为支撑。在构建流水线时遵循本文档中阐述的原则，将有效降低维护难度，使您能够在问题导致 SLA 违约之前将其捕获。这里的部分指导也有助于团队共享关于流水线关键重要性的知识。这在流水线生命周期的各个阶段都至关重要，从开发阶段一直到长期维护阶段。

本文档对流水线开发人员和维护人员均有参考价值。对于开发人员而言，如果流水线将直接投入生产环境，在开始构建之前参考本文档会大有裨益。同样，当概念验证流水线向生产流水线转型时，本文档也能作为转化指南。对于流水线维护人员而言，以下要素应作为进入维护模式的前提条件。

流水线定义与预期

在开始构建生产流水线之前，虽然未必总能对各项预期给出确切答案，但尽早将其纳入考量范围极具价值。强烈建议在确立这些预期的过程中，同步做好流水线定义与预期的文档化记录。

这些预期会影响生产流水线设计与部署的多个方面，包括：

  • 流水线设计与架构决策。
  • 调度机制应如何设置（因为这是控制流水线构建频率的主要手段）。
  • 需要执行哪些数据校验与监控。
  • 流水线处于维护模式时，应如何对问题进行优先级排序。

团队应着重探讨以下核心问题：

  • 流水线的确切范围是什么？起点和终点在哪里？应在何处接入其他流水线？
    • 若流水线的某些部分与其他流水线或用例存在重叠，建议将重叠部分作为独立的流水线进行处理。
  • 流水线刷新频率的要求是什么？
    • 数据是否需要在每天特定的时间点前完成刷新？
    • 流水线是否需要周末运行？
    • 数据严重滞后于时间的判定标准是什么？
    • 对于刷新频率和技术支持，预期标准是什么？
    • 需注意——这听起来简单，但该领域往往是流水线维护团队面临最多困难的地方。缺乏明确的定义，流水线维护者将难以合理安排工作优先级或采取正确的修复措施。
  • 端到端传播延迟的预期是多少？换言之，从数据接入 Foundry 起，到流水线输出结果完成更新，整个过程应耗时多久？
  • 所拉取的每个外部数据源的联系人是谁？外部数据源可以是数据连接同步，也可以是接入您流水线的独立上游 Foundry 流水线。
  • 数据正确性的功能性保障有哪些？流水线中是否存在必须确保其正确性的关键列或核心校验逻辑？
    • 注意：明确保障机制失效时的处理结果至关重要。校验失败是应阻止数据流向最终用户，还是仅触发告警而不阻断流水线更新？您可能更倾向于后者，以便允许其他最新数据继续在流水线中流转。这两种场景的实现方式截然不同。通过文档记录是早期追踪这些保障机制的良好起点。
  • 各项预期之间是否相互兼容？随着复杂系统的不断扩展，发生故障的概率也会增加。通常，您需要在告警触发后预留充足的时间来解决底层问题——无论是意料之外的流水线故障，还是上游数据源的数据缺失。
    • 示例 1：结合预期的流水线刷新频率来考量传播延迟至关重要。若预期刷新频率为每 2 小时一次，而流水线构建耗时 1.5 小时，可能难以满足 SLA 要求。
    • 示例 2：如果工作流要求每小时刷新一次，但上游数据源每天仅提供两次数据。这意味着您将无法实现每小时刷新的目标。

生产流水线原则

搭建成功生产流水线的关键原则可以概括为：“抱着你无法亲自维护的心态去构建它”。

以下是一些有助于实现这一目标的具体建议：

  • 对代码进行版本控制，并编写有意义的提交信息：这有助于更轻松地跟踪具体做了哪些修改、提交时间以及修改者。
    • 在生产流水线中，如果条件允许，建议使用 Java 和 Python。这两种语言拥有丰富的可用资源和强大的开发者社区。虽然 SQL 上手极易，但其结构很容易变得错综复杂，因此调试难度往往更大。
  • 始终将可读性放在首位：简单的流水线更易于维护。在转换代码中优先采用标准方案或现有解决方案，能够减轻后续的维护负担和开发复杂度。如果必须采用非传统做法，请务必进行全面详尽的文档记录。
  • 尽可能保持流水线呈线性：就整体架构而言，易于维护的流水线通常是那些直观且线性的流水线，但很难给出达到此目标的具体建议。请留意流水线的整体结构，并在客观条件允许下，尽可能针对可读性和简洁性进行优化。一旦考虑将流水线推向生产环境，花些时间理清流水线的逻辑通常是值得的。
  • 生产流水线往往具有较长的生命周期。因此，编写流水线的人通常并不是长期负责维护或管理流水线的人。请牢记，接下来接手流水线开发或维护的人员需要能够读懂代码，并理解流水线的配置逻辑。
  • 保持文档精简：
    • 逻辑文档：通常建议将与逻辑本身相关的说明保留在代码注释中。
    • 整体流水线文档（包括记录流水线中反复出现的问题）：也应保存在流水线附近、直观易找的位置。流水线所属的项目就是一个很好的存放处。流水线开发者和维护者都应能轻松找到这些文档。
    • 切记：过度文档化同样会降低可读性并削弱其实用价值。请保持精简，仅记录关键信息。如果需要记录篇幅较长且不常查阅的详细流水线信息，建议将其存放在独立的文档中。

开发流程与基础设施配置

开发

流水线进入生产环境后，关键在于确保围绕后续迭代的开发流程已建立完善，并有效传达给流水线开发者。这能避免流水线发生意外中断。

因此，我们建议阅读以下内容：

  • 《开发最佳实践》：至少应保证生产流水线具备良好的可读性、易于维护，并锁定主分支。若要让开发环境更加稳健，我们建议通读此文档中更为详尽的建议。
  • 《分支管理与流水线发布流程》：如果在流水线开发阶段尚未采用分支管理和发布流程，我们建议在将其推向生产环境时引入此流程。该文档将提供一个可供参考的示例流程。

基础设施（调度）

此外，尽早配置调度任务可以让你在开发时无需手动触发流水线各组件的构建。如果调度安排混乱，你可能会发现这会阻碍开发进度，拖慢工作节奏，因为代码变更无法自动在流水线中传递。

将流水线推向生产环境时，同样建议重新审查并调整调度设置，因为开发阶段使用的调度逻辑可能已不再合理，或存在反模式。此步骤应在转入维护模式前完成。

遵循 《调度最佳实践文档》，即可按照最佳实践完成调度配置的建立与审查。

尽早开启监控

流水线一旦进入常态化运行，就应立即开始监控其行为。这能防止技术债务不断累积，并帮助你评估对流水线的预期是否切实可行。受限于项目周期和团队产能，有时可能无法立刻启动监控，但只要条件允许，都强烈建议这么做。

有关如何设置监控的更多信息，请参阅《管道监控最佳实践》文档。
