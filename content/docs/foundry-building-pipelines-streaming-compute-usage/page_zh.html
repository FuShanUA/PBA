Foundry Streaming 的计算用量

Foundry Streaming 是一种高吞吐量、低延迟的计算形式，能够持续监听新传入的消息，应用用户定义的逻辑，并将其推送到管道的下一阶段。

Foundry 流依赖于基于分布式并行 worker 的架构，每个并行 worker 都拥有专用于其特定流处理作业的资源。流的资源需求会随着活动流的最大吞吐量及其处理的消息总数而动态扩展。

计算用量的衡量方式

流处理计算用量分为以下两类：

  • 实时处理计算： 对实时数据运行用户定义转换的过程。此源类型称为“streaming”。

  • 归档计算： 将数据从流处理层移动到 Foundry 存储层的过程。归档计算属于批处理过程，将显示为“transform”。

请注意，在支付 Foundry 使用费用时，默认的 streaming_usage_rate 为 0.5。这是对实时数据运行用户定义转换的费率。如果您与 Palantir 签订了企业合同，请在进行计算用量核算前联系您的 Palantir 代表。

在 Foundry 中，流的实时处理计算用量通过其在整个实际运行时间（wall-clock time）内消耗的计算秒数来衡量。因此，在流处理作业中占用更多计算资源（如 vCPU、内存、并行度）会增加作业成本。作业运行时间越长，消耗的计算量就越大。由于流的设计初衷就是持久运行以持续处理数据，因此它会持续消耗计算资源，直到由用户将其停止。

流采用静态分配；在流运行期间，每实际运行一秒都会消耗固定数量的计算秒数。此外，通常会对流进行调优以满足峰值需求，这意味着流消耗的计算用量不受数据量波动的影响。即使没有数据流过，流也会消耗计算秒数。

流的用量可按单个作业管理器和多个任务管理器消耗的总秒数来计算。请注意，每个并行 worker 将拥有相同的计算资源。

作业管理器的计算秒数按以下方式计算：

  • max(num_vCPU, gib_ram / 7.5) * streaming_usage_rate * stream_duration_seconds

任务管理器的计算秒数按以下方式计算：

  • max(num_vCPU, gib_ram / 7.5) * streaming_usage_rate * stream_duration_seconds * (num_parallel_task_managers + 1)
  • compute_seconds = job_manager compute_seconds + task_manager_compute_seconds

了解有关用于计算计算用量的值的更多信息，包括《内存与核心比率》。

归档作业是与每个流并行运行的批处理作业。归档作业会定期从流的热存储层读取数据，并将其移入 Foundry 存储中，以实现可靠的持久化存储和历史追踪。由于归档作业没有像流本身那样严格的低延迟要求，它们会按计划运行，并且仅在有数据需要归档时才消耗计算资源。

归档作业的用量基于单个 Spark driver 计算，可使用以下公式得出：

  • compute_seconds = max(num_vcpu, gib_ram / 7.5) * num_seconds
    • 归档作业规模较小。归档器始终以 1 个 vCPU 和 4 GiB RAM 的最低配置运行。

调查 Foundry Streaming 用量

要查看流的总用量，请先导航至《资源管理应用程序》。然后，在按资源划分用量部分下找到您的流，并选择详情以查看各个数据集的用量。

流的成本归因于该流生成的检查点数据集。此数据集作为该流处理过程的永久用量记录。在资源管理应用程序中，该数据集上的流处理用量归类于“streaming”类别下。

流每次启动后都会持续运行，直到被用户停止。当用户停止流时，该次运行将显示在数据集的历史记录选项卡下。您可以调查每个单独流的配置文件，以了解历史流运行的性能和计算用量。

每次运行历史归档时，它都会在《构建应用程序》中发布计算指标。使用《构建应用程序》来调查每个已运行归档的资源分配情况。

了解资源使用的驱动因素

流式处理资源使用的主要驱动因素是流本身的计算资源占用。这里的计算资源包括每个任务管理器的 vCPU 数量、每个分区的 RAM 大小（GiB）以及流中的分区数量。这些资源在流的配置文件中设定，并在整个运行周期内保持不变。

  • 流资源的分配应满足传入数据流的峰值吞吐量需求。如果传入的消息量过大，超出了计算资源的有效处理能力，流就会出现处理滞后的情况。
  • 要更改流所使用的资源，必须修改资源配置文件并重启该流。
  • 归档作业的计算使用量随流中传输的数据量而动态调整。归档作业会读取自上次归档以来的所有数据。如果没有数据流入，则归档作业不会消耗任何计算资源。在流处于活跃状态时，归档作业每 10 分钟运行一次。

通过 Foundry Streaming 管理资源使用

针对特定工作流，明确何时选择流式处理、何时选择批处理至关重要。流式处理专为要求秒级延迟和持续计算的工作流而设计。如果数据允许每隔几分钟运行一次，可以考虑采用小型微批处理作业，它能够推送与流式处理相同量级的数据，但能降低计算秒成本，代价是数据延迟会有所增加。

  • 流的规模会显著影响每次运行消耗的总计算秒数。应为流配置充足的资源，以应对该流预期的最大并发负载。
  • 合理确定流的规模，既要确保能够承载峰值负载，又要避免过度分配资源，这一点非常关键。这需要精细调整每个作业的规模（vCPU 和内存）以及流中任务管理器的总数。
  • 流会持续运行直到被手动停止。请密切监控流式计算的资源来源，确保仅在需要时才运行流。

计算资源使用量

以下示例展示了如何计算一个假设运行 10 分钟的流的计算使用量。请注意，大多数生产环境的流都是持续运行的。

流配置文件

  • 作业管理器 vCPU：0.5
  • 作业管理器 gib_ram：1
  • 任务管理器 vCPU：0.5
  • 任务管理器 gib_ram：2g
  • 并行度：2
  • gib_ram：4
  • 流持续时间：10 分钟（600 秒）
  • streaming_usage_rate：0.5

计算过程

  • 作业管理器计算秒数 = max(vCPU, gib_ram / 7.5 gib_ram) * streaming_usage_rate * 600s = max(0.5, 0.133) * 0.5 * 600s = 150 计算秒
    • 换算一下，即每秒 0.25 计算秒，或每小时 900 计算秒
  • 任务管理器计算秒数 = max(vCPU, gib_ram / 7.5 gib_ram) * (parallelism + 1) * streaming_usage_rate * 600s = max(0.5, 0.267) * 3 * 0.5 * 600s = 450 计算秒
    • 换算一下，即每秒 0.75 计算秒，或每小时 2700 计算秒

该流运行 10 分钟的总计算使用量为 150 个作业管理器计算秒和 450 个任务管理器计算秒。如需详细了解影响 《Foundry 中的计算使用量》的因素，请参阅相关文档。
