如何开始新的分析？

在Foundry中，每个用户都有一个名为您的文件的文件夹。这是一个用户可以使用数据进行原型设计并与其他人共享结果的文件夹。在此文件夹中，用户可以创建他们的第一个分析：

  1. 导航到工作区导航栏上的文件部分。
  2. 导航到您的文件选项卡。
  3. 通过选择+新建并从下拉菜单中选择分析来创建一个新的分析。
  4. 在新分析生成后，选择+创建新路径。
  5. 在相应的文件夹中选择您想要分析的数据集。

如何在平台上查找相关数据？

有两种主要的数据发现方法：

  • 搜索： Foundry 在工作区侧边栏的页面左侧有一个平台范围的搜索工具。此工具将搜索平台上的所有资源，是当您知道数据集名称时查找数据的绝佳方法。请注意，搜索工具可用于查找平台上的任何资源，包括电子表格、Contour 分析和代码笔记本。

  • 数据目录: Foundry 数据目录 包含已清理、整理的数据集，可供业务分析师和数据科学家使用。如果您对平台上已存在的数据感兴趣，可以访问 数据目录 作为良好的起点，并且可以直接从主页访问。您可以通过在工作区侧边栏中选择 首页 返回主页。

如何与同事共享分析？

与同事共享资源意味着他们必须能够访问您正在使用的项目。选择分享并发送共享链接或直接将您的同事添加到资源中以自动通知他们。如果您的同事收到权限被拒绝的错误，他们需要请求访问权限。一旦您的同事对项目的访问权限获得批准，他们就可以通过电子邮件查看您与他们共享的分析。

如何创建用于分析的初始数据集？

由于数据通常在Foundry平台中以原始形式存在，因此了解在开始创建分析之前如何对其进行筛选非常重要。数据筛选将使您专注于对分析重要的元素，而不会因不相关数据而分心。

  • 排序和过滤：直方图数据筛选：
    • 在 Foundry 中，排序和筛选是最常用的功能之一。您可以在直方图板上查看某一列中的所有不同选项，然后选择想要处理的特定类别。
  • 属性数据过滤：
    • 如果要仅筛选符合设定条件的数据，请使用保留。
    • 如果要仅排除符合设定条件的数据，请使用删除。
  • 多数据过滤：
    • 如果要筛选同时满足多个条件的数据，请使用与匹配。
    • 如果要筛选满足多个条件但不一定在同一时间/同一行数据中，请使用或匹配。
  • 调整过滤器：
    • 当用户为特定筛选条件（例如，carrier_code = DL）创建新的分析时，其他用户可以通过将筛选条件更改为自己的用例（例如，carrier_code = UA）或完全移除筛选条件以获得全局分析来轻松复制他们的分析。

分析操作默认应用于整个列，以便于对大型数据集进行分析。如果您想在较小的行选择范围内运行分析（类似于在 Excel 中选择特定的单元格区域），请在应用操作之前先将数据筛选到所需的行。

要查看更多的筛选选项，请查阅我们的 筛选数据 文档。

如何应用计算？

要在 Contour 中执行新的计算，您需要选择 添加列 在表达式板中。然而，与单元格级别的操作不同，Foundry 是在列级别上进行操作的。而不是公式乘以 A1 * B1 返回到单元格 C1，Foundry 会乘以 column1 * column2（将 column1 和 column2 中的每个对应行级条目相乘）来返回 column3。

在 Foundry 中，常见的 Microsoft Excel 分析功能有哪些等价项？

以下是 Microsoft Excel 中一些最常用的函数及其在 Contour 中的表达式等价项。您可以像在 如何应用计算？ 中讨论的那样应用这些计算。

  • Excel: IF(逻辑测试, 真值, [假值])

    • Foundry: 
      CASE
    WHEN logical_test THEN value_if_true
    ELSE value_if_false
END
    • 示例：如果我想要创建一个列，当航班开始于 JFK 时返回 yes，否则返回 no，则表达式看起来像： 
      CASE
    WHEN "origin" = 'JFK' THEN 'yes'
    ELSE 'no'
END

  • Excel: CONCAT(单元格1, [单元格2],…)

    • Foundry: CONCAT("col1", ["col2"],...)
    • 示例：如果我想通过连接timestamp和order_ID_number列来为每条订单创建一个唯一键，表达式将如下所示：
      • CONCAT("timestamp","order_ID_number")

  • Excel: VALUE(text)

    • Foundry: CAST("col1" AS DOUBLE)
    • 注意：您可以通过在表达式中替换DOUBLE类型，将您的列数据类型转换为STRING、INTEGER、BOOLEAN、DATE、TIMESTAMP或LONG
    • 示例：如果我想对某些列进行乘法运算，但其中一个必要的列cost被分类为字符串，表达式将如下所示：
      • CAST("cost" AS DOUBLE)

  • Excel: LEFT(text, [num_chars])

    • Foundry: SUBSTRING("col1", num2, num3)
    • 注意：num2是起始索引，num3是子字符串的长度
    • 示例：如果我想从一个airport_display_name列中提取括号中的字母，例如[ALB] Albany International + Albany, NY、[AZA] Phoenix - Mesa Gateway + Phoenix, AZ、[CLT] Charlotte Douglas International + Charlotte, NC，表达式将如下所示：
      • SUBSTRING("airport_display_name", 2, 3)，并将返回一个包含ALB、AZA和CLT的列。

阅读表达式板和 支持表达式语法 的文档以获取更多信息。

  • Excel: VLOOKUP(值, 表格, 列索引, [范围查找]) & 从其他数据集添加列
    • Foundry:JOIN 板。
    • JOIN 板允许您将当前工作数据集与另一个数据集进行连接，并将匹配结果合并到您的数据中。 示例：您正在使用数据集 flights，并希望从 aircraft 数据集中添加列 manufacturer 和 number_of_seats。

在Contour中保存数据集与在代码仓库或代码工作簿中创建数据集有什么区别？

保存Contour分析中的数据集的过程与从代码仓库或Code Workbook创建数据集的过程基本相同——逻辑被转换为一系列Spark转换操作，在集群上执行，并保存到存储在分布式文件系统中的数据集中。将分析保存为新数据集时，没有行数限制。但需要注意的是，数据规模越大，保存所需时间越长，因为底层计算会更耗费资源。请注意，从Contour导出数据时确实存在行数限制，这与将路径保存为数据集不同。

当我尝试导出时，它给了我一个100,000行的限制。这是正确的吗？

是的，这是正确的。Contour 有 100K 行的导出限制。如果您需要导出超过这个数量的数据，可以将 Contour 的结果保存为数据集，然后从该 数据集预览 页面上的 操作 下拉菜单中下载它。

根据合作伙伴的要求，Contour 导出和数据集下载的限制在不同的 Foundry 注册之间可能会有所不同。

我可以自动更新 Contour 分析路径吗？

目前没有方法可以自动更新 Contour 分析路径；这必须手动完成。但是，可以为分析的结果数据集设置一个计划。一旦您保存了数据集，就可以打开数据集预览，并从 操作 下拉菜单中选择 管理计划。然后，结果数据集将根据您配置的计划进行构建。

透视表未显示所有数据

透视表预览不显示表中的所有数据。透视表会对 整个 数据集进行聚合，并将输出减少到前 100 列或 10,000 个值，以防止浏览器性能变慢。要获得这些大型透视表的最终答案，您需要 切换到透视数据。您可以在 透视表文档 中阅读更多相关信息。

要排查问题，请执行以下步骤：

  1. 尝试 切换到透视数据 ，这将强制Contour计算整个数据集。
  2. 在表格板中查看数据。请注意，透视板将保持不完整。
  3. 如果可能，在透视表上方插入一个过滤板以减少您通过的数据（这应该也会提高整体性能），但再次说明，这不会改变透视表板仅对透视表前10,000个单元格进行计算的事实（出于性能原因）。
  4. 要获得完整的导出，请选择切换到透视数据，然后使用导出板或从当前路径的末尾使用导出选项。

在透视表中发现重复列

我的透视表由于重复列而无法计算。这通常是由于列名除了大小写外是相同的。

为了排查问题，请执行以下步骤：

  1. Check if any of the columns in the columns section of your pivot table contain values that are the same apart from casing (for example, Test and test). Foundry dataset column names are case-insensitive, so when the column is pivoted, the columns Test and test are considered duplicates.
  2. Map any such values to a single casing so that when the column is pivoted there are no collisions.

General performance issues

Your Contour is slow, and you would like to figure out what is causing the decreased performance.

To troubleshoot, perform the following steps:

  1. First, check to determine whether the input datasets used in your analysis are using Parquet or Avro files; if not, ensure that you are working on an appropriate, clean version of the dataset.

  2. Check if you are using a raw, ingested version of a dataset that is stored as a CSV file, which is non-performant. This is the most frequent cause of this issue.

  3. Check the partitions of your input dataset(s). If datasets used in your analysis are poorly partitioned, then this will result in slower performance.

  4. To check the size of files in your input datasets go to Dataset → Details → Files → Dataset Files.

  5. The files should be at least 128 MB each. If they are too small, or much too large, you will need to re-partition them.

  6. If you have a very long path, then you should materialize intermediate datasets and create new paths that begin with these newly materialized datasets. This will reduce redundancy in logic execution as each board executes the full query path required to create the board (such as the transformation logic used, if any, in all previous boards).

  7. Consider reducing the number of paths that you have in an analysis. This can slow down the browser performance specifically when using the path overview screen.

For further reading, review our Contour analysis performance optimization documentation.

count_distinct() fails in a window function in the expression board

The count_distinct() function is not available inside window functions due a limitation in Spark. Review official Spark documentation ↗.

You can potentially achieve the same thing (depending on the window logic you are looking to use) in the pivot table board, which offers a unique count aggregation option. You can define your "window" as the rows/column combinations and then generate a unique count for each intersection.
