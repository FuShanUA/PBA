Slate 调试简介

从诸多方面来看，Slate 是开发者可用的最复杂的 Foundry 工具。Slate 将构建单页 Web 应用的工具封装在一个易于上手的包中，但几乎没有提供内置的约束机制，而是允许在实现不同模式时拥有极大的自由度。这种灵活性是一项强大的特性，使得在 Slate 中能够构建出各式各样的应用、仪表板和工作流工具，但这同时也意味着，开发者极容易实现反模式，进而导致性能低下、逻辑混乱或应用行为不确定。

要找到此类最复杂的实现问题的根本原因，必须结合 Slate 开发的最佳实践以及 Web 应用开发的通用原则，对应用逻辑进行逆向工程分析。

我们将在下文逐一剖析这些内容，指出我们观察到的反模式，并推荐改进的实践或引导性的解决方案。首先从 Web 技术的相关问题切入：

1. 前端 Web 技术

  • Web 浏览器的工作原理。
  • 如何从浏览器发起网络请求，并由其他服务作出响应。
  • 浏览器如何为特定网站的渲染和 JavaScript 执行分配资源。
  • HTML 与 CSS 如何协同作用以定义网站的外观。
  • JavaScript 的编译与执行机制。

这些知识并非 Slate 专属，但许多关于 Slate 应用性能、布局和样式的疑问，往往源于对这些基础概念的误解，毕竟 Slate 应用本质上是定制构建的 Web 应用程序。

《W3Schools ↗》 等众多资源都提供了这些主题的基础教程。

2. Postgres 和 SQL

熟悉 SQL 以及 Postgres 数据库构建和执行查询的方式同样大有裨益，因为大多数应用都是通过查询 Postgres 实例来获取数据的，这些数据从 Foundry 数据集同步而来以供应用访问。

许多关于查询开发、报错和性能的疑问并非 Slate 专有，通过常规的搜索往往即可解答，搜索词通常格式为：Postgres + [问题]。

然而，Slate 在解析和呈现 Postgres 生成的报错信息方面并不算友好。建议查阅下文常见原因及补救措施，并参考《Postgres 性能与调优》指南以获取更详细的信息。

3. Slate 专属知识

在 Slate 中构建复杂应用需要一定的适应过程。许多反模式源于将其他应用开发概念“强加”于 Slate，而非在 Slate 提供的框架内进行开发。对于实现特定功能，往往存在隐含的“正确”模式，能够与这些 Slate 基础概念完美契合。

《理解依赖关系》中涵盖了上述大部分复杂情况，其中包括对 Slate 核心组件（依赖图、事件框架、微件、查询、函数等）、常见模式（及常见反模式）以及设计与实现最佳实践的探讨。

我们建议开发人员和支持工程师熟悉这份文档，因为许多常见问题都能在其中已有深入讲解的内容里找到答案。很多时候，解答问题的最佳方式并非去死磕具体实例——例如打开应用试图逆向推导用户的开发决策——而是要理解用户所遇问题属于哪一类常规故障，并引导他们参阅以下任意一项中的相关论述：

  • 《理解依赖关系》
  • 《构建复杂布局》
  • 《启用用户交互》
  • 与该问题相关的各类 Web 资源

针对部分最常见的情况，这份常见问题解答文档提供了简明解释与实用参考资料的捷径。

失败的查询

与查询相关的问题五花八门，查明根本原因有助于指引我们采取相应的解决策略。有时，问题会伴随明确的错误信息显现；但有时，可能根本没有错误提示，或者故障情况呈现非确定性（即时有时无）。下方的排查步骤列举了一些常见的根本原因及相应的解决策略。

要进行排查，请执行以下步骤：

有时，某些查询会有悖于同步网络请求的常规使用模式：

  • 查询试图向浏览器返回过多数据。
    • 作为一条粗略的经验法则，查询返回的行数不应超过 10,000 行。
      • 有时查询会明确报错并提示“行数过多”，但通常情况是查询虽然执行成功，却导致应用程序的其余部分运行极其缓慢。遇到这类情况，应排查返回大量数据行的查询，并设法减少返回量。
    • 如果你需要展示包含大量行的表格，或者需要向用户展示原始数据行，请使用 LIMIT 和 OFFSET 语句实现分页工作流，你可以利用 Table 组件的内置属性，也可以通过自定义工作流来完成。
  • 查询体量过大，导致仅仅发送请求就需要很长时间。
    • 查询往返时间应小于 5 秒。
    • 如果查询的有效负载过大，请考虑将嵌入在查询中的更多逻辑下沉到生成数据集的转换层中处理。

针对此类常见问题，请参阅《优化索引与模式设计》文档，并注意应用程序的结构设计应当确保返回至浏览器的数据已是最终形态，而无需在 JavaScript 函数中进行繁重的数据操作。

有时，问题出在 Postgres 如何解析和响应请求上：

  • 查询速度缓慢，原因是结构未经优化。
  • 查询速度缓慢，原因是数据集未建立索引或索引不正确。
  • 查询存在语法错误。

解决这类问题，通常可以通过搜索 Postgres + [错误信息] 或 Postgres + [期望查询语句] 来寻找答案。如需排查或调查查询性能问题，请参阅《Postgres 查询性能与调优》指南。

有时，问题出在 Slate 根据开发者的配置执行查询的方式和时机上：

  • 页面加载时查询出现非确定性失败。
    • 请参阅最佳实践指南中的《管理查询执行》章节。
  • 查询在 20 秒时返回无意义的错误信息（Slate 超时期限）
    • 通过 Slate 发起的查询请求会在 Slate 端于 20 秒时超时，这与请求服务器端的实际状态无关。要确切诊断此故障模式，请在 View 模式下打开 Google Chrome 网络检查器并刷新页面。这会显示每个查询请求的耗时情况；若你看到某条查询在 20s 处变红，即可断定故障原因是 Slate 查询超时。
    • 要解决此问题，需同时优化查询性能：请参阅《Postgres 查询性能与调优》与《控制查询执行》。
    • 有关超时的更多细节，请查阅下一节 Postgres 查询超时。
  • 在特定情况下查询失败（由于 Handlebars 渲染未得到妥善保护）。
    • 如果查询中包含 Handlebars 语句，请确保它们通过 {{#if}} 语句或函数内的逻辑，对 null 值或其他边缘情况进行了保护。
  • 查询在错误的时间执行或无法执行。
    • 请确保你 理解在 Slate 中各项操作发生的原因，并且已合理地 控制查询执行。对于复杂应用或过度依赖事件的应用，关于 Event Madness 的说明非常具有参考价值。

Postgres 查询超时

查询超时通常由两个主要因素导致：

  • 数据规模 —— Postgres 表中的行数以及查询返回的行数。
  • 查询优化 —— 查询的“开销”有多大。

作为应用开发者，必须理解这些因素及其对查询的影响。这两者均非 Slate 或 Foundry 特有的问题，因此你可以通过搜索 Postgres SQL query performance 获取进展，并在 《Postgres 查询性能与调优》 指南中找到通用最佳实践与技巧。

如需排障，请执行以下步骤：

基于应用开发者常遇到的问题，以下提示可能有所帮助：

  • 数据规模：限制同步至 Postgres 的数据集规模。例如，若仅需一年的数据，切勿同步所有时间的数据集然后在查询中进行过滤。

  • 数据规模：减少表中的列数。例如，若查询中仅选择少数几列，请在同步表之前通过转换将其剔除，以避免额外开销。

  • 数据规模：尽量减少返回的行数。若要展示原始行，请使用 OFFSET 和 PAGING 来获取更多行，而不是针对任何查询返回数百行以上的数据。

  • 查询优化：尽量避免开销昂贵的 SQL 命令。常见的性能“杀手”包括 COALESCE、DISTINCT，对“不完整”的列（例如某些值为 null）执行任何 JOIN 操作，或者在 WHERE 子句中使用前导通配符（%）或复杂匹配表达式进行过滤。

  • 查询优化：将计算逻辑向上游推移。这包括数据清洗（移除空值、格式化日期、映射显示值）和指标计算（月度总计、分桶平均值等），应尽可能将其推入转换层。如果需要允许用户通过过滤器自定义这些内容，你依然可以针对少数维度“预聚合”指标，随后只需执行一个简单得多的查询，对满足条件的少数几行数据进行求和/求平均值等操作即可。

  • 查询优化：使用 EXPLAIN ANALYZE 获取查询计划，了解 Postgres 是如何解析你的查询的。这有助于找出耗时的性能瓶颈，从而进一步针对性地进行优化。《如何阅读 EXPLAIN ANALYZE 查询计划》↗ 是一篇很有用的入门教程。

  • 查询优化：查阅《在 Postgres 中优化查询》。

Postgres 查询中无法引用列

在 Postgres 中，列名默认会被视为小写。由于 Foundry 允许在列名中使用灵活的大小写和空格，当表被索引到 Postgres 时，表名会被 "（双引号）字符包裹以保留其原始大小写。

这意味着，如果列名包含大写字符，你在 SQL 查询中引用该列时，也必须用双引号将其包裹起来。

你可以在 StackOverflow ↗ 上阅读有关此行为的更多信息。

若要排查此问题，请执行以下步骤：

列标识符处理不当的查询通常会报如下错误：

ERROR: column mixedcasecolumn does not exist HINT: Perhaps you meant to reference the column "Mixed Case Column"

遇到这种情况，最佳实践是更新相关数据集，使其使用符合 Postgres 默认设置的列标识符，即通常采用小写字符并用下划线替换空格。或者，直接确保标识符被正确地包裹在双引号中即可。

Phonograph 查询在提交时截断文本

在某种边缘情况配置下，您可能会发现引用一个或多个 Text Area 组件的回写查询，发送给 Phonograph 的请求仅包含部分文本。例如，用户输入了 This is a test message for writeback 这样的长文本后，迅速点击 Submit 按钮触发回写查询，实际可能只包含 This is a te。

根本原因在于 Text Area 组件使用了 0.5 秒的防抖延迟，以防止依赖图在每次输入按键时都重新评估。该延迟意味着，当且仅当以下条件满足时，查询将在未包含完整文本的情况下提交：

  • Text Area 组件是用户最后填写的输入项，且
  • 在文本框输入最后一个字符后不到 0.5 秒内按下了提交按钮。

排查问题，请执行以下步骤：

最简单的解决方案是优化用户体验，避免在输入长文本后 0.5 秒内使用提交选项。例如，在提交查询前增加确认对话框，或者直接调整表单元素顺序，使 Text Area 组件之后还有其他字段需要填写。

其他解决方案是将文本区域组件替换为 文本输入 组件，该组件不包含防抖延迟。还可以使用 Toast 组件作为提交按钮按下与查询执行之间的延迟缓冲，确保有足够的时间让完整文本包含在内。

布局

应用渲染问题

在某些配置下，若尝试将组件配置为“动态”宽度和高度，页面加载时该组件实际却会渲染在一个小方框内。触发 [w_myWidget.resize] 操作后，组件才会渲染为预期尺寸。

此现象源于 CSS 冲突：一方面通过百分比（通常为 100%）设定各组件元素的宽高，另一方面在一个或多个父容器上使用了 Flex 属性。由于这种冲突，Slate 在渲染时无法确定包裹组件的元素的正确尺寸，从而导致组件渲染过小。

排查此问题，请执行以下步骤：

实现组件响应式尺寸的合理模式是：仅在父容器和子组件上使用 Flex 配置项。切勿通过 CSS 手动设置其他 width 或 height 属性。要解决此问题，请从根子组件开始，逐一向上遍历所有父组件，移除所有的定位或尺寸 CSS 规则。

你可以使用 margin 和 padding 属性来缩进或对齐子组件。若熟悉 Flexbox CSS 规则，可借助 Chrome 开发者工具定位需调整的元素和类，并提供覆盖样式或补充 Flexbox 规则，以实现精确的预期效果。

Slate 应用无法打印 / 如何导出我的应用

Slate 专为快速应用开发而优化。其底层框架对布局和样式的可配置性几乎不加限制。因此，Slate 无法以适配打印媒介或导出的通用方式来渲染所有应用。应用开发者需针对每个应用投入时间，以适配打印支持。

排查此问题，请执行以下步骤：

首要且最佳的做法是思考为什么会有打印应用的需求。用户是否只是想保存已创建的视图，以便日后参考或在演示时使用？

若真如此，可以考虑使用内置的可共享视图功能，该功能会生成一个唯一的 URL，用于保留所有组件选择和变量的状态。用户可以在查看模式下的操作下拉菜单中访问此功能。

如果确实需要打印或导出，则需要定义额外的媒体 CSS 样式，以帮助应用正确渲染。

首先，至少需要在全局样式中添加以下内容：

@media print {
  @page {
    size: A4;
  }

  html, body , .ws-root-container.ws-sandbox {
    height: 600mm !important;
  }

  sl-app-view-actions {
    display: none !important;
  }
}

打印响应式应用或任何具有复杂样式或布局的应用会非常困难。在这种情况下，建议创建一个单独的“打印视图”应用，通过 URL 参数接收必要的输入，并生成针对打印优化的简化视图。这会增加开发成本和维护负担，但能带来令人满意的效果。

性能

我的应用加载缓慢或编辑模式下卡顿

与所有 Web 应用一样，Slate 使用终端用户计算机上的浏览器资源来执行 JavaScript 代码，并根据 CSS 指定的样式渲染 HTML 中定义的文档对象模型 (DOM)。这些资源中有一部分是固定开销，属于在 Slate 框架内工作的固有成本。然而，在性能不佳的情况下，根本原因往往是应用过载，即在应用开发过程中定义的配置要求浏览器提供过多资源来渲染应用、执行 JavaScript，或者仅仅是在内存中存储大量大型查询的结果。

有时，单个函数或查询可能是问题根源。此时，可以利用应用程序依赖项标签页中的性能指标，查看各个查询和函数的 `Time in Node`。该指标表示页面加载时解析特定节点所耗费的时长，有助于排查潜在的应用程序瓶颈。

通常而言，最佳实践是优先采用轻量级、边界清晰且功能单一的应用程序，并通过深度链接将多个小型应用程序组成的“集群”拼装成大型工作流或应用。

配置

文档过大导致无法保存

Slate 将应用程序配置保存为 JSON 文档，其大小限制通常为 2,000,000 字节的 UTF-8 编码字符。若尝试保存配置超出此大小限制的 Slate 应用程序将会失败，必须调整应用配置以缩减其体积。已打开的 Slate 应用程序的 JSON 文档可以通过导出方式获取。

Slate 应用程序的每个部分都会在一定程度上占用配置空间。编辑器开发者应充分关注所构建 Slate 应用程序的规模，以及嵌入到应用配置中的资源。以下是导致配置空间超限的常见原因及相应的解决方案：

Base64 编码的图片

将图片上传至 Foundry，并使用图片组件进行引用。

大型函数

考虑重构函数，将公共逻辑抽取到函数库中。

此外，大型函数更适合改写为Foundry 函数，并通过平台编辑器进行调用。

大型函数库

虽然函数库会在 Slate 应用的所有页面间共享以节省空间，但 JavaScript 库的体积可能十分庞大。建议对自定义库进行重构以复用共享逻辑，或者直接使用外部库的压缩版本。
