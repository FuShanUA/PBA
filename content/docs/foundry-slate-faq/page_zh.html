调试Slate简介

在许多方面，Slate 是开发人员可用的最复杂的 Foundry 工具。Slate 将构建单页 Web 应用程序的工具打包成一个易于使用的包，但提供了很少的内置 Rails（Rails 通常指框架中的一些约定和结构），而是允许在实现不同模式时有广泛的自由度。这种灵活性是一个强大的功能，使各种各样的应用程序、仪表板和工作流工具可以在 Slate 中构建，但它也意味着很容易实现反模式，导致性能缓慢或混乱，应用行为非确定性。

要找到这些实现问题中最复杂类型的根源，需要在 Slate 开发最佳实践以及 Web 应用开发一般原则的背景下，逆向工程应用程序逻辑。

我们将下面进行分解，指出我们看到反模式出现的地方，并提出改进的实践或指导解决方案。我们将从相关的网络技术问题开始：

1. 前端网络技术

  • 浏览器是如何工作的。
  • 浏览器如何发起网络请求并从其他服务中获得响应。
  • 浏览器如何为特定网站的渲染和JavaScript执行分配资源。
  • HTML和CSS如何协同工作以定义网站的外观。
  • JavaScript是如何被编译和执行的。

虽然这些知识并不特定于Slate，但对这些基础概念的理解不足是许多关于Slate应用性能、布局和样式问题的根本原因，因为Slate应用本质上是定制的Web应用。

W3Schools ↗ 等众多来源提供了这些主题的基本教程。

2. PostgreSQL 和 SQL

了解SQL以及PostgreSQL数据库如何构建和执行查询也是有帮助的，因为大多数应用程序通过针对PostgreSQL实例的查询来获取数据，这些数据是从Foundry的数据集同步过来供应用程序访问的。

关于查询开发、错误和性能的许多问题并不特定于Slate，可以通过常见的搜索查询来解答，通常形式为：Postgres + [question]。

然而，Slate在解析和提供Postgres生成的错误信息方面并不是特别友好，我们建议您查看下面的部分，了解这些常见问题的原因和解决方法，并参考Postgres性能和调优指南以获取更详细的信息。

3. Slate特定知识

在Slate中构建复杂的应用程序需要一些时间去适应，很多反模式可能源于“将其他应用程序开发概念强加到Slate上”，而不是利用Slate提供的框架进行工作。对于实现给定功能，通常有一些隐含的“正确”模式，这些模式最能与Slate的基本概念相结合。

这部分的大部分复杂性在理解依赖关系中有所介绍 ，包括Slate核心组件（依赖图、事件框架、小部件、查询、函数等）的讨论，常见模式（以及常见反模式）以及设计和实现的最佳实践。

我们建议开发人员和支持工程师熟悉此文档，因为许多常见问题都可以通过已深入解释的内容得到解答。在许多情况下，回答问题的最佳方法不是尝试解决特定实例——例如，通过打开应用程序并尝试逆向工程用户的开发决策——而是理解用户遇到的一般故障类别，并引导他们到以下任何相关的讨论中：

  • 了解依赖关系
  • 构建复杂布局
  • 启用用户交互
  • 与该问题相关的各种基于网络的资源

对于一些最常见的案例，本FAQ文档是快速获得简要解释和有用参考资料的捷径。

查询

失败的查询

与查询相关的问题有多种类型，找出根本原因可能会指向多种解决策略。在某些情况下，问题会以明确的错误信息呈现，但在其他情况下，可能没有错误信息，或者失败的条件可能是非确定性的（即有时会发生，有时不会发生）。以下的故障排除步骤列出了常见原因和相关的解决策略。

要进行故障排除，请执行以下步骤：

有时查询会违反同步网络请求的一般使用模式：

  • 试图返回太多数据到浏览器的查询。
    • 查询应遵循一个粗略的规则，即返回的数据行数应少于10,000行。
      • 有时查询会显式返回一个错误信息，提示“太多行”，但通常情况下，查询会成功执行，但整个应用程序会变得非常缓慢。在这种情况下，请查找返回大量行的查询，并尝试减少返回的数据量。
    • 如果您试图显示包含大量行的表格，或者需要向用户展示原始行数据，请使用LIMIT和OFFSET语句来实现分页工作流，可以使用表格小部件内置属性，也可以使用自定义的工作流。
  • 查询过大，导致请求本身耗时很长。
    • 查询的往返时间应小于5秒。
    • 如果您有一个具有大负载的查询，可以考虑将更多查询中的逻辑推送到生成数据集的转换层中。

对于这类一般性问题，请参考优化索引和模式设计 文档并考虑到应用程序应进行结构化，因此返回到浏览器的数据应为最终形式，而不是需要在JavaScript函数中进行大量处理。

有时问题则与Postgres如何解释和响应请求有关：

  • 由于查询结构不理想而运行缓慢的查询。
  • 由于数据集未被索引或错误地索引而运行缓慢的查询。
  • 语法不正确的查询。

这类问题通常通过搜索Postgres + [error message]中的Postgres + [desired query]来解答。对于所有与查询性能相关的故障排除或调查问题，请参考Postgres查询性能和调优指南。

有时问题则在于Slate根据开发人员的配置执行查询的方式和时机：

  • 页面加载时查询非确定性失败。
    • 请参考最佳实践指南中的管理查询执行部分。
  • 在20秒（Slate超时时间）时返回无用错误的查询
    • 通过Slate查询发出的请求会在Slate端超时
    • 在 20 秒时，无论请求的服务器端发生了什么。对此故障模式的最终诊断方法是在刷新页面时，使用 Google Chrome 网络检查器打开应用程序的 查看 模式。这将显示每个查询请求的持续时间，如果您看到某个查询在 20s 处变为红色，就可以确定故障是因为 Slate 超时了查询。
  • 通过提高查询性能来解决此问题： Postgres 查询性能和调优 和 控制查询执行。
  • 有关超时的更多信息，请参阅下一节 Postgres 查询超时。
  • 在某些情况下失败的查询（因为 Handlebars 渲染没有得到很好的保护）。
    • 如果您在查询中有 Handlebars 语句，请确保它们通过 {{#if}} 语句或通过函数中的逻辑针对 null 值或其他边缘情况进行保护。
  • 在错误的时间执行或未能执行的查询。
    • 确保您 了解在 Slate 中事物发生的原因 以及您是否适当 控制查询执行。对于复杂的应用程序或那些过度依赖事件的应用程序，关于 Event Madness 的注意事项是相关的。

    Postgres 查询超时

    当查询超时时，主要有两个因素在起作用：

    • 数据规模 - Postgres 表中有多少行数据，以及您的查询返回了多少行数据。
    • 查询优化 - 查询的“成本”有多大。

    作为应用程序开发者，有必要了解这些因素以及它们如何影响您的查询。这些并非 Slate 或 Foundry 特有的问题，因此您可以搜索 Postgres SQL query performance 并找到通用的最佳实践以及在 Postgres 查询性能和调优 指南中的技巧。

    要进行故障排除，请执行以下步骤：

    一些提示，可能对您有帮助，基于应用开发者常遇到的问题：

    • 数据规模：限制您同步到Postgres的数据集的规模。例如，如果您只需要一年的数据，请不要同步一个< strong >所有时间的数据集，然后在查询中进行过滤。

    • 数据规模：减少表中的列数。例如，如果您在查询中只选择几列，请在同步表之前通过转换操作将它们删除，以避免开销。

    • 数据规模：尽可能返回较少的行。如果您要显示原始行，请使用OFFSET和PAGING来获取后续行，而不是在每次查询响应中返回几百行以上的数据。

    • 查询优化：尽可能避免昂贵的SQL命令。常见的问题包括COALESCE、DISTINCT、任何在不“完整”列（如某些值是null）上的JOIN，或者涉及前导通配符（%）或复杂匹配表达式的WHERE子句筛选。

    • 查询优化：将工作上移。这包括数据清理（删除空值、格式化日期、映射显示值）和指标计算（每月总计、跨桶的平均值等），应在可能的情况下推送到转换层。如果您需要允许用户通过过滤器自定义这些内容，您仍然可以在几个维度上“预聚合”指标，然后对满足这些条件的少量行执行更简单的查询来求和/平均等。

    • 查询优化：使用 EXPLAIN ANALYZE 获取查询计划并了解Postgres如何解释您的查询。这将帮助识别时间花费的瓶颈，并进一步针对您的优化。

    • 查询优化：查看 在Postgres中优化查询。

    无法在我的Postgres查询中引用列

    在Postgres中，列名默认为小写。由于Foundry允许列名中使用灵活的大小写和空格，当一个表被索引到Postres时，该表会被双引号（"）包裹以保留大小写。

    这意味着如果列名中有大写字母，您在SQL查询中引用该列时也需要用双引号包裹。

    您可以在此处了解有关此行为的更多信息：StackOverflow ↗。

    要进行故障排除，请执行以下步骤：

    处理不当的列标识符的查询通常会报错，例如：

    ERROR: column mixedcasecolumn does not exist HINT: Perhaps you meant to reference the column "Mixed Case Column"

    在这种情况下，最佳做法是更新相关数据集，使其使用符合Postgres默认设置的列标识符，通常为小写字母并用下划线代替空格。否则，请确保这些标识符被正确地用双引号包裹。

    Phonograph查询在提交时截断文本

    在边缘情况配置中，你可以观察到引用一个或多个 Text Area 小部件的写回查询，会发送一个仅包含部分文本的 Phonograph 请求。例如，如果用户输入如 This is a test message for writeback 的长文本，并快速按下 Submit 按钮以触发写回查询，可能只包括 This is a te。

    根本原因是 Text Area（文本区域）小部件使用的 0.5 秒防抖延迟，用于防止依赖图在每次小部件输入按键时重新计算。这意味着，只有当以下条件同时满足时，查询才会在没有完整文本的情况下提交：

    • 文本区域小部件是用户最后填写的输入项，并且
    • 在最后一次输入文本区域中的字符后，小于 0.5 秒就按下了提交按钮。

    要进行故障排除，请执行以下步骤：

    最简单的解决方案是调整用户体验，使得在输入长文本后的0.5秒内无法使用提交选项。例如，在提交查询之前添加一个确认对话框，或者只是重新排序表单元素，以便在文本区域小部件之后还有一个字段需要填写。

    其他解决方案是将文本区域小部件替换为Text Input小部件，该小部件不包含防抖延迟。也可以使用Toast小部件作为提交按钮按下和查询执行之间的延迟缓冲器，以确保足够的时间过去，从而包含完整的文本。

    布局

    应用程序渲染问题

    在某些配置中，如果你尝试将小部件设置为“动态”宽度和高度，那么在页面加载时，你会发现小部件被渲染成一个小的盒子。触发[w_myWidget.resize]操作会导致小部件按照预期大小进行渲染。

    这种行为是由于CSS中指定各种小部件元素的宽度和高度为百分比（通常是100%）与一个或多个父容器上的Flex属性之间发生冲突导致的。在这种冲突下，Slate无法在渲染时确定包含小部件的元素的正确尺寸，因此会将小部件渲染得太小。

    要进行故障排除，请执行以下步骤：

    实现响应式小部件尺寸的正确模式是在父容器和子小部件上仅使用Flex配置设置。不要通过CSS手动设置其他width或height属性。要解决此问题，应从根级子小部件开始，然后依次向上遍历所有父小部件，删除任何位置或尺寸的CSS规则。

    您可以使用 margin 和 padding 属性来插入或以其他方式对齐子控件。如果您熟悉 Flexbox CSS 规则，可以使用 Chrome 调试工具来识别要定位的元素和类，并提供覆盖或额外的 Flexbox 规则，以实现所需的精确行为。

    My Slate app will not print / How do I export my App

    Slate 是为快速应用程序开发而优化的。底层框架对布局和样式的可配置性限制很少。因此，Slate 无法以适合打印媒体或导出的方式通用地渲染所有应用程序。应用程序开发人员需要在每个应用程序的基础上投入时间来支持打印功能。

    要进行故障排除，请执行以下步骤：

    第一个也是最好的选择是考虑为什么会有打印您应用程序的要求或请求。您的用户是否只是需要一种方法来保存他们创建的视图并在以后参考它，或者在演示过程中使用它？

    在这种情况下，请考虑使用内置的 可共享视图 功能，该功能会生成一个唯一的 URL，以保留所有小部件选择和变量的状态。用户可以从 视图 模式中的 操作 下拉菜单中访问此功能。

    如果您确实必须打印或导出，则需要定义额外的媒体 CSS 样式，以帮助您的应用程序正确渲染。

    首先，您需要至少添加以下内容到 全局样式 中：

    @media print {
  @page {
    size: A4;
  }

  html, body , .ws-root-container.ws-sandbox {
    height: 600mm !important;
  }

  sl-app-view-actions {
    display: none !important;
  }
}

    打印响应式应用程序或任何具有复杂样式或布局的应用程序将非常困难。在这些情况下，建议创建一个“打印视图”作为单独的应用程序，通过 URL 参数接收必要的输入，并生成优化用于打印的简化视图。这将增加开发成本和维护负担，但会带来满意的行为。

    性能

    我的应用加载缓慢或在编辑模式下运行迟缓

    和所有Web应用程序一样，Slate最终使用终端用户的计算机中的浏览器资源来执行JavaScript代码并根据HTML中定义的文档模型（DOM）以及CSS中指定的样式进行渲染。这些资源的一部分是恒定的；它们是使用Slate框架工作的开销。然而，在性能不佳的情况下，根本原因通常是应用程序过载的一种形式，即在开发过程中定义的配置要求浏览器提供过多的资源来渲染应用程序、执行JavaScript，或者仅仅是存储许多大型查询的结果。

    In some cases, a single function or query might be the cause. In this case, you can use the performance metrics in the Dependencies tab of an application to look at the Time in Node for various queries and functions. This metric represents the duration spent resolving that particular node on page load, which can help identify likely candidates for application bottlenecks.

    In general, a best practice is to favor lightweight, well-bounded, single-serving applications and create large workflows or applications out of “clusters” of small application with deep linking between them.

    Configuration

    The document is too large to save

    Slate saves the application configuration as a JSON document, which generally has a size limit of 2,000,000 bytes of UTF-8 encoded characters. Attempts to save a Slate application with a configuration above the size limit will fail, and changes will need to be made to the application configuration to decrease the size. The JSON document of an opened Slate application can be retrieved through export.

    Every part of a Slate application takes up configuration size to some degree, and editors should be conscious about both the scale of Slate applications being built, and the resources being embedded in the application configuration. The following are prevalent reasons for the overuse of configuration size, along with proposed solutions:

    Base64 encoded images

    Upload the image to Foundry and reference the image using the Image widget.

    Large functions

    Consider refactoring functions to extract common logic into function libraries.

    Additionally, large functions may be better suited as Foundry Functions consumed via the Platform editor.

    Large function libraries

    While function libraries are shared across all pages of a Slate application to conserve space, javascript libraries can be quite large. Either refactor custom libraries to use shared logic, or use minified versions of external libraries.
