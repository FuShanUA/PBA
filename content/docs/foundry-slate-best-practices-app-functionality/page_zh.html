理解依赖关系

理解Slate中的“为什么事情会发生”

在Slate中，所有小部件（widgets）、函数（functions）、变量（variables）和查询（queries）都被建模为图中的节点。每个节点计算出一个JSON输出，其他节点则模板化以引用该输出。这些引用定义了节点之间的依赖关系，Slate利用这些依赖关系来理解当某些内容发生变化时（例如新的用户输入；查询执行；变量值更新）何时需要重新计算节点。

“开发”你的Slate应用程序的主要过程就是通过添加小部件（用于显示输出和捕获输入）；函数（用于对数据进行轻微操作以供显示和处理应用程序登录）；以及查询 - 用于对数据或与其他Foundry服务API的连接进行查询，然后通过 Handlebar 语法配置它们的依赖关系。

依赖图

你可以始终使用 依赖项 标签页查看此 依赖图 的表示形式，以理解Slate如何根据你的 Handlebar 引用解释配置的关系。这个图表表示 就是 你的应用程序 - 如果它显示的内容看起来“有问题”，那么你的起点是理解 为什么，而不是假设图表没有正确表示你的应用程序。

当你的应用程序中发生意外情况时，从依赖项标签页开始，了解任何行为异常的组件的上游节点。查找可能导致某个节点基于不相关的查询或用户输入重新评估的意外关系。

有时可以将依赖图视为“惰性”的——它通过仅在上游引用发生变化时重新计算节点来避免不必要的工作 值。当上游节点更新为与之前相同的 值 时，这会导致潜在的令人困惑的行为——下游查询不会执行，下游小部件也不会重新渲染。这在涉及重置用户输入的模式中很常见——在 Foundry 中搜索 依赖图和事件框架简介 教程和 重置小部件选择 示例，以了解其实际表现以及如何使用 Math.random() 向依赖图评估中注入熵，确保节点重新计算。

事件

借助依赖图，Slate 能够处理确定函数何时需要重新计算或查询何时需要重新运行的复杂性，这取决于它们上游依赖项的状态。大多数工作流程都可以使用此功能进行配置，并且它应该 始终 作为Slate开发人员，应尽可能首先依赖依赖图。

然而，在某些情况下，使用显式的触发/操作对来补充依赖图是有意义的，Slate将这些称为“事件”。一些简单的例子包括在查询完成后触发显示横幅或提示，当用户点击某处以查看更多信息时打开对话框，或在配置了多个输入后通过按钮点击运行查询。

了解有关事件框架以及常见基本事件模式示例的更多信息。

事件狂热

一旦你意识到事件框架的力量，可能会倾向于通过基于事件的解决方案来看待Slate中的每个开发任务。这可能导致一种常见的失败模式，即单个应用程序包含数百个事件/操作对。虽然这在技术上是可能的，但这种复杂性使得难以逐步跟踪应用程序的预期行为，并且甚至会妨碍简单的调试。

通常情况下，应尽可能保持事件“链”简短，并倾向于依赖依赖图（Dependency Graph）而非事件（Events）的解决方案。如果你发现你的应用增长到超过50个事件，特别是这些事件不仅仅是触发提示或启动查询时，就该停下来仔细思考你的应用程序架构，寻找重构和简化开发的机会，或者进一步将应用分解为子应用。

自定义事件触发器

除了内置的事件类型外，你可以在HTML和Table类型的控件中创建自定义事件，以允许这些控件中的不同元素通过用户点击广播不同的事件。

这些自定义触发器的配置分为两个步骤：

  1. 在HTML元素中使用slClickEvent属性定义触发器：
Copied!
1<div class='pt-button' slClickEvent='delete'>Delete Me</div>
  1. 通过将名称添加到点击事件名称数组中，在控件配置中注册触发器。

一旦新事件被注册，您将在任何可以选择事件触发的地方看到一个新选项，例如 myWidgetName.click.delete。请注意，即使您动态生成HTML来定义点击事件，这些事件的注册仍需要硬编码，以便Slate能够跟踪潜在的触发事件，即使当前条件不实际生成该按钮。

您可以使用这些事件在HTML中构建自定义按钮组，或将其与下面讨论的事件值结合，以创建每个项目都有自己操作集的列表或表格。

条件事件

如果您需要添加额外的条件（除了简单的触发器触发），可以使用事件配置的JavaScript部分执行额外的逻辑。评估您的逻辑，如果事件不应执行操作，请返回特殊的{{slDisableAction}}值。此值不会永久禁用事件，而是中断该特定的 事件操作的实例。

作为最佳实践，避免在您的事件 JavaScript 内部开发复杂的逻辑 内部 - 没有代码检查或错误报告，因此任何错误或未覆盖的边界情况都可能导致您的代码静默失败并导致意外行为。相反，可以考虑实现一个函数来封装事件有效性的逻辑 - 然后您的事件 JavaScript 可以非常简单，例如：

unless ({{f_checkValidAction}})
    return {{slDisableAction}}

然而，您可以在事件中放入 debugger() 语句，然后使用 Chrome 开发者工具来捕获它们的执行并逐步调试您的逻辑。

传递事件值

除了 slClickEvent 属性外，您可以为给定的 HTML 元素定义一个 slClickEventValue。每当此事件被触发时，您可以使用特殊的 {{slEventValue}} 变量在您的事件 JavaScript 中引用关联的值。

这为您打开了各种动态交互模式：

  1. 创建一个函数，生成您的 文本 或 表格控件并为每个元素定义slClickEvent和slClickEventValue。
  2. 创建必要的事件，并使用这些事件将一个或多个单个变量的值设置为事件中的值。
  3. 在variable.changed事件触发器上触发一些进一步的操作 - 最常见的是查询 - 并引用变量的值。

例如，您可以使用此模式在元素中表的行旁边添加一个小的'x'，其中slClickEvent为“删除”，而slClickEventValue为行主键。然后当myTable.click.delete事件触发时，可以将名为v_idToDelete的变量的值设置为事件值。然后根据v_idToDelete.changed事件运行一个q_deleteRow查询以完成这个小链。

请注意，不要使链条比这更长或更复杂 - 请参见下面关于循环依赖的警告 - 并且应优先选择简短、明确的事件，而不是大型、嵌套的或以其他方式复杂的结构，因为这些结构难以推理和调试，并且通常会导致意外行为。

事件和循环依赖

Slate 依赖图在技术上是一个“有向无环图”（DAG），这意味着节点关系是定向的（即存在从根节点到叶节点的层次结构，节点解析按此方向进行），并且整个图必须是无环的，这意味着 Slate 将尝试阻止您在依赖关系图中配置任何“循环”——无论多长。

然而，使用 事件 和设置变量值或触发查询，有可能在您的应用中添加一个循环，这将导致非确定性行为。如果您有一个似乎需要循环依赖的工作流，请退一步考虑替代模式——几乎总是有办法实现所需的工作流功能。

管理查询执行

依赖图框架的一个影响是，默认情况下整个依赖图会在页面加载时进行解析。这意味着每个查询和函数将按照每个节点解析所需的顺序执行。实际上，这通常意味着如果你在开发过程中没有注意，你会逐渐注意到页面加载性能下降，直到开始出现间歇性超时。根本原因几乎总是许多“根”节点的查询，因为它们没有上游查询依赖项——这意味着所有这些查询都将尝试同时运行。这可能导致由于大量同时浏览器连接、对同一张表的大量同时查询以及返回到应用程序的大量网络流量而产生的排队和负载削减。最终状态是无法确定的查询失败，以及页面加载时应用显得迟缓，甚至完全崩溃。

使用 Postgate 的连接池和排队

在 Postgate 中，您应用程序的每个用户都有一个 连接池 专门为他们创建（如果他们最近已经使用过该应用程序，则从缓存中恢复）。每个连接池同时最多限制为10个连接，这防止了任何一个应用实例过度加载Postgate，因为Postgate被您应用程序的所有用户以及任何使用已同步数据的其他应用程序所共享。从性能角度来看，这个限制实际上比一次性允许所有查询通过（如果有超过10个查询）会带来更好的性能，因为Postgres在响应的并发查询越少时整体速度越快。

一旦所有10个连接都被使用完，剩余的查询请求将进入队列，一旦有连接可用就会获得一个连接。然而，为了避免连接无限期等待，连接池会在查询等待连接超过五秒时进行超时处理。这意味着如果您有一系列需要数秒的查询耗尽了连接池，那么原本可能很快的查询——例如，当您使用测试按钮单独运行它时——可能会花费很长时间甚至被超时。

这里的解决方案包括几个部分：

  1. 开发您的应用程序工作流，使查询间隔开并依赖于用户输入。一个常见模式是针对预聚合或DISTINCT表运行几个小查询以填充一组筛选条件，但“强制”用户进行一些选择并点击提交按钮以运行实际获取数据的查询。
  2. 谨慎地开发查询，注意平衡查询总数 - 特别是页面加载时运行的“重型”（平均返回时间>5秒）查询的总数，与每个查询返回的数据量。确保您的查询没有做不必要的工作 - 请参见上面关于索引和架构和Postgres调优的部分。
  3. 通过指定在查询执行前必须满足的额外条件，或者将查询设置为manual，然后通过事件触发查询 - 常见的是按钮点击。
简洁的查询

在所有开发过程中，你应该努力编写简洁的代码——你的代码应该以最有效的方式完成必要的工作。这通常是Slate查询的缺点，因为它们会花费大量时间重复做同样的工作。请考虑这个简单的查询，用于为用户选择类别填充下拉框：

Copied!
1SELECT DISTINCT(COALESCE(category_col)) FROM "my_table"

这意味着在每次页面加载时，Postgres都必须重复查找每行中的 category_col 值，以查看它是否为 null，然后生成所有不同值的列表。通过使用转换和一个新的派生数据集，当上游有新数据添加时，这项工作只需执行一次，然后查询就变为：

Copied!
1SELECT * FROM "my_table_categories"

即使你不能完全去除 SQL 表达式，例如当需要适应用户输入时，你也应努力将尽可能多的工作提取到上游转换中，然后构建更简单的查询。始终牢记“尽可能少地执行工作”这一原则，你将走上更加稳定和高效的应用程序开发之路。

第三部分，让我们更详细地看一下两种常见的模式。在你的应用程序中使用这些模式可以大大减少页面加载时运行的查询数量，并确保应用程序的行为符合预期。

条件查询

条件查询为查询执行提供了最细粒度的控制，并且可以与“事件触发”或“手动”配置（如下所述）结合使用，以添加查询执行必须满足的逻辑条件。通过在In addition...下拉菜单旁边的Run下方勾选In addition...复选框，可以为查询开启执行条件。

在最简单的情况下，All dependencies are not null选项会阻止查询的执行，如果查询或其部分中的任何handlebar表达式没有指定值。这对于需要一些用户输入后才有意义的查询特别有用，并且可以防止因WHERE语句不完整而失败的查询不必要的数据库往返。

手动查询

一种简单控制查询运行时间的策略是将它们设置为manual，使用下拉菜单中Run按钮旁边的复选框。

设置为手动运行的查询仅在由事件或手动指定的依赖项触发时才会执行。

这是应由用户操作触发的查询的正确模式，但如果你开始“链式”连接太多这样的查询，可能会导致不必要的复杂性；请参见上面事件部分中的注意事项。你仍然可以依靠依赖关系图来处理手动查询下游的所有节点 - 在查询从其触发器运行后，任何下游依赖项都会自动更新；无需通过q_queryName.success事件添加事件来增加复杂性。

JavaScript 函数的最佳实践和常见模式

函数可以让你根据用户的选择和应用程序状态对数据进行轻量级操作或执行逻辑。一些经常被忽视的模式和特性可以使你的函数工作更清晰、更简单。

通过返回复杂对象避免重复工作

如果你查看 验证用户输入 的示例，你会发现该验证函数返回的是一个 JavaScript 对象（JavaScript 对象），而不是单个值。Slate 可以解析这些对象，并通过 Handlebars 在你应用程序的其他任何地方使用它们。

这意味着，例如，如果你需要派生几个新的显示列来在不同的小部件中使用，你不需要为每个列都创建一个函数，而是可以创建一个函数，该函数以不同格式生成所有所需的显示值，并将它们全部返回到一个对象中。我们将在下面更详细地看一下这个例子。

在将逻辑集中到更少的函数中与拆分逻辑之间总是存在一种平衡，前者有助于减少重复工作并保持相似的工作集中，后者有助于封装独立的操作，并使找到必要的代码变得更加容易。请记住这种权衡，并在完善应用程序时定期重构您的函数。

生成显示列

通常你会发现需要做一些额外的工作来对数据进行处理，以便在图表中显示或生成一些HTML用于表格中显示。尽可能地，在格式化日期等情况下，您应该探索在数据转换和准备阶段完成这些工作——这遵循了尽可能少地执行工作（即在转换中执行一次，而不是在每个页面加载时为每个用户执行）的原则。

Where that's not possible, you can write a simple function to map through each value in a column and do some work to generate a new value, all while preserving the data structure that Slate expects.

Copied!
1`// f_deriveDisplayColumns
2
3var data = {{q_myQuery.results.[0]}}
4data.newColA = _.map(data.primaryKey, (value,key,index) => {
5    // Work to derive new column
6    return newColValue
7});
8return data;`

This snippet uses the Lodash ↗ version of the _.map()function; you could equally use the build in Array.map() for this case. It is generally useful to familiarize yourself with Lodash helper functions as they are tremendously helpful in manipulating JavaScript objects and often implement complex algorithms that could be bug-prone to re-implement from scratch.

With the function complete, rather than referencing {{q_myQuery}} directly, you can instead reference {{f_deriveDisplayColumns}} anywhere in your app, since it has the same structure as the original query response object, just with additional “columns” added in.

Factoring code with custom libraries

If you do find that you need to implement a rather complex piece of JavaScript, consider moving it to a custom library at the bottom left of the function editor. Libraries make pure JavaScript functions available in any Slate “function”, so you can avoid repeatedly implementing the same helper functions multiple times.

Since Libraries are pure JavaScript, you can't use Handlebars inside them; rather like normal JavaScript you can pass parameters into the library functions when they are called from a Slate “function”. These parameters can then be filled with Handlebars or otherwise dynamically generated.

Documenting your functions

In both your Library code and your functions, always take the time to organize your code cleanly and document the potentially unintuitive sections. You can always strive to write “self-documenting” code, but it is always kind to future developers and maintainers to be in the habit of cleanly and succinctly documenting your logic.
