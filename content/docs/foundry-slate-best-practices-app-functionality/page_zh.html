理解依赖关系

理解 Slate 中“事情发生的原因”

在 Slate 中，所有的微件、函数、变量和查询都被建模为图中的节点。每个节点都会求值生成 JSON 输出，而其他节点则通过模板化的方式来引用该输出。这些引用定义了节点之间的依赖关系，Slate 借此判断在发生变更时（例如接收到新的用户输入、查询执行完毕、变量值更新）何时对节点进行重新求值。

“开发” Slate 应用的一项主要工作就是定义这些节点：添加微件用于展示输出和捕获输入；添加函数用于对展示的数据进行轻量级处理以及处理应用登录逻辑；添加查询用于检索数据或连接其他 Foundry 服务的 API。随后，通过 Handlebar 语法来配置它们之间的依赖关系。

依赖图

你可以随时通过“依赖关系”选项卡查看这个依赖图的可视化表示，了解 Slate 是如何解析你通过 Handlebar 引用配置的各项关系的。这个图结构本身就代表了你的应用——如果它呈现出看起来“错误”的内容，你的出发点应该是去理解为什么会这样，而不是假设依赖图曲解了你的应用。

当应用中出现任何意外情况时，首先查看“依赖关系”选项卡，从出问题的微件入手理清其上游节点。留意那些意料之外的关联，它们可能会导致某个节点因为无关的查询或用户输入而被重新求值。

有时可以把依赖图想象成“懒散”的——它只在上游引用的值发生改变时才重新对节点求值，以此避免无谓的计算。然而，当上游节点更新后的值与之前完全相同时，这种机制就会导致一些令人困惑的行为：下游查询不会执行，下游微件也不会重新渲染。在涉及重置用户输入的模式中经常会遇到这种情况——在 Foundry 中搜索《依赖图与事件框架入门》教程和《重置微件选择》示例，可以实地观察这一现象，并学习如何利用 Math.random() 向依赖图的求值过程引入熵，从而确保节点能够顺利重新求值。

事件

借助依赖图，Slate 能够处理复杂的逻辑判断，根据上游依赖的状态决定何时重新计算函数或重新执行查询。大多数工作流都可以通过这一功能完成配置，因此作为 Slate 开发者，应始终首选尽可能依赖依赖图的方式来解决问题。

不过在某些情况下，利用显式的触发器/动作对来补充依赖图的功能会更为合理，Slate 将其称为“事件”。一些简单的场景包括：在查询完成时触发显示横幅或提示框、当用户点击某处以查看更多详情时打开对话框，或者在配置好一系列输入项后通过点击按钮来运行查询。

了解有关事件框架的更多信息以及常见的基础事件模式示例。

事件狂热

当你认识到事件框架的强大之处后，很容易陷入一种误区：试图通过基于事件的解决方案的视角来审视 Slate 中的每一项开发任务。这会导致一种常见的失效模式——单个应用中包含数百个事件/动作对。虽然技术上可行，但这种复杂性会让你难以梳理应用预期的行为逻辑，甚至连简单的调试都无法进行。

作为一条经验法则，请尽量缩短事件“链”，并优先考虑依赖图而非事件的解决方案。如果你发现应用的事件数量增长超过 50 个，特别是当这些事件的作用不仅限于触发提示框或发起查询时，就应该暂停下来，仔细审视应用架构，寻找重构和简化开发的机会，或者将应用进一步拆分为多个子应用。

自定义事件触发器

除了内置的事件类型外，您还可以在 HTML 和 Table 类型的微件中创建自定义事件，使这些微件中的不同元素能够在用户点击时广播不同的事件。

配置这些自定义触发器分为两个步骤：

  1. 在 HTML 元素中使用 slClickEvent 属性定义触发器：

1<div class='pt-button' slClickEvent='delete'>Delete Me</div>
  1. 在微件配置中，将名称添加到 Click Event Names 数组以注册触发器。

注册新事件后，只要您可以选择事件触发器，就会看到一个新选项，例如 myWidgetName.click.delete。请注意，即使您通过动态生成 HTML 来定义点击事件，这些事件的注册仍需硬编码，以便 Slate 能够跟踪潜在的触发器，即使在某些条件下您当前并未实际生成该按钮。

您可以使用这些事件在 HTML 中构建自定义按钮组，或将它们与下文讨论的事件值结合使用，以创建每个项目都有独立操作集的列表或表格。

条件事件

如果需要添加额外条件（超出简单的触发器触发范围），您可以使用事件配置中的 JavaScript 部分来执行额外的逻辑。评估您的逻辑，如果该事件不应执行操作，请返回特殊的 {{slDisableAction}} 值。此值不会永久禁用该事件，而是中断该事件操作的特定实例。

最佳实践是，避免在您的事件 JavaScript 内部开发复杂的逻辑——因为这里没有代码检查或错误报告功能，任何错误或未覆盖的边缘情况都可能导致代码静默失败并引发意外行为。相反，建议实现一个封装事件有效性逻辑的函数——这样您的事件 JavaScript 可以简化为：

unless ({{f_checkValidAction}})
    return {{slDisableAction}}

不过，您可以在事件中放入 debugger() 语句，然后使用 Chrome 开发者工具来捕获其执行过程并逐步调试您的逻辑。

传递事件值

除了 slClickEvent 属性外，您还可以为给定的 HTML 元素定义 slClickEventValue。每当触发此事件时，您都可以在事件 JavaScript 中使用特殊的 {{slEventValue}} 变量来引用关联的值。

这为动态交互开辟了多种模式：

  1. 创建一个函数，为您的 Text 或 Table 微件生成 HTML，并为每个元素定义 slClickEvent 和 slClickEventValue。
  2. 创建必要的事件，并使用这些事件将一个或多个单变量的值设置为事件传来的值。
  3. 基于 variable.changed 事件触发器触发进一步的后续操作（通常是查询），并引用该变量的值。

例如，你可以利用这种模式，在表格行旁添加一个小的 'x'，所在元素的 slClickEvent 设为 'delete'，并将 slClickEventValue 设为该行的主键。随后，在 myTable.click.delete 事件触发时，将名为 v_idToDelete 的变量值设为该事件值。接着，基于 v_idToDelete.changed 事件运行 q_deleteRow 查询，即可完成这一小段链式逻辑。

切忌将此类链式逻辑设置得过长或过于复杂——请参阅下文关于循环依赖的警告——建议多使用简短、独立的事件，而非庞大、嵌套或错综复杂的组合，因为后者不仅难以推理与调试，还常常引发意外行为。

事件与循环依赖

从技术上讲，Slate 的依赖图是一个“有向无环图”（DAG），这意味着节点关系是有向的（即从根节点到叶节点存在层级结构，且节点按此方向进行解析），并且整个图必须是无环的，也就是说 Slate 会阻止你在依赖图中配置任何形式的“环路”，无论该环路有多长。

然而，如果使用事件并设置变量值或触发查询，确实有可能在应用中引入环路，从而导致非确定性行为。如果你的工作流似乎必须依赖循环依赖，不妨退一步考虑其他替代模式——几乎总能找到一种方法来实现所需的工作流功能。

管理查询执行

依赖图框架的一个特性是，默认情况下整个依赖图会在页面加载时完成解析。这意味着每个查询和函数都会按照图中各节点解析所需的顺序运行。在实际开发中，如果你开发时没有注意这一点，会逐渐发现页面加载性能不断下降，直到开始出现间歇性超时。根本原因几乎总是出在大量作为“根”节点的查询上——由于它们没有上游查询依赖，这意味着所有这些查询都会尝试同时运行。这种情况会导致大量并发的浏览器连接引发排队和负载丢弃，对同一张或多张表发起过多同时查询，以及返回应用的网络流量过大。最终结果是引发非确定性的查询失败，且应用在页面加载时会显得卡顿，甚至完全失去响应。

Postgate 的连接池与排队机制

在 Postgate 中，应用的每位用户都会获得一个专门为其创建的连接池（如果该用户最近使用过应用，则会从缓存中恢复连接池）。每个连接池限制最多 10 个并发连接，这能防止单个应用实例过载 Postgate，因为 Postgate 是由应用的所有用户及其他任何使用已同步数据的应用共享的。在性能方面，相比一次性放行所有查询（如果超过 10 个），这种限制实际上能带来更好的性能，因为 Postgres 响应的并发查询越少，全局处理速度就越快。

一旦 10 个连接全部用完，剩余的查询请求就会排队，等待有可用连接时再依次执行。然而，连接池不会让连接无限期等待，对于排队超过 5 秒的查询将会超时。这意味着，如果有多个耗时数秒的查询占满了连接池，那些原本可能很快执行的查询——例如你使用测试按钮单独运行时——可能会耗费很长时间，甚至直接超时。

针对此问题的解决方案包含以下几个方面：

  1. 规划应用工作流时，将查询错开执行并使其依赖于用户输入。一种常见模式是先对预聚合表或 DISTINCT 表运行几个小查询来填充一组筛选器，然后“强制”用户进行选择并点击提交按钮，再去执行实际获取数据的查询。
  2. 谨慎开发查询，平衡查询总数——尤其是页面加载时运行的“重型”查询（平均返回时间 >5 秒）总数——与每次查询返回的数据量。确保查询没有执行多余的工作——请参阅前文关于索引与模式和Postgres 调优的章节。
  3. 通过指定查询执行前必须满足的附加条件，或将查询设置为 manual 并随后通过事件（通常是点击按钮）触发查询，来控制查询的执行时机。
    精简查询

    在所有的开发工作中，你都应该力求代码精简——以尽可能高效的方式只完成必要的工作。这通常是 Slate 查询的败因所在，它们会耗费大量时间反复执行相同的工作。来看下面这个简单的查询，其目的是填充一个供用户选择类别的下拉列表：

    已复制！
    1SELECT DISTINCT(COALESCE(category_col)) FROM "my_table"

    这意味着在每次加载页面时，Postgres 都得重复执行这项工作：检查每一行中 category_col 的值是否为 null，然后生成所有去重值的列表。如果使用转换逻辑并生成一个新的派生数据集，这项工作就只需在上游新增数据时执行一次，随后的查询便简述如下：

    已复制！
    1SELECT * FROM "my_table_categories"

    即使你无法完全摒弃 SQL 表达式（例如需要处理用户输入的情况），也应尽可能将繁重的工作剥离到上游转换逻辑中，从而构建更简单的查询。时刻牢记“尽量减少重复工作”这一原则，你的应用就会具备更高的稳定性和性能表现。

    在第三部分中，我们来详细看看两种常见模式。在应用中使用这些模式，可以大幅减少页面加载时运行的查询数量，并确保应用按预期运行。

    条件查询

    条件查询为查询执行提供了最细粒度的控制能力，并且可以与“事件触发”或“手动”配置（下文详述）相结合，从而添加查询执行必须满足的逻辑条件。勾选 Run 旁下拉菜单下的 In addition... 复选框，即可为查询开启执行条件。

    在最简单的情况下，所有依赖项均不为空选项可以防止查询执行，前提是查询或其局部模板中的任何 Handlebars 表达式没有指定值。这对于需要一定用户输入才能产生实际意义的查询尤为实用，同时也能避免因 WHERE 语句不完整而注定失败的查询进行不必要的数据库往返。

    手动查询

    控制查询执行时机的一种简单策略是，使用 运行 按钮旁边下拉菜单中的复选框将其设置为 手动。

    设置为手动运行的查询仅在由事件或手动指定的依赖项触发时才会执行。

    对于应由用户操作触发的查询而言，这是一种正确的模式。但如果开始将过多查询“链接”在一起，可能会导致不必要的复杂性——请参阅上文 事件 部分的说明。你依然可以依靠依赖图来处理手动查询下游的所有节点——查询由触发器运行后，任何下游依赖项都会自动更新；无需基于 q_queryName.success 事件添加额外的事件来增加复杂性。

    JavaScript 函数的最佳实践与常见模式

    函数能够根据用户选择和应用状态对数据进行轻量级处理或执行逻辑。一些常被忽视的模式和特性可以让你的函数代码更简洁、更清晰。

    通过返回复杂对象避免重复工作

    查看《验证用户输入》示例，你会注意到校验函数返回的并非单个值，而是一个 JavaScript 对象。Slate 能够解析这些对象，并通过 Handlebars 使其在应用程序的其他任何位置可用。

    这意味着，例如当你需要派生几个新的显示列用于不同组件时，无需为每个列单独编写函数，你可以构建一个单一函数，按不同格式创建所需的所有显示值，并将它们打包在一个对象中返回。我们将在下文详细探讨此示例。

    将逻辑整合到更少的函数中，有助于减少重复工作并保持相似逻辑的集中管理；而拆分逻辑则有助于封装独立操作，让你更轻松地找到所需代码。这二者之间始终需要权衡。请在完善应用的过程中牢记这一取舍，并定期重构你的函数。

    生成显示列

    你常常需要做一些额外工作来调整数据结构，以便在图表中展示，或者生成一些 HTML 在表格中呈现。在条件允许的情况下（例如格式化日期），你应该考虑在数据转换和准备阶段完成这些工作——这遵循了尽可能减少工作执行次数的原则（即在转换中执行一次，而不是在每次页面加载时为每个用户执行）。

    如果无法做到这一点，你可以编写一个简单的函数来遍历列中的每个值并进行处理以生成新值，同时保持 Slate 所期望的数据结构。

    

    已复制！
    1`// f_deriveDisplayColumns
2
3var data = {{q_myQuery.results.[0]}}
4data.newColA = _.map(data.primaryKey, (value,key,index) => {
5    // Work to derive new column
6    return newColValue
7});
8return data;`

    这段代码使用了 Lodash ↗ 版本的 _.map() 函数；在这种场景下，你同样可以使用内置的 Array.map()。熟悉 Lodash 的辅助函数通常很有用，它们在处理 JavaScript 对象时极为方便，并且通常实现了复杂的算法，如果从零开始重新实现这些算法，很容易产生 Bug。

    完成该函数后，你无需直接引用 {{q_myQuery}}，而是可以在应用的任何地方引用 {{f_deriveDisplayColumns}}，因为它的结构与原始的查询响应对象相同，只是额外增加了“列”。

    利用自定义库重构代码

    如果你确实需要实现一段相当复杂的 JavaScript 代码，建议将其移至函数编辑器左下角的自定义库中。库可以将纯 JavaScript 函数提供给任何 Slate“函数”使用，从而避免重复多次实现相同的辅助函数。

    由于库是纯 JavaScript，你无法在其内部使用 Handlebars；而是像处理普通 JavaScript 一样，在从 Slate“函数”调用库函数时，可以向其传递参数。随后，可以使用 Handlebars 填充这些参数，或者以动态方式生成这些参数。

    为函数编写文档

    无论是库代码还是函数代码，务必花时间将代码整理清晰，并为那些可能不直观的部分编写文档。你固然可以致力于编写“自文档化”的代码，但养成清晰、简洁地记录逻辑的习惯，对未来的开发者和维护者而言总是一种善意。
