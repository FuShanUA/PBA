时间序列查询计算用量

Foundry 将时间序列数据以高度优化的格式缓存在磁盘上，专为时间序列查询设计。查询这些数据需要消耗计算秒数（compute-seconds）。

时间序列查询通过以下方式消耗计算资源：

  1. 每次查询具有固定的计算秒数消耗。
  2. 每次查询还会根据读取的时间序列数据点数量产生额外的计算秒数度量。

针对较大的时间序列索引进行查询会读取更多数据点。以下各节将详细说明其计算方式。

计算资源度量方式

请注意，在为 Foundry 用量付费时，每次查询会消耗固定的最低计算秒数。默认数量为 4 个计算秒数。这是为查询提供结果所需的基础计算用量。如果您与 Palantir 签有企业级合同，请在进行计算用量计算前联系您的 Palantir 代表。

时间序列数据的存储归入 Foundry 存储范畴进行度量。存储时间序列数据不会消耗任何计算资源；只有对时间序列进行索引以及主动查询数据时才会消耗计算资源。

时间序列查询计算资源仅用于查询存储在 Foundry 中的时间序列数据。时间序列查询通过两种方式消耗计算资源：

  1. 每次查询使用最低数量的计算秒数。在基于用量的 Foundry 实例中，每次查询消耗 4 个计算秒数。
  2. 根据查询本身涉及的数据点数量，查询开销会在固定最低成本的基础上进行扩展。查询期间扫描的数据点数量取决于所查询序列的规模以及所执行逻辑的复杂度，这可能会引发对数据点的大量比较和操作。

以下公式用于从查询中推算计算秒数：

compute-seconds = 4 + points_scanned / 25000

调查时间序列查询用量

《资源管理》应用程序允许您向下钻取至数据集用量信息，这应作为调查 Foundry 平台用量问题的起点。

用户可以使用多种工具在 Foundry 中查询时间序列数据。时间序列查询用量始终归因于各工具生成或修改的资源对象。

  • 在 Quiver 中，时间序列查询将归因于发起查询的个别 Quiver 分析或仪表板。
  • 在 Workshop 中，查询将归属于发起查询的 Workshop 应用。当 Workshop 应用嵌入了查询时间序列的 Quiver 组件时，其用量仍归属于上级 Workshop 应用。
  • 对于使用 FoundryTS 库查询时间序列的转换操作，这些查询的计算用量将归属于编写该转换代码的代码库。

了解用量的主要驱动因素

在按用量合同为 Foundry 付费时，时间序列查询的计算资源消耗受三大主因驱动：

  1. 查询数量

    • 每次查询的最低用量为 4 个计算秒数。这意味着用户执行的查询越多、运行查询的 Quiver 仪表板越多、以及运行 FoundryTS 查询的定时构建任务越多，消耗的计算秒数就越多。

  2. 所查询序列的规模

    • 减少读取的数据点数量通常能降低计算开销和延迟。例如，对包含 10,000,000 个数据点的序列执行聚合查询，其计算消耗将高于对 100,000 个数据点的序列执行相同查询。然而，实际用量取决于具体查询操作。减少读取数据点数量最常用的方法是为每个序列添加时间范围过滤器。

  3. 查询的复杂度

    • 与底层序列进行更复杂交互的查询，将在该序列之上运行更多操作。在执行这些操作时，随着扫描数据点的增加，查询用量也会随之上升。

    有关《查询复杂度》的更多信息和示例，请参阅以下章节。

管理时间序列用量

管理查询总数对于管理时间序列查询的总计算用量至关重要。在 Foundry 中使用时间序列时，请考虑以下实践：

  • 在构建基于时间序列的分析或仪表板时，应考虑每次更新时必须运行的查询总数。还应考虑拆分更新路径，避免在参数变更时所有查询同时更新。
  • 运行 FoundryTS 构建时，务必合理设置调度，避免构建运行频率过高。
  • 对时间序列应用时间范围后，Foundry 的时间序列后端将仅读取该时间范围内的数据点。因此，可以根据查询的范围来优化 Foundry 中的时间序列查询。限制查询的时间范围不产生计算开销；合理选择时间范围能显著减少整体扫描的数据点数，从而降低查询中的计算用量。
  • 为任务选择正确的粒度；并非所有工作流都需要最大粒度的数据。有时，这意味着需要为特定工作流进行预聚合。在其他场景下，则意味着将不同粒度存储在不同的序列中。维护多个最新序列虽然会增加索引计算量，但存储多个序列本身并不产生计算开销。因此，持有多个不同粒度的序列可能比始终查询最高粒度序列的成本更低。

计算用量

要预测时间序列查询的成本，务必清楚所查询序列的大小。

请参考以下示例，其中对包含 100,000 个数据点的序列执行了三次查询，每次查询都扫描了该序列中的所有数据点：

series_size: 100,000 points
minimum_query_usage: 4-compute seconds
points_per_compute_second: 25,000 points
total_queries: 3

 
compute-seconds = num_queries * minimum_query_usage + total_points / points_per_compute_second
compute-seconds = 3 queries * 4 compute-seconds + 100,000 points * 3 queries / 25,000 points-per-second
compute-seconds = 3 * 4 + 300,000 / 25,000
compute-seconds = 24 compute-seconds

查询复杂度示例

随着对查询的时间序列应用更多的嵌套操作，时间序列查询的复杂度也会随之增加。

举例来说，参考以下 FoundryTS 代码，该代码将两个时间序列相加，并以 Pandas dataframe 的形式返回新序列在 1 年时间范围内的所有数据点：

series_1 = N.TimeseriesNode('series_1')
series_2 = N.TimeseriesNode('series_2')
result = F.dsl(program='a+b', return_type=float)([series_1, series_2]).time_range(start='2022-01-01', end='2023-01-01')
result.to_pandas()

此代码将向 Codex 发起如下结构的查询：

{
  id: dsl-fomula
  children: [
    { id: timeseries },
    { id: timeseries }
  ]
}

执行 to_pandas 调用将产生扫描 result 时间序列在所请求的 1 年时间范围内的数据点费用，以及为计算结果所需的两个组成序列中数据点（本例中即为各自的 1 年时间范围）的扫描费用。

现在，请参考以下应用了更多嵌套操作的 FoundryTS 代码。首先，我们定义一个由另外两个序列相加得到的新序列。然后，我们将该序列与其 7 天滚动平均值进行比较，并将结果中一年的数据点加载为 Pandas dataframe：

series_1 = N.TimeseriesNode('series_1')
series_2 = N.TimeseriesNode('series_1')
intermediate_1 = F.dsl(program='a+b', return_type=float)([series_1, series_2])
intermediate_2 = intermediate_1.rolling_aggregate('mean', '7d')
result = F.dsl(program='a-b', return_type=float)([intermediate_1, intermediate_2]).time_range(start='2022-01-01', end='2023-01-01')
result.to_pandas()

此代码将向 Codex 发起如下结构的查询：

{
  id: dsl-fomula
  children: [
    {
      id: dsl-fomula
      children: [
        { id: timeseries },
        { id: timeseries }
      ]
    },
    {
      id: rolling-aggregate
      children: [
        {
          id: dsl-fomula
          children: [
            { id: timeseries },
            { id: timeseries }
          ]
        }
      ]
    }
  ]
}

查询树中的每个节点都会因扫描 1 年时间范围的数据点以生成最终结果而产生费用。
