Connection security

Data Connection establishes outbound TLS connections to external data sources. Not all cipher suites used in TLS connections are supported throughout the platform.

Supported suites

Data Connection supports the following cipher suites for outbound connections, provided they are supported by the underlying Java version. Additional cipher suites may be available depending on your environment's configuration and connection origination point.

Contact Palantir Support with questions about additional cipher suites that may be available based on your environment's configuration and connection origin point.

Verify your external system accepts a supported cipher suite

To verify that your external system accepts a supported cipher suite, run openssl s_client -connect your-system.example.com:<port> -tls1_3 </dev/null (or -tls1_2 for TLS 1.2). The negotiated suite appears on the cipher line of the SSL-Session block in the output. To test a specific suite from the supported list, add -ciphersuites '<cipher_iana_name>' for TLS 1.3 or -cipher '<cipher_openssl_name>' for TLS 1.2. An SSLHandshakeException containing handshake_failure, no cipher match, or protocol is disabled or cipher suites are inappropriate indicates a cipher mismatch, so you should update the cipher configuration on your external system to enable one of the supported suites. You can also run openssl against the same network path your source uses from the source terminal.

Vulnerability scan findings on agent hosts

A vulnerability scanner running on an agent host may report findings against files that belong to earlier versions of the agent software or connector plugins. In almost all cases these are inactive files that are no longer loaded or run. The agent uses only the current version of each component, so a finding against a superseded file does not mean the agent is running vulnerable code.

These files are left behind by normal upgrades. When an agent upgrades to a new version, the version it replaces is kept on the host for up to 30 days and then deleted automatically. Security fixes, including for known Common Vulnerabilities and Exposures (CVEs), are delivered as new versions, and agents upgrade to the latest available version automatically during their upgrade window. Configuring an upgrade window for each agent ensures it is running the latest fixes. To learn more about how Palantir identifies, discloses, and patches vulnerabilities, see Monitoring for vulnerabilities.

To resolve a finding without waiting for the automatic 30-day cleanup, you can delete the affected backups, caches, and archived plugin versions yourself. See Backups and caches for instructions, including how to identify which versions are still in use.
