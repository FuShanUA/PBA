优化索引与架构设计

本页文档重点介绍如何通过调整索引来优化 Postgres 和 ElasticSearch 的性能。我们通常建议在 Ontology 功能的基础上构建 Slate 应用，并使用对象集和操作等功能进行数据读写。

高性能且易于维护的应用离不开合理拆分的数据集和正确的索引。

Postgres 的架构设计在绝大多数方面与 Slate 毫无关联。这些概念在其他资料中已有详尽说明，数据库设计的通用最佳实践不在本文讨论范围内。如果你对不同架构模式或如何选择索引列尚不熟悉，搜索引擎会迅速带你深入这个话题。

相反，我们将重点探讨一些与 Slate 和 Foundry 联系更为紧密的最佳实践。

尽可能将计算逻辑向上游推移

在应用开发流程的每个阶段，都要问自己一个问题：“这项工作放在哪里处理最合适？”并且始终倾向于将其尽量向上游推移。举例来说，如果你正在编写复杂的 JavaScript 函数来聚合数据或提取指标，不妨问问：“我能在查询中完成这个操作吗？”

如果每次加载页面时都要在查询中执行相同的操作，例如使用 SUM() 计算年度总计，或使用 DISTINCT() 生成列表，这时你应该思考：“我能否在派生数据集中处理这些逻辑？”

在 Foundry 中，相比于在查询或用户浏览器中执行的操作，分布式存储和计算的成本更低，因此请尽可能进行预计算。

Postgate (Postgres)

Postgate 封装了 Postgres 或 RDS，以便通过 PostgreSQL 查询对 Foundry 数据集进行简单的只读访问。请注意，应用查询的局限性通常取决于关系型数据库（而非平台其余部分使用的 Spark/HDFS）的特性，尤其是 Postgres 本身的特性。

在编写数据检索查询时，请使用 EXPLAIN ANALYZE 来分析查询性能并找出瓶颈。你可以在这篇《Thoughtbot 博文 ↗》中详细了解 Postgres 如何解析查询以及如何解读 EXPLAIN ANALYZE 的结果。如需了解更多信息，也可以阅读我们关于《优化 Slate 的 Postgres 查询》的详细说明。

你可以使用查询片段构建模块化查询，从而避免代码重复并更好地封装共享逻辑。我们建议结合使用查询片段和函数，在保持查询精简与确保其具备足够的可读性之间取得平衡。

Phonograph (ElasticSearch)

Phonograph 提供了一个读写数据存储，支持使用 ElasticSearch 风格的语法进行查询和聚合，同时还允许对 Foundry 数据集执行 CRUD 操作。有关 Phonograph 实践的更多内容，请参阅《从 Slate 向 Foundry 写回数据》章节。
