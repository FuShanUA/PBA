实时部署计算用量

Foundry 机器学习的实时部署是一种用于模型发布的持久且可扩展的部署形式，可通过 API 端点进行交互。实时部署会持续预留专用的计算资源，以确保能够快速响应传入流量。因此，托管实时部署在处于活跃状态期间会消耗 Foundry 的计算秒数。

以下文档仅涵盖由建模目标支持的实时部署。本文不涵盖基于 JavaScript 函数的部署和《直接部署》。直接部署由计算模块支持，因此遵循相应的原则；然而，对于《基于容器的模型》需要同时满足部署和模型资源的要求，这一限制仍然适用。

在实时运行时，Foundry 机器学习的计算用量会归属于建模目标本身，并在包含该建模目标的项目层级进行汇总。如需深入了解 Foundry 中计算秒数的定义及用量计算公式的来源，请查阅《用量类型》文档。

计算秒数度量

Foundry 机器学习实时部署将其基础设施托管在运行于 Foundry 基于 Pod 的计算集群中的专用“副本”上。每个副本都会分配一定的计算资源，以 vCPU 和 GiB 内存进行计量。每个副本在本地托管模型，并利用计算资源来处理传入请求。

导入到建模目标中的 Foundry 机器学习实时部署在活跃期间会消耗计算秒数，无论其接收到多少传入请求。部署一旦启动即被视为“活跃”，并且直到通过图形界面或 API 将其关闭之前，都会保持“活跃”状态。如果与实时部署关联的建模目标被移至平台回收站，该实时部署将被关闭。另一种情况是，从模型资源而非建模目标启动的《直接部署》，会根据请求量进行自动扩缩容，并且可以配置为在没有使用量时支持零最小副本。如需了解更多关于《建模目标的实时部署与直接部署之间的差异》以及《直接部署的自动扩缩容行为》，请查阅相关文档。

实时部署将消耗的计算秒数取决于三个主要因素：

  • 每个副本的 vCPU 数量
    • 对于实时部署，vCPU 以毫核为单位进行计量，每毫核等于 1/1000 个 vCPU
  • 每个副本的 GiB 内存
  • 每个副本的 GPU 数量
  • 副本数量
    • 部署中的每个副本将具有相同数量的 vCPU 和 GiB 内存

在支付 Foundry 使用费时，默认的用量费率如下：

vCPU / GPU用量费率
vCPU0.2
T4 GPU1.2
A100 GPU1.3
L4 GPU2.1
A10G GPU1.5
V100 GPU3
H100 GPU4.7

这些费率是实时模型根据其在 Foundry 并行计算框架下的配置来使用计算资源的标准。如果您与 Palantir 签有企业合同，请联系 Palantir 支持团队以探讨计算用量的计算方式。

以下公式用于度量 vCPU 计算秒数：

live_deployment_vcpu_compute_seconds = max(vCPUs_per_replica, GiB_RAM_per_replica / 7.5) * num_replicas * live_model_vcpu_usage_rate * time_active_in_seconds

以下公式用于度量 GPU 计算秒数：

live_deployment_gpu_compute_seconds = GPUs_per_replica * num_replicas * live_model_gpu_usage_rate * time_active_in_seconds
基于容器的模型

对于《基于容器的模型》，每个副本都配备一个容器，该容器拥有其专用的 GPU 或 vCPU 资源，具体如容器《模型版本的运行时配置》中所指定。

查看建模目标用量

平台中的所有计算秒数用量均可在《资源管理应用程序》中查看。

部署的计算用量归属于发起该部署的建模目标。任何给定的建模目标都可以同时有多个活跃的实时部署。建模目标的实时部署可以在部署部分下找到。有关示例，请参见下面的屏幕截图。

导致用量增加或减少的因素

实时部署在活跃期间会消耗计算秒数。以下策略有助于控制部署的总体用量：

  • 确保部署配置已针对预期的请求负载进行正确调优。部署应根据预期的最大并发请求数进行调优。如果部署资源不足，将开始对请求返回失败响应。然而，为部署分配过多资源会导致计算秒数的不必要消耗。
    • 建议线上部署管理员在将模型部署到关键业务运营环境之前，对线上部署端点进行压力测试，以确定正确的资源配置。
  • 线上部署将持续运行，直到被明确停止或取消。监控线上部署的使用情况至关重要，可避免在不需要时仍让部署保持运行状态。这种情况在预发布部署中较为常见。
  • 在不更改部署配置文件的情况下增加或减少 API 负载，不会影响其计算资源用量。线上部署会在其资源允许的范围内尽可能多地处理请求，而不会改变所消耗的计算秒数。

配置用量

线上部署的资源用量由其配置文件定义。可以在创建线上部署时设置配置文件。在部署处于活动状态时也可以更改配置文件。部署将自动接收更新后的配置文件，且不会产生停机时间。

管理用量

在通过资源队列管理计算资源的租户中，资源管理管理员可以按照以下步骤查看特定队列中当前正在运行的线上部署：

  1. 打开资源管理应用程序。
  2. 选择资源队列。
  3. 从左侧边栏中选择一个队列。
  4. 在页面中间选择持续计算选项卡。
  5. 选择最右侧的过滤图标，然后选择按线上部署进行过滤。

用量计算示例

示例 1：vCPU 计算

以下示例展示了一个采用默认双副本配置文件的线上部署，在激活 20 秒并使用“低 CPU 最低内存”配置文件时的情况：

resource_config:
    num_replicas: 2
    vcpu_per_replica: 0.5 vCPU
    GiB_RAM_per_replica: 1 GiB
seconds_active: 20 seconds
live_model_vcpu_usage_rate: 0.2

compute seconds = max(vcpu_per_replica, GiB_RAM_per_replica / 7.5) * num_replicas * live_model_vcpu_usage_rate * time_active_in_seconds
                = max(0.5vCPU, 1GiB / 7.5) * 2replicas * 0.2 * 20sec
                = 0.5 * 2 * 0.2 * 20
                = 4 compute-seconds

示例 2：GPU 计算

以下示例展示了一个线上部署的使用率。该线上部署采用默认的双副本配置文件，并在激活 20 秒期间使用 GPU V100 配置文件。

resource_config:
    num_replicas: 2
    gpu_per_replica: 1 V100 GPU
seconds_active: 20 seconds
live_model_gpu_usage_rate: 3

compute seconds = gpu_per_replica * num_replicas * live_model_gpu_usage_rate * time_active_in_seconds
                = 1 * 2replicas * 3 * 20sec
                = 1 * 2 * 3 * 20
                = 120 compute-seconds

示例 3：容器化模型计算

Deployment resource_config:
    num_replicas: 2
    vcpu_per_replica: 0.5 vCPU
    GiB_RAM_per_replica: 1 GiB

Model image resource_config:
    CPU: 4 cores
    Memory: 30 GiB

seconds_active: 20 seconds
live_model_vpcu_usage_rate: 0.2

total compute seconds = (compute seconds deployment resource) + (compute seconds model resource)

compute seconds deployment resource = max(vcpu_per_replica, GiB_RAM_per_replica / 7.5) * num_replicas * live_model_vcpu_usage_rate * time_active_in_seconds
                = 0.5 * 2replicas * 0.2 * 20sec
                = 0.5 * 2replicas * 0.2 * 20sec
                = 4 compute seconds

compute seconds model resources = max(cores, Memory / 7.5) * num_replicas * live_model_vcpu_usage_rate * time_active_in_seconds
compute seconds model resources = 30 / 7.5 * 2 * 0.2 * 20
compute seconds model resources = 32 compute seconds

total compute seconds = 36 compute seconds
