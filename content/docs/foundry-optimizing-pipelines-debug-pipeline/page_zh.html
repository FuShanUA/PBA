调试失败的管道

快速调试和解决管道问题的能力是管道维护工作的核心部分。它确保为重要组织工作流提供支持的生产管道保持可靠且有意义。

此页面提供了一个框架，您可以将其作为标准操作程序（SOP）的基础，当在值班轮班期间收到健康检查失败通知时，作为管道维护人员进行处理。

调试框架概览

始终按顺序提出以下三个问题：

  1. 缓解措施： 我能否尽快缓解这个问题？一些例子包括：
    • 重新构建计划。我们建议重新构建计划而不是单个数据集或失败的任务，因为这将出现在您的计划历史中。计划历史允许您跟踪管道的历史，而不是单个数据集的历史。
    • 如果有太多排队任务，寻找并取消重叠或密集的计划。
    • 如果手动上传失败，回滚到已知稳定版本。
  2. 分类： 什么问题是类别？
    • 我们对问题进行分类的原因是帮助确定根本原因，并确定解决该问题是否需要其他团队的参与。
    • 有关如何思考问题类别并识别它们的更多细节 如下。
  3. 更广泛的影响：这个问题是否可能影响平台的其他部分？
提示

阅读你的管道（Pipeline）文档！也许这个问题之前已经被解决了。或者在缓解过程中可能有警告说明不要做什么。例如，某些构建可能会非常昂贵，并且可能在高峰使用时间影响你环境的性能。这类细节应该被很好地记录下来供整个团队参考。

问题类别分类

在尝试缓解问题后，作为管道维护者，您需要更深入地了解并解决根本原因。在调试过程中对问题进行分类是有帮助的，因为它有助于我们识别根本原因，并且最重要的是，可以帮助您快速确定您是否可以修复该问题，或者是否需要联系其他团队。

问题有三类：

  • 上游问题：与由他人管理的基础设施或工件有关
    • 不在Foundry中：上游数据源的问题
    • 在Foundry中：由所讨论管道上游的数据集/项目引起的问题
  • 平台问题：由于Foundry平台服务未按预期工作而引起的问题。
  • 变更：任何在您负责监控的管道范围内发生的变更。这是最常见的问题类别，通常由用户变更引起。一些例子包括：
    • 代码变更
    • 计划变更
    • 管道中的数据量增加

按问题类别进行问题解决的步骤：

详细来说，上面突出显示的步骤是：

  • 识别：在执行上述步骤时，需要非常精确地确定哪里出了问题。回答以下问题：

    • 这个问题是什么时候开始的？
    • 在管道的哪个步骤出现了故障？
    • 哪些健康检查失败了？
    • 管道中是否有更改导致了这种损坏状态？

    这有助于在需要时与其他团队有效沟通上游问题和平台问题，并减少解决问题的时间。它还可以提高平台的调试技能。

  • 操作：

    • 上游问题：一旦你确认问题是由于上游项目或数据源的延迟、缺失或错误数据造成的，请联系负责上游数据的团队。
      • 提示： 在开始监控之前，记录上游团队的联系方式会很有帮助。这样，值班人员就可以轻松地联系他们。
    • 平台问题：如果你发现Foundry出现了意外行为，并且排除了你的管道中任何用户更改的可能性，请联系你的Palantir代表。尽可能提供有关问题的详细信息，包括任何观察到的更改的细节。有关如何识别这些问题的一些提示，请参见下面的 部分。
    • 更改：在确定监控的管道中有某些变化后，通常可以采取措施进行修复。在某些情况下，可能需要联系做出更改的人以获取更多信息。有关如何识别管道中是否发生变化的帮助，请参见下一节。

  • [可选] 下游用户沟通： 一个在上述图表中未提及的步骤是，当一个问题已被分类并进一步找到根本原因时，可能需要通知管道的下游消费者。这取决于问题的影响范围、持续时间以及管道的使用场景。

  • 临时解决方案：如果另一个团队或用户提供的修复需要一些时间，那么实施中期的临时解决方案可能会很有帮助，以确保你的管道中健康的部分继续为下游消费者运行。具体的临时解决方案取决于问题和用户的需求。例如包括：

    • 通过从计划中移除损坏的数据集或管道分支来隔离问题。
    • 如果这是问题的根本原因，则可以指定另一个 Python 库的版本。这可以通过在 meta.yml 中在库名称旁边指定明确的版本号来完成。

识别你管道中的变化

管道维护者最常见的问题往往是由于您监控的管道内部发生了某些变化而产生的意外后果。这也是作为管道维护者，您最有控制权并能够直接解决问题的类别，而无需依赖其他团队。

更详细地说，需要采取的步骤如下：

  1. 尽可能准确地追踪问题在您的管道中出现的位置。例如，尝试识别计划、数据集、事务、代码更改等。

  2. 将健康的先前运行与当前的故障状态进行比较，以确定发生了哪些变化。 有一个心理检查清单可能会很有帮助。以下是一个示例问题集，以及一些可以帮助您找到答案的示例工具：

    • 比平常慢吗？这是由排队引起的，还是构建实际上计算时间变长了？
    • 文件/数据大小的变化？
    • 代码更改？模式更改？
    • 计划更改？
    • 平台持续事件？

  • 计划指标 哪允许您查看计划的历史运行情况，以及用于比较运行的指标和图表

识别平台问题

如果您不确定问题是什么，可以检查其他作业、构建或相关平台组件中的类似症状，这可能是一个有用的调查路径。

特别是，您应该寻找以下问题的答案：

  • 它是否是 可重复的？
    • 这个问题是否一直发生？
    • 它是否遵循某种模式？例如，每周在周末之后的周一上午9点是否会失败？
  • 什么是范围？
    • 如果您的任务运行缓慢或失败，您是否在其他转换任务中也看到这种情况？或者可能只是其他Python任务中出现这种情况？

使用构建（Builds）应用可以在整个平台上筛选作业历史记录，有助于回答上述问题。
