什么是操作型应用？

在 Palantir，我们经常将某些工作流和应用称为“操作型”的。这是什么意思？

操作型应用用于驱动特定的决策流程，允许用户通过数据回写来记录其决策。传统的仪表板和报告侧重于向用户提供只读的洞察，而操作型应用则赋予了用户采取行动的能力。

我们发现，驱动决策的工作流往往更容易被用户采纳，并对组织绩效产生实质性影响。本页的其余部分将介绍如何利用 Foundry 的应用构建功能，在实践中创建操作型应用。

操作类型

在 Foundry 的本体（Ontology）中，操作类型提供了一种集中且受治理的方式，用于定义组织中的用户如何将数据回写到系统中。配置操作类型时，你可以定义用户需要输入的参数，并灵活地配置表单，向用户呈现丰富的选项，包括创建子区域、添加描述等。

对于谁能执行哪些操作的治理，是通过提交条件来控制的。你可以定义任意复杂程度的规则，确保数据的写入完全符合组织的约束条件。

除了基本的表单录入工作流外，操作类型还支持一系列高级选项：

  • 上传附件允许用户在提交信息时上传图片、PDF 或其他文件。
  • 副作用允许用户发送通知（包括通过电子邮件），或使用Webhooks将数据编排发送到 Foundry 之外的其他系统。
  • 函数支持的操作允许你使用代码定义任意复杂度的操作类型，从而控制对象应如何变化。
  • 触发定时构建允许用户触发数据集成构建任务。

在应用中使用操作类型

在本体中定义操作类型后，在 Foundry 的应用构建工具中使用它变得非常无缝。Workshop 和 Slate 都原生支持将操作表单直接嵌入到面向用户的应用中。

  • 在 Workshop 中，可以配置按钮组微件，让用户轻松地将数据回写到平台中。
  • 在 Slate 中，操作微件会向用户展示操作表单。你可以将其嵌入到对话框中以模拟 Workshop 的体验，也可以根据需要将其灵活编织到你的应用中。
  • 如果你想完全脱离 Foundry 的应用构建框架来开发自定义应用，可以使用 Foundry 的 REST API 搜索并应用操作。

结论

在数据集成和管理方面的投入，使得你组织中的操作人员能够基于数据做出更优决策，并将这些决策记录到系统中以便进一步学习。借助 Foundry 的本体和应用构建能力，你仅需数小时即可创建闭环工作流。
