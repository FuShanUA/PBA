分支与发布流程

本文档的目的是概述在Code Repositories中的Projects中管理的数据管道的推荐最佳分支管理和发布流程。

高层次目标是设计一个流程，以在对新数据功能/数据质量修复的快速迭代和拥有稳定且符合流程的变更管理流程之间取得平衡。

发布范围

在我们深入发布管理流程内部之前，我们需要更准确地定义“我们要发布的到底是什么产品”？为了我们的目的，我们将product定义为作为一个概念单位发布的资产集合。

参考 推荐的项目结构 作为我们在Foundry中构建管道的参考，我们看到管道是由多个不同类型的项目定义的——数据源、转换、本体论和工作流。然而，在大多数情况下，我们不会将每个项目单独作为一个产品进行管理，而是每个项目会将其资源的一个子集定义为以下产品类型之一：

  • 本体论产品：此产品是数据源、管道和本体论项目创建的输出数据集的集合。
  • 用例产品：每个用例项目代表一个独立的用例。每个用例都被定义为一个产品。

我们在这里给产品的定义可以涵盖在多个仓库和其他Foundry资源中管理的代码。

产品分支管理

本节内容涉及我们希望部署的分支策略，用于管理每个产品的发布。

分支命名

建议在产品内的所有仓库中使用相同的分支名称。这将确保下游分支从正确的上游分支读取。

我们推荐遵循的一般分支策略基于一种名为GitFlow ↗的常见实践，但进行了一些修改。主要更改是建议跳过release分支。由于Foundry中的工件部署方式，我们可以在测试后直接将更改从dev合并到master中。

公共分支

master - 这是生产分支。因此，它是一个受保护的分支，只有发布管理器角色可以批准将PR合并到此分支。假设master分支使用生产数据。

dev - dev（也称为开发）分支是一个暂存分支，来源于 master 分支。当您想使用测试数据测试完整功能时，它通常相关。dev分支可以从master（这会通过回退分支自动完成）或从UAT数据源获取。

feature-[X] - 功能分支是特定功能正在开发和测试的分支。此分支来源于dev 分支。这是一个短期分支，意味着一旦该分支被合并到 master，就可以删除。由于该分支来源于dev，它将使用与dev分支相同的数据。 请注意，只要回退分支没有被重新配置，并且功能分支上不存在输入数据集，这一点就一直成立。

其他特殊分支 其他特殊分支

major-release-[X] - 特殊分支，由于源系统的schema发生变化（在特定频率下发生），该分支整合了所有必要的更改。

hot-fix-[X] - 特殊分支用于修复在代码库已升级到新版本时在生产环境中发现的问题。我们假设dev和master是同步的（除非有正在积极测试的功能），因此不太可能需要这样的分支。

分支管理流程

如上所述，一些分支是永久性的，而其他分支是短期的。以下部分描述了分支之间的推荐流程。

  • 在 t0 时，创建 master 和 dev 分支。

  • 在 t1 时，从 dev 创建 feature x 分支。

  • 在 t2 时

    • , feature y 分支是从 dev 创建的

  • 在 t3 时，feature x 被合并到 dev

  • 在 t4 时，dev 被合并到 master, 并且 feature x 分支被删除

  • 在 t5 时，dev 被合并到 feature y 以获取 dev 的最新状态

  • 在 t6 时，特性 y 被合并到 dev 并稍后从 dev 合并到 master 如前所述

除了开发并发布到 dev 和 master 的功能外，还正在开发一个用于主要版本发布的长期分支，例如 major-release-[X] 。

建议持续将 dev 合并到 major-release-[X] ，以便最新功能保持同步。我们建议尽可能保持本体论（Ontology）模式的稳定性，并通过数据源和转换项目中的转换来隐藏后端模式的变化。

仓库升级

仓库升级 是定期提示，用于更新代码仓库中的配置文件。这些更新会应用修复程序或升级语言包依赖项的版本，最佳做法是及时应用这些推荐的升级。

请注意，如果您错过了提示中的“升级”按钮，您也可以在右上角的“...”选项菜单中找到该按钮：

升级过程应与功能开发周期同样对待。这意味着：

  1. master 应通过 dev 进行升级。

  2. 当提示升级 dev 时，将创建一个自动化功能分支。您可以在 CI 检查完成后在此分支上运行构建以测试配置更改。确认升级后转换仍能执行后，将配置更改合并到 dev 中。一旦 dev 升级完成，应使用 PR 将这些更改推送到 master。

  3. 要升级所有打开的功能分支，将 dev 合并回每个分支以包含这些配置升级。

维护窗口

从想法到生产快速迭代的循环在开发实践中很常见，并且对依赖最终用户反馈来确定下一次迭代循环的项目来说至关重要。为了支持这样的流程，我们建议建立一个预先定义好的维护窗口。

例如，一旦所有功能PR都被合并，所有数据集就会在dev分支上构建，以便在周三和周五进行验证。在这几天内，只有在验证过程中发现的问题修复才会被合并到dev中，直到dev最终在周四和周一被合并到master中，即“发布日”。

建议与所有相关方达成这样的节奏，以确保他们在这些天内投入足够的时间推动新功能的进展。

功能测试和最终用户验证

一旦功能开发完成，测试并验证功能结果的正确性和对任何依赖资源的影响至关重要。在代码仓库中，我们考虑输出数据集作为功能结果，应对其进行测试。

根据我们上面定义的分支结构，当构建输出数据集时，它将使用同一功能分支上的输入数据集的数据，或者通过回退逻辑从源分支读取，在简化版本中是master，在其他情况下是dev。

开发人员级别的测试可以使用数据集 健康检查、依赖测试数据集，或手动使用如 Contour 的工具进行。

测试方法

通过预览或SQL辅助进行此测试是不推荐的。预览辅助在执行代码时只使用数据的一个样本，这可能无法揭示所有情况——特别是在涉及一个或多个连接操作的代码中。SQL辅助测试代码的正确性，但不会测试构建的输出，而在某些情况下，构建的输出可能不同。

功能开发完成后，在将新功能代码合并到dev分支之前需要进行审查，并且功能结果需要由最终用户进行验证以确保其正确性和完整性。

我们建议如下面的图示所示，第一种使用拉取请求（Pull Request）审查，第二种使用Foundry问题（Foundry Issues）。

  1. 开发者从他的功能分支创建一个拉取请求到dev分支并添加实现注释。

  2. 开发负责人审查PR，与开发者合作进行必要的更改，并最终将PR合并到dev。

在拉取请求审查期间，请确保检查所有受影响数据集的模式更改。模式应被视为您的数据API，因此任何列名或列类型的变化都被视为对任何使用者的“破坏性”更改。通过创建新列而不是删除或修改旧列来尽可能限制破坏，同时宣布旧列的弃用并提供给数据使用者（通过数据连接识别）更新到新列的说明。在下一个主要版本中，可以在上一个主要版本中弃用的列被删除。

要审查拉取请求中的模式更改，请在 受影响的数据集 选项卡中使用模式差异视图。

  1. 开发负责人在 dev 分支上运行构建以创建功能输出数据集。
  2. 开发人员在输出数据集上创建 Foundry 问题 ，记录任何测试和验证请求，并将问题分配给相关最终用户进行验证。
  3. 最终用户收到该问题，拥有相关数据集和要测试的分支的完整上下文和视图。他们与开发人员协作进行所需的更改，直到问题解决并被标记为可以合并。然后将该问题分配给发布经理。
  4. 发布经理关闭该问题，并将dev合并到master中。
  5. 发布经理删除功能分支。

上述工作流程允许各方之间相对快速的协作周期，而不会丢失上下文。已关闭的问题会保留在平台上，可以作为变更管理过程的审计追踪。

Foundry 问题特性审查工作流程

以下是一个团队可以如何使用Foundry Issues来跟踪来自其他利益相关者的审查请求、围绕新功能进行讨论，并控制和记录将代码合并到master的过程，从而使所有相关方保持知情和最新状态的示例。请将此视为一种方向性指导，而非强制性的规定，并根据需要适应现有的操作和技术流程。

示例工作流

开发人员第一次创建特性审查请求问题时，该问题的状态被设置为Feature Review: Request Review。如果该特性是特定的，可以将问题附加到数据集或特定列上。

与最终用户的协作通过评论部分进行，并通过在Request Change或Ready to Merge之间更改状态以及更改问题的负责人来实现。

问题是存储特性审查工作流的好地方，因为最终用户也可以记录他们的验证过程（用于审计目的）并附加到其他资产的链接。

下图显示了如何在构建输出数据集上创建一个问题。

  1. 在 Foundry 浏览器中，选择输出数据集。

  2. 通过点击报告问题来对整个数据集报告问题。

  3. 如果该功能与特定列有关，您可以使用列级别的报告问题功能，为审核人员提供您所做更改的更好上下文信息。

  4. 按照表单提示填写问题字段。

  5. 问题创建完成后，请确保将标签更改为 Feature Review: Request Review。 （如果您没有这些标签可用，请联系您的 Palantir 代表以获得帮助。）

随着审核流程的继续，包括所有状态变更和验证证据的协作线程都会存储在问题中。
