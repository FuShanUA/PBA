分支与发布流程

本文档旨在概述通过代码仓库中的项目管理的分支管理最佳实践和数据管道发布流程。

其高层目标是设计一套流程，在快速迭代新数据特性与数据质量修复之间取得平衡，同时保持稳定且符合流程规范的变更管理机制。

发布范围

在深入探讨发布管理流程的内部细节之前，我们需要更明确地定义究竟“产品是什么”。就本文档而言，我们将产品定义为作为概念单元发布的一组资产集合。

以《推荐的项目结构》作为我们在 Foundry 中构建管道的参考，可以看到管道由多种不同类型的多个项目定义——数据源、转换、本体和工作流。然而，在大多数情况下，我们不会将每个项目单独作为产品进行管理，而是将每个项目中的部分资源定义为以下产品类型之一：

  • 本体产品：该产品是由数据源、管道和本体项目创建的输出数据集集合。
  • 用例产品：每个用例项目代表一个独立的用例。每个用例都定义为一个产品。

我们在此对产品给出的定义，可以涵盖由多个仓库管理的代码以及其他 Foundry 资源。

产品分支管理

以下部分涉及我们计划部署的分支策略，用于管理每个产品的发布。《了解更多关于分支的一般概念，以及分支在代码仓库和数据集之间的运作方式。》

分支命名

强烈建议在产品内的所有仓库中使用相同的分支名称。这能确保下游分支从正确的上游分支读取数据。

我们推荐遵循的通用分支策略基于名为 GitFlow ↗ 的常见实践并作了一些调整。主要变更是建议跳过 release 分支。由于 Foundry 中工件的部署方式，我们可以在测试后直接将变更从 dev 合并到 master。

常用分支

master - 这是生产分支。因此，它是受保护分支，只有发布管理员角色才能批准将 PR 合并到此分支。假定 master 分支接入的是生产数据。

dev - dev（即开发）分支是预发分支，派生自 master 分支。当您想使用测试数据测试完整功能时，它尤为重要。dev 分支的数据可以来自 master（通过回退分支自动实现）或 UAT 数据源。

feature-[X] - 功能分支用于开发和测试特定功能。该分支派生自 dev 分支。这是一个短生命周期分支，意味着一旦合并到 master 即可删除。由于该分支派生自 dev ，它将接入与 dev 分支相同的数据。 请注意，只要不重新配置回退分支，且功能分支上不存在输入数据集，此情况就始终成立。

其他特殊分支

major-release-[X] - 特殊分支，用于整合因源系统模式变更（以特定周期发生）而需要进行的所有变更。

hot-fix-[X] - 特殊分支，用于修复在生产环境中发现的问题，同时代码库已推进至新版本。由于我们假定 dev 和 master 保持同步（除非有正在测试的功能），因此不太可能需要此类分支。

分支管理流程

如前所述，部分分支是长期存在的，而另一些则是临时的。以下章节介绍了推荐的分支间工作流。

  • 在 t0 时，创建 master 和 dev 分支。

  • 在 t1 时，从 dev 分支创建 feature x 分支。

  • 在 t2 时，从 dev 分支创建 feature y 分支。

  • 在 t3 时，将 feature x 合并到 dev 分支。

  • 在 t4 时，将 dev 合并到 master，并删除 feature x 分支。

  • 在 t5 时，将 dev 合并到 feature y 分支，以同步 dev 的最新状态。

  • 在 t6 时，将 feature y 合并到 dev 分支，随后按照前面的方式，从 dev 合并到 master。

除了将开发的功能发布到 dev 和 master 分支外，还可以为重大版本发布开发一个长期分支，例如 major-release-[X]。

建议持续将 dev 合并到 major-release-[X] 中，以保持最新功能的同步。我们建议尽可能保持本体结构稳定，并通过数据源和转换项目中的转换来隐藏后端结构变更。

代码仓库升级

代码仓库升级 是定期提示用户更新代码仓库中配置文件的操作。这些更新会应用修复程序或升级语言包依赖的版本，最佳实践是及时应用这些推荐的升级。

请注意，如果错过了提示中的“升级”按钮，该按钮也可在右上角“...”选项菜单下的任何分支中找到：

升级过程应与功能开发周期同等对待。这意味着：

  1. master 应通过 dev 进行升级。

  2. 当提示升级 dev 时，系统会自动创建一个功能分支。待 CI 检查完成后，您可以在该分支上运行构建以测试配置变更。确认升级后转换操作仍能正常执行，即可将配置变更合并到 dev 中。一旦 dev 完成升级，应使用 PR 将这些变更推送到 master。

  3. 要升级所有未合并的功能分支，请将 dev 合并回各个分支以包含这些配置升级。

维护窗口

在开发实践中，从创意到生产的快速迭代循环十分常见，对于依赖终端用户反馈来决定下一轮迭代的项目而言，这更是项目成功的关键。为了支持这一流程，我们建议设立预定的维护窗口。

例如，当所有功能的 PR 都合并后，所有数据集会在 dev 分支上进行构建，以便在周三和周五验证结果。在这几天里，只有验证期间发现问题的修复程序才会合并到 dev 中，直到 dev 最终在周四和周一（即“发布日”）合并到 master 中。

建议与所有相关方就这种节奏达成一致，以确保他们在这几天内能投入足够的时间来推进新功能的落地。

功能测试与终端用户验证

功能开发完成后，对功能输出的正确性及其对任何相关依赖资源的影响进行测试和验证至关重要。在代码仓库中，我们将输出数据集视为需要进行测试的功能输出。

基于我们上文定义的分支结构，当我们构建输出数据集时，它要么会使用该功能同分支上的输入数据集数据，要么通过回退逻辑从源分支读取数据——在简化版中是 master 分支，在其他情况下则是 dev 分支。

开发人员级别的测试可以通过数据集《健康检查》、依赖测试数据集来完成，或者使用 Contour 等工具手动进行。

测试方法论

强烈建议不要通过预览或 SQL 助手进行此类测试。预览助手在执行代码时仅使用数据样本，可能无法揭示所有情况——尤其是涉及一个或多个连接操作的代码。SQL 助手仅测试代码的正确性，而不测试构建输出的正确性，而在某些情况下两者可能存在差异。

功能开发完成后，新功能代码在合并入 dev 分支前需要进行代码审查，同时功能的最终结果需由最终用户验证其正确性和完整性。

我们建议将拉取请求审查用于前者，将 Foundry 工单用于后者，结合两者使用，具体说明如下图所示。

  1. 开发人员从其功能分支向 dev 分支发起拉取请求，并添加实现说明。

  2. 开发主管审查拉取请求，与开发人员协作完成必要的修改，最后将拉取请求合并至 dev 分支。

在拉取请求审查期间，务必检查任何受影响数据集的模式变更。模式应视为你的数据 API，因此对列名或列类型的任何更改均视为对使用者的“破坏性”更改。尽量限制破坏性更改，建议创建新列而非删除或修改旧列，同时声明旧列弃用，并向数据使用者（通过 Data Connection 识别）提供更新至新列的指引。在下一个主要版本发布时，可以移除上一个主要版本中声明弃用的列。

要审查拉取请求中的模式更改，请在受影响数据集选项卡中使用模式差异视图。

  1. 开发主管在 dev 分支运行构建，生成功能输出数据集。
  2. 开发人员在输出数据集上创建 Foundry 工单，记录所有测试与验证请求，并将工单分配给相关最终用户进行验证。
  3. 最终用户收到工单后，可获取完整上下文并查看待测试的相关数据集与分支。他们与开发人员协作完成所需修改，直至问题解决并标记为“准备合并”。随后，该工单将分配给发布经理。
  4. 发布经理关闭工单，并将 dev 分支合并至 master 分支。
  5. 发布经理删除功能分支。

上述工作流允许各方之间进行相对快速的协作周期，同时不丢失上下文。已关闭的工单会保留在平台上，可充当变更管理流程的审计追踪记录。

Foundry 工单功能审查工作流

以下示例展示了团队如何利用 Foundry 工单来跟踪来自其他利益相关者的审查请求，开展围绕新功能的讨论，并控制和记录代码合并至 master 分支的流程，确保所有相关方保持知情与同步。请将此视为指导性建议而非强制性规范，可根据现有的操作和技术流程进行适当调整。

示例工作流

开发人员首次创建功能审查请求工单时，该工单会被赋予 Feature Review: Request Review 状态。该工单可附加至整个数据集，若功能针对特定列，也可附加至特定列。

与最终用户的协作通过评论区完成，可在 Request Change 或 Ready to Merge 之间更改状态，并变更工单的处理人。

工单是存储功能审查工作流的理想场所，最终用户还可以在此记录其验证过程（用于审计目的）并附加指向其他资产的链接。

下图展示了如何在构建输出数据集上创建工单。

  1. 在 Foundry Explorer 中，选择输出数据集。

  2. 点击报告问题，对整个数据集报告工单。

  3. 如果功能涉及特定列，可以使用列级别的报告问题功能，为审查者提供有关所做更改的更好上下文。

  4. 根据表单提示完成工单字段填写。

  5. 创建工单后，务必将标签更改为 Feature Review: Request Review。（如果没有可用的标签，请联系您的 Palantir 代表寻求协助。）

随着审查流程的推进，包含所有状态变更与验证证据的协作记录都会保存在该 issue 中。
