常见问题解答

本页面探讨了在使用 Foundry Forms 时可供参考的一些常见问题与调试步骤。

除了 Foundry Forms，我还能用什么？

对于 Foundry 上的数据录入或回写工作流，Foundry Forms 已不再是推荐方案。建议改用 Foundry Ontology 构建用户输入工作流，将相关数据结构表示为对象类型，并通过 Actions 配置回写交互。

Actions 针对新增、编辑和删除数据相关的权限提供了更稳健、更精细的控制能力，包括对受限视图的遵循以及复杂条件权限的配置。此外，Actions 可由 Foundry Functions 提供支撑，从而实现表现力更丰富的回写逻辑。

除了 Actions 配置中内置的表单构建器外，Workshop 和 Slate 也原生支持 Actions，借助于全套应用构建工具，可以在这两个组件中打造复杂的数据录入用户体验。

Actions 还会为 Foundry API 自动生成 API 绑定，便于外部应用将数据写入 Foundry；同时支持与 Webhook 交互，通过 Webhook，Actions 能够将数据写入外部数据系统或触发其他下游效应。

目前尚无废弃 Foundry Forms 的时间表，现有的 Foundry Forms 实现将继续获得支持。强烈建议新工作流采用基于 Ontology 的方案，Foundry Forms 预计不会再获得新功能、增强特性或与非安全相关的修复。

如何存储多个值？

多种字段类型允许填写者选择多个值（例如 checkboxes、dropdown 和 list）。使用 Fusion 表格时，这些值会以数组形式自动存储在单个单元格中。而使用对象类型时，则需进行一些额外配置：

  • 在源数据集和回写数据集的 Schema 中，相关列的类型必须设为 Array<X>，其中 X 是基础类型，如 String 或 Integer。
  • 在 Ontology 配置中，相关属性必须具备相同的基础类型 X，并勾选 Allow multiple values 复选框。

如何将多个值转换为目标数据集中的多行记录？

配置字段以存储多个值后，你可以使用 explode ↗ 函数将每个值拆分至单独的行中。此函数可用于数据集的 SQL/Python 转换或 Contour 表达式中。

如何生成由其他字段拼接而成的字段？

使用模板字段进行如下配置：

- uri: display.Text1
  name: Text 1
  type: Text
  tag: X
- uri: display.Text2
  name: Text 2
  type: Text
  tag: Y
- uri: display.Template
  name: Text 1_Text 2
  type: Template
  options:
    inputs:
      a: X
      b: Y
    template: '{{a}}_{{b}}'

如何调试提交失败的问题？

在保存基于对象的表单时，你可能会看到此错误：Submitting failed! Please try again or contact your Palantir support.。

若要进一步调试，请按照以下步骤操作：

  1. 右键单击页面并选择检查。然后，打开控制台选项卡。
  2. 找到以 Submitting failed! Reason ... 开头的消息。
  3. 展开名为 e 和 body 的消息组。
    • 最常见的 errorName 是 FormEntries:PhonographEntryParseError，通常由表单与数据集 Schema 或 Ontology 配置不一致引发。
  4. 展开名为 parameters 的消息组。
    • 在 FormEntries:PhonographEntryParseError 这个具体示例中，此处会指出引发问题的字段（PropertyId）。

举个例子，如果 String 类型的列或属性与允许多值的字段进行了绑定，用户就需要将类型更改为 Array<String>，或者将该字段更新为仅允许单个值。

是否支持移动端提交？

Forms 采用响应式 Web 设计，可在移动设备上运行；但 Forms 并非专为移动端设计，因此不提供官方支持。
