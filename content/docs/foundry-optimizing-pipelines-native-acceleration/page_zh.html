原生加速

您可以通过启用 Velox ↗ 的原生加速来提升 Spark 的性能。

原生加速是一种利用底层硬件优化来提升批处理作业性能的技术。这种性能提升的原理是将计算任务从 Java 虚拟机（JVM）语言转移到原生语言（如 C++），这些原生语言会被编译为机器码并直接在硬件上运行。通过使用特定平台的功能，原生加速旨在大幅缩短处理大规模数据工作负载所需的时间，从而加快作业执行速度并提高资源利用率。

原生加速可用于 Python 转换 和 管道构建器 (Pipeline Builder)。

构建分析

您可以在 Spark 详情 页面对原生加速构建进行基础分析。在查询计划选项卡下，选择物理计划；您将看到类似以下内容：

== Physical Plan ==
AdaptiveSparkPlan
+- == Final Plan ==
   Execute InsertIntoHadoopFsRelationCommand
   +- WriteFiles
      +- CollectMetrics
         +- VeloxColumnarToRowExec
            +- ^ ProjectExecTransformer
               +- ^ InputIteratorTransformer
                  +- ^ InputAdapter
                     +- ^ RowToVeloxColumnar
                        +- ^ HashAggregate
                           +- ^ VeloxColumnarToRowExec
                              +- ^ AQEShuffleRead
                                 +- ^ ShuffleQueryStage
                                    +- ColumnarExchange
                                       +- ^ ProjectExecTransformer
                                          +- ^ RowToVeloxColumnar
                                             +- * ColumnarToRow
                                                +- BatchScan parquet

虽然这与常规的 Spark 查询计划大体相似，但您会注意到一些关键差异。原本的 ProjectExec 节点被替换为了 ProjectExecTransformer。这意味着该操作将在 Velox 查询引擎中原生执行。查询计划中所有卸载的节点都会在树结构中用 ^ 符号标记。原生执行块被 RowToVeloxColumnar 和 VeloxColumnarToRowExec 夹在中间。这些节点负责将 Spark 数据集转换为 Arrow DataFrame，反之亦然。这种序列化/反序列化操作会带来极大的开销。

通常有两种模式表明原生加速性能不佳：

  • 以 ^ 符号标记的原生执行节点比例很低。
  • 大量 RowToVeloxColumnar 和 VeloxColumnarToRowExec 节点导致高昂的序列化开销。

如果性能未达预期，这项分析将非常有用。对管道进行微小的改动，可能会对卸载的计算量产生巨大影响。您可以使用检查点 等功能，将能够全部原生执行的构建片段手动组合在一起。

原生加速的实现与架构

Foundry 的原生加速实现基于 Apache Gluten ↗ 项目。Foundry 原生加速利用 Velox ↗ 查询引擎在运行时加速 Spark 作业。Velox 使用 C++ 编写，并且明确以数据库加速为设计初衷 ↗，提供了开发者 API 以便在 Arrow DataFrame ↗ 上执行算子级别的操作。Gluten 提供了必要的“粘合剂”，将 Spark 运行时与 Velox 绑定在一起。

在此设置中，管道首先像常规构建（无原生加速的构建）一样生成 Spark 查询计划。随后，会对该计划应用额外的优化规则，以识别查询中是否有一部分可以通过 Velox 运行。此决策取决于 Velox 是否具有等效的实现，以及 Gluten 中是否存在该实现的映射。查询可以在算子级别进行卸载：这大致对应于 SELECT、FILTER 或 JOIN 等 SQL 语句。在此阶段，查询计划中任何可以卸载的部分都会被标记。

规划步骤完成后，查询将通过常规的 Spark 引擎执行。这意味着所有的任务调度、执行器编排和生命周期管理都会照常进行。当执行器运行到查询计划中被标记为原生执行的部分时，差异就体现出来了。此时，不再调用 Spark 中的默认实现，而是调用 Velox 的实现。

这种架构极具优势，因为它支持并非所有计算都能通过 Velox 完成的查询。这是因为卸载决策是在算子级别而非整个计划级别做出的。支持的算子数量在不断增加，但用户编写的代码（如 UDFs）永远无法被卸载，因为不存在对应的原生实现。

为什么原生加速更快？

Spark 采用 Scala 编写，这是一种 JVM 语言，并包含诸多优化机制，例如代码生成 ↗ 以提升其性能。此外，JVM 本身也内置了优化机制，例如 C2 编译器 ↗，旨在尽可能利用平台特定的特性。然而，C++ 等原生语言在性能上依然更具优势，主要基于以下三个基本原因：

  • 编译期优化：Java 和 Scala 会编译成字节码，随后交由 JVM 执行；而 C++ 等原生语言则直接编译为机器码。这使得 C++ 编译器能够在编译期实施大范围优化，显著降低运行期开销。相比之下，JVM 语言依赖即时编译，该过程发生在程序执行期间，由于必须在编译耗时与快速启动之间取得平衡，往往难以达到同等水平的优化程度。
  • 无垃圾回收（GC）：在 C++ 中，内存管理由开发者手动控制，从而免去了垃圾回收带来的额外开销。而在 JVM 语言中，GC 过程可能引发不可预测的停顿与系统开销，这极易影响程序性能，尤其是在内存密集型应用中表现尤为明显。
  • 直接访问硬件及支持向量化 API：C++ 允许直接访问硬件特性与底层系统资源，开发者得以充分利用平台专属优化及向量化 API，如 SSE、AVX 等 SIMD（单指令多数据）指令。这种直接的底层访问能力带来了精细化的性能调优，而在 JVM 语言中，由于抽象层的存在阻碍了同等程度的硬件交互，这类优化往往难以轻松实现。

原生加速的内存配置考量

在 Foundry 中以原生加速模式运行 Spark，所需的配置与常规批处理流水线略有不同。Spark 支持使用堆外内存 ↗ 来执行部分操作。堆外内存不受 JVM 管理，规避了 GC 开销，从而带来更优的性能。默认情况下，我们在 Foundry 中并不启用堆外内存，因为这样做会为流水线引入额外的维护成本。对于原生加速而言，开启堆外内存是必要条件，因为经 Velox 修改的 DataFrame 必须位于堆外，才能供原生进程访问。除了 Velox 数据转换之外，Foundry 的其他所有操作仍需充足的堆内内存（例如，编排、调度及构建管理代码依然在 JVM 中运行），但在理想状态下，大部分计算任务此时应在堆外完成。将流水线配置为使用原生加速，会带来额外的维护成本，主要体现在需要平衡堆内与堆外内存。
