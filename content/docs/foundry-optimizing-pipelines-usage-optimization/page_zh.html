Foundry 使用量优化

Foundry 使用量优化最佳实践

本指南旨在提供优化 Foundry 使用量的方法和最佳实践。文档首先介绍 Foundry 使用量的衡量方式，其次说明如何识别使用浪费并进行流水线优化。其中的一般性建议同样适用于项目经理或平台管理员，帮助他们监控和优化组织层面的使用消耗。

除了此处列出的最佳实践外，Linter 会检查 Foundry 的运行状态以发现反模式，并提供针对性的改进建议，帮助提升资源状态。你可以评估并采纳这些建议来降低成本、优化本体，同时提升流水线的稳定性和鲁棒性。

何时进行优化

在考虑如何将这些最佳实践应用到工作流中时，务必避免过早优化流水线或工作流这一常见误区。用户应避免过早优化，也不要期望一种放之四海而皆准的优化策略。

建议通过以下思维步骤来验证优化方案的合理性：

  • 工作流是否已经完成并能正常运行？如果尚未完成，需警惕过早优化可能会影响工作流的功能。
  • 本次优化是否有明确的目标（例如数据可视化耗时或成本），并将其作为衡量后续决策成功与否的指标？
  • 针对上述目标，是否已经明确了具体的瓶颈所在？

如果对上述问题尚无明确答案，说明还需要做更多准备工作，才能开展有效的优化。

基于上述原则，以下是优化工作的正反示例：

  • [正面示例] 某正常运行流水线的成本为 X 美元，工程师被询问是否可以降低该项成本。资源管理应用显示大部分成本源自计算资源消耗。通过查看构建频率，工程师发现该调度每天都会运行，尽管并非每天都有人使用。将调度调整为较低频率后，资源消耗量降低了约 28%。优化后，该流水线不再是平台上最昂贵的流水线，工程师便可以将精力转向处理下一个瓶颈。
  • [反面示例] 工程师被要求设计一套高级策略来优化某流水线的存储成本，该任务耗费了大量工时。策略实施后，存储成本下降了，但整体基础设施账单却没有变化。原因何在？在以节约成本为目标的情况下，对“存储”这一瓶颈的判断出现了失误。尽管数据集的存储体积确实过大，但相对于计算资源而言，存储成本较低，这意味着该调整对工作流的总使用成本影响微乎其微。

一般性最佳实践

优化 Foundry 使用量的一些通用实践包括：

  • 建立项目机制，以便在资源管理应用中追踪使用量
  • 充分利用资源队列
  • 尽可能使用增量流水线
  • 管理调度计划，确保其运行能够满足而不超出组织的需求
  • 优化 Spark 使用方式（视你对具体实现的熟悉程度而定）

第一步：了解 Foundry 使用量的组成部分

Foundry 使用量由三个部分组成：Foundry 计算量、本体数据量和 Foundry 存储量。

大多数账户均采用这种三维模型；然而，部分账户的使用量衡量标准可能有所不同。请与 Palantir 代表核实相关条款以确认具体情况。

1. Foundry 计算量

Foundry 计算量由数据集成与分析工具驱动。Foundry 计算量主要分为三种类型：批处理计算、交互式计算和流式计算。

批量计算代表以“批量”方式运行的查询或作业，即按特定调度周期或临时触发在后台运行。批量计算作业在不运行时不会消耗任何计算资源。批量计算的几个示例包括所有转换作业、通过 Contour 构建的数据集、代码工作簿、数据健康检查以及同步到本体/索引存储的操作。

交互式计算代表实时执行的查询，通常作为交互式用户会话的一部分。为了向用户提供快速响应，交互式计算系统会维持常驻待机的计算资源，这意味着交互式查询往往比批量计算消耗更多的计算资源。交互式计算的主要形式是 Contour —— Contour 仪表板、分析以及嵌入的 Contour 图表都是交互式计算的示例。

流式计算代表持续接收消息并使用任意逻辑处理消息的常驻处理作业。流式计算的计算量是根据流准备好接收消息的时间长度来衡量的；与批量计算和交互式计算相比，流式计算的成本最高。流式计算的示例包括流式转换和 管道构建器 流。

批量计算、交互式计算和流式计算的计算资源使用量受以下因素驱动：

  • 数据逻辑：应用于数据的逻辑是影响 Foundry 使用量的最大因素之一，也是用户最能施加影响的因素。
  • 转换速度：转换速度是通过并行化作业来实现的。Foundry 可以扩展到数千台并发机器，以快速应对针对海量数据集的大规模计算。然而，更快的结果和并行化作业可能会引入额外开销和低效问题，从而导致使用量增加。
  • 计算类型：不同类型的计算在处理相同数据时所需的计算资源量不同。例如，批量处理通常比流式数据处理消耗更少的计算资源，因为批量处理仅在实际运行期间占用计算资源，而流式处理则是常驻开启的。
  • 数据规模与类型：数据量越大，处理所需的计算资源就越多。
  • 数据时效性：计算新结果的频率越高、调度的转换作业越多，执行所需的计算资源就越多。

2. 本体数据量

Foundry 使用量的第二个组成部分是 本体数据量。本体层是 Foundry 的独特能力之一。本体是企业数据与组织关注对象之间的转换层。本体是对数据世界的分类，它允许组织以更具体的形式看待其数据，例如“飞机”或“汽车”，而不是描述它们的大量行和列的聚合。如果您对本体的概念不太熟悉，可以从文档中了解更多信息。

本体数据量受以下因素驱动：

  • 对象数量：Foundry 的本体层可以扩展到每种对象类型数十亿个对象。某种对象类型的本体数据量总量与对象总数直接相关。
  • 对象大小：本体对象各自可以具有任意类型的数百个属性。具有更多或更大属性（例如长文本或图像）的对象会占用更多的本体数据量。
  • 对象间链接数量：与其他对象存在大量链接的对象，由于其链接元数据的大小，可能会占用更多的本体数据量。

3. Foundry 存储

Foundry 存储用于衡量存储在 Foundry 非本体转换层中的通用数据，有时也被称为“冷存储”。

数据集分支和先前的事务（及视图）会影响单个数据集占用的磁盘空间大小。Foundry 提供了多种保留策略，帮助您保持 Foundry 实例的精简。当文件通过 DELETE 事务从视图中移除时，这些文件并不会从底层存储中删除，因此会继续产生存储成本。降低存储容量的唯一方法是使用保留策略来清理不必要的事务。提交 DELETE 事务或更新分支并不会减少已使用的存储空间。

第二步：了解如何追踪 Foundry 使用量

《资源管理》应用程序为组织提供了可视性与透明度，帮助其了解 Foundry 的使用消耗情况。该应用程序支持用户查看按各类 Foundry 使用类型（Foundry 计算、本体存储卷和 Foundry 存储）划分的用量明细。用户可以按资源（项目）、来源（应用程序）以及用户来查看使用情况。

在试图找出可优化的 Foundry 用量时，首先应查看《资源管理》应用程序。借此，您可以发现哪些资源消耗了最多的计算资源，并定位瓶颈所在。在此基础上，您可以参考《Foundry 用量优化最佳实践》来寻找潜在的低用量方案，但请始终牢记——聚焦瓶颈。

步骤 3：设置项目以追踪 Foundry 用量

项目与《资源管理》应用程序

如上所述，在《资源管理》（RMA）中，计算资源默认在项目级别进行管理；在 RMA 中，我们可以看到按项目计量的 Foundry 计算、本体存储卷和 Foundry 存储指标。因此，为了有效跟踪整个数据管道中的使用指标，合理设置项目至关重要。合理的设置使数据工程师或平台管理员能够监控管道关键阶段的使用指标，从而识别可优化的环节。若设置不当，将无法识别数据管道中资源消耗大且计算成本高的部分。

用于追踪 Foundry 用量的推荐项目结构

应利用 Foundry 项目来构建结构合理的数据管道。《推荐的项目与团队结构》文档中深入介绍了项目设置及管道阶段的最佳实践。确保项目遵循推荐的结构——从数据源导入原始数据到实际的工作流——能够让用户分析管道关键阶段的计算和存储指标。

权限

在寻求降低用量的策略时，管理员应考虑团队中应有哪些人员有权创建项目和资源。将此权限限制在少数掌握设置最佳实践的人员范围内，可以减少项目和资源的泛滥，从而削减不必要的存储和计算开销。若允许任何用户在平台上创建项目，极有可能导致创建出的项目不符合追踪用量的推荐结构，进而产生昂贵且不必要的管道，最终推高用量。各组织可根据平台用户数量和数据访问限制，采取不同的项目创建权限管理方式；建议制定一套权限审批流程，并对拥有权限的人员进行项目结构规范的培训，以实现对这类资源用量的妥善监控。

步骤 4：资源队列

《资源管理》应用程序中有一项帮助您控制 Foundry 支出的核心功能，即资源队列。为了限制关联至特定项目或多个项目的计算能力，您可以将项目打包至队列中。每个队列将被分配特定的资源限额，用于定义同时使用的最大 vCPU 数量。例如，您可以为某个队列分配 XXX 个 vCPU，这便是该队列中已分配项目在任意时刻运行的最大 vCPU 数量。这能确保您清晰掌握每个项目将消耗的用量。

步骤 5：增量管道

增量管道通常用于处理随时间发生显著变化的输入数据集。通过避免对未发生变更的所有数据行或文件执行不必要的计算，增量管道能够降低端到端延迟，同时将计算成本降至最低。实现这一目标的方法在于理解 SNAPSHOT 与 APPEND 事务之间的区别。

快照事务

默认的事务类型为 SNAPSHOT。快照事务被视为对数据集中所有数据的全量替换。这意味着，当你打开一个最新事务类型为 SNAPSHOT 的数据集时，预览结果将仅包含该最新快照事务中接收到的数据。在数据转换或 Contour 分析中读取该数据集时情况也是如此——你只能看到来自最新事务的数据。

快照之所以是默认的事务类型，是因为它最容易上手——每次同步运行时，系统会下载数据库查询返回的所有数据，并创建一个快照事务，从而有效替换数据集此前的所有数据。既往事务中的文件依然可在数据集的历史版本中查看，但数据预览及下游数据转换默认将访问新事务。

当然，快照事务虽然简单易用，但每次都全量复制数据可能效率极低。一种潜在的提效方案是改用 APPEND 事务类型。

追加事务

当数据集由追加事务构成时，其默认视图是所有事务的汇总。这意味着在使用 APPEND 事务类型时，无需重新同步已上传的文件——只有新增数据会被同步至 Foundry 中。这能有效减少 Foundry 的存储占用，因为每个事务仅包含新增文件，而非源系统中所有可用数据的快照。

第 6 步：管理计划任务

优化 Foundry 使用的另一种方式是通过计划任务。在调度工具中配置的计划任务用于定期运行构建，从而保持数据在 Foundry 中的持续流转。

计划任务的设置应符合组织需求，但为了优化 Foundry 的使用，务必高效设置计划任务，避免不必要的运行。例如，如果你将数据集的计划设置为每天早上 8 点刷新，但组织实际上并不需要每天早上 8 点获取更新数据——这就造成了 Foundry 资源的浪费。相反，你应根据实际所需的更新频率来设置计划，比如每隔一天在早上 8 点刷新。仅此一项调整就能将 Foundry 的使用量减半。

在考量优化计划任务的最佳实践时，需牢记两大核心原则：一是消除重复计划，二是消除不必要的计划。

消除重复计划任务

要识别冗余的计划任务，首先进入数据血缘关系应用，将节点颜色更改为 Schedule Count。如果某些节点关联了多个计划任务，选中该节点并查看管理计划工具。在此处，你可以查看关联的计划任务，确定其归属者，并判断是否可以进行整合。

最佳实践是确保管道中的每个数据集仅关联一个计划构建。由两个不同计划任务构建同一数据集会导致排队，拖慢两个计划的执行进度，并造成批处理计算资源的浪费。

减少冗余计划任务的另一项最佳实践是避免全量构建，转而使用连接构建。以一个本体管道为例，该管道包含原始数据集、清洗后数据集、数据转换，最终生成本体。你无需设置三个计划任务（分别运行原始数据集、清洗后数据集和转换后数据集），只需设置一个计划任务，以原始数据集为触发器，以本体数据集为目标即可。

消除不必要的计划任务

要识别不必要的计划任务，进入数据血缘关系应用，按 Time since last built 对节点进行着色。借此可查看哪些数据的更新最为频繁，并判断该频率对组织而言是否为最优。

计划任务的频率与时机是优化使用的关键因素。你的组织究竟需要多高的数据更新频率？

  • 如果调度计划设为每日执行，请考虑周末是否需要更新数据。若不需要，将调度改为周一至周五更新，可为该管道或数据集节省近 30% 的 Foundry 批处理计算资源消耗。
  • 如果调度计划设为每日运行三次，请评估组织是否确实每天使用三次刷新后的数据，还是夜间刷新一次即可满足需求？
  • 是否必须使用基于时间的触发器，还是基于条件的触发器也能胜任？是需要调度管道每天凌晨 2 点刷新，还是可以在特定输入数据集刷新时再触发？相比基于时间的触发器，基于事件的触发器往往效率更高，也能节省更多资源消耗。

此外，最好不要将所有构建任务集中在同一时间调度，这有助于提高调试效率并降低计算资源消耗。在设定频率和时间时，关键是要回归组织实际需求，确保刷新频率与组织需求相符，而非盲目超出实际需要。

最后，在设置调度时务必查看高级选项。建议启用失败时中止构建功能，以减少批处理计算资源的浪费。还可以将失败作业的最大重试次数设为三次或更少，而非默认的最大五次。此外，建议将两次重建之间的间隔时间设置为至少一到三分钟，以便为导致失败的短暂故障留出自行恢复的时间。

第 7 步：优化 Spark

Spark 是一个开源的分布式集群计算框架，专为快速处理大规模数据和分析而设计。它通过将任务拆分给不同系统并行处理，免去了线性等待全部完成的时间，从而让计算机处理海量数据或分析变得更快、更便捷。

首先需要声明，作为优化方面的通用最佳实践，我们建议仅在明确识别到特定瓶颈时才手动调整 Spark 配置，而非在整个平台上将其作为常规操作。这是因为 Foundry 会不断为转换任务新增并启用优化功能，这些功能在手动覆盖的情况下是不生效的。一个简单的例子是 Spark 3 引入的动态分配机制。过去，手动设置转换任务的执行器数量至关重要，而如今该数量会在执行时自动调整，从而避免资源浪费。

优化 Spark 的操作仅应由熟悉 Spark 概念并深刻理解其在管道中应用机制的用户执行。

优化 Spark 的第一步是审查并理解 Spark 配置文件。Spark 配置文件是 Foundry 用于配置分布式计算资源（如驱动器和执行器）并为其分配合适数量 CPU 核心与内存的配置集合。大多数情况下，我们建议使用自动配置文件，而非尝试手动微调；不过有时您可能会发现系统使用了不必要的大型驱动器。

您可以利用数据血缘关系上的 Spark 使用情况着色标记，来识别可能使用了较高配置文件的数据集。

在开始优化之前，请先了解以下核心的 Spark 概念与术语。

  • 驱动器核心数：控制分配给 Spark 驱动器的 CPU 核心数量。
  • 驱动器内存（JVM 堆内内存，非堆外内存）：控制分配给 Spark 驱动器的内存大小。
  • 执行器核心数：控制分配给每个 Spark 执行器的 CPU 核心数量，进而决定每个执行器可并发运行的任务数。
  • 执行器内存：控制分配给每个 Spark 执行器的内存大小，该内存由执行器运行的所有任务共享。
  • 执行器数量：控制运行作业所需的执行器数量。

除了 Spark 配置文件外，以下最佳实践可在旨在减少 Foundry 使用量的 Spark 优化工作中为您提供切入点。

  • 在代码前期尽早减少数据量，以辅助 Spark 的查询优化器。
    • 顺序很重要 - 重新调整操作顺序，将数据过滤操作前置到早期阶段执行。
    • 丢弃无用数据 - 如果直到最后才丢弃不需要的数据，会白白耗费 Spark 的处理能力和时间。
  • 关注并尽量减少执行器之间的数据交换，即所谓的“洗牌”过程。
  • 注意尽量减少 Spark action 操作（如 `count`、`collect`、`take`）的使用。与 `filter`、`select` 等延迟执行的 transformation 不同，action 操作会对计算图施加限制，从而阻碍潜在的优化。 - 明确 repartition（通过 shuffle 重新组织全量数据）与 coalesce（合并已有分区）之间的区别。 - 时刻警惕并留意数据倾斜问题。解决数据倾斜的技术手段很多，例如使用 Broadcast join（前提是左侧数据集足够小）或“join salting”（连接加盐）技术。 - 尽可能最大化作业的并行度。通过 **Spark details** 界面可以查看每个步骤的执行并行度。需要注意的是，如果启用了 Spark 动态 Executor 分配（默认开启），系统会在条件允许时自动缩减多余的 Executor，避免资源浪费。 - 留意启动 Spark 任务的开销（可在 Foundry 的 **Spark Details** 界面中查看）。大量微小任务会导致额外开销和不必要的数据交换。经验法则来看，单个任务执行时间在 30 秒到 60 秒之间较为合适（需结合该阶段的总耗时综合评估）。 如果在工作流中难以决断是使用多个单输出转换还是多输出转换，请查阅我们的指南《优化多输出转换》。
