目标

建模目标是一个项目，它围绕特定的业务运营问题，将上下文、相关数据源、元数据和模型集中管理。可以将目标视为建模问题的定义——即问题的接口，而提交的模型则为其提供实现。

建模目标充当建模生态系统的沟通枢纽，并作为记录系统，用于随着时间推移评估、审查和逐步将模型解决方案投入生产。

目标与指标相关联，这些指标定义了如何针对提交给目标的模型进行相互评估。

提交

当模型提交到建模目标进行管理和评估时，系统会创建该模型版本的副本。这种不可变的提交类似于代码的 Pull Request——提交模型意味着请求全面的审查。建模目标在功能上相当于一个潜在具备生产价值的模型版本的 数据目录。

发布

模型提交后即可进行发布。发布是包含模型提交代码的、带版本号、已打包且具备生产就绪状态的资产。可以将发布视为针对目标中定义的建模问题所通过的解决方案。

发布包含可配置的环境标签（如“预发布”或“生产”）、用户定义的版本号以及简短的描述字段——即发布说明。

已发布的模型为部署提供支持，这些部署即为生产推理流水线或交互式端点。可以将部署配置为自动拉取最新打标签的发布版本。例如，带有“生产”环境标签的部署将获取最新的“生产”发布版本。

发布系统通过要求经由打标签和带版本的发布进行显式升级，为生产部署提供了一种保护机制，同时为生产流水线提供了可审计的模型版本历史记录。

部署

部署支持将选定并发布的模型交付给消费者，包括生产流水线、业务运营应用和 API 订阅者。

部署为 Foundry 中的模型提供了一个受治理的持续集成和持续交付（CI/CD）层。在模型提交经过审查并发布后，相应的部署会自动拉取新的模型版本，整个过程无需停机，且能保留血缘关系。

部署类型

批处理部署

批处理部署通过在指定的 Foundry 输入数据集上执行模型，并将结果发布到输出数据集，从而在流水线中运行模型。它们利用分布式计算，适用于生产流水线（例如，预测某个县所有地址的房价），以及大规模非实时处理（如批量计算机视觉或文档分析）。即使切换了部署的模型，消费者也可以从一致的输出数据集中读取数据。

批处理部署通常通过构建计划进行管理，并利用 Foundry 的内部构建工具。权限从输入数据集、模型、目标和包含它的项目向下传递。阅读有关如何设置批处理部署的内容。

实时部署

在低延迟或交互式场景下，可以通过实时部署来提供模型服务，该部署提供支持交互式查询的无服务器 REST API 端点。

实时端点可以独立配置权限，并以配置好的副本和资源执行。它们还具备高可用性，这意味着可以通过 CI/CD 更新模型，而不会导致端点停机。

您可以通过为模型创建交互式实时端点或使用模型函数，将实时部署集成到业务运营应用中。您还可以将模型集成到情景分析和模拟中。

指标

建模过程中最关键的步骤之一，是了解模型的性能表现及其适用条件。在 Foundry 中，你可以基于评估数据集及其内部的逻辑子集，为模型生成并追踪指标。这些指标及其关联的输入数据集，为《建模目标》中的评估工作流提供支撑。

你可以采用临时方式生成指标（例如在开发阶段），也可以将其作为建模目标《自动模型评估》的一部分进行系统化生成。随后，这些指标会在模型预览中展示，并可在《评估看板》中与其他模型进行对比。

MetricSet 封装了单次模型评估的数值指标、图像和图表。MetricSets 包含对相应模型（及版本）的引用，以及计算指标时所基于的特定数据集和事务（即版本）。该数据集引用被称为 MetricSet 的“输入数据集”。查看指标集支持的各项配置选项。

元数据

每个目标都关联了高度可配置的元数据，可用于支持广泛的管理与协作工作流。每次提交模型时都可以采集自定义的元数据字段，这有助于实现不同模型之间的相互对比。
