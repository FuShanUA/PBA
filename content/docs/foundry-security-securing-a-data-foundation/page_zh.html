保障数据底座安全

Foundry 的核心目标是提供组织内客观事实的规范化视图。构建这一现实的首要步骤是建立并配置数据底座的权限。为了说明如何保障数据底座的安全，我们将通过一家名为 Sky Industries 的飞机制造商的假设案例来进行演示。我们将扮演 Sky Industries 的一名开发者，负责整合原始的航班、跑道、机场、告警、航线和乘客数据，将其准备好映射至本体，随后提供给整个 Sky Industries 组织使用。

创建项目

设计数据底座的关键环节在于，确定需要为生产数据管道创建哪些项目，以便于未来的扩展和日常维护。在我们的示例中，我们将采用推荐的项目设置，并创建 3 个具有不同用途的项目：

  • Flight Control System [Datasource]： 从运输统计局（BTS）接入原始数据并在此项目中进行清洗。
  • Flight Delays [Transform]： 从 Flight Control System [Datasource] 项目导入数据集，并进行转换以生成可复用的数据集。
  • Aviation [Ontology]： 从 Flight Delays [Transform] 项目导入数据集，并进行转换以表示离散的组织对象。

根据您的 Foundry 实例情况，您可以同时创建项目和/或用户组。我们的目标是为每个项目配置三个用户组，分别映射到一个默认角色（例如，Viewer）。我们希望项目能被组织内的其他用户发现，因此我们将项目的默认角色设为 Discoverer，并且不为项目添加任何标记。此外，我们将在每个项目中创建一个代码仓库，用于存放我们的转换逻辑。以下是我们为 Flight Delays [Transform] 项目配置的最终设置。

项目引用

既然已经有了用于构建管道的项目，我们就可以开始使用 Foundry 来编写业务逻辑了。有关最佳实践的详细信息，请查阅《数据集成文档》。

在使用其他项目的数据之前，我们需要在当前项目中创建对该项目的引用。完成此操作后，我们就可以在构建中使用该数据集，并允许项目中的任何人基于这些数据编写转换逻辑，即使他们本身没有源项目的访问权限。

由于我们的管道跨越了 3 个项目，我们可以为每个项目分别向用户授予特定的角色权限。

例如，我们将第一位业务用户 Eric 添加到 Aviation [Ontology] - Viewer 组中，这将为他在 Aviation [Ontology] 项目中授予 Viewer 权限。假设默认角色为 Discoverer，这些用户在 Flight Control System [Datasource] 和 Flight Delays [Transform] 项目中将仅拥有 Discoverer 权限，而在 Aviation [Ontology] 项目中则拥有 Viewer 权限。需要注意的是，拥有数据源项目的发现者权限，并不会阻碍用户获取下游数据的访问权限，例如本体项目中的数据。

共享数据

我们建议通过将同事添加到正确的项目组来与其共享数据和资源，以便他们获得对该项目的统一访问权限。相比直接共享资源，这种方式更清晰且更易于管理。除了通过正确的项目组成员身份分配适当的角色外，还应检查是否满足其他访问要求，例如密级标识和组织成员身份。有关更多详细信息，请参阅检查权限部分。
