我的自定义环境初始化缓慢

慢速初始化通常表示环境定义太大或太复杂。随着环境中包的数量增加，初始化时间往往会呈超线性增长。此外，Foundry 通常会预先初始化常用环境，因此如果你是基于默认配置创建了自定义环境，速度变慢可能是因为你不再获得预初始化的环境，必须等待新环境被创建。有关环境创建过程的更多详情，请参阅 环境创建简介。

尝试以下方法：

  1. 撤销更改 - 如果在对环境配置进行了更改后，初始化变慢了，可以尝试恢复该更改。在某些情况下，添加一个包就可能显著增加初始化时间。
  2. 简化配置 - 从环境定义中删除任何不需要的包。小型、专用的环境比大型通用环境性能更好。
  3. 指定版本 - 在环境配置面板中的一些包上添加版本约束，如 选择一个环境 中所述。
. 对于像python这样有许多可能版本的包，以及像scipy这样依赖关系复杂的包来说，这样做最有效。如果添加某个特定包导致环境变慢，可以尝试为该包固定一个版本。
  • 注意事项: 添加版本限制可能会导致环境无法满足。如果环境无法初始化，可以尝试放松限制。例如，python=3.6.10将仅匹配该版本，而python=3.6将匹配python 3.6.x中的任何版本，因此更有可能满足。
  • 一个选项是等待你的环境（无限制）解析完成，然后查看选择了哪些版本（参见 查看已解析的环境）。然后你可以为这些特定版本添加限制，直到初始化性能足够快。通常，对于依赖项较多的包固定版本，而其他所有包保持未固定是最优的。

环境无法初始化

如果您的环境无法初始化，您可以查看环境启动错误日志以获取有关出错原因的更多信息。

日志的第一行将显示 Execution failed with non-zero exit code:，然后是一个整数错误代码。此错误代码表示具体的失败模式。

代码 1 - 环境解析错误

此错误表示指定包（及其依赖项）的某些子集不兼容或无法安装。如果最近在 Environment Configuration 面板中的更改导致初始化开始失败，请尝试恢复更改。以下是常见环境解析错误列表。

新的 Mamba 错误信息

Palantir 贡献 ↗ 通过提供更好的环境初始化错误信息格式，为开源 Mamba 社区做出贡献。截至 2023 年 2 月，Foundry 服务受益于更贴近被环境故障侵犯的依赖树的错误信息。

有关理解这些新消息、如何解释它们以及如何修复它们的指导，请参阅 新错误消息。

旧版 Mamba 错误信息

以下列出了在引入 新版 Mamba 错误信息 之前常见的旧版错误信息。

找不到包

在这种情况下，没有配置的通道提供 A 依赖项。

Copied!
1Problem: nothing provides requested A.a.a.a

如果您的包的固定版本不存在，就可能发生这种情况。如果是这种情况，请尝试在meta.yml中放松或删除包的版本信息；例如，matplotlib 1.1.1 可能会变成 matplotlib。或者，这可能意味着环境管理器无法找到要安装指定版本的必要包。

未找到依赖项

在这种情况下，包 A 包含了一个必需的依赖项 B，但没有任何渠道提供该依赖项。

Copied!
1Problem: nothing provides B-b.b.b needed by A-a.a.a:

请注意，B 可以是用户显式请求的包（在 meta.yml/foundry-ml/vector 配置文件中），也可以是一个传递依赖项。

如果 B 在您的注册中不可用于安装，就可能发生这种情况；例如，B 可能已被召回，因此无法安装。

如果是这种情况，请尝试在 A 中移除固定版本，以防 A 有所有依赖项都可用的版本，或者联系您的 Palantir 代表将必要的包导入 Foundry。

重复包错误
Copied!
1cannot install both X.a.b.c and X.d.e.f

如果你尝试安装相同的包 X 并且为两者指定了不同的版本，就会出现此错误。例如，如果你尝试在你的 meta.yml 中同时指定 python 3.6.* 和 python 3.7.* 的版本，可能会收到此错误。你可以通过删除其中一个重复的包版本来解决此问题。

权限错误
Copied!
1CondaService:ReadRepositoryPermissionDenied

如果你收到这个错误，可能是因为你需要的一些资产和包在你的注册中不可用。

包冲突

在这种情况下，包 A 对包 B 的版本有要求，但此版本的 B 与其他包存在冲突：

Copied!
1Problem: package A-a.a.a requires B >=2.2.5,<2.3.0a0, but none of the providers can be installed

请注意，包 B 可能是指 A 的传递依赖项，而您并未在环境中显式列出该依赖项作为需求。有关传递依赖项的更多信息，请参阅 已解析的环境 文档。

修复 Code 1 错误

要修复环境失败问题，请考虑以下步骤：

  1. 检查环境启动日志中的 STDOUT。
  2. 创建一个失败解决的最小示例。移除包直到环境初始化，然后重新添加包直到确定哪些包是阻塞项。
  3. 尝试放松约束，移除包的固定版本。
    • 您可以同时打开多个工作簿或同一工作簿的不同分支，以加快此过程。
    • 我们建议使用二分查找来发现这些问题。在一个工作簿中移除所有约束，在另一个工作簿中移除一半的约束，然后在另一个工作簿中移除另一半，依此类推。

代码 137 - 模块内存不足

这个错误通常表明环境初始化所需的内存超过了 Spark 模块的可用内存。如果最近在环境配置面板中的更改导致初始化开始失败，请尝试撤销更改。否则，尝试以下建议之一或多个：

  • 从环境中删除不需要的包。环境中的包数量与内存使用量呈超线性增长关系，因此即使删除几个包也可能足以解决问题。
  • 如“环境初始化缓慢”部分第 3 步所述，为一些包添加版本约束。您固定每个包版本将减少环境求解的复杂度，从而减少初始化环境所需的内存量。

如果在移除了所有不需要的包并将一些剩余的包固定到特定版本后，您的环境仍然无法初始化，请联系您的 Palantir 代表，并附上您尝试的所有调试步骤的详细信息。

其他代码

如果您的环境出现上述未列出的错误代码并且重试后仍然无法初始化，请联系您的 Palantir 代表，并附上完整的错误日志文本。

默认环境初始化速度不一致

我们预计默认配置文件有时会快速初始化，有时则需要更长时间。在后一种情况下，您需要等待 Mamba 解析您的环境规范并将其安装到支持您工作簿的 Spark 模块中。这是默认行为。在前一种情况下，您可能获得一个已预先初始化的模块，或者您的环境规范与缓存中已解析的环境匹配，可以立即安装。如果许多用户在同一时间打开工作簿，您不太可能获得一个预初始化的模块，需要等待一个新的模块被启动。

同时需要 Conda 包和 JAR 的包

一些包需要同时拥有 Conda 包和 JAR 才能使用。一个常见的例子是 graphframes 包。此类包需要特殊的设置过程，如果此包尚未为您的实例进行配置，则在构建时可能会遇到以下错误：

Copied!
1o257.loadClass.: java.lang.ClassNotFoundException:<Class>

JAR 文件可以添加到控制面板（Control Panel）的 Conda 环境配置标签页的 高级 部分中该配置文件的配置中（参见 配置代码笔记本配置文件）。如果您需要帮助配置此类包，请联系您的 Palantir 代表。
