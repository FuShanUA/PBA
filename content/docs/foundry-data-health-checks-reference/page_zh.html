构建状态检查反映了最终数据集构建所经历的整个过程的状态。因此，它将针对促成该最终数据集创建或更新的各个步骤提供状态反馈。需要注意的是，若在此过程中更新或创建的中间数据集也配置了构建状态健康检查，这些中间数据集的构建状态将不会更新。不过，所有这些中间数据集的作业状态均会得到更新。

作业状态检查独立于引发数据集刷新或创建的构建行为触发。换言之，无论相关数据集是否为某次构建的最终输出，作业状态检查都会针对特定数据集的每一次构建运行。

何时使用作业状态、构建状态或调度状态检查

通常建议所有调度都配置调度状态检查。若调度已有调度状态检查，不建议在同一调度构建的其他数据集上安装作业状态检查，因为该调度上的任何作业失败都会触发调度状态检查。

若需检查中间数据集是否已更新，且无需关心同一构建中的其他数据集是否成功更新，请对中间数据集使用作业状态检查。如有需要，当数据集作为构建输出，且需验证整个构建及包含该数据集在内的所有数据集均成功时，请使用构建状态检查。

如果数据集是构建的唯一输出，则构建状态与作业状态等同。如果该数据集属于中间数据集，或者构建包含多个输出，且该数据集对应的作业成功（或未运行），但同一构建中的其他作业失败并导致整体构建失败，两者的状态便会出现差异。

可用的模式检查类型如下：

值比较允许规则

近似标准差

由于数据集构建过程极易出现异常值，因此我们不采用真实标准差，而是使用绝对中位差（MAD），这是一种对数据离散程度更为稳健的度量方式。

MAD 定义为数据点与其中位数绝对偏差的中位数。对于中位数为 X 的数值 x_1, ..., x_n，其计算公式为 MAD = median(|x_i - X|)。

乘以一个常数后，绝对中位差可用于近似估算标准差。

我们的计算方式为 σ = MAD * 1.4826。

详细信息请参阅 《绝对中位差 - 维基百科》 ↗。
