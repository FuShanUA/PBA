构建复杂布局

设计应用程序的外观和感觉是一个令人兴奋的前景，Slate 为您提供自定义和覆盖许多小部件外观方面的灵活性。然而，许多项目在实现与设计草图完全匹配时会偏离轨道，这比高效、简洁地实现工作流功能更重要。

另一种思考方式是，每次样式覆盖都是对应用程序简单性和可维护性的权衡。在选择布局和样式时要谨慎且有意识，并在继续开发新功能之前花时间定期重构和清理您的应用程序。

构建复杂布局

使用 容器小部件（Container widgets） 用来布局你的应用程序。在足够多的容器以逻辑地组织你的应用程序和不必要的嵌套容器之间找到一个平衡点。请注意，当你构建应用程序时，每个小部件最终都会向文档模型（DOM） ↗添加多个层次，而DOM的复杂性与浏览器资源使用、页面加载和应用程序的整体响应性有直接关系——也就是说，你的应用程序有多“重”。如果有疑问的话，尽量使用更少的小部件，并尝试避免将容器嵌套超过3或4层。 在几乎所有情况下，你应该避免使用CSS来影响你应用程序的布局——远离position，或者使用静态的小部件定位或弹性容器进行相对定位。在这里有一些特殊情况的例外，但就应用程序的一般布局而言，通过坚持使用内置的定位工具，你可以避免不必要的复杂性并提高你应用程序的可读性。

对齐小部件 对齐控件

Slate 的一个经常被忽视的功能是能够将控件相互对齐。按住 CTRL（Mac 上为 CMD）键并点击以选择一个或多个控件。选择多个控件后，右侧面板将不再显示控件配置，而是提供若干选项来对齐或分布控件。

此外，如果您希望布局“自动”进行，可以使用 Flex 容器 来布置子控件。有关如何使用 Flex 容器的更多信息，请参见下面关于响应式布局的部分。

显示和隐藏控件

动态布局通常包括根据应用程序的状态显示或隐藏内容。有多种方法可以实现这一点。以下三种模式分享了一些选项，并讨论了各自的优缺点。

简单：通过类来隐藏

在最简单的情况下，你可以通过将小部件显示配置中的Additional Classes属性模板化为hidden来隐藏一个小部件。常见的模式是使用一个函数（或你的状态变量）来确定是否应该显示一个小部件，然后将该函数的输出模板化到小部件中。虽然这很简单，但这种方法在多个小部件上无法很好地扩展，并且你会看到“隐藏”小部件在页面加载时有短暂的闪烁，因为它们会在依赖关系图解析并应用类之前保持可见。

改进：通过标签页容器来隐藏

特别是当有多个小部件时，更好的做法是使用一个标签页容器，其中一个标签页为空，另一个包含要显示的小部件。这里的模式是应用一些CSS来隐藏容器边框和背景（即让容器相对于背景“不可见”）：

Copied!
1// This will remove the border and background from any container with 'minimal' as
2// an 'Additional Class'
3
4.minimal sl-app-container {
5    border: none;
6    background: none;
7}

然后将selectedTabIndex属性模板化，以在隐藏和可见之间切换。使用此模式，在页面加载时不会有闪烁，并且小部件会立即隐藏。

复杂：从根元素中隐藏

如果您需要一个小部件不仅“隐藏”，而是从DOM中完全移除，可以通过在父容器上模板化一个额外的类，通过根元素选择器来隐藏该小部件。这种模式最常见于响应式布局中，您需要隐藏一个小部件并且使同一层级的其他子小部件重新定位。

每个slate小部件的根元素都有一个ID选择器，格式如下：widget-[widgetName]。因此，假设我们有一个名为w_myHTMLWidget的小部件，它是w_sidebar容器的子项。要完全隐藏该小部件，您可以在全局样式中定义一个类：

.hide-myHTMLWidget {
    #widget-myHTMLWidget {
        display: none;
    }
}

然后在w_sidebar中 你可以有条件地添加这个类，整个小部件就会隐藏。有关使用条件CSS类的更多模式和示例，请参见下面的使用样式部分。

“多页面”应用程序

分割应用程序最常见的方式是使用标签容器。每个容器标签可以视为一个单独的页面，但重要的是要记住这些“页面”只是UI功能 - 例如，这不会影响依赖关系图的解析，因此即使图表在不同的标签页上，用于填充该图表的查询仍将按照依赖关系图的解析来运行。

如果你正在使用标签容器构建一个多页面应用程序，请确保检查Lazy rendering enabled选项 - 这会将标签上的小部件渲染移到当标签打开时，而不是当页面加载时 . 这对于非常大的应用程序尤其有用，因为在任何给定时间，大多数小部件都“隐藏”在另一个标签页中，可以显著提高页面加载时间。

即使您不希望允许用户手动选择某个标签页，也应使用带标签的容器 - 例如，如果您正在构建一个工作流应用程序，并且需要控制用户何时被移动到下一个标签页。在这种情况下，您可以取消选中 Show tab titles 选项，然后使用 Handlebars 语句来模板化在任何给定时间应显示哪个标签页（从零开始索引）。一种常见模式是将此标签页作为应用程序状态的一部分进行跟踪（参见上面的 构建有状态的应用程序），以便您可以简单地更新状态变量中的当前标签页值，应用程序中的标签页就会发生变化。
