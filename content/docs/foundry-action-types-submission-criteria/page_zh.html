提交条件

提交条件（以前称为验证）是决定某个操作是否可以提交的条件。提交条件支持将业务逻辑编码到数据编辑权限中，确保本体论数据质量与编辑治理。

提交条件是通过结合基于上下文（如用户或参数）的条件和静态信息来创建一个逻辑语句。提交条件可以将对象、关系甚至用户信息纳入逻辑语句，以确定某个操作是否可以被提交。

示例

例如，一家航空公司可能希望更改特定航班列出的飞机。配置的操作允许用户更改与Flight对象关联的Aircraft对象。然而，航空公司只希望选定的用户（如航班调度员）能够使用此操作，以确保仅使用仍在运行的飞机。通过提交条件，构建者可以确保只有在满足条件时，才能提交更改航班飞机的操作，这可以通过将用户的组成员身份与操作提交时飞机的状态相结合来实现。

提交条件由条件和运算符组成。条件是控制参数或用户属性值的单一陈述。运算符用于组合和嵌套不同的条件。

使用不同类型的运算符，我们可以创建更复杂的语句，以映射其业务流程和需求。只有在满足所有提交条件的情况下，才能提交操作。这与决定用户是否可以编辑操作类型本身的权限无关。虽然一种对象类型可以有多个操作类型，包括添加、修改和删除对象，但每个操作类型都有独立的提交条件。

条件

条件是两个值之间的单个比较检查。每个条件根据其参数或用户输入通过或失败。可以使用两种条件模板之一来配置条件：“基于当前用户”或“基于参数”。这些模板为其余的条件提供了一个框架。每个条件都是使用中间运算符在两个值之间进行的简单比较。

示例

继续我们的示例，可以使用 当前用户 模板来设置飞行控制器要求，因为它需要提交操作的人员上下文。为了知道飞机是否仍在运行，需要通过 参数 模板使用 飞机 对象。

当前用户

当前用户 模板根据提交操作的用户定义权限。当前用户 输入可用于检查用户的 ID、通过组 ID 的组成员身份，或其他任何多通道属性（如用户的组织）。Foundry 将用户 ID 视为字符串，可以与静态定义的用户 ID 列表或存储用户 ID 的任何字符串参数进行比较。

组 ID 选项允许您使用动作用户所属的组（无论是直接还是继承的成员资格）创建条件。这些组可以与静态选择的组或由其他参数提供的组 ID 进行比较。

多值属性被视为字符串列表，只能与其他字符串或字符串列表进行比较。用户将拥有一个其可访问的多值属性列表。使用 其他用户属性 字段，可以针对用户没有访问权限的属性配置条件。如果用户没有访问某个属性的权限，他们将无法通过该条件。

示例

为了知道我们示例中的用户是否是飞行控制员，我们需要检查该用户是否是飞行控制员组的成员。

参数

提交标准也可以使用参数部分中定义的参数。参数由其他应用程序或用户自身传递到操作类型中。对参数使用条件可以让构建者将业务逻辑嵌入到操作类型中，并防止用户在不符合业务要求的数据上提交操作。

示例

在我们的示例中，操作状态通过 Aircraft 对象提供，并且可以随着每架飞机而改变。条件需要基于 Aircraft 对象类型参数构建。

提交标准不支持附件和对象集合参数。这些参数类型已从选择面板中移除。

选择一个值

在选择条件模板后，选择要比较的值。一些参数（如列表或对象参数）需要更细致地选择用于比较的值。我们也可以选择比较列表的长度而不是其内容。

示例

在飞机示例中，飞机的操作状态存储在 Aircraft 对象的属性中。

由于我们示例中的用户属于多个组，但比较的是单个组，因此需要选择 includes 运算符以检查重叠。然而，操作状态必须与预期状态完全匹配，因此必须设置 is 运算符。

值

值表示比较的另一侧。该值可以基于现有参数、静态值或无值。无值用于检查第一个值是否为空（或 null）。与运算符一样，可用选项取决于第一个值的类型。

逻辑运算符

逻辑运算符可用于组合不同的条件。逻辑运算符也可以嵌套以创建更复杂的逻辑，并且可能需要满足其下的所有、任何或没有条件才能通过。

失败信息

失败消息支持定义在操作无法提交时应显示的错误信息。根级别的每个条件和逻辑运算符都有自己的失败消息。如果较低级别的条件未满足，则显示相应根级别（父级）的失败消息。每当条件不满足时，失败消息会在Foundry（对象资源管理器、Workshop或Quiver）中显示给最终用户。失败消息会告知用户为何他们被阻止提交操作。
