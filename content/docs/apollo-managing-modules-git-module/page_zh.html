在Git仓库中存储模块定义

一旦你有多个模块定义，你可能希望将它们存储在一个Git仓库中。将模块定义存储在Git仓库中可以实现更好的协作和对更改的控制。它允许用户在Git仓库中协作编辑模块定义，并且任何更改都需要通过拉取请求来审核。这种方法确保所有更改在应用之前都经过审查和批准。如果你希望单个用户能够独立创建模块，则不需要将模块定义存储在Git仓库中。

在Git仓库中存储模块定义的好处包括：

  • 协作： 多个用户可以同时对模块定义进行工作。
  • 审核流程： 使用拉取请求在合并前审核和批准更改。
  • 自动化： 与CI/CD管道集成以自动验证和更新。

通过利用Apollo CLI，您可以有效地管理这些定义。Apollo CLI 提供了多个命令，可用于验证、更新和锁定您的模块定义，确保一致性，并在团队间实现高效的协作。这些命令可以集成到CI步骤中，或作为Git预钩子使用，以自动化模块定义的管理。

这种方法只有在您小心不要直接授予所有人创建模块的权限时才有效。通过限制模块创建权限，您可以确保所有更改都经过代码仓库和审核流程，从而保持控制和一致性。

apollo-cli module create

apollo-cli module create 命令用于在Apollo中创建或更新一个模块。该命令处理模块定义文件并在Apollo中创建模块。

有关创建模块的更多详细信息，请参阅 创建一个模块 部分。

使用方法

Copied!
1apollo-cli module create --module-definition-file /path/to/module-definition.yml

预演

您可以使用 --dry-run 标志来模拟 Module 的创建，而不会实际发布它。这在对 Module 定义进行任何更改后验证其有效性时非常有用。

Copied!
1apollo-cli module create --module-definition-file /path/to/module-definition.yml --dry-run

创建所有模块

您也可以为目录中的所有模块创建新版本，而不是分别创建每个模块。只有在定义发生更改时才会创建新版本：

Copied!
1apollo-cli module create --module-directory /path/to/module/directory [--dry-run]

apollo-cli module lock

《apollo-cli module lock》的 apollo-cli module lock 命令返回一个扁平化的模块定义视图，包括所有子模块，使查看将包含在您的模块中的所有实体变得更容易，即使它们是嵌套的。它还包括模块在安装期间所需的变量列表，包括任何子模块中的变量。此扁平化视图保存为与模块定义同名并带有.lock后缀的文件。

当在Git仓库中存储模块定义时，您应该在预合并CI作业中运行此命令，以确保锁文件必须存在且是最新的。这允许代码审查者查看模块定义的更改，包括对来自子模块的实体或变量的更改。

用法

Copied!
1apollo-cli module lock --module-definition-file /path/to/module-definition.yml

验证

您可以使用 --verify 标志对模块定义执行验证检查。此标志通过将其与生成的文件进行比较，确保现有的 .lock 文件是最新的。如果两者不匹配，则命令会失败。

Copied!
1apollo-cli module lock --verify --module-definition-file /path/to/module-definition.yml

创建所有锁定文件

您可以通过使用 --module-directory 标志一次创建或验证目录中所有模块定义的锁定文件：

Copied!
1apollo-cli module lock [--verify] --module-directory /path/to/module/directory

apollo-cli module update

apollo-cli module update 命令用于更新 复合模块 中的子模块版本

. 因为子模块定义包括确切的版本，此命令可用于定期将所有版本更新为最新版本。一旦执行此操作，还必须运行 module lock 命令以根据子模块更新可能带来的更改来更新锁定文件。

使用方法

Copied!
1apollo-cli module update --module-definition-file /path/to/composite-module-definition.yml
Copied!
1name: example-composite-module
2displayName: Example Composite Module
3description: Example Composite Module
4submodules:
5   - name: "example-module-1"
6-    version: 0.0.0
7+    version: 0.2.0
8   - name: "example-module-2"
9-    version: 0.1.0
10+    version: 0.3.0

在本例中，example-module-1 已从版本 0.0.0 更新到 0.2.0，而 example-module-2 已从版本 0.1.0 更新到 0.3.0。更新子模块版本后，应运行 module lock 命令，以根据子模块更新可能带来的更改来更新锁定文件。

version: local

在你的子模块引用中，你可以使用 version: local 代替具体的 Maven 坐标（例如，0.3.0）。Apollo 然后总是引用仓库中的最新版本，并自动将 local 版本解析为 Maven 坐标。 如果一个子模块被更新，Apollo 会创建父复合模块的新版本，因此不需要手动更新复合模块（例如 0.3.0 到 0.4.0）。 要使用 version: local，你需要向 Apollo 提供所有的 Module 定义。Apollo 使用这些作为 local 版本。为此，请使用 --module-directory 标志：

Copied!
1apollo-cli module create [--dry-run] --module-directory /path/to/module/directory

同样的标志也适用于 module lock：

Copied!
1apollo-cli module lock [--verify] --module-directory /path/to/module/directory
