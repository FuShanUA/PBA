权限参考

Data Connection中的所有主要概念——代理（Agents）、源（Sources）和同步（Syncs）——在Foundry中作为 资源（resources） 进行管理。这使得可以在项目和文件夹之间灵活地组织这些资源，并允许对Data Connection资源应用安全原语，例如 标记（markings）。

平台所有者应小心不要过度共享Data Connection资源。具有源（Sources）和代理（Agents）查看和编辑权限的用户应被信任为管道（Pipeline）开发人员。同步（Syncs）产生的数据集可以根据需要导入到下游转换（Transforms）项目中，从而在不共享对代理或源的访问权限的情况下授予对数据的访问权限。

本页面记录了所有Data Connection资源的权限工作方式，包括：

  • 代理（Agents）
  • 源（Sources）
  • 同步（Syncs）
  • 插件（Plugins）
  • Webhooks
  • 代码中的外部连接（External connections from code）

资源权限

以下列出的权限是默认设置。在某些环境中，可能通过 自定义角色 更改了默认权限行为。

代理

所有者

代理的所有者通常是其创建者，并拥有以下权限：

  • 重新下载代理或为该代理重新生成令牌
  • 继承编辑者的权限

编辑者

代理的编辑者可以：

  • 将源代码部署到该代理。
  • 使用代理管理器（更改插件和配置，并重启代理）配置代理。
  • 与其他用户共享代理。
  • 继承查看者权限。
  • 删除代理。

用户必须是项目的编辑者才能在该项目中创建代理，但必须是项目的所有者才能管理该项目中的代理。这意味着用户可能创建了一个代理，但之后无法生成下载链接或对代理执行其他管理任务。

查看者

代理的查看者可以：

  • 查看代理配置和状态。

源

所有者

默认情况下，源所有者继承Editor权限。

源所有者还可以：

  • 如果配置导出的用户有权限从任何组织或他们添加为可导出到所选源的安全标记中移除标记，则可以修改导出配置。
  • 允许将源导入到代码资源中。

编辑器

源的编辑器可以：

  • 删除源。
  • 更改源分配的代理（添加代理时还需要对每个要添加的代理具有“编辑”权限）。
  • 更新源配置（对分配给源的代理还需要“编辑”权限）。
  • 重命名源。
  • 为该源创建、编辑和删除同步。
  • 在数据库源上运行 SQL 查询。
  • 与其他用户共享源。
  • 探索并预览支持这些操作的源，例如 JDBC 和目录源。
  • 继承查看者权限。
  • 将源导入到代码资源
警告

请记住，如果源凭据允许，同步操作可能会更改源系统（例如，从目录或S3源删除文件，或通过任意SQL从数据库中删除数据）。您只能将源（或包含该源的项目）的编辑访问权限授予那些您也会授予其账户全部访问权限的用户。

查看者

  • 源的查看者可以查看源配置，包括分配的代理。

同步

同步权限来源于相关的源和输出数据集。

  • 查看一个同步需要源和数据集的查看权限。
  • 编辑一个同步需要源和数据集的编辑权限。
  • 删除一个同步需要源和数据集的编辑权限。
  • 运行一个同步需要数据集的编辑权限。

插件和JDBC驱动程序

所有者

  • 继承编辑者权限。

编辑者

  • 删除插件 / 驱动程序（插件 / 驱动程序不能被分配给任何代理，才能成功删除它）。
  • 继承查看者权限。

查看者

  • 查看插件 / 驱动程序。
  • 下载插件 / 驱动程序。
  • 将插件添加到代理中（还需要代理的编辑者权限）。

Webhooks

Webhook 的权限完全从其关联的源继承。

  • 查看 Webhook 配置 需要源的查看权限。
  • 创建 Webhook 需要源的编辑权限。
  • 编辑 Webhook 配置 需要源的编辑权限。
  • 删除一个Webhook需要对源数据有编辑权限。
  • 配置一个操作以使用Webhook需要对源数据有编辑权限。
  • 执行一个Webhook需要对源数据有编辑权限。请注意，当通过一个操作执行Webhook时，不需要执行Webhook的权限。

默认情况下，只有执行Webhook的用户可以使用Data Connection应用程序中的历史记录选项卡查看响应。可以将webhooks:read-privileged-data权限添加到自定义或默认的角色中，以便具有该角色的用户可以查看他们可以访问的所有Webhook的完整请求和响应历史记录。标准推荐的设置是添加一个名为“Webhook特权数据查看器”的新角色，并确保需要查看完整Webhook历史记录的用户在相关源上拥有此角色。

从代码发起的外部连接

外部转换（transform）、管道（pipeline）和计算模块

当源被导入到代码资源中时，具有仓库编辑权限的用户可以使用为该源配置的连接详细信息编写自定义代码，包括访问密钥值。

这意味着将源导入代码以及授予Editor对源的访问权限的能力应谨慎管理，只有可信用户才能获得对源和仓库的访问权限。

  • 在代码资源中导入一个源（转换（transform）、管道（pipeline） 或 计算模块（compute module））需要对源拥有 Editor 权限。
  • 触发导入源的构建需要对代码资源拥有 Editor 权限（对源本身没有任何权限）。
  • 从代码资源中移除已导入的源需要对源 或 代码资源拥有 Editor 权限。

外部函数

有关在函数仓库中使用webhooks的权限的详细信息，请参阅 外部函数权限 的文档。

标记传播

如上所述，资源可以应用 组织（Organizations） 和 标记（Markings） 作为额外的访问控制级别。将组织和标记应用于源时，它们会传播到从该源同步生成的数据集。这意味着用户无法查看任何来自该源的数据集中的数据，除非他们有权访问该源上应用的所有组织和标记。

Foundry到Foundry同步异常

当使用 Foundry 连接器在 Foundry 实例之间同步数据时，标记（Marking）和基于列的访问控制（CBAC）策略不会自动传播到目标堆栈。虽然在源堆栈上会尊重 CBAC —— 用户只能同步他们被允许查看的数据 —— 但这些安全配置在同步过程中不会被传输。数据到达后，您必须手动在目标堆栈上配置适当的权限和标记。

由于源可能包含不同敏感级别的数据，我们建议您将源标记为适用于该源中可用数据的所有组织（Organization）和标记（Marking），然后使用转换（Transform）功能通过 stop_propagating 和/或 stop_requiring 功能来清理并去除数据的标记。有关去标记的更多信息，请参阅关于 移除标记 和 移除继承的标记 的指南。

请注意，在数据同步后应用组织（Organization）和标记（Marking）不应被视为充分的；因为对源的访问等同于对数据的访问，源应标记为适用于其包含的任何内容的组织和标记。

最佳实践

作为快速提醒，主要的Data Connection资源是代理（agent）和源（source）。插件和驱动程序被部署到代理中以扩展功能。

代理最佳实践

对于代理，有两种推荐的权限设置：每个组织（Organization）中的所有代理都放在一个项目（Project）中，或者每个代理都单独放在一个项目中。上面的图表展示了前一种情况；下面的指导说明了你可能选择每种情况的原因。

选项1：将每个组织的所有代理放在一个项目中。

将所有代理放在一个项目中是最简洁的方法，如果有一组用户负责管理所有代理，这是推荐的做法。这使您可以在项目级别控制对代理的访问，并且可以分别在代理下游对源和同步进行权限设置。

选择此选项仅当每个代理管理器和每个源编辑器都可以访问所有代理时是可接受的。请记住，要在一个代理上部署一个源，用户必须对该代理具有编辑权限。

选项 2：将每个代理放入其自己的项目中。

这种方法允许代理权限的完全细粒度控制。它还保证了在源管理演变过程中，结构永远不会偏离“分组”关系。例如，您可以安全地将代理分配/取消分配给一个源，而无需向用户授予整个代理组的访问权限，并且将来无需重新调整项目以适应组权限的变化。

如果您有多个组负责管理代理，并且希望分别授权他们对代理的访问权限，则建议采用此方法。

警告

项目中的编辑者可以创建新的数据源和在该代理上运行的同步任务。请确保不要将“编辑者”角色授予不应该执行此操作的用户。

数据源最佳实践

每个数据源应位于 数据源 层的单独项目中。在数据源上定义的每个同步任务应输出到与数据源同一项目的 数据集 中。如果数据源包含敏感数据（如个人身份信息），您可以对数据源应用一个 标记，该标记会传播到输出的数据集。

警告

包含数据源的项目的编辑者可以为该数据源创建新的同步任务和编辑现有的同步任务，还可以创建 Webhooks 并配置它们在 Foundry 其他地方的使用。

请注意，如果源凭证允许，同步可能会更改源系统；例如，从目录或S3源中删除文件，或通过任意SQL从数据库中删除数据。

您应该只将编辑权限授予源（或包含该源的项目），这些用户也应获得源所使用的账户的完整访问权限。

插件和JDBC驱动程序

对于插件和JDBC驱动程序，有两种推荐的权限设置：每个组织在一个项目中包含所有资源，或者每个JDBC/驱动程序在一个项目中，该项目包含一个或多个需要它的代理。

选项1：每个组织在一个项目中放置所有插件和驱动程序。

如果有一组用户负责管理所有代理程序，建议采用此方法。这允许代理管理员访问所有已上传的插件和驱动程序。只有在所有代理管理员之间需要平等访问插件和驱动程序时，才应选择此方法。

实施此方法的推荐流程是：

  1. 为每个代理程序创建一个文件夹。
  2. 将当前的插件和驱动程序迁移到相应的代理程序文件夹中。
  3. 为每个插件和驱动程序选择一个权威副本，并将其移动到项目根目录。
  4. 将每个代理程序迁移以使用所需插件和驱动程序的根目录副本。
  5. 弃用单独的代理程序文件夹。

选项 2：将每种插件和驱动程序类型放入其自己的项目中。

这种方法允许某些“敏感”的插件或驱动程序与其他插件或驱动程序分开进行权限设置。如果采用此方法，必须确保给每个项目的相应代理程序所有者足够的权限，以便他们可以在代理程序上使用这些插件和驱动程序。
