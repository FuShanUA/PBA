权限参考

数据连接（Data Connection）中的所有核心概念——智能体、来源和同步——均在 Foundry 中作为资源进行管理。这使得可以跨项目和文件夹灵活组织这些资源，并允许将标记等安全机制应用于数据连接资源。

平台数据主人应谨慎行事，避免过度共享数据连接资源。拥有来源和智能体查看及编辑权限的用户，应为受信任的管道开发者。同步生成的数据集可根据需要导入到下游的 Transforms 项目中，从而在不共享智能体或来源访问权限的情况下授予数据访问权。

本页面记录了所有数据连接资源的权限运作方式，包括：

  • 智能体
  • 来源
  • 同步
  • 插件
  • 网络钩子
  • 来自代码的外部连接

资源权限

以下列出的均为默认权限。在某些环境中，默认权限行为可能已通过自定义角色进行了更改。

智能体

数据主人

智能体的数据主人通常是其创建者，并拥有以下权限：

  • 重新下载智能体或重新生成该智能体的令牌
  • 继承编辑者权限

编辑者

智能体的编辑者可以：

  • 将来源部署到该智能体。
  • 使用智能体管理器配置智能体（更改插件和配置，并重启智能体）。
  • 与其他用户共享该智能体。
  • 继承查看者权限。
  • 删除该智能体。

用户必须成为某个项目的编辑者才能在该项目中创建智能体，但必须成为该项目的数据主人才能管理该项目内的智能体。这意味着用户可能创建了智能体，随后却无法生成下载链接或对该智能体执行其他管理任务。

查看者

智能体的查看者可以：

  • 查看智能体配置和状态。

来源

数据主人

默认情况下，来源数据主人继承 编辑者 权限。

来源数据主人还可以：

  • 如果配置可导出标记的用户有权从其添加为可安全导出到所选来源的任何组织或安全标记中移除标记，则可以修改导出配置。
  • 允许将来源导入到代码资源中。

编辑者

来源的编辑者可以：

  • 删除该来源。
  • 更改分配给该来源的智能体（还需要对添加的每个智能体具有编辑权限）。
  • 更新来源配置（还需要对分配给该来源的智能体具有编辑权限）。
  • 重命名该来源。
  • 为该来源创建、编辑和删除同步。
  • 对数据库来源运行 SQL 查询。
  • 与其他用户共享该来源。
  • 探索和预览支持这些操作的来源，例如 JDBC 和目录来源。
  • 继承查看者权限。
  • 将来源导入到代码资源
警告

请注意，如果来源凭据允许，同步可以更改源系统（例如，从目录或 S3 来源中删除文件，或通过任意 SQL 从数据库中删除数据）。您只应向那些您也会授予其来源访问数据所用账户完全访问权限的用户，授予对该来源（或包含该来源的项目）的编辑权限。

查看者

  • 来源的查看者可以查看来源配置，包括分配的智能体。

同步

同步权限派生自相关的来源和输出数据集。

  • 查看同步需要对来源和数据集具有查看权限。
  • 编辑同步需要对来源和数据集具有编辑权限。
  • 删除同步需要对来源和数据集具有编辑权限。
  • 运行同步需要对数据集具有编辑权限。

插件和 JDBC 驱动程序

数据主人

  • 继承编辑者权限。

编辑者

  • 删除插件 / 驱动（插件 / 驱动未分配给任何智能体才能成功删除）。
  • 继承查看者权限。

查看者

  • 查看插件 / 驱动。
  • 下载插件 / 驱动。
  • 将插件添加到智能体（同时需要该智能体的编辑者权限）。

Webhooks

Webhook 的权限完全继承自其关联的源。

  • 查看 Webhook 配置需要源的查看权限。
  • 创建 Webhook需要源的编辑权限。
  • 编辑 Webhook 配置需要源的编辑权限。
  • 删除 Webhook需要源的编辑权限。
  • 配置操作以使用 Webhook需要源的编辑权限。
  • 执行 Webhook需要源的编辑权限。请注意，通过操作执行 Webhook 时不需要执行 Webhook 权限。

默认情况下，只有执行 Webhook 的用户可以使用数据连接应用中的历史记录标签页查看响应。可以将 webhooks:read-privileged-data 权限添加到自定义或默认角色中，以允许拥有该角色的用户查看其可访问的所有 Webhook 的完整请求和响应历史记录。标准的推荐做法是添加一个名为“Webhook 特权数据查看者”的新角色并赋予此权限，并确保需要查看完整 Webhook 历史记录的用户在相关源上拥有此角色。

代码的外部连接

外部转换、管道和计算模块

当源导入到代码资源中时，拥有该代码仓库编辑访问权限的用户可以使用为该源配置的连接详情编写自定义代码，包括访问机密值。

这意味着将源导入代码以及授予对源的 编辑者 访问权限应当谨慎管理，仅允许受信任的用户访问源和代码仓库。

  • 在代码资源（转换、管道或计算模块）中导入源需要该源的 编辑者 权限。
  • 触发导入源的构建需要代码资源的 编辑者 权限（不需要源本身的权限）。
  • 从代码资源中移除已导入的源需要源或代码资源的 编辑者 权限。

外部函数

有关在函数仓库中使用 Webhook 权限的详细信息，请参阅外部函数权限文档。

标记传递

如上所述，资源可以应用组织和标记作为额外的访问控制层。应用于源的组织和标记将传递到该源上的同步操作生成的数据集中。这意味着用户无法查看从某个源生成的任何数据集中的数据，除非他们拥有对该源应用的所有组织和标记的访问权限。

Foundry 到 Foundry 同步异常

使用 Foundry 连接器在 Foundry 实例之间同步数据时，标记和基于列的访问控制 (CBAC) 策略不会自动传递到目标技术栈。虽然在源技术栈上会遵守 CBAC —— 用户只能同步允许其查看的数据 —— 但这些安全配置不会在同步期间传输。数据到达后，您必须手动在目标技术栈上配置适当的权限和标记。

由于源可能包含不同敏感级别的数据，我们建议您使用适用于该源中可用数据的所有组织和标记来标记源，然后使用转换通过 stop_propagating 和/或 stop_requiring 功能清理和取消标记数据。要了解有关取消标记的更多信息，请参阅有关移除标记和移除继承标记的指南。

请注意，在数据同步后应用组织和标记不应被视为足够的安全措施；由于对源的访问实际上是对数据的访问，因此应使用适用于其包含的任何内容的组织和标记来标记源。

最佳实践

在此简要提醒，数据连接的主要资源包括智能体和数据源。插件和驱动程序会部署到智能体上以扩展功能。

智能体最佳实践

针对智能体，我们推荐两种权限设置方案：每个组织将所有智能体放置在单个项目中，或者每个智能体独立放置在各自的项目中。上图展示了前一种方案；下文的指导说明将阐述选择每种方案的原因。

方案一：每个组织将所有智能体放置在单个项目中。

将所有智能体集中放置在单个项目中是最清晰的做法。如果由同一组用户管理所有智能体，则推荐采用此方案。这样可以在项目层级控制对智能体的访问权限，并在智能体下游分别对数据源和同步任务进行权限配置。

仅当允许所有智能体管理者和所有数据源编辑者访问全部智能体时，才应选择此方案。请注意，要在智能体上部署数据源，用户必须对该智能体具备编辑者权限。

方案二：每个智能体独立放置在各自的项目中。

这种方案能够实现最细粒度的智能体权限控制。它还能保证在数据源管理演进的过程中，结构永远不会偏离原有的“分组”设计。例如，您可以安全地为数据源分配或取消分配智能体，而无需向用户授予整组智能体的访问权限，未来也无需为了适应组权限变更而重构项目。

如果存在多个管理智能体的团队，且希望分别配置他们对智能体的访问权限，则推荐采用此方案。

警告

包含该智能体的项目编辑者可以创建在该智能体上运行的新数据源和同步任务。请确保不要将项目的编辑者角色授予不应执行此操作的用户。

数据源最佳实践

在数据源层中，每个数据源都应独立放置在各自的项目内。为数据源定义的每个同步任务应将输出结果写入与该数据源位于同一项目内的数据集中。如果数据源包含个人身份信息（PII）等敏感数据，您可以对该数据源应用标记，该标记将传播至输出数据集。

警告

包含该数据源的项目编辑者可以为该数据源创建新的同步任务、编辑现有同步任务，还可以创建 Webhook 并将其配置为在 Foundry 的其他地方使用。

请注意，如果数据源凭据允许，同步任务可能会更改源系统；例如，从目录或 S3 数据源中删除文件，或者通过任意 SQL 语句删除数据库中的数据。

您只应将数据源（或包含该数据源的项目）的编辑访问权限授予那些您同样愿意授予其对数据源访问数据所用账户拥有完全访问权限的用户。

插件和 JDBC 驱动程序

针对插件和 JDBC 驱动程序，推荐两种权限设置方案：每个组织将所有资源放置在单个项目中，或者将每个 JDBC 驱动程序/驱动放置在包含一个或多个所需该驱动的智能体的项目中。

方案一：每个组织将所有插件和驱动程序放置在单个项目中。

如果由同一组用户管理所有智能体，则推荐此方案。这使得智能体管理者能够访问所有已上传的插件和驱动程序。仅当所有智能体管理者对插件和驱动程序拥有同等访问权限时，才应选择此方案。

实施此方案的推荐流程如下：

  1. 为每个智能体创建一个文件夹。
  2. 将当前的插件和驱动程序迁移到相应的智能体文件夹中。
  3. 为每个插件和驱动程序选择一份副本作为各资产的权威副本，并将其移动到项目根目录下。
  4. 将每个智能体迁移为使用所需插件和驱动程序的根目录副本。
  5. 弃用各个单独的智能体文件夹。

方案二：将每种插件和驱动程序类型放置在各自独立的项目中。

这种机制允许对部分“敏感”插件或驱动程序进行独立于其他组件的单独授权。若采用此方案，务必在各项目中为相关的智能体数据主人授予必要权限，以便其在智能体上正常调用这些组件。
