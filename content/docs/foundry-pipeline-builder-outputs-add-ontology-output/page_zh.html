添加本体论输出

您可以选择添加一个本体论（Ontology）输出，以引导您的管道（pipeline）集成，从而生成定义您全局本体论新元素的干净、结构化的数据。添加输出指导可以帮助节省计算检查的时间，并有助于指导您工作流的端到端创建。

Pipeline Builder 目前支持从批处理和流式数据集创建本体论（Ontology）对象和关联。然而，在 Pipeline Builder 中，无法对由流式数据集支持的对象启用编辑或配置多对多关联。

添加对象类型输出

有两种方式可以添加对象类型输出。您可以通过管道（pipeline）输出面板创建一个空白对象，或者在与转换节点连接的图中创建一个已填充的对象。

使用管道输出面板

打开位于Pipeline Builder右侧的管道输出面板，然后选择对象类型 > 添加以创建一个空白的对象模板，在其中可以编辑对象名称、图标和属性。

请注意，复数名称和对象类型ID将从名称自动填充，以便于使用。

您可以在Pipeline Builder中定义新对象，但不能修改或删除现有对象，除非它们是在Pipeline Builder中创建的。首次部署后，对象类型ID无法再修改。

现在您可以选择手动输入数据来为您的对象填充数据。 或通过将现有数据集或转换的输出拖动到此节点的输入中。

通过图表

选择您要创建对象的转换节点，然后选择添加输出 > 新对象类型。

如果您的注册包含多个本体论（Ontology），则在创建对象时需要选择一个对象所属的本体论（Ontology）。否则，将会显示错误信息 An ontology must be specified to publish ontology type outputs。要设置本体论（Ontology），请在底部面板或对象上选择设置本体论（Set ontology），如下所示。

在对象类型上启用编辑

如果您在Pipeline Builder中创建了对象，并且该对象由批量数据集支持，您可以在Pipeline Builder或Ontology Manager中启用对象类型的编辑。

流支撑的对象类型 目前不支持编辑。要启用流支撑的对象类型上的编辑，请使用 直接数据源 功能。

要启用编辑，请将鼠标悬停在右侧面板中的对象类型上并选择 编辑。切换 允许对这类对象进行编辑。保存并部署您的管道以使更改生效。

启用直接数据源 [测试版]

测试版

直接数据源目前处于< a href="/docs/foundry/platform-overview/development-life-cycle/">测试版阶段，可能在您的注册中不可用。在开发过程中功能可能会发生变化。

直接数据源提供低延迟、低吞吐量的写入操作到本体论（Ontology），包括基于流式数据的< a href="/docs/foundry/object-edits/overview/">对象类型编辑。

要启用一个直接数据源：

  1. 创建一个具有对象类型输出的流管道。您应该有选项来 使用直接数据源来支持此对象类型 和 允许对此类型的对象进行编辑。启用这两个设置。
  2. 根据需要配置对象目标，然后部署。一旦部署完成，该对象类型将出现在< a href="/docs/foundry/ontology-manager/overview/">本体论管理器中，并且其对象可以在< a href="/docs/foundry/insight/overview/">洞察和< a href="/docs/foundry/object-explorer/overview/">对象浏览器中查看。

了解有关部署后监控和调试直接数据源的更多信息。.

如果您 放弃 由直接数据源支持的对象目标，则图中被放弃的节点将显示为流数据集。但是，此节点表示的是支持对象类型的直接数据源，而不是流数据集。与流数据集不同，直接数据源不会直接存储数据；数据存储在本体论中。放弃后，您仍然可以导航到本体论管理器或洞察中的对象类型。

已知限制

作为一个处于积极开发中的测试功能，直接数据源目前有以下限制：

  • 重放会清除对象视图： 重放流式管道会移除现有的对象视图以立即提供新视图。由于切换是即时的，新视图尚未完全填充，因此对象视图可能会暂时不完整或为空。完整的视图将随时间逐渐填充。针对此限制的重放功能正在开发中，此文档将相应更新。
  • 初始部署超时： 流式作业在初始部署时可能在提交前失败，或者在使用重置输出进行重放时失败，因为部署期间执行的初始设置可能导致超时。
  • 不支持分支： 直接数据源不能用于分支。
  • 回填可能较慢： 重放后回填数据可能较慢。正在开发改进措施。

有关补救步骤和其他常见问题，请参见

直接数据源常见问题。

添加链接类型输出

一旦添加了对象类型输出，就可以定义链接类型输出。与对象输出类似，这些可以在面板或图表中创建。

使用管道输出面板

在 管道输出 面板中，选择 链接类型 > 添加。这将打开一个表单，您可以在其中定义链接类型的左右两边。

在 Pipeline Builder 中定义的链接只能使用在同一管道内创建的对象。在 Ontology Manager 应用程序本身中，您可以创建在 Pipeline Builder 中创建的对象和在 Ontology Manager 中创建的对象之间的链接。
首次部署后，链接类型ID或源和目标对象类型将无法再进行修改。

图例

链接类型ID：这是一个自动生成的ID，唯一地表示此链接。

选择一个对象类型输出：从下拉菜单中选择一个对象类型。如果您在管道中没有对象输出，您需要先配置一个对象输出类型。

基数：选择链接类型的每一侧的基数。支持两种基数：一对一和多对多。目前不支持一对一链接类型。在下面的例子中，假设有两个通过基数相关的对象类型：一个 Aircraft 对象类型和一个 Flight 对象类型。

  • 一对多基数：这表示一个 Aircraft 可以与多个 Flights 相关联。
  • 多对多基数：这表示一个 Aircraft 可以链接到多个 Flights，并且一个 Flight 可以链接到多个 Aircraft。
    • 请注意，当您选择多对多基数时，会看到一个提示，要求您 点击并拖动数据集或转换的输出以用作连接表。此表应包含第一个对象类型（本例中为 Aircraft）和第二个对象类型（本例中为 Flight）主键的所有关联组合。
    • 以下是一个多对多基数的示例，其中一个 Class 对象可以链接到多个 Student 对象，而一个 Student 对象可以链接到多个 Class 对象：

主键/列：选择要用于创建链接左侧的属性。

  • 在一对一基数链接类型中，主键将由对象类型确定。
  • 在多对多基数的链接类型中，选择连接表中的哪些列映射到每个关联对象类型的主键。

外键/列：选择要用于创建链接右侧的属性。

  • 在一对多基数的链接类型中，一个对象类型的外键属性必须引用另一个对象类型的主键属性。
  • 在多对多基数的链接类型中，选择连接表中哪些列映射到每个关联对象类型的主键。

名称：为链接类型的每一侧填写显示名称。链接类型的某一侧表示到该对象类型的链接。Aircraft 对象类型的显示名称描述了从 Flight 到 Aircraft 的链接。在此示例中，您可以选择显示名称“分配的飞机”，因为一个 Flight 有一个分配的 Aircraft。

复数显示名称： 对于具有多对多基数的链接类型的一侧，您还将被提示填写一个复数显示名称，以便用户应用程序在显示链接对象时可以显示正确的名称。在我们的示例中，Flight 对象类型的复数显示名称将描述从 Aircraft 到 Flight 的链接。我们可能会选择复数显示名称为“已安排的航班”，因为一个 Aircraft 有多个已安排的 Flights。

解决本体论验证错误

用户可能对管道进行更改，导致本体论中出现验证错误。一些错误示例包括：

  • 在Pipeline Builder和Ontology Manager中对同一对象类型或链接类型进行了冲突的更改。
  • 更改了被管道外的链接类型引用的对象类型的主要键。
  • 对对象类型的模式进行了重大更改，使得需要进行 模式迁移。

任何错误都会在右侧面板下的部署此管道部分显示。在错误选项卡中，会有一条消息描述在部署之前需要进行哪种类型的解决。

在您能够部署管道之前，必须解决这些错误。

当解决按钮显示箭头时，选择它会将您带至本体论管理器以解决验证错误。否则，您可以在Pipeline Builder中直接解决这些错误。

在本体论管理器中解决合并冲突

按照本体论管理器中的提示应用合并冲突的解决方案。如果您的更改会影响潜在的编辑，您可以决定是放弃编辑还是使您的编辑与对象的最新版本兼容。否则，无需额外的说明即可解决合并冲突，您可以选择应用。

应用更改后，本体论（Ontology）将更新为最新更改，您可以选择 返回到管道。

在解决冲突后，您可能会偶尔遇到错误 Ontology type outputs are out of date with the ontology 返回到您的管道时。这是因为管道尚未获取通过上述冲突解决工作流对您的本体论（Ontology）输出所做的更改。

选择 解决 应该可以修复此问题，并允许您部署您的管道。

在Pipeline Builder中解决冲突

一些本体论验证错误不需要切换到本体论管理器。在下面的示例中，需要进行模式迁移，这发生在Pipeline Builder中的对象类型模式与本体论主分支上的最新模式发生更改时。选择解决直接在Pipeline Builder中选择迁移策略。

对于受影响的属性，您可以选择是否要 从属性中删除编辑 或迁移您的编辑，以使其与属性的最新版本兼容。

一旦您选择了最佳方法，请选择 应用迁移。

将对象类型输出重新引入Pipeline Builder

按照以下步骤，您可以将由数据集输出支持的对象类型从您的管道中带入Pipeline Builder：

在右侧的面板中，选择Pipeline outputs旁边的齿轮图标。

选择替换为对象。这将打开用本体论类型输出替换数据集对话框。

选择您希望在管道中转换回相应对象的数据集输出，并选择导入所选输出。

保存您的更改并部署您的管道。

在Pipeline Builder中解除本体论类型输出 [已弃用]

在Pipeline Builder中取消对本体论类型输出的拥有权，将把这些本体论类型的管理权转移到本体论管理器。这意味着对这些本体论类型的任何未来修改只能在本体论管理器中进行。当您需要Pipeline Builder不支持的对象类型配置时，例如共享属性类型或运行时生成的属性，这会很有用。要取消对本体论类型的拥有权，请选择您想要取消拥有的对象类型。任何引用所选对象类型的链接类型将自动转移到本体论管理器。

只有在main分支的最新保存版本成功部署后，才能将本体论类型输出替换为数据集输出。

选择取消对选定对象和链接类型的拥有权。

在此管道中，对象类型输出节点将转换为数据集 输出节点，这些节点在本体论管理器中支撑对象类型。从右侧的面板中，通过选择输出旁边的图标来导航到本体论管理器中的对象类型。

在放弃所选对象后部署您的管道。

支撑数据源中的列名必须与给定对象类型的属性配置的支撑列名完全匹配。列名不匹配将导致数据缺失或索引作业失败。例如，一个具有支撑列 routeId 的属性需要在支撑数据源中有一个名为 routeId 的列。

配置细粒度安全权限

您可以通过使用受限视图策略或对象安全策略来对对象配置细粒度安全权限：

  • 受限视图 （RVs）可实现行级安全。
  • 对象安全策略可同时实现行级和列级安全。

与Ontology Manager中的对象安全策略相比，Pipeline Builder中的对象安全策略目前仅支持部分功能。请参见下方支持的功能。

要设置您的细粒度安全策略，请转到输出对象并选择配置细粒度安全。

在弹出窗口中选择受限视图策略或对象安全策略。

对象安全策略

对象安全策略 选项允许您使用行级策略、属性安全策略、标记删除管理和分类更改来配置单元格级别的细粒度安全。

默认情况下，对象安全策略应用于所有属性。当属性安全策略包含一个属性时，用户必须同时通过对象安全策略和属性安全策略才能查看该属性的值。

要添加行级策略，请选择细粒度安全旁边的加号（+）图标。

在策略编辑器中，您可以创建规则来比较属性值、用户属性或常量值，以定义您的安全条件。

请注意，如果基于标记或分类用户属性创建策略，请确保值类型设置为标记或分类。

要强制正确的类型，请使用

Logical Type Cast expression in the transform node.

Additionally, markings and classifications must be wrapped in an array for compatibility with the ontology. Mapping a single marking or classification directly to an object property will result in an error.

Markings must use the associated marking ID. Learn more about the expected format of marking IDs.

After configuring your granular security policy, select Back (bottom left) to return to the main configuration window. If no further changes are needed, select Apply to save your policy.

You can go to Mandatory controls to remove any inherited markings or Classification to adjust the classification.

Pipeline Builder does not currently support adjusting classifications for manually entered tables.

Property security policies can be added in a similar way. Select Add in the Property security policies section. This will create a new property security policy. To configure the property security policies, select the arrow on the policy you want to edit.

Under Policy name, you can rename your property security policy. Select one or more columns you want to apply this policy to under Properties affected, and then configure the granular security, mandatory control, or classification in the same way you would do so above.

After configuring your property security policy, select Back (bottom left) to return to the main configuration window. Use Apply to save your policy.

Restricted view policy

To configure which rows a user can view on your output, you can create a restricted view policy to apply to your object. The policy is backed by a Restricted View, which is built after your changes are merged and deployed to main.

You can configure your policy directly in the restricted view policy window. The same rules for markings and classifications apply here as stated above. Learn more about restricted view policies and best practices. You also have the option to convert the restricted view policy into an object policy for greater configurability.

Once you have finished creating your granular security policy, save your pipeline and propose your changes. The backing security policies will only be created and go into effect after you have deployed successfully on the main branch.
