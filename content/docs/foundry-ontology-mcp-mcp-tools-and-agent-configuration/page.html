MCP tools and agent configuration

Action tool description

Use the Agent tool description field in the Ontology Manager application to update the description the agent sees when using the action as a tool. This allows you to provide specific guidance to AI agents about when and how to use each action. For example, an action that creates a new task and gets the project ID for linking the tasks could include instructions on how to obtain the project ID.

Using Ontology MCP with Claude skills

Claude skills ↗ are reusable instruction sets that extend what Claude can do. You can integrate Ontology MCP (OMCP) tools with your skills to encode more complex business logic into your agent.

In the example below, a skill named get-or-create-task guides the agent on how to use both the search tool and the create task action tool in combination. This prevents duplicate tasks from being created by first searching for an existing task before creating a new one.

Copied!
1---
2name: get-or-create-task
3description: Searches for a task by title in a project, returns it if found, or creates a new task if not found
4---
5
6# Get or create task
7
8Find an existing task by title within a project, or create a new task if no match is found.
9
10## Instructions
11
121. Search for the task by title using the search tool:
13   - Tool: `search-osdk-todo-task`
14   - Filter results by `project_id` if provided
15
162. Evaluate results:
17   - If a match is found, return the existing task details
18   - If multiple matches are found, use the first match
19   - If no matches are found, proceed to create a new task
20
213. Create a new task using the create task action tool:
22   - Tool: `create-osdk-todo-task`
23   - Required parameters: `title`, `project_id`
24   - Optional parameters: `description`, `status`, `assigned_to`, `start_date`, `due_date`
25
264. Return the task details including ID, title, project, status, and assignee

Microsoft Copilot Studio integration

Microsoft Copilot Studio integration only supports authorization code grant in a Confidential Client. This means that when creating the Developer Console application for your Ontology MCP integration with Microsoft Copilot Studio, you should choose Backend service and User's permissions. This will create the required service user that Copilot Studio uses to issue the token on behalf of your users.

Using Adaptive Cards to format and display Ontology MCP data

Developed by Microsoft, Adaptive Cards ↗ are a versatile UI framework for Teams, Copilot, and Outlook integrations. When paired with Ontology MCP tools, Adaptive Cards let your agent return structured data from your ontology and display it as a formatted card rather than plain text.

Additionally, you can use Action.OpenUrlDialog in your Adaptive Cards to trigger a URL that opens a custom dialog in Teams or Copilot, allowing users to interact with the data returned from your Ontology MCP tools.

Prerequisites

Before you begin, install the following VS Code extensions:

  • Microsoft 365 Agents Toolkit ↗
  • Adaptive Card Previewer ↗

Configure your Ontology MCP server in VS Code

Create a .vscode/mcp.json file in your project to connect VS Code to your Ontology MCP server:

Copied!
1{
2  "inputs": [],
3  "servers": {
4    "todo-confidential": {
5      "type": "http",
6      "url": "<your-ontology-mcp-server-url>",
7      "headers": {
8        "Authorization": "Bearer ${env:FOUNDRY_TOKEN}",
9        "Accept": "application/json, text/event-stream"
10      }
11    }
12  }
13}

Replace <your-ontology-mcp-server-url> with your Ontology MCP server's URL, which you can find in your Developer Console application.

Create an adaptive agent

  1. Select Start in the Microsoft 365 Agents Toolkit panel and connect to your Ontology MCP server.

  1. Follow the guided steps in the Microsoft 365 Agents Toolkit to create a new adaptive agent.

Add capabilities to your agent

To render an Adaptive Card from the Ontology MCP tool output, add a capabilities block to your agent's tool definition. The static_template property points to an Adaptive Card template file that defines how the returned data is displayed.

The following example configures a GetCustomerList tool to render its response using an Adaptive Card:

Copied!
1{
2    "name": "GetCustomerList",
3    "description": "List all customers.\n\n    Returns:\n        A list of CustomerSummary objects with key customer properties.",
4    "parameters": {
5        "type": "object",
6        "properties": {},
7        "required": []
8    },
9    "capabilities": {
10        "response_semantics": {
11            "data_path": "$",
12            "static_template": {
13                "file": "adaptiveCards/customerList.json"
14            }
15        }
16    }
17}

The data_path property specifies the JSON path to the data in the tool response, and static_template.file points to the Adaptive Card template that renders the data.

Optional: Generate Adaptive Cards with a Claude skill

You can create a Claude skill that automatically generates Adaptive Card templates based on the schema of your Ontology MCP tools. First, follow the instructions on the Developer Console MCP page to connect your Ontology MCP server to Claude Code. Then, save the following skill file to your project's .claude/skills directory, such as the .claude/skills/build-adaptive-card.md example below:

Copied!
1---
2name: build-adaptive-card
3description: Build an Adaptive Card for a specific MCP tool connected to Claude Code
4---
5
6# Build Adaptive Card for MCP Tool
7
8You are helping the user build an Adaptive Card for a specific MCP tool that is connected
9to Claude Code in this session.
10
11## Instructions
12
13Follow these steps precisely:
14
15### Step 1: Identify the target tool
16
17Ask the user which MCP tool they want to build an Adaptive Card for. List the available
18MCP tools you know about from the connected MCP server. If the user has already named the
19tool, skip asking.
20
21### Step 2: Call the tool to get a real response sample
22
23Call the named MCP tool with minimal or sample arguments to get a real response. If the
24tool requires arguments, use the most minimal and representative ones possible. Capture the
25full JSON response.
26
27If the tool requires arguments you cannot guess, ask the user for sample argument values
28before calling.
29
30### Step 3: Validate the response shape
31
32Inspect each property of the returned object. If any property value is a string that is
33itself valid JSON (starts with `[` or `{` and parses as JSON), the tool is double-serializing
34its output. Stop and tell the user:
35
36> The tool is returning its data as a JSON-encoded string instead of a native JSON
37> object or array. Adaptive Card templates cannot bind to a string. Fix the MCP server
38> tool implementation to return a native Python dict or list instead of `json.dumps(...)`.
39
40Do not proceed until the user confirms the tool has been fixed.
41
42If the response is valid JSON, determine:
43- Is the top-level response an object with a `value` array? (MCP envelope pattern)
44- Or is it a flat array or a single object?
45- What are the field names, types, and which fields are nullable?
46- Which fields are numeric and might benefit from formatting (currency, percentages)?
47
48### Step 4: Generate the Adaptive Card JSON
49
50Create an Adaptive Card (version 1.5) following these rules:
51
52**Template binding:**
53- If response is `{"value": [...]}`, use `"$data": "${value}"` on the repeating Container
54- If response is a flat array, use `"$data": "${$root}"` on the repeating Container
55- If response is a single object, bind fields directly with `${fieldName}`
56
57**Card structure:**
58- Start with a header Container (`style: "emphasis"`, `bleed: true`) with a title TextBlock
59- For list responses: use a repeating Container with a ColumnSet inside
60- For single-object responses: use a FactSet or labeled rows
61- Use `"wrap": true` on all TextBlocks
62- For nullable fields, add `"$when": "${field != null}"` guards
63- For currency fields: use `formatNumber` for display formatting
64- For status color coding: use conditional `if()` expressions
65- Do NOT use `.length` on arrays (not supported in template language)
66- Do NOT use date functions, structs, arrays, or window functions
67
68**File naming:** `appPackage/adaptiveCards/<toolName>.json` (camelCase the tool name)
69
70Write the card JSON to `appPackage/adaptiveCards/<toolName>.json`.
71
72Next, write a companion `.data.json` file at `appPackage/adaptiveCards/<toolName>.data.json`
73containing the actual sample response for design-time preview data.
74
75### Step 5: Wire up response_semantics in ai-plugin.json
76
77Read `appPackage/ai-plugin.json`. Find the function entry matching the tool name. Add or
78update the `capabilities` block:
79
80```json
81"capabilities": {
82    "response_semantics": {
83        "data_path": "$",
84        "static_template": {
85            "file": "adaptiveCards/.json"
86        }
87    }
88}
89```
90
91### Step 6: Report to the user
92
93Print a summary of the card file created, the data sample file created, the function updated
94in `ai-plugin.json`, the `data_path` used, and key design decisions made.
95
96## Key constraints
97
98- `response_semantics` must be inside `capabilities`, not directly on the function
99- `static_template` uses `"file": "adaptiveCards/foo.json"` (the toolkit inlines at build time)
100- Adaptive Card template language does not support `.length`
101- `$schema` for the card: `"http://adaptivecards.io/schemas/adaptive-card.json"`
102- Card version: `"1.5"`
