单元测试

本页所述的 Python 仓库单元测试仅适用于批处理管道，不支持流处理管道。

Python 仓库可以选择在检查环节运行测试。这些测试使用流行的 Python 测试框架 pytest ↗ 来运行。

CI 任务：condaPackRun

所有 CI 检查均包含 condaPackRun 及其他任务。

condaPackRun 任务负责安装环境。每个构件都从相应的频道中获取，conda 随后使用这些构件来构建环境。

该任务包含以下三个阶段：

  1. 下载并解压已解析环境中的所有包。
  2. 验证包内容。根据配置情况，conda 将使用校验和或验证文件大小是否正确。
  3. 将包链接到环境中。

环境规范会作为缓存存储在以下隐藏文件中，供后续构建使用。

  • conda-version-run.linux-64.lock
  • conda-version-test.linux-64.lock

缓存会保留 7 天。如果 meta.yaml 文件发生任何更改，缓存将被重新生成。

该任务的执行速度很大程度上取决于向仓库中添加了多少个包。添加的包越多，任务运行速度就越慢。

启用样式检查

可以通过在 Python 项目的 build.gradle 文件中应用 com.palantir.conda.pep8 和 com.palantir.conda.pylint Gradle 插件来启用 PEP8 和 Pylint 样式检查。对于 transforms 仓库，该文件位于 Python 子项目中；对于库仓库，该文件位于根文件夹下。

以下是一个 transforms 的 build.gradle 文件示例：

可以在 Python 项目的 src/.pylintrc 文件中配置 Pylint。例如，可以按照以下方式禁用特定消息：

[MESSAGES CONTROL]
disable =
    missing-module-docstring,
    missing-function-docstring
Pylint 的限制

并非所有 Pylint 配置都能保证在 Foundry 中正常工作。如果 src/.pylintrc 中的某项功能未在检查中显示，说明该功能不受支持。

编写测试

Pytest 会从以 test_ 开头或以 _test.py 结尾的 Python 文件中识别测试用例。我们建议将所有测试放置在项目 src 目录下的 test 包中。测试本质上是以 test_ 为前缀命名的 Python 函数，其中包含通过 Python assert 语句实现的断言。Pytest 同样能够运行基于 Python 内置 unittest ↗ 模块编写的测试。有关编写测试的详细信息，请参阅 pytest 文档 ↗。

并行化测试

默认情况下，pytest 会串行执行测试。为了提升测试运行效率，可以将测试任务分发到多个 CPU 核心并行处理。

修改 transforms 的 build.gradle 文件，将 numProcesses 设置为所需使用的进程数即可。


```python
apply plugin: 'com.palantir.transforms.lang.python'
apply plugin: 'com.palantir.transforms.lang.python-defaults'

// 应用测试插件
apply plugin: 'com.palantir.transforms.lang.pytest-defaults'

tasks.pytest {
    numProcesses "3"
}
```

测试并行化是通过使用 pytest-xdist ↗ 测试插件来运行的。

并行化测试涉及将待处理的测试发送给任何可用的 worker，无法保证执行顺序。对于需要依赖全局或共享状态，并且预期其他先行测试会对其进行修改的测试，应当进行相应调整。
