推荐的项目与团队结构

本页概述了在 Foundry 中使用项目构建数据管道的推荐方式，旨在实现：

  • 在整个管道中构建结构清晰的权限体系
  • 跨角色的高效协作
  • 清晰且易于维护的管道结构

此外，我们还涵盖了在生产环境中成功管理管道所需履行的角色与职责范围。

本文提供了对各个管道阶段及其用途的概念性概述。如需了解在 Foundry 中分步设置此结构的详细指南，请参阅平台安全文档中的《保护数据基础》。

管道阶段

这些阶段定义了构成有序管道的项目逻辑分离。我们将在下面逐一详细说明。

  1. 数据连接：同步任务将源系统的原始数据落入数据源项目中。
  2. 数据源项目：每个逻辑数据源对应一个数据源项目，用于定义每个原始数据集的基本清洗步骤并应用一致的 schema。
  3. 转换项目：从一个或多个数据源项目导入数据集，并进行转换以生成规范的、可复用的数据集。
  4. 本体项目：从一个或多个转换项目导入数据集，并进行转换以生成代表离散操作对象的规范表。通常，单个本体项目会将特定操作域的相关对象集合归为一组，以便于管理。
  5. 工作流项目：工作流项目从本体项目导入数据，以实现特定目标。这通常涉及操作工作流、数据科学研究、商业智能分析与报告，或应用开发项目。

每个管道阶段都是一个独立的单元，其输出的是可供下游项目导入并用于其他用例、管道开发、分析等场景的数据集。在每个项目内部，负责团队除了要实现转换步骤外，还应管理其输出的稳定性和完整性。这包括管理构建计划、配置和监控数据健康检查，以及在相关情况下编写单元测试或额外的数据完整性测试。

以下各节将详细介绍每种下游项目类型，以追踪数据在管道中的流转过程。在设计管道时，应从本体层反向推导，以确定需要连接的源系统和需要实现的管道转换。

项目提供了一种在 Foundry 中组织数据和代码的方式。它们是管理围绕特定工作范围进行协作的个人和团队访问权限与许可的主要单元。此外，项目还提供了捕获和共享元数据的功能，例如项目活动的事件日志、项目文档空间，以及项目资源消耗和计算指标。

在思考什么是一个好项目时，可以类比什么是一个好的微服务：明确的目的、供下游依赖使用的清晰输出数据集/API，以及由规模足够小的人员群体拥有，以便他们能够有效协调并为项目管理制定自己的标准。

在决定是创建新项目还是在现有项目中构建时，请考虑：

  • 项目与代码仓库应保持 1:1 的比例。范围越大，协作越深入，但也会引入更具挑战性的操作要求，以确保无缝的版本控制。
  • 项目是您为所有相关资源施加文件结构和权限的层级。如果单个概念上的“项目”包含需要单独权限的子组件，请考虑在 Foundry 中创建多个项目。

1. 数据连接

在 Foundry 中流经管道的数据通常源自外部源系统。数据连接服务提供了一个管理界面，用于注册源系统。对于每个源系统，会配置一个或多个独立的同步任务，将数据落入数据源项目的raw数据集中。

为了能够初步配置源系统或同步任务，智能体管理员通常需要配置一个数据连接智能体。每个智能体应存储在一个仅限智能体管理员访问的专用项目中。

管理新数据源的新连接是一项跨部门的工作，需要数据源数据主人、智能体管理员和数据源开发者协同合作，不过通常还需要合规性/法务专员签字批准将数据从源系统移动到新环境。

源系统配置完成后，数据源开发者即可独立工作，配置各项同步任务。这极大减少了团队间的反复沟通，并确保连接新系统是一项低成本的、一次性的工作。

对于每个连接，一个关键的考量是在 SNAPSHOT（每次同步替换当前数据）和 APPEND（每次同步向现有数据累加）之间做出选择。要更深入地理解这些概念以及它们对构建高效管道的影响，请参阅《增量管道》部分。

延伸阅读

有关数据源和同步的原则、架构及配置的更多信息，请访问《数据连接文档》。

有关监控连接并确保流入管道的数据准确且符合预期的更多思考，请查阅《维护管道》部分。

2. 数据源项目

每个源系统都应将数据落库至平台内匹配的项目中。常规模式是以尽可能“原始”的格式落库每次同步的数据。向“清洗后”数据集转换的步骤定义在项目的代码仓库中。

建立这种“每个数据源对应一个项目”的模型有诸多便利优势：

  • 鼓励在一致的位置探索新数据源。
  • 提供统一的位置来指定对数据源的访问控制。
  • 最大程度降低不同团队重复同步相同数据的风险。
  • 代码仓库体积小巧且具有针对性，专门用于处理源系统数据的类型和格式。
  • 允许在管道层访问数据之前，对数据进行匿名化、清洗和标准化。

数据源项目目标

在每个数据源项目内部，目标是为整个组织内广泛的用户、用例和管道准备好同步后的数据以供消费。

输出数据集及其模式应被视为一种 API，也就是说，它们应当在逻辑上组织有序且随时间推移保持稳定，因为数据源项目的输出数据集是所有下游工作的基石。输出应与源表或源文件保持 1:1 的映射关系；数据集中的每一行都应在源系统中有直接对应的行。

为此，从原始数据到清洗数据转换的一些典型目标包括：

  • 采用一致的列命名规范。
  • 确保为列应用合适的数据类型。
  • 处理缺失、格式错误或类型不匹配的值。
  • 建立主要的《健康检查》。
  • 移除不适合通用场景的《个人身份信息（PII）或其他敏感数据》。

即使源系统提供了列数据类型信息，有时也有必要将值以 STRING 类型导入。需要特别注意 DATE、DATETIME 和 TIMESTAMP 类型，这些类型在源系统中常以非标准格式和数值类型表示，偶尔会引发问题。如果这些类型在源系统中的格式确实不可靠，以 STRING 类型导入可提供一种选项，以便实现更稳健的解析并定义逻辑来处理格式错误的值或丢弃不可用或重复的行。

除了这些程序化步骤外，清洗后的数据集还应得到严谨的文档记录。对数据集的定性描述、源系统出处、相关联系人及管理协议，以及适用时描述数据特征的逐列元数据，这些都将确保未来的开发者能够正确使用数据。

实施数据源项目

  1. 系统管理员为目标数据源创建一个新项目，并将相应的数据源开发者添加到新的《用户组》中来管理该项目。
    • 只有特定数据源的开发者组成员才应拥有在项目内构建数据集的权限。
  2. 数据源数据主人和智能体管理员协同合作，在源系统上《部署数据连接智能体》，并在界面中配置数据源连接。
    • 确保配置适当的权限，以便数据源开发者能够预览源系统中的表和文件。
  3. 数据源开发者为项目中的一个或多个原始数据集配置同步，需考虑正确的同步频率和同步类型，并为每个原始数据集添加适当的数据健康检查。
  4. 数据源开发者执行清理转换操作，生成可供其他项目使用的“干净”数据集。
  5. 下游开发者通过向项目或特定干净数据集提交工单，来请求源系统中的新数据集、对干净数据集的增强，或反馈数据质量问题。

尽管每个数据源都有其独特性，但清理和准备源系统的步骤往往存在共性，例如将原始字符串值解析为特定类型以及处理错误。当数据源包含大量相似的清理转换操作时，最佳实践是使用 Python 定义一个库，以提供一套一致的工具并减少代码重复。

推荐的文件夹结构

  • /raw：用于来自数据连接同步的数据集。
  • /processed（可选）：用于文件级转换，从非表格文件创建表格数据。
  • /clean：用于与 /raw 中数据集保持 1:1 对应关系，但已应用清理步骤的数据集。
  • /analysis：用于为测试或记录清理转换操作而创建的资源，并展示数据的形态。
  • /scratchpad：在构建或测试清理转换操作过程中创建的任何临时资源。
  • /documentation：数据血缘关系图表，用于展示清理的流水线步骤，以及使用 Foundry Notepad 编写的额外文档（超越顶层项目 README 的部分）。

3. 转换项目

转换项目的目标是封装一组共享且具有语义意义的数据，并生成规范视图以输入到本体层。

这些项目从一个或多个数据源项目导入已清理的数据集，将它们与查找数据集连接以扩展值，规范关系或反规范化关系以创建以对象为中心或以时间为中心的数据集，或者聚合数据以创建标准的共享指标。

推荐的文件夹结构

  • /data
    • /transformed（可选）：这些数据集是转换项目中中间步骤的输出结果。
    • /output：这些数据集是转换项目的“输出”结果，也是下游本体项目中唯一应依赖的数据集。
  • /analysis：用于为测试或记录清理转换操作而创建的资源，并展示数据的形态。
  • /scratchpad：在构建或测试清理转换操作过程中创建的任何临时资源。
  • /documentation：数据血缘关系图表，用于展示清理的流水线步骤，以及使用 Notepad 编写的额外文档（超越顶层项目 README 的部分）。

4. 本体项目

本体力图在整个 Foundry 平台中贯彻统一的通信层，而精心梳理一个干净、有序且稳定的本体是重中之重，这能确保在丰富通用运营全景图的同时，推动大量有价值的项目并行推进。有关设计和管理本体的概念与实践详情，请参阅《本体概念文档》。

本体项目是任何流水线的核心节点，代表了生成数据集所需的最终转换，这些数据集需符合本体中定义的单个对象或相关对象组的定义。这些项目还会（独立）生成同步的数据集，用于支持对象浏览器视图。

由于本体数据集代表了它们所描述的运营对象的规范事实，因此它们构成了所有“消费”项目的起点。尽管上游清理和流水线步骤的数据溯源与转换逻辑是可见的，但在概念层面上，对于那些仅从本体项目消费数据的项目开发者、分析师、数据科学家和运营用户而言，这些步骤和中间数据集可能是一个黑盒。从这个意义上说，本体层充当了运营对象的 API。

实施本体项目

与数据源项目类似，维护本体项目的关键因素包括：

  • 完善的文档
  • 有效的健康检查
  • 定期的调度计划
  • 在《数据目录》中进行梳理并添加适当的标签

这些内容将在《管道维护》以及下文关于记录项目文档的提示框中进行更详细的讨论。

此外，随着本体日益健壮且参与贡献本体层的团队不断增多，维护数据集“API”的完整性变得愈发关键。为此，应考虑实施额外的检查机制，以确保提议的变更能够保持输出数据集模式的完整性。后续即将发布的文档将对此展开深入探讨。

推荐的文件夹结构

  • /data
    • /transformed（可选）：这些数据集是本体项目中中间步骤的输出结果。
    • /ontology：这些数据集是本体项目的“输出”结果，也是下游用例或工作流项目中唯一应当依赖的数据集。
  • /analysis：为测试或记录清理转换而创建的资源，用于展示数据的形态。
  • /scratchpad：在构建或测试清理转换过程中创建的任何临时资源。
  • /documentation：展示清理管道步骤的数据血缘关系图谱，以及以报告形式编写的额外文档（超越顶层项目 README 的内容）。

5. 工作流项目

工作流项目也称为用例项目，其设计应针对当前语境保持灵活性，但通常围绕单一项目或团队构建，以形成高效的协作单元，并划定职责与访问权限的边界。

通常，工作流项目应引用本体项目中的数据，以确保运营工作流、商业智能分析和应用项目都能共享统一的业务视图。如果在开发工作流项目时发现本体层提供的数据源不足以作为来源，这表明需要丰富和扩展本体。避免引用管道中靠前（或靠后）环节的数据，因为这会导致特定类型数据的单一事实来源发生碎片化。

项目文档记录

每个项目在开发过程中都应配备详尽的文档记录。以下是一些常见的模式与最佳实践：

  • 在每个项目的根目录保存一个或多个数据血缘关系图资源，以捕获最重要的数据集与管道关系。不同的图表可以采用不同的配色方案来突出相关特征，例如数据刷新的规律性、相关数据集的分组方式，或是数据集的大小等特征。
  • 为项目添加简短描述——该内容直接显示在项目标题下方，将用于所有项目的列表视图中。重点应放在为不熟悉该项目的用户提供背景信息。
  • 使用顶层项目文件夹右侧边栏的添加文档按钮来编写 Markdown 语法。这是解释项目目的、主要输出以及项目使用者所需其他相关信息的最佳位置。
  • 将更多的全局项目文档放入本地创建的 Markdown (.md) 文件中，并上传至文档文件夹或《记事本》中，然后从顶层文档链接至这些文件。
  • 在项目的代码仓库中添加 README.md 文件，并确保每个转换文件都包含描述和额外的内联代码注释。

推荐的文件夹结构

工作流项目的结构将比其他项目类型更加多样化，应侧重于确保主要资源能够被即时访问并具备完善的文档。

  • /data
    • /transformed（可选）：这些数据集是工作流项目中中间步骤的输出结果。
    • /analysis（可选）：这些数据集是分析工作流的输出结果。
    • /model_output（可选）：这些数据集是模型工作流的输出结果，随后可对其进行分析以确定模型的拟合度。
    • /user_data（可选）：如果工作流允许用户创建自己的数据“切片”，请将其存储在此处的用户专属文件夹中。
  • /analysis：为驱动决策和反馈循环而创建的资源与报告。
  • /models：创建的所有模型均应存储于此。
    • /templates：用于存储为项目特定用途而共享的代码工作簿模板。
  • /applications：额外的工作流应用或子应用存储于此。主应用应存储在项目根目录，以便于醒目访问。
    • /develop：处于开发阶段的应用、新功能和模板存储于此。
  • /scratchpad：在构建或测试清理转换过程中创建的任何临时资源。
  • /documentation：数据血缘关系图展示了清理工作的管道步骤，以及报告里编写的额外文档（超出顶层项目 README 的部分）。

管道管理角色

以下角色展示了在 Foundry 中参与管道规划、设计、实现和管理的常见用户画像。并非在所有情况下都需要配备所有角色，尤其是在平台开发初期，而且一个人可能会同时承担多个角色。不过，总体上考虑这些角色描述及其交互方式，有助于构建秩序井然、高效运作的团队。

下图展示了主要角色与其最常活跃的管道环节之间的对应关系。

主要角色

  • 智能体管理员：智能体管理员负责创建和配置数据连接 智能体，并将其共享给相关的 数据源数据主人。这集中了智能体安装和配置的控制权，从而提升了安全性和管控能力。
  • 数据源开发者：数据源开发者是负责各数据湖相关数据源采集流程的工程师。这项工作可以由单一团队完成，也可以按源系统进行拆分。
    • 每个数据源开发者拥有以下内容：(1) 一个具备与数据源技术连接的数据连接数据源；(2) 一个用于接收同步数据的 Foundry 项目。
    • 数据源项目的产出是经过清洗和准备的数据集，可供一个或多个下游用例项目直接使用。数据源开发者还需定义其负责的各数据源的刷新频率，并与 发布经理 或 运营经理 协作，安排 管道构建策略。
  • 数据管道开发者：数据管道开发者是负责构建通用本体层的数据工程师，该本体层为所有用例提供数据资产。数据管道开发者负责定义此本体层，并构建从源结构到本体结构所需的转换逻辑。
  • 发布经理：发布经理拥有生产管道的拥有权，并管理发布流程，以确保生产管道符合组织标准且可靠执行。
    • 对于管道的代码变更，通常通过向主干分支提交 Pull Request 的审批流程，以及批准在主干分支上运行管道构建来完成。为了保证持续的可靠性，发布经理会监控与管道和数据源开发者共同为关键数据集定义的 健康检查。
  • 合规官 / 法务官：合规官负责在用例开发前对其进行审批。此类审批应综合考虑：(1) 需要使用的数据；(2) 项目的目的；(3) 将有权访问数据的用户，并需结合其司法管辖区的数据保护制度和组织自身政策进行评估。
  • 用例开发者：用例开发者是负责用例开发的工程师——包括软件开发人员和/或数据科学家。他们使用从本体层获取的数据，这些数据已由合规官批准供其使用。

其他角色

  • 运营经理：该角色负责监控生产管道的健康执行情况。他们监控不同数据源的数据刷新、不同管道构建的执行情况，并协助对支持请求进行分诊。该角色为 发布经理 提供支持。
  • 权限数据主人 / 身份管理员：在大多数情况下，将用户分配到用户组的操作是在客户的身份管理系统中完成的，相关信息通过 SAML 集成传入 Foundry。在这种情况下，权限管理员负责拥有该流程。
  • 数据官：负责将本体层建模为以用户为中心的视图，拥有数据质量，并寻找应予集成以扩展本体层的新数据源。数据官还负责监控在用例中完成的数据扩充，并将其泛化至本体层以提升复用性。
  • 项目经理：项目经理与参与项目的工程师协作，确保技术与业务目标保持一致，遵循平台开发的最佳实践标准，并协调与其他项目团队的合作。项目经理还负责项目的文档记录工作。
  • 数据源数据主人：这是特定源系统的主要联系人。在配置新的数据连接时，他们需要与智能体管理员密切合作；在此之后，他们需要对“上游”问题承担一定程度的责任——无论是运行在其服务器或文件系统上的智能体服务出现问题，还是数据本身的完整性出现问题。
