推荐的项目和团队结构

本页概述了在Foundry中使用< a href="/docs/foundry/data-integration/data-pipeline/">数据管道（data pipeline）来构建的推荐方式，通过< a href="/docs/foundry/getting-started/projects-and-resources/">项目（projects）实现以下目标：

  • 在管道（pipeline）中跨角色的结构良好的权限管理
  • 跨角色的有效协作
  • 可读性强且易于维护的管道结构

此外，我们还涵盖了需要在生产环境中成功进行管道（pipeline）管理所必须履行的< a href="#pipeline-management-roles">角色与职责（roles and responsibilities）。

这提供了对各种管道阶段的概念性概述以及每个阶段的作用。如需在Foundry中设置此结构的逐步指南，请参阅平台安全文档中的< a href="/docs/foundry/security/securing-a-data-foundation/">确保数据基础的安全性（Securing a data foundation）。

管道阶段

这些阶段定义了构成有序管道的项目的逻辑分离。我们将在下面详细介绍每个阶段。

  1. 数据连接: 将原始数据从源系统同步到一个 Datasource project。
  2. 数据源项目: 每个逻辑数据源对应一个 Datasource project，定义每个原始数据集的基本清理步骤并应用一致的模式。
  3. 转换项目: 从一个或多个 Datasource projects 导入数据集，并进行转换以生成规范的、可重用的数据集。
  4. 本体论项目: 从一个或多个 Transform projects 导入数据集，并进行转换以生成代表离散操作对象的规范表。一个本体论项目通常会将特定操作域中相关的对象集合进行分组，以便于管理。
  5. 工作流项目: 工作流项目从本体论项目导入数据以实现特定目标。这通常是操作性工作流、数据科学调查、商业智能分析和报告，或者应用程序开发项目。

每个管道阶段是一个独立的单元，其输出是可供下游项目导入并用于其他用例、管道开发、分析等的数据集。在每个项目中，负责团队除了实现转换步骤外，还应 管理输出的稳定性和完整性。这包括管理构建计划、配置和监控数据健康检查，并在相关情况下 编写单元测试 或额外的数据完整性测试。

以下各节将更详细地介绍每种下游项目类型，以跟踪数据通过管道的流程。在设计管道的过程中，应从本体论层开始逆向工作，以确定需要连接的源系统和要实施的管道转换。

项目 在Foundry中提供了一种组织数据和代码的方式。它们是管理个人和团队围绕特定工作范围进行协作时访问和权限的主要单元。此外，项目还提供了捕获和共享元数据的功能，例如项目活动事件日志、项目文档空间以及项目资源消耗和计算指标。

在考虑什么是好的项目时，可以类比什么是好的微服务：明确的用途，下游依赖项使用的清晰输出数据集/API，由一组足够小的人负责，他们可以有效地相互协调，并为项目管理设定自己的标准。

在决定是否创建一个新项目或在现有项目中构建时，请考虑以下几点：

  • 项目应与代码仓库一一对应。更大的范围可以实现更深入的合作，但会带来更多的操作要求以确保无缝的版本控制。
  • 项目是您希望对所有相关资源施加文件结构和权限的层级。如果一个概念上的“项目”包含需要单独权限的子组件，那么在Foundry中考虑创建多个项目。

1. 数据连接

在Foundry中的管道中流动的数据通常来自外部源系统。 数据连接 服务提供了一个管理界面，用于注册源系统。对于每个源系统，可以配置一个或多个单独的同步操作，将数据导入到raw数据集中的数据源项目。

为了首先配置源系统或同步操作，通常需要代理管理员配置一个数据连接 代理。每个代理应存储在一个仅对代理管理员可访问的专用项目中。

管理新数据源的连接是一项多学科的工作，需要数据源所有者、代理管理员和数据源开发人员的合作，尽管有时还需要合规/法律官员参与。 签署数据从源系统移动到新环境的流程。

但是，一旦源系统配置完成，数据源开发者就可以独立配置各个同步操作。这大大减少了团队之间的来回沟通，并确保连接新系统是一项低门槛、一次性的工作。

每个连接的关键考虑因素是选择使用 SNAPSHOT，每次同步时会用新数据替换当前数据，还是使用 APPEND，每次同步时会将数据累积添加到现有数据中。为了更全面地理解这些概念以及在 增量管道 部分创建高效、高性能管道的影响。

进一步阅读

有关数据源和同步操作的原则、架构和配置的更多信息，请访问 数据连接文档。

有关监控您的连接并确保流入管道的数据准确且符合预期的更多想法，请查看 维护管道。

2. 数据源项目

每个源系统应将数据放入平台内的匹配项目中。通常的模式是尽可能以“原始”格式将数据从每次同步中存入。转换为“干净”的数据集步骤在项目 代码仓库 中定义。

建立这种每个数据源一个项目的模型有许多便利的优势：

  • 鼓励在一致的位置探索新的数据源。
  • 提供了一个单一的位置来指定对数据源的访问控制。
  • 减少了不同团队重复同步相同数据的可能性。
  • 代码仓库规模小且专为处理源系统数据的类型和格式而设计。
  • 允许在数据被管道层访问之前进行匿名化、清理和标准化。

数据源项目目标

在每个数据源项目中，目标是为组织内各种用户、用例和管道准备同步的数据。

输出数据集及其模式应被视为一个API - 也就是说，它们应该逻辑上组织良好且随时间保持稳定，因为数据源项目的输出数据集是所有下游工作的核心。输出应该与源表或文件一一对应；数据集中的每一行都应该在源系统中有直接对应的行。

为此，原始→清理转换的一些典型目标包括：

  • 实施一致的列命名方案。
  • 确保对列应用适当类型。
  • 处理缺失、格式错误或类型错误的值。
  • 建立主要健康检查。
  • 移除不适合通用消费的PII（个人可识别信息）或其他敏感数据。

即使源系统提供了列数据类型信息，有时也需要将值作为 STRING 类型导入。特别注意 DATE、DATETIME 和 TIMESTAMP 类型，这些类型在源系统中通常以非标准格式和数值类型表示，有时会带来麻烦。如果这些类型从源系统来看格式不可靠，将其导入为 STRING 类型可以提供更强大的解析方式，并定义处理错误值或删除不可用或重复行的逻辑。

除了这些编程步骤外，干净的数据集应进行严格文档记录。对数据集的定性描述、源系统的来源、适当的联系人和管理协议，以及相关时的每列元数据，描述数据特征，都将确保未来的开发人员正确使用数据。

实现一个数据源项目

  1. 系统管理员为目标数据源创建一个新项目，并将适当的 数据源开发者 添加到一个新的 用户组 用于管理项目。
    • 只有特定数据源的数据源开发人员组的成员才应具有在项目内构建数据集的权限。
  2. 数据源所有者和代理管理员合作，在源系统上< a href="/docs/foundry/data-connection/set-up-agent/">部署数据连接代理，并在UI中配置数据源连接。
    • 确保配置了适当的权限，以便数据源开发人员能够预览源系统中的表和文件。
  3. 数据源开发人员配置同步项目中一个或多个原始数据集，考虑正确的同步频率和同步类型，并为每个原始数据集添加适当的数据健康检查。
  4. 数据源开发人员实施清理转换以生成“干净”的数据集，供其他项目使用。
  5. 下游开发人员通过向项目或特定的干净数据集提交一个问题来请求从源系统获取新数据集、对干净数据集进行增强或报告数据质量问题。

虽然每个数据源都会有些独特，但清理和准备源系统的步骤通常有一些共同的步骤，例如将原始字符串值解析为给定类型并处理错误。当数据源有许多类似的清理转换时，最佳做法是在 Python 中定义一个库，以提供一组一致的工具，并减少重复代码。

推荐的文件夹结构

  • /raw: 用于从数据连接同步的数据集。
  • /processed（可选）：用于从非表格文件创建表格数据的文件级转换。
  • /clean: 用于与 /raw 中的数据集一对一对应，但应用了清理步骤的数据集。
  • /analysis: 用于创建资源以测试或记录清理转换并展示数据的结构。
  • /scratchpad: 在构建或测试清理转换过程中创建的任何临时资源。
  • /documentation: 数据血缘 显示清理流程步骤的图表，以及在 Foundry Notepad 中编写的额外文档（超出顶级项目 README 的内容）。

3. 转换项目

转换项目的目的是将共享的、语义上有意义的数据分组封装起来，并生成规范视图以供本体论（Ontology）层使用。

这些项目从一个或多个数据源（Datasource）项目中导入清洗后的数据集，与查找数据集进行关联以扩展值，对关系进行标准化或反标准化以创建以对象为中心或以时间为中心的数据集，或者对数据进行聚合以创建标准的、共享的指标。

推荐文件夹结构

  • /data
    • /transformed(可选)：这些数据集是转换项目中间步骤的输出结果。
    • /output: 这些数据集是“转换项目”的“输出”，并且是在下游本体论项目中应依赖的唯一数据集。
  • /analysis: 用于测试或记录清理转换并展示数据形状的资源。
  • /scratchpad: 在构建或测试清理转换过程中创建的任何临时资源。
  • /documentation: 显示清理管道步骤以及额外文档（超出顶级项目 README 的内容）的 数据谱系 图表，这些文档是在记事本中编写的。

4. 本体论项目

本体论在Foundry中强制执行一个共享通信层，维护一个干净、有组织且稳定的本体论是最高的优先事项，以确保一系列有价值项目可以同时推进，并丰富共同的操作画面。有关设计和管理您的本体论的概念和实际操作的更多信息，请参考 本体论概念文档。

一个本体论项目是任何管道的核心，它代表了生成符合本体论中定义的一个或多个相关对象的数据集所需的最终转换。这些项目还会（独立地）生成与对象浏览器视图同步的数据集。

由于本体论数据集代表了它们所表示的操作对象的规范真实情况，因此它们成为所有“消费”项目的起点。虽然上游清洗和管道步骤的来源和转换逻辑是可见的，但概念上，这些步骤和中间数据集对于仅从本体论项目中获取数据的项目开发人员、分析师、数据科学家和操作用户来说可能是一个黑箱。从这个意义上说，本体论层充当了操作对象的API。

实现一个本体论项目

与数据源项目类似，维护本体论项目的关键因素包括：

  • 健全的文档
  • 有意义的健康检查
  • 定期的计划
  • 在数据目录中的整理 和适当的标签

这些内容在维护管道中以及下面关于记录项目的提示中进行了更详细的讨论。

此外，随着您的本体论变得更加完善，并且越来越多的团队参与本体论层，维护数据集“API”的完整性变得越来越关键。为此，建议实施额外的检查，以确保提议的更改能够保持输出数据集模式的完整性。这将在即将发布的更多文档中深入讨论。

推荐的文件夹结构

  • /data
    • /transformed（可选）：这些数据集是本体论项目中间步骤的输出结果。
    • /ontology：这些数据集是本体论项目的“输出”，只有这些数据集应在下游用例或工作流项目中被依赖。
  • /analysis：用于测试或记录清理转换并展示数据形状的资源。
  • /scratchpad: 在构建或测试清理转换过程中创建的任何临时资源。
  • /documentation: 显示清理管道步骤以及额外文档（超出顶级项目 README 的内容）的 Data Lineage 图表，这些文档是用报告编写的。

5. 工作流项目

工作流项目，也称为 用例项目，应根据当前上下文灵活设计，但通常围绕一个项目或一个团队建立有效的协作单元，并明确责任和访问的边界。

通常，工作流项目应引用本体论项目的数据，以确保操作流程、商业智能分析和应用项目都对世界有一个共同的视图。在开发工作流项目的过程中，如果本体层中可用的数据源不足以作为来源，这表明需要丰富和扩展本体论。避免引用管道中较早（或较晚）阶段的数据，因为这可能会导致特定类型数据的真相来源出现碎片化。

文档项目

每个项目在整个开发过程中都应该得到充分的文档记录。以下是一些常见的模式和最佳实践：

  • 在每个项目的根目录下保存一个或多个数据血缘图资源，以捕获最重要的数据集和管道关系。不同的图表可以使用不同的颜色方案来突出显示相关特征，例如数据刷新的规律性、相关数据集的分组，或者数据集的特性，如大小。
  • 在项目下方直接添加一个简短的描述。这将在所有项目的列表视图中显示。重点是为可能不熟悉该项目的用户提供上下文信息。
  • 在顶级项目文件夹右侧侧边栏中使用添加文档按钮，编写Markdown ↗语法。这是解释项目目的、主要输出和其他对项目使用者相关的信息的最佳位置。
  • 将更多的全局项目文档放在本地创建的Markdown（.md）文件中，并上传到文档文件夹，或在记事本中，并从顶级文档中链接到这些文件。
  • 在项目的代码仓库中添加README.md文件，并确保每个转换文件都有描述和额外的内联代码注释。

推荐的文件夹结构

工作流项目的结构将比其他项目类型更加多样化，应着重于使主要资源立即可访问并有良好的文档说明。

  • /data
    • /transformed（可选）：这些数据集是工作流项目中中间步骤的输出结果。
    • /analysis（可选）：这些数据集是分析工作流的输出结果。
    • /model_output（可选）：这些数据集是模型工作流的输出结果，然后可以对其进行分析以确定模型的适用性。
    • /user_data（可选）：如果您的工作流允许用户创建他们自己的“数据切片”，请将它们存储在此处的用户专用文件夹中。
  • /analysis：为驱动决策和反馈循环而创建的资源和报告。
  • /models：任何创建的模型都应存储在此处。
    • /templates：用于项目特定用途的任何 代码手册模板 应在此处存储。
  • /applications：额外的工作流应用程序或子应用程序存储在此处。主应用程序应存储在项目根目录中以便于访问。
    • /develop：正在开发的应用程序、新功能和模板存储在此处。
  • /scratchpad：在构建或测试清理转换过程中创建的任何临时资源。
    • /documentation: 显示清理管道步骤以及在报告中编写的额外文档（超出顶级项目 README 的内容）的数据血缘图。

管道管理角色

以下角色是Foundry中在管道的范围定义、设计、实施和管理中通常涉及的典型配置文件示例。并非所有角色在所有情况下都是必需的，尤其是在平台开发的早期阶段，而且个人可能同时承担多个角色。然而，一般来说，考虑这些角色描述及其相互作用将有助于创建有序且高效的团队。

下图将主要角色与他们最常活跃的管道部分相关联。

主要角色

  • 代理管理员： 代理管理员负责创建和配置数据连接 agent，并将其与相关 数据源所有者 共享。这集中了代理安装和配置的控制权，以提高安全性和控制能力。
  • 数据源开发者： 数据源开发者是负责数据湖中每个相关数据源的采集过程的工程师。这可以作为一个团队来完成，也可以按不同的源系统进行分工。
    • 每个数据源开发者拥有（1）一个具有与数据源技术连接的数据连接数据源，以及（2）一个从同步数据中获取数据的 Foundry 项目。
    • 数据源项目的输出是一个经过清理和准备的数据集，可供一个或多个下游用例项目使用。数据源开发者还定义他们所拥有的每个数据源的刷新频率，并与 发布经理 或 运维经理 合作，安排一个 管道构建策略。
  • 数据管道开发者： 数据管道开发人员是负责构建通用本体层的数据工程师，该层为所有使用场景提供数据资产。数据管道开发人员负责定义这一本体层，并构建从源结构到本体结构所需的转换。
  • 发布经理： 发布经理负责生产管道并管理发布流程，以确保生产管道符合组织的标准并可靠执行。
    • 对于管道的代码更改，通常是通过将拉取请求提交到主分支的审批流程以及在主分支上运行管道构建的审批流程来完成的。为了确保持续可靠性，发布经理会监控与管道和数据源开发人员共同定义的关键数据集的健康检查。
  • 合规官员/法律官员： 合规官员负责在开发之前批准使用场景。这种批准应考虑：(1) 需要使用的数据，(2) 项目的目的，以及 (3) 在其管辖范围内的数据保护制度和组织自身政策下，谁将有权访问这些数据。
  • 用例开发者： 用例开发者是负责用例开发的工程师 - 软件开发人员和/或数据科学家 -。他们使用来自本体论层的数据，并由合规官员批准给他们。

其他角色

  • 运维经理： 此角色负责监控生产管道的健康执行。他们监控不同来源的数据刷新、不同管道的构建执行并协助处理支持请求。此角色支持发布经理。
  • 权限所有者 / 身份管理员： 在大多数情况下，用户分组的分配在客户的身份管理系统中进行，并通过SAML集成传递到Foundry。在这种情况下，权限管理负责该过程。
  • 数据官： 负责将本体论层建模为面向用户的对世界视图，负责数据质量，并寻找应集成以扩展本体论层的新数据源。数据负责人还负责监控用例中的增强内容，并将其泛化到本体论层，以提高可重用性。
  • 项目经理：项目经理与参与项目的工程师合作，确保技术目标和业务目标的一致性，遵守平台开发的最佳实践标准，并协调其他项目团队之间的协作。项目经理还负责记录项目。
  • 数据源负责人：这是特定源系统的首要联系人。在配置新的数据连接期间，他们需要与代理管理员紧密合作，之后需承担一定程度的“上游”问题的责任——无论是他们服务器或文件系统上运行的代理服务，还是数据本身的完整性。
