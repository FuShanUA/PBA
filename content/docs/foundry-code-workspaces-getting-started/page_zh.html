代码工作区入门

本教程通过为 JupyterLab® 或 RStudio® Workbench 配置工作区，来介绍代码工作区的基础用法。

底层代码仓库

所有代码工作区均由 Palantir 平台的代码仓库提供底层支持。这使得代码工作区具备分支管理、合并和提交历史等行业标准的版本控制功能，允许其他用户查看代码并在同一工作区中安全地进行操作。

此前，代码工作区及关联的代码仓库表现为独立的资源。现在，两者已整合表现为单一资源，即代码仓库。在 Foundry 的任意文件夹中选择该代码仓库，将自动打开关联的 Jupyter 或 RStudio® 工作区。

此变更适用于新建的代码工作区，不影响现有的代码工作区和代码仓库，后者仍将保持为独立资源。

启动代码工作区

要启动代码工作区，请打开代码工作区应用并进入常规选项卡，随后选择用于创建代码工作区的文件夹。

在项目中拥有 Editor 或 Owner 角色的用户能够修改代码工作区的设置。拥有 Viewer 角色的用户可以启动工作区。您可以选择将主文件夹作为存储位置，但这可能会导致您的代码工作区和数据被限制在某个项目内。

选择要启动的工作区类型。目前支持的工作区为 JupyterLab 和 RStudio® Workbench。

接着，为要创建的工作区设置底层代码仓库的名称和位置。若要创建 RStudio® 工作区，您可以选择相应的许可证。在大多数情况下，系统会自动选择许可证。

每个启动工作区的用户都将运行各自独立的会话。如果有两名或以上用户使用同一工作区，请确保所有用户对底层代码仓库均具有 Viewer 或更高权限。有关代码工作区底层代码仓库的更多信息，请参阅上文。

高级设置

在高级设置中，您可以为代码工作区定义空闲持续时间和资源规格。您可以随时通过选择右上角的齿轮图标，在工作区的设置面板中修改这些配置。

  • 自动关闭
    • 自动关闭设置用于定义容器保持活动的持续时间。达到该非活动时间段后，工作区将自动关闭以降低资源消耗。自动关闭后，需要将软件包和数据集重新部署至容器中。请注意，代码在自动关闭后仍会保留。
    • 默认的自动关闭时间为 30 分钟。自动关闭的最长时间设置为 6 小时，会话的最大时长为 24 小时。
    • 较长的自动关闭时间可能会导致更高昂的成本。建议设置较短的自动关闭时间以降低成本，并依赖提交至 Foundry 的代码，而非使用需要跨会话还原的未提交更改（请参阅下方的代码工作区生命周期文档）。
  • 计算资源
    • 更大的资源规格可能会导致更高昂的成本。
  • 网络策略
    • 如果管理员启用了此功能，您可以选择网络策略并将其附加到工作区，以便从工作区发起外部 API 调用。

数据转换

用户无法直接在底层代码仓库中发布转换，因为该仓库已定制为代码工作区的基础设施使用。所有代码必须从代码工作区进行发布。

导入数据

您可以直接从 Foundry 的数据源导入数据，方法是导航至数据选项卡，然后选择添加 > 读取数据。这包括 Foundry 数据集、Iceberg 表和虚拟表。从代码工作区当前项目外部（包括用户主文件夹）导入的任何数据，都需要添加引用。

或者，对于尚未接入 Foundry 的数据，选择上传数据可直接从您的设备导入数据。

数据集视图不能作为导入的数据集在代码工作区中使用。如有需要，请将数据集视图复制到新的或现有的数据集中。

导入包

您可以通过导航至左侧面板的包选项卡将包导入到您的代码工作区，也可以直接使用 Conda 或 PyPI（针对 Jupyter®）以及 CRAN（针对 RStudio®）安装包。

JupyterLab® 包

JupyterLab® 工作区支持从 Conda 或 PyPI 安装包。默认情况下，代码工作区会开放 conda-forge 和 pypi.org。您可以通过将其他渠道添加到底层代码仓库并重启代码工作区，从而将其添加到代码工作区中。

通过使用托管环境，JupyterLab® 中的 Conda/PyPI 环境可以在 JupyterLab®、应用程序或转换的工作流中一致地恢复。

您可以从包选项卡浏览包，并选择您希望安装的包。这将在新的 JupyterLab® 终端中自动为您执行安装命令。

要手动安装包，您可以在 JupyterLab® 终端中对 Conda 包使用 maestro env conda install <packages>，对 PyPI 包使用 maestro env pip install <packages>。

有关更多信息，请查阅 Jupyter® 笔记本中的托管 Conda/PyPI 环境。

已弃用直接使用 mamba 和 pip 安装包的方式。

CRAN

RStudio® 工作区支持从 CRAN、Posit™ Package Manager 或 Bioconductor 安装包。默认情况下，代码工作区开放了 Posit™ Public Package Manager ↗ 以及 Bioconductor ↗ 的全部资源。您可以通过将其他渠道（如 MRAN）添加到底层代码仓库并重启代码工作区来添加这些渠道。

您可以使用 renv 来安装包，因为 renv 支持经过身份验证的 CRAN 渠道。

您可以从包选项卡浏览包，并选择您希望安装的包。这将生成一个安装命令，供您复制粘贴到 RStudio® Workbench 中执行。通常，CRAN 包使用 renv::install("<package>") 安装，而 Bioconductor 包使用 renv::install("bioc::<package>") 安装。

发布代码

为了让同一工作区的其他用户能够查看您的代码，您必须将代码发布到您正在工作的分支上。

请注意，发布代码会触发 CI 检查，这将发布每个已注册代码工作区应用程序的新版本。作为最佳实践，我们建议将代码工作区聚焦于单一类型的任务（例如构建应用程序、分析数据、编写数据转换或构建模型），而不是试图在单个代码工作区中完成多个独立的工作流。

未发布的更改可以在源代码控制侧边栏中查看。可选择性地提供提交信息，然后选择同步更改来发布您的代码。折叠时，文件更改的数量会显示在源代码控制图标上。

对代码工作区进行分支操作

您可以使用分支管理器对代码工作区进行分支操作、检出分支，以及执行其他版本控制活动。

要创建或查找分支，请使用屏幕顶部的分支选择器。点击分支选择器会弹出一个菜单，显示现有分支列表。此外，您也可以选择创建新分支来新建分支。如果您有未发布的代码更改，系统会提示您在检出所选分支前先进行同步。

与《Foundry 其他位置的分支机制》类似，代码工作区中可用的数据来自于您当前所处的分支（包含回退分支）。如果您将数据写入输出数据集，数据也会写入到您当前的分支中。

您可以通过选择提议更改将您的分支合并至主分支。提议更改后，系统会显示拉取请求 (PR) 预览。

共享您的代码工作区

您可以在界面右上角选择共享来分享您的代码工作区。

如果代码工作区和代码仓库是独立的资源，您还必须共享底层仓库的权限，其他人才能查看您工作区中的代码。

在项目中拥有 Editor 或 Owner 角色的用户可以修改代码工作区的设置。在项目中拥有 Viewer 角色的用户则可以启动该工作区。

重启代码工作区

您可以在界面右上角选择Active > Restart workspace来手动重启代码工作区（例如，为了使配置更改生效）。

此操作将生成一个处于初始状态的新会话。工作区会自动重启，并包含所有已提交到底层代码仓库的更改。但是，工作区不会保留重启前的所有状态，包括已导入的数据、存储在内存中的变量、已安装的软件包等。

在重启工作区之前，应用程序会自动创建一个检查点，并提示您同步最新更改。

您可以选择Active > Restore checkpoint将工作区恢复到先前检查点的状态。更多详情请参阅下文的检查点部分。工作区重启后，Code Workspaces 不会自动将工作区恢复到先前的检查点。

代码工作区生命周期

首次打开 Code Workspaces 的用户将获得一个空白的工作区，其中底层仓库已克隆至 /home/user/repo。

当没有用户活动时，Code Workspaces 会话会自动关闭，且最长可持续 24 小时（有关自动关闭的更多信息，请参阅高级设置文档）。

当会话关闭时，所有相关状态都会以检查点的形式持久化保存在 Foundry 中，并在同一用户下次打开代码工作区时恢复。这使得用户能够从关闭时的位置继续工作。

在代码工作区中保存工作

Code Workspaces 有两种不同的保存现有工作的方式：手动代码同步和自动检查点。

我们建议您依赖已同步到底层代码仓库的代码（将必需的软件包安装作为代码的一部分），而不是依赖工作区检查点。代码同步有助于确保工作流的弹性，并促进与可能无法访问特定检查点的其他用户进行协作。

代码同步

代码同步是一项手动操作，也是在代码工作区中保存现有工作的推荐方式。同步代码意味着将您的文件提交到工作区关联的后端代码仓库。通过代码同步保存的所有工作都将永久存储在仓库中，并且 /home/user/repo 文件夹中的所有内容都会作为同步的一部分进行存储。代码同步不会保存现有状态，例如变量内容、环境中已安装的包、数据或代码单元格的输出。工作区的同步历史记录可以通过工作区关联的后端代码仓库进行查看。我们通常建议您频繁同步代码，这既能维护变更历史记录，又能确保所有相关工作得到妥善且永久的存储。

若要保存在工作区中所做的全部现有更改，请选择工作区右上角的 同步更改。操作成功后，更改将提交到工作区关联的后端仓库，您可以通过选择右上角的齿轮图标，然后选择 打开代码仓库 来访问该仓库。

检查点

检查点是一项自动对工作区现有状态进行快照的功能，以便在多次会话之间恢复状态。检查点分为两类：数据检查点和代码检查点。

数据检查点

与仅存储已提交文件的代码同步不同，数据检查点会存储 /home/user/repo（包含未提交的代码更改）、/home/user/libs（包含已安装的包）、/home/user/envs（Conda 环境）和 /home/user/data（用户下载的额外数据）中的所有内容。在 RStudio® 中，存储在内存中的变量会保存在 /home/user/repo/.RData 里。

默认情况下，代码工作区每十分钟会为给定用户的会话保存一次数据检查点。数据检查点不会在不同用户之间共享。代码工作区在工作区关闭前（例如通过手动重启、手动关闭或自动关闭）也会创建一个数据检查点。处于非活动状态的代码工作区不会生成任何新的数据检查点，并且任何给定的数据检查点最多保留 30 天。因此，如果工作区闲置超过 30 天，将不再有可用的数据检查点，此时只能从已同步到后端仓库的内容或代码检查点中恢复工作。数据检查点旨在作为一种安全保障机制，用于在多次会话之间持久化工作区的状态；不应将其作为主要的保存方式。我们强烈建议您频繁同步工作区中的现有工作。

恢复数据检查点将会重启工作区，并将主文件夹恢复到创建检查点时的状态。

代码检查点

代码检查点通过保存 /home/user/repo（包含未提交的代码更改）的内容来仅存储工作区代码（不包含数据），同时会排除超过特定大小或扩展名通常与数据相关的文件。这种过滤尽最大努力进行。代码检查点仅在工作区关闭前创建。与数据检查点不同，代码检查点没有保留期限。

代码检查点不受过滤限制，该过滤机制原本是为了防止您将常见的数据文件提交到工作区的后端仓库。您必须确保代码文件中不包含数据。例如，.py 代码文件绝不应包含 Foundry 数据集中存在的表格数据。

管理数据文件

代码工作区会自动扫描您的 /home/user/repo/ 文件夹，查找大文件或具有常见数据文件扩展名的文件，例如 .csv、.gz、.parquet、.pdf 或 .xlsx。将这些文件移动到 data-checkpoint 或 data-tmp 目录可以优化工作区启动时间并提升安全性，作为一种防范措施，防止您或其他用户无意中将这些文件提交到工作区的后端仓库。

  1. data-checkpoint：这是存放需要在多次会话之间保存和恢复的数据文件的推荐位置。这些文件将被保存到检查点中，并在后台恢复，不会阻碍工作区启动。此外，该文件夹将排除在同步范围之外，以防止将大型且未经安全处理的数据提交到后端代码仓库。
  2. data-tmp：用于存储临时文件的推荐路径，这些文件无需跨会话保留。相关文件不会保存至检查点或底层代码仓库中；工作区关闭时即会被清理。

若需让数据可供已发布的仪表板、转换任务或其他用户使用，建议将数据持久化至 Foundry 数据集，而非保留在工作区中。

工作区检测到数据文件后，将弹出警告提示，引导您将其移至 data-checkpoint 或 data-tmp 目录。选择需迁移的文件并按对话框提示完成移动。请记得在代码中更新新的路径前缀，否则在 /home/user/repo/ 文件夹下写入新数据文件时，该警告会再次出现。
