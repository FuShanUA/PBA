配置参考

本页记录了与 Webhooks 相关的详细配置选项和权限。

配置输入和输出

每个 Webhook 都可以灵活配置，将 输入参数 映射到将要发送到外部系统的请求。然后，系统返回的响应可以作为 输出参数，这些参数可以在 Palantir Foundry 的其他地方使用。

输入参数

输入参数 表示在用户执行 Webhook 时可以传入的输入。通常，Webhook 是为在 操作类型 中使用而配置的。 ，您可以指定 操作参数 和 Webhook 输入参数 之间的映射。

Webhook 输入参数有多种数据类型和约束：

  • 布尔值（Boolean） 参数可以是 true 或 false 。
  • 整数（Integer）、长整型（Long） 和 双精度浮点数（Double） 参数表示数值。
  • 字符串（String） 参数表示文本值，并且可以被限制为仅允许特定的值。
  • 日期（Date） 和 时间戳（Timestamp） 参数表示基于时间的数据。

除了这些基本类型之外，还有几种集合类型可用：

  • 列表（List） 参数表示特定类型的有序输入集合。
  • 记录（Record） 参数允许传递键/值对，并且可以被限制为要求特定键，其值具有特定类型。
  • 可选（Optional） 参数表示可能存在的输入。

如果您想使用 对象上的函数 要从操作参数映射到webhook输入，如果您的映射输入的函数返回undefined，您也可以有条件地不触发webhook。例如，WebhookInput | undefined。

最后，附件类型可用于传递在操作表单中上传的文件。请注意，此功能不适用于代理工作者源。

输出参数

输出参数允许您捕获从外部系统返回的数据，以便在Foundry的其他地方使用。例如，当您在外部系统中创建一条新记录时，系统可能会返回该新记录的ID。通过将这个新ID捕获到输出参数中，您可以将其传播到操作中，并立即将其写入Foundry对象。

捕获的输出参数取决于您使用的任务类型。

为了方便，某些情况下Webhook任务的结果会自动转换为适当的输出参数类型：

  • 如果配置了字符串（String）输出参数，而Webhook任务结果不是字符串，则结果将被转换为JSON字符串。
  • 如果配置了记录（Record） 如果输出参数配置为字符串，则 Webhook 服务会尝试将字符串解析为 JSON。
  • 如果配置了 Double（双精度浮点） 输出参数，则 Webhook 任务结果如果是整数或长整型，将自动转换。

可以使用通过“测试连接”侧边栏发出的请求所接收到的响应来配置输出参数。一旦成功执行测试请求，添加新输出参数时将自动显示从响应中解析出的建议输出。

任务主体

某些 Webhook 实现为任务（Task）。对于这些 Webhook，任务主体 表示在 Webhook 执行时将发送到任务的数据。

每个任务的结构取决于将要列出的具体任务类型，如果相关的话，会在Webhook 类型条目下说明。任务正文应使用Handlebars ↗语法定义。在任务正文内，您可以引用上面文档中定义的 Webhook 的任何输入参数。

Webhook 类型

每种类型的 Webhook 支持不同的配置选项。本节记录了各种 Webhook 类型可用的选项，并提供了基本示例以帮助您开始。

REST API

Webhook 最常见的用途是向 REST APIs 发起 HTTP 请求。在配置好 REST API 源后，您可以使用请求构建器界面来构造您想要发起的请求。

REST API 源的可用选项包括：

选项必需描述
方法是请求的 HTTP 方法。
相对路径否指定请求端点的相对路径。这应相对于 REST API 源中配置的一个域名。
查询参数否提供要包含在请求中的任何查询参数的键值对。某些查询参数可能根据源配置在运行时包含。这些参数将在此处显示为只读。
授权否授权详细信息基于源配置。任何编辑应通过返回到源进行。
头信息否提供要包含在请求中的任何头信息的键值对。某些查询参数可能根据源配置在运行时包含。这些参数将在此处显示为只读。
正文否 对于接受正文的请求，您可以从可用类型中包含一个正文。这些类型包括Raw JSON、Form Data、Form URL Encoded、二进制File、XML、Plain Text 和 HTML。

多个请求

一个单一的webhook可以包含多个请求。请求可以链接在一起，先前调用的响应值可以在后续调用中引用。

REST API（旧版）

此处记录的旧版REST API选项以及使用magritte-rest-v2插件的是历史参考内容。新的工作流应使用REST API源来实现。

在设置好magritte-rest-v2源后，您可以使用generic-rest-webhook-task创建REST Webhooks。

对于REST Webhook，Task主体的结构必须是一个calls数组，每个元素代表一个将发送到外部系统的REST请求。

对于webhooks，唯一支持的call类型是magritte-rest-call。

下面是REST Webhook的基本Task主体模板示例，表示单个HTTP调用：

Copied!
1{
2    "calls": [
3        {
4            "type": "magritte-rest-call",
5            "method": "POST",
6            "requestMimeType": "application/json",
7            "path": "your/request/path",
8            "body": { "text": {{json message}} }
9        }
10    ]
11}

此Task主体模板会向your/request/path端点发送POST请求，请求体为{ text: <message> }，其中<message>是Webhook中的字符串输入参数。

提取输出参数

从REST Webhook中提取输出参数主要有两种方法：1）通过名称捕获JSON响应中的顶级字段；2）定义JSON提取器以实现更定制化的提取逻辑，并显式列出您想要输出的字段。

您可以选择将完整的响应 JSON 作为字符串输出。这在使用函数进行后续编辑或通知渲染时提供了额外的灵活性。

REST 插件支持多种提取器类型，包括 JSON、XML、HTML、HTTP 状态等。Webhook 需要一个返回 JSON 的外部端点，因此在 Webhook 任务配置中只能使用 json 提取器。

对于捕获顶级字段，假设您已配置了一个 REST 调用，其返回以下响应：

Copied!
1{
2    "id": "c52fd6e4-6eb5-4da1-8908-4845e51c801b"
3}

您可以使用与 JSON 响应中的键相同的 ID 定义输出参数以捕获它们。在此示例中，您可以添加一个参数 ID 为 id 的字符串输出参数，此字段将从响应中被捕获。

如果需要从响应中捕获嵌套字段，可以通过指定 extractors 来提取值。提取的值也可以在后续调用中用于串联调用。如果需要捕获整个响应，可以通过指定 extractors 并将根路径作为目标："result": "/"。

下面是任务正文模板的示例，它发出两个请求：一个是对GET端点的请求以获取一些数据，然后是对POST端点的请求，使用前一个调用中的数据。

Copied!
1{
2    "calls": [
3        {
4            "type": "magritte-rest-call",
5            "method": "GET",
6            "path": "path/to/fetchData",
7            "extractor": [
8              {
9                "type": "json",
10                "assign": {
11                  "request_output": "/json/path/to/output"
12                }
13              }
14            ]
15        },
16        {
17            "type": "magritte-rest-call",
18            "method": "POST",
19            "path": "your/request/path",
20            "body": { "text": {%request_output%} },
21            "extractor": [
22              {
23                "type": "json",
24                "assign": {
25                  "result": "/json/path/to/result"
26                }
27              }
28            ]
29        }
30    ],
31    "output": ["result"]
32}

第一个调用向 path/to/fetchData 的端点发送一个 GET 请求，然后从 /json/path/to/output 的 JSON 路径中提取数据到名为 request_output 的状态变量中。然后，request_output 状态用于第二个调用的正文。从第二个调用中，我们从 JSON 响应中提取另一个字段到名为 result 的状态变量中。最后，配置中的 "output" 字段定义了哪些提取的字段应作为输出参数返回。

其他选项

除了指定 calls 和 output 之外，REST Webhook 任务还有几个额外的配置选项：

  • 如果您在 REST Webhook 中进行多次调用，只允许一次调用使用不安全的 HTTP 方法。不安全的调用是指可能在外部系统中修改状态的调用。默认情况下，只有 GET、OPTIONS 和 HEAD 请求被视为安全。您可以通过将 "isHttpMethodSafe": true 指定为调用的一个字段来覆盖其他类型调用的安全性（例如 POST 或 retryable-status-codes）。如果您进行多个请求，其中早期的一个请求是 external-system-not-changed-status-codes，并且您在后续请求中使用其响应时，这会很有用。
  • 您可以指定一个 create-record-salesforce-webhook-task 数组，该数组表示可以重试的 HTTP 状态码。
    • 例如，您可以配置 Webhook 在从服务器接收到 503 响应时重试外部请求。此选项默认为空列表。
  • 您可以指定一个 update-record-salesforce-webhook-task 数组，表示您连接的服务器尽管请求失败，但并未更改任何数据。
    • 此选项默认包含400到431之间的所有状态码。
    • 当Webhook执行并失败时，会捕获外部系统是否可能已更改的指示，以支持写入失败的调试。

限制

对于每个 Webhook，您可以设置三种类型的限制，这些限制会约束 Webhook 的执行方式：时间限制、并发限制和速率限制。

一个 时间限制 允许您设置Webhook应执行的最大持续时间。这使您可以确保终端用户应用程序保持响应，并在您连接的外部系统响应过长时显示超时错误。如果没有提供值，默认超时时间为20秒。允许的最大超时值为180秒。

一个并发限制指定了同时运行的Webhook执行的最大数量。这可以定义为避免向外部系统发送太多并发请求而使其过载。

一个速率限制限制了您指定的时间窗口内Webhook可以执行的次数。如果您希望保证Webhook每秒、每分钟、每小时或每天只执行特定次数，您可以启用这种类型的限制。

Beta

将webhook作为函数发布仍处于开发的 beta 阶段。从webhook发布的函数不能打包到Marketplace产品中。

  • Machinery 和 Automate

其他选项

存储响应

默认情况下，webhook 响应将被存储并在历史视图中显示六个月。完整的响应只能被执行 webhook 的用户以及拥有 webhooks:read-privileged-data 权限的管理员看到。

OAuth 2.0与Webhooks

授权码授予

Palantir Webhooks 支持使用 OAuth 2.0 授权码授予流程调用端点。这需要使用 出站应用程序（outbound application） 来定义与 OAuth 2.0 服务器的交互。一旦配置完成，出站应用程序可以作为 REST API Webhook 的认证，并且在尝试执行 Webhook 时会提示单个用户与 OAuth 服务器进行认证。

客户端凭证授予

Webhooks 可用于使用客户端凭证执行 OAuth 流程。客户端凭证授权流程通常使用一个长期有效的密钥来获取一个短期有效的访问令牌，该令牌可用于对目标系统执行操作。REST API Webhooks 支持连续进行多个 API 调用，并可用于执行客户端凭证握手，然后作为单次执行完成所需的请求。

以下指南将解释如何针对一个假设的示例系统配置客户端凭证授权流程。

要继续操作，您需要以下信息：

  • OAuth 2.0 服务器域名 和 令牌端点 的路径。令牌端点用于获取用于向 资源域名 发起请求的有效令牌。
    • 在许多情况下，OAuth 2.0 服务器域名和资源域名是相同的。例如，当使用 第三方应用程序 对 Foundry API 进行身份验证时，就是这种情况。
  • 资源域名 和包含从OAuth 2.0服务器获取的承载令牌的授权标头的所需端点路径。
  • 客户端ID for the OAuth 2.0 server.
  • 客户端密钥 for the OAuth 2.0 server.

并非所有的OAuth 2.0服务器都严格遵循OAuth 2.0标准。Webhooks请求构建器界面旨在足够灵活，以适应您可能需要进行的任何非标准配置，以便成功进行身份验证。在尝试遵循本教程之前，我们建议您查看您要连接的系统的文档。
