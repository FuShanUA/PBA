配置参考

本页面记录了 Data Connection 中与 Webhook 相关的详细配置选项和权限。

配置输入与输出

每个 Webhook 都可以灵活配置，将输入参数映射到将要发送给外部系统的请求中。随后，系统的响应可以被捕获为输出参数，进而在 Palantir Foundry 的其他地方使用。

输入参数

输入参数代表用户执行 Webhook 时可以传入的输入。通常，Webhook 会被配置用于操作类型，在这些类型中，你可以指定操作参数和 Webhook 输入参数之间的映射关系。

Webhook 输入参数支持多种数据类型和约束条件：

  • 布尔型参数可以是 true 或 false。
  • 整数型、长整型和双精度型参数表示数值。
  • 字符串型参数表示文本值，并且可以设置约束，仅允许特定的值。
  • 日期型和时间戳型参数表示基于时间的数据。

除了这些基本类型外，还提供了几种集合类型：

  • 列表型参数表示特定类型的有序输入集合。
  • 记录型参数允许传入键值对，并且可以设置约束，要求必须包含特定的键且其值为特定类型。
  • 可选型参数表示可能存在也可能不存在的输入。

如果你想使用对象上的函数将操作参数映射到 Webhook 输入，当用于映射输入的函数返回 undefined 时，你也可以选择完全不予触发 Webhook。例如，WebhookInput | undefined。"

最后，可以使用附件类型来传递在操作表单中上传的文件。请注意，智能体 worker 源不支持此功能。

输出参数

输出参数允许你捕获从外部系统返回的数据，以供 Foundry 其他地方使用。例如，当你在外部系统中创建新记录时，系统可能会返回该记录的 ID。通过将这个新 ID 捕获到输出参数中，你可以将其传递给操作，并立即将其写入 Foundry 对象。

捕获的输出参数取决于你正在使用的任务类型。

了解更多关于如何从 REST Webhook 捕获输出参数并在操作中使用这些参数的信息。

为方便起见，在某些情况下，Webhook 任务的结果会自动转换为相应的输出参数类型：

  • 如果配置了 String 输出参数，且 Webhook 任务结果不是字符串，则该结果将转换为 JSON 字符串。
  • 如果配置了 Record 输出参数，且 Webhook 任务结果是字符串，则 Webhooks 服务将尝试把该字符串解析为 JSON。
  • 如果配置了 Double 输出参数，则类型为 Integer 或 Long 的 Webhook 任务结果将自动进行转换。

可以使用通过“测试连接”侧边栏发出的请求所收到的响应来配置输出参数。成功发出测试请求后，在添加新的输出参数时，系统将自动显示从响应中解析出的建议输出项。

任务主体

某些 Webhook 会以任务形式实现。对于这些 Webhook，任务主体表示执行 Webhook 时将发送给任务的数据。

每个任务所需的结构取决于具体的任务类型，如果相关，该结构会在相应 Webhook 类型的条目下列出。任务主体应使用 Handlebars ↗ 语法定义。在任务主体内，你可以引用为 Webhook 定义的任何输入参数（如上文所述）。

Webhook 类型

每种类型的 Webhook 支持不同的配置选项。本部分记录了各种 Webhook 类型的可用选项，并提供了基本示例以帮助你入门。

REST API

Webhook 最常见的用途是向 REST API 发起 HTTP 请求。配置 REST API 源后，你可以使用请求构建器界面来构造要发起的请求。

REST API 源的可用选项包括：

选项是否必填说明
Method是请求的 HTTP 方法。
Relative path否指定请求端点的相对路径。该路径应相对于 REST API 源中配置的某个域名。
Query Params否为请求中包含的任意查询参数提供键值对。某些查询参数可能会在运行时根据源配置自动加入，这些参数将在此处以只读形式显示。
Authorization否授权详情基于源配置。如需编辑，请返回源配置页面进行修改。
Headers否为请求中包含的任意标头提供键值对。某些查询参数可能会在运行时根据源配置自动加入，这些参数将在此处以只读形式显示。
Body否对于接受主体的请求，你可以从可用类型中选择并包含一个主体。这些类型包括 Raw JSON、Form Data、Form URL Encoded、二进制 File、XML、Plain Text 和 HTML。

多重请求

单个 Webhook 可以包含多个请求。请求之间可以串联，即将前一次调用的响应值在后续调用中引用。

提取输出参数

从 REST Webhook 提取输出参数主要有两种方式：一是按名称捕获 JSON 响应中的顶层字段；二是定义 JSON 提取器以实现更定制化的提取逻辑，并显式列出希望输出的字段。

你也可以选择将完整的响应 JSON 提取为字符串输出。这为在后续执行编辑或使用函数渲染通知时遍历响应提供了额外的灵活性。

REST 插件支持多种提取器类型，包括 JSON、XML、HTML、HTTP 状态码等。由于 Webhook 需要一个返回 JSON 的外部端点，因此在 Webhook 任务配置中只能使用 json 提取器。

对于捕获顶层字段，假设你配置了一个返回以下响应的 REST 调用：

其他选项


除了指定 calls 和 output 之外，REST Webhook 任务还提供了一些额外的配置选项：


    

  • 如果在 REST Webhook 中发起多个调用，仅允许其中一个调用使用不安全的 HTTP 方法。不安全的调用是指可能会修改外部系统状态的调用。默认情况下，只有 GET、OPTIONS 和 HEAD 请求被视为安全方法。你可以通过在该调用中指定 "isHttpMethodSafe": true 字段，来覆盖其他调用类型（如 POST 或 PUT）的安全性设定。如果你在发起多个请求时，其中一个较早的请求是 POST，并且你需要在后续请求中使用其响应结果，那么此设置会非常实用。
    • 

  • 你可以指定一个 retryable-status-codes 数组，表示哪些 HTTP 状态码可以触发重试。

      

    • 例如，你可以配置 Webhook 在收到服务器返回的 503 响应时重试外部请求。此选项默认为空列表。
      • 

    

    • 

  • 你可以指定一个 external-system-not-changed-status-codes 数组，用于表明你所连接的服务器在请求失败的情况下并未更改任何数据。

      

    • 此选项默认包含从 400 到 431 的所有状态码。
      • 

    • 当 Webhook 执行失败时，系统会记录外部系统状态是否可能已被修改的指示信息，以便调试写入失败的问题。

Limits


For each Webhook, you can set three types of limits that constrain how the Webhook can be executed: time limits, concurrency limits, and rate limits.


A time limit allows you to set the maximum duration a Webhook should execute for. This enables you to ensure that end-user applications remain responsive and show a timeout error when the external system to which you are connecting is taking too long to respond. The default timeout is 20 seconds if no value is provided. The maximum allowed timeout value is 180 seconds.


A concurrency limit specifies the maximum number of Webhook executions that run at a single time. This can be defined to avoid overwhelming an external system with too many concurrent requests.


A rate limit restricts how many times a Webhook can be executed within a time window that you specify. If you want to guarantee that a Webhook will only execute a specific number of times every second, minute, hour, or day, you can enable this type of limit.

Webhook history


The webhook history displays a timeline of when webhooks were triggered. The history metadata will always include the user ID who triggered the webhook, a timestamp, and the success or failure status of the execution, as shown below:




By default, inputs passed to the webhook and the full response will only be visible to the user who called the webhook. This protects any sensitive data passed in or returned from the webhook call. The webhooks:read-privileged-data permission will allow access to the full history, and is not granted to any users by default. A custom role with this permission is required to access the full history for a webhook.


Other options


Store responses


By default, webhook responses will be stored and displayed in the history view for six months. The full response will only be visible to the user who executed the webhook and any admin with the webhooks:read-privileged-data permission.


This option may be disabled entirely for a webhook that is known to return sensitive information that should not be stored in the webhook history.


OAuth 2.0 with Webhooks




授权码许可


Palantir Webhooks 支持使用 OAuth 2.0 授权码许可流程调用端点。这需要使用出站应用来定义与 OAuth 2.0 服务器的交互。配置完成后，出站应用可用作 REST API Webhook 的身份验证方式，并会在尝试执行 Webhook 时提示用户单独通过 OAuth 服务器进行身份验证。


客户端凭证许可


可以使用 Webhook 执行基于客户端凭证的 OAuth 流程。客户端凭证许可流程通常使用长期有效的密钥来获取短期有效的访问令牌，进而使用该令牌对目标系统执行操作。REST API Webhook 支持连续链式调用多个 API，可借此执行客户端凭证握手及后续的目标请求，将其作为单次执行过程。
