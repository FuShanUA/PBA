包安全

Apollo 的包安全功能使您可以通过自动扫描查看在包工件中检测到的漏洞和病毒。导出应用程序中的 包安全 选项卡提供即时访问安全报告，从而快速评估风险并减少手动扫描。

包安全是一个 实验性功能。请联系 Palantir 代表以启用此功能。

设置包安全报告

要将安全报告包含在您的管道的包中，请确保您的管道的 Vulnerability Scan Report 包含在 资源选择 设置中。在此设置下，如果您选择了预设选项之一，则应默认包含该报告。如果您选择了 自定义 选项，请确保选择了 Vulnerability Scan Report。

安全报告是按照每个Bundle进行评估的，并在构建时包含在Bundle中。这意味着只有那些在构建时包含安全报告的Bundle才会拥有有效的安全报告。其他所有Bundle将不会包含关于Bundle扫描结果的准确信息。

要查看一个Bundle是否包含漏洞扫描报告：

  1. 进入Bundle收件箱面板中的概览选项卡。
  2. 向下滚动到构建设置部分。
  3. 找到YAML内容中的resourceTypeOptions部分。
  4. 如果type是exclusionList，请确保VULNERABILITY_SCAN_REPORT不在resourceTypes:下的列表中。

    如果type是inclusionList，请确保VULNERABILITY_SCAN_REPORT包含在resourceTypes:下的列表中。

配置您的管道以避免病毒、漏洞和未扫描的制品

为了避免导出未扫描或扫描失败的制品，您可以将以下标签要求添加到您的Pipeline的必需资源标签设置中：

  • security-scan-outcome = pass: 确保仅包含通过安全扫描的工件（无活动病毒或漏洞）。
  • security-scanned = true: 确保所有工件都已进行漏洞和病毒扫描。有些工件Apollo仅扫描漏洞，不扫描病毒。

这将确保Pipeline仅导出具有通过安全扫描的工件的产品发布。

访问安全报告

要访问Bundle的安全报告，请导航到该Bundle的导出收件箱页面。然后选择安全选项卡。在这里您可以查看扫描报告和扫描结果。

扫描报告部分提供了Bundle安全性的概述，将告知您该Bundle是否安全可以传输。

  • 最左边的卡片显示Bundle是否包含病毒。
  • 中间的卡片显示Bundle是否包含未抑制的过期漏洞。
  • 最右侧的卡片显示了该捆绑包中缺少病毒扫描、漏洞扫描或两者都缺少的工件数量。

在扫描报告部分下方是扫描结果表格。该表格显示扫描器发现的具体病毒和漏洞。您可以对扫描结果进行过滤，以查看特定的发现结果，例如仅查看病毒发现结果。

如需了解有关特定病毒或漏洞的更多信息，请选择该行以打开包含更多详细信息的对话框。在那里，您可以获取有关安全问题的深入信息，例如扫描失败的原因以及发现了什么问题。您还可以采取措施来修复或抑制该问题。

如果一个工件缺少扫描或存在错误或不完整的扫描，您可以在安全选项卡顶部看到一个红色横幅，以及底部的未扫描的工件表格。此表格允许您查看受影响工件缺失的扫描类型，这可以是病毒扫描、漏洞扫描或两者都有。您必须解决该问题并重新运行扫描器。

如果您的捆绑包中包含任何安全问题，您必须在导出捆绑包之前< a href="#remediate-security-issues">修复这些问题。

修复安全问题

对于病毒、漏洞或未扫描的工件，您需要查明该工件所属的产品版本，修复问题，并正确扫描该工件。然后，您需要重新构建管道以包含以下内容之一：

  • 如果所有发现的问题都可以被忽略，那么相同的版本发布。
  • 如果需要修复一个问题，那么新发布的版本。

您必须重新构建管道，因为捆绑包的安全报告在构建过程中直接嵌入到捆绑包中。因此，该报告是静态的，不会反映实时更新，例如当漏洞或病毒已被消除时。为了获得包含您的修复工作的更新报告，您需要重新构建管道以生成一个新的捆绑包和新的捆绑包报告。

对于增量管道，下一个构建的捆绑包不会包括您当前可以发送但未发送的全部内容。在对当前捆绑包中的安全问题进行修复后，后续的捆绑包通常只包含由于这些纠正措施而产生的更新或新的产品版本。为了解决这个问题，您有几个选项：

  • 使用紧急管道发送遗漏的发布版本： 如果您需要将遗漏的数据导出到您的目标中心，可以使用紧急管道导出缺失资源。为此，请转到紧急管道的设置，并使用 数据选择规则 包含缺失数据。然后重新构建管道并导出所构建的包。

  • 临时将管道构建设置切换为快照模式： 如果您需要将遗漏的数据导出到您的目标中心，但尚未设置紧急管道：

    1. 确保受影响的发布版本被撤回，或修改您的 管道的数据选择设置 以排除它。
    2. 将管道的构建设置切换为快照模式。
    3. 重新构建管道以获取下一个要导出的包。

    与增量构建策略不同，快照策略在每次构建时都会完全重新评估数据选择规则。快照策略会包含所有Bundle的数据，无论这些信息是否已在导出管道中的前一个Bundle中被发送。因此，排除受影响的发布版本并使用快照构建策略重新构建管道，将为您提供一个包含所有遗漏数据以及您更新或新的产品发布版本的Bundle。一旦您发送了快照Bundle，请切换回增量策略，以利用增量Bundle相关的大小和传输时间优势。

  • 跳过该Bundle及其包含的所有新发布版本： 如果您不需要立即导出Bundle中的遗漏数据，可以跳过它，并等待未来构建包含这些更新的Bundle。只有在您确定不需要前一个Bundle中包含的信息时，才选择此选项。

  • 即使存在安全问题也发送该包： 如果安全问题是一个误报，或者不违反合规要求，您可以仍然发送该包。只有在您确定该包无论报告如何都满足安全要求时，才选择此选项。

修复策略

病毒修复

如果您的包中包含病毒，除非发现是假阳性，否则不应导出该包。如果是假阳性，您可以：

  1. 对该特定镜像抑制该发现。
  2. 重新运行相关产品版本的漏洞扫描，以确认该发现已被抑制。
  3. 重新构建该包。

如果病毒发现是真正的阳性结果，您必须修复该病毒发现并发布包含修复内容的新版本。如果container-vuln-scanner没有自动召回之前的受影响版本，请确保召回该版本。然后您可以重新构建Bundle以包含您新发布的版本。

漏洞修复

如果您有严格的漏洞要求，需要确保在捆绑的工件中检测到的漏洞被抑制。如有必要，您可以像处理病毒发现一样修复过期的漏洞。可以通过抑制漏洞（如果没有可用的修复程序或风险可接受）或者发布包含漏洞修复的新版本的相关产品版本来实现。

未扫描工件修复

确定该工件所属的产品发布版本，并验证该产品发布版本已被扫描。请注意，只有属于产品发布版本的工件才会被扫描。如果手动选择单独的工件，请确保它们仍属于某个产品发布版本以进行扫描。
