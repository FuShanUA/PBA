保持高性能

增量转换能够确保管道更新时仅处理新增数据行，从而有助于最大程度地减少资源消耗。在 Foundry 中，增量转换只能向数据集添加新行，无法替换或删除现有行。数据接入过程可能同样如此。由于每次运行通常只处理少量数据，这些新增的数据量很小，但在渲染活动视图时，必须将所有这些新增数据合并考量。

通常情况下这不会有问题，但当数以万计甚至十万计的小规模更新累积下来（取决于更新频率，这可能在连续构建数周或数月后发生），读写性能可能开始下降。这可能是由于以下几个相关因素造成的，下文将对此进行详细讨论。

可能的原因

文件数量过多

数据集中存在过多文件会导致后端在访问当前视图数据时响应缓慢。这可能引发预览不可用问题、Contour 运行失败，以及转换（尤其是 Python 转换）过程极度缓慢，这是因为在该语言环境中加载文件存在额外的开销。

随着时间推移，任何涉及活动数据集视图的操作都将受制于低效的数据读取，情况严重时，原本仅需几分钟即可完成的转换最终可能耗时数小时甚至数天。在极端情况下，后端请求甚至可能超时，进而导致构建失败。

此外，作为将 Spark DataFrame 写入 APPEND 或 UPDATE 事务输出的一部分，增量转换需要构建一个包含输出当前视图中所有现存文件的内存列表。对于文件数量庞大的输出而言，此操作耗时较长，可能需要使用高驱动程序内存的 Spark 配置 来避免内存溢出错误。

事务数量过多

与文件数量过多类似，视图中存在过多事务同样会导致响应缓慢。然而，这种缓慢不仅发生在请求文件时，在解析视图范围时也会出现，这会影响数据集的各项功能，如统计信息计算、历史记录加载等。在转换过程中，这种缓慢现象会体现在预计算阶段，此时环境正在搭建且尚未产生任何 Spark 详细信息，导致问题难以排查。

通常事务与文件会同步累积，但在某些情况下，即使文件数量相对较少，空事务也可能引发问题。反之，如果每个事务包含大量文件，那么即使事务数量相对较少，同样可能导致文件总数极其庞大。

可能的解决方案

定期快照

保持增量管道高性能的最简且在某些情况下最有效的方案是定期运行快照。快照会重新处理全量数据，并用一个可高效分区的新视图替换输出。快照属于计算密集型操作，这也是人们通常偏好增量处理而非常规处理的原因，但偶尔进行快照有助于防止同一视图中文件或事务过度堆积。为了确定快照频率，管理员需要在数据读写性能之间取得平衡。

快照事务具有级联效应，即任何使用了已快照输入的增量转换都会触发自身快照。我们无需直接强制增量转换执行快照，只需对其某个输入进行快照，转换应用程序即会判定该增量转换同样需要快照。因此，设置增量管道定期快照的一种实用方法是：创建一个按预定快照计划构建的虚拟输入，并使该虚拟输入始终在快照模式下运行。

定期快照存在以下几个缺点：

  • 快照会使血缘关系变得复杂，因为虚拟输入会出现在搜索结果和文件夹结构中，这可能引发混淆。
  • 即使快照频率不高，其成本也可能非常昂贵。
  • 快照运行期间，无法通过管道传播任何更新，这可能会影响 SLA（服务级别协议）。
  • 快照不适用于原始数据集，例如由数据连接同步支持的数据集。
  • 并非所有转换在编写时都支持快照（尽管我们建议始终予以支持）。

数据集投影

处理增量堆积最推荐的方法是为相关数据集添加《数据集投影》。数据集投影提供了一种查询数据集中数据的替代机制，并且独立于规范数据集进行存储和更新。得益于这些特性，数据集投影能够跳出增量计算的纯追加模型，并随着数据量的增长自动重组其内部数据表示。这被称为压缩——压缩确保从投影进行的读取始终保持高性能，无论规范数据集中包含多少文件或事务。

这对于增量管道尤其有用，因为数据集投影无需与规范数据集完全同步，读取器即可受益于性能的提升。所有 Foundry 产品都知道如何将来自投影和规范数据集的数据结合起来，以重建视图，效果与没有投影时完全相同。例如，如果一个数据集有 100 个增量事务，并且有一个基于前 99 个事务构建的投影，那么将从投影中读取 99 个事务，而仅从数据集中读取 1 个。因此，通常每天或每周更新一次数据集投影就足够了，这使得其维护的计算成本非常低。

请注意，由于数据集投影是独立于规范数据集的资源，因此可以随时构建，即使规范数据集本身正在构建也可以（反之亦然）。读取器只会使用根据有效事务确定的当前状态。例如，如果数据集正在构建第 10 个事务，而投影同时开始构建，它将读取第 9 个事务。在这种场景下查询数据的读取器将从投影读取事务 1-8，从数据集读取事务 9，最终看到的数据与直接从数据集读取的数据完全相同。

数据集投影的缺点包括：

  • 数据集投影比仅使用数据集本身消耗更多的存储空间。
  • 数据集投影无法与 Foundry Retention 开箱即用。
  • 数据集投影不会以任何方式修改规范数据集（因此，如果移除投影，读取效率将再次降低）。

保留策略

有时管道的用例并不要求在平台内永久保留历史数据，仅保留最近的事务即可，这可以使用 Foundry Retention 自动完成。在这种情况下，只要转换逻辑不包含任何跨事务依赖（例如聚合或差分计算），就可以构建增量管道而无需特殊考虑。必须在 Python Transforms 中为增量装饰器设置一个特殊的 allow_retention 标志（否则 DELETE 事务将触发快照运行）。

保留策略变更的缺点包括：

  • 丢失历史数据。
  • 如果从数据角度来看事务不是自包含的单元，保留策略可能会导致状态不一致（例如，没有匹配开始事件的结束事件）。

其他可考虑的选项

中止事务

在某些情况下，提交的事务将包含零个文件，或者仅包含空文件。这些事务对视图没有影响，但它们被视为有效更新，并会触发计划以及所有相关的副作用，这可能导致计算浪费。空事务也会大大增加文件和事务的数量。

最好从源头上避免空事务，这通常是数据连接《同步》。数据连接总是会中止没有文件的事务，但仍可能生成空文件。空事务在使用自定义插件时可能是一个特殊问题；有时可能无法通过修改插件来避免空事务（例如，如果需要非空事务来更新数据连接增量元数据）。在其他情况下，无文件事务可能由转换或其他方式提交。

为了尽量减少这些空事务或无文件事务的影响，我们可以在流水线中最上游的转换节点显式中止事务。当检测到接收到的输入为空，或最终产生的输出为空时，我们可以在输出对象上调用 .abort()；这将导致作业及其对应的事务一并被中止。中止的事务会被有效取消，既不会触发后续调度，也不会产生任何副作用。中止事务能够切断链条，阻止空事务继续向流水线下游传播。同时，中止事务也不会计入失败统计（而有意让构建失败则会被计入失败统计）。

需要注意的是，中止的构建会被视为成功执行，并会推进输入事务指针。因此，在输入非空的情况下执行中止操作会导致数据丢失；如果因其他原因（例如前置条件校验失败）需要停止构建，这种数据丢失可能并非期望的结果。

变更日志数据集

变更日志逻辑允许我们在仅追加事务上实现编辑语义，从而在增量流水线中可靠地执行连接和聚合操作。然而，除了前文提到的文件和事务数量问题外，在仅追加事务上实现编辑语义可能会导致行数无限制增长，使得转换在进入状态解析阶段（或需要独立快照时）的性能日益恶化。

在这类流水线中控制行数稍显复杂。生成快照是可行的，当存在部分状态时，它对中间转换环节大有裨益（因为这些状态在完全重建时基本会消除）。但要想充分发挥快照的优势，逻辑必须将数据行收敛至其最新状态，但这并不总是我们想要的（在某些场景下，我们可能需要推算数据行在任意时间点的状态，而不仅仅是最新状态）。数据集投影的作用终究有限。此外，如果数据行并非原子单位（例如，最终可能只有结束事件而缺少匹配的开始事件），保留策略也可能无法达到预期效果。因此，在设计此类流水线时必须格外谨慎。
