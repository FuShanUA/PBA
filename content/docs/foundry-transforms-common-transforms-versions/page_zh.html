Transforms 版本

在 transforms 中，构建时所使用的 Spark 和 Java 版本取决于 transforms 的版本。Transforms 会在后台自动升级，以确保作业能够保持最新的安全性和性能改进。根据《Spark 版本控制策略》，各版本间的 API 保持一致，并启用旧版配置以防止行为变更。这些配置已在《Spark 迁移指南》中进行了说明。

环境页面

通过作业追踪器选择 Spark Details > Environment，即可访问环境页面。在此处，您可以查看构建的关键配置详情，包括 transforms 和 Spark 版本。

sparkModuleVersion 对应 transform 版本，而 sparkVersion 对应构建中使用的 Spark 版本。

运行时版本控制

每次运行构建时都会选择一个 transforms 运行时版本。系统会使用一种名为“裁决”的机制，以确保您的构建不会升级到无法正常工作的版本；裁决过程会择优测试升级方案，并在必要时回退至旧版本。当推出新的 Spark 版本时，如果条件允许，系统会自动进行测试并将其应用到您的管道中，全程无需用户干预。如果升级失败，您的管道将继续在旧的 Spark 版本上运行。每次 Spark 版本升级都会伴随一条公告。若要临时保留现有的 Spark 版本，请使用模块固定功能将管道锁定在固定版本上。

在运行时版本发布大约一个月后，系统会发送仓库升级提示，以告知用户在无法自动升级时出现的问题。您应该能够根据仓库中的检查错误来调试这些情况。

调试作业

如果您怀疑 transforms 升级导致管道中断，请使用调试作业来诊断问题。对于失败的作业，选择 Actions 和 Rerun as debug job，指定一个之前成功运行的版本，以确认 transform 版本的变更是否为导致中断的原因。如果遇到 transforms 升级导致管道中断的情况，请联系 Palantir 支持团队以获取进一步的帮助。

以下情况目前不支持运行调试作业：

  • 使用带有机密信息的外部 transforms。
  • 使用凭据的旧版外部 transforms。

对于上述两种情况，调试作业预计会失败并报 OneTimeAccessKeyService:OneTimeAccessKeyNotFound 错误。
