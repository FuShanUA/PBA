用例生命周期

一个用例是 一个有时间限制 的努力，由 专门的团队 执行，以在平台上为一组 用户 提供 新功能。

一个用例不是“集成源系统 X”或“应用 ML 技术 Y”。相反，一个 用例 关注的是我们需要为用户提供的 功能，而不是过程中的实现细节。将开发工作视为一个用例迫使我们立即面对项目的价值主张。由于这个新功能，谁的工作生活会得到改善？我们可以衡量什么来知道我们是否成功？

在构建我们的用例的过程中，我们将解锁 Foundry 的许多累积功能，例如：

  • 通过集成、数据科学技术和业务逻辑来丰富数据；
  • 构建一个灵活、可重用的 数据资产，作为我们组织和业务流程的数字孪生体；
  • 并创建 可重用的用户界面，通过将决策记录为数据来闭合操作循环。

以我们的用例及其结果为指导，我们保持与现实世界价值的联系，即使在深入研究Foundry中的具体实现复杂性时也是如此。

启动一个用例

在Foundry上开发用例的工艺涉及分解结果的功能需求

下面描述的解决方案设计过程是应对这一挑战的一种方法。它专注于将用例需求提炼成一种格式，以指导接口实现和数据增强的决策。这些决策反过来又影响本体论设计，该设计作为用例API，并从最终用户交互中抽象出管道实现的细节。

通过这种方式，解决方案设计框架鼓励如下面图像所示的整体方法：

Palantir, Foundry, Apollo, Gotham, AIP 保持不变。 Foundry -> Foundry Apollo -> Apollo Gotham -> Gotham AIP -> AIP Ontology -> 本体论 ontology -> 本体论 Pipeline -> 管道 pipeline -> 管道 Transform -> 转换 transform -> 转换 Workshop -> Workshop workshop -> Workshop Code Repositories -> 代码仓库 Code Workspaces -> 代码工作区 Data Integration -> 数据集成 Model Integration -> 模型集成 Object -> 对象 object -> 对象 Dataset -> 数据集 dataset -> 数据集 Tenant -> 租户 tenant -> 租户 Marking -> 标记 marking -> 标记 Permission -> 权限 permission -> 权限 Role -> 角色 role -> 角色 Function -> 函数 function -> 函数 Action -> 操作 action -> 操作 Property -> 属性 property -> 属性 Interface -> 接口 interface -> 接口 内容： 将解决方案设计围绕 转换、结构化 和 与用例数据交互 进行分组，可以将开发流程从纯粹的以用户为中心的需求转向以平台为中心的组件。有了这个框架，我们可以进一步考虑以下内容：

  • 常见模式的蓝图，
  • 表示额外开发复杂性的标志，以及
  • 在可用选项之间的默认选择和选择替代方案时的考虑因素。

我们还可以评估开发复杂性与严格遵守业务需求之间的权衡。通过解决解决方案设计框架，应该能够重新评估优先级。

例如，假设原始的业务需求是呈现一个特定的图表并包含一些特定的表单输入，这可能需要奇特的本体论配置和在 Slate 中大量的开发工作量。如果改用功能等效的可视化效果和操作（满足功能需求），那么我们可以使用 Workshop，在前端开发上的投入仅为原来的十分之一，并且保持更简洁的本体论数据结构。
