流

与数据集类似，流代表了从数据接入 Foundry 到被下游系统处理这段时间内的数据形态。流是对一系列行的封装，这些行同时存储在持久化的“热缓冲区”和由文件系统支撑的“冷存储”中。使用 Foundry 流的优势在于，它们不仅提供了与 Foundry 数据集相同的基础操作（如分支管理、版本控制、权限管理、模式管理等），还能提供低延迟的数据视图。

流本质上是表格型的，因此也是结构化的。它们以 Avro ↗ 等开源格式存储，并附带列本身的元数据。这些元数据作为模式与流一起存储。

流存储

热缓冲区

当记录流入 Foundry 流时，它们会被存储在热缓冲区中，以低延迟状态提供给所有支持读取流的下游应用程序。这个热缓冲区对于实现低延迟转换和数据可用性至关重要。它为数据接入提供了“至少一次”的语义，并为平台内的数据处理提供了可选的“精确一次”语义。

冷缓冲区

Foundry 流中的所有数据每隔几分钟就会从热缓冲区转移到冷存储中。我们将此过程称为“归档”，它会使数据以标准 Foundry 数据集的形式可用。这意味着任何 Foundry 应用程序都可以对流数据进行操作，即使它不实时处理来自热缓冲区的数据。Foundry 流的数据集视图在平台中的表现与标准的 Foundry 数据集完全一致。

流处理

从流中读取数据

启用了低延迟功能的 Foundry 产品能够读取数据的混合视图。通过同时读取热存储层和冷存储层的数据，产品可以提供完整的数据视图。该视图使产品能够访问仍处于热存储中的低延迟记录，以及已转移到冷存储中的较旧数据。这样一来，Foundry 流就能同时兼具热存储的低延迟优势和冷存储的低成本优势。

事务

与标准的 Foundry 数据集不同，流本身没有固有的事务边界。相反，每一行都被视为独立的事务，并且状态是按行跟踪的。这使得流可以在细粒度级别上被读取，从而让 Foundry 能够支持基于推送的转换，而无需进行批处理或轮询。

流类型

您可以根据每个流的吞吐量需求为其配置流类型。这些流类型设置适用于流向上文提到的热缓冲区存储写入数据的方式。只有当流指标表明流在写入热缓冲区存储时遇到瓶颈，您才需要修改流类型设置。延迟和吞吐量之间存在权衡关系，因此请在检查流指标后再设置高吞吐量/压缩流类型。 我们支持以下流配置：

  1. 高吞吐量：适用于每秒发送大量数据的流。启用此流类型可能会以牺牲一定的延迟为代价来换取更高的吞吐量。因此，在启用之前，您应先检查流指标。如果平均批次大小等于最大批次大小，或者作业因 Kafka 生产者批次过期而失败，您可能需要启用高吞吐量设置。
  2. 压缩：启用此配置后，在将数据生产至热存储缓冲区时，会对消息批次进行压缩。压缩有助于减小发送数据的体积，从而降低网络带宽和存储开销，代价是进行压缩与解压缩会占用额外的 CPU 资源。仅当数据流包含大量重复字符串，且出现网络带宽受限症状（如延迟非零、吞吐量低于预期或记录丢失）时，我们才建议启用此流类型。

在定义页面创建新数据流时，可以设置流类型。对于已有数据流，也可以在其流设置中更新流类型。具体操作为：在 Foundry 中导航至您的流式数据集，并在工具栏中选择详情。随后，进入流设置。在此处即可更改流类型并启用或禁用压缩。

分区

为保持高吞吐量，Foundry 会将输入流拆分为多个分区以进行并行处理。创建数据流时，您可以通过吞吐量滑块来控制生成的分区数量。需要注意的是，尽管数据已被分区，但对该流的所有读写操作仍表现得如同在单一分区上进行一样。此行为为 Foundry 数据流的生产者和消费者提供了设计上的透明度。

对于特定数据流，每增加一个分区，都会提升该流可处理的吞吐量上限。一个实用的经验法则是，每个分区大约能提升 5mb/s 的吞吐量。

流式作业

所有流式作业在内部均呈现为作业图，用以直观展示您的流处理管道。在数据处理过程中，数据会沿着有向边在作业图中流动，直至抵达数据接收端。

检查点

Foundry 流式处理通过将活动状态和当前处理位置存储在检查点中，来提供数据处理过程中的容错能力。

检查点由数据源周期性地生成，并与来自数据源的数据一道在作业图中流转。 一旦检查点到达作业图末端的所有数据接收端，数据源在该检查点之前发出的所有行记录必然也已抵达接收端。

检查点允许流式作业从最新检查点的位置重启，免去了重新处理已消费数据的麻烦。检查点不仅存储了作业图中每个算子的状态，还记录了流中最后处理的数据点。在流式作业的作业详情页面上，您可以实时查看该流最近几次检查点的状态、体积及耗时。

流式一致性保证

Foundry 的流式处理提供两种一致性保证：AT_LEAST_ONCE（至少一次）和 EXACTLY_ONCE（精确一次）。

AT_LEAST_ONCE 语义

AT_LEAST_ONCE 语义确保消息至少被向下游传递一次，但在发生检查点失败或重试的情况下，同一消息可能会传递多次。这意味着可能会出现重复数据，下游消费应用的设计必须能够处理或容忍重复消息。

AT_LEAST_ONCE 语义的优势

  • 确保不会丢失任何消息，提供高级别的消息持久性。
  • 通常比 EXACTLY_ONCE 语义的延迟更低，因为消息无需因等待记录簿记而阻塞，即可直接传递。

AT_LEAST_ONCE 语义的劣势

  • 要求下游消费应用具备处理重复消息的能力。

EXACTLY_ONCE 语义

`EXACTLY_ONCE` 语义确保每条消息仅被投递和处理一次，杜绝消息重复或丢失。这是最高级别的消息投递保障，能够大幅简化消费端应用的设计。

`EXACTLY_ONCE` 语义的优势

  • 确保消息零丢失，提供高水准的消息持久性。
  • 免去消费端应用处理重复消息或实现幂等性处理的负担。
  • 由于每条消息仅处理一次，确保了处理结果的一致性。

`EXACTLY_ONCE` 语义的劣势

  • 由于需要额外的协调和跟踪机制来防止消息重复，其延迟通常高于 `AT_LEAST_ONCE` 语义。Foundry 流处理通过检查点机制解决了这一问题。

延迟权衡

在 `AT_LEAST_ONCE` 和 `EXACTLY_ONCE` 语义之间做选择，往往需要在延迟与处理复杂度之间进行权衡。`AT_LEAST_ONCE` 语义通常具备较低的延迟，因其无需复杂的协调或跟踪机制，但这会将处理重复数据及维持一致性的责任更多地推给消费端应用。启用 `EXACTLY_ONCE` 后，数据记录只有在每次检查点完成后（默认时长为两秒）才会向下游可见。值得注意的是，记录依然以流式速度进行处理，只是在“最终确认”后才会在下游显现。

另一方面，`EXACTLY_ONCE` 语义提供了更强的保障，通过确保每条消息仅被处理一次，能够简化消费端应用的设计。然而，这种保障是以额外的系统开销导致延迟增加为代价的。

目前，Foundry 中的流式数据源在数据抽取和导出方面仅支持 `AT_LEAST_ONCE` 语义。流式管道则同时支持 `AT_LEAST_ONCE` 和 `EXACTLY_ONCE` 语义，可在 Pipeline Builder 的 Build settings 部分进行配置。
