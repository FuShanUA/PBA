SAP ERP

SAP ERP 连接器允许您将 Foundry 连接到 SAP 的本地 ERP 中央组件（ECC）和 S/4 HANA（本地和 云私有版 ↗）。SAP ERP 连接器使 Foundry 能够与各种类型的 SAP 数据进行交互，包括：

  • SAP 应用程序表（SAP Application Tables）
  • SAP CDS 视图（SAP CDS Views）
  • SAP HANA 信息视图（SAP HANA Information Views）
  • SAP BW 内容提取器（仅远程连接）
  • SAP 事务代码和 ABAP 报表（SAP Transaction Codes and ABAP Reports）
  • SAP 函数模块（SAP Function Modules）
  • SAP 媒体（DMS 文档）

使用 SAP ERP 源需要在目标 SAP 应用程序层上安装 Palantir Foundry SAP 应用程序 2.0 插件。

设置

  1. 打开 数据连接 应用程序，并在屏幕右上角选择 + 新建源。
  2. 从可用的连接器类型中选择 SAP ERP。
  3. 按照后续配置提示继续使用以下部分中的信息设置您的连接器。

了解更多关于 在Foundry中设置连接器 的内容。

连接类型

SAP ERP 连接器支持两种连接类型：

连接类型描述
直接直接连接到 SAP 系统。这是默认设置。
远程（通过网关）通过网关连接到远程 SAP 系统。启用 通过网关连接 切换并提供一个 上下文 值以标识远程系统。

连接类型决定了哪些 SAP 对象类型 可用。了解更多关于 远程连接 和 远程代理配置 的信息。

认证

SAP ERP 连接器支持以下认证方法：

认证方法描述
基本认证提供安装 连接器 时创建的技术用户的用户名和密码。
身份验证令牌提供一个用于身份验证的令牌。
自定义身份验证头提供一个自定义的身份验证头。
无身份验证如果在代理机上通过证书设置了身份验证，请使用此选项。

网络与连接

请确保正确配置 出站策略，以允许 Foundry 访问 SAP 系统。对于本地 SAP 环境，通常需要 代理代理策略 来正确路由流量。

许多 SAP 系统使用自签名证书，在首次配置连接时可能会导致 SSL 握手异常。请确保您从系统中获得了正确的自签名证书，并 将它们添加到源中。

SAP 数据加速器

通过 SAP 数据加速器（SAP 通过 SAP Cloud Connector 暴露系统的托管连接服务）访问此 SAP 系统，将源的 主机 和 端口 设置为您的云连接器虚拟主机和端口。然后，分配 Palantir 提供的客户端证书，允许出站到 SAP 数据加速器控制平面和数据平面，并添加 SAP 数据加速器 属性以及您的控制平面 URL。此路径需要使用 Foundry 工作程序。

批量同步

创建一个同步

  1. 从源概览页面中，选择 + 新增在批量同步旁边。
  2. 配置标准设置：名称、目标数据集和计划。
  3. 将事务类型设置为以下之一：
    • APPEND — 用于增量更新。有关详细信息，请参阅 增量同步。
    • SNAPSHOT — 用于全量加载。
  4. 从下拉菜单中选择一个 SAP 对象类型（请参见下方的 SAP 对象类型）。
  5. 输入 对象名称。当你输入时，字段会根据所选的对象类型建议匹配的对象。
  6. 可选地，在 附加项 选项卡中配置其他参数。

SAP 对象类型

增量同步

增量同步支持从同一张表进行追加式事务。要启用增量同步，请将事务类型设置为APPEND。

可用的增量模式取决于对象类型：

增量模式ERP 表CDS 视图HANA 视图BW 内容提取器
多个字段✓✓✓
拼接字段✓ ✓✓
更改文档表（Change document table）✓
双表（Twin table）✓
SAP 内置增量（SAP built-in delta）✓
  • 多个字段（Multiple fields）: 导入其中任意指定字段大于或等于已导入最大值的行。用逗号分隔字段。
  • 拼接字段（Concatenate fields）: 与多个字段相同，但将字段值拼接在一起，而不是使用 OR 组合。
  • 更改文档表（Change document table）: 根据 SAP 更改文档表中的更新导入行。有两种子模式： 仅 CDPOS 或 CDHDR 和 CDPOS。
  • 双表（Twin table）: 当一个单独的“双表”中的字段满足增量条件时，从目标表导入行。
  • SAP 内置增量（SAP built-in delta）: 使用提取器原生的变化数据捕获机制。

增量字段应为一个单调递增的值。系统使用“大于等于”比较，以避免在同步过程中遇到特定日期中途停止时丢失数据。重复值可能出现在Foundry数据集中，应在转换管道的第一步中将其删除。

交易代码和功能模块对象类型不支持增量同步；它们始终作为快照运行。

对于表类型的增量同步，每次同步的最大行数设置限制了每次运行返回的近似行数。如果间歇性问题（如网络故障）中断长时间运行的同步，可以使用它将大型表的初始同步拆分为一系列较小、更稳定的运行。

双表配置

当使用双表作为增量模式时，增量双表设置指定双表，增量双表映射 设置定义了主表和副表之间的连接条件。映射条目使用以下形式：

{PRIMARY_TABLE_NAME}-{FIELD_NAME}={TWIN_TABLE_NAME}-{FIELD_NAME}

使用分号（AND 运算符）组合多个连接条件。

重置增量状态

要强制进行全面重新加载并重新初始化增量摄入，请在同步中启用 重置增量状态 切换开关。这将对 SAP 的数据进行完整快照，替换数据集中的所有文件。同步完成后，请禁用切换开关，以便后续同步恢复增量追加。

重置增量状态支持 ERP 表、CDS 视图、HANA 视图和 BW 内容提取器对象类型。

Filters

Add condition groups, then configure field, operator, and value rows within each group. All field names must match the SAP data dictionary.

  • For date columns, use the format YYYYMMDD (for example, 20210101 for January 1, 2021).
  • The is like and is not like operators support the * wildcard (for example, A*12* matches any string starting with A and containing 12).
  • Filter values can also use dynamic filter keywords and date calculation functions.

Dynamic filters

Dynamic filter values use special keywords and date calculation functions for more flexible filter expressions. Available from add-on version SP26 and later. Use them anywhere a filter value is accepted.

Fixed keywords

KeywordDescription
[CURRENTYEAR]Current year in YYYY format.
[TODAY]Today's date in YYYYMMDD format.
[LASTDAYOFMONTH] Last day of the current month in YYYYMMDD format.
[LASTDAYOFLASTMONTH]Last day of the previous month in YYYYMMDD format.
[FIRSTDAYOFMONTH]First day of the current month in YYYYMMDD format.
[FIRSTDAYOFLASTMONTH]First day of the previous month in YYYYMMDD format.

Date calculation functions

FunctionDescription
[ADDDAY]Adds days to the selected date. Example: [ADDDAY(22102022,1)] → 23102022.
[ADDMONTH]Adds months to the selected date.
[ADDYEAR]Adds years to the selected date.
[GETMONTH]Returns the month of the selected date as a 2-digit value (01–12).
[GETDAY]Returns the day of the month as a 2-digit value.
[GETYEAR]Returns the year of the selected date.

Functions can be used directly with fixed keywords or nested. For example, [ADDDAY([TODAY], 1)] or [GETDAY([ADDDAY([FIRSTDAYOFMONTH], -1)])].

Transaction Code parameters

When the object type is Transaction Code, the object name refers to either an SAP transaction code or an Advanced Business Application Programming (ABAP) report, and the following additional fields are available:

ParameterDescription
Program typeSpecifies whether the object name refers to a transaction code or an ABAP report. Required.
Selection variantName of an SAP selection variant to apply before running the transaction or report.
Output variantName of an SAP output variant (layout) to control the structure of the result list.
Use spool ingestionWhen enabled, captures the report output from the SAP spool system rather than from runtime memory. Required when the report output exceeds 2 GB.
Hide subtotal and summary rows When enabled, drops aggregate subtotal and summary rows from the ingested data.

CDS view parameters

Core Data Services (CDS) views declared with the WITH PARAMETERS clause require their parameters to be supplied at sync time. When the object type is CDS View, the CDS view parameters optional property accepts rows of parameter name and parameter value pairs. Parameter names must match the names declared on the CDS view; unknown or missing parameters cause the sync to fail. Empty rows are discarded on save.

Function module parameters

When the object type is Function Module, the object name refers to a function module, typically a Business Application Programming Interface (BAPI). The following additional fields are available:

ParameterDescription
Function input parametersRows of parameter name and parameter value pairs that are passed as the function module's IMPORTING parameters. Required for most BAPIs.
Commit after function callWhen enabled, issues a COMMIT WORK statement after the function call. Required for standard BAPIs that rely on a commit to persist their changes. Off by default.

Media sets

The SAP ERP connector supports media sets for ingesting media files and documents stored in SAP's Document Management System (DMS).

To create a new media ingest:

  1. Navigate to the source overview page.
  2. Select + New next to Media set syncs.
  3. Define your media set format. Providing a specific format will enable you to use transformation steps specific to that format in downstream data transformation. Read more about media set formats.
  4. Use the source exploration tool, which organizes files per document types, to select the files you want to ingest, or define your own filters directly in the Filters section.

Exploration

When you select Subfolder or media file, you can either select a specific Document Type (like CPD) to ingest all associated documents, or select a single document to ingest only that specific file.

Media filters

The following filters can be applied to media set syncs:

FilterDescription
Exclude files already syncedSkips files whose path already exists in the media set.
Path matchesOnly syncs files matching a regular expression pattern. Defaults to .pdf.
Document typeFilters by SAP document type.
File size limitOnly syncs files within the specified size limit (in bytes).

Example: Select a Document Type

SAP Media Document Types are designated by an acronym and a plain English description of their purpose. When filtering by Document Type, you can use the acronym representation to narrow the scope of the media ingest.

Webhooks

SAP webhooks allow you to write data back to SAP by invoking BAPIs (Business APIs) from Foundry. See Webhooks for an overview of how to set up a webhook.

SAP webhooks support Foundry worker connections in Foundry-managed cloud compute. We recommend this runtime for SAP webhooks instead of executing them through a data connection agent. Existing SAP sources configured on a Foundry worker automatically benefit from cloud execution with no migration required; SAP sources running on a data connection agent should be switched to a Foundry worker to take advantage of cloud-executed webhooks.

The only task type available for SAP webhooks is sap-run-function-webhook-task-v0. The following example invokes BAPI_SALESORDER_CHANGE to modify the purchase date for a given sales document:

Copied!
1{
2    "function-name": "BAPI_SALESORDER_CHANGE",
3    "inputs": {
4        "SALESDOCUMENT": {{json sales-doc-id}},
5        "ORDER_HEADER_IN": {
6            "PURCH_DATE": {{json purchase-date}}
7        },
8        "ORDER_HEADER_INX": {
9            "UPDATEFLAG": "U",
10            "PURCH_DATE": "X"
11        }
12    },
13    "output": "RETURN"
14}

To target a remote SAP system, add a remote field to the task body:

Copied!
1{
2    ...
3    "output": "RETURN",
4    "remote": {
5        "context": "<SAP_CONTEXT_NAME>"
6    }
7}

Use in code repositories

SAP ERP sources can be imported into Python external transforms or functions , giving your code direct access to the SAP add-on's REST API. Use this when batch syncs, media set syncs, or webhooks do not provide enough control. For example, to invoke a BAPI synchronously from a Workshop action, loop over a large input dataset and call a BAPI per row, fetch specific DMS documents based on Foundry-side data, or chain multiple SAP calls in a single build.

The endpoints exposed by the add-on are defined in the foundry-sap-connector ↗ repository. The examples below use the following endpoints, all relative to the source URL (https://<host>:<port>/sap/palantir):

EndpointPurpose
POST /v2/function/{functionName}/writebackSynchronously invoke a single function module (BAPI).
POST /v2/function/{functionName}/parallel_writebackSynchronously invoke a function module with bulk input, parallelized server-side.
GET /v2/get_documentsList DMS documents, optionally filtered by document type, file format, file name, or object link.
GET /v2/get_documentDownload a single DMS document by ID.

Before using these examples, import the SAP ERP source into your repository:

  • For transforms, follow Set up external transforms and use the external_systems decorator.
  • For functions, follow Make API calls from functions and declare the source in the @function(sources=[...]) decorator.

Example: Writeback to SAP via BAPI (transform)

The following external transform invokes BAPI_SALESORDER_CHANGE once per row of an input dataset to update the purchase date on each sales document, and records the SAP RETURN table in an output dataset for review.

Copied!
1from transforms.api import Input, Output, transform
2from transforms.external.systems import external_systems, Source, ResolvedSource
3
4
5@external_systems(
6    sap_source=Source("<source_rid>"),
7)
8@transform(
9    sales_order_updates=Input("<input_dataset_rid>"),  # columns: sales_doc_id, purchase_date (YYYYMMDD)
10    writeback_results=Output("<output_dataset_rid>"),
11)
12def update_purchase_dates(ctx, sap_source: ResolvedSource, sales_order_updates, writeback_results):
13    base_url = sap_source.get_https_connection().url
14    client = sap_source.get_https_connection().get_client()
15
16    results = []
17    for row in sales_order_updates.dataframe().collect():
18        response = client.post(
19            f"{base_url}/v2/function/BAPI_SALESORDER_CHANGE/writeback",
20            json={
21                "commit": True,
22                "input": {
23                    "SALESDOCUMENT": row.sales_doc_id,
24                    "ORDER_HEADER_IN": {"PURCH_DATE": row.purchase_date},
25                    "ORDER_HEADER_INX": {"UPDATEFLAG": "U", "PURCH_DATE": "X"},
26                },
27                "output": ["RETURN"],
28            },
29        )
30        response.raise_for_status()
31        results.append({
32            "sales_doc_id": row.sales_doc_id,
33            "purchase_date": row.purchase_date,
34            "status_code": response.status_code,
35            "sap_return": str(response.json().get("RETURN", [])),
36        })
37
38    writeback_results.write_dataframe(ctx.spark_session.createDataFrame(results))

Set "commit": true for standard BAPIs that require a commit statement to persist their changes. Use the metadata endpoint (GET /v2/function/{functionName}/metadata) to discover the expected input and output structure of a function before calling it.

Example: Writeback to SAP via BAPI (function)

To invoke a BAPI synchronously from a Workshop action, an Ontology action backend, or any other caller of a Foundry function, expose the writeback as a Python function with a source. The function takes typed inputs and returns the SAP RETURN structure to the caller.

Copied!
1from functions.api import function
2from functions.sources import get_source
3
4
5@function(sources=["sap_source"])
6def update_sales_order_purchase_date(sales_doc_id: str, purchase_date: str) -> str:
7    """Updates the purchase date on a sales order via BAPI_SALESORDER_CHANGE.
8
9    Args:
10        sales_doc_id: SAP sales document number.
11        purchase_date: New purchase date in YYYYMMDD format.
12
13    Returns:
14        A string representation of the SAP RETURN table, useful for surfacing
15        any error or warning messages back to the caller.
16    """
17    source = get_source("sap_source")
18    base_url = source.get_https_connection().url
19    client = source.get_https_connection().get_client()
20
21    response = client.post(
22        f"{base_url}/v2/function/BAPI_SALESORDER_CHANGE/writeback",
23        json={
24            "commit": True,
25            "input": {
26                "SALESDOCUMENT": sales_doc_id,
27                "ORDER_HEADER_IN": {"PURCH_DATE": purchase_date},
28                "ORDER_HEADER_INX": {"UPDATEFLAG": "U", "PURCH_DATE": "X"},
29            },
30            "output": ["RETURN"],
31        },
32        timeout=30,
33    )
34    response.raise_for_status()
35    return str(response.json().get("RETURN", []))

The source must have code import enabled and exports enabled for functions to use it. Surface SAP RETURN messages with a non-zero severity as user-facing errors so callers can react to them.

Example: Parallel writeback (transform)

Use the parallel_writeback endpoint from an external transform to send many BAPI calls in a single request. The SAP system spawns multiple background jobs and distributes the calls across them, which is significantly faster than sequential writeback calls for large batches.

Copied!
1from transforms.api import Input, Output, transform
2from transforms.external.systems import external_systems, Source, ResolvedSource
3
4
5@external_systems(
6    sap_source=Source("<source_rid>"),
7)
8@transform(
9    sales_order_updates=Input("<input_dataset_rid>"),
10    writeback_results=Output("<output_dataset_rid>"),
11)
12def update_purchase_dates_parallel(
13    ctx, sap_source: ResolvedSource, sales_order_updates, writeback_results
14):
15    base_url = sap_source.get_https_connection().url
16    client = sap_source.get_https_connection().get_client()
17
18    bulkinput = [
19        {
20            "SALESDOCUMENT": row.sales_doc_id,
21            "ORDER_HEADER_IN": {"PURCH_DATE": row.purchase_date},
22            "ORDER_HEADER_INX": {"UPDATEFLAG": "U", "PURCH_DATE": "X"},
23        }
24        for row in sales_order_updates.dataframe().collect()
25    ]
26
27    response = client.post(
28        f"{base_url}/v2/function/BAPI_SALESORDER_CHANGE/parallel_writeback",
29        json={
30            "commit": True,
31            "output": ["RETURN"],
32            "bulkinput": bulkinput,
33            "parallel": True,
34            "paralleljobs": 5,
35            "functioncallperjob": 10,
36        },
37    )
38    response.raise_for_status()
39
40    results = [
41        {"index": i, "result": str(r)} for i, r in enumerate(response.json())
42    ]
43    writeback_results.write_dataframe(ctx.spark_session.createDataFrame(results))

Tune paralleljobs (background jobs spawned in SAP) and functioncallperjob (calls handled per job) based on the workload and the available capacity on the SAP system. Set "parallel": false to fall back to sequential execution while still using a single bulk request.

Example: Fetch DMS documents to a media set (transform)

The following external transform combines the DMS list and download endpoints to fetch documents from SAP's Document Management System and write them to a media set. Use this pattern when the built-in media set sync does not provide enough control, for example, when document selection depends on Foundry-side data, or when you need to combine multiple filters before downloading.

Copied!
1import io
2from transforms.api import transform
3from transforms.external.systems import external_systems, Source, ResolvedSource
4from transforms.mediasets import MediaSetOutput
5
6
7@external_systems(
8    sap_source=Source("<source_rid>"),
9)
10@transform(
11    documents=MediaSetOutput("<media_set_rid>"),
12)
13def fetch_dms_documents(ctx, sap_source: ResolvedSource, documents):
14    base_url = sap_source.get_https_connection().url
15    client = sap_source.get_https_connection().get_client()
16
17    # List documents in DMS, filtered by document type and file format.
18    list_response = client.get(
19        f"{base_url}/v2/get_documents",
20        params={"DOCUMENTTYPE": "CPT", "FILEFORMAT": "pdf"},
21    )
22    list_response.raise_for_status()
23
24    for doc in list_response.json():
25        document_id = doc["DOCUMENT_ID"]
26        document_name = doc["DOCUMENT_NAME"]
27
28        download_response = client.get(
29            f"{base_url}/v2/get_document",
30            params={"DOCUMENT_ID": document_id},
31        )
32        download_response.raise_for_status()
33
34        documents.put_media_item(io.BytesIO(download_response.content), document_name)

To drive ingestion from a Foundry input dataset, for example, fetching only the documents referenced by a list of EBELN (purchase order) values, call /v2/get_documents with an OBJECTLINK filter, or pass each DOCUMENT_ID directly to /v2/get_document from the input rows.
