API: ModelAdapter 参考

一个 模型适配器 是一个发布的 Python 库，它在 Foundry 和存储的模型工件之间提供通信层，使 Foundry 能够加载、初始化并运行任何模型。

要实现一个 ModelAdapter，您必须了解以下列出的类：

ModelAdapter 实现

ModelAdapter 类是一个抽象基类，所有模型适配器实现都必须继承。所有模型适配器必须实现四个抽象方法：

  1. load()
  2. save()
    • 请注意，load() 和 save() 对于使用 默认序列化程序 定义的模型适配器不是必需的。
  3. api()
  4. predict() 用于单输出，run_inference() 用于多输出

模型 save() 和 load()

palantir_models.serializers 库提供了许多 默认序列化程序，可用于常见建模框架的模型序列化（保存）和反序列化（加载）。

在某些情况下，用户可能需要为模型序列化或反序列化实现自定义逻辑。例如，当您使用的建模框架没有可用的默认序列化程序，或者您需要手动控制在任何给定时间加载到内存中的模型时，这可能是必要的。

在这些更复杂的情况下，请参见下面 保存 和 加载 的实现。

Save

ModelStateWriter

一个 ModelStateWriter（模型状态写入器） 会被传递给 ModelAdapter.save 方法，以便 ModelAdapter 可以将模型工件保存/序列化到 Foundry 存储中。

8        """
9        Open a file-like object to serialize model artifacts and parameters.

13        """
14        Put a local file in this model's repository.
15        :param asset_file_path: If provided, the file will be placed at this path in the repository.
16            Otherwise the file will be placed in the root directory of the repository.
17        """

19    def put_directory(self, local_root_path, asset_root_path = "/"):

每当通过 model.publish() 在转换中发布模型适配器时，就会调用 save() 方法，此时 ModelStateWriter 的内容会被打包成一个 zip 文件，并存储在由新创建的模型版本引用的工件仓库中。

如果模型由容器镜像或镜像支持，则该字段是可选的，并且仅在该情况下才会提供给ModelAdapter.load方法。上下文对象包括一个共享目录挂载路径和从容器名称到其服务URI的映射。每个容器可以有多个服务URI，因为拥有多个开放端口是有效的。

模型输入
ModelInput（模型输入）类型包含可以在 ModelAdapter.api 方法中定义的输入类型。模型适配器支持以下输入类型：
  1. 表格数据
  2. 参数
  3. 文件系统
  4. 媒体引用 [Beta]
  5. 对象
  6. 对象集
Beta
MediaReference（媒体引用）支持处于 Beta（测试） 阶段，仅在Python转换中支持推理，可能在您的注册中不可用。带有模型的MediaReferences（媒体引用）不支持自动评估、批量部署或建模目标应用程序中的实时部署。功能可能会在开发过程中发生变化。

表格输入
TabularInput用于指定提供给model.transform()的输入应为表格类型。在该模型适配器推理逻辑的上下文中，此输入的类型将作为df_type参数，如api()方法中所指定。如有必要，将执行适当的数据类型转换。允许的以下表格类型：
  1. TransformInput（转换）
  2. Pandas DataFrame（熊猫数据框）
  3. Spark DataFrame（火花数据框）

    • 注意：仅支持适配器API中指定df_type=DFType.SPARK的表格输入。对于指定DFType.PANDAS的表格输入，不支持火花数据框的转换。

参数输入
ParameterInput 用于指定提供给 model.transform() 的输入应为参数类型。参数是用 type 参数指定类型的常量值输入。以下类型被接受为参数输入：
  1. str
  2. int
  3. float
  4. bool
  5. list (*)
  6. dict (*)
  7. typing.Any (*)
  8. palantir_models.NDArray 用于 NumPy 多维数组
参数输入也可以指定一个 default 值，当 model.transform() 对应于模型适配器的 api() 方法中定义的参数输入未提供时使用。



查看 使用这些类型时的具体要求，当 将模型作为函数发布 时。
文件系统输入
FileSystemInput 用于指定提供给 model.transform() 的输入应为文件系统类型。只有 TransformInputs（转换输入）支持作为 FileSystemInput 的输入类型。
媒体引用输入
MediaReferenceInput 用于指定提供给 model.transform() 的输入应为媒体引用。媒体引用应为 str 类型，并包含完整的媒体引用对象定义。模型适配器会将此媒体引用字符串转换为 MediaReference 对象，该对象包含与被引用媒体项进行交互的方法。



对象和对象集输入
对于 Object 或 ObjectSet 输入，对象类型是在定义模型适配器 API 中的输入时指定的。此对象类型将从为选定的 本体论 SDK（Ontology SDK） 导入。
在进行模型推理时，可以通过以下方式将对象或对象集传递给模型：
API 强制执行
作为一般原则，在推理时模型适配器 API 并不严格强制执行，尽管对于 实时推理 通常更为严格，并且特别是对于 函数（Functions），它们是强类型的。
参数类型始终被强制执行，任何输入到 model.transform() 的参数如果不符合指定的类型，将抛出错误。


模型适配器允许在推理时通过表格输入传递API中未指定的附加列。Foundry在将数据传递给predict方法之前，不会从输入中删除额外的列。这一点对于确保模型适配器与评估一起工作非常重要，如模型适配器API指南中所述。在批量推理的情况下，如果API中指定的一些列缺失，适配器不会抛出错误。MediaReference类型（目前处于Beta阶段）是一个例外。它要求列中的每个元素都是一个媒体引用字符串。在批量推理期间，它会在将每个元素传递给该模型适配器的推理逻辑（用于批量和实时推理）之前将其转换为MediaReference对象。
然而，在实时推理情况下，模型输入将被转换为其声明的类型（如果已定义），以确保类型安全。这适用于任何表格输入中的列。这种行为的一个副作用是，如果输入数据中没有找到具有指定类型的列，或者数据无法转换为指定类型，则会抛出错误。


对于对象输入
  1. 模型适配器 API 的 Object 输入中指定的 OntologyObject 类型的一个实例。
  2. 指定对象类型的主键。
对于 ObjectSet 输入
  1. 模型适配器 API 的 ObjectSet 输入中指定的 OntologyObject 类型的 ObjectSet 实例。
  2. 指定对象类型的 ObjectSet rid。

模型推理
predict() 方法
对于具有单个表格或参数类型输出的模型，可以使用 predict() 方法来定义模型适配器的推理逻辑，而不是使用 run_inference() 方法。此方法的参数将是模型适配器的 api() 方法中定义的输入对象的名称。虽然参数不需要保留与定义时相同的顺序，但名称必须匹配。
predict() 方法示例


以下示例定义了一个predict()方法，用于多表输入、单表输出的模型适配器。


Copied!
1import palantir_models as pm
2
3class ExampleModelAdapter(pm.ModelAdapter):
4    ...
5
6    @classmethod
7    def api(cls):
8        inputs = {"input_1": pm.Pandas(), "input_2": pm.Pandas()}
9        outputs = {"output_dataframe": pm.Pandas()}
10        return inputs, outputs
11
12    def predict(self, input_1, input_2):
13        resulting_dataframe = ... # Some inference logic using input_1 and input_2
14        return resulting_dataframe


在上面的例子中，两个输入（input_1和input_2）在predict()方法的签名中通过名称进行引用。该函数返回的数据框（resulting_dataframe）将被写入名为output_dataframe的单个输出。
run_inference()方法
对于多输出模型或写入文件系统的模型，必须通过run_inference()方法定义自定义推理逻辑。此方法接受两个参数：inputs和outputs。这两个参数都是NamedTuples，其名称对应于api()方法中定义的输入和输出的name参数。
输入
通过名称引用输入将访问传递给model.transform()的对象，该对象对应于同名的api()输入。

模型适配器的 run_inference() 方法中的 my_input_df 对象（对象引用了名为 "input_dataframe" 的输入，该输入是 Pandas 类型的表格输入）将等于从转换中传入的 my_input_data TransformInput 的 pandas 表示。
输出
通过名称引用输出将提供一个与同名的 api() 输出对应的可写对象。这些对象中的每一个都有一个 .write() 方法来指定将写入每个输出的数据。
示例：输出



给定以下 ModelAdapter 定义：

20        my_output_dataframe_two = do_something_else_to_input_and_return_a_new_dataframe(my_input_df)

以及以下对 .transform() 的调用：


Copied!
1@transform(
2    my_input_data=Input(...),
3    my_output_data_one=Output(...),
4    my_output_data_two=Output(...),
5    my_model=ModelInput(...)
6)
7def compute(my_input_data, my_output_data_one, my_output_data_two, my_model):
8    my_model_outputs = my_model.transform(my_input_data)
9    my_output_dataframe_one = my_model_outputs.output_dataframe_one
10    my_output_dataframe_two = my_model_outputs.output_dataframe_two
11    my_output_data_one.write_pandas(my_output_dataframe_one)
12    my_output_data_two.write_pandas(my_output_dataframe_two)


在转换中的 my_output_dataframe_one 和 my_output_dataframe_two 对象将等于写入模型适配器的 output_dataframe_one 和 output_dataframe_two 输出的物件（在本例中为 my_output_dataframe_one 和 my_output_dataframe_two）。
媒体引用
关于Foundry中的媒体引用信息，请参见 这里。
