分支操作类型

本体论操作与全局分支集成，使您可以在不影响生产环境的情况下在分支上测试操作类型。您可以运行操作、验证其配置，并在合并更改到 main 之前在隔离的分支上下文中观察编辑。

有关全局分支概念和工作流程的一般信息，请参阅 全局分支文档。

在分支上运行操作

您可以在分支上的 Workshop 模块中测试操作，以验证它们是否已正确配置。当所有相关对象类型在分支上被索引后，您可以运行该操作类型并在分支上查看结果编辑。

前提条件

要在一个分支上运行一个操作，所有该操作类型所编辑的对象类型都必须在该分支上进行索引。您可以通过单个对象类型页面或在本体论管理器中的操作类型页面对对象类型进行索引。

在分支上运行操作仅作为测试机制。分支上的操作更改不会合并回main。

在分支上使用函数支撑的操作

在分支上的函数支撑的操作的行为取决于它们是否是分支感知的。分支感知的函数可以在分支上进行修改并从该分支读取模式，而非分支感知的函数只从main读取模式。无论是否为分支感知，所有的函数支撑的操作只能在分支上执行，并且不会将更改写回到main。请参阅< a href="/docs/foundry/global-branching/integrations/">全局分支文档中的支持函数类型。

在分支上的副作用管理

以下信息特别适用于通过 操作 应用于分支的 webhook、具有外部调用的函数以及通知。通过其他方式应用的副作用将按照这些消费者定义的行为执行。

Webhooks

默认情况下，当在分支上应用操作时，webhook 不会执行。这种行为是为了防止在测试环境中意外写入外部系统。

在这种情况下，您将看到一个提示通知，表明此行为。

然而，在某些情况下，希望在分支上测试 webhook，例如访问 READ 端点。

要覆盖默认行为，请在本体论管理器中配置操作类型的安全和提交标准选项卡，以在分支上启用webhook执行。

如果在分支上启用了webhooks，则webhook将像在main上一样运行。因此，如果webhook被配置为访问外部生产环境，即使该操作在分支上执行，它也会继续这样做。

具有外部调用的函数

默认情况下，当在分支上应用操作时，具有外部调用的函数支持的操作逻辑不会执行；整个操作都会失败。这种行为是为了防止在测试环境中意外写入外部系统。

在这种情况下，您会看到一个提示失败的弹出通知，并附有对该行为的解释。

然而，有时在分支上进行测试是必要的，例如调用READ端点时。您可以通过在Ontology Manager中启用动作类型< strong >安全与提交标准选项卡中的外部调用功能来覆盖此限制。

如果启用了在分支上具有外部调用的功能，该功能将像在main上一样执行相同的外部调用。因此，如果该功能配置为访问外部生产环境，即使操作是在分支上执行的，它仍然会继续这样做。

通知

默认情况下，当在分支上应用操作时，不会发送通知。这种行为是为了防止在测试环境中意外通知收件人。

在这种情况下，您会看到一个提示此行为的弹出通知。

然而，有些情况下在分支上测试通知是可取的。

要覆盖默认行为，您可以在 Ontology 管理器中动作类型的 安全和提交标准 选项卡中启用分支上的通知。

此外，您还可以在动作在分支上运行时指定通知接收者：

  • 分支所有者： 将所有通知发送给分支所有者。
  • 默认接收者： 通知原始通知中配置的接收者。

已知限制

  • 对分支上的动作进行编辑仅用于测试，不会合并回 main。
  • 当在分支上运行动作时，默认情况下不会执行 Webhook 和电子邮件通知。
  • 当在分支上运行动作时，默认情况下调用外部系统的函数将失败。
