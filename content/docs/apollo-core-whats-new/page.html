Introducing Export and Import

Apollo Export and Import features are now available on all Apollo Hubs. The documentation for these workflows is now available in the public documentation.

Export and Import provide a seamless way to deploy software in environments that lack network access or connectivity to your source Apollo Hub. By managing the source of truth for the software catalog and configurations in a Source Hub accessible to most users, you do not need to solely rely on users with access to a restricted environment. This allows developers, operators, and other Apollo users to have a unified view for configuring, managing, and monitoring deployment of software across various network boundaries.

To achieve this, designate a Target Hub from another Apollo Hub, which will act as the Source Hub. The Source Hub will serve as the source of truth for some or all Product Release information for the Target Hub. Changes are initially made in the Source Hub, and then transferred to the Target Hub using an Export Pipeline.

Get started by creating an Export Pipeline.

2026-05-21

Configure Module rollout with promotion pipelines and Release Channel tracking

Modules control critical properties of Apollo Environments, such as what Entities are installed and their configuration. Previously, Module updates were instantly propagated to every Environment where the Module was installed. This meant releasing a new Module version had a large impact on Environments, with no clear path for mitigation if something went wrong.

The following updates enable you to control how Module updates are released in your Environments.

Module promotion pipelines

Module releases now support promotion pipelines, and can be promoted to Release Channels like Products. Note that only time-based promotion stages are supported, though adding support for canary-based promotion is on the development roadmap.

You can configure promotion pipelines for Modules by navigating to the Releases tab of the Module page and selecting Edit from the promotion pipeline section.

A Module with a promotion pipeline configured.

Module installations track Release Channels

All Module installations now track the Environment default Release Channel. Apollo will only propose a Module update once a Module release has been promoted to that channel. You can override the tracked Release Channel for both new and existing Module installations.

Override the default Release Channel for new Module installations using the Select Release Channel dropdown.

Override the default Release Channel for existing Module installations by selecting Update Release Channel.

We recommend keeping the default channel since Module installations should have the same risk tolerance as their Environment.

Recall Modules

Recalling a Module release will cancel all open change requests to update installations to that version, and prevents Apollo from proposing upgrades to that version.

Apollo will not automatically downgrade a Module installation on a recalled version to a non-recalled version.

Recall a Module from the Module page by selecting Recall Releases from the Actions dropdown.

Introducing Apollo Debug Terminals

Apollo Debug Terminals are now available on all Apollo Hubs, providing an interactive shell for debugging your Environments directly from Apollo. You can run kubectl commands to inspect the configuration, health, and logs of containers running in Kubernetes directly from your browser.

Each terminal session is time-bound to four hours, isolated from other users' sessions, and uses Kubernetes RBAC scoped to your Apollo roles. By default, terminals grant read-only access to a limited set of Kubernetes resources such as Pods, Services, and ConfigMaps. To launch a terminal session, navigate to the Terminal tab in the Environment you want to debug — you will need the Environment Terminal User role.

Example terminal session on an Apollo Environment.

When a debugging task requires elevated permissions, you can request a just-in-time (JIT) access policy from the same tab. Once an Environment Administrator approves the request, the additional permissions apply within seconds to both active and new terminal sessions. Each grant is time-bound and expires automatically after the policy's configured duration — you can extend an active grant if you need more time, or revoke it earlier at any point.

JIT access request dialogue.

Stagger Product upgrades with ramped rollouts

Ramped rollouts are now available on all Apollo Hubs, allowing you to stagger Product upgrades within a Release Channel. A ramped rollout is a rate-limiting mechanism that distributes Entity upgrades across a specified time window.

Ramped rollouts are useful for Products with a large number of Entities. Rather than upgrading all the Entities at once, ramped rollouts allow upgrades to happen gradually. This enables earlier detection of issues and reduces the risk of widespread incidents.

Diagram that shows how Product upgrades are staggered throughout the rollout window when ramped rollouts are used.

You can configure ramped rollouts in your Product's Upgrade settings. For each Release Channel in the promotion pipeline, you can define a rollout window over which upgrades should be distributed.
