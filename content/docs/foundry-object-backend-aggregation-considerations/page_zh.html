聚合注意事项

在Foundry中，根据您处理的数据的复杂性和数据量，由于高基数聚合的性质，应用程序可能无法以完全准确的方式显示结果（也称为“不精确聚合”）。

object-set-service API 有以下与聚合准确性相关的字段，您可以使用它们来在可用时提高准确性：

  • API调用者可以在请求中添加AggregationExecutionMode并将其设置为PREFER_ACCURACY。当设置时，API响应会变慢，但会提供更准确的结果，但不保证完全准确。
  • 聚合响应包含一个AggregateResultAccuracy字段，用于指示结果是否准确。

Foundry中的每个产品都有略微不同的行为，表明聚合可能是不精确的。请查看以下部分，了解不精确聚合在Foundry中可能的表现形式，以及用户在追求准确性时应考虑的问题。

如需更多指导，请联系Palantir支持。

对象浏览器和工作区

考虑以下两个直方图截图，显示来自包含2000万对象的数据集的数据。第一张截图是拉入对象浏览器的数据集，第二张是工作区内的工作区筛选列表小部件。

这两个直方图是通过两个聚合请求构建的。第一个请求尝试获取计数最多的前100个桶，接收的是一个近似响应。第二个请求再次获取相同100个桶的计数，但除此之外，还会过滤到仅这100个桶以确保计数准确。

虽然显示的计数是准确的，但第二个直方图在某种意义上仍然不准确，因为所显示的桶并不是实际计数最多的前100个桶，因为第一个聚合响应并不准确。

对象资源管理器和工作区请求OSS时未指定AggregationExecutionMode，OSS默认为PREFER_SPEED。

Quiver透视表和Workshop透视表

在Quiver和Workshop的透视表中，按计数降序排序列时，顶部桶不会以正确的顺序显示。在这种情况下，不精确的聚合可能由以下错误信息之一表示：

  • "column值过多，未全部显示"
  • "由于计算限制，显示的是近似结果"
  • "每属性仅加载前1,000个值。请过滤您的数据以获得更准确的结果。"

在下面的示例中，Example Bucket列没有按降序排列。

这些不是由计数确定的真实顶级桶，因为 Quiver 和 Workshop 在支撑透视表的聚合请求中没有指定排序。排序是在前端使用返回的桶完成的。

本体论 SDK（OSDK）

OSDK 设置为 PREFER_ACCURACY，其有限的聚合复杂性意味着每个查询响应将为 ACCURATE。

函数

函数始终使用 PREFER_ACCURACY，因此给定桶的值将是正确的。目前尚无法在函数调用期间同时 groupBy 和 orderBy。以下代码片段演示了后端中 groupBy 的当前示例以及内存中的顺序。

    @Function()
    public async aggregateOnMoreBucketsThanAuthorized(): Promise<TwoDimensionalAggregation<string, Double>> {
        // Aggregate and sum
        const aggregation = await Objects.search()._af20mInstancesObv2()
                 .groupBy(o => o.exampleBucket.topValues())
                 .count()

        aggregation.buckets.sort((b1, b2) => b2.value - b1.value);
        return aggregation;
    }
