聚合注意事项

在Foundry中，根据您处理的数据的复杂性和数据量，由于高基数聚合的性质，应用程序可能无法显示具有完整准确性的结果（也称为“不精确聚合”）。

object-set-service API 有以下与聚合精度相关的字段，您可以使用这些字段来在可用时提高精度：

  • API调用者可以在请求中添加AggregationExecutionMode，并将其设置为PREFER_ACCURACY。如果设置，则API响应较慢，但提供更精确的结果，但不保证完全准确性。
  • 聚合响应包含一个AggregateResultAccuracy字段，用于指示结果是否准确。

Foundry中的每个产品在指示聚合可能不精确方面都有略微不同的行为。请查看以下部分，了解不精确聚合在Foundry中可能出现的示例，以及用户在追求准确性时应考虑的问题。

如需更多指导，请联系Palantir支持团队。

对象资源管理器和工作区

请考虑以下两个直方图截图，显示来自包含2000万个对象的数据集的数据。第一张截图是将数据集导入对象资源管理器后的结果，第二张是工作区中的工作区筛选列表小部件。

这两个直方图是通过两个聚合请求构建的。第一个请求尝试获取按计数排名前100个桶，接收的是一个近似响应。第二个请求再次获取这100个桶的计数，但同时还会过滤到仅这100个桶以确保计数的准确性。

虽然显示的计数是准确的，但第二个直方图仍然不准确，因为显示的桶并不是实际按计数排名前100的桶，因为第一个聚合响应并不准确。

对象资源管理器和工作区向OSS发出的请求未指定AggregationExecutionMode，OSS默认使用PREFER_SPEED。

Quiver透视表和Workshop透视表

在Quiver和Workshop透视表中，按计数降序排序列时，顶级桶不会以正确的顺序显示。在这种情况下，不精确的聚合可能会出现以下错误信息之一：

  • "column的值过多，未全部显示"
  • "由于计算限制，显示的是近似结果"
  • "每属性仅加载前1,000个值。请过滤您的数据以获得更准确的结果。"

在下面的示例中，示例桶列并未按降序排列。

这些并不是Quiver和Workshop实际的顶级桶，因为Quiver和Workshop在支持透视表的聚合请求中没有指定排序。排序是在前端使用返回的桶完成的。

本体论 SDK (OSDK)

OSDK 默认设置为 PREFER_ACCURACY，其有限的聚合复杂度意味着每个查询响应都将为 ACCURATE。

函数

函数始终使用 PREFER_ACCURACY，因此给定桶的值将是正确的。目前无法在函数调用期间同时进行 groupBy 和 orderBy。以下代码片段演示了后端中 groupBy 的当前示例和内存中的排序。

    @Function()
    public async aggregateOnMoreBucketsThanAuthorized(): Promise<TwoDimensionalAggregation<string, Double>> {
        // 聚合并求和
        const aggregation = await Objects.search()._af20mInstancesObv2()
                 .groupBy(o => o.exampleBucket.topValues())
                 .count()

        aggregation.buckets.sort((b1, b2) => b2.value - b1.value);
        return aggregation;
    }
