如何使用 Apollo

在继续阅读本文档之前，我们建议您先了解 Apollo 的核心概念。

本文档解释了 Spoke 环境中的更改如何以 计划 的形式在系统中流动。本文档涵盖以下内容：

  • 发布通道的提升、环境设置、报告状态和约束条件。
  • 编排引擎如何考虑上述因素，决定为特定 Spoke 发布哪些计划，以及代理如何执行这些计划。

环境设置

Apollo Hub 使用环境设置信息来管理环境和实体。有关 环境设置 和 托管实体设置 的更多信息，请参阅相关文档。

如需了解更多关于查看或编辑环境设置的信息，请参见 管理环境 和 管理实体。

报告状态

在托管的 Spokes 中运行的代理会收集有关所有托管实体的信息，包括已部署的版本、当前配置、存活状态和就绪状态以及其他元数据，并将其返回到 Apollo Hub。此信息称为实体的报告状态。有关更多信息，请参阅 实体报告状态。

约束条件

约束条件是计划在被代理执行前必须满足的前提条件。Apollo 会评估这些约束条件，以确保只有在安全和可接受的情况下才会运行计划。有关更多信息，请参阅 约束条件。

编排引擎

编排引擎负责 产品发布渠道提升 过程，同时也会为运行在 Spoke 环境中的代理发出 计划。

发布通道推广

为了决定何时应将发布从一个发布通道推广到另一个，编排引擎会持续评估该发布是否已经通过或未能通过健康推广标准，这基于来自报告状态的信息。

如果评估成功，该发布将会被添加到在 发布通道推广管道 中配置的下一个发布通道中。

在评估未通过标准的情况下，编排引擎会自动召回该发布，以确保所有环境都回滚到该版本。

对 spoke 环境进行更改

根据 计划生命周期 所述，编排引擎会评估在 spoke 环境中的控制平面中运行的代理应生成哪些计划。

请注意，Apollo 没有为 Environment 或 Entities 设置特定的目标状态。相反，Apollo 会提出满足 Environment 或 Entities 配置约束的 Plans。例如，一个 Entity 可以定义 Product 和 Release Channel，但不定义具体版本。

执行变更

在 Spoke 的 Control Plane 中运行的 Agents 会持续轮询 Orchestration Engine 以获取新的 Plans。一旦一个 Plan 通过了所有相关的 Constraints，Orchestration Engine 将会将该 Plan 发送给 Environment 中的相关 agent 执行。

一旦 agent 执行完 Plan，agent 会反馈该 Plan 是否成功。如果 Plan 失败，并且 Environment 的状态与 Plan 开始前的状态不同，Orchestration Engine 将会发出一个 rollback Plan 以恢复到 Plan 开始前的状态。
