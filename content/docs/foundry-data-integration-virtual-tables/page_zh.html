虚拟表

虚拟表允许你查询受支持数据平台中的表，而无需事先将数据存储在 Foundry 数据集中。

虚拟表充当指向 Foundry 外部源系统中表的指针。虚拟表屏蔽了底层源系统和存储格式，让你能够构建无缝结合来自不同源系统数据的工作流。虚拟表还可以与存储在 Foundry 中的数据集相结合，作为灵活架构的一部分，在这种架构下，数据无需集中存放在同一位置。你还可以创建新的虚拟表作为 Foundry 数据转换的输出，从而实现存储完全外部化、由 Foundry 负责编排、安全及其他功能的工作流。

虚拟表由以下元素定义：

  • 与源存储系统的连接（例如，源 URL 或凭证）。此连接通过在 Foundry 的数据连接应用程序中设置源来建立。
  • 用于在源系统中标识该表的定位器（例如，数据库、模式和表名）。

与 Foundry 中的任何资源一样，虚拟表受 Foundry 的安全与权限模型管辖，并可在各种 Foundry 应用程序中打开或使用。

为虚拟表设置连接

支持虚拟表的数据源在数据连接应用中进行设置。选择您要使用的数据源，然后在源配置中导航至虚拟表标签页。请遵循源文档以及其中列出的使用虚拟表的任何要求。

创建虚拟表

所有受支持的数据源都允许您在 Foundry 中从源系统注册单个表。表格型数据源还支持一次性批量注册多个虚拟表。部分数据源还支持自动注册，系统会定期将配置凭证可访问的源系统中的所有表注册到指定项目中。

要注册虚拟表，请在数据源的虚拟表标签页中选择创建虚拟表。浏览可用表并选择要注册的表。除非您选择其他位置，否则虚拟表将创建在数据源的默认输出文件夹中。

批量注册

在使用 Databricks 平台、BigQuery 和 Snowflake 平台等表格型数据源时，您可以一次性批量注册多个虚拟表。首先，从左侧面板中选择一个或多个外部表。使用右侧面板可以更改新虚拟表的保存位置或更新其名称。需要注意的是，在 Foundry 中更改虚拟表的名称并不会更改源系统中的表名。

自动注册

启用自动注册后，您将创建一个新的 Foundry 项目，虚拟表会自动创建于其中。该项目中的文件夹层级将映射源系统的结构，并随着源系统中新表的创建而定期更新。当源表被删除时，项目中相关的虚拟表不会被自动删除，但访问它们将无法加载任何数据。

要启用自动注册，您必须在 Foundry 中拥有项目创建权限。

该项目由 Foundry 管理，用户无法在其中手动创建或更新资源。该项目中注册的虚拟表可导入其他项目，用于工作流开发。

启用自动注册后，可针对该项目设置权限和访问控制，数据主人随后可通过访问侧边栏进行管理。

Databricks 平台数据源的标签过滤

为 Databricks 平台数据源配置自动注册时，可选择指定一组表标签进行过滤。只有在 Databricks 平台 TABLE_TAGS ↗ 系统表中设置了至少一个此类标签的表才会被注册。

代码仓库中的虚拟表

在代码仓库中使用虚拟表时，使用这些虚拟表的转换操作会根据在数据源上配置的出口策略自动获取网络出口权限。为连接到数据源，系统必然会使用该数据源上配置的凭证。此行为与外部转换类似。

必须在数据源上启用以下设置：

  1. 代码导入：允许将数据源导入并在代码仓库中使用。有关此设置的更多详细信息及启用方法，请参阅此处。
  2. 导出控制：用于控制允许在基于虚拟表的 Python 转换输入中使用哪些安全标记和组织。有关此设置的更多详细信息及启用方法，请参阅此处。

配置数据源并将其导入代码仓库后，即可使用 transforms.api.Input 将虚拟表作为Python 转换的输入，其使用方式与数据集完全一致。增量计算拥有与数据集一致的 API，并受部分数据源支持。更多信息请参阅特定数据源文档。

通常，虚拟表支持作为 Python、SQL 和 Java 转换的输入。仅 Python 转换支持将新建虚拟表作为转换输出，而 SQL 和 Java 转换仅支持向已有虚拟表写入数据。

查看虚拟表详情

在数据集预览的详情面板中，您可以查看虚拟表的关键信息。

这包括：

  • 增量：若支持此功能，您可以基于该表配置增量管道，使下游构建仅处理新增或变更的数据，无需重新处理所有行。
  • 版本控制：若支持此功能，该表将提供版本控制能力，允许 Foundry 检测更新，并在数据未发生变更时跳过不必要的下游构建。

虚拟表输入的更新检测

更新检测功能可监控虚拟表，并检测源系统中该表的变更情况。借此，您可以将虚拟表作为依赖资源（如管道或对象）的构建触发器。

仅当您的虚拟表输入已用于创建下游资源（如下游表、数据集或对象）时，更新检测才可用。

更新检测面向由外部管理并在 Foundry 中注册的虚拟表输入。对于虚拟表输出（由 Foundry 管道创建），则无需进行更新检测，因为 Foundry 会利用管道构建信息自动跟踪其更新状态。

启用更新检测后，Foundry 会定期轮询源系统以检查表更新。若源表格式支持版本控制（例如 Delta 或 Iceberg），Foundry 可检测变更并仅在必要时触发下游构建。若不支持版本控制，每次轮询都会被视为潜在更新，这可能导致不必要的下游构建。

您可以通过查看虚拟表详情来确认虚拟表是否支持版本控制。

启用更新检测

要为虚拟表输入启用更新检测，请按以下步骤操作：

  1. 在《数据集预览》中打开虚拟表。
  2. 转到左侧面板的更新检测部分。
  3. 启用更新检测并设置所需的轮询计划（例如每小时或每天）。

启用后，即可将虚拟表输入用作下游表和数据集的《计划触发器》。检测到源数据更新时，任何基于该虚拟表的对象都会自动重新索引。

配置基于虚拟表的对象

你可以在《本体管理器》中配置直接基于虚拟表的对象。

如果底层虚拟表在 Foundry 外部更新，应在虚拟表上启用《更新检测》，以确保对象能从源系统定期获取更新。

使用虚拟表与同步至数据集的对比

选择使用虚拟表还是同步至 Foundry 数据集，取决于你的架构目标和所要支持的目标工作流。我们建议按具体工作流逐个评估合适的集成模式。这两种方式也可以结合使用，互为补充。

以下是关于使用虚拟表与将数据同步至数据集在潜在优势、缺点和局限性方面的一些考量。

使用虚拟表的优势

虚拟表具有诸多优势，包括：

  • 避免在 Foundry 中存储源数据，减少重复存储。需要注意的是，对于任何下游创建的资源（如 Foundry 管道输出的数据集和对象），Foundry 仍会存储其数据。
  • 可将查询下推至源系统，以限制总数据传输量。请注意，计算下推的可用性因源系统和查询类型而异。
  • 对于体量极大的表而言，虚拟表可能尤为有利，因为此时重复存储的成本会变得相当可观。
  • 使用虚拟表时，数据会在使用时直接进行查询，无需同步数据，也无需考虑数据陈旧的风险。
  • 虚拟表提供了更多选择性，有助于使 Foundry 的实施与目标架构模式保持一致。

使用虚拟表的缺点

虚拟表并不一定适用于所有场景。需要考量以下几点：

  • 与操作存储在 Foundry 数据集中的数据相比，交互性能可能会更慢。
  • 计算用量可能会增加，具体取决于在虚拟表上运行的查询类型。例如，作为计划管道输入的表，其产生的计算量可能少于在 Contour 分析中频繁进行交互式访问的表。
  • 虚拟表无法受益于 Foundry 数据集的诸多功能，例如数据集版本控制或分支管理。

使用虚拟表的局限性

虚拟表的局限性包括：

  • 并非所有数据源都支持虚拟表。
  • 虚拟表需要 Foundry worker 源和直接出口策略。
  • 通常不支持使用自助服务私有链接出口策略的连接。但是，由 Palantir 代表设置的私有链接则受支持。
  • 不支持使用智能体代理策略的连接。
  • 不支持使用存储桶端点出口策略的连接。
  • 不支持使用智能体 worker 的连接。
  • 并非所有 Foundry 应用程序和功能都支持将虚拟表作为输入。然而，在虚拟表下游创建的任何物化资源（例如管道输出的数据集和对象），在 Foundry 生态系统中依然受到全面支持。
  • 使用 use_external_systems 装饰器的转换目前与虚拟表不兼容。请改用基于源的 external_transforms，或者将转换拆分为多个转换：一个使用虚拟表作为输入，另一个使用 use_external_systems 装饰器。

虚拟表查询的计算资源

对于直接在虚拟表上运行的查询，计算任务可能会在 Foundry 与源系统之间进行分配。具体的执行逻辑取决于查询本身以及源系统所支持的算力下推程度。详情请参阅特定源系统的文档。
