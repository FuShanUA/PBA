启用用户交互

几乎所有应用都需要用户交互，Slate 提供了三个主要功能来实现交互性：

  • 输入组件：简单的组件，提供常见的 UX 元素来捕获输入，例如下拉列表、文本字段、单选按钮等。
  • 其他组件内的选择状态：大多数其他组件类型都支持某种形式的选择。每种组件的选择机制略有不同，因此需要实际操作尝试，以了解支持哪种选择类型。
  • 事件触发器：所有组件都有一组事件触发器，当组件状态发生变化或用户执行操作时，这些触发器会广播事件。查阅《事件》文档，详细了解如何将事件集成到应用中。

捕获用户输入最简单的模式是静态表单：添加输入组件并提供静态选项，随后即可在查询和函数中引用用户选择，从而呈现动态视图。然而通常情况下，会存在“链式”输入的复杂性，即一个或多个输入中的选择会影响下一组输入的可用选项。务必保持这些链条简短且易于理解——需要设置的参数过多会导致应用难以直观操作且性能不佳。请参阅下文的“开放式探索”反模式。

在依赖输入的这种更为复杂的配置中，最佳实践是将配置过滤器的工作流与分析结果数据的工作流分离。简而言之，这意味着应将所有依赖这些用户输入的查询设置为 manual，并为用户提供一个更新数据按钮组件。这种模式可确保应用不会在每次更改过滤器时重新运行所有查询，从而浪费资源（和用户时间）。如果应用中包含任何形式的自由文本输入——文本字段或文本区域类型组件——这一点尤为关键，否则查询等下游依赖项会在用户的每次按键时重新求值。

数据回写工作流

当应用需要捕获用户输入以回写数据时，你必须将查询配置为手动运行，并由明确的用户操作来触发该查询。否则，持久化数据的查询会在每次输入变化时执行，包括文本输入组件中的每一次敲击键盘，这会导致极其异常的行为。

在会话间保持用户选择

如果你的应用需要配置大量不同的输入项才能获取有用的视图，用户可能希望有一种方法能“保存”他们的配置，以便分享或日后再次查看。

你可以自行构建该功能的自定义版本，但直接使用内置的可共享视图功能会更加简便。

“动态输入”

有时，你需要获取那些看似无法用静态输入组件处理的输入内容。许多情况下，转换一下思路就能找到简单的解决方案。但假设你需要允许用户构建更复杂的筛选器，或者你的应用场景似乎无法通过任何富有创意的静态输入组合来实现。

你可能会想在这种工作流中使用 重复容器（Repeating Container）组件，但这类组件仅限于展示用途。你可以在重复容器内使用输入组件，但其作用域 仅 限于该容器内部，除非通过容器内的其他组件，否则你无法引用该输入组件的任何实例。

相反，你可以只保留输入组件的一个实例，但允许用户多次进行选择，方法是让他们“保存”自己的选择。你可以在 HTML 组件中展示累积的选择结果，甚至可以加入移除先前选择的功能。

要实现这种模式，请使用一个设置为空数组（[]）的变量 v_userSelections。然后，每次用户点击按钮保存选择时，你都可以更新此变量：

可共享视图

可共享视图允许用户捕获 Slate 应用的当前状态，并生成一个可分享给他人的唯一 URL。当其他用户打开该 URL 时，应用加载时会保持视图创建时的选择和配置。

可共享视图会存储以下内容：

  • 所有组件 交互属性 的当前值；这些属性可由用户交互更改，例如 selectedValue。
  • 所有局部 变量 的当前值。

加载可共享视图 URL 时，保存的值将覆盖组件和局部变量的默认值，下游依赖也会随之初始化。

创建可共享视图

若要手动创建可共享视图，请在查看模式下打开应用，并从操作下拉菜单中选择获取可共享视图。系统会生成一个带有视图 ID 的唯一 URL 供你分享。

Slate 应用构建者可以通过 slate.saveView 和 slate.loadView 操作，以编程方式创建和加载可共享视图。视图保存后会触发 slate.viewSaved 事件，并提供可传递给 slate.loadView 的视图 ID。

限制

可共享视图存在以下限制：

  • 仅限主页：可共享视图目前仅保存和恢复 Slate 应用的主页状态。仅包含本地变量；共享变量不予保存或恢复。
  • 组件与变量名称：可共享视图与组件和变量名称绑定。若在创建视图后重命名组件或变量，加载该视图时将无法恢复对应组件或变量的已保存状态。
  • 组件类型变更：若组件的类型在保存视图与打开视图期间发生变更，该组件的已保存状态将不予应用。
  • 版本无关：可共享视图适用于应用的任意版本。在某一应用版本中创建的视图，只要引用的组件和变量名称依然存在，即可在任何其他版本上加载。
