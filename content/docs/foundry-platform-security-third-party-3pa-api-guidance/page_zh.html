Foundry 第三方应用程序 & API 指南

Foundry 的第三方应用程序身份验证和授权功能使非 Foundry 应用程序和脚本能够安全地与 Foundry 的 API 进行交互。这些功能的核心是支持外部应用程序的 OAuth2。

本文档提供了 Palantir 建议使用这些功能的指导，以及可能不恰当使用的示例。

通过授权第三方应用程序和 API，用户同意遵循 Palantir 与客户之间书面协议中商定的适当使用条款。如果您对您的预期用途有任何疑问，或者不确定您的计划是否适当、安全或符合规范，请联系您的 Palantir 代表。

适当用途

  • 替换服务用户账户
    • OAuth2 授权码流程允许外部应用程序代表 Foundry 用户执行操作。这确保了权限的正确性，并且有明确的审计路径。使用 OAuth2 流程也确保了用户已明确授予应用程序代表其执行操作的权限，再次提供清晰的审计路径。
    • 我们强烈建议现有使用服务账户在 Foundry 中执行操作的应用程序尽可能迁移到 OAuth2 授权流程。
  • 与外部系统的交互
    • 示例：一个监视内部客户系统更改并在 Foundry 本体论中执行操作的应用程序。
  • 针对特定用户工作流的自定义应用
    • 示例：一个与 Foundry 的本体论和操作 API 交互的移动应用，为关键工作流提供简化的用户体验。
  • 监控或控制 Foundry 管道或工作流
    • 示例：一个连接到 Foundry 监控和数据健康 API 的应用程序，用于评估关键管道的状态，并允许用户在需要时触发构建。

不适当的使用

第三方应用程序的集成和使用Foundry API可能会对数据安全构成风险，只有在充分了解技术及合同考虑因素的情况下才应进行。当规划一个访问数据或代表用户执行操作的开发项目时，请联系您的系统管理员，以确定您的计划是否适当、安全且符合Foundry的适当使用条款。

以下示例列出了API不当使用可能损害Foundry中管理的数据完整性或安全性的代表性场景。

  • 规避数据控制
    • 示例：使用一个用户的令牌读取数据，并使用另一个用户的令牌写入数据。
    • Foundry具有先进的细粒度用户授权功能。在您的应用程序中在用户账户之间共享数据可能会绕过这些控制。
    • It is important you keep the use of individual Foundry accounts fully isolated. You must not access data with one user's tokens and allow another user to read, discover, write, or in any way interact with that data. If you wish to allow users to share data, this should be done in Foundry, not in the third-party application.
  • 在没有用户理解和同意的情况下执行操作。
    • 您的应用程序不得欺骗您的用户。当使用 Foundry 用户的账户时，您的应用程序必须清楚且准确地描述它正在执行的操作。
    • 您的应用程序必须请求其功能所需的最小角色和权限集，且不能更多。同时，还必须明确告知用户在 Foundry 中将执行哪些操作以及这些操作将做什么。
    • 任何不明确、意外、恶意或有害的应用程序行为都是被禁止的。
  • 在未考虑访问控制的情况下从 Foundry 外部检索或存储数据。
    • Foundry 的客户依赖 Foundry 安全存储关键数据；您的应用程序必须尊重该数据的敏感性和价值，并仅检索其功能所需的最小数据集。
    • 您的应用程序也应尽量避免在Foundry中存储或缓存检索到的数据。这可能有一个例外情况是离线缓存，但应谨慎处理并获得明确的用户同意。
    • 如果将Foundry中的数据移动到其他数据存储系统中，请承认这将不再按照您组织在Foundry内的数据保护配置进行访问控制和审计。
