容器转换

前提条件

以下文档假定您具备容器化基础设施和容器镜像等概念的知识。如果您不熟悉这些主题，我们建议您查阅 Docker 概述文档 ↗。

Foundry 以两种方式与推送到平台的容器进行交互：

  1. 使用 transforms sidecar 装饰器。
  2. 构建并使用 基于容器的模型资产。

在这两种用例中，第一步是按照下面列出的图像要求将图像推送到 Foundry 内置的 Docker 注册表中。

镜像要求

  1. 该镜像具有数字 userID。
  • The userID defined within the Dockerfile must be numeric and not '0'. Foundry does not allow commands to run within the container as the root user; '0' gets interpreted in some systems as zero, and non-numeric IDs can be set to function as root.
  • For bring your own container transforms, the userID must be specifically '5001'.
  1. The image is built for linux/amd64 platform.
  • Foundry supports the execution of containers built for this platform only. The default platform for Docker is linux, so add --platform linux/amd64 for the Docker build command ↗ to set the platform specifically.
  1. The image is pushed with digest or any tag other than latest.
  • The executed Docker push command should specify the digest of the image or use a tag that is not latest. Foundry will not execute images tagged as latest since there is no mechanism to ensure any given image is actually the latest one.
  1. The maximum image layer size is less than ~10 GB.
  • It is strongly recommended that each layer is smaller than ~10 GB. If your use case requires a larger layer size, contact your Palantir representative.
  1. 任何暴露的端口都在 1024 到 65535 之间。
  • 0 到 1023 的端口是众所周知的端口，因此被保留给 root 用户。Foundry 不允许在容器中以 root 用户身份运行命令；因此，当镜像在 Foundry 中启动时，此范围内的任何端口将不可用。
  1. [可选] 该镜像已启用遥测功能。
  • 要从您的容器启用遥测日志：
    • 镜像必须在 /bin/sh 中包含一个 shell 可执行文件。
    • 镜像必须支持 shell 命令 set 和 tee。
