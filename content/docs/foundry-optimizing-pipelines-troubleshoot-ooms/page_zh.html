排查内存溢出 (OOM) 错误

作业中出现内存溢出错误通常表现为以下几种形式：

  • 出现 “作业因阶段失败而中止”
  • 出现 “ExecutorLostFailure”
  • 出现 “作业 [jobID] 正在使用时 Spark 模块死亡。 (ExitReason: MODULE_UNREACHABLE)”
  • 出现 与驱动程序的连接丢失

这些错误信息表明已超出当前构建所允许的最大内存限制。这通常不是平台故障，而是要求平台运行的构建本身存在问题。可以通过以下几个步骤来减少构建运行所需的内存。

如需排查，请执行以下步骤：

  • 如果转换逻辑使用 Python 或 Pandas 编写：
    • 尽量将计算转移到 PySpark 中，以充分利用整个计算集群的算力。原生 Python 和 Pandas 中的逻辑会在驱动程序中依靠单处理器执行，速度可能比你的本地笔记本电脑还要慢。
  • 如果转换逻辑中使用了连接操作：
    • 排查“空值连接”——即针对大量行值为空的列进行连接。这会显著增加连接操作的内存消耗。要解决此问题，可以在当前转换或前置转换中过滤掉相关列的空值。
    • 排查导致输出数据集行数剧增的连接操作，并确认这种扩展是否必要。一种实用技巧是运行分析，计算数据集中每个键对应的行数以及连接后的结果行数。
    • 排查可设置检查点的递归连接。对单个表的重复连接会导致查询计划快速膨胀。应使用检查点来缓存中间结果，以避免此问题。
  • 检查输入数据集中的文件大小 （数据集 → 详情 → 文件 → 数据集文件）。每个文件应至少为 128MB。如果文件过小或过大，都需要进行重分区。
  • 将大型转换拆分为多个较小的转换。这也有助于定位引发故障的具体转换环节。
  • 从输入数据集中移除不必要的列，或对数据集进行预过滤以剔除不需要的行，从而减少 Spark 需要在内存中保存的数据量。
  • 在条件允许的情况下，简化转换逻辑。
  • 若代码优化仍无法确保作业成功构建，可以分配额外资源。请注意，增加资源会导致该构建相关的计算成本上升。有关分配额外资源的更多信息，请参阅：
    • 《何时修改默认 Spark 配置》：根据具体错误提供应增加哪类 Spark 配置的指导。
    • 《Spark 配置参考》：列出 Foundry 中可用的 Spark 配置。
    • 《应用 Spark 配置》：指导如何为作业应用自定义 Spark 属性。
