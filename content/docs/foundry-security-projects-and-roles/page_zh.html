项目与角色

项目是 Foundry 中组织工作的主要方式，也是主要的安全边界。您可能需要将管道设置为一系列项目。为了与他人协作，您需要为每个项目授予各个用户和群组相应的角色。项目与角色共同构成了 Foundry 中自主访问控制的管理基础。

了解有关在 Foundry 中保护数据基础的更多信息。

项目与资源

项目是 Foundry 中的主要安全边界，可以看作是共享工作的容器。由于其边界执行安全策略，项目是组织数据并在安全空间内实现开放协作的关键手段。

每个项目都是一个协作空间，为特定目的组织用户、文件和文件夹。项目的设计应确保在其中协作的用户对内容具有基本一致的访问权限，但对这些内容的具体权限可以有所不同——例如，项目中的所有人都可能拥有默认的查看者角色，但只有一个特定的群组对所有项目资源拥有编辑者角色。项目强制执行安全边界，从而使得工作能够包含在特定空间内。工作（转换、分析等）在项目中进行，其产出与逻辑一同存放在该项目内。

由于工作及其产出必须位于同一个项目中，因此要重用或基于其他项目中的工作进行构建，您必须使用引用。

为了简化权限管理，我们建议在项目级别授予群组角色。通过群组管理项目权限，可以将具备相同权限的一组用户集中管理，减少单个角色授权带来的混乱。项目内容将继承在项目级别授予的角色，从而为用户提供对项目中内容的一致访问权限。

创建项目

用户需要对空间拥有 Editor 或 Owner 权限，才能在该空间中创建项目。

可以在空间设置页面管理空间权限。

申请项目访问权限

用户可以为其无权访问的项目提交访问申请。访问申请将包含授予用户项目访问权限所需的所有变更，包括任何必需的标记。

可以从 Foundry 文件系统视图的多个访问入口提交项目访问申请：

  • 申请访问权限
    • 位于项目与文件视图中项目名称的旁边。
    • 在用户打开无权查看的资源（例如通过直接链接）时出现。
  • 申请项目访问权限
    • 如果用户仅对项目拥有发现者角色，则该选项位于项目视图中。
  • 申请额外访问权限
    • 如果用户已拥有项目访问权限，则该选项位于项目视图的操作下拉菜单中。

选择上述任一入口后，用户将看到申请访问权限弹窗。他们必须提供申请访问的理由、需要授予权限的用户和群组（如果是代表他人申请），以及授予权限的方式。

如上文所述，我们建议通过群组管理项目权限。在申请访问权限弹窗中，用户可以选择加入一个在项目中拥有适当角色的群组。对于在 Foundry 内部管理的群组，用户可以选择要加入的群组，该申请将被路由至群组管理员进行审批。可以为群组配置自定义申请流程，以帮助申请用户选择合适的群组加入。有关自定义审批访问申请配置的更多信息，请参阅平台安全管理文档。此外，您可以隐藏群组以防止用户申请其访问权限。具体操作请参阅管理群组文档。

对于在 Foundry 外部管理的群组，系统会向用户展示一条提示信息和 URL，引导他们前往 Foundry 平台之外申请所需群组的访问权限。了解如何配置此提示信息和 URL，请参阅外部群组文档。 如果项目未分配任何群组，用户可以直接申请以特定角色加入该项目。这将创建一个项目访问申请任务，并需要拥有该项目数据主人角色的用户予以审批。 用户提交申请后，系统会弹出一条提示信息，表明申请已成功提交。在此提示信息上选择查看详情即可查看已创建的申请；或者前往 Foundry 工作区侧边栏的审批收件箱，并在左侧的筛选器中选择我的申请进行查看。

文件和文件夹访问申请

当用户对项目内的文件或文件夹选择申请访问权限时，访问申请将直接针对项目本身提交，而非针对特定的具体资源。在审核申请时，系统会显示提交申请时所在的文件或文件夹，以提供额外的上下文信息。

共享和移动资源

为了将项目作为安全边界来落实，我们建议将资源移动到已根据您的使用场景配置好权限的项目中，而不是直接从您的文件进行共享。这样可以明确访问权限，并清晰呈现谁拥有对哪些资源的访问权。点击访问面板即可管理对该项目具有访问权限的用户和群组：

引用

项目是 Foundry 中的核心安全边界，这一机制也延伸至 Foundry 的构建系统。一次构建可以接收任意数量的输入数据集，并生成任意数量的输出数据集。这些输入和输出_必须_位于同一个项目中。然而，为了构建实用的数据流水线，您可能需要使用来自其他项目的数据集。在使用其他项目的数据集时，我们建议采用文件引用方式。

只要您在该用例下具备所需的角色，添加文件引用便允许您在项目中使用上游数据集。导入后，只要您的同事满足数据集上的组织和标记要求，他们便无需具备上游项目的访问权限，即可查看您项目中派生的数据集。

在上图中，我们在 `Flight Delays [Transform]` 项目中引用了来自 `Flight Control System [Datasource]` 项目的航班数据集。如果我们向该转换项目中添加用户，他们将能够查看 `delays` 数据集，并在此基础上构建额外的转换。然而，若要查看原始的 `flights` 数据集或在其基础上构建任何额外的转换，他们仍需在上游项目中具备 `Viewer` 角色。 项目和引用有助于组织协作，项目数据主人可以轻松地将用户添加到自己的项目中，并确保后者拥有正确的权限。 要向项目添加文件引用，通常您必须在源项目中具备 `Viewer` 角色，并在目标项目中具备 `Editor` 角色。有关向项目添加引用所需权限的更多信息，请参阅《项目引用与权限》。 引用可以从 Foundry 的多个位置添加，例如代码存储库、代码工作簿、流水线构建器、Fusion 和 Contour。 您无法对位于您的文件中的文件添加引用，因为这可能导致权限问题。若要引用位于您的文件中的文件，请首先将该文件移动到项目中，以便您的同事能够看到它。

通过标记实现集中数据治理

在某些场景下，我们可能需要更集中地控制对某一类数据的访问。比如，任何有权访问某种航班数据的人，可能都需要先完成一项强制培训。在原始的 flights 数据集上应用标记（Marking），将要求用户必须通过中央机构才能访问 flights 及从中衍生的任何数据，例如 delays 数据集。有关使用标记的更多信息，请参阅标记。

角色

角色是一组权限集合，用于授予对不同级别资源的访问权限。角色属于自主访问权限，通常在项目级别授予，以便对项目范围内的所有资源提供统一的能力。然而，强制控制机制——组织和标记——将始终阻止无权限用户访问资源，无论该用户拥有什么角色。

在 Foundry 中，默认角色按权限从大到小依次为：数据主人、编辑者、查看者和发现者。每个角色都可以将相同或更低级别的角色分配给其他用户。例如，数据主人可以向其他用户授予数据主人、编辑者、查看者或发现者角色，而发现者只能向其他用户授予发现者角色。这些默认角色可以自定义，以包含全新的角色。

重要的是，与强制控制机制一样，角色授权也会继承到子级资源。例如，向用户授予某个项目或文件夹的查看者角色，会使其获得该项目或文件夹下所有资源的查看者权限。通常，我们是以用户组的形式在项目级别被授予角色。

详细了解有关配置组织的角色的信息。

文件夹和文件的角色授权

如上所述，我们建议仅在项目级别授予角色，以便对项目范围内的所有资源提供统一的能力。为了强制执行此行为，你可以在项目视图的设置部分使用开关来禁用文件夹和文件的角色授权。当此设置被禁用时，角色授权只能在项目级别进行，不能在文件夹或文件级别进行。该开关可由项目中拥有数据主人角色的用户进行设置。

如果对已经包含角色授权资源的项目禁用了角色授权设置，针对这些独立资源的角色授权将被移除。一旦移除了现有的角色授权，在重新启用该设置之前，将无法再次添加。类似地，如果将带有角色授权的资源移动到禁用了角色授权设置的项目中，资源级别的角色授权也将被移除。在禁用角色授权设置以及将资源移动到禁用角色授权设置的项目时，系统会向用户发出有关此行为的警告。

此外，项目的链接共享功能也将被移除，因为链接共享会赋予链接接收者对特定文件夹或文件的直接角色授权。

文件夹和文件的角色授权默认处于禁用状态。空间管理员可以在空间级别更改此默认行为。我们建议保持禁用文件夹和文件的角色授权。
