项目与角色

项目（Project） 是在 Foundry 中组织工作的主要方式，也是主要的安全边界。您可能希望将您的管道设置为一系列项目。为了与其他用户合作，您将在每个项目上授予各种用户和组角色。项目和角色一起是 Foundry 中实现自主访问控制的方式。

了解更多关于 在 Foundry 中保护数据基础架构 的信息。

项目与资源

项目（Project） 是 Foundry 中的主要安全边界，可以看作是共享工作的容器。因为它们的边界强制执行安全措施，所以项目是组织数据并在安全空间内实现开放协作的关键方式。

每个项目是一个协作空间，用于组织用户、文件和文件夹以实现特定目的。项目应设计为使在该项目内协作的用户对内容具有大致相同的访问权限，但对内容的权限各不相同——例如，您项目的每个人可能都有默认的查看者角色，但只有特定的一组用户可能对所有项目资源具有编辑者角色。项目强制执行安全边界，这使得工作能够被包含在一个空间中。工作（转换、分析等）是在项目中完成的，该工作的输出与它的逻辑一起存在于项目中。

由于工作及其输出必须存在于同一个项目中，要重用或基于其他项目的工作，您必须使用 引用。

为了便于权限管理，我们建议授予 组 在项目级别设置角色。通过组来管理项目权限，可以将具有相同权限的一组用户统一管理，从而减少单独分配角色的杂乱情况。项目内容将继承在项目级别授予的 角色，为用户提供对项目内内容的统一访问权限。

创建项目

用户需要在某个空间中拥有 Editor 或 Owner 权限才能在该空间中创建项目。

空间权限可以从 空间设置 页面进行管理。

请求访问项目

用户可以提交他们未被授权访问的项目的访问请求。该访问请求将包括使用户能够访问该项目所需的所有更改，包括任何所需的 标记（Marking）。

项目访问请求可以从 Foundry 文件系统视图中的多个访问点提交：

  • 请求访问
    • 位于 项目与文件（Projects & files） 视图中项目名称旁边。
    • 当打开用户没有查看权限的资源时出现（例如，直接链接）。
  • 请求项目访问
    • 如果用户仅在项目中拥有 Discoverer 角色，则位于项目视图中。
  • 请求额外访问
    • 如果用户已拥有对项目的访问权限，则位于项目视图中的 操作（Actions） 下拉菜单中。

选择上述任一入口后，用户将看到一个 请求访问 弹出窗口。他们必须提供访问理由，应授予访问权限的用户和组（如果代表其他人请求），以及如何授予访问权限。

如上文 所述，我们建议通过组来管理项目的权限。在 请求访问 弹出窗口中，用户可以选择通过项目上的适当角色获得对组的访问权限。对于 在 Foundry 内部管理的组，用户可以选择加入一个组，这将把请求转交给组管理员进行审批。可以配置自定义的请求流程，以帮助请求用户选择合适的组进行加入。您可以在 平台安全管理文档 中找到有关自定义审批访问请求配置的更多信息。此外，您可以隐藏组，以防止用户请求访问这些组。请参阅 管理组文档 了解具体操作方法。

对于在Foundry外部管理的组，用户将看到一条信息并被重定向到一个URL，以请求在Foundry平台之外访问该组的权限。请参阅外部组文档，了解如何配置此信息和URL。

如果没有分配给项目任何组，则用户可以请求直接以指定角色添加到该项目。这将创建一个项目访问请求任务，并需要具有项目所有者（Owner）角色的用户进行批准。

一旦用户创建了请求，应显示一条提示信息表明请求成功。通过选择消息中的查看详细信息，或导航至Foundry工作区侧边栏中的审批收件箱，然后从左侧筛选器中选择我的请求来查看已创建的请求。

文件和文件夹访问请求

当用户在项目内的文件或文件夹上选择请求访问时，访问请求将提交到项目本身（而不是特定资源）。在审核请求时，会显示提交请求的文件或文件夹，以提供额外的上下文。

项目是Foundry中的中心安全边界，它扩展到Foundry的构建系统。构建过程可以接受任意数量的输入数据集，并生成任意数量的输出数据集。这些输入和输出必须位于同一项目中。然而，要创建一个有用的管道，您可能需要使用其他项目的数据集。当使用其他项目的数据集时，我们建议应用文件引用。

添加文件引用允许您在您的项目中使用上游数据集，只要您拥有该用例所需的角色。一旦导入，您的同事就不需要访问上游项目以查看在您的项目中派生的数据集，只要他们满足数据集上的任何组织和标记。

在上面的图片中，我们从 Flight Control System [Datasource] 项目引用了航班数据集，用于我们的 Flight Delays [Transform] 项目。如果我们向转换项目添加一个用户，他们将能够查看 delays 数据集，并在其基础上构建额外的转换。然而，要查看原始 flights 数据集或在其基础上构建任何其他转换，他们仍然需要上游项目的 Viewer 角色。

项目和引用有助于组织协作，因为项目所有者可以轻松地将用户添加到自己的项目中，并确保后者拥有正确的权限。

要为一个项目添加文件引用，您通常必须在源项目上拥有 Viewer 角色，并在目标项目上拥有 Editor 角色。有关添加项目引用所需权限的更多信息，请参阅 项目引用和权限。

可以从 Foundry 的许多地方添加引用，例如 代码仓库、代码笔记本、管道构建器、融合 和 轮廓。

您不能对位于 您的文件 中的文件添加引用 因为这可能会导致权限问题。如果您想引用位于您的文件中的文件，首先请将该文件< a href="/docs/foundry/compass/move-and-share-resources/">移动到一个项目中，以便您的同事可以查看它。

通过标记实现的中央数据治理

在某些情况下，我们可能希望更集中地控制对某一类数据的访问。也许任何拥有某种飞行数据访问权限的人都需要先完成强制培训。将标记应用于原始flights数据集将要求用户通过一个中心机构才能获得对flights以及由此生成的任何内容（例如delays数据集）的访问权限。有关使用标记的更多信息，请参见 标记。

角色

角色是授予不同资源访问级别的一组权限。角色是一种可选的权限，通常在项目级别授予，以在项目范围内的所有资源上提供统一的功能。然而，强制性控制、组织和标记将始终阻止不符合条件的用户访问资源，无论用户的角色如何。

从最强大到最弱，Foundry中的默认角色依次为：所有者、编辑者、查看者和发现者。每个角色可以将相同或更低的角色分配给其他用户。例如，所有者可以将所有者、编辑者、查看者或发现者的角色授予任何其他用户，而发现者只能将发现者的角色授予其他用户。这些默认值可以自定义，以包含完全< a href="/docs/foundry/platform-security-management/manage-roles/#customizing-the-default-roles">新的角色。

重要的是，与强制性控制一样，角色授予会继承到子资源。例如，向用户授予项目或文件夹的查看者权限，会使他们在该项目或文件夹包含的所有资源上拥有查看者权限。通常，一组用户会在项目上被授予角色。

了解更多关于< a href="/docs/foundry/platform-security-management/manage-roles/"> 配置您的组织的角色。

文件夹和文件上的角色授予

如上所述，我们建议仅在项目（Project）级别授予角色，以在项目范围内的所有资源上提供统一的功能。为了强制这种行为，您可以在项目视图的设置（Settings）部分中使用切换选项来禁用文件夹和文件的角色授予。当此设置被禁用时，角色授予只能在项目级别进行，不能在文件夹或文件级别进行。此切换选项可以由具有Owner角色的用户在项目上进行设置。

如果项目中已包含具有角色授权的资源，那么角色授权设置将被禁用，针对这些单个资源的角色授权将被移除。一旦现有的角色授权被移除，在该设置重新启用之前，无法重新添加。同样，如果具有角色授权的资源被移动到角色授权设置被禁用的项目中，则资源级别的角色授权将被移除。当禁用角色授权设置以及将资源移动到具有禁用角色授权设置的项目时，用户将收到此行为的警告。

此外，项目链接共享功能也将被移除，因为链接共享会使链接的接收者直接获得对单个文件夹或文件的角色授权。

文件夹和文件上的角色授权默认是禁用的。空间管理员可以在空间级别更改默认行为。我们建议保持文件夹和文件上的角色授权处于禁用状态。
