排查调度问题

调度器指标页面

排查调度问题的最佳起点之一是查看调度器指标页面。该指标页面能够帮助定位失败的根源，常见的故障模式包括：

  • 计划构建失败。在 运行历史 选项卡中会看到构建失败的记录，点击这些构建记录将跳转至构建应用程序中的构建报告以查看完整日志。
  • 计划构建被忽略。对于已触发但未实际执行的构建，运行历史 选项卡的状态列会显示 已忽略。
  • 计划未在预期时间或按预期频率触发。这种情况下，运行历史 选项卡可能不会显示原本预期触发的构建记录。

版本 选项卡展示了过往的调度版本及编辑记录。如果调度表现突然异常，该选项卡会非常有用。请检查是否有与此次异常表现相关的版本变更，并考虑将调度回滚至先前正常工作的状态。

计划构建失败

通过查看调度器指标页面上的 运行历史 选项卡，可以确认调度是否在预期的时间被触发。

如果调度已触发，但随后的构建过程失败，可以按照调试指南对该构建进行排查。

如果未设置适当的权限，调度构建也会失败。调度的权限取决于其所使用的令牌模式。有关更多信息，请参阅项目级作用域调度。

计划构建被忽略

通过查看调度器指标页面上的 运行历史 选项卡，可以确认调度是否在预期的时间被触发。该页面通常也会给出调度被忽略的原因。

所有数据集均为最新

如果所有目标数据集均为最新状态，即自上次构建这些数据集以来其输入数据未发生更新，那么此次调度运行将被忽略。如果是这种情况，在 运行历史 选项卡中会看到该原因。在调度编辑器中，导航至调度列表。随后可以选择按 已过期 为数据血缘关系图着色，从而总览哪些作业规格被视为过期状态。

在特殊情况下，可以通过高级设置中的 强制构建 选项覆盖此行为，但在非必要场景下使用此选项会造成计算资源浪费。如果任何目标数据集是由 Phonograph 同步、带有 API 调用的转换或数据连接同步所构建的，它们可能不会显示为过期状态，这就需要启用 强制构建 选项才能使调度正常运行。

调度仅构建部分数据集

如果调度仅触发了部分数据集的构建，在调度器指标页面上的 运行历史 选项卡中会看到相关记录。

导致该情况的一个原因是，仅有部分数据集处于过期状态。调度器将仅构建过期的数据集，而处于最新状态的数据集在构建过程中会被忽略。有关更多排查细节，请参阅所有数据集均为最新。如果所有这些数据集都处于最新状态，就会出现构建被 已忽略 的情况。

另一个原因可能是该数据集未被包含在构建的数据集图中。在调度编辑器中，当选中某个特定调度时，待构建的数据集会在数据血缘关系图中高亮显示。数据集的选择取决于构建类型。如果使用的是 连接构建，在多个分支上使用相同数据集的调度中，务必仔细核实是否存在连接数据集。

计划未触发

您可以通过查看调度器指标页面上的 运行历史 标签页，来验证计划是否在预期时间被触发。以下是一些常见的排查步骤：

  • 检查计划是否已暂停。已暂停的计划在恢复之前不会触发。
  • 检查计划触发器配置。如果之前运行成功，请检查计划历史记录，查看最近是否对触发器进行了更改。
  • 如果计划使用事件触发器，请确认预期事件是否确实发生。例如，如果构建应在输入更新时触发，请检查输入上的最后一次构建是否成功运行，以及该构建中的事务是否已在数据集预览历史记录视图中成功提交。

计划重试次数与配置不符

请注意，并非所有类型的失败都可以重试。计划运行时的重试次数将限制为管理员配置的最大值。有关更多信息，请参阅高级设置。

计划失败并提示 JobSpecInputsTrashed 或 JobSpecOutputsTrashed，或者数据血缘关系警告部分数据集已被废弃

这意味着计划包含或读取了已废弃的资源。您可以采取以下任一方法来解决：

  • 从回收站恢复已删除的数据集。
  • 从计划中排除已删除的数据集。如果此数据集用作计划中另一个下游数据集的输入，您还需要执行以下操作之一：
    • 将下游数据集与已废弃的数据集一同排除。
    • 修改下游数据集的逻辑，使其不再将已废弃的数据集作为输入。

调度器权限

如果您遇到无法编辑计划的问题，项目范围权限可能是根本原因。

要在项目范围模式下编辑计划，用户必须对目标数据集具有 编辑 权限，对触发器数据集具有 查看 权限，并且对计划所属的项目具有 编辑 权限。如果有某个数据集您已失去权限，请在保存更改之前从计划中移除该数据集。

要编辑、删除或暂停计划，用户需要对目标数据集具有 编辑 权限，并对计划所属的项目具有 编辑 权限。要查看计划，用户需要对目标数据集具有 查看 权限。
