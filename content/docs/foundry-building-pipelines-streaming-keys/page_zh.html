流式键

Foundry 流允许您将一个或多个列指定为键列，并使用主键来标识已解析的记录。以下章节将说明如何为您的流式管道设置和维护分区与主键。

分区键

通常情况下，Palantir 平台中的主键用于唯一标识数据库记录。然而，Foundry 流中的分区键会将具有相同键的记录进行分组，例如特定设备的所有读数，或特定客户的所有交易记录。与批处理管道中的主键不同，流中的分区键不会对记录去重，因为它们并不能唯一标识记录。

Foundry 中的键允许用户确保特定键的所有记录保持顺序。记录进入 Foundry 后，会在内部存储于 Kafka 中。Kafka 保证写入同一分区的记录在读写时保持顺序。如果发送到 Foundry 的记录没有键，该记录将以轮询方式写入内部 Kafka 主题的任意分区。但如果用户为记录指定了键，该记录将被写入该键的专用分区，从而在下游消费时保持顺序。

同理，Flink 流处理作业会根据输入源上设置的分区键列自动维护顺序。Flink 转换作业在运行时，每个转换操作可能包含一个或多个并行分区。对于按分区分键的输入流，Flink 作业会自动将所有键列值相同的记录发送到同一个并行算子实例。除非专门重新分键（例如使用 Key by 转换），否则整个管道全长的分区和顺序均由源端的键列和行值决定，即使转换逻辑删除了列或覆盖了值也是如此。

主键：变更数据捕获（CDC）模式

Foundry 流中的主键与关系型数据库和批处理数据集中的主键工作原理类似。不过，流式主键由一组去重列组成，用于唯一标识已解析的记录，且并不指定排序列。

主键是 Foundry 中流模式里的一项元数据。该键不会影响所存数据的内容，也不会影响流式数据管道和转换的应用方式。主键控制的是某些 Foundry 消费者读取数据的方式；完整的数据可视为一个变更日志，而主键则告知消费者如何在应用所有变更后，计算出数据的去重当前视图。

当前视图是数据的过滤视图，仅包含每个键最近流式传输的记录。某个键完整且最近的记录会被始终保留，即使该记录包含 nulls。如果指定了已删除列，且某个键最近流式传输的记录将该值设置为 true，则该记录将被过滤掉。

以下示例展示了一个主键和数据流，其中最近流式传输的行位于表格的上方。

主键：{Dedeuplication column(s): [Key], isDeletedColumn: Optional[isDeleted]}

| Key | Value | isDeleted |
|------|-----------|-----------|
| Key2 | null | false | | Key1 | thirdVal | true | | Key2 | secondVal | false | | Key1 | firstVal | false |

这串变更日志流在支持 CDC 的消费者读取时，将被读取为以下当前视图：

| Key | Value | isDeleted |
|------|-----------|-----------|
| Key2 | null | false |

如果您选择为流式数据设置主键，就必须将相同的列设置为分区键，以维持顺序并正确解析去重后的数据视图。在流式设置界面中设置主键后，相同的列将被自动添加为分区键列。

以下两个支持 CDC 的作业会自动将带主键的流式数据读取为当前视图：

  1. 使用带主键的流为本体注入数据：新的更新记录将按需触发新对象的创建、更新或删除。
  2. 任何 Spark 转换作业都可以读取流的归档数据集： 在进行任何转换之前，会对源数据进行去重处理。值得注意的是，几乎所有与归档数据集的交互都会运行 Spark 作业，包括数据集预览和轮廓分析。查看归档数据集时，显示的始终是去重后的当前视图，尽管数据本身包含了完整的变更日志（如果你将数据作为文件下载的话）。任何不运行 Spark 的流式作业都会处理完整的变更日志，包括重放操作。

键的传递

Pipeline Builder 会跟踪分区键和主键列在管道转换过程中的演变，并将所有有效的键写入输出流的 schema 中。如果你的转换操作没有使任何键列失效，那么在任意数量的下游管道序列中，将自动维持相同的分区和去重指令。

如果你重命名了某个键列，该键会更新为新的列名。同理，如果你应用的转换操作移除或覆盖了某个键列，该列将从键中移除。由于覆盖键列内容可能意味着新的排序保证或去重策略，Pipeline Builder 会直接将该列从键中完全移除。如果你想保留先前被覆盖的键列，必须再次应用 Key by 转换。即使分区键列因移除或覆盖而丢失，除非你重新设置键，否则管道的剩余部分仍将维持原有的排序保证。

如果所有分区键或去重 CDC 键被移除或覆盖，该键将被完全丢弃。删除或覆盖键列内容时务必谨慎；这样做可能会导致意外结果，包括丧失排序保证或去重策略。

目前，键绝不会针对用户定义函数（UDF）进行传递。由于函数由用户定义，系统无法推断用户意图采用何种键传递策略（如果有的话）。如果你打算传递键，请务必在 UDF 之后应用 Key by 转换。

此外，主键不会针对有状态转换进行传递（大多数转换是无状态的）。

使用流式键

在 Pipeline Builder 应用程序的流式管道中，将 Key by 转换添加到你的图谱中。值得注意的是，你在此处设置的任何键都会覆盖并替换输入数据上原有的键。

如果你只想设置分区键，请关闭 CDC mode 选项，仅提供 Key by columns 列表。除非你处于 CDC 模式下，否则其他参数不是必需的，也不被允许使用。

要同时设置主键和分区键，请开启 CDC mode 选项。如果你有 isDeleted 列，可以选择在 Primary key is deleted 字段中指定它。对于流式处理场景，我们强烈建议将可选的 Primary key ordering columns 参数留空。该转换会同时设置分区键和主键，其中分区键列与主键去重列相同。Primary key ordering columns 参数仅在批处理作业中使用归档数据集时才起作用，绝不会影响流式处理的去重逻辑。指定排序列的选项是为了向后兼容、供批处理转换用户使用，或供打算以特定纯批处理方式使用流归档的用户使用。

检查当前键

以下部分介绍如何在你的 Foundry 流中查找并验证流式键逻辑。

检查分区键

打开你的流式数据集，导航至 Details 标签页，然后打开 Schema 标签页以查看 JSON 格式的数据 schema。

搜索 includeInParitioning，它会出现在作为主键组成部分的每一列的 fieldSchemaList 元素中：

"customMetadata": {
"includeInPartitioning": true
}

如果未看到带有 `includeInParitioning` 的模式字段，说明流未设置键，系统无法保证数据存储和处理时的顺序。要手动添加键，请将模式以 JSON 文本格式编辑，并在需要设为分区键列的各个模式字段（列）中插入自定义元数据（如前所述）。

如果流已经拥有一个或多个分区键，新增分区键列会导致顺序保证减弱，因为分区数量会增多；系统仅保证在所有分区键列中具有相同值的行能够维持原有顺序。

部署前，若在管道输入流上设置了分区键列，除非有意重新设置键，否则系统将对该数据源及其在整个管道中的所有转换操作保证顺序。在数据预览中，分区键列可能会带有键符号。

检查主键

打开流式数据集，进入详情选项卡，随后打开模式选项卡以 JSON 格式查看数据模式。

此时会看到一个名为 `primaryKey` 的 JSON 属性。假设流包含名为 `uniqueCol1` 和 `uniqueCol2` 的去重列，且 `isDeleted` 列对应 `isDeletedCol`，模式应如下所示：

  "primaryKey": {
    "columns": [
      "uniqueCol1",
      "uniqueCol2"
    ],
    "resolution": {
      "type": "duplicate",
      "duplicate": {
        "resolutionStrategy": {
          "type": "customStrategy",
          "customStrategy": {}
        },
        "deletionColumn": "isDeletedCol"
      }
    }
  }

若未设置主键，模式将显示为 `null`：

"primaryKey": null

要手动设置或移除主键，可以编辑模式 JSON 以上述格式指定键，或使用 `null` 移除键。如果手动设置主键，强烈建议将相同列设为分区键列。

保证顺序的唯一方法是在整个流式血缘关系中设置分区键列。设置完成后，分区键列将自动向下游传递。即使流配置为仅包含单一分区，受 Flink 应用程序扩展及非确定性处理记录方式的影响，顺序也未必能得到保证。

流处理键的最佳实践

谨慎选择分区键，因为导致记录分布低效的键会人为增加系统负载并限制吞吐量。若顺序对业务场景至关重要，应将分区键设为需要维持顺序的通用分组标识符，例如电子邮件 ID、客户 ID 或组织 ID。若顺序无关紧要，可以选择使用唯一 ID 作为键，或不设键。

流式输出的最终顺序保证强度，取决于在通往输出的流式链路（以 Kafka 主题为底层支持）及转换操作（Flink 作业）中最弱的一环。因此，从将记录抽取至 Foundry 的初始流式提取开始，需确保整个数据血缘关系维持所需顺序并正确设置分区键。顺序保证强度不会高于抽取数据至 Foundry 的源系统。例如，若使用 Kafka 连接器从 Kafka 抽取数据，应将分区键列设置为与 Kafka 键列相同，以便 Foundry 能够在系统上提供同等的顺序保证。

此外，在一系列数据转换过程中完全更改分区列（及顺序保证）也可能引发问题。若在应用新的 `Key by` 转换前已保证某种顺序，转换操作接收到的记录相对于新增键列将处于乱序状态；这些记录在转换链路期间将始终保持错误的顺序。
