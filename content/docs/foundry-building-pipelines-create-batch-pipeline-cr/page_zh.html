使用代码仓库创建数据集批量管道

本指南将引导您通过一个简单的数据转换示例来使用代码仓库应用程序。您将学习如何编写和编辑SQL代码以及构建您的数据集。您还将学习如何在分支上工作，以便与同事协作。

1. 创建仓库

首先创建一个新的仓库。为此，请导航到Foundry中的一个项目，然后在右上角选择+ 新建，接着选择代码仓库。

在本指南中，我们将编写一个SQL转换。为您的仓库命名，然后在语言模板下拉菜单中选择SQL。然后选择初始化仓库。

Download titanic.csv

3. 创建一个分支

通过创建个人 分支 要进行更改，而不是直接编辑代码的主版本，您可以与同事安全地在同一个代码仓库中协作。您也可以跟踪和撤销更改，并将更改合并到主代码中。代码可以逐行审查，这意味着对生产管道的更改可以在团队成员之间轻松讨论。有关Foundry中分支的更多信息，请参见 此页面。

当您导航到一个代码仓库时，默认情况下您将在 master 分支上。最佳实践是 master 分支受到保护，这意味着无法直接在该分支上编辑代码。请注意，您可以读取受保护分支上的文件，但不能编辑或创建文件。

在您可以将更改添加到您的代码仓库之前，您必须首先创建自己的分支，该分支包含 master 分支上的代码副本。要创建您的分支，请点击当前分支名称旁边的 图标。

这会打开一个对话框，用于选择现有分支并为新分支指定自定义名称：

创建新分支后，您将在左侧看到相同的文件树。您只是复制了您开始的 master 分支上的代码。现在您可以编辑您分支中的文件。

4. 编辑代码

创建新文件

现在您已经在自己的分支中工作，可以通过将鼠标悬停在文件夹上并点击省略号图标来创建一个新的 SQL 文件，然后选择 新文件。一旦您选择 新文件 ，您将被提示选择文件类型并为其命名。在此示例中，选择SQL转换，并选择一个文件名（不包含空格或特殊字符）：

请注意，您的新SQL文件在文件树中被突出显示，该位置是构建后结果数据集所在的位置。

如果文件树中的文件名是绿色的，表示它是您分支中新建的文件，在您开始的主分支上不存在。如果文件名是橙色的，表示它是在master上存在的文件，但在您的分支中已被修改。

您新创建的.sql文件将根据您提供的文件名声明一个输出数据集。例如，如果您在/Public/Authoring内部的代码仓库中创建了titanicAnalysis.sql，那么您的新文件将自动声明一个输出数据集/Public/Authoring/titanicAnalysis。

编辑您的文件

接下来，将占位符文本替换为您的实际数据转换代码。首先，用实际输入数据集的路径替换`SOURCE_DATASET_PATH`。在这个示例中，您将使用在本教程第（2）步中导入的titanic数据集。

请注意，当您开始键入反引号时，自动完成功能会显示一个交互式菜单，列出您可以使用的数据集。当您从列表中选择一个数据集时，数据集引用将被替换为该数据集的唯一ID。这样即使以后数据集被移动，您的转换代码仍能正常工作。

在反引号中输入您的项目名称，找到titanic数据集，并从菜单中选择它。

继续编写SQL代码以对您的数据执行转换。您在输入SQL函数时会看到各种帮助对话框出现。例如，假设您想在“titanic”数据集中创建一个表示乘客性别的一位字母缩写的列。您可以查看有关如何使用SUBSTRING函数的信息：

在继续之前，请完成您的数据转换代码，以选择“Name”（名称）、“Age”（年龄）、“Survived”（是否幸存）和“Ticket”（票号）列以及一个名为“Gender”（性别）的派生列。“Gender”（性别）列表示乘客的性别缩写；要创建此列，请在“Sex”（性别）列上调用SUBSTRING函数。

请注意，您必须为任何在SQL中创建的派生列定义别名。有关编写SQL数据转换的更多信息，请参阅 Spark SQL语言参考。

5. 测试您的更改

在Foundry中，数据集可以分叉（类似于代码）。这对于测试多步骤数据管道的设计很有用。例如，您可以在不破坏其他人下游依赖的情况下，在隔离环境中测试数据管道部分的更改。

现在您已经编写了数据转换代码，应该在您的分支中测试所做的更改。在将更改合并到master分支之前，测试您的更改非常重要，以确保代码按预期工作。

使用预览功能迭代您的更改

在编写代码时，您可以使用预览 加速开发周期并快速迭代更改的功能。预览会使用采样的输入运行您的代码，并提供一个示例输出，而无需提交更改、运行检查或在 Foundry 中生成数据集。示例输出可能不代表构建的结果，但可以为您提供一种确认代码是否正常工作并产生预期结果的方法。

要使用该功能，请点击页眉中的预览，或在代码编辑器底部栏中打开预览辅助工具。您可以预览基于文件的数据集或带有模式的数据集。

对于带有模式的数据集，您可以通过点击您想要编辑的输入的设置图标来自定义预览时使用的输入。选项包括：

  • 原始输入：使用原始输入的样本。
  • 上一次预览：对于在同一仓库中生成的数据集，您可以链接预览的更改，并将数据集的预览作为您预览的输入。
  • 应用自定义过滤器： 在支持的列上应用过滤器，以在输入的特定子集上测试您的更改。例如：“时间戳在给定窗口内的所有行”，或“具有给定字符串值的所有行”。
  • 选择不同的数据集： 选择包含您的代码所需所有列的另一个数据集，以测试特定情况或提供任何其他自定义样本作为您的输入。

当首次在包含文件的数据集上运行预览时，必须配置将在样本中使用的文件。一旦选择了样本文件，可以通过选择相关输入重新配置它们。保存配置后，预览将对所选的文件样本执行代码。

再次运行预览时，无需重新配置输入文件。预览执行后，您可以将输出查看为行或文件。如果您有相应的权限，还可以选择下载输出文件。

提交您的更改

在编写新代码后，您可以提交更改。在代码仓库中，当您想要标记已完成的工作时，可以提交更改。即使在您进行提交之前，默认情况下您的工作也会自动保存。当您到达一个停止点时，提交会特别标记您的更改集。

提示

点击提交按钮将提交您的更改并运行代码的自动检查。点击构建按钮也会提交您的更改。具体来说，点击构建将运行自动代码检查并开始构建您的输出数据集。如果您希望快速测试您的更改而不构建数据集以确保代码通过代码检查，请点击提交。否则，您可以跳到构建您的数据集。

要提交您所做的更改，请点击 在右上角的按钮上点击，输入你所做的更改的摘要。提交更改会触发对你的代码进行自动检查。右上角的一个图标表示这些检查的状态；悬停在此图标上以查看详细信息。

在你的分支上构建你的数据集

要测试你的更改，请点击屏幕顶部的 按钮。

一旦你点击了构建按钮，会发生两件事：自动检查会在你的代码上运行，并且你的输出数据集将开始构建。在此期间，将从你分支中的代码创建一个新的输出数据集，或者现有的输出数据集将更新以反映你的更改。你可以在 构建 助手里查看运行任务的进度。任务完成后， 图标表示每个任务已成功完成。如果您看到 图标，请点击 Build 辅助工具中的 details 按钮，以了解出错的更多信息。这将带您到 Checks 选项卡，在这里您可以查找错误信息并重新触发您的构建。

以下是有关测试您的更改和构建您的数据集的一些重要信息：

  • 点击右上角的 Build 按钮等同于点击代码仓库界面底部的 Build 辅助工具中的 Build 按钮。
  • 在实际点击构建按钮运行检查并开始构建输出数据集之前，您必须选择包含数据转换逻辑的文件。
  • 每次在您的分支上触发构建时，它将在现有正在运行的构建之后排队。

预览您的数据集

一旦您的任务成功完成并且您的数据集已构建，您可以在Build辅助工具中预览已构建的数据集：

点击Build辅助工具中您的数据集的链接以打开完整数据集。

现在，您已在自己的分支上构建了数据集。继续阅读本指南的其余部分，了解如何将您的数据转换代码合并到主分支。

6. 提交您的更改以供审核

到目前为止，您已经：

  • 创建了自己的分支以便进行更改，
  • 创建了一个新的 SQL 文件，包含您的数据转换代码，
  • 测试了您的更改并在您的分支中构建了数据集，并且
  • 预览了已构建的输出数据集。

现在，您将向您的团队成员提出更改以供审查。在您测试了更改并满意于最终的数据集输出后，您可以向您的团队成员提出更改以供审查。

提示

拥有所有者权限的用户在创建一个拉取请求时可以启用“自动合并更改”的选项。只有当至少有一个必需的检查已为您的仓库分支配置并通过时，此选项才可用。如果您启用了“自动合并更改”的选项，您的拉取请求在创建后将自动合并到主代码中。一旦您的分支中的更改合并到主代码中，您的分支也将被自动删除。

要创建一个新的拉取请求，请点击右上角的按钮。这将打开“新建拉取请求”页面，在这里您可以编写更改的描述，并点击创建拉取请求 按钮。

这会创建一个新的拉取请求，包含您提出的更改。拉取请求页面提供了多种工具，您可以使用这些工具来审查这些更改将如何影响您的数据管道：

  • 更改的文件选项卡允许团队成员逐行审查您的代码
  • 影响分析选项卡显示了在您的分支上受影响的数据集发生了哪些变化，包括数据集模式和健康检查的变化
  • 管道审查选项卡显示了您的数据管道的图表，并以可视化方式突出显示此拉取请求中的更改如何影响管道。

了解有关理解拉取请求中更改影响的更多信息。

In general, it's important to invite others to review your changes before they’re merged into the master branch. Users can approve or reject on a file-by-file basis to keep track of which files still need to be adjusted before the pull request can be merged. To see which users have already approved or rejected a particular file, hover over the corresponding indicator icon.

Using the individual file review buttons is optional, but when you reject one file, this automatically rejects the pull request. Similarly, when you approve the pull request, all individual files get approved.

7. Merge your changes into the master branch

Finalizing your changes

Once your changes have been reviewed, a user with the appropriate permissions (by default, Owners and Editors) can merge the changes on your pull request to combine them into master.

For the purposes of this tutorial, proceed by selecting the Squash and merge button at the bottom-right of your screen, then select Squash and merge again in the confirmation dialog.

Your repository may have different policies that must be met before changes are merged. Policies are defined by the repository owner in the branch settings page and presented to code authors in the Pull Request page.

Validating that your changes are on master

Once your proposed changes have been accepted, you should validate that your the changes you made in your branch are reflected on the master branch. To do so, click the Code tab, select the master branch, and browse the files. Make sure you see the changes that you made in your branch.

Deleting your branch

Warning

Do not delete any branches that you did not create. For others working in the same Code Repository, this could result in lost work!

Now you can delete the branch you created at the start of this tutorial to reduce clutter. Since your changes have been merged into the master branch, there is no need to keep your branch. Navigate to the Branches tab, and look under “Personal branches”. Delete the branch you created by clicking the icon:

The pull request page offers a "Delete branch after merge" option to allow for quick clean-up of branches. This option is unavailable for protected branches.

To delete a protected branch, you first need to unprotect it and then follow the steps above.

8. Build your dataset on the master branch

The final step is to build your new dataset on the master branch. Similar to build the dataset in your branch, click the button at the top of the screen when your SQL code file is selected.

Once your tasks successfully complete and your dataset gets built, you can click on the link to your dataset in the Build helper to open your full dataset.

9. Revert changes

If you notice problems with a code change you have already merged into the master branch, there is an easy way to undo those changes. You can revert a specific commit by locating it in the commit history of the master branch. On the Branches tab of your repository, click on master to see the chronological list of all commits.

You can view a certain commit's code changes by clicking on the commit hash. Once you have located the commit you want to revert, click Revert . This will open a pull request into the master branch which you can review and merge.
