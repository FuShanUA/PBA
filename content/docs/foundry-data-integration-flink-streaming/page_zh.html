Flink 基础

Apache Flink 是一款分布式计算引擎，能够以低延迟处理无界数据集，适用于常见的流处理工作流。Foundry 流处理采用 Flink 作为底层引擎，执行用户代码及平台内其他流处理应用，例如实时向本体补充数据以及流式时间序列摄入。

要判断流处理场景是否需要额外的作业配置，了解 Flink 的基本工作原理会很有帮助。

有关 Flink 的更多详细文档，请访问《Flink 主页》 ↗。

Flink 作业

所有流处理作业都描述为一系列作用于数据源集合并将结果写入数据接收器的操作。这些操作包括聚合、连接以及其他行级操作，如字符串解析或算术运算。Flink 将每个操作抽象为算子。在 Flink 中，源和接收器也通过算子来描述。

Flink 作业在内部表示为“作业图”或“逻辑图”。作业图是有向图，节点由算子组成，边定义了算子之间的关系。提交作业到 Flink 时，会生成作业图。在 Foundry 作业追踪器的详情部分，会渲染 Flink 作业图的预览。

实际执行作业时，Flink 会将逻辑图转换为物理图，展示算子在计算运行时的执行方式。物理图由任务构成，任务是 Flink 作业的基本工作单元，可以表示算子的单个实例或多个算子链接在一起。

作业管理器与任务管理器

与 Spark 类似，Flink 的运行时架构包含不同类型的工作节点。Spark 使用驱动程序协调管理作业，使用执行器执行任务；而 Flink 采用作业管理器和任务管理器，其角色与 Spark 驱动程序和执行器大致类似。

Flink 作业管理器负责调度任务、为任务分配资源，处理已完成或失败的任务，协调《作业检查点》与故障恢复，并提供对作业信息的编程访问接口。通常情况下，同一时刻只有一个活跃的作业管理器作为主节点，并保持备份节点处于就绪状态，以应对不可恢复的故障。

Flink 任务管理器负责执行任务，以及在流之间缓冲和交换数据。至少会有一个任务管理器，但为了实现流处理的并行化，可能会有多个任务管理器。当任务管理器需要处理非常大的记录时，可能需要额外资源。如果流的吞吐量极高，可能需要提高作业的并行度，这会导致任务管理器数量增加。

作业状态

有些 Flink 操作只需单独查看单个事件，而另一些操作则需要跨多个事件记住信息。这些是《有状态操作》。有状态操作的一些示例如下：

  • 聚合： 例如，统计滚动五分钟窗口内的事件总数，或计算所有已知事件的滚动平均值。
  • 连接： 执行引擎需要了解之前见过的事件，才能将它们与当前摄入的事件进行连接。

有状态操作所需的信息称为作业状态，由 Flink 使用状态后端存储。状态由任务管理器管理和存储，并由作业管理器以《检查点》的形式进行协调。当状态较大时（例如聚合或连接的窗口非常大），作业管理器和任务管理器可能需要额外资源。
