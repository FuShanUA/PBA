故障排除

本页面包含使用代码工作区时可能遇到的错误的故障排除提示。如果您有其他问题或无法通过此指南解决您的问题，请联系您的帕兰提尔代表。

错误信息和建议的解决方案

错误：未授权 - 会话不再有效。可能已超时。请刷新页面。

如果您遇到“未授权”错误，表示您的会话可能已超时，请按照以下步骤进行修复：

  1. 刷新页面。
  2. 确保您使用 Google Chrome 浏览器访问代码工作区。Safari 浏览器 不受支持。
  3. 如果之前在此代码工作区中加载了带有安全标记的数据，而您失去了对标记数据的访问权限，可能会收到此错误。如果是这种情况，请选择 运行中，然后选择 重新启动工作区 以获得一个没有数据的新会话。
  4. 如果您是第一次使用代码工作区，可能需要联系IT管理员，以确保代码工作区已正确列入白名单。

错误：内容被阻止。请联系网站所有者以解决此问题。

Foundry 采用严格的Content Security Policy（内容安全策略）来确保在代码工作区中渲染的内容得到适当控制。例如，代码工作区不允许从CDN加载任意JavaScript，就像某些Python包如folium中的情况一样。

如果您在关键工作流中收到此错误，请联系您的Palantir代表以讨论该工作流的潜在支持。一些工作流已经得到支持，但可能需要您运行稍微不同的命令：

  • 构建 Dash 应用程序
  • 构建 Streamlit 应用程序
  • 预览 TensorBoard™

错误：包或命名空间加载失败

您可能会遇到如下包或命名空间加载失败的错误：

> library("hdf5r")
Error: package or namespace load failed for 'hdf5r' in dyn.load(file, DLLpath = DLLpath, ...):
 unable to load shared object '/home/user/libs/r/hdf5r/libs/hdf5r.so':
  libhdf5_serial_hl.so.100: cannot open shared object file: No such file or directory

这可能是因为 Foundry 为了减少潜在漏洞的暴露，限制了服务器上预先安装的 Linux 包的数量。然而，一些 R 库依赖于特定的系统 Linux 库，这些库可能未在服务器上预先安装。

请注意，某些包是 不被 Code Workspaces 支持的

请联系 Palantir 的代表，讨论所需 Linux 包的可用性。

错误：安装 pip 包时构建 wheel 失败

Pip 可能需要一些系统依赖在安装前已经预装，而 conda 会将运行 Python 包所需的所有依赖都打包。一般建议是：

  1. 如果可能的话，始终从 conda 安装而不是 pip。
  2. 如果可能的话，使用 conda-build 构建 pip 包并将其发布到 Artifacts 仓库。
  3. 否则，尝试使用以下命令从 pip 安装：
Copied!
1maestro env pip install <package>
  1. 如果 pip 包无法安装，错误信息通常会指出缺少哪个系统依赖或文件。首先找到提供所需依赖的 conda 包，并使用以下命令安装：
Copied!
1maestro env conda install <package>
  1. 重复步骤 3 和 4 直到安装成功

示例

octis 包存在于 Pypi 中但不存在于 conda 中。 运行 maestro env pip install octis 时，我们遇到以下错误：

...

ERROR       error: command 'g++' failed: No such file or directory    
ERROR       [end of output]    
ERROR       
ERROR   note: This error originates from a subprocess, and is likely not a problem with pip.    
ERROR   ERROR: Failed building wheel for libsvm    
 INFO   Running setup.py clean for libsvm

在这种情况下，依赖项（libsvm）似乎需要编译器来构建wheel，因此我们需要首先通过相应的conda包安装编译器。

上述错误表明缺少的依赖项是g++，因此我们应该使用maestro安装相应的conda包（在本例中为gxx_linux-64）：

这次命令应该会成功。

错误：尝试将数据同步到代码仓库失败

当正在同步的git提交太大时会发生此错误。如果提交包含非代码文件（如输出数据），就会发生这种情况。Code Workspaces内的版本控制不支持大文件。相反，可以将数据交互式地写回到Foundry，使您能够轻松地使用和共享代码工作区的输出。

要找出导致此错误的文件，请在工作区中打开终端并执行以下命令：

Copied!
1git rev-list --objects --all |
2   git cat-file --batch-check='%(objecttype) %(objectname) %(objectsize) %(rest)' |
3   sed -n 's/^blob //p' |
4   awk '{sum[$3]+=$2} END {for (i in sum) print sum[i], i}' |
5   sort --numeric-sort --key=1 |
6   $(command -v gnumfmt || echo numfmt) --field=1 --to=iec-i --suffix=B --padding=7 --round=nearest

要解决此错误，请从 git 历史记录中删除任何大文件。例如，要删除所有大于 10 MB 的文件，可以运行终端命令：git filter-repo --strip-blobs-bigger-than 10M。或者，要从 git 历史记录中删除特定文件，请使用 git filter-repo --path path/to/file.ext --invert-paths。

为防止再次发生此错误，请用不应同步的大文件路径或扩展名更新 .gitignore 文件。例如，要防止任何输出 CSV 文件被同步，请添加 *.csv。请注意，您可能需要确保在 IDE 中显示隐藏文件以查看 .gitignore 文件。

要应用这些更改，请在终端中运行 git push --force。按照这些步骤后，同步更改过程应按预期工作。

文件下载或上传错误

出于安全原因，Foundry 通常会限制文件的下载和上传，以确保这些操作在适当的数据治理框架内进行。这意味着用户不能在 Shiny® 或 Streamlit 应用程序中添加“下载”按钮或类似功能。

要从 Jupyter® 或 RStudio® 工作区下载数据，请先将数据写入 Foundry 中的输出数据集。您可以使用 Foundry 内置的过程从输出数据集中下载数据，这可以确保应用适当的数据治理限制。
