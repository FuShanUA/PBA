代码库升级

Foundry 会定期为活跃的代码库生成升级拉取请求。这些升级包含对 Transforms 模板的重要更新以及运行时改进。升级拉取请求将在专用分支上发起，并请求合并至默认分支。

自动升级

在代码库中启用自动升级后，Foundry 会尝试自动合并这些升级拉取请求。所需的检查成功通过后，拉取请求将被合并，默认分支的提交历史中将出现一个新的合并提交。在代码库设置中选择 自动合并升级拉取请求 即可启用此功能。

若未启用此选项，或拉取请求合并失败，当有新的升级拉取请求时，代码库将给出提示消息，并要求具备相应权限的用户手动完成合并。

警告

强烈建议启用自动升级，以确保代码库能够及时更新至最新的运行时改进版本。

在以下几种情况下，代码库自动升级在合并前需要人工干预：

  • 代码库的模板类型不支持合并。
  • 用户在升级拉取请求所在分支发起后，向该分支提交了任何更改。
  • 升级拉取请求覆盖了任何包含用户手动修改内容的文件（参见文件覆盖）。
  • 受此拉取请求直接影响的部分数据集尚未使用最新版本的代码进行构建（参见影响分析）。
  • 该代码库产生的作业尚未使用升级拉取请求中引入的运行时 Spark 模块版本进行构建（参见影响分析）。
  • 代码库当前所处的模板版本过旧，无法支持自动升级。

文件覆盖

创建代码库时，系统会根据所选的模板类型，使用默认的模板文件对其进行初始化。升级代码库时，部分默认文件将被覆盖，以与最新模板版本保持一致。这是因为默认文件对于代码库的正常运行至关重要，用户不应对其进行修改覆盖。

影响分析

你可以利用代码库升级拉取请求中的影响分析选项卡，审查受影响数据集的任何潜在变更，从而在合并前确保这些变更安全无误。部分升级可能会影响所使用的运行时 Spark 模块版本，进而可能对该代码库构建的数据集产生影响。
