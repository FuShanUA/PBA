示例

基于作者团队成员身份自动批准

要自动批准特定团队成员所做的更改，请检查 author.teams。使用策略变量使团队名称可配置，而无需编辑表达式：

type == "UPDATE_ENTITY" && {{ policyVariable.approvedTeam }} in author.teams
  ? approve("由已批准团队的成员编写的更改。")
  : no_op()

将 approvedTeam 定义为一个 STRING 变量，并将其值设为团队名称。这样，您可以通过策略配置更新已批准的团队，而不是直接编辑表达式本身。

如果要根据作者已经在该资源上拥有角色而非团队成员身份来自动批准，请使用 authorRoles 从 referencedResources 中获取：

type == "UPDATE_ENTITY" && "entity:operator" in referencedResources.installation.authorRoles
  ? approve("作者是此安装的实体操作员。")
  : no_op()

为特定实体类型要求特定团队

要编写一个仅适用于特定实体类型的策略，请首先对type进行保护，然后对request.apolloEntityType进行保护：

type == "UPDATE_ENTITY" && request.apolloEntityType == "network-security"
  ? require_team("platform-security", "安全团队必须批准网络配置更改。")
  : no_op()

此策略仅对network-security实体产生审查要求。对于所有其他更改类型和实体类型，它返回no_op()，Apollo将继续处理下一个策略。

跨策略组合要求

由于所有启用的策略都会被评估，并且它们的结果使用AND语义进行组合，因此添加新策略会增加一个必须与现有策略的任何要求同时满足的要求。

例如，如果默认策略已经要求实体操作员批准UPDATE_ENTITY更改，那么启用此附加策略意味着每个UPDATE_ENTITY更改也必须得到安全团队的批准：


type == "UPDATE_ENTITY"
  ? require_team("platform-security", "Security team must approve all entity config updates.")
  : no_op()

在应用更改之前，必须满足所有要求。单个策略不需要了解其他策略的上下文。

自动批准安装特定模块

模块安装会为模块中的每个对象生成单独的对象更改请求。所有启用的策略都会评估这些更改，因此添加一个返回 approve() 的新独立策略来信任某个模块是不够的。现有的对象策略仍会运行并产生它们自己的要求。

要自动批准特定模块的安装，请将模块检查添加到每个现有对象策略的开头，这些策略原本需要批准。在其他分支之前添加此检查，以便在达到默认要求之前立即 approve()：

type == "UPDATE_ENTITY" && request.apolloEntityType == "service" ? (
  is_trusted_module_request(author, metadata, editors)
    && "module" in referencedResources
    && referencedResources.module.name == "my-trusted-module"
    ? approve("Trusted installation of my-trusted-module.")
    : require_operation(referencedResources.installation.rid, "entity:approve-config-update", "Entity operators must approve entity config updates.")
) : no_op()

守卫 "module" in referencedResources 是必需的，因为 module 键仅在变更请求涉及单个模块安装时才存在。将相同的添加应用到覆盖模块中包含的实体类型的每个默认策略，通常是实体配置更新、实体设置更新和实体删除策略。

对模块发布版本的门控

对于属于模块安装的一部分的实体更改，referencedResources 包括 module 和 moduleRelease 键。使用 referencedResources.module.name 通过产品 ID 识别模块，并使用 referencedResources.moduleRelease.name 检查发布版本：

type == "UPDATE_ENTITY" && "module" in referencedResources
  && referencedResources.module.name == "my-module"
  && referencedResources.moduleRelease.name == "1.0.0"
  ? approve("Approved installation of my-module version 1.0.0.")
  : no_op()

此策略会自动批准特定版本的特定模块的安装。对于其他所有更改，它返回 no_op()，其他策略将继续正常评估。

自动批准到受信任主机的出站流量

使用 request.data.all(...) 来检查 CREATE_EGRESS_POLICIES 请求中的每条策略是否在自动批准前指向特定的主机和端口：

type == "CREATE_EGRESS_POLICIES" && request.data.all(policy,
    policy.address.type == "dns"
      && policy.address.dns == "example.com"
      && policy.port.port == 443)
  ? approve("所有策略均指向 example.com 的 443 端口。")
  : no_op()

all(...) 宏确保只有当请求中的每条策略都匹配时才会触发批准。如果任何策略指向不同的主机或端口，该表达式将转到 no_op()，其他策略将继续正常评估。
