示例

基于作者团队成员的自动批准

要自动批准特定团队成员所做的更改，请查看 author.teams. 使用策略变量可以使团队名称可配置而无需编辑表达式：

type == "UPDATE_ENTITY" && {{ policyVariable.approvedTeam }} in author.teams
  ? approve("Change authored by a member of the approved team.")
  : no_op()

将 approvedTeam 定义为一个 STRING 变量，其值为团队名称。这样，您可以通过策略配置更新批准的团队，而不是直接编辑表达式本身。

如果要根据作者已经在资源上拥有角色而不是团队成员身份进行自动批准，请使用 authorRoles 中的 referencedResources ：

type == "UPDATE_ENTITY" && "entity:operator" in referencedResources.installation.authorRoles
  ? approve("Author is an entity operator for this installation.")
  : no_op()

对特定实体类型要求特定的团队

要编写一个仅适用于特定实体类型的策略，请首先对 type 进行保护，然后对 request.apolloEntityType 进行保护：

type == "UPDATE_ENTITY" && request.apolloEntityType == "network-security"
  ? require_team("platform-security", "Security team must approve network configuration changes.")
  : no_op()

此策略仅对 network-security 实体类型产生审查者要求。对于所有其他更改类型和实体类型，它返回 no_op()，Apollo 会继续执行下一个策略。

跨策略组合要求

由于所有启用的策略都会被评估，并且它们的结果以 AND 语义进行合并，因此添加一个新的策略会增加一个必须与现有策略中的任何要求同时满足的要求。

例如，如果默认策略已经要求实体操作员批准 UPDATE_ENTITY 更改，那么启用此附加策略意味着每个 UPDATE_ENTITY 更改也必须得到安全团队的批准：

type == "UPDATE_ENTITY"
  ? require_team("platform-security", "Security team must approve all entity config updates.")
  : no_op()

在更改可以应用之前，两个要求都必须满足。单个策略不需要了解其他策略的上下文。

自动批准特定模块的安装

模块安装会为模块中的每个实体生成单独的实体更改请求。所有启用的策略都会评估这些更改中的每一个，因此添加一个返回approve()的新独立策略对于受信任的模块是不够的。现有的实体策略仍然会运行并产生自己的要求。

为了自动批准特定模块的安装，请将模块检查添加到每个现有实体策略的开头，这些策略原本需要批准。在其他分支之前添加此检查，以便在达到默认要求之前立即approve()：

type == "UPDATE_ENTITY" && request.apolloEntityType == "service" ? (
  is_trusted_module_request(author, metadata, editors)
    && "module" in referencedResources
    && referencedResources.module.name == "my-trusted-module"
    ? approve("Trusted installation of my-trusted-module.")
    : require_operation(referencedResources.installation.rid, "entity:approve-config-update", "Entity operators must approve entity config updates.")
) : no_op()

由于module键仅在更改请求涉及单个模块安装时才存在，因此需要"module" in referencedResources保护。将相同的添加应用到覆盖模块中包含的实体类型的每个默认策略上，通常是实体配置更新、实体设置更新和实体删除策略。

对模块发布版本的闸门

对于属于模块安装的一部分的实体更改，referencedResources 包括 module 和 moduleRelease 键。使用 referencedResources.module.name 通过产品 ID 来识别模块，使用 referencedResources.moduleRelease.name 来检查发布版本：

type == "UPDATE_ENTITY" && "module" in referencedResources
  && referencedResources.module.name == "my-module"
  && referencedResources.moduleRelease.name == "1.0.0"
  ? approve("Approved installation of my-module version 1.0.0.")
  : no_op()

此策略会自动批准特定模块的特定版本的安装。对于其他所有更改，它返回 no_op()，其他策略继续正常评估。

自动批准出站到可信主机

使用 request.data.all(...) 检查在自动批准之前，请求中的每个策略是否都针对特定的主机和端口：

type == "CREATE_EGRESS_POLICIES" && request.data.all(policy,
    policy.address.type == "dns"
      && policy.address.dns == "example.com"
      && policy.port.port == 443)
  ? approve("All policies target example.com on port 443.")
  : no_op()

all(...) 宏确保仅当请求中的每个策略都匹配时才触发批准。如果任何策略针对不同的主机或端口，则表达式将传递到 no_op()，其他策略将继续正常评估。
