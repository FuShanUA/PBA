调试失败的流

在编写流处理管道时，可能会遇到流故障。本页提供了调试失败流的工作流建议，并介绍了 Foundry 中提供的各项工具，帮助你排查流失败的原因。

故障类型

故障可以分为多种类型，不同类型的故障指向的潜在原因各不相同。以下列举了一些最常见的故障类型及其成因。

  • 即时故障：流在启动后 5 分钟内失败。
  • 瞬时故障：流运行一段时间后失败一次，但重启后成功运行并继续处理。
  • 持续故障：流运行一段时间后，连续发生一系列即时故障，每次都在流重启后的 5 到 10 分钟内失败。

即时故障

迅速失败的流通常存在用户编写逻辑方面的问题。例如，失败可能源于无效的类型转换、解析异常或其他阻碍流处理集群成功启动的问题。

如果之前运行成功的流现在失败，请检查逻辑和上游流是否有变动，以排查潜在问题。

要确定这些问题的根本原因，请按照下文检查流处理集群日志工作流中列出的步骤进行操作。

瞬时故障

对于之前成功运行但在重启后再次运行的流，其故障通常指向由网络中断引起的基础设施问题。中断往往由托管 Foundry 实例的云服务商引发（例如 AWS、Azure 或 GCP 宕机）。为流配置调度计划后，Foundry 将自动重新运行流，并从上次中断的位置继续处理。

持续故障

对于之前成功运行但如今未修改逻辑却反复失败的流，其问题通常最难定位。不过，原因一般属于以下两者之一：流入的新数据阻碍了流继续推进（例如类型无效、数据损坏等），或者存在持续的基础设施问题（如 AWS 宕机等）。

要检查错误是否由管道逻辑引起，请按照下文检查流处理集群日志工作流中列出的步骤进行操作。

常见调试工作流

下面概述了可用于解决常见流处理问题的几个工作流。这些检查操作无需按特定顺序执行。

检查流处理集群日志

按照提取流日志中的步骤操作，并在日志中搜索任何异常、错误或“throwables”。这些日志通常会通过“ERROR”、“throwable”或“Exception”等关键字描述潜在的底层问题。

检查管道逻辑

用户编写代码中的 Bug 往往会导致管道逻辑变动引发中断。通常，上一步获取的日志异常会指向引发问题的特定代码。如果存在这种情况，可以尝试回滚所有改动，看看能否解决问题。

检查输入数据

偶尔，流处理管道中的逻辑在遇到意外数据时会抛出异常。例如，管道将两列相除，而除数为 0 或 null 时，就会触发此类行为。检查输入流末尾的记录，确认数据是否发生了可能影响流处理管道的变化。若发现数据异常，可考虑添加过滤器或相应逻辑来清洗或剔除这些数据。

检查流详情

当流处理失败时，Foundry 会检测常见异常并在流详情页面中展示。请查看详情，确认是否有错误信息。

有时，流日志中的错误信息比详情页面更加详尽。建议同时查看这两处，以全面了解流处理失败的原因。

获取流日志

流处理过程中可能会遇到失败，此时需要获取构建日志来定位原因。构建日志通常是排查各类流处理故障的最佳切入点。

要获取流处理作业的日志，请先前往《数据集预览》中的流详情页面。

接下来，找到需要查看的作业，然后点击下载日志。

选择下载日志时，是将最早的日志行放在前面，还是将最新的日志行放在前面。

流处理错误

作业因不可恢复错误反复失败

Caused by: com.palantir.logsafe.exceptions.SafeIllegalStateException: Start transaction rid is not part of the current view.

此错误通常是由于流处理中运行了输入数据集的新快照所致。

流处理会在收到记录后立即将其向下游发送，这使得记录变得不可撤销。生成新快照时，流处理作业会忽略之前的事务，仅使用新事务。若要丢弃旧记录以切换至新快照，请执行手动重放。

解决此问题的方法是，在《部署管道》向导中勾选部署时重放选项，重新部署管道以重放流。
