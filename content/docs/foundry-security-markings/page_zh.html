标记

标记为 Foundry 中的文件、文件夹和项目提供了额外的访问控制层级。标记定义了资格标准，将可见性和操作权限限制在满足这些标准的用户范围内。要访问某项资源，用户必须是应用于该资源的所有标记的成员。平台管理员通常在组织内管理标记。

对标记的访问是二元的（全有或全无）。无论角色如何，除非用户满足所有标记要求，否则无法以任何方式访问文件。

标记旨在允许数据保护官集中管理并审计究竟谁可以访问任何给定类别的数据。标记的一个常见用例是限制对个人身份信息（PII）的访问。例如，可能有一组用户只有在完成一系列培训后才有资格访问敏感的 PII 数据。平台管理员可以创建一个 PII 标记并将其应用于敏感数据集。此标记确保对 PII 的访问仅限于适当的用户，并且不能在该组之外共享。

标记是强制性控制，而角色是自主性控制。强制性控制通过要求用户拥有特定标记才能访问数据来限制访问权限。标记本身上的“扩展访问”权限是一个集中管理的权限，移除标记时必须具备此权限。例如，即使某用户在带有 PII 标记的数据集上拥有数据主人角色，其数据主人角色也不允许他们在没有该标记的“扩展访问”权限的情况下移除该标记。相比之下，自主性控制扩展访问权限，并通过数据共享工作流授予，不受集中限制。例如，在资源上拥有数据主人角色的任何用户都可以向另一个用户或组授予数据主人角色。

用户必须是文件、文件夹或项目上所有标记的成员才能拥有访问权限，因为标记是连接性的（布尔 AND）。

继承

标记沿文件层次结构和直接依赖关系继承，并通过转换和分析逻辑传播。从带有标记的文件、文件夹或项目派生的所有资源都将继承该标记，除非显式移除该标记。不同于基于数据在平台中位置的角色访问控制，标记与数据相伴而行。文件可以通过两种方式继承标记：通过文件层次结构和/或通过数据依赖关系。

文件层次结构

文件可以通过包含它的项目或文件夹继承标记。如果项目或文件夹带有标记，其中的每个文件或文件夹都会继承该标记。这意味着，限制对项目或文件夹的访问始终会限制对其内部所有内容的访问。

数据依赖关系

限制对数据集的访问始终会限制对从中派生的任何数据的访问。这是因为数据集文件可以从它依赖的数据集（例如上游数据集）继承标记。如果数据集带有文件标记，那么依赖它的每个数据集都会继承该标记，这种继承的标记被称为数据标记。

应用标记被视为一项敏感操作，因为标记将立即沿所有文件和数据依赖项继承。这可能会导致下游其他用户意外被锁定。使用前，请先了解如何安全地应用标记。

同样，移除标记也属于敏感操作。你可以从最初应用标记的文件、文件夹或项目中移除标记，这将立即从下游文件和数据依赖项中移除该标记。或者，也可以在转换中移除标记，但这只会沿数据依赖项移除标记。移除前，请先了解如何移除标记。

使用标记

标记旨在限制对文件、文件夹和项目等资源的访问。标记不应用于开通访问权限。当用户满足一组标记条件时，即可获得对该标记及关联资源的访问权。然而，有资格访问的用户并不总是应该拥有访问权。应基于项目上的基于角色的权限向用户授予文件访问权。

例如，假设使用 PII 标记来限制对所有包含员工 PII 的 Foundry 数据的访问。这些 PII 可能存在于财务记录（如社会安全号码）、健康记录（如有关年龄、性别或诊断的信息）或其他个人数据（如姓名或地址）中。

要处理 PII，用户需要接受适当的培训。假设财务部门的某位用户已完成所需培训，并且有资格获得 PII 标记的访问权。由于该用户来自财务部门，因此被授予了某个财务数据项目的 Viewer 角色。即使该用户有资格查看包含员工 PII 的其他数据，其在项目中的角色仍然决定了其访问级别。

标记应用于对需要额外保护的敏感数据定义访问限制。有多种方法可以应用与数据敏感度相关的标记：

  • 每个敏感数据类别一个标记
    • 在最常用的标记结构中，每个敏感数据类别创建一个标记。每个标记限制对包含该数据类别的所有资源的访问。如果数据具有多种敏感类型，则将所有相应的标记应用于该资源；只有有资格访问所有相关敏感类别的用户才能获得对该资源的访问权。
    • 为标记要求和敏感类型建立明确的标准可能会很有帮助。例如，你可以使用 PII 标记来标记任何包含性别、年龄、种族等个人属性的数据。
  • 每个敏感数据主人一个标记
    • 在这种结构下，数据主人可以决定如何限制对其拥有数据的访问。通过为每个敏感数据主人设置一个标记，由团队或用户组拥有的敏感数据会被标记，数据主人可自行决定向用户授予该标记的访问权。例如，销售团队生成、消费和管理所有数据都可以由数据主人用 Sales Data 标记进行标记，并且数据主人仅向符合条件的用户授予销售数据的访问权。
    • 为了让数据主人对其数据资产拥有更多控制权，标记会向下传播。这意味着，如果拥有数据访问权的用户尝试从该数据创建派生资源以与另一用户共享，数据主人需要向另一用户授予该标记的访问权，才能解锁其对派生资源的访问。
  • 用于不同流水线阶段的标记
    • 数据集可能以原始形式摄入到 Foundry 中，并且通常需要经过处理和转换才能共享给最终用户。原始数据可能包含不适合下游用户的敏感信息，管理员可以选择应用 Raw Data 标记来限制未经授权用户的访问。经过处理移除 PII（例如哈希或加密）后，管理员可以移除 Raw Data 标记并应用其他相关标记，以进一步保护流水线中的数据。
  • 使用标记限制数据发现
    • 由于标记以全有或全无的方式限制访问，如果必须隐藏文件、文件夹或项目等资源的存在，则应使用标记。标记可以确保无权访问该标记的用户不会在搜索结果或项目/文件夹视图中看到被标记的数据。

在 Foundry 中实现标记时，可以结合使用上述策略。例如，可以在流水线起始处使用 Raw Data 标记，然后在后处理阶段为每个敏感数据类别分别使用一个标记。

示例：保护医疗数据

假设某医疗机构拥有以下三个层级的敏感患者数据：

  • 合成数据：这是敏感度最低的数据，旨在模拟真实的患者记录。所有合成数据均使用合成数据标记进行标识。
  • 去标识化数据：此类数据包含一些敏感字段，但所有直接标识符均已移除。不过，此类数据可能与其他数据文件结合使用以识别患者身份。所有此类数据均使用去标识化数据标记进行标识。
  • 含标识符数据：此类数据可能包含可用于直接识别 individual 患者身份的标识符。这是患者数据的最高敏感级别。所有此类数据均使用可识别数据标记进行标识。

在此场景下，数据层级呈层级结构，拥有可识别数据标记访问权限的用户，同时也拥有去标识化数据标记和合成数据标记的访问权限。同理，拥有去标识化数据标记访问权限的用户，也拥有合成数据标记的访问权限。

在 Foundry 中，含标识符的数据（使用可识别数据标记）通过移除标识符字段，转换为去标识化数据（使用去标识化数据标记）。标记管理员或数据主人会审查转换逻辑中的所有变更，确保去标识化数据中不存在任何标识符。进一步的复杂转换会生成合成数据，并使用合成数据标记进行标识。在以下截图（使用示意数据）所示的每个转换阶段中，原有的标记会被移除，并添加一个反映数据更新状态的标记。

示例：保护调查数据

案件调查数据尤为敏感，反洗钱调查便是典型场景。不同案件的调查数据绝不能混淆。此外，当调查员在审查某一案件时，不应看到其他案件的数据。标记机制可实现以下案件访问限制：

  • 与特定案件相关的数据（包括资源、图像、数据集及其他证据）均使用唯一的案件 - xxxxxx标记进行标识，其中xxxxxx代表案件编号。
  • 仅特定案件的调查员才能获得该案件标记的访问权限。调查员可同时拥有多个案件的访问权限，但这些权限通过不同的标记进行区分。

示例：保护银行业务数据

在一家假设的银行中，每个团队或部门对其生产或管理的数据拥有完全控制权。即各团队自主决定其他团队能否访问其数据资产（无论整体或部分）。在此示例中，为简化组织架构，我们设定了以下具有对应标记的团队：消费者金融、内部合规与市场营销。

  • 假设消费者金融团队与市场营销团队分别向内部合规团队授予消费者金融和市场营销标记的访问权限。这样内部合规团队即可验证数据是否已正确用于预先批准的工作流，并开展季度审计。
  • 季度审计的结果记录在一份带有内部合规标记的报告中，同时移除另外两个标记。该审计结果仅限内部合规团队访问。若内部合规团队需与数据保护办公室共享该季度审计报告，内部合规团队可向数据保护办公室授予内部合规标记的访问权限，以便其审查合规报告。

请查阅有关如何配置标记的《管理文档》。

使用作用域会话

《作用域会话》允许用户在 Foundry 会话期间选择预定义标记的子集进行访问，从而在不同类型的工作之间实现视觉隔离。如果您的组织已启用作用域会话，您在登录 Foundry 后可能需要选择一个作用域会话，这会将您在 Foundry 中的访问权限限制在该作用域会话的标记子集内。

选择受限会话后，工作区会出现一个横幅，显示该受限会话的名称。

如果你拥有多个受限会话的访问权限，可以将鼠标悬停在工作区横幅上并选择 更改受限会话。系统会弹出登录时显示的受限会话对话框，供你选择受限会话。如果你选择了与当前受限会话不同的会话，页面将会刷新，且你的权限会被限制在所选的新受限会话范围内。

部分用户可能有权使用 无受限会话 选项。该选项允许用户绕过受限会话限制，从而访问其所有标记。
