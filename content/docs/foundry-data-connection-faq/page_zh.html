计划构建从未运行

一个构建已计划在指定时间运行，但并未尝试执行。

如需排查此问题，请执行以下步骤：

  1. 该数据集和分支是否还有其他正在运行的同步任务？请确认没有其他作业在同时运行，因为无法在同一个数据集和分支上并行运行两个同步任务。

  2. 计划是否已暂停？请在该数据集的计划概览页面中确认此计划未被暂停。你可以通过同步任务的在调度器中编辑视图，或数据集预览的管理计划选项访问该页面。

  3. 此同步任务的智能体是否已被禁用？请导航至与该同步任务数据源关联的智能体，并确认该智能体未被禁用。

运行 SNAPSHOT 事务时数据接入导致数据集中出现重复行

同步任务已运行，但导致了数据行重复。

  1. 创建新的同步任务时，请选择运行 APPEND 类型的同步，而不是 Snapshot。
  2. 在声明增量设置时，如果 last_upd_in_appl_ts 是唯一且始终递增的列，请将其设为指定列，然后选择一个小于该列所有其他值的初始值。
  3. 此后无需进行任何额外操作，因为数据连接会追踪已同步的最新值，并且仅接入后续较新的数据行。这里的“较新”是指值大于先前的值，且较新的时间戳大于较早的时间戳。

增量加载运行时信息

我在运行增量同步时使用了什么值？

如需排查此问题，请执行以下步骤：

  1. 进入该数据集。
  2. 在屏幕顶部附近选择历史记录。
  3. 在屏幕左侧选择你关注的相关事务。
  4. 查看构建。
  5. 查看事务。
  6. 在屏幕底部展开自定义元数据部分。
  7. 检查 incrementalMetadata 块并验证其正确性。

数据库与数据集之间的列类型不一致

同步完成后，数据集上显示的类型与我的原始类型不一致。

如需排查此问题，请执行以下步骤：

  1. 如果原始列类型为 TIMESTAMP，请确认在 Foundry 中结果类型是否为 LONG。如果为 LONG，你需要使用首选的数据准备工具（代码仓库、Preparation 或其他应用程序）将该类型解析为 TIMESTAMP。这是由于数据库创建者提供的许多驱动程序会将类型还原为最安全的表示形式，从而导致了这种副作用。
  2. 如果原始列类型为 DECIMAL，且其精度与原数据库不同，我们建议在数据库本身的查询中将数字转换为指定的精度和小数位数，或者在查询中将该列转换为 VARCHAR，然后在 Foundry 中重新进行类型转换。

TIME 类型列在数据库与数据集之间丢失亚秒级精度

我的数据库包含具有亚秒级精度的 TIME 类型列，且其值带有非零的亚秒级分量，但数据集中的值仅能体现到秒级分量的差异。

对于大多数基于 JDBC 的数据源（包括 PostgreSQL 和 Microsoft SQL Server），数据库中的 TIME 值在 Foundry 输出数据集中会反映为一个 INTEGER，表示自午夜以来的毫秒数（向下取整到最接近的整秒）。因此，数据库中的 05:00:00.000、05:00:00.200 和 05:00:00.800 等值，在输出数据集中都将变为 18000000。

为了保留亚秒级的差异，可以考虑在同步配置的 SQL 语句中将该值转换为字符串，如下方的 Postgres 示例所示：

Copied!
1SELECT record_id, CAST(column_with_time_type AS text) FROM table_name

运行中查询的状态

查询开始运行后，我该如何查看其状态？

要进行排查，请执行以下步骤：

  1. 打开 Job tracker 应用程序，并选择正在运行的同步任务。
  2. 此处会显示该同步任务最详细的状态。
  3. 如果可能，请在源数据库中验证查询行为。

同步因架构不匹配而失败

如果在增量 APPEND 事务之间，文件或 JDBC 表的架构发生了变化，您的数据集就会开始报错并提示架构不匹配。Data Connection 确实会推断 JDBC 抽取的架构，但对于基于文件的抽取仅传播现有架构。在这种情况下，如果架构相同，您必须再次应用架构推断。如果架构在 APPEND 事务之间确实发生了变化，则需要为新架构创建新的数据集。

要进行排查，请执行以下步骤：

基于文件

  1. 如果文件是 XLSX 或 CSV 表格数据，可以在已同步的数据集上重新推断架构，这应该不会出现问题。如果此架构与之前匹配，数据集将毫无问题地添加追加的行。

  2. 如果在推断架构后仍然出现架构错误（在数据集预览或其他应用程序中），则需要将此新文件同步到一个新的数据集，因为它代表了对表格的完全不同的视图。

  3. 如果数据集已经追加了新架构，我们建议联系 Palantir 支持团队以回退此事务。此外，需要转到同步概览页面并暂停与此同步关联的任何计划，从而暂停文件到当前数据集的同步。

  4. 具有新架构的后续文件应同步到与原始数据集不同的另一个数据集，因此我们建议将原始同步中的信息复制到新的同步中，但将目标数据集替换为另一个数据集（并在数据集名称中注明新版本）。

  5. 此外，最好删除原始同步，以避免未来发生任何架构不匹配错误并损坏 Foundry 中的现有数据。

JDBC

  1. 如果您预期架构在某个时刻发生变化并在未来保持其新形式，最好将原始表存入一个名称能指示架构版本的数据集中。（例如 account_transactions_v1.0）。
  2. 如果架构在同步执行之前在原始表中发生了变化：
    • 暂停同步的计划（如果存在）
  3. 如果架构在同步执行之后在原始表中发生了变化：
    • 暂停同步的计划（如果存在）
    • 联系 Palantir 支持团队回滚此可能已损坏目标数据集的事务
  4. 同步暂停后，若准备迁移至新架构，须先将新架构接入新数据集：
    • 将同步任务克隆为新同步，并用新数据集替换原目标数据集（例如 account_transactions_v1.1）。随后可将此新数据集与原数据集执行 union 操作，以容纳完整数据。
    • 若业务场景需要，在验证新同步行为正确后，可删除原同步。此举可确保旧数据集不会落入损坏数据，代价是会降低先前加载配置的透明度。

重新添加已删除同步作业规格时出现权限错误

删除同步作业规格后，若尝试重新添加，将导致权限错误。

解决此问题：

创建输出至同一数据集的新同步。这会生成具备正确权限的新作业规格，同时保留目标数据集中的现有数据。

Bootvisor 状态为 Unknown

我的 Bootvisor 卡在 Unknown 状态，无法停止或启动。

操作前请联系 Palantir 支持团队检查您的配置。执行以下步骤将暂时阻止同步在此智能体上运行。请确保数据源配置多个智能体，或在维护窗口期执行这些步骤，以避免停机。

  1. 暂停该智能体上的同步。
  2. 等待所有正在运行的同步任务完成。
  3. 停止该智能体（此步骤可能因智能体当前状态而失败）。
  4. 通过 SSH 登录该智能体机器。
  5. kill 所有与数据连接相关的 JVM 进程。
  6. 执行 ./service/bin/init.sh start 启动 Bootvisor。
  7. 在数据连接中启动智能体，并确认智能体未被 paused。
