在 Pipeline Builder 中使用 LLM 节点

在 Pipeline Builder 中，使用 LLM 节点提供了一种便捷的方法，可大规模地对您的数据执行大语言模型（LLM）处理。将该节点集成到 Pipeline Builder 中，您可以在各种数据转换之间无缝插入 LLM 处理逻辑，无需编写代码即可将 LLM 轻松集成到您的数据流水线中。

使用 LLM 节点内置了预先配置好的提示词模板。这些模板汲取了资深提示词工程师的专业经验，为初学者使用 LLM 提供了友好的入门途径。您还可以在输入数据集中选取几行数据进行试运行，从而在整个数据集上运行模型之前，对提示词进行迭代调优。这种预览功能可在几秒钟内完成计算，有效加快了反馈循环，并提升了整体开发流程的效率。

若要使用此功能，平台管理员必须为您授予自定义工作流的 AIP 功能权限。

选择数据集

若要将 LLM 应用于数据集，请在工作区中选择一个数据集节点，然后选择 Use LLM。

选择提示词

以下是可用提示词模板的不同示例。若要创建自定义提示词，请选择 Empty prompt。

分类

当您需要将数据划分到不同类别时，应使用分类提示词。

以下示例展示了如何填写提示词以实现我们的预设目标：将餐厅评论分为服务、食物和氛围三个类别。

多重性字段允许您选择输出列是包含单个类别、多个类别，还是确切数量的类别。在示例中，我们希望涵盖一条评论可能涉及的所有类别，因此选择 一个或多个类别 选项。

在 上下文 字段中，输入对数据的描述。在示例中，我们输入 Restaurant Review。

在 类别 字段中，输入您想要分配给数据的不同类别。在示例中，我们指定了 Food、Service 和 Atmosphere 这三个类别，因为我们希望将餐厅评论归类到这三个类别中。

在 要分类的列 字段中，选择包含待分类数据的列。在示例中，我们选择 review 列，因为该列包含了我们的餐厅评论。

摘要提取

您可以使用摘要提取模板将数据摘要为指定长度。

在此模板中，您可以指定摘要的长度。您可以选择按词数、句数或段落数来衡量，并在 摘要大小 字段中指定具体数值。

在示例中，我们希望得到一句话的餐厅评论摘要，因此将摘要大小指定为 1，并从下拉菜单中选择 句子。

翻译

若要将数据翻译为其他语言，请使用翻译提示词。在 语言 字段中指定目标语言。在以下示例中，我们希望将餐厅评论翻译为西班牙语，因此在 语言 字段中指定 Spanish。

情感分析

当您需要根据数据的正面或负面情感倾向为其分配数值分数时，请使用情感分析提示词。

在此模板中，你可以配置输出分数的量程。在以下示例中，我们需要一个 0 到 5 之间的数字，其中 5 表示评价最正面，0 表示最负面。

实体提取

当你想从数据中提取特定元素时，请使用实体提取提示词。在以下示例中，我们需要提取餐厅评价中所有的 food、service 和 times visited 元素。

具体而言，我们希望将所有食物元素提取为 String Array，将服务质量提取为 String，并用一个 Integer 表示该顾客光顾餐厅的次数。

为了获得这些结果，我们需要更新 Entities to extract 字段。在 Entity name 下输入 food、service 和 number visited，并配置以下属性：

  • 对于 food，将 Type 指定为 Array，并选择 String 作为该数组的类型。
  • 对于 service，选择 String 作为类型。
  • 对于 number visited，选择 Integer。

现在，已将 LLM 输出配置为符合本示例中指定的类型。

你还可以在提示词页面的 Output type 下，调整结构体中已提取实体的类型。

空提示词

如果现有的提示词模板都无法满足你的使用场景，你可以通过选择 Empty prompt 来创建自定义模板。

视觉

管道构建器还支持视觉功能，允许兼容视觉的模型分析图像并根据视觉输入回答问题。若要检查模型是否具备视觉能力，请在模型选择器中查看该模型的 Capability 下是否带有 Vision 标签。

要使用视觉功能，请在空提示词模板的 Provide input data 部分输入媒体引用列，并选择所需的视觉模型。

目前，视觉提示词不支持将媒体集作为直接输入。请使用 Convert Media Set to Table Rows 转换操作获取 mediaReference 列，然后将其输入到 Use LLM 节点中。

可选配置

输出类型

在提示词页面上，你可以指定 LLM 输出需要遵循的输出类型。选择屏幕底部附近的 Output type 选项，然后从下拉菜单中选择首选类型。

包含错误

同样在提示词页面上，你可以配置输出，使其在输出值旁边显示 LLM 错误信息。此配置会将你的输出类型更改为一个结构体，其中包含原始输出类型和错误字符串。若要包含 LLM 错误，请勾选 Include errors 旁边的复选框。

若要将输出恢复为不包含错误的原始输出，请取消勾选 Include errors 复选框。

跳过已处理行的计算

为了节省计算成本和时间，您可以通过开启跳过重新计算行选项，跳过对已处理行的重新计算。

开启跳过重新计算行后，系统会根据传入输入提示词的列和参数，将当前行与之前处理过的行进行比较。在后续部署中，列和参数值完全匹配的行将直接获取缓存的输出值，无需重新处理。

跳过重新计算行不适用于上一次部署中输出值为错误或空值的行。在这种情况下，这些行将在下次运行时重新计算。

如果对提示词进行了需要重新计算所有行的更改，可以清除缓存。此时，LLM 视图上方会出现一条警告横幅。

要清除缓存，请选择红色的垃圾桶图标。清除缓存后，所有行将在下次部署时重新处理。

如果更改了输出类型，缓存将自动清除。发生这种情况时，会出现一条警告横幅。如果这是误操作，您可以在横幅中选择撤销更改。

在合并分支时，缓存状态的任何更改都会显示在更改页面中。

如果开启了跳过重新计算行的 LLM 节点具有多个下游输出，您必须将这些输出放在同一个作业组中。否则，在尝试部署时会遇到以下错误：

在默认作业组之外创建一个新作业组即可修复此错误。

[高级] 显示模型配置

对于每个提示词，您可以配置该“使用 LLM”节点所使用的模型：

  • 模型类型：GPT 实例的模型类型，例如 3.5 或 4。“使用 LLM”节点还支持像 Mistral AI 的 Mixtral 8x7b 这样的开源模型。
  • 温度：较高的温度值会使输出更加随机，而较低的值则会使输出更加集中和确定。
  • 最大 Token 数：用于限制输出中的 token 数量。您可以将 token 视为语言模型用来理解和处理书面语言的小文本块。
  • 停止序列：如果遇到任何停止序列值，将停止 LLM 生成。您最多可以配置四个停止序列。

试运行

在每个“使用 LLM”面板底部，您可以使用示例测试特定的 LLM。选择试运行选项卡，并在左侧输入要测试的值，然后选择运行。

要测试更多示例，您可以选择添加试运行。

要直接从输入数据中添加示例，请转到输入表选项卡并选择要在试运行中使用的行。选择使用行进行试运行，系统将自动返回到试运行选项卡，并将您选择的行填充为试运行内容。

运行试运行后，您可以查看发送给模型的原始提示词和原始输出。只需选择所选试运行右侧的 </> 图标即可。

此操作会打开一个对话框，包含以下详细信息：

  • 初始提示词：发送给模型的确切提示词，包括后端所做的任何修改或补充。
  • LLM 中间输出：LLM 的所有输出，包括任何失败的输出。
  • 修正信息：针对任何失败所做修正的详细信息。
  • 最终输出：LLM 提供的最终结果。

在左侧选择相应的标题，即可在面板右侧查看原始文本。

预览与创建

如果你使用五个模板之一，可以通过选择预览选项卡，在创建提示词前预览 LLM 提示词指令。在预览选项卡中，你只能查看指令而无法进行编辑。如需编辑模板，请返回至配置选项卡。

你应该选择创建提示词，以编辑新输出列的名称并在输出列中预览结果。

一旦选择了创建提示词，你将无法返回该特定看板的模板。

要更改输出列名，请编辑输出列部分。要查看已应用到输出表预览的更改，请选择应用。要预览输出表，请选择输出表选项卡。此预览仅显示前 50 行。

最后，完成使用 LLM 节点配置后，请在右上角选择应用。这样你就可以为 LLM 看板的输出添加转换逻辑，并在主 Pipeline Builder 工作区中选择 LLM 看板时查看前 50 行的预览。

Spark 批处理流水线中的计算资源使用情况

Spark 通常以 CPU 核心粒度进行并行处理；一个有用（虽略有简化）的心智模型是，每个核心一次处理一行数据。对于算术等大多数操作，这种架构效率很高，因为所有计算都在本地完成。然而，当进程与外部 LLM 交互时，大部分计算都在外部进行，进程会将大量时间花在等待响应上。在这种情况下，通过每个核心使用多线程来实现并发会高效得多，能充分利用等待时间来发送新请求。

基于上述原因，使用 LLM 节点会利用每个核心的多线程来发送大量并发请求。即使使用默认的超小型计算配置（仅使用单核），并发度也非常高。

你可以通过使用提供更多核心的计算配置（增加执行器数量或每个执行器的核心数）来进一步提高并发度，但这存在一种风险：随着核心数增加，你的作业可能会触及 LLM 提供商或AIP 设定的速率限制。使用 LLM 节点的实现包含重试逻辑，即使在遇到这些速率限制时作业也能成功，但如果配置了多个核心，而它们却将大量时间花在重试失败请求上，这种做法可能效率低下或造成资源浪费。

在调整并发度时，我们建议将默认的超小型计算配置作为基线。若想进一步提高并发度，请非常缓慢地进行扩容（每次增加的核心数不要超过两个），并在作业的Spark 详情页面中监控“所有任务的总运行时间”与“作业持续时间”之间的相关性趋势。理想情况下，随着核心数增加，作业持续时间应当减少，而所有任务的总运行时间应保持不变或仅略微增加。如果所有任务的总运行时间开始显著增加，尤其是在作业持续时间没有相应减少的情况下，说明你的核心数过多，应当缩减规模。
