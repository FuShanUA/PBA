CEL 策略

Apollo 使用 通用表达语言（CEL） 策略来确定在更改请求被应用之前必须由谁批准。

当打开更改请求时，Apollo 会评估该空间的所有启用策略，并使用结果来设置审查者要求。

CEL 是一种开源标准，用于在应用程序中编写嵌入式策略，被行业中的许多其他平台使用，包括 Kubernetes ↗ 和 Google Cloud。CEL 语言具有类似于 C、C++、Java、JavaScript 和 Go 中的表达式的简单语法。要开始使用 CEL，请参考他们的 语言定义 ↗。

评估模型

策略在单个实体更改的级别运行，而不是在整个更改请求的级别。一个修改两个实体的更改请求会产生两次独立的策略评估，并且可能为每个实体产生不同的审核要求。

所有启用的策略都会针对每个输入进行评估。它们产生的审核要求通过AND语义组合，这意味着在更改被批准之前，必须满足每个策略的每个要求。

如果某个策略不适用于给定的更改，则必须返回 no_op()。这表示该策略对该更改没有意见，其结果将被丢弃。其他策略将继续正常评估。如果对于给定的更改，所有启用的策略都返回 no_op()，Apollo将返回错误。

策略结构

策略包含：

  • 一个表达式字符串，必须返回如approve()、reject()、no_op()或require_role()的结果。有关完整列表，请参见响应类型。
  • 可选的 命名变量，每个变量都有一个名称、类型和值。变量在表达式中使用 {{ policyVariable.variableName }} 占位符进行引用，在评估之前这些占位符会被配置的值替换。这使得管理员可以在不修改其表达式的情况下配置策略。

输入字段

以下字段可在每个策略表达式中使用。

字段类型描述
type字符串（string）更改类型，例如 CREATE_ENTITY 或 UPDATE_ENTITY。参见 更改类型 API 参考。
author对象（object）打开更改请求的用户。包含 username（字符串）和 teams（字符串集合）字段。
onBehalfOf对象（object） 当服务账户代表其他用户打开更改请求时，会显示该字段。在这种情况下，author 包含服务账户，onBehalfOf 包含实际的请求者。结构与 author 相同。否则，该字段不存在。
editorsset在更改请求被打开后编辑该请求的用户的用户名。每个条目是一个字符串。
metadatamap内部更改请求元数据。直接传递给诸如 is_trusted_module_request 和 is_trusted_import_request 的谓词函数。
requestobject特定于类型的请求数据。可用字段取决于更改类型。参见 更改类型 API 参考。
diffmap更改的字段的稀疏映射。只有更改的字段存在；被删除的字段的值为 "<<unset>>"。
referencedResourcesmap与更改相关联的资源，按资源名称进行键控。可用的键取决于更改类型。参见 更改类型 API 参考。
createdAttimestamp更改请求被打开的时间。
now时间戳当前策略被评估的时间。

示例

以下表达式会自动批准来自可配置信任作者列表的实体配置更新，并要求其他所有作者的操作员批准。该列表定义为一个名为 trustedAuthors（类型 STRING_LIST）的变量，因此可以在不修改表达式的情况下进行维护：

type == "UPDATE_ENTITY" ? (
  author.username in {{ policyVariable.trustedAuthors }}
    ? approve("Change authored by a trusted team member.")
    : require_role(referencedResources.installation.rid, "entity:operator", "Operator approval required.")
) : no_op()

有关更多示例，请参阅 示例。
