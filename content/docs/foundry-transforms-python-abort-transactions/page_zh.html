终止事务

Python 转换支持终止事务，允许在输出数据集未发生更改（即没有向数据集写入新数据）的情况下让作业成功完成。这通过使用 transform 装饰器并在转换中可用的 ctx 对象上调用 .abort_job() 来实现。

如果你需要在特定条件下阻止输出数据集和下游数据集更新，就可以使用终止事务的功能。输出数据集一旦更新，下游数据集就会被视为过期（stale），并在下次构建时（无论是手动还是通过计划构建）进行更新。这为构建失败提供了一种替代方案，让你更容易分辨出到底是什么地方真正出现了故障。

在数据集的事务历史中，已终止的事务会显示为灰色的成功作业。这让你能够一眼分辨出成功的构建究竟是提交了事务还是终止了事务。

以下是你可能需要终止事务的几种情况：

  • 你有一个基于输入数据集内容来更新数据集的自定义条件。
  • 你的数据集需要强制构建，因为它不会过期。例如，通过调用 API 获取数据的数据集，而不是依赖输入数据集的数据集。
  • 你有一个在计划调度时总是更新的回写数据集（详细示例见下文）。

使用 abort_job() 为总是更新但输出并不总是发生变化的数据集添加验证，可以通过避免下游不必要的更新来节省计算资源。

示例：概念性自定义条件

以下是一个基础的概念性示例，在这个例子中，我们希望确保只有当今天的数据到达输入数据集时，才更新输出数据集。

使用 if (len(todays_trips_df.head(1)) == 0) 通常会比 if (todays_trips_df.count() == 0) 返回结果更快，因为它只会检查是否至少存在一行数据，而不会不必要地对所有行进行计数。

中止的事务与被忽略的作业有何区别？

当作业被标记为“忽略”时，计算过程不会运行，因为 Foundry 判定作业规格（JobSpecs）并未过期。而当事务被中止时，作业实际上会运行并成功完成，只是输出数据集保持不变，且不会提交任何事务。

中止的事务与增量事务有什么关系？

增量转换仅读取已提交事务对应的输入和输出数据集视图。这意味着在执行增量计算时，它们会忽略已中止的事务。

当增量转换的整个作业被显式中止事务时，下一次构建将读取（并因此重新处理）输入数据，就像那个中止的事务从未发生过一样，从而能够继续以增量方式运行。

警告

使用 PySpark 时，可以对单个数据集调用 abort()，而不是对整个作业执行该操作。但这种做法并不推荐，因为它可能导致增量计算出现问题。

具体而言，如果仅对部分输出中止事务，构建将无法以增量方式运行。对于中止了事务的输出，其输出作业规范将使用先前的输入事务范围，因为中止的事务会被忽略；而对于提交了事务的输出，其输出作业规范将使用当前的输入事务范围。这种输入事务范围的不一致，意味着转换操作将无法再进行增量运行。

在多输出增量转换中，如果显式地对部分输出中止事务，下一次构建将以快照方式运行，并触发失败的增量计算检查，提示 Provenance records for the previous build are inconsistent。如果设置了 require_incremental=True，构建将失败并报错 InconsistentProvenanceRecords。这是因为输出的当前视图此时是由不同的输入事务生成的。

在运行增量构建时，应使用 TransformContext 上的 .abort_job() 方法来中止整个作业，而不是中止单个输出。此操作将中止构建的所有输出。如果使用 v2_semantics，这是中止输出同时允许后续构建进行增量运行的唯一方式。
