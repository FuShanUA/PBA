本体论权限

查看、编辑和管理本体论资源的权限通过 Compass 进行管理，这是 Palantir 平台的文件系统。本体论资源保存在项目中，所选项目决定了谁可以查看、编辑和管理它们。

此功能已为新本体论启用。对于现有本体论，本体论所有者必须 手动启用此功能，并且现有的本体论资源需要迁移。此功能尚未适用于默认本体论。如果您不确定自己的本体论类型，请联系 Palantir 支持。

这种基于项目的权限方法取代了以前的权限模型：本体论角色和数据源派生的权限。它带来了多项好处：

  • 统一的权限模型： 本体论资源使用与其他资源类型相同的权限系统，因此您只需在一个地方学习和管理权限。
  • 批量管理： 在项目或文件夹级别设置权限，以一次控制多个资源的访问权限，无需为每个单独的项目设置权限。
  • 权限可解释性： 安全 选项卡显示查看和编辑对象类型所需的权限，以及查看实例或运行操作所需的权限。
  • 额外的隐私控制： 通过应用标记或将它们放在用户没有角色授权的项目中，来隐藏敏感的本体论资源。
  • 指南针编目原语： 使用组合和标签对本体论资源进行组织，并使用角色授权或标记来从用户处隐藏不相关的资源。

迁移到项目不会改变谁有权限访问支撑数据源。要查看对象，用户仍然需要在对象类型和数据源上拥有权限。

基于项目的权限示例

考虑一个名为 Building 的对象类型，它被保存为项目 A 中的一个文件。您查看、编辑或管理 Building 的能力取决于您在项目 A 中的角色。如果您是项目 A 中的 Editor，您可以编辑 Building 对象类型。要查看特定的 Building 对象（如 Empire State Building），您需要在该对象类型上拥有 Viewer 角色，并且要么可以访问支撑数据源，要么可以通过 对象和属性安全策略 获得访问权限，具体取决于该对象类型的权限配置。

如果您仅对该对象类型有查看权限，您只能看到诸如模式和联系信息之类的信息，而不能看到实际数据。如果您需要帮助理解所需的权限，请查看 Compass 项目的侧边栏。

查看对象类型和对象

对象类型与对象的权限设置方式不同。要查看对象类型，您必须拥有该对象类型的查看权限，但不需要拥有支撑数据源的查看权限。

要查看对象，您必须拥有该对象类型的查看权限并能访问数据。对底层数据的访问权限由对象类型的的安全配置决定：

  • 对象和属性安全策略: 对象可见性由直接配置在对象类型上的策略控制，与支撑数据源的权限无关。
  • 数据源策略: 对象可见性由支撑数据源上的权限控制。您必须拥有支撑数据源的查看权限才能看到这些对象。

有关配置对象安全的更多信息，请查阅 管理对象安全的文档。有关对象类型（模式）和对象（数据）之间区别的更多信息，请查阅 对象权限的文档。

当对象处于项目中时，必须将支撑数据源导入到项目中，以便对象可以进行索引。如果支撑数据源尚未在项目中，您将在创建对象时被提示导入它。

编辑关联和操作的权限

您需要根据要编辑的资源类型拥有相应的编辑权限：

  • 对于关联： 您必须同时拥有该关联类型和关联对象类型的编辑权限。
  • 对于操作： 您必须拥有该操作类型的编辑权限以及操作所编辑的所有本体论资源类型的编辑权限。

使用不同权限模型进行打包和安装

市场产品使用其打包时所采用的相同权限模型进行安装。 要求将新的本体论资源保存在项目中 的切换选项仅在 本体论配置 中影响新本体论资源的创建 —— 它不会改变市场如何在 目标环境（产品安装的位置）相对于 源环境（产品打包的位置）安装产品。

  • 使用本体论角色打包： 使用本体论角色进行安装。
  • 使用项目权限打包： 使用项目权限进行安装。
  • 发布后源迁移到项目权限： 在下次升级时，目标的本体论资源将迁移到项目权限。

基于分类的访问控制（CBAC）

当本体论资源保存在受 基于分类的访问控制（CBAC） 管理的项目中时，以下规则适用：

  • 创建资源时必须指定文件分类。
  • 文件分类必须等于或低于 项目最大分类。
  • 对象类型实例化失败，如果未指定分类。

市场和文件分类

文件分类不特定于本体论——它们适用于所有文件。下面的规则在此特别说明，因为它们影响项目权限与基于分类的访问控制之间的交互。市场本身不会携带文件分类。

  • 在发布时： 市场不跟踪文件分类。如果市场的分类低于将要丢弃的任何分类，则会检查用户是否具有对这些标记的解密权限。
  • 在安装时： 每个已安装的文件采用目标项目的分类。如果目标项目的分类低于市场的分类，则安装用户必须拥有对这些标记的解密权限。
  • 在升级时： 文件分类不会被更改。在安装时设置的分类可以手动编辑，这些手动编辑在后续升级后仍然有效。

之前的权限模型

以前，权限配置的本体资源根据您的本体授权模型而有所不同。下表总结了每种模型中资源的当前管理方式。有关这些旧权限系统的更多信息，请参阅文档。

旧本体权限模型描述
本体角色- 在本体管理器中使用特定于本体的角色（本体 viewer，本体 editor，以及本体 owner）对本体资源进行权限设置。它们不是项目的资源。
数据源派生 - 本体论资源的权限来源于对象的支撑数据源。例如，如果你是支撑数据源的编辑者，则你对对象类型具有editor。
