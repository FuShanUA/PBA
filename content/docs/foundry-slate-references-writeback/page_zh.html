将数据回写至 Phonograph [遗留]

对象存储 V1（Phonograph）的回写功能目前处于遗留阶段，预计不会再进行额外的开发。任何新的工作流都不再支持 Phonograph 回写。

我们建议迁移至对象存储 V2，以便利用本体的改进与演进的架构。此外，我们建议将操作与操作微件结合使用，以对本体应用更新和更改。有关更多信息，请参阅操作类型文档。

要从 Slate 中的用户输入/数据修改中获取数据并将其保存至 Foundry，需要三个组件：

  1. 源数据集：这是 Foundry 中已有且您希望开放编辑的数据集。该数据集必须具有主键。
  2. Phonograph 同步：Phonograph 是一个手动编辑缓存，允许 Slate 捕获用户对源数据集的输入更改。您需将数据集同步至 Phonograph（而非 Postgres），这会使得该数据集在 Slate 中可用并允许用户进行编辑。
  3. 回写数据集：这是您将创建的 Foundry 数据集，用于存储源数据集经用户修改后的版本。源数据集将始终保持未修改状态，而存储在 Phonograph 缓存中的所有用户编辑内容都将保存至 Foundry 的回写数据集中。

添加源数据集

首先，在 数据集 标签页中将源数据集添加至您的 Slate 应用程序。我们将使用 + 添加 按钮添加一个名为 asteroid_notes 的简单派生数据集。这个示例数据集包含小行星名称以及一个名为 research_notes 的空白列，该列用于存放用户输入的小行星相关信息。

添加数据集后，您将查看该数据集的同步设置。我们将配置与名为 Phonograph 的手动编辑缓存的同步。

选择一个主键。所选主键必须能够唯一标识单行数据。通常，您必须在转换中组合多个列，以便在建立索引前生成合适的主键。您也可以将一组列定义为联合主键；但这会增加数据编辑的复杂性。

配置好主键后，我们将创建一个回写数据集，用于在 Foundry 中存储用户的编辑内容。选择“浏览”，在 Foundry 中创建一个数据集，以便保存您的编辑。在弹出的窗口中，您需要指定数据集的名称以及希望在 Foundry 中创建该数据集的位置。

关于移除同步的说明

目前无法通过 UI 注销（删除）同步。但是，若要在 Slate 中使用，您可以向“表注册 - 注销”端点添加查询。该查询的唯一参数是表 RID，执行结果将是删除该表——这不会影响输入或输出数据集。在关联数据集的同步 UI 中，您将看到不再有已注册的回写同步。此操作将永久删除表中的所有数据，请谨慎使用，并确保删除该查询或将其设置为手动运行。

读取更新时的搜索服务与存储服务对比

搜索服务的接口使用关联 Phonograph 表的搜索索引。该搜索索引与存储服务的 Post Event 接口并非同步更新，这意味着从变更写入到该变更出现在 Search 接口的查询结果中，会存在短暂延迟。一种简单的处理方式是使用 Toast 组件创建一个定时器，如下所示：

  1. q_postEvent.success -> w_successToast.open
  2. w_successToast 超时时间设置为 5000（5秒）
  3. w_successToast.close -> q_getAllRows

你可能需要适当调整 Toast 的持续时间，以确保能准确获取已变更的行。同时，请务必在 getAllRows 查询中添加 sort 块，因为默认排序是按行文档的最后更新时间戳进行的，这意味着编辑过的行会“移至”所有结果的末尾。

或者，你可以为应用设计一种将数据展示与数据持久化到 Phonograph 分离开来的模式。具体做法是，使用一个状态变量来追踪当前视图，随后在用户触发时定期将该数据保存回 Phonograph，但不要自动重新运行查询来获取所有行。这种模式能带来更好的用户体验（更新看似即时完成），但需要更为复杂的设计。

任何发往 Table Storage Service 的请求（包括通过主键值列表获取行）都保证包含所有编辑操作，因此串联如下事件是安全的：q_postEvent.success -> q_getRowByPrimaryKey.run

在这种情况下，通过主键查询所获取的特定行的任何已写入变更，都会确保包含在结果中。

下面我们将讨论如何利用批量端点和事件框架，构建更强大的方案来处理变更时的延迟问题。
管理延迟
受限于手动编辑缓存的底层架构，成功调用 Post Event 后，编辑内容不一定会立即写入主索引。目前团队正致力于将 Get All Rows 和 Get Rows 调用改为同步执行，这样只要将 q_getAllRows 设定为在 q_postEditEvent 查询成功后触发，就能确保所有编辑内容如期呈现。然而，若依赖 search 端点，则需要预留几秒钟的延迟时间，等待搜索索引完成更新。
最简单的方法是在编辑查询与获取所有行查询之间插入一个提示框。将提示框的触发绑定至 q_postEditEvent.success 事件，并持续 3 至 4 秒（即毫秒值设为 3000-4000）。随后，在 w_editSuccessToast.closed 事件上触发 q_getAllRows 查询，这段延迟时间足以让索引完成更新。
面对更复杂的业务流程，建议进一步将 Slate 应用的展示状态与编辑索引的状态解耦。具体而言，在页面加载时，将 q_getAllRows 查询的结果赋值给一个 Slate 变量，以此作为“前端状态”。后续所有的编辑、删除、更新事件均针对该前端状态变量进行操作，而非直接向后端发起查询。随后，可定期将前端变量的状态与查询返回的数据进行比对，并生成批量查询的载荷，从而一次性提交所有变更。


你可能还希望通过提示框（toast）提供一些用户反馈，或者设置一个“未保存更改”的计数器，并配有一个手动按钮来保存所有更改。这种方案较为笨重，仅适用于复杂且长期运行的应用程序。在大多数情况下，提示框方案已经足够，而且对 Get Rows 和 GetAll Rows 端点的改进很快就会让这种方案在几乎所有场景中变得多余。因此，如果你正在考虑这一选项，最好直接联系 Palantir 团队作进一步探讨。


聚合


search 端点还支持聚合功能，可针对已索引的数据生成统计信息并对数值进行分桶。


系统支持部分《ElasticSearch 2.0 Aggregation ↗》语法：value_count、max、geohash_grid、terms、top_hits、sum、cardinality、avg、nested、filter、histogram、min。


请注意，受限于 Elasticsearch，部分聚合结果可能为近似值。在使用任何类型的聚合之前，请查阅 Elasticsearch 官方文档，明确该结果是精确值还是近似值。若为近似值，需在继续操作前评估该误差是否在你的业务场景中可接受。
