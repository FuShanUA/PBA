API：查询模型或建模目标的实时部署

在直接模型部署和建模目标中，您都可以创建一个实时部署，并通过HTTP端点托管您的模型。您可以通过使用生产环境中的模型来测试托管模型，方法是通过 模型上的函数，或者直接使用实时部署API。

要直接查询托管模型，请从以下端点选项中选择：

  • 多I/O（输入/输出）端点： 一个灵活的端点，可以处理复杂的托管模型和输入类型，以及具有单一输入和输出的简单模型。多I/O端点的URL以 /v2 结尾，也被称为“v2端点”。
  • 单I/O端点 [过时]：
    • 一个只能处理简单托管模型的遗留端点，其API由单个输入和输出组成。单输入/输出端点也称为“v1端点”，将来将被弃用。

单输入/输出端点已弃用，仅在建模目标部署中受支持。

多输入/输出端点

多输入/输出端点 - 有时被称为“v2端点” - 是一个灵活的端点，支持一个或多个输入和一个或多个输出。

  • URL： <ENVIRONMENT_URL>/foundry-ml-live/api/inference/transform/ri.foundry-ml-live.<LIVE_DEPLOYMENT_RID>/v2
    • <ENVIRONMENT_URL>：有关更多信息，请参见 下方 部分。
  • HTTP方法： POST
  • 认证类型： Bearer令牌
  • 必需的HTTP头：
    • Content-Type：必须为 "application/json"。
    • Authorization：必须为 "Bearer <BEARER_TOKEN>"，其中 <BEARER_TOKEN> 是您的认证令牌。
  • 请求体：包含要发送到模型的信息的JSON对象。此对象的预期结构取决于部署模型的API。
  • 响应：成功的响应将返回200状态码和一个表示模型返回的推理响应的JSON对象。此对象的结构将反映当前部署模型的API。

示例：使用多输入输出端点查询实时部署

在以下示例中，我们将使用一个具有简单API的模型，该API包含一个输入和一个输出。

此示例中的托管模型期望一个名为inference_data的单个输入，这是一个包含text列的数据集。在这种情况下，预期的请求格式如下：

Copied!
1{
2  "inference_data": [{
3    "text": "<Text goes here>"
4  }]
5}

模型返回一个名为output_data的数据集，其中包含一个prediction列。这表示如下响应：

Copied!
1{
2  "output_data": [{
3    "prediction": "<Model prediction here>"
4  }]
5}

Copied!
1curl --http2 -H "Content-Type: application/json" -H "Authorization: <BEARER_TOKEN>" -d '{ "inference_data": [ { "text": "Hello, how are you?" } ] }' --request POST <ENVIRONMENT_URL>/foundry-ml-live/api/inference/transform/ri.foundry-ml-live.<LIVE_DEPLOYMENT_RID>/v2

单输入/输出端点

不再推荐在新实现中使用单输入/输出端点。请优先使用上述描述的更灵活的多输入/输出端点。

  • URL: <ENVIRONMENT_URL>/foundry-ml-live/api/inference/transform/ri.foundry-ml-live.<LIVE_DEPLOYMENT_RID>
  • <ENVIRONMENT_URL>: 有关更多信息，请参见下面的 部分。
  • HTTP 方法: POST
  • 认证类型: Bearer token
  • 必需的 HTTP 头信息:
    • Content-Type: 必须为 "application/json"。
    • Authorization: 必须为 "Bearer <BEARER_TOKEN>"，其中 <BEARER_TOKEN> 是您的认证令牌。
  • 请求体: 包含以下字段的 JSON 对象：
    • requestData: 一个包含要发送到模型的信息的数组。此数组的预期形状取决于已部署模型的API。
  • 响应： 成功的响应将返回状态码为 200 和一个包含以下字段的JSON对象：
    • modelUuid: 一个标识模型的字符串。
    • responseData: 一个对象数组，每个对象代表模型的推理响应。这些对象的形状取决于已部署模型的API。

环境URL

在上面显示的示例中，<ENVIRONMENT_URL> 占位符表示您环境的 URL。要检索您的环境 URL，请从部署的 查询 选项卡中复制 curl 请求并提取 URL。

错误处理

最常见的 HTTP 错误代码如下所示：

  • 400：由无效请求或模型失败引起。
    • 通常是在模型推理期间发生异常导致的。堆栈跟踪通常会在响应体中以帮助调试。
    • 如果请求不符合模型 API，也会返回此错误代码。
  • 422：由格式不正确的请求引起。确认请求中的 JSON 是有效的，并且格式符合预期。
  • 429：由过多的请求引起。使用退避重试或减少并发量。速率限制取决于您的环境。
  • 500： 由于内部服务器或部署错误导致。可能需要手动干预。
  • 503: 服务或部署不可用。当部署处于资源不足、内存不足、无法处理更多并发请求或正在自动尝试重启时，可能发生此情况。请使用退避重试。可能需要手动干预。
