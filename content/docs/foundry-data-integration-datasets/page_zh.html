数据集

数据集是数据从接入 Foundry 到映射至本体过程中最基本的表现形式。从根本上讲，数据集是存储在底层文件系统中的一组文件的封装。使用 Foundry 数据集的优势在于，它为权限管理、模式管理、版本控制以及随时间的更新提供了内置支持。本文余下部分将探讨支撑这些功能的核心概念。

在 Foundry 的数据集成层，数据集用于存储和表示各类数据——结构化、非结构化以及半结构化数据：

  • 结构化（表格型）数据包含以 Parquet ↗ 等开源格式存储表格数据的文件，以及有关数据集中列的元数据。这些元数据作为模式与数据集一同存储。
  • 非结构化数据集包含图像、视频或 PDF 等文件，但未关联模式。
  • 半结构化数据集包含 XML 或 JSON 等文件格式。虽然可以为这些文件格式应用模式，但为了提升性能和易用性，最好在下游数据转换中推导表格模式。

事务

数据集会随时间推移发生变化。当你在 Foundry 中打开数据集并看到行和列时，实际看到的是最新的数据集视图。

作为终端用户，你通常会使用构建来修改数据集，该功能允许你根据指定的逻辑更新数据集内容。然而在底层，数据集是通过事务随时间进行更新的，事务代表了对数据集内文件的修改。事务具有简单的生命周期：

  • 事务启动后，会处于 OPEN 状态。在此状态下，可以在数据集的底层文件系统中打开并写入文件。
  • 事务可以被提交，这将使其进入 COMMITTED 状态，所有已写入的文件此时将出现在最新数据集视图中。
  • 事务可以被中止，这将使其进入 ABORTED 状态。事务期间写入的所有文件都将被忽略。

如果你是软件工程师，可能对 Git 较为熟悉。数据集事务是 Foundry 支持数据版本控制的基础，有时也被称为“面向数据的 Git”。事务类似于 Git 中的提交：对数据集内容进行的原子变更。

事务类型

在事务中修改数据集文件的方式取决于事务类型。共有四种可能的事务类型：SNAPSHOT、APPEND、UPDATE 和 DELETE。

SNAPSHOT

SNAPSHOT 事务会用一套全新的文件替换当前的数据集视图。

SNAPSHOT 事务是最简单的事务类型，也是批处理流水线的基础。

APPEND

APPEND 事务会向当前数据集视图添加新文件。

APPEND 事务无法修改当前数据集视图中的现有文件。如果开启了 APPEND 事务并覆盖了现有文件，随后尝试提交该事务将会失败。

APPEND 事务是增量流水线的基础。只需将新数据同步至 Foundry 并在整个流水线中仅处理这部分新数据，就能以高性能的方式端到端处理大型数据集的变更。然而，构建和维护增量流水线伴随着额外的复杂性。了解更多关于增量流水线的信息。

UPDATE

UPDATE 事务与 APPEND 类似，会向数据集视图添加新文件，但也可以覆盖现有文件的内容。

UPDATE 事务破坏了《增量管道》的仅追加要求。如果数据集接收到 UPDATE 事务，下游管道将无法增量处理数据，必须回退至 SNAPSHOT（批处理）方式。仅在必须修改现有文件时，才应使用 UPDATE 事务。有关更多详细信息，请参阅《增量转换与仅追加输入》。

在配置《基于文件的同步》以从外部系统摄入数据时，所选的事务类型决定了数据在管道中的流动方式。以下是三种常见的摄入模式：

  • 批量镜像（SNAPSHOT）： 每次同步运行都会摄入所有文件并提交 SNAPSHOT 事务。这是最简单的模式，适用于文件总量较小、每次运行均可重新加载的场景。
  • 增量镜像（APPEND）： 每次同步运行仅摄入新文件并提交 APPEND 事务。此模式支持端到端的增量管道，也是在文件仅添加不修改时的推荐方案。
  • 增量镜像（UPDATE）： 每次同步运行会摄入新文件和已修改文件，并提交 UPDATE 事务。仅当外部系统会修改现有文件时才使用此模式，因为它会阻碍下游的增量处理。

有关这些模式的详细配置指南（包括过滤设置和冲突选项），请参阅《基于文件的同步》。

DELETE

DELETE 事务会移除当前数据集视图中的文件。

请注意，提交 DELETE 事务并不会从底层文件系统中删除实际文件，它只是从数据集视图中移除了对该文件的引用。

在实践中，DELETE 事务主要用于实现数据保留工作流。通过基于保留策略（通常基于文件保留时间）删除数据集上的文件，可以将数据从 Foundry 中移除，这既有助于最小化存储成本，也能满足数据治理的要求。

事务类型示例

假设数据集某分支具有以下事务历史，从最早的事务开始：

  1. SNAPSHOT 包含文件 A 和 B
  2. APPEND 添加文件 C
  3. UPDATE 修改文件 A，使其内容变为 A'
  4. DELETE 移除文件 B

此时，当前数据集视图将包含 A' 和 C。如果我们添加第五个包含文件 D 的 SNAPSHOT 事务，当前数据集视图随后将仅包含 D（因为 SNAPSHOT 事务会开启新视图），而前四个事务将保留在旧视图中。

保留

由于 DELETE 事务实际上并未从《底层文件系统》中移除旧数据，您可以使用《保留策略》来移除不再需要的事务中的数据。

分支

数据集事务旨在使数据集内容能够随时间变化，而要实现协作——允许多个用户同时试验对数据集的修改——还需要额外的功能。数据集中的分支即专为支持此类工作流而设计。

要了解 Foundry 中的分支机制（包括单个数据集和跨整个管道的分支），请参阅《分支》概念页。

数据集视图

数据集视图表示某个分支在特定时间点上数据集的有效文件内容。历史视图类似于数据集的历史版本。要计算视图中包含哪些文件，需遵循以下步骤：

  1. 从空文件集开始。
  2. 特定时间点的视图始于该时间点之前最近的 SNAPSHOT 事务。如果不存在 SNAPSHOT 事务，则改用数据集的最早事务。
  3. 对于视图中的第一个事务以及随后的每个事务，执行以下操作：
    • 对于 SNAPSHOT（仅限第一个事务）或 APPEND 事务，将该事务的所有文件添加到集合中。
    • 对于 UPDATE 事务，将该事务的所有文件添加到集合中并替换现有文件。
    • 对于 DELETE 事务，从集合中移除该事务中的所有文件。

最终生成的文件集合构成了一个数据集视图。

如果数据集仅包含 SNAPSHOT 事务，视图数量等于事务数量。对于增量数据集，视图数量则等于 SNAPSHOT 事务的数量。

一个视图可能包含来自多个分支的事务。例如，假设一个增量数据集以 SNAPSHOT 起始，并在 master 分支上包含一些 APPEND 事务，如果该分支上的后续事务也是 APPEND（严格来说是非 SNAPSHOT）事务，这些事务将构成该分支上数据集视图的开端。

模式

模式是数据集视图上的元数据，定义了应如何解析视图中的文件。这包括视图内文件的解析方式，以及文件中列或字段的命名与类型设定。Foundry 中最常见的模式是表格型模式——它们描述了数据集中的列，包括列名和字段类型。

需要注意的是，数据集中的文件并不保证一定符合指定的模式。例如，可以将 Parquet 模式应用于包含 CSV 文件的数据集。在这种情况下，试图读取该数据集内容的客户端应用程序会因为部分文件不符合模式而遇到错误。

模式存储在数据集视图上，因此会随时间变化。这一机制非常实用，因为数据集的内容在结构上可能随时间推移而发生改变。例如，新事务可能会为表格型数据集引入新列，或者更改某个字段的类型。

在 Foundry 中，你可以进入《数据集预览》应用，导航至详情选项卡并选择模式，以查看任意数据集的模式。

受支持的字段类型

以下为数据集可用的字段类型列表：

  • BOOLEAN
  • BYTE
  • SHORT
  • INTEGER
  • LONG
  • FLOAT
  • DOUBLE
  • DECIMAL
  • STRING
  • MAP
  • ARRAY
  • STRUCT
  • BINARY
  • DATE
  • TIMESTAMP

部分字段类型需要额外参数：

  • DECIMAL 需要 precision 和 scale。如果不确定这些参数的设置值，推荐使用默认值 precision: 38 和 scale: 18。38 是可能的最高精度值。
  • MAP 需要 mapKeyType 和 mapValueType，两者均为字段类型。
  • ARRAY 需要 arraySubType，这是一个字段类型。
  • STRUCT 需要 subSchemas，即字段类型列表。

有关上述字段类型的更多信息（包括描述和示例），请参阅《Spark 数据类型文档 ↗》。

模式选项

在数据集预览的详情选项卡中的模式部分，你可以在模式元数据底部的 options 块中为 CSV 文件添加可选的解析配置。有关更多信息，请参阅《CSV 解析》文档。

文件格式

模式包含有关数据集中文件底层存储格式的信息。最常用的三种格式为：

  • Parquet
  • Avro
  • 文本

文本文件格式可用于表示多种文件类型，包括各种 CSV 格式或 JSON 文件。关于文本解析方式的其他信息存储在名为 customMetadata 的模式字段中。

实际上，对于 JSON 或 XML 等非表格型格式，我们建议将文件存储在非结构化（无模式）数据集中，并按照《模式推断文档》中的说明在下游数据转换中应用模式。

后端文件系统

数据集内追踪的文件并不直接存储在 Foundry 中。系统会维护文件在 Foundry 中的逻辑路径与后端文件系统中的物理路径之间的映射关系。Foundry 的后端文件系统由 Hadoop FileSystem ↗ 中的基础目录指定。这可以是自建的 HDFS 集群，但更常见的做法是配置使用云存储服务，例如 Amazon S3。所有数据集文件均存储在后端文件系统基础目录之下的文件夹层级中。
