基于文件的同步

使用数据探查创建基于文件的同步任务后，可以在同步页面的配置选项卡中更新配置。

基于文件的数据接入模式概念

尽管 Foundry 基于文件的同步提供了底层设置以实现更高的灵活性和可配置性，但大多数用例都会遵循已知的模式。下表记录了已知模式及其实现预期行为所需的底层设置，以及可能与这些模式产生冲突的设置。

使用 SNAPSHOT 的批量镜像（默认）

  • 事务类型： SNAPSHOT
  • 过滤器： 无

每次运行都会接入外部系统子目录中嵌套的所有文件，包括之前运行中已接入的文件，并向输出数据集提交包含确切这些文件的 SNAPSHOT 事务。输出的 Foundry 数据集视图将包含一个包含所有文件的单个 SNAPSHOT 事务。

冲突设置

  • 过滤器： 排除已同步的文件
    • 会导致尾部窗口数据接入模式。
  • 过滤器： 限制文件数量
    • 如果限制值小于可用文件总数，会导致输出的 Foundry 数据集视图仅包含所需文件的一个不确定性子集。
  • 过滤器： 至少 N 个文件
    • 如果外部系统指定子文件夹中的嵌套文件数量不足 N 个，此设置将产生一个空事务，导致接入 0 个文件。否则，此设置无效。

使用 APPEND 的增量镜像

  • 事务类型： APPEND
  • 过滤器： 排除已同步的文件

输出的数据集视图将包含一系列 APPEND 事务，这些事务聚合了指定子文件夹中任意任务运行期间曾存在的所有嵌套文件。每次运行都会以文件路径名为键，接入所有尚未接入的文件，并向输出数据集提交一个 APPEND 事务。

冲突设置

  • 过滤器： 启用 最后修改日期 或 文件大小 选项的 排除已同步的文件
    • 当已有文件的 最后修改日期 或 文件大小 发生变化时，这些选项会尝试在 APPEND 事务中以文件路径名为键，错误地重新接入这些已存在的文件。如需支持对现有文件的更新，请查阅使用 UPDATE 的增量镜像（incremental with UPDATE）数据接入模式。

使用 UPDATE 的增量镜像

  • 事务类型： UPDATE
  • 过滤器： 排除已同步的文件
  • 以下一项或两项：
    • 过滤器： 启用 最后修改日期 选项的 排除已同步的文件
    • 过滤器： 启用 文件大小 选项的 排除已同步的文件

输出的数据集视图将包含一系列 UPDATE 事务，这些事务聚合了指定子文件夹中任意任务运行期间曾存在的所有嵌套文件的最新版本。每次运行都会以文件路径名为键，接入所有尚未接入或此后发生更改的文件，并向输出数据集提交一个 UPDATE 事务。

注意事项

仅当对现有文件的修改是外部系统不可协商的固有行为时，才使用此模式。虽然从某种意义上说数据接入是增量的，即每次运行只接入新增或已更改的文件，但下游管道无法以增量方式运行，因为输出数据集（下游管道的输入）并非仅追加。

使用 SNAPSHOT 的尾部窗口

  • 事务类型： SNAPSHOT
  • 过滤器： 排除已同步的文件

输出的数据集视图将包含单个SNAPSHOT事务，其中仅包含从未在任何先前作业运行中出现过的文件。每次运行都会以文件路径名为键，摄入所有尚未摄入的文件，并向输出数据集提交一个 SNAPSHOT 事务，其中仅包含这些文件。

当只有“近期”文件（即在外部系统中于倒数第二次和最后一次运行之间创建的文件）与下游流水线和操作相关时，此模式非常有用。在先前运行中已摄入的文件将不会在输出数据集视图中显示。

矛盾设置

  • 过滤器： 限制文件数量
    • 当在给定时间窗口内外部系统中创建的文件数量超过指定限制时，系统将摄入这些文件的一个非确定性子集，其余文件将推迟到后续窗口处理。随着时间推移，该数量可能迅速增长，从而破坏输出数据集中预期的“时效性”。

除了限制输出中所需文件类型的过滤器外，指定子文件夹和可选正则表达式始终是安全的。此类过滤器包括使用 在此时间之后修改 排除过时文件，或使用 路径不匹配 排除特定文件扩展名的文件（例如 .sh 可执行文件）。

只有 排除已同步的文件、至少 N 个文件 和 限制文件数量 这三个过滤器与所选同步模式紧密耦合，可能会对其产生干扰。

同步失败行为

与所有批量同步一样，基于文件的同步是事务性的。如果同步在任意环节失败，事务将中止，且该次运行中的所有文件都不会提交到数据集。

例如，如果同步配置为上传一百万个文件，而在上传完最后一个文件之前发生网络错误，整个事务将回滚。已上传的文件都不会被提交，数据集视图也不会发生改变。

由于这种行为，在单次运行中处理大量文件的同步更容易受到瞬时故障的影响。为降低同步失败的影响，建议使用带有文件限制过滤器的增量 APPEND 同步，以便每次运行处理较小批次的文件。这样，发生故障时只需重新同步当前批次，而无需同步所有文件。

过滤器

过滤器允许你在源文件导入 Foundry 之前对其进行筛选。支持的过滤器类型包括：

  • 排除已同步的文件： 仅同步自上次同步以来添加或大小/日期发生修改的文件。

排除已同步的文件 存在已知的规模限制，因为每次同步运行都需要扫描所有文件。对于需要摄入大量文件的同步，我们建议将其拆分为针对历史数据的 SNAPSHOT 同步，以及带有 排除已同步的文件 的 INCREMENTAL 同步，从而确保该过滤器仅应用于较小批次的近期文件。

  • 路径匹配： 仅同步路径（相对于连接器根目录）与正则表达式匹配的文件。
  • 路径不匹配： 仅同步路径（相对于连接器根目录）与正则表达式不匹配的文件。
  • 在此时间之后修改： 仅同步在指定日期和时间之后修改过的文件。
  • 文件大小介于： 仅同步大小在指定的最小和最大字节值之间的文件。
  • 任一文件路径匹配：如果任一文件的相对路径匹配该正则表达式，则同步该子文件夹中未被其他规则过滤的所有文件。
  • 至少 N 个文件：仅当剩余的过滤文件至少达到 N 个时，才同步所有过滤后的文件。
  • 限制文件数量：限制每次事务保留的文件数量。此选项可提高增量同步的可靠性。

转换器

转换器允许在上传至 Foundry 之前执行基本的文件转换（例如压缩或解密）。同步过程中，选定的待接入文件将根据所选转换器进行相应修改。

与其使用数据连接转换器，我们建议在 Foundry 中使用《管道构建器》和《代码仓库》执行数据转换，以便利用数据溯源和分支功能的优势。

数据连接支持以下转换器：

  • 使用 Gzip 压缩
  • 合并多个文件
    • 将多个文件合并为单个文件。
  • 重命名文件
    • 将文件名中所有出现的指定子字符串替换为新子字符串。
    • 将 ^(.*/) 替换为 /，从而去除文件名中的目录路径。
  • 使用 PGP 解密
    • 对使用 PGP 加密的文件进行解密。
    • 要求智能体系统已配置 PGP 密钥。
    • 不适用于在 Foundry worker 上运行的同步任务。
  • 在文件名中追加时间戳
    • 为每个接入文件的文件名添加自定义格式的时间戳。

完成策略

完成策略目前处于《开发生命周期》的遗留阶段，预计不会有进一步的开发。现有同步任务中的完成策略为只读状态，且无法在新的同步任务中进行配置。我们建议使用下游的《外部转换》作为替代方案。

完成策略是一种机制，用于在文件成功批量同步到 Foundry 数据集后，尝试删除这些文件及其空的父目录。完成策略为只读状态，无法在新的同步任务中进行配置。现有的带有完成策略的同步任务将继续运行，但其配置无法再进行修改。

当数据从中间暂存区（如文件存储系统）同步，而非直接从外部记录系统同步时，此工作流可能会有用。但不推荐使用此工作流。Foundry 应直接连接到外部记录系统。如果 Foundry 本身是记录系统，数据应直接推送到 Foundry，而不是推送到中间暂存区。

完成策略的局限性与替代方案

完成策略存在若干重要的局限性与注意事项。下文描述了这些局限性及其潜在的缓解措施或替代方案。

完成策略支持

完成策略目前处于《开发生命周期》的遗留阶段，预计不会有进一步的开发。现有同步任务中的完成策略为只读状态，且无法在新的同步任务中进行配置。完成策略仅适用于使用智能体工作节点的数据源。我们建议将完成策略提供的功能作为下游转换来实现。

举个例子，假设您与某个 S3 存储桶建立了《直接连接》，其中包含文件 foo.txt 和 bar.txt。您希望通过文件批量同步将它们复制到数据集中，然后从 S3 删除这些文件。请勿使用完成策略，而应采取以下操作：

  • 配置一个不带任何完成策略的批量同步任务，并安排其运行。
  • 编写一个下游《外部转换》作业，安排其在同步输出数据集更新时运行，并将同步的数据作为输入。
  • 在该外部转换中，编写 Python 转换代码，遍历同步数据集中出现的文件，并调用 S3 接口从存储桶中删除这些文件。

请注意，如果有任何删除调用失败，此方法支持重试，并且能够保证在尝试执行任何删除操作之前，数据已成功提交至 Foundry。此方法同样兼容增量文件批量同步。

完成策略采用尽力而为机制

对于使用完成策略的现有同步任务，请注意完成策略仅提供尽力而为的保证。这意味着它们无法保证数据一定会被有效删除。以下是一些可能导致完成策略失败的情况：

  1. 如果智能体工作进程在批量同步将数据提交至 Foundry 之后、但在执行完成策略之前发生崩溃或重启，完成策略将不会重试。
  2. 如果用于连接的凭据没有写入权限，批量同步可能会成功读取数据并提交至 Foundry，但无法执行完成策略指定的删除操作。

通常情况下，我们建议尽可能使用完成策略的替代方案。目前已不再支持自定义完成策略。

特别是对于目录数据源，如果您需要在数据摄入后确保文件被删除，请考虑改用通过 SSH 删除文件的外部转换。

优化基于文件的同步

警告

本指南专为设置新同步任务或排查缓慢、不稳定同步问题的用户提供建议。如果您的同步任务已经稳定运行，则无需进行任何操作。

将大量文件同步到单个数据集中可能会非常缓慢，且容易受到瞬时故障的影响。如同步失败行为中所述，任何环节出现故障都会中止整个事务，这意味着同步规模越大，丢失工作进度的风险就越高。

如果数据集随着时间推移不断增长，使用 SNAPSHOT 同步数据的时间也会增加，因为 SNAPSHOT 事务会将所有数据同步到 Foundry 中。相反，请使用配置为 APPEND 事务类型的同步来增量导入数据。由于您同步的是较小且独立的数据块，因此可以建立有效的检查点；发生同步故障时，只需极少的重复工作，而无需完全重新运行。此外，您的数据集同步速度也会更快，因为无需每次同步都上传所有数据。

配置增量 APPEND 同步

APPEND 事务需要额外配置才能成功运行。

默认情况下，同步到 Foundry 的文件不会进行过滤。但是，APPEND 同步需要使用过滤器来防止重复导入相同文件。我们建议使用 排除已同步的文件 和 限制文件数量 过滤器，来控制单次同步导入到 Foundry 的文件数量。最后，请对同步任务进行调度，以保持与源系统的数据同步。
