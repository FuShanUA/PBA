对于以下检查失败，您可以在“代码仓库”中的“检查”选项卡中查看检查日志。使用这些日志，您可以发现检查失败的原因。

新 Mamba 错误信息

Palantir 贡献 ↗ 通过提供更好的环境初始化错误信息格式化方式，为开源 Mamba 社区做出了贡献。截至 2023 年 2 月，Foundry 服务从更贴近被环境故障侵犯的依赖树的错误中受益。

依赖树

在下面的例子中：

packageA
   ├─ packageB
   └─ packageC
        └─ packageD

包 packageB 和 packageC 是 packageA 的 直接依赖项。packageD 是 packageC 的直接依赖项，但却是 packageA 的 传递依赖项。虽然 packageA 对 packageD 没有直接约束，但是 packageA 对 packageC 的直接要求间接地对 packageD 强加了约束。以下是该概念的一个真实示例：

statsmodels
├─ numpy
├─ scipy
├─ matplotlib
│  │  ├─ libpng
│  │  ├─ setuptools
│  │  ├─ cycler
│  │  ├─ dateutil
│  │  └─ kiwisolver

虽然可能一开始并不明显，但statsmodels确实对libpng施加了约束，这是通过对其matplotlib施加约束而间接实现的。

直接冲突

当在同一个环境中请求同一包的不同版本时，就会发生直接冲突。想象一下，请求一个同时包含python 3.7和python 3.8的简单环境。这是一个直接冲突，会导致环境失败。新的错误信息将提供以下信息：

  Could not solve for environment specs
  The following packages are incompatible
  ├─ python 3.7**  is requested and can be installed;
  └─ python 3.8**  is uninstallable because it conflicts with any installable
   versions previously reported.

上面的信息正确解释了python 3.7确实可以安装。但是，如果安装该版本，任何尝试安装其他不同版本的操作都会导致冲突。可以通过从环境中移除python的任一版本约束来解决此环境问题。

然而，直接冲突非常少见，与由依赖项和传递依赖项引起的冲突相比。

直接依赖冲突

直接依赖冲突

NumPy 文档 ↗ 特别指出，numpy >=1.22.0 需要 python >=3.8。因此，尝试创建同时请求 python 3.7.* 和 numpy 1.22.0 的环境会导致 直接依赖冲突。将出现以下错误信息：

Could not solve for environment specs
The following packages are incompatible
├─ numpy 1.22.0**  is installable with the potential options
│  ├─ numpy 1.22.0 would require
│  │  ├─ python >=3.9,<3.10.0a0 , which can be installed;
│  │  └─ python_abi 3.9.* *_cp39, which can be installed;
│  ├─ numpy 1.22.0 would require
│  │  ├─ python >=3.10,<3.11.0a0 , which can be installed;
│  │  └─ python_abi 3.10.* *_cp310, which can be installed;
│  └─ numpy 1.22.0 would require
│     ├─ python >=3.8,<3.9.0a0 , which can be installed;
│     └─ python_abi 3.8.* *_cp38, which can be installed;
└─ python 3.7**  is uninstallable because there are no viable options
   ├─ python [3.7.10|3.7.11|...|3.7.9] conflicts with any installable versions previously reported;
   ├─ python [3.7.0|3.7.1|3.7.2|3.7.3|3.7.6] would require
   │  └─ python_abi * *_cp37m, which conflicts with any installable versions previously reported;
   └─ python [3.7.10|3.7.12|...|3.7.9] would require
      └─ python_abi 3.7.* *_cp37m, which conflicts with any installable versions previously reported.

此错误信息表明 numpy 1.22.0 可以与 Python 3.8、3.9 或 3.10 一起安装，这与请求的 3.7.* 版本冲突。可以通过放松对 Python 或 NumPy 的约束来解决此环境问题。例如，python 3.8.* 和 numpy 1.22.0，或者 python 3.7.* 和 numpy 1.* 都会生成成功的环境。

出于排查环境的目的，您可以假设 python 和 python_abi 版本是配套的。您可以通过调整您的 python_abi 版本来更改您的 python 版本，而不是在环境中指定 python_abi。

传递依赖冲突

类似地，由于传递依赖，可能会出现包冲突。由于传递依赖的性质，仅请求少量的包可能导致数百个约束被添加到环境求解操作中。请考虑以下statsmodels包：

statsmodels
├─ numpy
├─ scipy
├─ matplotlib
│  │  ├─ libpng
│  │  ├─ setuptools
│  │  ├─ cycler
│  │  ├─ dateutil
│  │  └─ kiwisolver

请求特定的statsmodels版本将对setuptools允许的版本施加限制，因为该包依赖于matplotlib。因此，如果一个环境请求了statsmodels和setuptools不兼容的版本，就会发生传递依赖冲突，这是由于statsmodels本身对setuptools施加了限制。

例如，考虑以下请求了huggingface-adapter 0.1.1*和transforms 1.645.0的环境：

The following packages are incompatible
  ├─ huggingface-adapter 0.1.1** * is installable and it requires
  │  └─ palantir_models >=0.551.0 *, which requires
  │     └─ pyspark-src >=3.2.1,<3.3.0a0 *, which can be installed;
  └─ transforms 1.645.0 * is uninstallable because it requires
     └─ pyspark-src 3.2.1_palantir.36 *, which conflicts with any installable versions previously reported

起初可能不容易看出这两个包为何不兼容。错误信息有助于识别问题来源于huggingface-adapter的传递依赖与transforms直接依赖于pyspark-src之间的冲突。可以通过放松对transforms或huggingface-adapter的约束来解决此环境，这样 Mamba 就可以找到一对版本，使pyspark-src的要求相容。

解释新的错误信息

以下是新 mamba 错误消息可能提供的所有可能表述，如何解读它们，以及如何修复它们的一些指导：

  • The following packages are incompatible 或 The following package could not be installed：这些中的任何一个通常是详细错误消息的第一句话。这表明整体问题是由不兼容版本引起的包 冲突，还是必要的包安装本身的问题。
    • 如何修复：请直接在该陈述下方阅读相关信息。
  • can be installed：这句话通常出现在树形依赖的顶部，意味着该包本身可以安装而无需额外约束。然而，假设安装了这个特定版本，由于其他包依赖与该已安装版本冲突，可能会出现一组冲突。例如，python 3.7** is requested and can be installed 意味着可以在环境中安装 Python 3.7。此陈述下方列出的每一种其他冲突都假设环境已安装 Python 3.7。
  • no viable options: 可安装的版本中没有符合所请求环境中其他规范施加的约束条件的。例如，消息 python 3.7** is uninstallable because there are no viable options 表示如果其他包或约束没有特别要求不同的 Python 版本，则可以安装 python 3.7.* 的版本。
    • 如何解决：放宽导致此包外允许版本范围的包的约束。
  • does not exist, perhaps a (typo or a) missing channel: 未找到要安装的包。这可能意味着该包被错误地指定，或者配置的通道（设置 > 库 在代码仓库中，或 环境配置 在代码工作区）中没有该包。例如，请求一个包含 python 3.423.*（不存在的版本）和 pthon 3.8.*（明显的拼写错误）的环境会导致以下错误信息：
 Could not solve for environment specs
  The following packages are incompatible
  ├─ pthon 3.8**  does not exist (perhaps a typo or a missing channel);
  └─ python 3.423** does not exist (perhaps a typo or a missing channel).

  • 如何解决：确保包中没有拼写错误，或者确保所有请求的包都可用于该环境。有关如何在 Foundry 中查找包存在的建议，请参阅 发现 Python 库。

  • is installable and it requires：该包及其附加版本可以安装到环境中，但会向环境中引入一组额外的约束。

  • is uninstallable because it requires：无法安装该包及其附加版本，因为其某些依赖项在环境中产生冲突。

    • 如何修复：检查消息下方列出的依赖项的有问题的要求，并放宽它们的约束，或者此特定包的约束。

  • is installable with the potential options：该包有一系列可用的、不冲突的版本可以在环境中安装。选择其中一个版本可能会导致进一步的约束。

  • conflicts with any installable versions previously reported：通常与上述行中提到的版本相对比，这表示该包的某个已假定的版本已被确定在其他地方，因此该版本将无法满足。

    • 如何修复：确保同一环境不会直接或间接请求同一包的两个不同版本。

  • is missing on the system：这特别指的是缺少 虚拟包 ↗ . 一些包只能在特定的OS环境中运行。例如，cudatoolkit ↗ 包需要特定版本的 __cuda，这是一个系统级功能，必须作为虚拟包存在于环境中，以确保该包可以在现有架构上运行。

  • which cannot be installed (as previously explained)：由于之前已经描述过的冲突，或者因为另一个包包含之前描述的完全相同的约束，该包无法安装。

为什么取消固定版本不能解决解析失败？

即使约束完全放松，所需的包或包版本可能仍然：

  • 不可用。您可以使用 包标签 检查可用的包和版本。如有需要，您可以通过联系您的Palantir代表请求添加这些包。
  • 不兼容。即使我们能够访问所有可能发布的版本，这些包定义可能永远不兼容。例如，您依赖的某个Conda包可能已升级（查看升级PR）到一个新版本，但该版本存在错误。
  • 与您meta.yml中的Python版本不兼容。

为了检查这种情况，您可以将成功检查的Conda锁定文件与失败检查的进行比较。要访问Conda锁定文件，请在设置齿轮中选择Show hidden files and folders选项，并导航到transforms-python/conda-versions.run.linux-64.lock。在meta.yaml文件中，将成功运行和失败运行之间发生变化的库版本固定为不同的版本。

请求包没有更改的失败

这可能发生的两个主要原因如下：

  1. 由于您的转换可能依赖于外部包（参见公开管理的Conda通道），不幸的是，如果上游发生了问题，它可能会出现故障。
  2. 升级PR合并将触发环境的重新解析，当Checks运行时，以及在极少数情况下可能导致包解析失败，特别是如果您有一个过度约束的环境。这是因为升级PR可能会引入新的依赖项，导致环境需要重新计算。您可以通过测试来确认升级PR是否是原因，即如果回退升级PR所应用的提交，检查是否通过。然后您可以按照上面的章节重新应用升级PR。

如果没有对您的meta.yml进行更改，调试缓慢问题

在代码仓库中：

  1. 升级您的分支（查看手动分支升级以获取更多信息）。
  2. 如果您仍然遇到缓慢问题，并且您进行了超出@transform_df标签之外的代码更改 ，那么这段代码可能减慢了检查速度。请评估此代码的性能（如适用）。

如果您的 meta.yml 发生更改，则作业可能会超时

如果您在长时间运行的任务中收到内存不足（OOM）错误，这可能是由于包版本不兼容导致无法解决的环境问题。为了解决这个问题，请首先升级您的分支，然后继续执行以下调试步骤。

出现 Conda 错误的构建失败

Copied!
1CondaEnvironmentSetupError

如果您遇到构建错误，解决步骤如下：

  1. 检查您的 驱动程序日志 中的错误。
  2. 如果您对您的代码/meta.yml 进行了更改（要观察这一点，请使用 作业比较） 工具）然后恢复它们，看看是否可以修复构建。如果恢复的meta.yml更改解决了该问题，那么很可能存在包冲突，这可以通过上述所述方法进行调试。如果没有更改您的meta.yml，则检查构建的Python模块版本（如以下步骤中所述）。如果这不能解决构建问题，可能是由于分支升级或底层升级（查看升级PRs）导致了无法解决的环境问题。在这种情况下，您应检查上述提到的调试步骤。
  3. 如果您没有做出任何更改，与失败的构建相比，是否有不同的Python模块版本运行了成功的构建？检查 "Infra details"、"Environment" 然后 "SparkModuleVersion"。关于恢复到此版本，请联系您的Palantir代表。

多下载失败

Copied!
1RuntimeError: Multi-download failed

此错误意味着您无法从用于下载包的 artifacts channel（构件通道） 访问该通道，或者该构件在注册中不可用。实际的失败原因可以在 驱动程序日志 中看到。

如果您正在尝试对该仓库进行模板化处理，请转到您的 Conda channels（Conda 渠道） 列表，并检查列出的渠道是否有任何警告。如果某个列出的渠道有警告，请按照以下步骤操作：

  1. 移除损坏的渠道（如适用，请向 Palantir 代表请求权限）。
  2. 重新触发检查。
  3. 重新触发构建。

带有入口点错误的构建失败

Copied!
1transforms._errors.EntryPointError: "Key {name} was not found, please check your repo's meta.yaml and setup.py files"

这个错误意味着缺少触发构建所需的根文件。您可能可以使用数据集预览，但构建将会失败。

要调试此问题，请检查meta.yaml或setup.py中是否缺少任何关键信息。作为参考，您可以创建一个新的Python代码仓库，并检查新仓库中的meta.yaml和setup.py文件。

在向meta.yaml和/或setup.py添加任何缺失的信息后，提交更改，等待检查成功，然后重新触发构建。

同时需要Conda包和JAR的包

某些包需要同时拥有Conda包和JAR才能可用。一个常见的例子是graphframes包（其中Conda包包含Python API，而JAR包含实际实现）。如果您只添加了Conda包，但没有添加必要的JAR，可能会遇到以下错误：

o257.loadClass.: java.lang.ClassNotFoundException:<Class>

其他情况下，您可能会遇到此错误：

避免依赖冲突的最佳实践

从上面提到的错误来看，最常见的错误原因是依赖冲突。为了减少依赖冲突，请遵循这些最佳实践。

避免依赖冲突的最佳实践：Python

为 Python 保持 主版本.次版本 版本控制。

  • 在 代码仓库 中，Python 3.9.*、3.10.* 和 3.11.* 是可用的。建议您不要固定 Python 版本，只需指定包名称 python 即可。但是，如果您的代码需要在特定版本的 Python 上运行，则所选版本应在构建和运行部分都进行固定。Python 3.8.* 已经被弃用；它仍然可以使用，但将在 2025 年 1 月之后不再支持。Python 3.6.* 和 3.7.* 不再受支持。

确保构建和运行部分中的 Python 依赖项相同。Python 依赖项之间的不匹配可能导致不良结果和失败。

范围如 python >=3.9 或 python >3.9,<=3.10.11 不支持 Python 版本。如果使用了不支持的 Python 版本，我们将默认使用 Python 3.6.*。请注意，Python 3.6.* 已弃用，因此请确保在 meta.yaml 中为支持的 Python 版本设置了有效的版本号。

  • 在 代码笔记本 中，您可以通过将 自动 更改为所需的版本控制来切换版本控制。

Python 版本支持

从 Python 3.9 开始，Foundry 将遵循 Python 生命周期 所定义的时间表，这意味着如果某个 Python 版本被声明为生命周期结束，该版本将不再受到平台支持。有关更多详细信息，请查看 Python 版本 页面。
