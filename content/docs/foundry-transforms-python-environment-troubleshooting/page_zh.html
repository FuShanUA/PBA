新版 Mamba 错误消息

Palantir 通过优化环境初始化错误消息的格式，向开源 Mamba 社区 做出了贡献 ↗。自 2023 年 2 月起，Foundry 服务所显示的错误信息已能更准确地反映出环境失败所破坏的依赖树状态。

依赖树

在以下示例中：

packageA
   ├─ packageB
   └─ packageC
        └─ packageD

包 packageB 和 packageC 是 packageA 的直接依赖。packageD 是 packageC 的直接依赖，但它是 packageA 的传递依赖。虽然 packageA 没有直接对 packageD 施加约束，但 packageA 对 packageC 的直接要求间接地对 packageD 强制施加了约束。以下是此类概念的真实示例：

statsmodels
├─ numpy
├─ scipy
├─ matplotlib
│  │  ├─ libpng
│  │  ├─ setuptools
│  │  ├─ cycler
│  │  ├─ dateutil
│  │  └─ kiwisolver

虽然乍看之下可能不明显，但 statsmodels 确实通过对 matplotlib 的约束，以传递的方式对 libpng 施加了约束。

直接冲突

当在同一环境中请求同一包的不同版本时，就会发生直接冲突。假设要请求一个同时包含 python 3.7 和 python 3.8 的简单环境。这就是会导致环境创建失败的直接冲突。新的错误消息会提供以下信息：

  Could not solve for environment specs
  The following packages are incompatible
  ├─ python 3.7**  is requested and can be installed;
  └─ python 3.8**  is uninstallable because it conflicts with any installable
   versions previously reported.

上述消息正确地解释了 python 3.7 确实可以安装。然而，如果安装了该版本，任何安装其他不同版本的尝试都会引发冲突。要解决此环境的问题，只需从中移除任一版本的 python 约束即可。

不过，与依赖和传递依赖引发的冲突相比，直接冲突相当罕见。

直接依赖冲突

《NumPy 文档》 ↗ 明确指出 numpy >=1.22.0 需要 python >=3.8。因此，如果尝试创建一个同时请求 python 3.7.* 和 numpy 1.22.0 的环境，将导致直接依赖冲突。随后会产生以下错误消息：

这条错误信息表明，numpy 1.22.0 可以搭配 Python 3.8、3.9 或 3.10 进行安装，但这与所要求的 3.7.* 版本产生了冲突。要解决此环境问题，可以放宽对 Python 或 NumPy 的限制。例如，python 3.8.* 搭配 numpy 1.22.0，或者 python 3.7.* 搭配 numpy 1.*，都可以成功构建环境。

在排查环境问题时，你可以假设 python 和 python_abi 的版本是直接绑定的。你可以通过调整 python 版本来改变 python_abi 版本，而无需在环境中显式指定 python_abi。

传递依赖冲突

类似地，软件包冲突也可能由传递依赖引起。由于传递依赖的特性，仅请求少数几个软件包就可能在环境求解操作中引入数百个约束条件。以 statsmodels 软件包为例：

statsmodels
├─ numpy
├─ scipy
├─ matplotlib
│  │  ├─ libpng
│  │  ├─ setuptools
│  │  ├─ cycler
│  │  ├─ dateutil
│  │  └─ kiwisolver

由于 statsmodels 依赖于 matplotlib，因此请求特定版本的 statsmodels 会对 setuptools 的可用版本施加限制。结果是，如果环境请求了不兼容的 statsmodels 和 setuptools 版本，就会因为 statsmodels 自身对 setuptools 的限制而引发传递依赖冲突。

例如，请看以下请求安装 huggingface-adapter 0.1.1* 和 transforms 1.645.0 的环境：

The following packages are incompatible
  ├─ huggingface-adapter 0.1.1** * is installable and it requires
  │  └─ palantir_models >=0.551.0 *, which requires
  │     └─ pyspark-src >=3.2.1,<3.3.0a0 *, which can be installed;
  └─ transforms 1.645.0 * is uninstallable because it requires
     └─ pyspark-src 3.2.1_palantir.36 *, which conflicts with any installable versions previously reported

乍一看，可能不太容易理解这两个包为何不兼容。错误信息有助于我们识别出，问题出在 huggingface-adapter 的传递依赖与 transforms 对 pyspark-src 的直接依赖发生了冲突。要解决此环境中的问题，可以放宽 transforms 或 huggingface-adapter 的限制条件，这样 Mamba 就能找出一组版本，使得它们对 pyspark-src 的要求能够兼容。

解读新的错误信息

下方列出了新的 Mamba 错误信息中所有可能出现的提示语、其含义解读，以及一些修复建议：

  • The following packages are incompatible 或 The following package could not be installed：这两者之一通常是详细错误信息的第一句话。它表明整体问题是由版本不兼容导致的包冲突，还是在安装所需包本身时出现了问题。
    • 修复方法：直接阅读该语句下方的必要信息。
  • can be installed：该语句通常出现在依赖树顶部，表示该包本身可以在无额外限制条件的情况下安装。然而，假设安装了该特定版本，由于其他包的依赖与已安装版本发生冲突，可能会引发一系列冲突。例如，python 3.7** is requested and can be installed 表示可以在该环境中安装 Python 3.7。在此语句下方列出的所有其他冲突，均假定环境中已安装了 Python 3.7。
  • no viable options：在所请求环境的其他规范施加的限制下，没有任何可安装的版本能够满足要求。例如，信息 python 3.7** is uninstallable because there are no viable options 表示，如果不是有其他包或限制条件明确要求使用不同的 Python 版本，原本是可以安装 python 3.7.* 各版本的。
    • 修复方法：放宽导致超出该包可行版本范围的依赖包约束。
  • does not exist, perhaps a (typo or a) missing channel：未找到要安装的包。这通常意味着包名指定有误，或者已配置的渠道（代码仓库中的 Settings > Libraries，或代码工作簿中的环境配置）均未包含该包。例如，请求配置一个包含 python 3.423.*（不存在的版本）和 pthon 3.8.*（明显的拼写错误）的环境，将会触发以下报错信息：
 Could not solve for environment specs
  The following packages are incompatible
  ├─ pthon 3.8**  does not exist (perhaps a typo or a missing channel);
  └─ python 3.423** does not exist (perhaps a typo or a missing channel).

  • 修复方法：确保包名没有拼写错误，或确认所有请求的包在该环境中均可用。关于如何在 Foundry 中查找包是否存在，请参阅《发现 Python 库》获取指导。

  • is installable and it requires：该包及其指定版本可以安装到环境中，但会给环境引入一组额外的约束条件。

  • is uninstallable because it requires：该包及其指定版本无法安装，因为其部分依赖项在环境中引发了冲突。

    • 修复方法：检查该报错信息下方列出的依赖项中存在问题的要求，放宽这些依赖项的约束，或放宽该特定包的约束。

  • is installable with the potential options：该包有一系列可用且不冲突的版本可供安装到环境中。选择其中某个版本可能会引入进一步的约束。

  • conflicts with any installable versions previously reported：通常与上文行中提及的版本相对照，这表示在别处已确定了该包一个被假定为可安装的版本，因此当前这个版本将无法满足条件。

    • 修复方法：确保同一环境不会直接或间接地请求同一包的两个不同版本。

  • is missing on the system：这特指缺少虚拟包 ↗。某些包只能在特定的操作系统环境中运行。例如，cudatoolkit ↗包要求环境中作为虚拟包存在特定版本的__cuda（一种系统级特性），以确保该包能够在现有架构上运行。

  • which cannot be installed (as previously explained)：无法安装该包，原因要么是前文已描述过的冲突，要么是另一个包包含了前文已描述过的完全相同的约束条件。

避免依赖冲突的最佳实践

从上述错误来看，最频繁的报错原因是依赖冲突。为减少依赖冲突，请遵循以下最佳实践。

避免依赖冲突的最佳实践：Python

请为 Python 坚持 主版本号.次版本号 的版本控制方式。

  • 在 Code Repositories 中，提供了 Python 3.9.*、3.10.* 和 3.11.* 版本。建议您不锁定 Python 版本，仅指定包名 python。但是，如果您的代码需要在特定 Python 版本上运行，则应在构建和运行部分同时锁定所选版本。Python 3.8.* 现已弃用；仍可继续使用，但在 2025 年 1 月之后将不再提供支持。Python 3.6.* 和 3.7.* 已不再支持。

请确保构建和运行部分中的 Python 依赖项完全一致。Python 依赖项之间的不匹配可能导致非预期结果和失败。

Python 版本不支持诸如 python >=3.9 或 python >3.9,<=3.10.11 之类的范围表达式。如果使用了不支持的 Python 版本，我们将默认使用 Python 3.6.*。请注意，Python 3.6.* 已弃用，因此请确保您的 meta.yaml 中针对受支持的 Python 版本进行了有效锁定。

  • 在 Code Workbook 中，您可以通过将 automatic 更改为所需版本来切换版本控制。

Python 版本支持

从 Python 3.9 开始，Foundry 将遵循 Python End Of Life ↗ 规定的时间线，这意味着如果某个 Python 版本被宣布停止维护（EOL），平台将不再对其提供支持。请查看 Python 版本 页面以获取更多详细信息。

避免依赖冲突的最佳实践：Python 以外的包

避免显式固定版本号，因为这会导致依赖冲突。即便是主版本号.次版本号也可能引发冲突。

请注意，在代码工作簿（Code Workbook）中，您可以为 pandas、matplotlib 和 numpy 添加最低版本限制，因为在自动设置下，代码工作簿默认使用 pandas=0、matplotlib=2 和 numpy<1.20。
