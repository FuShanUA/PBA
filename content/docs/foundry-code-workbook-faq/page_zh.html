    • 我可以轻松地将我所有的工作从一个工作簿移动到另一个工作簿吗？

    是的。您可以在工作簿之间复制粘贴节点（工作簿图中的元素）。

    1. 在选择您想要从一个工作簿中复制的节点时按住 Ctrl 。不支持复制粘贴Contour板节点。
    2. 使用右键上下文菜单复制所选节点。
    3. 选择目标工作簿的图表，并使用组合键 Cmd+V (macOS) 或 Ctrl+V (Windows) 粘贴节点。新节点将被导入到您的新工作簿中。

    我可以将转换节点设置为变量吗？

    不可以。代码工作簿图中的每个转换节点必须返回一个表格或数据框（如二维数据结构，包含列）。

    返回顶部

    为什么每个转换节点需要返回一个表格或数据框？

    Artifact在Foundry中的基本单位是数据集。每个转换节点都需要返回一个表或DataFrame（例如，一个二维数据结构，包含列），以便每个节点可以注册为一个Foundry数据集，因此在整个Foundry中都可用。此外，Code Workbook中的表或DataFrame必须带有有效的模式，这样它们将生成数据集（例如，至少有一个列存在，列名不重复，列名不包含无效字符等）。

    返回顶部

    Code Workbook能否处理合并和合并冲突？

    可以。有关更多信息，请查看关于分支与合并的文档。

    初始化我的环境需要多长时间？

    初始化我的环境需要多长时间？

    Code Workbook 的默认配置平均初始化时间为 3-5 分钟。但是，如果您在 Code Workbook 配置中添加了其他包，初始化时间可能会根据这些包的复杂性和相互依赖性而有所不同，通常在 20-30 分钟之间。随着环境中包的数量增加，初始化时间会显著增加。

    初始化速度慢通常表明环境定义过大或过于复杂。环境中的包数量与初始化时间呈超线性增长关系，因此您可能需要简化任何自定义环境。在某些情况下，Code Workbook 可能会预先初始化常用环境以加快初始化速度。如果您基于默认配置创建了自定义环境，较慢的初始化时间可能是由于缺乏预初始化。了解更多关于 优化自定义环境的初始化时间 的信息

    如果浏览器标签页超过 30 分钟未活动，由于不活动，环境可能会丢失。

    如何恢复我之前从代码笔记本中删除的代码？

    如果这些转换是构建在数据集中的，您可以使用生成的数据集预览的比较功能来查看当时代码。从那里您可以复制粘贴相关的转换。不幸的是，如果这段代码是在中间转换中，就无法恢复。

    代码笔记本比代码仓库更具有迭代性，后者具有完整的git提交和发布功能。如果您还有其他分支可用，我们建议您检查它们以查找已删除的转换。

    Spark环境和自定义库

    无法找到所需的Python库

    默认情况下，任何在 Conda Forge ↗ 中可用的Python包都可以添加到您工作簿的自定义环境中。如果该Python库包含在Conda Forge中，则您可以自定义环境以直接包含它。

    要进行故障排除，请执行以下步骤：

    1. 在右上角选择 环境 > 自定义环境。
    2. 搜索所需的库，并添加 自动（这将接受升级，但如果模块升级可能会导致意外中断）或此库的特定版本。
    3. 选择 更新Spark环境。

    请注意，环境可能需要一些时间才能重新加载，一般来说，使用自定义环境会比使用标准环境更慢，因为标准环境池会被“预热”，而每个自定义环境都是从零开始创建的。

    返回顶部

    导入 Python 库时遇到问题

    有时可能需要使用 Code Workbook 中尚未提供的库。可以将这些库添加到您的可用列表中，但这需要一些手动操作。

    要进行故障排除，请执行以下步骤：

    1. 确认 Python 库不在工作簿自定义 Spark 环境列表中。
    2. 如果不存在，则检查是否有其他可用库可以提供相同的功能。
    3. 如果必须包含此库，请联系 Palantir 支持。
    4. 一旦库可用，您可以参考之前的故障排除步骤来添加库。

    返回顶部

    无法创建环境

    我正在尝试更新我的工作簿环境，它显示“等待 Spark / 初始化环境”一段时间后，然后出现“无法创建环境”的错误信息。

    要进行故障排除，请执行以下步骤：

    1. 确认是否对工作簿的Spark环境进行了任何自定义更新。如果有的话，您的调查应集中在任何新增的自定义库上。
    2. 确保没有模块被固定到特定版本。通常建议使用模块的最新版本。
    3. 移除任何自定义库，看看Spark环境是否可以加载。
    4. 如果某个特定库阻止了Spark环境的加载，请升级到Palantir支持。

    重试器错误

    运行导入包或任何基本命令时返回以下错误：“com.github.rholder.retry.RetryException: 在3次尝试后，重试未能成功完成。 at com.github.rholder.retry.Retryer.call(Retryer.java:174)”。在使用Pandas的工作簿时，Code Workbook应用程序仍会在您的转换之前将Spark数据框转换为Pandas数据框，这会显著增加驱动程序的内存消耗，并可能导致内存不足（OOM）。

    要进行故障排除，请执行以下步骤：

    1. 确认您的转换单元格是否使用Pandas，并且此问题是否出现在专门运行Pandas的代码中。
    2. 如果此计算可以在Spark中进行，我们强烈建议重新配置您的代码以使用Spark，因为它具有并行性且更可扩展。
    3. 作为最后的手段，当此计算必须在驱动程序上执行时，我们建议使用一个增加驱动程序内存的配置文件。

    保存为数据集时失败

    此问题可能由多种原因引起，但最常见的状况是返回一个包含有效模式的表或数据框。如果以下任何步骤都无法帮助您确定您看到的具体错误，请参考我们关于 构建和检查错误 的指南。

    要进行故障排除，请执行以下步骤：

    1. 确保转换节点返回的是表或数据框。
    2. 确认该表或数据框符合作为数据集写入Foundry的基本要求：例如：
      1. 至少有一列。
      2. 所有列名都是有效的（如它们不包含空格或无效字符）。
      3. 没有重复的列名。
    3. 如果您确信此转换节点应该能够被写入为数据集，则尝试运行另一个转换节点或构建一个新的、非常简单的转换节点。
      1. 这可以帮助确定是本地问题（例如，特定转换节点出现意外情况）还是更系统性的问题（例如，转换节点普遍无法保存到Foundry）。
      2. 如果是本地问题，请继续调试用于创建此表/数据框的代码。
      3. 如果是系统性问题，请将问题升级到Palantir Support，并提供您调查中的所有相关信息。

    返回顶部

    在更新输入数据集预览后工作簿未自动重新计算

    当选择< strong >更新表预览< /strong >作为输入表时，仅更新输入数据集的视图，而Foundry中的底层数据集不会自动刷新。

    要进行故障排除，请执行以下步骤：

    1. 如果在为输入表选择< strong >更新表预览< /strong >后输入数据集没有刷新，那么：
      • 打开这些数据集，查看它们是否未能构建。
      • 如果输入的数据集未能构建，那么这将解释为什么您在工作簿中看不到任何更新的信息。请从这里继续调试。
      • 如果输入的数据集按预期构建，但您在代码工作簿中的预览与实际看到的内容不一致，那么：
        • 尝试从代码工作簿中删除输入表并重新添加。
        • 如果这不起作用，并且预览和实际表格之间仍然存在差异，请联系 Palantir 支持。
    2. 如果在选择更新表格预览时转换节点没有更新，这是预期的行为，因为这些节点不受此操作的影响。要更新代码工作簿中的所有转换节点，请选择run -> run all。这将在尊重构建依赖关系的同时运行代码工作簿中的所有转换节点。

    无法将我的分支合并到主分支

    当你发现自己无法将分支合并回主分支时，最常见的问题是master分支被保护了。可能存在合并冲突的问题，但这在另一部分有详细说明。

    要进行故障排除，请执行以下步骤：

    1. 确定该分支是否被保护。
    2. 如果该分支被保护，则确定所有者并请求他们合并你的分支。
      • 对于给定的工作区，在master上开启分支保护，并确保你希望限制合并的用户在工作区上只有compass:edit权限。
      • 内部，Code Workbook有四种与分支相关的权限级别：view、edit、maintain和manage。默认情况下，compass:read扩展为view，compass:edit扩展为edit，而compass:manage扩展为maintain和manage。
      • 创建一个分支并准备合并到父分支始终只需要edit权限。合并到受保护的分支需要maintain权限。更改分支保护设置需要manage权限。
      • 更多相关信息请参阅我们的 分支概述。
    3. 如果分支未受保护，那么您会遇到合并冲突，其中此代码为非加法代码，与 master 分支中的代码冲突。
      • 在这些情况下，请在将您的分支合并回 master 之前解决这些合并冲突。
      • 更多相关信息请参阅我们的 合并分支 文档。
    4. 如果 master 分支未受保护，并且没有合并冲突，但您仍然无法将您的分支合并到 master 中，请联系 Palantir 支持团队。

    加快单元格计算

    假设我有 (1000 行 * 30 列) + (100 万行 * 30 列) 的数据集输入，并且有一个包含大量窗口 / 列推导步骤的转换，如何才能使计算更快？

    要进行故障排查，请执行以下步骤：

    1. 重构

    对于实验或快速迭代，通常将代码重构为几个较小的步骤而不是一个大的步骤是一个好主意。

    这样，您首先计算上游单元格，将数据写回Foundry，并在后续步骤中使用这些预计算的数据。如果您一直重新计算而不改变这些早期步骤的逻辑，这会创建过多的工作。

    具体来说：

    workbook_1:
  cell_A:
    work_1 : input -> df_1
      (takes 4 iterations to get right): 4 * df_1
    work_2: df_1 -> df_2
      (takes 4 iterations to get right): 4 * df_2 + 4 * df_1
        = 4 df_2 + 4 df_1
    work_3: df_2 -> df_3
      (takes 4 iterations to get right): 4 * df_3 + 4 * df_2 + 4 * df_1
  total work:
    cell_A
     = work_1 + work_2 + work_3
     = 4 * df_1 + (4 * df_2 + 4 * df_1) + (4 * df_3 + 4 * df_2 + 4 * df_1)
     = 12 * df_1 + 8 * df_2 + 4 * df_3

    相反，如果您将work_1和work_2写入各自的单元格，则您的工作将如下所示：

    workbook_2:
  cell_A:
    work_1: input -> df_1
      (takes 4 iterations to get right): 4 * df_1
  cell_B:
    work_2: df_1 -> df_2
      (takes 4 iterations to get right): 4 * df_2
  cell_C:
    work:3: df_2 -> df_3
      (takes 4 iterations to get right): 4 * df_3
  total_work:
    cell_A + cell_B + cell_C
       = work_1 + work_2 + work_3
       = 4 * df_1 + 4 * df_2 + 4 * df_3

    假设df_1、df_2和df_3的计算成本相同，workbook_1.total_work = 24 * df_1，而workbook_2.total_work = 12 * df_1，因此您可以期望迭代速度提高约两倍。

    2. 缓存

    对于任何“小”数据集，您应该通过选择工作簿，然后选择操作 > 缓存来缓存它们。

    这会将行保留在内存中，供您的工作簿使用，而无需从写回的数据集中获取。“小”是根据多个因素做出的主观判断，但代码工作簿在尝试缓存时表现良好，并会在它太大时提醒您。

    3. 函数调用

    你应该尽可能使用原生的 PySpark 方法，而不要直接在数据上使用 Python 方法（例如循环遍历单个行，执行 UDF）。PySpark 方法调用底层用 Scala 编写的 Spark 方法，并直接对数据进行操作，而不是 Python 运行时；如果你只是将 Python 作为与该系统交互的层，而不是作为与数据交互的系统，那么你将获得 Spark 自身的所有性能优势。

    4. 降采样

    如果你能生成一个大型输入数据集的准确样本，这可以作为你转换操作的模拟输入，直到你完善了逻辑并希望用完整数据集进行测试。

    在编写任何 PySpark 代码之前，考虑对超过一百万行的数据集进行降采样和缓存；你可能会更快地得到结果，而不会因为数据集太大而缓慢地发现语法错误。

    5. 总结

    一个良好的代码工作簿看起来如下：

    • 离散的代码块，执行你期望以后重复使用的特定材料化操作，但不需要反复重新计算。
    • 已下采样到“小”尺寸。
    • 缓存的“小”数据集，以便快速获取。
    • 仅使用原生 PySpark 代码，利用快速的底层 Spark 库。
