我能将所有工作轻松地从工作簿 A 转移到工作簿 B 吗？

可以。你可以在不同工作簿之间复制粘贴节点（工作簿图谱中的元素）。

  1. 按住 Ctrl 键并选择你想从工作簿中复制的节点。不支持复制粘贴等高线面板节点。
  2. 通过右键上下文菜单复制选中的节点。
  3. 选中目标工作簿的图谱，使用 Cmd+V (macOS) 或 Ctrl+V (Windows) 组合键粘贴节点。新节点将被导入到你的新工作簿中。

我可以将转换节点设置为变量吗？

不行。代码工作簿图谱上的每个转换节点都必须返回表格或数据框（例如带有列的二维数据结构）。

返回顶部

为什么每个转换节点都需要返回表格或数据框？

Foundry 中产物的基本单元是数据集。每个转换节点都需要返回表格或数据框（例如带有列的二维数据结构），以便每个节点都能注册为 Foundry 数据集，从而在整个 Foundry 的其余部分可用。此外，Code Workbook 中返回的表格或数据框必须具备有效的模式，以确保能够生成数据集（例如，至少存在一列，列名不重复，列名不包含无效字符等）。

可以。如需了解更多信息，请查阅关于《分支与合并》的文档。

初始化我的环境需要多长时间？

Code Workbook 的默认配置平均初始化时间为 3 到 5 分钟。但是，如果你在 Code Workbook 配置文件中添加了额外的包，初始化时间可能在 20 到 30 分钟之间，具体取决于这些包的复杂性和相互依赖关系。随着环境中包数量的增加，初始化时间往往会显著增加。

初始化缓慢通常表明环境定义过大或过于复杂。初始化时间往往会随着环境中包的数量超线性增长，因此你可能需要简化任何自定义环境。在某些情况下，Code Workbook 可能会预初始化常用环境以加快初始化速度。如果你基于默认配置文件创建了自定义环境，较慢的初始化时间可能是由于缺少预初始化所致。了解更多关于《优化自定义环境的初始化时间》的信息

如果浏览器标签页超过 30 分钟处于非活动状态，环境可能会因闲置而丢失。

如何恢复之前从代码工作簿中删除的代码？

如果这些转换已构建到数据集中，你可以利用生成数据集预览的比较功能查看当时的代码，进而复制粘贴相关的转换。遗憾的是，如果该代码位于中间转换中，则无法恢复。

相比于代码仓库，代码工作簿是一个更具迭代性的平台，前者具备完整的 git 提交与发布功能。如果你还有其他可用分支，建议检查这些分支中是否存在被删除的转换。

Spark 环境与自定义库

找不到所需的 Python 库

默认情况下，Conda Forge ↗ 中提供的任何 Python 包都可以添加到工作簿的自定义环境中。如果该 Python 库包含在 Conda Forge 中，你可以直接自定义环境将其纳入。

要进行排查，请执行以下步骤：

  1. 在右上角选择 Environment > Customize Environment。
  2. 搜索所需的库，然后添加 automatic（将进行升级，但如果模块升级可能会引发意外中断）或该库的特定版本。
  3. 选择 Update Spark environment。

请注意，重新加载环境可能需要一段时间。通常情况下，使用自定义环境会比使用预置环境慢，因为预置环境会维持一个“预热”池，而每个自定义环境都必须从头启动。

返回顶部

导入 Python 库时遇到问题

有时可能需要使用代码工作簿中尚不存在的库。可以将其添加到你的可用列表中，但这需要进行一些手动操作。

要进行排查，请执行以下步骤：

  1. 确认该 Python 库不在工作簿的自定义 Spark 环境列表中。
  2. 如果不存在，请评估是否有其他可用库能够提供相同的功能。
  3. 如果必须引入该库，请联系 Palantir 支持团队。
  4. 该库可用后，你可以参考之前的排查步骤来添加库。

返回顶部

创建环境失败

我正在尝试更新工作簿环境，系统长时间显示“Waiting for Spark / Initializing environment”，随后报错并提示“Failed to create environment”。

要进行排查，请执行以下步骤：

  1. 确认是否对工作簿的 Spark 环境进行过任何自定义更新。如果有，请重点排查添加的任何自定义库。
  2. 确保没有将任何模块锁定在特定版本。使用最新版本的模块通常是一种好习惯。
  3. 移除所有自定义库，查看 Spark 环境是否能加载。
  4. 如果确定是某个库导致 Spark 环境无法加载，请将问题上报给 Palantir 支持团队。

重试器错误

运行导入包或任何基本命令时返回以下错误：“com.github.rholder.retry.RetryException: Retrying failed to complete successfully after 3 attempts. at com.github.rholder.retry.Retryer.call(Retryer.java:174)”。在工作簿中使用 Pandas 时，代码工作簿应用在执行转换之前，仍会将 Spark 数据帧转换为 Pandas 数据帧，这会占用驱动程序大量内存，极有可能导致内存溢出（OOM）。

要进行排查，请执行以下步骤：

  1. 验证您的转换单元格是否正在使用 Pandas，并确认该问题是否在专门运行 Pandas 代码时出现。
  2. 如果此计算可以通过 Spark 实现，我们强烈建议您重新配置代码以使用 Spark，因为 Spark 支持并行计算且更具可扩展性。
  3. 作为最后手段，如果此计算必须在驱动器上执行，我们建议您使用增加驱动器内存的配置文件。

无法保存为数据集

导致此问题的原因有多种，但最常见的情况是返回的表或数据框包含无效的架构。如果以下步骤均无法帮助您识别所遇到的特定错误，请参阅我们关于《构建与检查错误》的指南。

若要排查此问题，请执行以下步骤：

  1. 确保转换节点返回的是表或数据框。
  2. 确认表或数据框符合作为数据集写入 Foundry 的基本要求：例如：
    1. 至少包含一列。
    2. 所有列名均有效（例如不包含空格或无效字符）。
    3. 不存在重复的列名。
  3. 如果您确信此转换节点应该能够作为数据集写出，请尝试运行另一个转换节点，或者构建一个新的、非常简单的转换节点。
    1. 这有助于确定问题是局部问题（例如该特定转换节点发生了意外情况）还是系统性问题（例如所有转换节点普遍无法保存至 Foundry）。
    2. 如果是局部问题，请继续调试用于创建此表或数据框的代码。
    3. 如果是系统性问题，请将问题升级上报至 Palantir 支持团队，并提供您调查中的所有相关信息。

更新输入数据集预览后，工作簿未自动重新计算

当选择对输入表执行更新表预览操作时，仅会更新输入数据集的视图，Foundry 中的底层数据集并不会自动刷新。

若要排查此问题，请执行以下步骤：

  1. 如果对输入表选择更新表预览后输入数据集未刷新，则：
    • 打开这些数据集，查看它们是否构建失败。
    • 如果输入数据集一直构建失败，这就可以解释为什么您在工作簿中没有看到任何更新的信息。请从这里继续进行调试。
    • 如果输入数据集已按预期构建，但您在代码工作簿中看到的预览与之不符，则：
      • 尝试从代码工作簿中删除该输入表，然后重新添加。
      • 如果此操作无效，且预览与实际表之间仍存在差异，请联系 Palantir 支持团队。
  2. 如果选择更新表预览后转换节点未更新，这是预期行为，因为转换节点不受此操作影响。要更新代码工作簿中的所有转换节点，请选择run -> run all。这将在遵循构建依赖关系的前提下，运行代码工作簿中的所有转换节点。

无法将分支合并至主分支

当您发现无法将分支合并回主分支时，最常见的问题是 master 分支受保护。可能存在合并冲突的问题，但这将在另一章节中讨论。

若要排查此问题，请执行以下步骤：

  1. 确定该分支是否受保护。
  2. 如果分支受保护，请找到数据主人并请求他们合并您的分支。
    • 对于给定的工作簿，在 master 分支上开启分支保护，并确保您希望限制合并分支的用户仅具有该工作簿的 compass:edit 权限。
    • 在内部，代码工作簿具有四个与分支相关的权限级别：view、edit、maintain 和 manage。默认情况下，compass:read 展开为 view，compass:edit 展开为 edit，compass:manage 展开为 maintain 和 manage。
    • 创建分支并准备合并至父分支始终只需要 edit 权限。合并至受保护分支需要 maintain 权限。更改分支保护设置需要 manage 权限。
    • 更多信息请参阅我们的《分支概览》文档。
  3. 如果分支未受保护，你会遇到合并冲突，即此处的非累加代码与 master 分支中的代码发生冲突。
    • 在这些情况下，将分支合并回 master 之前，请先解决这些合并冲突。
    • 更多信息请参阅我们的《合并分支》文档。
  4. 如果 master 分支未受保护，且不存在合并冲突，但你仍无法将分支合并到 master，请联系 Palantir 支持团队。

加快单元格计算速度

假设我的数据集输入为（1000 行 * 30 列）+（100 万行 * 30 列），并且转换操作包含大量窗口计算/列派生步骤，如何才能让计算运行得更快？

若要排查此问题，请执行以下步骤：

1. 重构

为了便于实验或快速迭代，通常建议将代码重构为多个较小的步骤，而不是放在单个大步骤中。

这样，你可以先计算上游单元格，将数据写回 Foundry，并在后续步骤中使用这些预计算的数据。如果在未改变这些早期步骤逻辑的情况下不断重新计算，会产生大量额外的工作。

具体而言：

workbook_1:
  cell_A:
    work_1 : input -> df_1
      (需要 4 次迭代才能完成): 4 * df_1
    work_2: df_1 -> df_2
      (需要 4 次迭代才能完成): 4 * df_2 + 4 * df_1
        = 4 df_2 + 4 df_1
    work_3: df_2 -> df_3
      (需要 4 次迭代才能完成): 4 * df_3 + 4 * df_2 + 4 * df_1
  total work:
    cell_A
     = work_1 + work_2 + work_3
     = 4 * df_1 + (4 * df_2 + 4 * df_1) + (4 * df_3 + 4 * df_2 + 4 * df_1)
     = 12 * df_1 + 8 * df_2 + 4 * df_3

相反，如果你将 work_1 和 work_2 写入各自独立的单元格中，你执行的工作将变为：

workbook_2:
  cell_A:
    work_1: input -> df_1
      (需要 4 次迭代才能完成): 4 * df_1
  cell_B:
    work_2: df_1 -> df_2
      (需要 4 次迭代才能完成): 4 * df_2
  cell_C:
    work:3: df_2 -> df_3
      (需要 4 次迭代才能完成): 4 * df_3
  total_work:
    cell_A + cell_B + cell_C
       = work_1 + work_2 + work_3
       = 4 * df_1 + 4 * df_2 + 4 * df_3

假设 df_1、df_2 和 df_3 的计算成本相同，workbook_1.total_work = 24 * df_1，而 workbook_2.total_work = 12 * df_1，因此你可以预期迭代速度将提升约 2 倍。

2. 缓存

对于任何“小型”数据集，你应该通过选择工作簿并选择 操作 > 缓存 来对其进行缓存。

这会将数据行保留在工作簿的内存中，无需从写回的数据集中重新读取。考虑到必须权衡的几个因素，“小型”只是一个主观判断，但代码工作簿在尝试缓存方面做得很好，如果数据集过大，它会向你发出警告。

3. 函数调用

你应尽可能坚持使用原生的 PySpark 方法，绝不要直接在数据上使用 Python 方法（例如遍历单行或执行 UDF）。PySpark 方法会调用用 Scala 编写的底层 Spark 方法，并直接针对数据运行，而不是通过 Python 运行时；如果你仅仅将 Python 作为与该系统交互的层，而不是让它成为直接处理数据的系统，你就能获得 Spark 本身的所有性能优势。

4. 降采样

如果你能从大型输入数据集中提取出准确的样本，就可以将其用作转换操作的模拟输入，直到你完善逻辑并打算针对完整数据集进行测试时为止。

在编写任何 PySpark 代码之前，请先考虑对超过一百万行的数据集进行降采样和缓存；这样或许能获得更快的周转时间，避免因数据集过大而缓慢排查语法错误。

5. 总结

优秀的代码工作簿应具备以下特征：

  • 包含离散的代码块，用于执行特定的物化操作，这些操作后续会重复使用，但无需反复重新计算。
  • 降采样至“小”规模。
  • 缓存“小”规模数据集，以实现极速读取。
  • 仅使用原生 PySpark 代码，充分利用底层快速的 Spark 库。
