可选数据持久化

对于工作簿中的每个转换，用户可以选择是否将结果保存为数据集。

选择是否保存为数据集

默认情况下，新的转换不会保存为数据集。如需保存，请使用逻辑窗格中的切换开关。

这些截图使用了来自《纽约出租车与轿车委员会 ↗》的开源数据。

要同时更改多个转换的持久化设置，请使用左侧的批量编辑器。

如果选择将转换从“不保存”更改为“保存”，它将重新链接到之前保存的数据集。如果之前保存的数据集不存在，则会创建一个新的数据集。

已保存的转换会用一条蓝色横条标识。

执行模型

运行某个节点时，该节点上游所有未持久化节点的逻辑也会一并运行。在下图中，运行“已保存转换 C”时，“未保存转换 A”的逻辑也会执行。

如果更改了“未保存转换 A”中的代码但未运行，然后运行“已保存转换 C”，那么“已保存转换 C”的结果将反映该逻辑变更。

运行“未保存转换 D”时，将从 Foundry 数据集中读入“已保存转换 C”，并执行“未保存转换 B”的逻辑。

假设我们切换“已保存转换 C”的设置，使其不再保存为数据集。当运行“未保存转换 D”时，所有三个上游转换的逻辑都会执行。

在这种情况下，当运行这一系列转换时——如果最终目标是在“未保存转换 D”中查看结果，则无需预览 3 个上游未保存的转换。通过运行“未保存转换 D”，将使用所有四个转换中的最新逻辑以及导入数据集的最新事务。

常见问题

何时应该将转换保存为数据集？

如果某个转换的计算量非常大，并且位于许多其他转换的上游，您可能希望将其保存为数据集，以避免性能下降。

如果您想在工作簿之外（例如，在另一个 Code Workbook 或 Contour 分析中）使用转换的结果，则应将该转换的结果保存为数据集。

如果某个转换以非确定性方式计算函数（例如，使用 row_number 函数或调用当前时间的函数），您应该将该数据集持久化到 Foundry。这样可以保证下游转换使用写入数据集的确切结果。

为工作设置检查点

如果工作簿包含未持久化节点的长链，建议定期通过持久化中间节点来为工作簿设置检查点。

何时应该预览节点？

通常，在创建一系列转换时应使用预览功能，以验证其正确性并预览结果。一旦一系列转换的代码定型，就没有太多理由再使用预览功能了。

为什么未保存的转换没有出现在数据血缘关系中？

代码工作簿中未保存的转换属于逻辑块，而非项目中的资源。在探索数据血缘关系时，数据集会展示其执行的全部代码，包括已保存转换上游任何未保存转换中的代码。

可选数据持久化对性能和资源使用有何影响？

代码工作簿最初采用的执行模型会将每个转换保存为 Foundry 数据集。目前的执行模型允许用户自行选择是否将转换保存为数据集，这会对性能和资源使用产生影响。

定时构建

假设在之前的执行模型中运行了一次定时构建。在新的执行模型中，如果所有转换均保持持久化，性能将与以前完全一致。

如果确定某些中间转换无需保存为 Foundry 数据集，并选择取消其持久化，流水线的运行速度将得到显著的性能提升。原因在于，这些节点的中间结果不再需要写入 Foundry，并且在某些情况下，这还能帮助 Spark 查询规划器进一步优化下游计算。

交互式使用

在交互式场景下，旧的执行模型会同时计算预览结果和写入结果。采用可选持久化后，未持久化的节点仅计算预览，而持久化的节点仅计算写入。因此，不会产生重复计算工作，在新的执行模型中运行相同工作簿所消耗的资源将少于以往的执行模型。

在交互式场景中，可选持久化对性能的影响更为微妙。对于包含计算量非常大的转换（例如大型连接操作）的工作流，建议将该连接操作持久化，以避免每次运行下游转换时都重新计算这一大型连接。

无论何种情况，只要决定取消某个转换的持久化，即可通过不向 Foundry 写入数据来节省存储空间。
