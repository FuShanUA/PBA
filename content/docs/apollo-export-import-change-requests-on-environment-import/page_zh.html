环境导入时的变更请求

当包含环境更改的包导入到中心节点（Hub）时，Apollo会创建一个变更请求，以在应用更改之前进行审查和批准。这确保了敏感的基础架构更改（如网络安全规则的修改）在生效前获得适当的批准。

当您导入包含环境更改的包时，Apollo不会立即应用所有更改。相反，Apollo会打开一个变更请求，其中包含该包中的所有环境更改。这包括配置更新、实体更改（如Kubernetes命名空间、网络安全规则和实例组）以及模块安装。

环境更新的变更请求将包中的所有更改合并为一个操作。要么所有更改都被批准并一起应用，要么都不应用。这可以防止只应用包中部分更改的情况，这可能导致环境处于不一致的状态。

请注意，Apollo会为包中的每个环境创建单独的变更请求。

自动批准

在导入时创建的大多数更改请求会自动获得批准，不需要人工干预。除非包中包含影响环境安全态势的资源更改，例如：

  • 网络安全性规则
  • 具有特权安全上下文的Kubernetes命名空间

包含这些更改的更改请求需要经过授权用户的手动批准后才能应用更改。

查看待审批项

当导入的更改需要审批时，环境页面上会出现一个横幅，表示有一个更改请求需要处理。您可以点击该横幅以导航到更改请求并审查所提议的更改。

您可以从导入页面查看已导入的Bundle的审批状态。导航到历史选项卡并选择相应的Bundle。如果有需要审批的变更请求，它将在导入页面上显示。

被拒绝或取消的变更请求会在导入页面上显示。

Apollo将显示所有导入多个环境的Bundle的变更请求。

批准已导入的更改

要批准在导入时创建的变更请求：

  1. 从环境页面横幅或导入历史中导航到变更请求。
  2. 审查所提议的更改。变更请求会列出Bundle中包含的所有配置更新和实体更改。
  3. 选择批准变更请求以应用更改。

一旦获得批准，Apollo 将会把更改应用到环境中。

管理多个捆绑包导入

如果在前一个捆绑包的更改请求仍在审批中时导入了新的捆绑包，Apollo 会将最新的更改更新到现有的更改请求中。这意味着在任何时间点，每个环境最多只有一个待审批的更改请求。

导入历史记录将显示多个捆绑包与同一个更改请求相关联。一旦更改请求获得批准，所有捆绑包的更改将会一起应用。
