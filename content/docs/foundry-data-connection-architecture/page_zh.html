架构示例

采用直连策略的 Foundry worker

此类连接也可称为“直连”。

对于访问接受 Foundry 入站流量的外部系统（例如 REST API 和云/SaaS 系统），采用直连策略的 Foundry worker 是推荐的默认方式。

在此场景中，数据连接运行于一个被称为 Foundry worker 的隔离计算容器内，由该容器处理经过身份验证的加密网络请求。 外部系统必须接受来自 Foundry 的入站流量。来自 Foundry 的网络出站流量通过平台内的直连出站策略进行管理和控制。

外部系统凭据采用 AES-256-GCM 服务端加密进行存储，且仅能由授权用户触发的容器进行解密。

要创建直连，请参阅有关设置数据源的说明。

采用智能体代理策略的 Foundry worker

此类连接也可称为“智能体代理”或“智能体精简模式”连接。

对于访问托管在私有网络上的外部系统，采用智能体代理策略的 Foundry worker 是推荐架构。这需要使用数据连接智能体。

在此场景中，智能体充当简单的网络隧道，本身不执行任何数据处理。

智能体向 Foundry 发起网络请求以建立 websocket 连接。进一步了解智能体上可用的网络控制功能。

所有数据连接和计算能力均由 Foundry worker 执行，这是一个具有可扩展计算资源的隔离容器，负责处理数据并通过提供的 websocket 与外部系统通信。

将智能体用作代理的功能，可通过平台内的智能体代理出站策略进行细粒度的管理和控制。

可使用多个智能体在多个外部系统之间实现工作负载均衡。

与直连类似，外部系统凭据采用 AES-256-GCM 服务端加密进行存储，且仅能由授权用户触发的容器进行解密。

要创建智能体代理连接，您需要：

  • 设置智能体。
  • 创建一个或多个智能体代理策略。
  • 使用上述策略设置 Foundry worker 数据源。

旧版架构：智能体 worker

旧版

智能体 worker 处于开发生命周期的旧版阶段。对于本地部署和私有托管系统，我们推荐使用采用智能体代理的 Foundry worker。有关对比，请参阅Foundry worker 与智能体 worker 对比。

此类连接也可称为“智能体胖模式”。

智能体 worker 连接是历史上用于从 Foundry 访问托管在私有网络上的外部系统的架构。这需要使用数据连接智能体。进一步了解智能体 worker 的已知限制。

要将现有的智能体 worker 数据源迁移至 Foundry worker，请遵循迁移指南。

在此场景中，智能体通过由 HTTPS 保护的单向出站连接不断轮询 Foundry，以获取待执行的新任务。接收到任务后，任务由智能体本身（也称为智能体 worker）执行，结果通过同一单向连接发回 Foundry。智能体与内部源系统通信所需的所有网络配置均在智能体宿主机上进行配置。

外部系统凭据采用 AES-128-GCM 加密存储在平台上，密钥则保存在智能体上。在执行功能期间，智能体从 Foundry 检索加密凭据，在本地进行解密，并将其用于查询。执行完成后，解密后的凭据将自动从内存中删除。

要创建智能体 worker 连接，您需要：

  • 设置智能体。
  • 将其配置为智能体 worker。
  • 设置智能体 worker 数据源并为其分配该智能体。

摘要

连接类型能力执行网络系统凭证
直连在 Foundry 中从 Foundry 直连至源系统。在平台中加密存储，加密密钥保留在平台中。
智能体代理在 Foundry 中从智能体到 Foundry 的出站连接，用于建立 websocket 并代理流量。在平台中加密存储，加密密钥保留在平台中。
智能体工作节点（旧版）在智能体主机上从智能体到 Foundry 的出站连接以轮询任务。从智能体到源系统的入站连接。在平台中加密存储，加密密钥保留在智能体上。
