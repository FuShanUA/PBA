管理标记

在 Foundry 设置的“标记”部分中可以管理标记。然后可以在整个平台上的资源上应用这些标记。

创建标记类别

类别删除

一旦创建，标记类别无法被删除。

管理员用户可以创建标记和标记类别并控制它们的元数据、可见性和成员关系。访问平台设置中的标记部分需要特殊的平台权限。

如果您是具有必要权限的管理员用户，可以通过点击平台设置中“标记”部分的新建标记类别按钮来创建标记类别。

当您创建一个标记（Marking）分类时，可以设置 分类可见性（category visibility） 并分配分类权限。默认情况下，分类的创建者将成为管理员。分类的创建者可以选择移除自己作为创建者，并添加另一个管理员。

分类可见性（Category visibility）

标记（Marking）的可见性是按分类来分配的，但组织（Organization）分类例外。每个组织都有自己的可见性设置。

在大多数情况下，标记和分类的名称和描述不是敏感信息，即使是没有标记访问权限的用户也应该能够看到这些信息。这种行为由分类可见性决定，默认情况下是 Visible。

如果分类可见性设置为 Visible，组织中的所有用户都可以在标记界面中看到该分类及其标记的存在。如果用户未满足标记权限，则他们将看不到该标记或分类的存在。

如果一个标记分类的可见性为 Hidden，则此分类及其标记的存在被视为敏感信息。Hidden 分类可以对未被明确授予 Category Viewer 权限的所有用户隐藏。

标记分类确保以下可见性规则：

  • 可见性是针对分类及其所有标记一起定义的；不能按每个标记单独分配。
  • 所有有权访问标记的用户都可以查看该分类以及该分类中的所有其他标记的存在。这些用户不会在 API 结果中显示为“分类查看者”，但应视为已知。
  • 所有在标记上拥有角色（管理员、删除者）的用户都可以查看该分类以及该分类中所有标记的存在。这些用户不会在 API 结果中显示为“分类查看者”，但应视为已知。
  • 所有在某个分类中拥有（管理员、查看者）角色的用户都可以查看该分类以及该分类内所有标记的存在。

分类权限

用户可以拥有指定他们如何与标记分类交互的权限：

  • 分类管理员： 可以更改分类描述和权限，并在分类中创建标记的用户。
  • 分类查看者： 可以看到分类的存在以及其中所有标记的用户。

按组织限制分类

一个标记分类可以被限制为仅属于一个组织，以确保它永远不会对那个组织以外的用户可见。

创建标记

标记删除

一旦创建，标记无法被删除或移动到其他类别。

如果您是具有必要权限的管理员用户，可以通过点击 新建标记 按钮在平台设置的标记部分创建新的标记。然后您可以指定标记管理员和标记移除者。

当您在 Foundry 内创建标记时，会自动获得“管理权限”访问权限。我们建议为新标记分配不同的团队成员这些权限。这样可以避免依赖单一管理员来管理所有标记的权限。

标记权限

以下权限适用于资源级标记。组织有单独的权限模型，拥有自己的 应用组织 和 扩展访问 权限。有关组织级权限的详细信息，请参阅 组织权限。

当您创建一个新的标记时，可以添加具有不同访问级别和权限的用户和组：

  • 管理权限： 可以授予管理此标记、其成员及其元数据权限的用户。
  • 应用标记： 可以将此标记应用于项目和资源的用户。此权限仅授予应用标记的能力，并不授予标记的成员资格。
  • 移除标记： 可以从项目和资源中移除此标记的用户。要移除一个标记，用户也必须能够应用该标记。
  • 成员： 可以查看受此标记保护的资源和项目的用户。

所有上述权限都是独立的，并不会自动为用户赋予成员权限。例如，一个用户可以在“标记”上拥有“应用标记”和“管理权限”的访问权限，但并不一定是该“标记”的成员。在这种情况下，用户可以将该标记应用到平台中的文件、文件夹和项目，并管理该标记，但他们无法查看用该标记标记的数据。

向用户授予标记

标记会全局授予用户。当用户被授予一个标记时，该用户就具有全局权限来查看由该标记限制的内容类型。然而，拥有对标记的访问权限并不意味着用户可以查看所有带有该标记的内容；用户仍必须通过其角色获得权限。除非用户在该标记上有额外的管理权限，否则用户不能将标记的访问权限授予其他用户。

如果将标记权限授予一个组，那么该组的新用户将继承这些权限。

应用标记

如果您满足以下两个要求，可以在资源、文件夹或项目上应用标记：

  1. 您在标记（Marking）上拥有“应用标记”权限。
  2. 您拥有“更新资源上的标记”权限，默认情况下该权限包含在所有者（Owner）角色中。

应用标记是一项敏感操作，可能限制下游用户的使用。在对现有管道（Pipeline）应用标记之前，我们建议采取以下步骤：

  1. 确认标记的存在：确保您要应用的标记已被创建，或者 创建该标记。
  2. 为您的管道创建分支： 创建一个新分支以继续这些步骤。当将标记应用于现有管道时，您可能需要在管道的某个点停止标记的传播（例如，以便清理了PII的数据集可以被最终用户打开）。stop_propagating 语法仅对受保护分支有效。在您的新分支合并到受保护分支之前，您不能 stop_propagating 继承的标记。
  3. 审查数据血缘关系：确认您在数据血缘图中拥有完整的节点集合，并确认您理解潜在更改对下游的影响。
  4. 审查管道中的事务类型： 标记会随着数据依赖关系在事务级别上传播。增量构建的数据集（类型为APPEND或UPDATE）需要特殊处理。具体来说，使用APPEND事务构建的数据集的最新视图将包含来自旧上游事务的依赖；相比之下，使用SNAPSHOT事务构建的数据集的最新视图（Foundry默认），仅依赖于上游数据集的最新事务。如果您只处理SNAPSHOT事务，可以继续按照这些步骤操作。如果不是，请参阅APPEND和UPDATE事务的文档以及标记如何传播的文档。
  5. 确定何时停止继承标记：探索管道的数据血缘中的节点，并审查列和数据预览，以确定您应停止继承要应用的标记的点。例如，您可能希望在管道中移除DOB（出生日期）列后停止传播PII标记。在此步骤中，您可以激活模拟模式，以查看标记将传播到何处以及如何传播。
  6. 提前进行必要的转换更改： 在你的分支中，将stop_propagating语法添加到你所识别的转换中。将你的分支合并到受保护的分支中，以便stop_propagating语法生效。
  7. 构建你的生产管道： 构建所有下游的转换，包括你所识别的需要stop_propagating的节点。你需要重新构建包含stop_propagating语法的数据集以及stop_propagating转换的所有下游数据集。如果你有下游的APPEND或UPDATE数据集，请查看此指南。
  8. 审查模拟变更： 审查你管道的最终模拟数据血缘关系。确认你stop_propagating转换的下游数据集不会受到新标记应用的影响。确保敏感列不会出现在不会接收传播标记的数据集中。
  9. 应用标记： 上述步骤应能确保应用标记将保护包含敏感数据的数据集，并且标记不会传播到不必要的范围。此时，你可以应用该标记。

如果在应用标记后发现错误（例如，一大组用户无法看到他们应该能看到的数据），您可以移除该标记，它会立即停止传播。然后，您应再次审查模拟的更改，以确定问题的原因。

移除标记

您可以移除标记以给予更多用户访问权限、重新分类转换后的数据，或从继承的标记中分离派生资源。

例如，假设数据集A包含个人身份信息（PII），并且有一个标记保护从数据集A派生的数据集。如果数据集B是从数据集A派生的，但已经过转换以去除PII，您可能需要从数据集B中移除该标记，以便更广泛地使用数据。

要移除一个标记，您必须拥有对该特定标记进行应用和删除的权限。您还需要通过允许更改其标记的角色来访问相关资源。如果您使用的是默认角色，这种访问权限可通过Owner角色获得。

直接移除标记

直接从文件、文件夹或项目中移除标记会立即从任何继承了该标记的依赖项中移除该标记。在直接移除标记后，您不需要重新构建下游数据集，因为该标记会在下游立即被移除。下面是一个直接应用的PII标记的示例。

移除继承的标记

您只能从受限制视图和数据集移除继承的标记。如果在生成派生文件时，受限内容被删除或隐藏，您可以从派生文件中移除标记。当移除继承的标记时，下游数据集将不再受到该标记的保护，更多用户可能访问该数据集。用户仍然需要拥有标记的权限才能查看上游数据。

要从受限制视图中移除继承的标记，请编辑受限制视图并点击相应标记旁边的停止传播。

要从数据集中移除继承的标记，请在转换代码中使用stop_propagating语法。为了安全地移除继承的标记，请按照以下步骤操作：

  1. 确定移除敏感数据的位置： 审查您的完整管道，并决定您希望在哪里移除敏感数据。
  2. 创建分支： stop_propagating 语法仅对同时受保护且已启用设置 Require security approvals before merging 的分支有效。在您的分支合并到受保护的分支之前，您无法 stop_propagating 继承的标记。
  3. 进行必要的转换更改： 在您选择的数据集内，删除敏感数据，并将 stop_propagating 语法添加到转换中。
  4. 构建您的分支： 构建您的分支并确认输出数据集中已删除敏感数据。此时，由于您处于非受保护分支上，stop_propagating 语法不会阻止标记的继承。
  5. 合并您的分支： 在确认敏感数据已删除后，您可以安全地将转换更改合并到主分支。合并您的分支将应用 stop_propagating 语法。
  6. 构建您的主分支和下游依赖项： stop-propagating 的更改必须沿着您管道中的最新事务传播，这要求您重新构建具有 stop_propagating 语法的数据集以及 stop_propagating 转换下游的所有数据集。所有数据集构建完成后，请确认已移除继承的标记。如果存在下游的 APPEND 或 UPDATE 事务，请查阅 更多文档。

调查标记继承

标记是 继承的 沿文件层次结构和数据依赖关系。这意味着文件本身可以通过标记进行保护，或者数据本身可以通过标记进行保护。有时你需要调查标记的来源，并且概念性地理解标记可以继承的不同方式是很重要的。

你可以通过选择文件以打开右侧详细信息面板，然后查看访问要求 > 标记部分来查看文件的标记。

检查文件层次结构

要查看文件的标记，请选择它并打开资源面板。在访问部分下查找。

沿文件层次结构继承的标记由文件夹旁车图标表示。

调查数据依赖关系

要确定标记来自哪个数据依赖，可以使用数据血缘应用程序，或者在打开数据集文件后使用比较选项卡。

由于标记是在 文件级别 应用的，但沿着 事务 传播，因此很难确定何时为数据集引入了标记。我们建议首先检查数据血缘应用程序，以查看标记的来源。

使用数据血缘应用程序，点击 图例 > 权限类型：数据集中的数据访问 然后展开数据血缘中的所有上游节点，并遍历数据血缘以查看标记的来源。沿数据依赖继承的标记由数据血缘侧车图标表示。
