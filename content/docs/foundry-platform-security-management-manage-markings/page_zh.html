管理标记

标记在 Foundry 设置的标记部分进行管理，随后会应用于平台上的各类资源。

创建标记类别

类别删除

标记类别一旦创建，便无法删除。

管理员用户能够创建标记和标记类别，并控制其元数据、可见性及成员关系。访问平台设置中的标记部分需要具备特殊的平台权限。

如果您是具备相应权限的管理员用户，可以在平台设置的标记部分，通过点击 新建标记类别 按钮来创建标记类别。

创建标记类别时，您可以设置类别可见性并分配类别权限。默认情况下，类别创建者将成为管理员。类别创建者可以选择移除自己的创建者身份，并转而指定其他用户为管理员。

类别可见性

标记可见性按类别进行分配，组织（Organization）类别除外。每个独立的组织各有其专属的可见性设置。

在多数情况下，标记及类别的名称和描述并不属于敏感信息，即便是未拥有标记访问权限的用户也应能够看到。此行为由类别可见性决定，其默认值为 可见。

若类别可见性设为 可见，组织内的所有用户均可在标记界面中看到该类别及其标记的存在。如果用户不满足标记权限要求，则无法看到相应标记或类别的存在。

若标记类别可见性为 隐藏，则该类别及其标记的存在本身即被视为敏感信息。对于未显式授予 类别查看者 权限的所有用户，可以将 隐藏 类别设为不可见。

标记类别遵循以下可见性规则：

  • 可见性是针对类别及其所有标记统一定义的；无法按单个标记进行分配。
  • 所有拥有某项标记访问权限的用户，均可查看该类别及该类别下所有其他标记的存在。这些用户不会在 API 结果中显示为“类别查看者”，但应默认其具备此权限。
  • 所有在某项标记上拥有角色（管理员、移除者）的用户，均可查看该类别及该类别下所有标记的存在。这些用户不会在 API 结果中显示为“类别查看者”，但应默认其具备此权限。
  • 所有在某类别上拥有角色（管理员、查看者）的用户，均可查看该类别及该类别下所有标记的存在。

类别权限

用户可拥有决定其如何与标记类别交互的权限：

  • 类别管理员：能够更改类别的描述和权限，并在该类别中创建标记的用户。
  • 类别查看者：能够查看类别及其中所有标记是否存在的用户。

按组织限制类别

可将标记类别限定于单一组织，以确保该组织之外的用户绝对无法看到它。

创建标记

标记删除

标记一旦创建，便无法删除或移至其他类别。

如果您是具备相应权限的管理员用户，可以在平台设置的标记部分，通过点击 新建标记 按钮来创建新标记。随后，您可以指派标记管理员和标记移除者。

在 Foundry 中创建标记时，系统会自动授予您“管理权限”的访问权限。我们建议为新的标记分配不同的团队成员来管理这些权限。这样，您可以避免依赖单一管理员来管理所有标记权限。

标记权限

以下权限适用于资源级别的标记。组织拥有独立的权限模型，包含自身的应用组织和扩展访问权限。有关组织级别权限的详细信息，请参阅组织权限。

创建新标记时，您可以添加具有不同访问级别和权限的用户和用户组：

  • 管理权限：用户可以授予管理此标记、其成员及其元数据的权限。
  • 应用标记：用户可以将此标记应用到项目和资源。此权限仅授予应用标记的能力，不授予标记的成员资格。
  • 移除标记：用户可以从项目和资源中移除此标记。要移除标记，用户还必须能够应用该标记。
  • 成员：用户可以查看受此标记保护的资源和项目。

上述所有权限相互独立，不会自动为用户提供成员权限。例如，用户可以在某个标记上拥有“应用标记”和“管理权限”访问权限，但不是该标记的成员。在这种情况下，用户可以将标记应用到平台上的文件、文件夹和项目中，并管理该标记，但他们无法查看带有该标记的数据。

向用户授予标记

标记是向用户全局授予的。当用户获得某个标记时，该用户即拥有全局权限来查看受该标记限制的内容类型。然而，拥有标记的访问权限并不意味着用户可以查看带有该标记的所有内容；用户仍必须通过其角色获得相应权限。除非用户对该标记拥有额外的管理权限，否则他们无法向其他用户授予标记访问权限。

如果标记权限被授予给某个用户组，那么加入该组的新用户将继承这些权限。

应用标记

如果满足以下两个要求，您可以对资源、文件夹或项目应用标记：

  1. 您拥有该标记的“应用标记”权限。
  2. 您拥有“更新资源标记”权限，该权限默认包含在数据主人角色中。

应用标记是一项敏感操作，可能会限制下游用户。在将标记应用于现有管道之前，我们建议采取以下步骤：

  1. 确认标记存在：确保您要应用的标记已创建，或者创建该标记。
  2. 为管道创建分支：创建一个新分支以继续执行这些步骤。在将标记应用于现有管道时，您可能需要在管道中的某个位置停止标记传播（例如，以便最终用户可以打开已清理个人身份信息的数据集）。stop_propagating 语法仅在受保护分支上生效。在您的新分支合并到受保护分支之前，您无法对继承的标记使用 stop_propagating。
  3. 审查数据血缘关系：确认您的数据血缘关系图包含完整的节点集合，并确认您了解潜在变更对下游的影响。
  4. 审查管道中的事务类型：标记在事务级别沿着数据依赖关系传播。增量构建的数据集（APPEND 或 UPDATE 类型）需要特殊处理。具体而言，使用 APPEND 事务构建的数据集的最新视图将包含来自旧上游事务的依赖关系；相比之下，使用 SNAPSHOT 事务（Foundry 默认设置）构建的数据集的最新视图仅取决于上游数据集的最新事务。如果您仅处理 SNAPSHOT 事务，可以继续执行这些步骤。如果不属于这种情况，请参阅有关APPEND 和 UPDATE 事务以及标记如何传播的文档。
  5. 确定何时停止继承标记： 探索管道数据血缘关系中的节点，审查列和数据预览，以确定应在何处停止继承要应用的标记。例如，一旦管道中移除了 DOB（出生日期）列，您可能希望停止传播 PII 标记。在此步骤中，您可以激活模拟模式，以查看标记将传播的位置和方式。
  6. 预先进行必要的转换更改： 在您的分支中，在已识别的转换中添加 stop_propagating 语法。将您的分支合并到受保护分支中，以便 stop_propagating 语法生效。
  7. 构建生产管道： 构建下游的所有转换，包括您指定为 stop_propagating 的节点。您需要重新构建包含 stop_propagating 语法的所有数据集，以及这些 stop_propagating 转换下游的所有数据集。如果您有下游的 APPEND 或 UPDATE 数据集，请查阅该指南。
  8. 审查模拟更改： 检查管道最终的模拟数据血缘关系。确认 stop_propagating 转换下游的数据集不会受到应用新标记的影响。确保敏感列不会出现在未接收传播标记的数据集中。
  9. 应用标记： 上述步骤应已验证：应用标记将保护包含敏感数据的数据集，且标记不会超出必要范围进行传播。此时，您可以应用标记了。

如果在应用标记后发现有误（例如，大量用户无法看到他们本应能查看的数据），您可以移除该标记，它会立即停止传播。随后，您应再次审查模拟更改，以查明问题的原因。

在以下示例中，DOB 列已被移除，并且在本体 passengers 数据集上应用了 stop_propagating 语法。PII 标记被应用在原始的 passenger 数据集上，且仅传播至清洗后的 passenger 数据集。

移除标记

您可以移除标记，以便向更多用户授予访问权限、对转换后的数据进行重新分类，或者将派生资源与继承的标记分离开来。

例如，假设数据集 A 包含 PII，并且某个标记保护着派生自数据集 A 的数据集。如果数据集 B 派生自数据集 A，但其经过转换已移除了 PII，您可能希望从数据集 B 中移除该标记，以允许更广泛地使用该数据。

要移除标记，您必须同时拥有应用和移除该特定标记的权限。您还需要通过允许更改其标记的角色获取对目标资源的访问权限。如果您使用的是默认角色，则可以通过 数据主人 角色获得此访问权限。

移除直接应用的标记

直接从文件、文件夹或项目中移除标记，将立即从继承该标记的所有依赖项中移除该标记。直接移除标记后，您无需重新构建下游数据集，因为标记会立即在下游被移除。以下概念性示例展示了一个直接应用的 PII 标记。

移除继承的标记

您只能从受限视图和数据集中移除继承的标记。如果在派生依赖文件时移除或混淆了受限内容，您可以从派生文件中移除该标记。移除继承的标记后，下游数据集将不再受该标记保护，可能会有更多用户获得该数据集的访问权限。用户仍需拥有该标记的访问权限才能查看上游数据。

要从受限视图中移除继承的标记，请编辑受限视图，并点击相应标记旁边的 停止传播。

要从数据集中移除继承的标记，请在您的转换代码中使用 stop_propagating 语法。要安全地移除继承的标记，请按照以下步骤操作：

  1. 确定移除敏感数据的位置： 检查完整管道，决定在何处移除敏感数据。
  2. 创建分支： stop_propagating 语法仅在受保护且启用了 合并前需要安全审批 设置的分支上生效。在分支合并到受保护分支之前，无法对继承的标记使用 stop_propagating。
  3. 进行必要的转换更改： 在选定的数据集中移除敏感数据，并在转换中添加 stop_propagating 语法。
  4. 构建分支： 构建分支并确认输出数据集中已移除敏感数据。由于当前处于非受保护分支，此时 stop_propagating 语法不会阻止标记被继承。
  5. 合并分支： 确认移除敏感数据后，即可安全地将转换更改合并到主分支。合并分支后会应用 stop_propagating 语法。
  6. 构建主分支及下游依赖： stop-propagating 更改必须沿着管道中的最新事务传播，这需要重新构建包含 stop_propagating 语法的数据集以及 stop_propagating 转换下游的所有数据集。所有数据集构建完成后，确认已移除继承的标记。如果存在下游 APPEND 或 UPDATE 事务，请查阅《附加文档》。

在以下示例中，DOB 列已被移除，并在本体 passengers 数据集上应用了 stop_propagating 语法，以阻止 PII 标记继续向下游传播。

调查标记继承

标记会沿着文件层级和数据依赖关系进行继承。这意味着文件本身可以受标记保护，且/或数据本身也受标记保护。有时需要调查标记的来源，因此从概念上理解标记继承的不同方式非常重要。

选中文件以打开右侧详情面板，然后在 访问要求 > 标记 部分下查看文件的标记。

调查文件层级

要查看文件的标记，请选中并打开资源面板。在 访问 部分下查看。

通过文件夹侧边栏图标可指示沿文件层级继承的标记。

要查明标记在文件层级中的起源位置，请在 Foundry 工作区中遍历文件层级。在以下示例中，从初始文件和文件夹资源开始遍历文件层级，可以验证 flight data 文件夹具有 PII 标记。资源侧边栏上会显示访问信息。

调查数据依赖关系

要查明标记源自哪个数据依赖关系，可以使用数据血缘关系应用程序，或在打开数据集文件后使用“比较”选项卡。

标记在文件级别应用，但会沿事务传播，因此很难确定数据集何时引入了标记。建议先检查数据血缘关系应用程序，以查看标记的来源。

使用数据血缘关系应用程序时，点击 图例 > 权限类型：数据集中的数据访问。然后展开数据血缘关系的所有上游节点并遍历数据血缘关系，查看标记的来源。数据血缘关系侧边栏图标表示沿数据依赖关系继承的标记。
