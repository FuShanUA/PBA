使用新的自定义模型实现，而不是函数接口来在 AIP 中注册模型

发布日期：2026-05-21

一种新的自定义模型（BYOM）实现，也称为注册模型，现已全面可用，提供了针对最常见的和标准的模型提供者 API 的优化流程。通过注册模型，您可以将自己的 LLM 或账户连接到 AIP，跨越 Palantir 开发者产品，如 AI FDE、AIP Analyst、AIP Chatbot Studio、AIP Logic、Workshop 等。而不是 使用函数接口，您可以在 控制面板 中注册一次模型，以便它在 AIP 中原生流动。一旦注册，该模型会出现在所有应用程序的模型选择器中，并遵守与 Palantir 提供的模型相同的速率限制、权限和可观察性。

新增内容?

新的实现方式取代了之前注册大型语言模型（LLM）的 函数接口方法。与传统的函数接口方法相比，注册的模型有以下优势：

  • 更易于设置： 注册管理员只需在 控制面板中注册一次模型，无需编写和维护 TypeScript 函数或 Webhook。
  • 支持更多 AIP 应用程序： 新的 BYOM 实现方式 支持多个 AIP 和 Foundry 应用程序，并在控制面板和资源管理中提供基础设施工具。
  • 支持更多模型功能： 注册的模型支持工具调用、推理、结构化输出、视觉输入和流式传输。
  • 与 AIP 基础设施集成： 在注册、项目和用户级别查看 LLM 的使用限制；在 资源管理 应用程序中观察使用情况；在 控制面板中进行权限管理和启用；并通过 AIP 应用程序中的共享模型选择器访问您的模型。

在AIP 设置扩展中提供对您注册模型的访问权限。

何时应使用注册模型？

基于大语言模型（LLM）的支持和可行性，我们通常建议使用Palantir提供的来自模型提供商（如Anthropic或OpenAI）的模型，或者Palantir自托管的开源模型（如Llama模型）。然而，您可能更希望将自己拥有的模型或账户带入AIP。我们建议仅在由于法律和合规原因无法使用Palantir提供的模型时，或当您拥有自己的微调或其他独特的LLM，希望在AIP中利用它时，才使用注册模型。

遗留的函数接口方法仍可用于现有集成，但不再推荐用于新用例。如果您有注册模型无法很好解决的使用场景，请联系Palantir支持团队。

CBAC 注册现在可以在 Compass 项目中存储本体论资源

发布日期：2026-05-21

基于分类的访问控制（CBAC）现在适用于本体论资源，使您可以通过 Compass，Palantir 平台的文件系统 配置谁可以查看、编辑和管理这些资源。要使用此功能，必须首先由本体论所有者启用。

当启用该功能时，用户必须将本体论资源保存到一个项目中。所选位置决定了查看、编辑和管理权限。这种基于项目的权限方法取代了以前的本体论权限模型：本体论角色和数据源派生的权限。主要优势包括以下内容：

  • 统一的权限模型： 本体论资源现在使用与其他资源类型相同的权限系统。
  • 批量管理： 在项目或文件夹级别设置权限，以一次控制多个资源的访问。
  • 权限可解释性： 在 安全 选项卡中显示查看和编辑对象类型所需的权限，以及查看实例或运行操作所需的权限。
  • 额外的隐私控制: 通过应用标记或将其放置在用户未获得角色授权的项目中，隐藏敏感的本体论资源。
  • 指南针（Compass）整理原语: 用户可以使用投资组合、标签来组织本体论资源，并使用角色授权或标记来从用户处隐藏不相关的资源。

查看对象的权限仍然需要对象类型和数据源的权限。迁移到项目不会改变谁有访问数据源的权限。

当在具有 基于分类的访问控制（CBAC） 的项目中使用本体论时，用户必须在创建文件时指定文件分类。 该分类必须等于或低于 允许的最大分类 。如果没有指定分类，则对象类型实例化会失败。

此功能对新本体论已启用。对于现有本体论，必须 手动启用，且现有本体论资源需要迁移。

在Pipeline Builder中升级从PDF板提取文本以提高性能

发布日期：2026-05-19

Pipeline Builder 现在包含一个 v2 版本的“从 PDF 提取文本”板，基于 PyMuPDF 构建。v2 板用单个基于 PyMuPDF 的提取路径替换了 v1 板的 Tesseract 和 Docling 后端。在与 v1 Tesseract 和 Docling 流程的内部基准测试中，PyMuPDF 在电子（嵌入式文本）PDF 上产生了更高的提取准确性，并且处理文档的速度大约提高了 10 倍。v2 板还能处理比 v1 更大的批量大小，而 v1 受限于 Docling 支撑的布局模型。

Pipeline Builder 中的从 PDF 提取文本转换。

有什么新功能？

  • 灵活的输出格式： 除了纯文本外，还可以以 Markdown 或 HTML 格式导出提取的内容，使与下游 LLM 应用程序的集成更加简便。
  • 自动 PDF 类型检测： 系统智能地为每份文档选择最佳的提取方法，最大化准确性和速度。
  • 文档理解： 利用基于混合图神经网络（GNN）方法（PyMuPDF-layout）的高级布局分析，实现对文档元素（标题、段落、表格和图形）的精确识别。
  • CPU 执行： 所有提取方法——原始文本和 OCR——现在都由 PyMuPDF 处理，确保结果一致且性能提升，同时在 CPU 上高效运行。

如何升级

  • 启用功能标志 在管道构建器中启用提取文本 v2 表达式 以默认方式开始创建 v2 板。
  • 打开 警告 选项卡（底部面板）以列出您管道中的每个 v1 从 PDF 提取文本板。
  • 导航到每个板并选择 升级。
  • 保存并部署 您的管道。

使用参数化功能，在您的Python转换中声明类型参数，其值在计划上设置。

在标准模式下，计划直接设置值，使像增量转换的周期性压缩等用例成为可能。在并行化模式（高级） ，计划管理许多独立的运行设置，每个设置都有自己的参数值，从而通过并行运行转换来实现高规模场景模拟等用例。

创建一个参数化计划，并为目标数据集转换中声明的参数选择值。

通过在 操作类型上设置计划规则，当应用操作时可以触发引用计划的构建，并将操作的参数值传递给底层转换。这使得可以直接在 操作应用程序 中重新计算数据集，或者作为 Automate 中的操作效果 的一部分。

    . 从5月11日那一周开始普遍可用，该库提供了React钩子和内置的性能优化功能，例如全局缓存、乐观更新等，以帮助您更轻松地构建与Foundry交互的前端OSDK应用程序。

    库的优势

    当您的应用程序主要通过OSDK消费数据时，请使用@osdk/react来简化您在本体论中查询数据和执行操作的方式。该库提供了以下优势，以增强您的开发体验：

    • 标准化对象缓存：该库将每个对象存储一次。当您调用编辑对象的操作时，所有引用该对象的列表、链接和视图都会自动更新。
    • 基于操作的失效机制：操作响应会指定哪些对象被添加、修改或删除。无需手动失效，列表会根据其where条款重新评估。
    • 乐观更新及自动回滚： 在创建、编辑或删除对象后，立即更新本地缓存，然后再响应服务器。乐观更改将应用于临时存储层，并在操作失败时自动回滚。
    • 跨组件去重：运行相同查询的两个组件共享一个网络请求和一个缓存条目。

    为了提高开发的简便性，该库还为每个本体论原语提供了符合习惯的 React 钩子，以及你可以使用 OSDK React 钩子的平台 API，以及通过 AGENTS.md 文件获得的 LLM 默认指导。

    介绍 SQL Studio，Foundry 的专用 SQL 应用程序

    发布日期：2026-05-07

    更新：截至 2026 年 6 月 15 日这一周，此产品现已正式发布。

    SQL Studio，Foundry 用于编写和运行 SQL 查询的专用应用程序，现已进入公测阶段。SQL Studio 在 Foundry 上为表格数据和本体论对象类型提供了交互式 SQL 分析功能，由专门构建的 SQL 引擎和 AI 辅助查询编写功能支撑。

    SQL Studio 为表格数据和本体论对象提供了一个交互式、AI辅助的 SQL 分析界面。

    SQL Studio 基于嵌入在 Dataset Preview（数据集预览）、Data Lineage（数据血缘）和 Ontology Manager（本体论管理器）等应用程序中的上下文相关 SQL 控制台，现在提供了一个专用应用，支持对表格数据进行读写 SQL 操作，支持对本体论对象类型进行读取操作，并能够发布可重复使用的本体论 SQL 函数。

    由本体论 SQL 和 Furnace 支持

    SQL Studio 基于两个共享相同 Spark SQL 语法 的 Foundry SQL 引擎：用于查询本体论对象类型的本体论 SQL，以及用于查询表格数据的 Furnace。

    本体论 SQL 是 Foundry 用于查询本体论对象类型和多对多关联的 SQL 引擎。查询直接针对对象存储执行，使用内存计算路径以实现对受支持查询形状的快速响应时间，更复杂的查询会自动路由到 Spark。

    Furnace 是 Foundry 的用于表格数据的 SQL 引擎。它在 Trino 和 Spark 之间动态路由查询，为合适的负载提供显著更快的查询速度。Furnace 支持读取和写入操作。

    关键特性

    SQL Studio 将完整的 SQL 分析体验集中在一个地方：

    • 统一的数据和对象查询： 从一个应用程序中查询表格数据和本体论对象类型，在数据模式和对象模式之间切换，同时使用通用的 Spark SQL 语法。
    • 低延迟、交互式分析： SQL Studio 是为迭代式、交互式开发工作流程而设计的。查询会动态路由到最合适的计算引擎，包括针对支持的查询结构的更快的非 Spark 选项，并在预热的、完全管理的计算资源上运行。
    • AI 辅助代码生成： 一个对话式的 AIP 侧边栏可以帮助您编写、解释和调试查询。它理解 Foundry 支持的 SQL 语法，并可以查看您的编辑器，包括当前代码和任何引用的数据集、表、对象类型的模式。
    • 以表格或图表预览结果： 将查询结果以表格形式查看，或使用内置的折线图和柱状图进行可视化。结果限制是可配置的，具有适当权限的用户可以选择返回最多 10,000 行而不是默认的 1,000 行预览。
    • 保存和共享 SQL 工作表： 使用便签进行一次性分析，或者将您的工作保存为 SQL 工作表。工作表可以私有保存供个人使用，或在项目中与同事共享。保存会创建一个新版本，您可以稍后审查和恢复；未保存的更改会自动暂存，因此用户的进行中的工作会在会话之间持续存在。
    • 读写支持： 除了 SELECT 查询，SQL Studio 还支持对数据集的 CREATE TABLE 操作，以及对 Iceberg 表的 CREATE、INSERT、UPDATE 和 DELETE 操作。
    • 本体论 SQL 函数（测试版）： 在对象类型上定义可重用的、带参数的 SQL 查询，并将其发布为 SQL 函数 . 支持的查询结构可以在与本体论SQL查询相同的内存路径上以低延迟执行。在Foundry中使用它们，包括在Workshop、Actions、Automate和本体论SDK中。SQL函数是实验性的，并非所有Foundry环境都启用，请联系你的Palantir代表进行注册。

    开始使用

    要开始使用SQL Studio，Foundry管理员应从控制面板的 应用访问 页面启用该应用程序。启用后，SQL Studio可以直接从应用菜单访问。

    有关SQL Studio功能的信息，请参阅 SQL Studio文档。关于语法指导，请参考 SQL方言文档。要了解底层引擎的更多信息，请参阅 Furnace 和 本体论SQL 概述。

    下一步有什么 接下来有什么

    SQL Studio 正在积极开发中，一些功能将在近期路线图中推出，包括：

    • 全局分支 和 市场包装 对 SQL 工作表的支持。
    • 应用程序可用性改进，例如多个编辑器标签、改进的查询历史记录和更多的图表类型选项。
    • 与 Foundry 的生产管道工具和基于 Git 的 CI/CD 开发工作流集成。

    全局分支功能现已全面推出（5月18日）：跨应用进行分支、测试和合并

    发布日期：2026-05-05

    从2026年5月18日这一周开始，全局分支（以前称为Foundry分支）将对所有用户的所有注册账户全面可用。全局分支提供了一个共享的工作流程，可在单个分支上跨多个应用程序进行更改，无需中断生产工作流即可端到端地测试这些更改，并将它们合并回 Main。查阅 全局分支文档 以了解更多详情。

    在本体论管理器中查看主分支更新和资源检查状态。

    支持的应用和工作流

    全局分支功能适用于转换、TypeScript v1 函数仓库、Pipeline Builder（管道构建器）、本体论、Workshop（研讨会）、AIP Logic（AIP逻辑）、对象视图。

    一个显示资源、审批状态和“不要合并提案”设置的提案概览。

    对于这些应用，支持以下工作流程：

    • 创建分支并修改资源： 创建或访问一个分支，并对资源进行更改而不影响 Main 分支。
    • 删除资源： 从分支中删除资源以将其重置为 Main 分支上的状态。
    • 从 Main 包含新更新并解决冲突： 将你的分支资源重新定位以用 Main 分支上的最新更改更新它。如果存在冲突，你将被重定向到相应应用来解决它们。
    • 保护资源：资源拥有者可以保护资源，使得更改必须通过分支并在合并前获得批准。
    • 定义审批策略：项目拥有者定义审批策略，指定合格的审查者、所需的审批以及贡献者是否可以自行批准。对于代码仓库和管道构建器，您可以在资源本身内定义策略。
    • 查看分支资源：使用数据血缘来检查资源的分支状态并追踪更改如何传播。
    • 清除所有检查并合并更改：当创建一个提案时，系统会对每个资源运行检查以暴露冲突或缺少审批等问题。一旦所有检查通过，您可以合并您的提案。

    更新的安全模型和分支生命周期

    安全模型更新随全局分支功能正式发布（GA）推出。分支生命周期的变化将在接下来的几周内陆续推出。

    • 合并权限（随GA推出） : 合并不再需要专用的分支角色。任何可以查看提案的用户，在所有审批和检查通过后都可以合并。一个新的 不要合并 设置允许分支所有者在准备就绪前阻止合并。
    • 分支生命周期（将在接下来的几周内推出）: 当一个分支变得不活跃时，系统将取消其本体论资源的索引，并最终删除其数据；所有逻辑都将被保留。分支将不再自动关闭，而是您可以随时手动归档和恢复它们。

    受限视图、自动化及未来展望

    受限视图和自动化现已进入测试阶段。核心分支工作流已可用，但一些正式发布级别的功能——例如审批集成以及从分支中移除资源——尚未提供。请联系Palantir支持团队以启用这些功能，并查阅特定应用的文档以了解每个应用当前的范围。
