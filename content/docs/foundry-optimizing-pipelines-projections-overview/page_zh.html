数据集投影

数据集投影能够提升大范围查询的性能。如果需要针对多种查询模式（例如在两个独立的列上进行过滤）优化数据集，应考虑为其添加投影。下方列出了投影的使用场景具体示例。

每个投影通常仅支持单一的查询模式，侧重于优化一组列上的过滤操作或连接操作。单个数据集可添加多个投影，且所有列类型均支持投影。此外，快照数据集或增量构建的数据集均可添加投影。系统使用 Noho 服务来管理投影，并在设置投影时于数据集架构中引用该服务。

投影存在以下限制：

  • 仅支持为仅追加（append-only）的数据集添加投影。在仅追加的数据集中，只能向其添加（即追加）行；无法手动从其中删除文件或事务。具体而言，仅追加数据集仅能执行将行添加至数据集的 APPEND、SNAPSHOT 和 UPDATE 事务。投影系统会自动识别违反此要求的数据集并禁用其投影，但作为用户仍应尽量避免出现此类情况。
  • 投影不支持架构演进，即使对于添加列这类增量一致的操作也不例外。

查看以下使用场景示例，以确定投影是否适用于您的场景。请先了解如何设置投影。

使用场景

多种类型的查询均可从投影中获益。以下示例将演示以下场景：

  • 在多个预先指定的维度上对大型数据集进行过滤
  • 在预先指定的一组列上对大型数据集进行连接
  • 在预先指定的一组列上对大型数据集进行聚合
  • 从频繁写入的增量数据集或流式数据集中读取数据
  • 直接查询上传的数据

基于列列表进行过滤

给定一个有序的列列表，可针对该列表的任意前缀优化过滤操作。投影仅能加速将列与常量值进行比较的过滤操作。此外，对字符串列的任何过滤操作均必须区分大小写。

例如，假设存在一个针对有序列列表 ["x", "y", "z"] 上的过滤操作进行优化的投影。 该投影将加速以下类型的查询：

  • SELECT * FROM dataset WHERE x = 5 AND y = 10 AND z = '15'
  • SELECT * FROM dataset WHERE x = 5 AND y = 10
  • SELECT * FROM dataset WHERE x = 5 AND q = 3
    • 如果其他列上存在未包含在已配置列表中的附加过滤条件（如本例中的 q = 3），Spark 会自动尝试拆解过滤条件，将可“下推”至数据源的部分（本例中为 x = 5）先行处理，随后再应用其余条件。

但以下类型的查询不会被优化：

  • SELECT * FROM dataset WHERE abs(x) == 10
    • abs(x) == 10 未将列与常量值进行比较。
  • SELECT * FROM dataset WHERE x % 100 == 10
    • x % 100 == 10 未将列与常量值进行比较。
  • SELECT * FROM dataset WHERE y = 10
    • ["y"] 不是 ["x", "y", "z"] 的前缀。
  • SELECT * FROM dataset WHERE z = '15'
    • ["z"] 不是 ["x", "y", "z"] 的前缀。
  • SELECT * FROM dataset WHERE x = 5 OR q = 3
    • Spark 无法将 x = 5 过滤条件“下推”至数据源，因为这样会漏掉 x = 5 为假而 q = 3 为真的行。

范围查询

投影可选择性地加速过滤列上的范围查询，例如：

  • SELECT * FROM dataset WHERE x > 5 AND x < 10
  • SELECT * FROM dataset WHERE s LIKE 'SOME_PREFIX%'

局限性

  • 无法创建既包含非基本类型（例如数组）列，又支持对筛选列进行范围查询的投影。

基于列集合的连接

给定一组无序的列和一个分桶数量，仅针对该确切的列集合和分桶数量优化连接操作。

例如，针对 {"x", "y"} 上的连接优化的投影将优化以下类型的查询：

  • SELECT * FROM dataset1 INNER JOIN dataset2 ON dataset1.x = dataset2.x AND dataset1.y = dataset2.y

但以下查询不会被优化：

  • SELECT * FROM dataset1 INNER JOIN dataset2 ON dataset1.x = dataset2.x

连接投影与非投影数据集

在 Foundry 中，大型数据集的连接通常执行排序合并连接。这涉及根据连接键对每个数据集进行分区，按该键对每个分区进行排序，然后合并具有相同键的（已排序）分区。

  • 在一般情况下，这涉及对两个数据集进行洗牌和排序。
  • 如果其中一个数据集具有针对在连接列上进行连接而优化的投影，则它不会被洗牌或排序，但另一个数据集会。
  • 如果两个数据集都具有针对在连接列上进行连接而优化的投影，并且使用相同数量的分桶，则两个数据集都不会被洗牌或排序。这可以带来显著的性能提升。将具有连接优化投影的数据集与显式分桶的数据集（无投影）进行连接也是如此，但同样，分桶数量和连接列必须完全匹配。

在 Foundry 中，大多数投影消费者在执行排序合并连接时，默认不会利用投影已经排序这一事实，因此你可能仍然会在 Spark 查询计划中看到排序操作。在转换中，你可以使用 Spark profile BUCKET_SORTED_SCAN_ENABLED，基于投影已排序的事实来修改 Spark 的行为，但这并不总能提升性能，甚至可能导致性能下降。

基于预指定列集合的聚合

在 Foundry 中，聚合操作通常涉及根据聚合键对数据集执行洗牌交换（即对数据集进行分区）。当从哈希分桶的投影读取数据时（所有连接优化的投影都属于这种情况，但筛选优化的投影可能并非如此），消费者可以避免这种洗牌，从而在聚合期间带来显著的性能提升。对于以仅追加方式增量更新的数据集，其上的主键预期检查是一个特别受益于投影（在主键列上）的场景，这不仅因为投影执行的压缩操作，还因为投影的哈希分桶可以在用于计算预期的聚合中加以利用。

从增量管道读取

增量管道可能会导致非常高的文件数量，进而导致读取性能下降。例如，一个每五分钟写入十个分区的管道每年将写入超过一百万个文件。由于诸多原因，读取这些文件非常困难，包括仅仅生成输入分区列表这样的操作。

投影提供了一种透明地压缩此类增量管道的方法。只需设置一个针对筛选或连接优化的投影，读取操作就会使用该投影。《了解有关在增量管道中使用投影的更多信息》。

请注意，即使投影未与规范数据集完全保持同步，消费者仍然可以利用投影的筛选或连接优化，前提是：在投影上次构建时的最新事务与规范数据集上的当前最新事务之间，规范数据集上没有发生 SNAPSHOT 或 DELETE 事务（或修改现有文件的 UPDATE 事务）。

查询上传的数据

对于数据查询而言，CSV 是一种低效的文件格式，但上传至 Foundry 的数据（如通过文件系统源的 Data Connection 导入，或手动上传等）通常最初都是 CSV 格式。创建转换作业是将这些 CSV 文件转换为更高效格式（如 Parquet）的一种方法。或者，你也可以添加投影；读取操作会使用优化后的投影以获得更好的性能，而数据集仍会保留原始的 CSV 文件。
