增量转换

警告

增量计算是一个高级功能。在使用此功能之前，请确保您了解用户指南的其余部分。

到目前为止，用户指南中显示的转换每次运行时都会重新计算其整个输出数据集。这可能导致大量不必要的工作。考虑以下示例：

2

9   @Compute
10   public void myComputeFunction(
11           @Input("/examples/students_hair_eye_color") FoundryInput myInput,
12           @Output("/examples/students_hair_eye_color_filtered") FoundryOutput myOutput) {
13       Dataset<Row> inputDf = myInput.asDataFrame().read();
14       myOutput.getDataFrameWriter(inputDf.filter("eye = 'Brown'")).write();
15   }
16}

如果向 /examples/students_hair_eye_color 输入数据集添加了新数据，则 filter() 将在整个输入上执行，而不是仅对添加到输入的新数据进行操作。这会浪费计算资源和时间。

如果一个转换可以了解其构建历史，它就可以更智能地计算其输出。更具体地说，它可以利用对输入所做的更改来修改输出数据集。在重新生成表时使用已经生成的数据的过程称为增量计算。没有增量计算，输出数据集总是会被转换的最新输出所替换。

编写一个增量转换

我们将逐步指导您如何使用transforms-java编写一个增量转换。与transforms-python相比，transforms-java 不使用注解来自动验证增量性并以增量方式应用转换。在 Java 中编写增量转换的过程由用户更直接地控制，用户可以明确决定在何种情况下转换应以增量方式执行，何时不应如此。通过解释输入数据集是如何被修改的，用户可以决定是采用增量方式还是快照方式更新输出数据集。

解释您的输入

第一步涉及对您输入的解释。输入数据集可以通过多种方式进行修改，我们只能在某些特定情况下应用增量转换。

  • APPENDED（追加）
  • UPDATED（更新）
  • NEW_VIEW（新视图）
  • UNCHANGED（未更改）

根据输入的变化情况，我们可以决定从输入数据集中读取什么以及写入输出数据集的内容。

读取输入

了解输入是如何被修改的，使我们能够相应地读取它。如果 事务仅追加了数据，我们可以确定可以安全地进行增量操作并只读取已修改的内容。如果相反，我们对输入数据集进行了更改，包括对现有行的修改，我们可能需要重新读取整个视图。 Transforms-Java API 通过 readForRange() 方法允许对输入数据集使用不同的读取模式。 ReadRange 暴露了可能的读取范围。不同的模式包括：

  • UNPROCESSED（未处理）
  • PROCESSED（已处理）
  • ENTIRE_VIEW（整个视图）

通过解释输入修改类型，我们可以决定如何读取我们的数据，如下面的示例所示。

Copied!
1private ReadRange getReadRange(FoundryInput input) {
2     switch (input.asDataFrame().modificationType()) {
3         case UNCHANGED:
4             LOG.info("No changes in input dataset, read only unprocessed");
5             return ReadRange.UNPROCESSED;
6         case APPENDED:
7             LOG.info("Append-only changes in input dataset, read only unprocessed");
8             return ReadRange.UNPROCESSED;
9         case UPDATED:
10             LOG.info("Update-type changes in input dataset, read entire view");
11             return ReadRange.ENTIRE_VIEW;
12         case NEW_VIEW:
13             LOG.info("New view in input dataset, read entire view");
14             return ReadRange.ENTIRE_VIEW;
15         default:
16             throw new IllegalArgumentException("Unknown ModificationType for input dataset "
17                 + input.asDataFrame().modificationType());
18     }
19}

然后我们可以根据需要修改我们的 compute 方法。

Copied!
1@Compute
2public void myComputeFunction(
3        @Input("/examples/students_hair_eye_color") FoundryInput myInput,
4        @Output("/examples/students_hair_eye_color_filtered") FoundryOutput myOutput) {
5    Dataset<Row> inputDf = myInput.asDataFrame().readForRange(getReadRange(myInput));
6    myOutput.getDataFrameWriter(inputDf.filter("eye = 'Brown'")).write();
7}
警告

此时我们只读取输入数据集的不同部分，而不会对输出数据集采取不同的操作。运行到目前为止的代码，无论你读取的是输入的哪一部分，都会在输出上产生一个快照事务。在应用增量转换之前，请继续完成教程的全部内容，以了解如何正确修改输出数据集。

转换数据

在此步骤中，用户需要根据需要应用数据的转换。请记住，根据输入的修改，读取的数据会有所不同。在我们的例子中，转换是一个简单的过滤器，用于筛选棕色眼睛的数据，我们可以将其隔离为：

Copied!
1inputDf = inputDf.filter("eye = 'Brown'");

写入输出

一旦我们解释了输入数据集中的修改，读取所需的输入部分并根据我们的转换逻辑转换数据，就可以相应地编写输出。 WriteMode 提供了不同的写入模式。不同的模式包括：

  • SNAPSHOT
  • UPDATE

例如，在我们的情况下，可以根据输入修改类型选择输出类型。

Copied!
1private WriteMode getWriteMode(FoundryInput input) {
2     switch (input.asDataFrame().modificationType()) {
3         case UNCHANGED:
4             LOG.info("No changes in input dataset, writing in update mode");
5             return WriteMode.UPDATE;
6         case APPENDED:
7             LOG.info("Append-only changes in input dataset, writing in update mode");
8             return WriteMode.UPDATE;
9         case UPDATED:
10             LOG.info("Update-type changes in input dataset, writing in snapshot mode");
11             return WriteMode.SNAPSHOT;
12         case NEW_VIEW:
13             LOG.info("new view in input dataset, writing in snapshot mode");
14             return WriteMode.SNAPSHOT;
15         default:
16             throw new IllegalArgumentException("Unknown ModificationType for input dataset " + input.asDataFrame().modificationType());
17     }
18 }
警告

不要混淆 WriteMode.UPDATE 和 DataFrameModificationType.UPDATED。前者会导致输出数据集的增量修改，这将导致下游数据集的 DataFrameModificationType.APPENDED。后者是对输入数据集的修改，包括追加和现有行的修改。

最后，write() 函数可以被修改以包含一个写入模式：

Copied!
1@Compute
2public void myComputeFunction(
3        @Input("/examples/students_hair_eye_color") FoundryInput myInput,
4        @Output("/examples/students_hair_eye_color_filtered") FoundryOutput myOutput) {
5    Dataset<Row> inputDf = myInput.asDataFrame().readForRange(getReadRange(myInput));
6    myOutput.getDataFrameWriter(inputDf.filter("eye = 'Brown'")).write(getWriteMode(myInput));
7}

最佳实践

在快照和增量之间切换

假设你希望主要运行增量转换，但有时需要重新运行你的数据集的快照。

为了避免手动硬编码所需的结果，当此输入被修改时，您可以使用SNAPSHOT写模式添加一个新输入，该输入将导致输出。这个新输入基本上会充当快照触发数据集。请注意，您需要根据这个新快照触发数据集的修改类型来调整转换的其他输入的读取范围。

也可以通过创建一个没有来源的空追加事务来外部强制进行快照。然而，transforms-java 不提供这种功能，因此超出了本指南的范围。

忽略增量删除

忽略模式变更

警告

请注意，输入数据集的模式修改与增量转换结合使用时可能会产生意外后果。

请阅读下面的所有文档，并在使用此功能前确保您理解所有潜在影响。

警告

仅在低级别转换中可以忽略模式变更。

如果增量构建所依赖的数据集的模式发生更改，该更改将导致DataFrameModificationType.NEW_VIEW，可能会破坏增量性。

但是，如果设置了IncrementalOptions.USE_SCHEMA_MODIFICATION_TYPE选项，模式更改不会导致新视图的生成。 相反，输入数据集中的模式更改将被解释为DataFrameModificationType.UNCHANGED，并将设置一个SchemaModificationType标志SchemaModificationType.NEW_SCHEMA，允许用户显式处理此特殊情况。

2        @UseIncrementalOptions(IncrementalOptions.USE_SCHEMA_MODIFICATION_TYPE)
3        public void myComputeFunction(
4               @Input("/Users/admin/students_data") FoundryInput myInput,
5               @Output("/Users/admin/students_data_filtered") FoundryOutput myOutput) {
6        ...
7        }

如果您手动注册了转换，请按照以下代码块中的方式将属性添加到构建器中。

Copied!
1    LowLevelTransform lowLevelManualTransform = LowLevelTransform.builder()
2               .computeFunctionInstance(new MyLowLevelManualFunction())
3               .putParameterToInputAlias("myInput", "/path/to/input/dataset")
4               .putParameterToOutputAlias("myOutput", "/path/to/output/dataset")
5               .useSchemaModificationType(true)
6               .build();

与转换相关的构建将成功或失败，这取决于转换如何依赖于输入数据集。 更具体地说，如果转换依赖于涉及模式更改的列，则对这些列的修改会使增量转换失败。 在这种情况下，需要一个新的快照才能再次使用增量转换。

转换 依赖 某个列的条件是：

  • 它包含明确依赖于该列的修改（例如，如果我们有filter("eye = 'Brown'")，并且在RAW数据集中的列eye被重命名或删除，则如果我们重新触发我们的FilterTransform，增量更新将失败）。
  • 修改后的列出现在输出数据集中（例如，如果我们从示例RAW数据集中删除列hair，我们的FilterTransform将失败）。

如果转换不依赖于模式更改，增量构建将成功。

例如，如果我们首先在转换中为id和eye添加一个select语句，并从RAW数据集触发初始快照构建，然后在RAW数据集中删除列hair，增量构建将成功，并且模式更改不会对增量转换产生任何影响。 如果模式有新增更改（例如，添加新列），构建也将始终成功。
