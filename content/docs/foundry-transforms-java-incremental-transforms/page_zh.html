增量转换

警告

增量计算是一项高级功能。在使用此功能之前，请确保您已充分理解用户指南的其余部分。

迄今为止，用户指南中展示的转换操作在每次运行时都会重新计算整个输出数据集。这可能导致大量不必要的工作。请参考以下示例：


```java
package myproject.datasets;

import com.palantir.transforms.lang.java.api.*;
import org.apache.spark.sql.Dataset;
import org.apache.spark.sql.Row;

public final class FilterTransform {

   @Compute
   public void myComputeFunction(
           @Input("/examples/students_hair_eye_color") FoundryInput myInput,
           @Output("/examples/students_hair_eye_color_filtered") FoundryOutput myOutput) {
       Dataset inputDf = myInput.asDataFrame().read();
       myOutput.getDataFrameWriter(inputDf.filter("eye = 'Brown'")).write();
   }
}
```

如果向 `/examples/students_hair_eye_color` 输入数据集中添加了任何新数据，`filter()` 会对整个输入执行操作，而不仅仅是针对新增数据。这在计算资源和时间上都是一种浪费。

如果转换逻辑能够感知其自身的构建历史，它就能更聪明地计算输出。更具体地说，它可以利用输入的变更来修改输出数据集。这种在重新物化表时利用已物化数据的过程，被称为*增量计算*。如果没有增量计算，输出数据集总是会被转换的最新输出所替换。

让我们回到上面展示的转换示例。该转换对 `students` 数据集执行 `filter()`，筛选出棕色头发的学生。有了增量计算后，如果两条新学生数据被追加到 `students` 中，该转换可以利用其构建历史信息，仅将新的棕色头发学生追加到输出中：

## 编写增量转换

我们将逐步指导您如何使用 *transforms-java* 编写增量转换。与 *transforms-python* 不同，*transforms-java* 不使用注解来自动验证增量性并应用增量转换。在 Java 中编写增量转换的过程由用户更直接地控制，用户可以明确决定在何种情况下转换应以增量方式执行，何时不应如此。通过解析输入数据集是如何被修改的，用户可以决定是以增量方式还是以类似快照的方式更新输出数据集。

### 解析输入

要采取的第一步是解析输入。输入数据集可以通过多种方式进行修改，仅在特定场景下我们才能应用增量转换。
DataFrameModificationType（或 FilesModificationType）表达了数据集修改的不同方式。各模式如下：


    

  • APPENDED
    • 

  • UPDATED
    • 

  • NEW_VIEW
    • 

  • UNCHANGED
    • 



根据输入的变化情况，我们可以决定从输入数据集中读取什么内容，以及向输出数据集写入什么内容。


读取输入


了解输入的修改方式后，我们就能以相应的方式进行读取。如果事务仅追加了数据，我们可以确信地采取增量处理，仅读取修改的部分。相反，如果输入数据集发生了包含修改已有行的变化，我们可能需要重新读取整个视图。Transforms-Java API 提供了 readForRange() 方法，支持对输入数据集采用不同的读取模式。
ReadRange 暴露了可能的读取范围。各模式如下：


    

  • UNPROCESSED
    • 

  • PROCESSED
    • 

  • ENTIRE_VIEW
    • 



通过解析输入修改类型，我们可以决定如何读取数据，如下例所示。




已复制！
1private ReadRange getReadRange(FoundryInput input) {
2     switch (input.asDataFrame().modificationType()) {
3         case UNCHANGED:
4             LOG.info("输入数据集无变化，仅读取未处理部分");
5             return ReadRange.UNPROCESSED;
6         case APPENDED:
7             LOG.info("输入数据集仅有追加更改，仅读取未处理部分");
8             return ReadRange.UNPROCESSED;
9         case UPDATED:
10             LOG.info("输入数据集有更新类型更改，读取整个视图");
11             return ReadRange.ENTIRE_VIEW;
12         case NEW_VIEW:
13             LOG.info("输入数据集有新视图，读取整个视图");
14             return ReadRange.ENTIRE_VIEW;
15         default:
16             throw new IllegalArgumentException("输入数据集的修改类型未知 "
17                 + input.asDataFrame().modificationType());
18     }
19}




We can then modify our compute method accordingly.

警告

一旦解析了输入数据集中的变更，读取了所需部分的输入内容，并按照转换逻辑处理完数据后，我们就可以相应地写出输出。WriteMode 提供了不同的写入模式。具体模式如下：


    

  • SNAPSHOT
    • 

  • UPDATE
    • 



例如，在我们的场景中，可以根据输入的变更类型来选择输出类型。

1private WriteMode getWriteMode(FoundryInput input) {
2     switch (input.asDataFrame().modificationType()) {
3         case UNCHANGED:
4             LOG.info("输入数据集无变更，以更新模式写入");
5             return WriteMode.UPDATE;
6         case APPENDED:
7             LOG.info("输入数据集仅有追加变更，以更新模式写入");
8             return WriteMode.UPDATE;
9         case UPDATED:
10             LOG.info("输入数据集为更新类变更，以快照模式写入");
11             return WriteMode.SNAPSHOT;
12         case NEW_VIEW:
13             LOG.info("输入数据集存在新视图，以快照模式写入");
14             return WriteMode.SNAPSHOT;
15         default:
16             throw new IllegalArgumentException("输入数据集的变更类型未知 " + input.asDataFrame().modificationType());
17     }
18 }



警告
请勿混淆 WriteMode.UPDATE 与
DataFrameModificationType.UPDATED。前者会导致输出数据集的增量修改，
并对下游数据集产生 DataFrameModificationType.APPENDED 的效果。
而后者是对输入数据集的修改，既包含追加操作，也包含对现有行的修改。


最后，可以修改 write() 函数以包含写入模式：

最佳实践


在快照与增量之间切换


假设您希望主要运行增量转换，但有时需要重新运行数据集的快照。


为避免手动硬编码预期结果，您可以添加一个新输入，使得每当该输入被修改时，输出便采用 SNAPSHOT 写入模式。这样，该新输入实际上就充当了快照触发数据集。请注意，您需要根据这个新的快照触发数据集的修改类型，调整转换其他输入的读取范围。


也可以通过创建一个不带溯源信息的空追加事务，从外部强制执行快照。不过，transforms-java 并未提供此类功能，因此这不在本指南的讨论范围之内。

忽略增量删除




如果增量构建依赖于无限增长的只追加数据集，且没有足够的磁盘空间来容纳这种增长，可能就需要删除上游数据集的部分内容。
然而，这可能会破坏增量机制，因为原始数据集的修改将不会产生 APPENDED 修改类型。
IncrementalOptions.IGNORE_INCREMENTAL_DELETES 可以避免这种情况，不会将上游数据集中的删除操作视为破坏性更改。

与转换相关的构建是成功还是失败，取决于转换对输入数据集的依赖方式。


具体而言，如果转换依赖于发生模式变更的列，那么对这些列的修改将导致增量转换失败。在这种情况下，必须先生成新的快照，才能继续使用增量转换。


在以下情况下，转换会依赖于特定列：


    

  • 转换中包含显式依赖该列的修改（例如，如果代码中使用了 filter("eye = 'Brown'")，而在 RAW 数据集中 eye 列被重命名或删除，那么当我们重新触发 FilterTransform 时，增量更新就会失败）。
    • 

  • 发生修改的列出现在了输出数据集中（例如，如果我们在示例 RAW 数据集中移除了 hair 列，我们的 FilterTransform 就会失败）。
    • 



如果转换不依赖于这些模式变更，增量构建将会成功。


例如，如果我们先在转换中添加针对 id 和 eye 的 select 语句，并基于 RAW 数据集触发初始快照构建，随后在 RAW 数据集中移除 hair 列，此时增量构建依然会成功，模式变更不会对增量转换产生任何影响。此外，在面对模式的新增型变更（例如添加新列）时，构建也始终能够成功。
