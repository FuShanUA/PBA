本体论角色迁移 [旧版]

旧版

本页描述了一个已过时的流程，不再是本体论资源权限设置的最新方法。现在可以使用 Compass文件系统来为本体论资源设置权限。要进行迁移，请 查看迁移指南。

本体论角色 是一种用于本体论资源的旧版授权模型，取代了 数据源派生权限。本体论角色适用于对象类型（object type）、关联类型（link type）和操作类型（action type），这些统称为“本体论资源”。为了使用本体论的最新功能，包括共享属性类型、接口等，必须使用本体论角色。默认情况下，自2023年9月以来创建的所有新租户都已使用本体论角色。

本体论角色允许您直接授予角色 直接 在每个本体论资源及其元数据上。所有元数据权限都在本体论管理器中进行管理。本体论角色允许您将本体论资源（如一个Aircraft对象类型）与这些资源的数据解耦，例如对象实例（如以本体论对象表示的具体飞机）和链接实例；这些数据实例仍由其输入数据源权限进行管理。了解更多关于本体论权限。

每个本体论都与一个或多个Foundry组织相关联。

角色权限在本体论角色文档中有详细说明。

所有迁移到本体论角色的对象类型、链接类型和操作类型都将对访问该特定本体论的所有用户拥有查看权限。因此，本体论资源信息对于该本体论的所有用户都是可访问的。

将本体论资源迁移到本体论角色没有明确的顺序要求，使得可以混合使用使用基于数据源的权限的对象类型。 ，并使用本体论角色。因此，为了此次迁移的目的，我们定义able to edit an object type如下：

  • 使用数据源派生权限的对象类型： 用户对该对象类型的输入数据源具有Editor权限。
  • 使用本体论角色的对象类型： 用户对该对象类型具有Ontology Editor权限，并不需要任何输入数据源的权限。用户应注意，这仅允许编辑本体论资源及其元数据，并不授予对数据/数据源本身的任何权限。对对象数据（不是元数据）的访问仍然由授予输入数据源的权限来管理。

迁移到本体论角色

如果您的注册启用了本体论角色，但您尚未将您拥有的本体论资源迁移到角色授权模型，您将在本体论管理器中看到一个提示您执行此操作的横幅。 或者，当查看本体论资源的表格时，您可以筛选出仍使用“与支撑数据源相同权限”的资源。

本体论角色尚未适用于所有Foundry注册。如果您没有访问本体论角色的权限，请联系Palantir支持。

前提条件

要执行迁移，用户必须具有以下权限：

  1. 对Foundry本体论进行更改的权限。

  2. 每个单独资源的以下权限：

    • 对于对象类型，用户必须在输入数据源上拥有Owner权限。
    • 对于一对多关联类型，用户不需要额外的权限。只需要在关联类型中引用的两种对象类型上具有Viewer权限，并且用户必须是该关联类型本身的Owner或在本体论级别上的Owner。
    • 对于多对多关联类型，用户还必须在关联类型的输入数据源上具有Owner权限。
    • 操作类型不需要额外的权限。

迁移至本体论角色后，修改操作类型需要额外的权限。您需要对操作类型所编辑的任何对象类型以及操作类型本身拥有权限，才能通过角色修改操作类型。

迁移通知

迁移到本体论角色是通过 升级助手 进行管理的

. 一个专用的升级助手任务会显示必须迁移的本体论资源列表，以及拥有执行迁移权限的分配者。从升级助手打开资源会将用户引导至 安全 标签页，在本体论管理器应用程序中迁移该资源。

如果用户有设置本体论角色的权限，尚未迁移的本体论资源将在 安全 标签页的迁移界面中出现，用于首次设置本体论角色。在本体论资源被迁移后，本体论角色选择器会在 安全 标签页下显示，并可以通过 Ontology Owners 进行更新。

使用本体论管理器进行迁移

有两种方式可以将本体论资源（对象类型、操作类型、关联类型）迁移到角色中： 批量迁移 本体论资源，或对所有本体论资源进行 逐个迁移。

本体论（Ontology）资源批量迁移

您可以将本体论（Ontology）资源批量迁移到角色（Role）。在继续之前，请参考以下< a href="#prerequisites">前提条件。您每次只能迁移500个资源。

要进行批量迁移，请进入本体论（Ontology）管理器中的高级，并在“迁移”部分选择继续使用迁移助手。

迁移向导将出现，并包含以下步骤。

  1. 选择资源：选择您希望应用相同角色（Role）的所有对象（Object）类型、关联（Link）类型或操作（Action）类型。下图显示了迁移向导中选择资源的界面：

  1. 相关资源：如果存在可以迁移的相关资源（关联（Link）类型或操作（Action）类型），则会显示此步骤。
  2. 分配角色： 您可以手动设置要分配这些对象类型的组/用户和默认角色。目前，角色迁移器无法批量建议角色。我们强烈建议在授予角色时使用用户组而不是用户。
  3. 摘要：查看摘要，确认迁移的影响，并完成迁移以应用角色。

当您确认迁移的影响时，您将确认从支撑数据源之前获得的权限将被新分配的角色授权覆盖，并且迁移是不可逆的。

批量迁移的例外情况

某些操作类型不能在没有进一步步骤的情况下进行批量迁移。

如果满足以下条件，操作类型不能迁移到角色：

  • 该操作类型没有检查当前用户权限的提交标准。
    • 解决方案：在本体论管理器（Ontology Manager）的“安全与提交标准”（Security & Submission criteria）选项卡中（在“提交标准”（Submission criteria）部分），根据当前用户添加一个条件，保存到本体论（Ontology）后重新尝试迁移。
  • 该操作类型由一个函数支持，在迁移前必须使用@edits decorators重新发布。
    • 解决方案：参阅 装饰器 文档中的步骤。

为了解决这些问题，请转到 操作类型 页面，并筛选出 与支撑数据源权限相同 的操作类型，这将显示由于问题未被迁移的操作类型。解决所有未决问题后，再对剩余的操作类型进行批量迁移。

警告

在迁移操作类型时，确保仅向那些对引用的对象类型和所有操作类型部署用例都有足够上下文的用户授予权限。

逐个迁移所有本体论资源

从本体论管理器主页，选择您要迁移的本体论资源（对象类型、关联类型或操作类型），并按Security setup到Same permissions as backing datasource以及按permissions = owner进行筛选。

选择要迁移的实体，然后在标题栏选择迁移到角色，或者在安全选项卡上选择开始使用角色。

迁移向导将显示两个建议的角色选项：数据源角色和本体论管理员和数据源角色。这些选项允许用户使用和配置与他们现有的本体论设置相一致的角色。选择其中一个选项会预填建议的角色列表；这些角色以后可以更改。

  • 数据源角色：此选项会添加所有在输入数据源上具有Editor或Owner角色的用户和用户组。
  • 本体论管理员和数据源角色：此选项会添加当前可以修改给定本体论资源的所有用户，基于 数据源派生权限。这些用户属于Ontology Administrators用户组，并且在输入数据源上具有编辑器或所有者角色。

例外情况

如果您遇到以下情况，可能无法尝试将操作类型迁移到角色：

  • 该操作类型没有提交标准来检查当前用户的权限。
    • 解决方案：在本体论管理器（提交标准部分）的动作类型中的安全与提交标准选项卡中根据当前用户添加一个条件，保存到本体论后重新尝试迁移。
  • 该动作类型由一个函数支撑，必须在迁移前使用@edits decorators重新发布。
    • 解决方案：参阅装饰器文档中的步骤。

当这些问题解决后，您就可以将动作类型迁移到角色中。

如果一些本体资源不再需要怎么办

如果您不再需要某个需要操作的本体资源，可以使用 本体清理工具 删除该资源。 . 该工具可根据一组标准（例如，数据源是否已删除、过期日期是否已过、索引是否失败、是否使用本体论角色等）帮助您识别哪些对象类型可以安全删除。

迁移基于函数的操作类型

基于函数的操作类型需要迁移支撑的函数。Foundry 将尝试自动迁移函数，使您可以同时迁移操作类型。然而，某些函数可能无法自动迁移，需要在函数中手动声明修改后的对象类型。如果在将操作类型迁移到本体论角色后，发现基于函数的操作出现故障，请查看 常见问题解答 部分。

迁移常见问题

对象感知的应用程序会有任何变化吗？

用户可能能够访问在 本体论管理器 中的对象类型和关联类型上的元数据，他们之前由于没有 Viewer 访问输入数据源而无法查看这些信息。这并不意味着他们将获得对输入数据源中的数据的访问权限，因为这些数据始终由这些数据源上的角色来管理。他们能够看到的元数据的具体内容由他们的 本体论角色 决定。

当对象类型迁移时，对象浏览器也将更新编辑对象视图的权限检查。在对象类型迁移之前，用户需要属于 Object View Administrators 组，并且拥有 Editor 访问输入数据源的权限才能编辑对象视图。在对象类型迁移之后，对象浏览器只会检查用户是否在该对象类型本身上具有 Ontology Editor 权限。
