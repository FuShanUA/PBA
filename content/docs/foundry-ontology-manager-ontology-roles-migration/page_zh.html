本体论角色迁移 [旧版]

旧版

本页描述了一个已过时的流程，不再是本体论资源权限设置的最新方法。现在可以使用 Compass 文件系统对本体论资源进行权限设置。要进行迁移，请 查看迁移指南。

本体论角色 是一种用于本体论资源的旧版授权模型，取代了 数据源派生权限。本体论角色适用于对象类型、链接类型和操作类型，这些统称为“本体论资源”。本体论角色需要使用本体论的最新功能，包括共享属性类型、接口等。默认情况下，自2023年9月起创建的所有新注册项都已使用本体论角色。

每个本体论都与一个或多个Foundry组织相关联。

角色权限详细信息请参见本体论角色文档。

所有迁移到本体论角色的对象类型、链接类型和操作类型都将对拥有该特定本体论访问权限的所有用户具有查看权限。因此，该本体论的资源信息对所有该本体论的用户都是可访问的。

将本体论资源迁移到本体论角色时不需要显式的顺序，这意味着可以同时存在使用数据源派生权限的对象类型和其他使用本体论角色的对象类型。因此，在此次迁移过程中，我们定义能够编辑一个对象类型如下：

  • 使用数据源派生权限的对象类型： 用户对该对象类型的输入数据源具有 Editor 权限。
  • 使用本体论角色的对象类型： 用户对该对象类型具有 Ontology Editor 权限，并不需要对输入数据源有任何权限。用户应注意，这仅允许编辑本体论资源及其元数据，不会授予对数据/数据源本身的任何权限。对对象 数据（而非 元数据）的访问仍由授予输入数据源的权限决定。

迁移到本体论角色

如果在您的注册中启用了本体论角色，但您尚未将您拥有的本体论资源迁移到角色授权模型，您将在本体论管理器中看到一个提示您进行迁移的横幅。 或者，当查看本体论资源的表格时，您可以筛选出仍然使用“与基础数据源相同权限”的您拥有的资源。

本体论角色尚未对所有Foundry注册用户开放。如果您无法访问本体论角色，请联系Palantir支持。

前提条件

要执行迁移，用户必须具有以下权限：

  1. 在Foundry本体论中进行更改的权限。

  2. 对每个单独资源的以下权限：

    • 对于对象类型，用户必须在输入数据源上拥有所有者权限。
    • 对于一对多链接类型，用户不需要额外权限。只需要在链接类型中引用的两个对象类型的查看者权限，并且用户必须是链接类型的所有者或在本体论级别上拥有所有者权限。
    • 对于多对多链接类型，用户还必须在链接类型的输入数据源上拥有所有者权限。
    • 操作类型不需要额外权限。

在迁移到本体论角色后，需要额外的权限才能修改操作类型。您需要对操作类型编辑的任何对象类型以及操作类型本身拥有权限，才能使用角色修改操作类型。

迁移通知

迁移到本体论角色是通过 升级助手 进行管理的。一个专用的升级助手任务会显示必须迁移的本体论资源列表，以及具有执行迁移权限的分配人员。从升级助手打开一个资源将引导用户到该资源的 安全 标签页，以便在本体论管理器应用程序中进行迁移。

如果用户有设置本体论角色的权限，尚未迁移的本体论资源将在 安全 标签页的迁移界面中显示，以首次为本体论资源设置本体论角色。本体论资源迁移后，本体论角色选择器会在 安全 标签页下显示，并可通过 本体论所有者 进行更新。

使用本体论管理器迁移

有两种方法可以将本体论资源（对象类型、操作类型、链接类型）迁移到角色中：本体论资源的批量迁移或 所有本体论资源的一次性迁移。

本体论资源的批量迁移

您可以将本体论资源批量迁移到角色中。在继续之前，请参考< a href="#prerequisites">先决条件列表。您一次只能迁移500个资源。

要进行批量迁移，请进入本体论管理器中的高级，并在“迁移”部分选择继续使用迁移助手。

    Exceptions to bulk migration

    Some action types cannot be bulk-migrated without further steps.

    Action types cannot be migrated to roles if:

    • The action type has no submission criteria that checks the current user’s permissions.
      • Solution: Add a condition based on the current user in the Security & Submission criteria tab of the Action type in the Ontology Manager (in the Submission criteria section) and save to the Ontology before re-attempting the migration.
    • The action type is backed by a function and must be republished with @edits decorators before the migration.
      • Solution: Refer to the steps in the decorators documentation.

    To address these issues, go to the Action Types page and filter down to Same permissions as backing datasource, which will display the action types that have not been migrated because of issues. Resolve all outstanding issues and then proceed with bulk migration again for any remaining action types.

    Warning

    When migrating action types, ensure that permissions are only granted to users with sufficient context on both the referenced object types and all use cases where the action type is deployed.

    One-by-one migration for all Ontology resources

    From the Ontology Manager home page, select the Ontology resources that you want to migrate (object type, link type, or action type) and filter by Security setup to Same permissions as backing datasource and by permissions = owner.

    Select the entity to migrate, then choose either Migrate to roles on the header or Start using roles on the Security tab.

    The migration wizard will appear with two suggested role options: Datasource roles and Ontology admins and datasource roles. These options allow users to use and configure roles that align with their existing Ontology setups. Selecting one of the options pre-fills the list of suggested roles; these roles can be changed later.

    • Datasource roles: This option adds all users and user groups with an Editor or Owner role on the input datasource.
    • Ontology admins and datasource roles: This option adds all users that can currently modify the given Ontology resource based on datasource-derived permissions. These users belong to the Ontology Administrators user group and have an Editor or Owner role on the input datasource.

    Exceptions

    What if some ontology resources are no longer needed

    If you no longer need an ontology resource that requires an action, you can delete the resource using the Ontology cleanup tool. This tool helps you identify which object types are safe to delete based on a set of criteria (for example, if the datasource is trashed, the deprecation date has passed, the index is failing, whether they are using Ontology roles or not, and so on.)

    Migrating function-backed action types

    Function-backed action types require the backing Function to be migrated. Foundry will attempt to migrate Functions automatically, allowing you to migrate the action type as well. Some Functions, however, might not be migrated automatically and require manually declaration of the edited object types in the Functions. Check the FAQ section if you see failures in your Function-backed actions after migrating an action type to Ontology roles.

    Migration frequently asked questions

    Will there be any changes in object-aware applications?

    Users may have access to metadata on object types and link types in the Ontology Manager that they could not previously view because they did not have Viewer access to the input datasources. This does not mean they will get access to the data in the input datasources which is always governed by the roles on those datasources. The specifics of what metadata they will be able to see is determined by their Ontology role.

    Object Explorer will also update the permission checks for editing Object Views when an object type is migrated. Before an object type is migrated, a user needs to be in the Object View Administrators group and have Editor access to the input datasources to edit Object Views. After an object type has been migrated, Object Explorer will only check that the user has Ontology Editor permissions on the object type itself.
