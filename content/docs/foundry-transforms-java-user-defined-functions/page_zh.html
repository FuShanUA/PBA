用户定义函数 (UDF)

Pipeline Builder 同时支持 Java 和 Python 用户定义函数 (UDF)。了解有关在 Pipeline Builder 中使用 Python UDF 的更多信息。

如果无法通过 Pipeline Builder 中现有的转换选项处理数据，或者需要引入外部 Java 或 Python 库，亦或是有需要在多个流水线中复用的复杂逻辑，你可以创建自己的用户定义函数 (UDF)。通过用户定义函数，你可以在 Pipeline Builder 或代码仓库中运行自定义的 Java 或 Python 代码，并对其进行版本控制与升级。

仅在确有必要时才应应用用户定义函数。建议尽可能使用 Pipeline Builder 或 Java 转换仓库中现有的转换功能。

租户环境必须部署于 Palantir 的云托管基础设施上才能使用 UDF。目前 Pipeline Builder 仅支持行映射 (row map) 和扁平映射 (flat map) 类型的 UDF，后续将增加对其他类型的支持。用户定义函数是一项高级功能。请务必仔细阅读以下文档，以充分了解在流水线中使用用户定义函数所带来的影响。

创建用户定义函数 (UDF)

要创建用户定义函数仓库，请先导航至你希望保存该仓库的项目。

接着，选择 New 并选择 Code Repositiory。在 Foundry UDF Definitions 选项下，选择 Map UDF Definition/Implementation，使用 UDF 模板初始化你的仓库。最后，选择 Initialize repository。

你可以选择在本地环境中进行开发（推荐），或直接在代码仓库中操作。

    只需在 src/main/resources/udfs/definitions 目录下添加新的定义文件，即可在此代码库中定义多个 UDF。

    实现行映射用户定义函数

    行映射 UDF 接收一行数据作为输入，每输入一行便精确输出一行。行映射是默认的 UDF 类型。

    要实现 UDF 创建和转换逻辑，请在 src/main/resources/java/myproject 下创建以下两个新的 Java 类文件：

    • YourUdfNameMapUdfFactoryImpl.java：负责创建 UDF 实例，并为其提供执行所需的参数。
      • 该类应实现生成的 YourUdfNameMapUdfFactory 接口。
      • 所有 UDF 参数均可通过 args.config() 参数获取。
      • 运行时将调用一次 create 方法来实例化 UDF。
    • YourUdfNameMapUdfImpl.java：实现从输入对象到输出对象的实际转换逻辑。
      • 该类应实现生成的 YourUdfNameMapUdf 接口。
      • 对于流经转换的每一行数据，都会调用一次此逻辑。
      • 输入和输出的类属性以及配置参数，可以通过同名方法进行访问，但方法名需采用驼峰命名法。例如，argument_one 对应的方法为 argumentOne()。

    代码库模板中包含了下列 Java 类的示例实现，你可以参考这些示例来编写自己的实现。编辑完成后，请重新运行代码生成，以确保没有问题。

    • src/main/java/myproject/MultiplyMapUdfFactoryImpl.java
    • src/main/java/myproject/MultiplyMapUdfImpl.java

    实现 flat map 用户定义函数

    Flat map UDF 接收一行数据作为输入，每行输入可以输出 0、1 或多行数据。

    要实现 flat map UDF 的创建和转换逻辑，请按照以下示例，在 src/main/resources/java/myproject 下创建两个新的 Java 类文件。

    YourUdfNameFlatMapUdfFactoryImpl.java：负责创建 UDF 实例，并为其提供执行所需的参数。

    • 该类应实现生成的 YourUdfNameFlatMapUdfFactory 接口。
    • 所有 UDF 参数均可通过 args.config() 参数获取。
    • 运行时将调用一次 create 方法来实例化 UDF。

    定义示例测试用例为可选项，但强烈建议执行此操作，因为它会显示在 Pipeline Builder 的转换文档下方，从而让用户更轻松地理解 UDF 的具体作用。若不想包含示例，请删除 src/test 目录并继续阅读下一节。

    部署用户定义函数

    您可以通过流水线构建器（推荐）或标准的 UDF 部署仓库来部署 UDF。

    发布至流水线构建器（推荐）

    目前，发布至流水线构建器仅适用于 Rubix 注册中的行映射和扁平映射 UDF。后续将增加对其他 UDF 类型的支持。

    请按照以下步骤将您的 UDF 发布至流水线构建器：

    1. 从底部面板打开 Task Runner。然后，选择 Java 标签页并输入 generateEddieLockfile，以生成一个包含唯一标识符的文件，供流水线构建器注册您仓库中的每个 UDF。

    generateEddieLockfile 命令会基于 UDF 名称返回一个随机的唯一标识符。如果在发布后重命名 UDF，代码将作为新的 UDF 发布，而不会覆盖之前的 UDF。同样，如果删除并重新生成锁定文件，UDF 将获得新的唯一标识符并注册为新的 UDF。在 Pipeline Builder 中导入 UDF 时弹出的窗口中，将显示该 UDF 的所有版本，包括旧名称、新名称以及新的锁定文件。

    1. 提交锁定文件及您的更改。
    2. 为您的发布版本打上标签（例如 0.0.1）。
    3. 验证检查是否通过。如果检查失败，UDF 将不会发布到 Pipeline Builder。

    如果不存在锁定文件，UDF 将不会注册到 Pipeline Builder 以便导入管道中。要生成锁定文件，请运行 generateEddieLockfile。如果您在 Code Repositories 中工作，则不需要锁定文件。

    1. 检查通过后，导航至当前或新的 Pipeline Builder 管道。
    2. 转到 Reusables > User-defined functions > Import UDF。
    3. 从列表中选择您的 UDF，然后选择 Add。您的 UDF 应出现在 Pipeline Builder 转换选择器中，并且可以像管道中的任何其他转换一样使用。

    要在 UDF 的第一个版本之后部署新的更改或修复（包括对定义 YML 的编辑），请重复上述实现和发布步骤。然后，在您的 Pipeline Builder 管道中：

    1. 转到 Reusables > User-defined functions。
    2. 为您的 UDF 选择 Edit version，并选择更新后的版本。

    发布到部署仓库

    请按照以下步骤将您的 UDF 发布到 Code Repositories：

    1. 选择 Commit 您的更改，并选择 Tag version 为您的发布设置版本。
    2. 标签检查通过后，使用 Foundry UDF Definitions > Foundry Streaming UDF Deployment 模板（用于流处理）或 Foundry UDF Definitions > Foundry Batch UDF Deployment 模板（用于批处理）创建另一个仓库。
    3. 按照模板中的说明部署 UDF。

    在首次版本的用户定义函数发布后，若要部署新的变更或修复（包括对定义 YAML 的编辑），请重复上述的实现与发布步骤。随后，更新部署仓库 build.gradle 依赖项中引用的标签版本。

    Schema 定义 YAML

    UDF schema 采用以下结构：

    
name: # (字符串) 模式的项目唯一名称，代码生成将使用它作为生成对象的前缀。
description: # (可选<字符串>) 此模式的描述，在 Pipeline Builder 中使用此 UDF 时将显示以提供上下文。
fields: # (列表) 此模式中的所有字段
  - name: # (字符串) 此字段的局部唯一名称
    nullable: # (布尔值) 该值是否可能为 null
    type: # (FieldType) 此字段的数据类型及其相关的元数据。例如：
      type: double
      double: {}


    如上所述，强烈建议您为每个模式添加描述说明，以便帮助 Pipeline Builder 的用户理解输入和输出的预期。

```yaml
type: string
string: {}
# 时间戳
type: timestamp
timestamp: {}
# 结构体
type: struct
struct:
  fields: [] # (列表)
```


    自定义类型
    

    UDF 中的自定义类型允许定义可在整个模式定义中重复引用的类型。
    

    

    已复制！
    1customTypes: # 该代码块为可选内容，若未使用自定义类型，则无需包含此部分。
2  types: # 切勿遗漏此 `types` 代码块！保留它是为了日后能在 `customTypes` 中支持 `imports` 等附加字段。
3    customType: # 该键名应为该自定义类型的唯一名称
4      # (FieldType) 该自定义字段的数据类型及其相关的元数据。例如：
5      type: double
6      double: {}
7    anotherCustomType:
8      # 定义自定义类型
9      # ...
    


    自定义类型可以定义为任何字段类型的别名，包括基本类型、数组、映射和结构体。然而通常情况下，自定义类型在定义贯穿于输入和输出模式中的重复结构体类型时最为有用。
    

    
customTypes:
  types:
    customStruct:
      type: struct
      struct:
        fields:
          - name: "doubleField"
            nullable: false
            type:
              type: double
              double: {}

自定义类型可以在模式中按如下方式引用：

```yaml
inputSchema:
  name: "Input"
  fields:
    - name: "inputStruct"
      nullable: false
      type:
        type: custom
        custom: customStruct # 上述自定义类型的名称
outputSchema:
  name: "Output"
  fields:
    - name: "outputStruct"
      nullable: false
      type:
        type: custom
        custom: customStruct # 上述自定义类型的名称
```

当 UDF 架构中引用了自定义结构体时（如上所述），代码生成机制并不会为 `InputStruct` 和 `OutputStruct` 创建独立的类。相反，输入对象和输出对象都会引用同一个针对所引用自定义类型（在下例中为 `CustomStruct`）的代码生成类：

```java
public final Output map(FoundryRowMapUdf.Context ctx, Input input) {
    // 一个空操作的 UDF，将输入结构体字段传递给输出结构体字段
    Output output = Output.create(ctx.getRowBuilderFactory());
    CustomStruct inputStruct = input.inputStruct(); // 返回一个 `CustomStruct`
    output.outputStruct(inputStruct); // 期望一个 `CustomStruct` 参数
    return output;
}
```


    使用自定义类型的一大优势在于支持自定义结构体；当输入与输出的代码生成类型在结构上一致时，无需进行任何额外的转换工作。

    能否取消注册 UDF？
    

    UDF 一旦发布，便无法从 Pipeline Builder 中取消发布。如果您删除了锁文件条目并在代码仓库中重新运行 generateEddieLockfile 命令，系统会为该 UDF 分配一个新的 ID，这将导致它在 Pipeline Builder 转换列表的 UDFs 部分中出现两次。
