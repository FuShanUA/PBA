样式指南

PySpark 是一种包装语言，允许你与 Apache Spark 后端进行交互以快速处理数据。Spark 可以在分布式服务器网络上处理非常大的数据集，当正确使用时，这能带来显著的性能和可靠性优势。即使对于有经验的 Python 开发者来说，它也带来了挑战，因为 PySpark 的语法借鉴了 Spark 的 JVM 传统，因此实现了可能不熟悉代码模式。

本 PySpark 代码风格的主观指南基于我们在 PySpark 仓库中遇到的最常见重复主题，提出了常见情况和相关最佳实践。

为了确保一致的代码风格，每个主仓库都应启用 Pylint ↗，并使用相同的配置。我们提供了一些 特定于 PySpark 的检查器 ↗，你可以将它们添加到你的 Pylint 中，以匹配本文档中列出的规则。请参阅 启用样式检查的文档 关于我们针对Python仓库的内置Pylint插件的详细信息。

除了PySpark特定内容外，编写干净代码的一般做法在PySpark仓库中也很重要 - Google PyGuide ↗ 是一个很好的起点。

优先使用隐式列选择而不是直接访问，除非是为了消除歧义

Copied!
1# bad
2df = df.select(F.lower(df1.colA), F.upper(df2.colB))
3
4# good
5df = df.select(F.lower(F.col("colA")), F.upper(F.col("colB")))

首选选项可能看起来更复杂、更长且更繁琐 - 这是正确的；事实上，最好完全避免使用F.col()。但在某些情况下，使用它或替代的显式选择是不可避免的。然而，有一个非常充分的理由，使第二例子优于第一个。

当像第一种情况那样使用显式列时，数据框名称和模式都明确绑定到数据框变量。这意味着如果 df1 被删除或重命名，引用 df1.colA 将会失败。

相比之下，F.col("colA") 总是引用正在操作的数据框中名为 “colA” 的列，而在这种情况下，该列的名称恰好是 df。它不需要跟踪其他数据框的状态，因此代码变得更加局部化，不易出现“远距离的幽灵交互”，而这些通常很难调试。

避免第一种情况的其他良好理由：

  • 如果数据框变量名很长，涉及它的表达式很快变得难以处理；
  • 如果列名包含空格或其他不支持的字符，需要通过方括号运算符访问，则 df1["colA"] 与 F.col(“colA”) 的编写难度相当；
  • 将像 F.col("prod_status") == 'Delivered' 这样的抽象表达式赋给变量，使其可以用于多个数据框，而 df.prod_status == 'Delivered' 始终绑定到 df

幸运的是，通常不需要使用带有 F.col() 的复杂表达式。唯一的例外是 F.lower、F.upper 等。以及这些 ↗。

注意事项

在某些情况下，可以访问多个数据框的列，并且可能会有名称重叠。一个常见的例子是在匹配表达式中，如 df.join(df2, on=(df.key == df2.key), how='left')。 在这种情况下，直接通过数据框引用列是可以的。您也可以使用数据框别名来消除连接的歧义（有关更多信息，请参阅本指南中的连接部分）。

避免遍历列，而采用列表推导

一般来说，在Spark中使用for循环效率较低。从高层次来看，这是因为Spark是惰性求值的，一次只能处理一个for循环。如果循环的所有部分都可以一次性处理，这可能会导致运行时间变慢，并可能导致Driver内存不足错误（OOM）。要将数据集中的所有列名从大写改为小写，而不是使用下面的第一个示例（标记为# bad），我们建议使用列表推导（如第二个示例，标记为# good）：

Copied!
1# bad
2for colm in df.columns:
3    df = df.withColumnRenamed(colm, colm.lower())
Copied!
1# good
2df = df.select(
3    *[F.col(colm).alias(colm.lower()) for colm in df.columns]
4)

使用列表推导式，如 # good 示例所示，可以避免上述讨论的性能缓慢和查询计划问题，同时仍获得相同的结果。

重构复杂的逻辑操作

逻辑操作通常位于 .filter() 或 F.when() 中，需要易于阅读。我们应用与链接函数相同的规则，将逻辑表达式保持在同一个代码块中，最多不超过 3 个表达式。如果它们变得更长，通常表明代码可以简化或提取出来。将复杂的逻辑操作提取到变量或函数中可以使代码更易于阅读和推理，这也可以减少错误。

Copied!
1# bad
2F.when( (df.prod_status == 'Delivered') | (((F.datediff(df.deliveryDate_actual, df.current_date) < 0) & 
3((df.currentRegistration.rlike('.+')) | ((F.datediff(df.deliveryDate_actual, df.current_date) < 0) & 
4(df.originalOperator.rlike('.+') | df.currentOperator.rlike('.+')))))), 'In Service')

我们可以用不同的方式简化上面的代码。首先，让我们专注于将逻辑步骤分组到几个命名变量中。Pyspark 要求表达式必须用括号包裹。这与实际用于分组逻辑运算的括号混合在一起可能会降低可读性。例如，上面的代码中有一个冗余的 (F.datediff(df.deliveryDate_actual, df.current_date) < 0)，原作者没有注意到，因为很难发现。 

Copied!
1# better
2has_operator = (df.originalOperator.rlike('.+') | df.currentOperator.rlike('.+'))
3delivery_date_passed = (F.datediff(df.deliveryDate_actual, df.current_date) < 0)
4has_registration = (df.currentRegistration.rlike('.+'))
5is_delivered = (df.prod_status == 'Delivered')
6
7F.when(is_delivered | (delivery_date_passed & (has_registration | has_operator)), 'In Service')

上面的例子更易于阅读，并且也删除了冗余的表达式。我们可以通过减少操作数量来进一步改进它。

Copied!
1# good
2has_operator = (df.originalOperator.rlike('.+') | df.currentOperator.rlike('.+'))
3delivery_date_passed = (F.datediff(df.deliveryDate_actual, df.current_date) < 0)
4has_registration = (df.currentRegistration.rlike('.+'))
5is_delivered = (df.prod_status == 'Delivered')
6is_active = (has_registration | has_operator)
7
8F.when(is_delivered | (delivery_date_passed & is_active), 'In Service')

请注意，现在 F.when 表达式简洁易读，期望的行为对任何审查此代码的人都很明确。如果读者怀疑某个地方有错误，只需查看单个表达式即可。如果您的代码中有单元测试，并希望将它们抽象为函数，那么每个逻辑块都很容易进行测试。

在最终示例中仍然有一些代码重复：如何消除这种重复，作为练习留给读者。

使用 select 语句来指定模式契约

在 PySpark 转换开始时进行 select，或在返回之前进行，是一种良好的做法。这个 select 语句会与 reader 和代码共同指定输入和输出的 dataframe 模式契约。

任何 select 都应被视为清理操作，为转换的下一步准备 dataframe。

始终力求使 select 语句尽可能简单。由于常见的 SQL 习惯用法，允许每个选定列最多使用一个来自 spark.sql.function 的函数，并且可以使用一个可选的 .alias() 来给它赋予有意义的名称。请注意，应谨慎使用此方法，如果在同一 select 中有超过 三个 这样的用法，则应将其重构为一个单独的函数，如 clean_<dataframe name>()，以封装该操作。

绝不允许在 select 中使用涉及多个 dataframe 的表达式，或者像 .when() 这样的条件操作。

Copied!
1# bad
2aircraft = aircraft.select(
3    'aircraft_id',
4    'aircraft_msn',
5    F.col('aircraft_registration').alias('registration'),
6    'aircraft_type',
7    F.avg('staleness').alias('avg_staleness'),
8    F.col('number_of_economy_seats').cast('long'),
9    F.avg('flight_hours').alias('avg_flight_hours'),
10    'operator_code',
11    F.col('number_of_business_seats').cast('long'),
12)

将相同类型的操作集中在一起。所有单独的列应该在前面列出，而对spark.sql.function中的函数调用应放在单独的行上。

Copied!
1# good
2aircraft = aircraft.select(
3    'aircraft_id', 'aircraft_msn', 'aircraft_type', 'operator_code',
4    F.col('aircraft_registration').alias('registration'),
5    F.col('number_of_economy_seats').cast('long'),
6    F.col('number_of_business_seats').cast('long'),
7    F.avg('staleness').alias('avg_staleness'),
8    F.avg('flight_hours').alias('avg_flight_hours'),
9)

select()语句本质上会重新定义数据框的模式，因此它自然支持包括或排除列，新旧列以及重新定义已有的列。通过在一个语句中集中所有这些操作，可以更容易地识别最终的模式，这使得代码更易读。它还使代码稍微简洁一些。

不要调用withColumnRenamed()，而是使用别名：

Copied!
1# bad
2df.select('key', 'comments').withColumnRenamed('comments', 'num_comments')
3
4# good
5df.select('key', F.col('comments').alias('num_comments'))

不要使用withColumn()来重新定义类型，在select中进行转换：

Copied!
1# bad
2df.select('comments').withColumn('comments', F.col('comments').cast('double'))
3
4# good
5df.select(F.col('comments').cast('double'))

但保持简单：

Copied!
1# bad
2df.select(
3    ((F.coalesce(F.unix_timestamp('closed_at'), F.unix_timestamp())
4    - F.unix_timestamp('created_at')) / 86400).alias('days_open')
5)
6
7# good
8df.withColumn(
9    'days_open',
10    (F.coalesce(F.unix_timestamp('closed_at'), F.unix_timestamp()) - F.unix_timestamp('created_at')) / 86400
11)

如果列将不会被使用，避免在select语句中包含它们，并选择显式的列集——这比使用.drop()更受推荐，因为它保证了模式更改不会导致意外的列增加，从而影响你的数据框。当然，不是所有情况下都禁止删除列；例如在连接之后经常是合适的，因为连接通常会引入冗余的列。

最后，而不是通过select语句添加新列，建议使用.withColumn()代替。

空列

如果您需要添加一个空列以满足模式，始终使用F.lit(None)来填充该列。不要使用空字符串或表示空值的其他字符串（例如NA）。

除了在语义上正确外，这样做的一个实际原因是保留使用像isNull这样的实用程序的能力，而不是必须验证空字符串、null和'NA'等。

Copied!
1# bad
2df = df.withColumn('foo', F.lit(''))
3
4# bad
5df = df.withColumn('foo', F.lit('NA'))
6
7# good - note necessary cast since `None` is typeless. Choose the appropriate type based on expected use.
8df = df.withColumn('foo', F.lit(None).cast('string'))

使用注释

虽然注释可以提供代码的有用见解，但通常对代码进行重构以提高其可读性更有价值。代码本身应该具有可读性。如果您使用注释来逐步解释逻辑，您应该对其进行重构。

Copied!
1# bad
2
3# Cast the timestamp columns
4cols = ["start_date", "delivery_date"]
5for c in cols:
6    df = df.withColumn(c, F.from_unixtime(df[c] / 1000).cast(TimestampType()))

在上面的例子中，我们可以看到这些列被转换为时间戳。注释并没有增加太多价值。此外，如果注释只提供了代码中已有的信息，那么更详细的注释可能仍然没有帮助。例如：

Copied!
1# bad
2
3# Go through each column, remove 1000 because millis and cast to timestamp
4cols = ["start_date", "delivery_date"]
5for c in cols:
6    df = df.withColumn(c, F.from_unixtime(df[c] / 1000).cast(TimestampType()))

与其留下仅仅描述你编写的逻辑的注释，你应该努力留下提供上下文的注释，解释你在编写代码时所做决定的“原因”。这对于PySpark尤其重要，因为读者可以理解你的代码，但通常并不了解输入到你的PySpark转换中的数据的上下文。一些小的逻辑可能涉及数小时的数据挖掘以理解正确的行为，在这种情况下，解释其理由的注释特别有价值。

Copied!
1# good
2
3# The consumer of this dataset expects a timestamp instead of a date, and we need
4# to adjust the time by 1000 because the original datasource is storing these as millis
5# even though the documentation says it's actually a date.
6cols = ["start_date", "delivery_date"]
7for c in cols:
8    df = df.withColumn(c, F.from_unixtime(df[c] / 1000).cast(TimestampType()))

UDFs（用户定义函数）

强烈建议在所有情况下都避免使用UDF，因为它们的性能远低于原生的PySpark。在大多数情况下，看似需要使用UDF的逻辑实际上可以重构为仅使用原生的PySpark函数。

Collect（以及相关函数）

始终避免使用将数据收集到Spark驱动程序中的函数，例如：

  • DataFrame.collect()
  • DataFrame.first()
  • DataFrame.head(...)
  • DataFrame.take(...)
  • DataFrame.show(...)

使用这些函数会消除像Spark这样的分布式框架的优势，导致性能下降或内存不足错误。我们强烈建议改用以下方式：

  • 在代码仓库中使用< a href="/docs/foundry/code-repositories/debug-transforms/">预览与调试功能，在开发转换时检查变量的状态（包括Spark DataFrames）。
  • 如果可能的话，使用原生的PySpark函数或UDF（在极少数情况下，当逻辑无法用原生函数编码时）来处理DataFrame中的map/aggregate值。

连接

小心处理连接。如果你执行一个左连接，且右侧对于某个键有多个匹配项，那么该行将被复制多次，次数等于匹配的次数。这被称为“连接爆炸”，可能导致数据集大小显著增加。始终要仔细检查你的假设，确保你进行连接的键是唯一的，除非你期望出现这种乘法效应。

错误的连接是许多问题的根源，这些问题可能很难调试。有一些方法可以有所帮助，例如明确指定how，即使你使用的是默认值（inner）：

Copied!
1# bad
2flights = flights.join(aircraft, 'aircraft_id')
3
4# also bad
5flights = flights.join(aircraft, 'aircraft_id', 'inner')
6
7# good
8flights = flights.join(aircraft, 'aircraft_id', how='inner')

还要避免right连接。如果你打算使用right连接，请交换数据框的顺序并改用left连接。这样做更直观，因为你在操作的数据框是你连接的中心。

Copied!
1# bad
2flights = aircraft.join(flights, 'aircraft_id', how='right')
3
4# good
5flights = flights.join(aircraft, 'aircraft_id', how='left')

在连接数据框时，避免使用会在输出中重复列的表达式：

Copied!
1# bad - column aircraft_id will be duplicated in the output
2output = flights.join(aircraft, flights.aircraft_id == aircraft.aircraft_id, how='inner')
3
4# good
5output = flights.join(aircraft, 'aircraft_id', how='inner')

避免重命名所有列以避免冲突。您可以只给整个数据框赋予一个别名，并使用该别名在最后选择您想要的列。

Copied!
1# bad
2columns = ["start_time", "end_time", "idle_time", "total_time"]
3for col in columns:
4    flights = flights.withColumnRenamed(col, 'flights_' + col)
5    parking = parking.withColumnRenamed(col, 'parking_' + col)
6
7flights = flights.join(parking, on="flight_code", how="left")
8
9flights = flights.select(
10    F.col("flights_start_time").alias("flight_start_time"),
11    F.col("flights_end_time").alias("flight_end_time"),
12    F.col("parking_total_time").alias("client_parking_total_time")
13)
14
15# good
16flights = flights.alias("flights")
17parking = parking.alias("parking")
18
19flights = flights.join(parking, on="flight_code", how="left")
20
21flights = flights.select(
22    F.col("flights.start_time").alias("flight_start_time"),
23    F.col("flights.end_time").alias("flight_end_time"),
24    F.col("parking.total_time").alias("client_parking_total_time")
25)

然而，在这种情况下，请记住：

  • 如果您不需要两个列，最好在连接之前删除重叠的列；
  • 如果您确实需要两者，最好在连接之前重命名其中一个；
  • 在输出数据集之前，您应该始终解决歧义列，因为在转换完成后，您将无法区分它们。

关于连接的最后一句话，不要使用 .dropDuplicates() 或 .distinct() 作为捷径。如果观察到意外的重复行，几乎总是有根本原因导致这些重复行出现。添加 .dropDuplicates() 只是掩盖了这个问题，并增加了运行时的开销。

表达式的链式调用

链接表达式是一个有争议的话题，但我们建议对链接的使用施加一些限制。有关此建议背后原理的讨论，请参阅本节的 结论。

避免将不同类型的表达式链接成多行表达式。特别是如果它们有不同的行为或上下文时。例如，不要将列创建或连接与选择和过滤混合在一起。

Copied!
1# bad
2df = (
3    df
4    .select("a", "b", "c", "key")
5    .filter(df.a == "truthiness")
6    .withColumn("boverc", df.b / df.c)
7    .join(df2, "key", how="inner")
8    .join(df3, "key", how="left")
9    .drop('c')
10)
11
12# better (separating into steps)
13# first: we select and trim down the data that we need
14# second: we create the columns that we need to have
15# third: joining with other dataframes
16
17df = (
18    df
19    .select("a", "b", "c", "key")
20    .filter(df.a == "truthiness")
21)
22
23df = df.withColumn("boverc", df.b / df.c)
24
25df = (
26    df
27    .join(df2, "key", how="inner")
28    .join(df3, "key", how="left")
29    .drop('c')
30)

将每个表达式组隔离到其自身的逻辑代码块中可以提高可读性，并使找到相关逻辑更加容易。

例如，代码下面的读者可能会跳转到他们看到数据框被赋值的地方 df = df...。

Copied!
1# bad
2df = (
3    df
4    .select('foo', 'bar', 'foobar', 'abc')
5    .filter(df.abc == 123)
6    .join(another_table, 'some_field')
7)
8
9# better
10df = (
11    df
12    .select('foo', 'bar', 'foobar', 'abc')
13    .filter(F.col('abc') == 123)
14)
15
16df = df.join(another_table, 'some_field', how='inner')

将表达式链接在一起有正当的理由。这些通常代表原子逻辑步骤，是可以接受的。应用一条规则，即在同一块中最多允许一定数量的链接表达式，以保持代码的可读性。我们建议链接不超过3-5条语句。

如果你发现你正在创建更长的链接，或者由于变量的大小而遇到困难，考虑将逻辑提取到一个单独的函数中：

Copied!
1# bad
2customers_with_shipping_address = (
3    customers_with_shipping_address
4    .select("a", "b", "c", "key")
5    .filter(F.col("a") == "truthiness")
6    .withColumn("boverc", F.col("b") / F.col("c"))
7    .join(df2, "key", how="inner")
8)
9
10# also bad
11customers_with_shipping_address = customers_with_shipping_address.select("a", "b", "c", "key")
12customers_with_shipping_address = customers_with_shipping_address.filter(df.a == "truthiness")
13
14customers_with_shipping_address = customers_with_shipping_address.withColumn("boverc", F.col("b") / F.col("c"))
15
16customers_with_shipping_address = customers_with_shipping_address.join(df2, "key", how="inner")
17
18# better
19def join_customers_with_shipping_address(customers, df_to_join):
20
21    customers = (
22        customers
23        .select("a", "b", "c", "key")
24        .filter(df.a == "truthiness")
25    )
26
27    customers = customers.withColumn("boverc", F.col("b") / F.col("c"))
28    customers = customers.join(df_to_join, "key", how="inner")
29    return customers

事实上，超过三个语句的链式结构非常适合分解为单独的、命名良好的函数，因为它们已经是封装好的、隔离的逻辑块。

理由

这些链式限制背后有几个原因：

  • 区分 PySpark 代码和 SQL 代码。链式操作与大多数（如果不是全部的话）其他 Python 风格相违背。你在 Python 中不会链式调用，而是进行赋值。
  • 鼓励避免创建大的单一代码块。这些代码块通常更适合作为命名函数提取出来。
  • 这并不需要非此即彼；三到五行的链式操作最大值在实用性和可读性之间取得了平衡。
  • 如果你使用的是 IDE，这使得使用自动提取或执行代码移动更容易（例如，在 PyCharm 中使用 Ctrl/Cmd+Shift+Up）。
  • 长链式结构难以阅读和维护，特别是当链式结构嵌套时。

多行表达式

你可以链接表达式的原因是PySpark是从Spark开发的，而Spark来源于JVM语言。这意味着一些设计模式被带入了，即链式调用。

然而，Python不支持优雅地处理多行表达式，唯一的替代方法是提供显式的换行符，或者将表达式括在括号中。只有当链式操作发生在根节点时，才需要提供显式的换行符。例如：

Copied!
1# needs `\`
2df = df.filter(F.col('event') == 'executing')\
3       .filter(F.col('has_tests') == True)\
4       .drop('has_tests')
5
6# chain not in root node so it doesn't need the `\`
7df = df.withColumn('safety', F.when(F.col('has_tests') == True, 'is safe')
8                              .when(F.col('has_executed') == True, 'no tests but runs')
9                              .otherwise('not safe'))

因此为了保持一致性，将整个表达式放入一个括号块中，并避免使用 \ :

Copied!
1# bad
2df = df.filter(F.col('event') == 'executing')\
3    .filter(F.col('has_tests') == True)\
4    .drop('has_tests')
5
6# good
7df = (
8  df
9  .filter(F.col('event') == 'executing')
10  .filter(F.col('has_tests') == True)
11  .drop('has_tests')
12)

其他注意事项和建议

  • 注意不要让函数变得太大。一般来说，文件不应超过250行，函数不应超过70行。
  • 尽量将你的代码保持在逻辑块中。例如，如果你有多个引用相同内容的行，尽量将它们放在一起。将它们分开会使读者更难获得上下文。
  • 测试你的代码。如果你可以运行本地测试，请这样做，并确保你的新代码被测试覆盖。如果你无法运行本地测试，请在你的分支上构建数据集，并手动验证数据是否符合预期。
  • 避免作为通用后备方案使用.otherwise(value)。如果你正在将一组键映射到一组值，并且出现了一些未知的键，使用otherwise会将所有这些键隐藏为一个值。
  • 不要将注释掉的代码保留在仓库中。这适用于单行代码、函数、类或模块。应依靠git及其分支或查看历史记录的功能。
  • 当遇到由多个不同源表集成组成的大型单一转换时，应将其拆分为自然的子步骤，并将逻辑提取到函数中。这有助于提高上层代码的可读性，并实现转换之间的代码重用性和一致性。
  • 在命名函数或变量时，应尽可能明确和描述性。除了顶层转换封装对象转换外，应努力捕捉函数实际执行的操作，而不是仅仅根据其中使用的对象进行命名。
  • 在引入新的导入别名时要三思而后行，除非有充分的理由。例如，如果一个导入的模块在不同的文件中被多次调用，并且该模块对几乎所有开发人员来说都是常见且熟悉的，那么这可能是合理的。一些已建立的别名包括types和functions，来自pySpark from pyspark.sql import types as T, functions as F。
  • 避免在过滤条件、新列值等中使用字面字符串或整数。相反，应将它们提取到变量、常量、字典或类中，以捕捉其含义。这使代码更易读，并在仓库中强制执行一致性。
