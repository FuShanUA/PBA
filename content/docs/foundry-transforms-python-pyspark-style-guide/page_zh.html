风格指南

PySpark 是一种包装语言，允许你与 Apache Spark 后端进行交互以快速处理数据。Spark 能够在分布式服务器网络中处理超大规模数据集，只要使用得当，就能在性能和可靠性上带来显著优势。然而，即便是经验丰富的 Python 开发者也会面临挑战，因为 PySpark 语法继承了 Spark 的 JVM 基因，因此实现了一些可能令人感到陌生的代码模式。

这份带有主观立场的 PySpark 代码风格指南，基于我们在各类 PySpark 代码库中遇到的最常见高频问题，总结了相应的场景与最佳实践。

为了贯彻一致的代码风格，每个主代码库都应启用 Pylint，并采用相同的配置。我们额外提供了一些针对 PySpark 的检查器，你可以将其集成到 Pylint 中，以匹配本文档所列规则。有关 Python 代码库内置 Pylint 插件的详情，请参阅关于启用风格检查的文档。

除了 PySpark 的特性之外，整洁代码的通用实践在 PySpark 代码库中同样重要——Google 的 PyGuide 是一个不错的起点。

首选方案可能看起来更复杂、冗长且杂乱——但这很正常；实际上，最好完全避免使用 F.col()。但在某些情况下，使用它或替代的显式选择是不可避免的。不过，有非常充分的理由倾向于选择第二种示例而非第一种。

在第一种情况下使用显式列时，dataframe 的名称和 schema 都会显式绑定到 dataframe 变量上。这意味着，如果 df1 被删除或重命名，引用 df1.colA 就会失效。

相比之下，F.col("colA") 将始终引用正在操作的 dataframe 中名为 “colA” 的列（在此例中恰好名为 df）。它完全不需要跟踪其他 dataframe 的状态，因此代码变得更加局部化，也不容易出现“远距离的神秘交互”，这类问题往往难以调试。

避免使用第一种情况的其他充分理由：

  • 如果 dataframe 变量名很长，涉及它的表达式很快就会变得难以处理；
  • 如果列名包含空格或其他不受支持的字符，需要通过中括号运算符来访问，那么写 df1["colA"] 和写 F.col(“colA”) 一样麻烦；
  • 将诸如 F.col("prod_status") == 'Delivered' 的抽象表达式赋值给变量，可以使其在多个 dataframe 中复用，而 df.prod_status == 'Delivered' 始终绑定在 df 上。

幸运的是，通常不需要使用 F.col() 构造复杂的表达式。唯一的例外是 F.lower、F.upper……以及这些 ↗。

注意事项

在某些场景下，可以访问多个 dataframe 中的列，并且名称可能会重叠。一个常见的例子是匹配表达式，如 df.join(df2, on=(df.key == df2.key), how='left'）。在这种情况下，直接通过 dataframe 引用列是没问题的。你也可以使用 dataframe 别名来消除 join 的歧义（详情请参阅本指南中的 Joins 部分）。

避免遍历列，推荐使用列表推导式

通常来说，Spark 中的 for 循环效率较低。宏观上看，原因在于 Spark 采用惰性求值，每次只会处理 for 循环中的一个迭代。如果循环中的所有部分都能一次性处理，这种机制可能会导致运行时间变慢，并可能引发 Driver 内存溢出错误（OOM）。为了将数据集中所有列名从大写改为小写，建议使用列表推导式（如第二个标有 # good 的示例所示），而不是采用下方第一个示例（标有 # bad）的写法：

已复制！

使用 select 语句指定模式契约

在 PySpark 转换操作的开头或返回结果前执行 select，被视为一种最佳实践。这条 select 语句向读者和代码明确约定了输入与输出预期的 DataFrame 模式。

任何 select 操作都应被视为一种数据清洗过程，旨在为转换流程的下一步消费准备好 DataFrame。

应始终力求保持 select 语句尽可能简单。按照常见的 SQL 习惯用法，允许每个选定列最多使用 一个 来自 spark.sql.function 的函数，并可附加一个可选的 .alias() 来赋予其有意义的名称。请切记，这种用法应当克制，如果同一个 select 中此类用法超过 三个，请将其重构为如 clean_<dataframe name>() 的独立函数以封装该操作。

绝不允许在 select 中使用涉及多个 DataFrame 的表达式，或是像 .when() 这样的条件操作。

UDF（用户定义函数）

强烈建议在所有情况下都避免使用 UDF，因为其性能远低于原生 PySpark。在大多数场景下，看似必须使用 UDF 的逻辑，实际上都可以重构为仅使用原生 PySpark 函数来实现。

Collect（及相关函数）

务必避免使用将数据收集到 Spark 驱动程序中的函数，例如：

  • DataFrame.collect()
  • DataFrame.first()
  • DataFrame.head(...)
  • DataFrame.take(...)
  • DataFrame.show(...)

使用这些函数会抵消 Spark 等分布式框架的优势，导致性能下降或引发内存溢出错误。强烈建议不要使用这些函数，而是采用以下替代方案：

  • 在开发数据转换逻辑时，使用代码仓库（Code Repository）中的预览与调试功能来检查变量（包括 Spark DataFrame）的状态。
  • 使用原生 PySpark 函数（如有可能）或 UDF（在极少数原生函数无法表达逻辑的情况下），来处理 DataFrame 中的映射/聚合值。

关联操作

执行关联操作时需谨慎。如果执行左关联，且右侧针对某个键存在多条匹配记录，那么该行数据将按匹配次数被复制多份。这种现象被称为“关联爆炸”，可能导致数据集规模急剧膨胀。务必反复核实你的假设，确认关联键是唯一的，除非你本身就期望这种数据乘积效应。

不当的关联操作是引发诸多棘手调试问题的根源。一些做法有助于规避此类问题，例如显式指定 how 参数，即使你使用的是默认值（inner）：

设计考量

对链式调用施加这些限制主要出于以下几个原因：

  • 区分 PySpark 代码与 SQL 代码。链式调用这种写法在绝大多数（甚至所有）其他 Python 风格规范中都格格不入。在 Python 中我们不做链式调用，而是进行赋值操作。
  • 避免构建庞大的单一代码块。这类代码块通常更适合提取出来作为一个具名函数。
  • 这并非非黑即白的规定；将链式调用的最大长度限制在三到五行之间，能在实用性与可读性之间取得平衡。
  • 如果使用集成开发环境（IDE），这种限制更便于执行自动提取操作或进行代码移动（例如在 PyCharm 中使用 Ctrl/Cmd+Shift+Up）。
  • 过长的链式调用难以阅读和维护，尤其是在存在嵌套链式调用的情况下。

多行表达式

之所以能够进行链式调用，是因为 PySpark 脱胎于 Spark，而 Spark 源自 JVM 语言。这意味着它沿袭了一些 JVM 语言的设计模式，即链式调用能力。

其他考虑与建议

  • 警惕函数体积膨胀。通常情况下，单个文件不应超过 250 行，单个函数不应超过 70 行。
  • 尽量将代码保持在逻辑块内。例如，如果多行代码引用了相同的内容，应将它们放在一起。拆分这些代码会增加读者获取上下文的难度。
  • 测试你的代码。如果能运行本地测试，请执行并确保新代码被测试覆盖。如果无法运行本地测试，请在你所在的分支上构建数据集，并手动验证数据是否符合预期。
  • 避免将 .otherwise(value) 作为通用的兜底方式。如果你正在将一组键映射为一组值，并且出现了未知的键，使用 otherwise 会把所有这些未知键都掩盖成一个值。
  • 不要将注释掉的代码保留在代码仓库中。这适用于单行代码、函数、类或模块。请依靠 git 及其分支或查看历史记录的功能来替代这一做法。
  • 遇到由整合多个不同源表组成的单一大型转换时，应将其拆分为自然的子步骤，并将逻辑提取为函数。这有助于提升顶层代码的可读性，并实现代码复用以及不同转换之间的一致性。
  • 命名函数或变量时，尽量做到明确且具有描述性。除了包裹对象转换的顶层转换之外，应努力体现函数的实际行为，而不是仅用内部使用的对象来命名。
  • 引入新的导入别名要三思，除非有充分的理由。例如，如果某个导入的模块在多个不同文件中被频繁调用，且该模块为几乎所有开发者所熟知，这样做可能是合理的。一些已有的约定包括 pySpark 中的 types 和 functions：from pyspark.sql import types as T, functions as F。
  • 避免在过滤条件、列的新值等场景中使用字面量字符串或整数。应视情况将其提取为变量、常量、字典或类，以明确其含义。这样可以提升代码的可读性，并在整个代码库中保持一致性。
