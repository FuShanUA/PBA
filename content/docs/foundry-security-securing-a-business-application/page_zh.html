保障业务应用的安全

在建立数据基础之后，你可以开始构建本体，并在此基础上开发一个“航班告警收件箱”应用。这样做的目的是为一线操作人员提供有价值的工作流，同时确保安全机制清晰且易于维护。

对象数据继承数据集权限

底层支撑数据完成转换后，你可以在本体中创建四个对象：Flight、Flight Alert、Delay 和 Aircraft。为此，请创建新的对象类型，并为每个对象选择已经构建好的底层数据集。在此过程中，你需要将数据集中的列映射为新的对象类型中的属性。底层数据集中的每一列都将成为新对象类型中的一个属性，而每一行则将成为该对象类型的一个实例。因此，对象类型内部的数据将受到底层数据集权限的保护。

对于 Aircraft 对象，其数据受 /Sky Industries/Aviation [Ontology]/aircraft 数据集权限的保护。因此，Aviation [Ontology] 项目中 Aviation [Ontology] - Viewer 组的用户将能够查看 Aircraft 对象中的数据。

创建完所需的对象后，你可以根据对象之间的关联方式，在对象之间创建所需的链接类型。以下是我们构建的本体结构。

为了完整支撑你的工作流，你可能需要为部分对象添加回写功能。回写功能允许编辑现有的属性值（例如，修正错误值或更新状态属性），但更重要的是，它允许记录新的数据和决策，从而实现价值的持续积累。

为此，只需在编辑器部分生成一个回写数据集。我们建议将编辑后的数据集与底层数据集放置在同一个本体项目中（例如 Aviation [Ontology]），以确保对象上的权限统一。编辑对象的推荐方式是使用操作，你也可以通过 API 调用这些操作。以下是带有回写数据集的 Flight Alert 对象。

授予用户访问本体数据的权限，将完全由你在数据基础中创建并应用的组来管理。例如，你只需将新用户添加到 Aviation [Ontology] - Viewer 组中，他们就能查看全部四个对象的数据：Flight、Flight Alert、Delay 和 Aircraft。

编辑本体

你可以通过管理资源所在项目的权限，来控制谁能够编辑特定本体类型的配置。对本体内资源（如对象类型、链接类型和操作类型）的访问权限，取决于谁有权访问 Compass 文件系统中的相关项目。例如，若要用户在本体中编辑 Flight Alert 对象，该用户只需对包含 Flight Alert 资源的项目具有适当的访问权限即可；无需额外的权限或角色。你可以在每个项目的安全部分管理这些权限。

向用户和/或组授予对象类型上的角色，仅赋予他们对对象类型定义及其元数据的权限，并不授予对数据本身的任何权限。数据的权限由底层数据源的权限管理。

限制对象实例

后续我们决定整合旅客数据，于是便着手接入并搭建旅客数据管道。

出于合规性考虑，我们可能希望运营用户只能查看其所在国家的乘客数据（美国用户只能查看美国乘客，德国用户只能查看德国乘客，以此类推）。由于我们的对象类型由底层数据集支撑，其权限与整个数据集绑定。若要限制单个数据行的权限，我们需要使用受限视图来支撑对象类型，受限视图是对单一数据集的行级权限视图。

创建受限视图时，必须先完成以下前置步骤：

  1. 创建全局用户组 passenger-data-global-access，以便为所有管理用户授予全局乘客数据的访问权限。
  2. 确认用户已具备必要的地理用户属性（例如 location:country - US）
  3. 为受限视图创建一个名为 Customer Information [Restricted] 的项目，用于存放当前及未来所有与客户信息相关的受限视图。这样我们就能将受限视图的访问权限与底层输入数据集分离开来单独管理。

用户属性可以在用户管理界面中手动配置，也可以通过 SAML 设置自动映射。

完成上述前置条件后，我们就可以创建受限视图了。导航至输入数据集资源，选择 Create restricted view，如下图高亮所示。

受限视图向导将引导你完成所有步骤。将受限视图保存到我们之前创建的新项目 Customer Information [Restricted] 中。

策略是受限视图的核心部分，它决定了用户能够查看哪些数据行。在本例中，我们希望实现：

将数据添加至数据目录

数据目录是 Foundry 中精选内容的浏览视图。尽管在 Foundry 中的每个数据集都可以被分析（前提是你拥有正确的权限），但数据目录中收录的数据集被认定为对用户最有用或最相关的数据，并集中于此以便于访问。

构建本体后，我们建议将所有本体数据集添加到数据目录中。特别是本体回写数据集，由于它们是对象数据的权威来源，更应该被添加到你的数据目录中。

授予用户应用访问权限

在构建完 Flight、Flight Alert、Delay、Aircraft 和 Passenger 对象后，我们参照 Workshop 文档，为运营中心构建了 Flight Alert Inbox 应用。为了将能够编辑 Flight Alert Inbox 应用的用户与能够编辑本体数据的用户区分开来，我们将专门为 Flight Alert Inbox 应用创建一个新项目，名为 Flight Alert Inbox。

用户必须满足所有访问要求才能访问该项目。由于该项目未关联任何标记，用户需要具备相应的角色，并且必须是 Sky Industries 组织的成员。

运营消费者将被添加到 Aviation [Ontology] - Viewer 用户组，以查看 Flight、Flight Alert、Delay、Aircraft 和 Passenger 数据；同时被添加到 Flight Alert Inbox - Viewer 用户组，以查看 Flight Alert Inbox 应用。

应用开发人员将被加入 航空 [本体] - 查看者 组以查看 Flight、Flight Alert、Delay、Aircraft 和 Passenger 数据，并被加入 航班告警收件箱 - 编辑者 组以编辑航班告警收件箱应用。

数据管道开发人员将被加入 航空 [本体] - 编辑者 组以编辑 Flight、Flight Alert、Delay、Aircraft 和 Passenger 数据集，并被加入 航班告警收件箱 - 查看者 组以查看其变更下游的工作流。
