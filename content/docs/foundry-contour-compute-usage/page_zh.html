Contour 的计算资源使用

在 Foundry 中基于数据运行 Contour 工作流需要使用 Foundry 的计算资源，该资源以“计算秒”为单位进行计量。本文档详细说明了 Contour 如何使用计算资源，并提供了在产品中排查和管理计算资源使用情况的相关信息。

每个展示数据的 Contour 看板背后，都有对 Contour 后端发出的查询请求作为支撑。Foundry 通过在自动伸缩的 Spark 集群上执行这些查询来响应请求。Contour 查询的计算秒数，完全根据查询实际执行期间所消耗的计算秒数来计量。然而，在交互模式下，为了提供更快的响应速度，所有 Contour 分析的计算资源均处于“常开”状态；这意味着每个交互式查询消耗计算秒数的速率要高于批量计算。

查询所消耗的资源量受多种因素影响，例如输入数据集的规模与布局，以及查询本身的复杂度（如所使用的 Contour 看板数量）。

计算资源的计量方式

Contour 借助 Foundry 的并行计算后端来执行针对 Foundry 数据的查询。因此，Contour 中的计算资源用量按以下公式进行计量。

请注意，默认的 interactive_contour_usage_rate 为 15。这是 Contour 基于常开的并行计算框架消耗计算资源的速率。如果您与 Palantir 签订了企业级合同，请在进行计算资源使用量核算前联系您的 Palantir 代表。

total_compute_seconds = contour_usage_rate * (driver_compute_seconds + executor_compute_seconds)

交互式 Contour 用量

Contour 的“常备”执行系统专为快速响应终端用户发起的即席查询而设计。在 Foundry 计算秒数维度上，单次 Contour 查询的成本取决于在 Spark 集群上执行该查询所需的时间，以及查询执行期间所占用的核心数量。核心数量则是根据查询在调度它的 Spark 应用中所分摊的资源份额来确定的。

在交互模式下，每次创建或更新 Contour 看板都会消耗计算资源。这包括会产生“级联”效应的计算消耗，即影响 Contour 路径中特定看板下游的其他看板。例如，在 Contour 分析中，以下操作均会消耗计算资源：

  • 创建新看板（可能会影响所有下游看板）
  • 更新看板（可能会影响所有下游看板）
  • 点击路径顶部的更新数据

批量 Contour 用量

Contour 路径可以“另存为”数据集。此操作会将 Contour 后端生成的底层代码保存到数据集作业规范中。随后，可以按计划或以即席方式运行该数据集构建任务。这类数据集作业将作为批量作业进行计量，不会像交互式 Contour 用量那样产生“常开”开销。

Contour 批量数据集作为通用的并行批量计算进行计量，类似于通过 Code Repositories 和 Code Workbook 创建及更新的数据集。这些批量计算与其他批量计算消耗计算资源的速率保持一致。

排查 Contour 用量

Contour 产生的交互式用量始终归属于触发该操作的分析任务。对于指定的 Contour 分析任务，其所有的交互式计算消耗均归属至该分析资源本身。

执行计算任务的各个看板会生成相应的 Spark 指标，可通过 Builds 应用程序进行查看。在每个看板上下文菜单的查看作业选项中即可进入此功能。来自 Contour 的批量计算用量归属于计算结果输出的目标数据集，这与 Code Repositories 和 Code Workbook 的机制类似。对于交互式和批量计算，您均可在 Resource Management 应用程序中查看整体的资源使用情况。

理解资源使用的驱动因素

Contour 运行于 Spark 计算框架之上，因此主要受两大因素影响：数据规模与查询复杂度（即对数据执行的操作数量与类型）。对数据执行的操作越多，后端计算结果所消耗的计算资源就越大。不过，由于某些操作的计算开销远高于其他操作，通过增加操作反而可能降低整体用量（例如，在高开销操作前添加过滤条件）。

对数据的操作次数取决于 Contour 路径中的看板数量。处于路径下游的看板依赖于上游看板的输出结果。在路径中新增看板会改变流经该路径的数据，从而要求重新计算看板，进而增加计算开销。

Contour 用户增多会带来查询量上升，可能导致计算开销增加。然而，Contour 提供了两项特性来降低多用户场景下的计算资源消耗：检查点（checkpoints）与缓存（caching）。

  • Contour 的检查点由 Contour 后端自动管理。检查点使得 Contour 能够快速提供路径下游的查询结果。随着时间推移，这有助于减少较长 Contour 路径所带来的计算量，但无法彻底消除计算开销。
  • Contour 还会对相同数据上的相同查询结果进行缓存。这意味着，如果两名用户先后打开同一个 Contour 分析，计算过程只会执行一次。这极大提升了仪表板的响应效率，并削弱了用户数量对计算资源用量的影响。
    • 需要注意的是，更改输入数据或参数会导致缓存失效，但在使用相同输入数据与参数进行后续计算时，将再次启用缓存。

使用 Contour 管理用量

优化分析过程与输入数据集可以降低成本并缩短分析时间。了解有关 Contour 优化的更多信息。

计算用量

Contour 遵循与平台中其他基于 Spark 的产品（如 Code Repositories）相同的 Spark 计量原则。借助基于成本的路由机制，Contour 会自动调整后端驱动器（drivers）和执行器（executors）的规格，以高效满足数据规模与查询类型的需求。

 示例 Contour 驱动器规格：
    num_vcpu: 2
    gib_ram: 11
 示例 Contour 执行器规格：
    num_vcpu: 2
    gib_ram: 12
    num_executors: 3
 查询时长：3 秒
 交互式 Contour 用量费率：15

driver_compute_seconds = interactive_usage_rate * driver_compute_seconds
                       = 15 * max(2vcpu, 11gib / 7.5gib) * 3sec
                       = 15 * 2 * 3 = 90 compute-seconds

executor_compute_seconds = interactive_usage_rate * executor_compute_seconds
                         = 15 * max(2vcpu, 12gib / 7.5gib) * 3sec * 3executors
                         = 15 * 2 * 3 * 3 = 270 compute-seconds


total_compute_seconds = driver_compute_seconds + executor_compute_seconds
                      = 90 compute-seconds + 270 compute-seconds = 360 compute-seconds
