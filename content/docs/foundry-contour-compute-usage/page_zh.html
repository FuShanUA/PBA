通过 Contour 使用计算资源

在 Foundry 中的数据上运行 Contour 工作流需要使用 Foundry 计算资源，该资源以计算秒（compute-seconds）为单位进行衡量。本文档详细说明了 Contour 如何使用计算资源，并提供了有关在产品中调查和管理计算资源使用情况的信息。

每个显示数据的 Contour 板块都由对 Contour 后端的查询支持。Foundry 通过在自动扩展的 Spark 集群上执行这些查询来提供 Contour 查询。Contour 查询的计算秒数仅根据查询实际执行时所使用的计算秒数进行测量。然而，在交互模式下，为了提供更快的响应，所有 Contour 分析的计算资源都是“始终开启”的；这意味着每个单独的交互式查询比批量计算使用计算秒数的速度更快。

查询使用的资源数量取决于许多因素，例如输入数据集的大小和布局以及查询的复杂性（如使用的 Contour 板块）。

如何衡量计算资源

Contour 使用 Foundry 的并行计算后端来执行对 Foundry 数据的查询。因此，Contour 中的计算资源是按照以下公式进行衡量的。

请注意，默认的 interactive_contour_usage_rate 是 15. 这是 Contour 基于始终在线的并行化计算框架使用的计算资源速率。如果您与 Palantir 签有企业合同，请在进行计算资源使用计算之前联系您的 Palantir 代表。

total_compute_seconds = contour_usage_rate * (driver_compute_seconds + executor_compute_seconds)

交互式 Contour 使用情况

Contour的“随时就绪”执行系统旨在对终端用户的临时查询做出反应。在Foundry中，Contour查询的成本以计算秒数为单位，基于在Spark集群上执行查询所需的时间以及查询执行期间使用的内核数量。内核数量是通过查看查询在Spark应用程序中的份额来确定的，其中查询被安排执行。

在交互模式下，每次创建或更新Contour板时都会使用计算资源。这包括影响Contour路径中给定板下游的所有板的“级联”计算。例如，以下操作将在Contour分析中使用计算资源：

  • 创建新板（可能影响所有下游板）
  • 更新板（可能影响所有下游板）
  • 点击路径顶部的更新数据

批量Contour使用

轮廓路径可以“另存为”数据集。这会将由轮廓后端生成的底层代码保存为数据集作业规范。此数据集构建之后可以按计划或临时运行。这些数据集作业被计为批量作业，不会产生交互式轮廓使用所具有的“始终在线”的开销。

轮廓批量数据集被计为通用并行化批量计算，类似于通过代码仓库和代码工作簿创建和更新的数据集。这些批量计算以与其他批量计算相同的速率使用计算资源。

检查轮廓使用情况

从轮廓进行的交互式使用总是与它所触发的分析相关联。对于给定的轮廓分析，所有交互式计算都与分析资源本身相关联。

运行计算的独立面板会产生Spark指标，可以从“构建”应用程序中查看。这可以通过 查看作业 访问。 每个看板的上下文菜单部分。从Contour批量计算的使用情况附着于计算生成的数据集，类似于代码仓库和代码工作区。对于交互式和批量计算，您可以在资源管理应用程序中查看总体使用情况。

理解使用情况的驱动因素

Contour 运行在 Spark 计算框架上，因此受两个主要因素影响：数据规模和查询复杂度（对数据执行的操作数量和类型）。必须对数据执行的操作越多，后端就必须使用更多的计算来计算结果。然而，由于某些操作比其他操作昂贵得多，可以通过添加操作来减少总体使用情况（例如，在一个昂贵的操作之前添加一个过滤器）。

数据操作的数量受Contour路径中板的数量影响。路径中越靠后的板依赖于路径中较早的板的输出。在路径中添加新的板可能会改变通过路径的数据，这需要重新计算板并增加计算量。

更多的Contour用户可能会由于查询数量的增加而产生更多的计算需求。然而，Contour有两个功能可以大幅减少多个用户的计算使用： 检查点 和 缓存。

  • Contour 检查点 由Contour后端自动管理。检查点使Contour能够快速提供路径中较远位置的查询结果。这可以在长时间内减少大型Contour路径的计算量，但不会完全消除计算。
  • Contour还会缓存相同数据上相同查询的结果。这意味着如果两个用户依次打开相同的Contour分析，计算只会运行一次。这使得仪表板能够更高效地提供，并减少了用户数量对计算使用的影响。
    • 注意，更改输入数据或更改参数将无法使用缓存，但对同一输入数据和参数的未来计算将再次使用缓存。

计算使用情况

轮廓在成本基础上路由的原则下与其他基于Spark的产品（如代码仓库）相同。通过成本路由，轮廓会自动优化后端驱动程序和执行器的大小，以有效满足数据量和查询类型的需要。

 Example Contour Driver Size:
    num_vcpu: 2
    gib_ram: 11
 Example Contour Executor Size:
    num_vcpu: 2
    gib_ram: 12
    num_executors: 3
 Query Duration: 3 seconds
 Interactive Contour Usage Rate: 15

driver_compute_seconds = interactive_usage_rate * driver_compute_seconds
                       = 15 * max(2vcpu, 11gib / 7.5gib) * 3sec
                       = 15 * 2 * 3 = 90 compute-seconds

executor_compute_seconds = interactive_usage_rate * executor_compute_seconds
                         = 15 * max(2vcpu, 12gib / 7.5gib) * 3sec * 3executors
                         = 15 * 2 * 3 * 3 = 270 compute-seconds


total_compute_seconds = driver_compute_seconds + executor_compute_seconds
                      = 90 compute-seconds + 270 compute-seconds = 360 compute-seconds
