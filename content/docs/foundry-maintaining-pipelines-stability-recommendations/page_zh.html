稳定性建议

本页面介绍了如何长期构建具备弹性和稳定性的数据管道，并提供了关键建议。

对所有变更进行版本控制

生成数据集的每一段逻辑都应纳入版本控制。这有助于更轻松地追踪数据管道中出现的退化与变更。具体操作上，可归结为以下几点：

  1. 在搭建生产级数据管道时，避免使用 Contour 或 Code Workbook，因为在这些工具中追踪修改更为困难。它们非常适合开发阶段，但随着管道逐渐成熟，相关功能应改用 Transforms 重写，理想情况下应使用 Python 或 Java。
  2. 确保所有代码仓库的 master 分支处于锁定状态，且所有变更必须提交 Pull Request 并经代码审查人员批准。
  3. 合并代码时，建议始终采用 Squash and merge 方式，这样可以在 master 分支上保留更清晰的提交历史。

隔离常见的数据质量错误

“敏感”或“不稳定”的数据集应通过校验步骤与数据管道的其他部分隔离。尤其是通过 Fusion 创建的数据集、手动上传的数据以及其他类型的上传数据，往往容易出现数据质量问题，最终可能影响整个数据管道。

在以下示例中，假设名为 Fusion 的数据集经常出现数据质量问题（如模式变更、解析错误、数据不完整等），并最终影响到数据管道的其余部分。

解决方案是创建一个 Fusion Validated 数据集，只有在某些校验步骤通过后，才从 Fusion 复制数据。

创建一个基于事件的调度任务，以便在 Fusion 更新时构建 Fusion Validated。

将 Fusion Validated 视为其他管道的输入。在构建时忽略它，并为其添加相关的健康检查。构建状态至关重要，因为你可能需要联系负责更新 Fusion 的人员，告知他们潜在的错误或问题。

同样的逻辑也适用于 Manual Upload 和 Manual Upload Validated 数据集。

处理共享资源

项目中可能存在某个数据集属于多个不同管道的情况。如果各构建任务未能完全对齐，部分管道可能会因等待构建该数据集而受阻。

在这种情况下，建议考虑新建一个管道，按所需频率构建此共享数据集，并将其作为其他管道的输入（在调度中将其忽略）。但请注意，此类操作会显著增加管道设置的复杂度。我们建议，除非确有必要，否则不要进行此操作。

另一种侵入性较低的变通方案是，仅在某个管道/调度中构建共享资源，在其他所有管道中将其视为输入。例如，在上一个示意图中，将 Shared 作为其中一个管道的输入（即在除该调度外的所有调度中将其排除）。

避免部分运行

应避免仅运行管道的一部分，或在数据集上执行“全量构建”。

管道的状态要么是最新的，要么不是。如果仅运行部分管道（例如仅执行导出阶段），可能会导致状态不一致，增加评估管道健康状况的难度。如果对终端数据集执行“全量构建”，该构建将缺失运行管道应具备的所有相关配置项（重试、失败策略、忽略项等）。每当需要手动启动构建时，应使用调度概览页面上的“立即运行调度”按钮，以便根据调度配置执行构建。
