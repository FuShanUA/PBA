稳定性建议

本页涵盖了创建稳定且具有弹性的管道的建议。

对所有更改进行版本控制

生成数据集的每一段逻辑都应该进行版本控制。这有助于更容易地跟踪回归和管道发生的更改。实际来说，这意味着：

  1. 在设置生产级别的管道时，避免使用 Contour 或 Code Workbook，因为这些工具中的修改追踪更具挑战性。这些工具非常适合开发阶段，但随着管道的成熟，应该将功能重写为转换（Transforms），最好是在 Python 或 Java 中。
  2. 确保 master 分支在所有的代码仓库中被锁定，所有更改都需要通过拉取请求（Pull Request）并获得代码审查员的批准。
  3. 在合并时，我们建议您始终 Squash and merge，因为这会在 master 上留下更清晰的提交历史记录。

隔离常见的数据质量问题

“敏感”或“不稳定”的数据集应通过验证步骤与管道其余部分隔离。特别是，在Fusion中创建的数据集、手动上传的数据以及其他类型的上传容易出现数据质量问题，这些问题可能会影响到整个管道。

在下面的例子中，假设名为Fusion的数据集经常出现数据质量问题（模式变化、解析错误、数据不完整等），这些问题最终会影响管道的其余部分。

解决方案是创建一个Fusion Validated数据集，该数据集仅在某些验证步骤通过后从Fusion复制数据。

创建基于事件的计划，在Fusion更新时构建Fusion Validated。

将Fusion Validated视为其他管道的输入。忽略其构建，并在其上添加相关的健康检查。构建状态很重要，因为您可能需要联系负责更新Fusion的人员，告知他们潜在的错误或问题。

同样的逻辑也适用于Manual Upload和Manual Upload Validated数据集。

处理共享资源

您可能有一个属于项目中多个不同管道的数据集。如果您的构建不完全对齐，某些管道可能会因构建此数据集而被阻塞。

在这种情况下，您应该考虑创建一个新的管道来按需频繁构建这个共享数据集，并让共享数据集成为其他管道的输入（忽略其排期）。然而，请注意，这种操作会显著增加您的管道设置的复杂性。我们的建议是< strong>不执行此操作，除非绝对必要。

另一种影响较小的方法是仅在一个管道/排期中构建共享资源，并将其作为所有其他管道的输入。例如，在之前的图表中，将共享视为其中一个管道的输入（即从除一个排期外的所有排期中排除它）。

避免部分运行

您应避免仅运行管道（pipeline）的一部分或对数据集（dataset）执行“完整构建”。

管道（pipeline）要么是最新的，要么不是。如果您只运行了管道（pipeline）的一部分（例如，仅运行导出阶段），可能会导致状态不一致，使评估管道（pipeline）是否健康变得更加困难。如果在终端数据集（terminal dataset）上执行“完整构建”，此构建将缺少管道（pipeline）应使用的所有相关选项（重试、失败策略、忽略项等）。每当需要手动启动构建时，应使用“立即运行计划”按钮，在计划概览页面根据计划配置运行构建。
