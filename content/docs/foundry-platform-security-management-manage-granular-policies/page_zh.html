管理细粒度策略

细粒度策略允许您为数据集和对象配置行级安全设置。它们被 受限视图 和 对象安全策略 使用，以确定用户有权访问哪些特定行或对象实例。

细粒度策略是一组规则和逻辑运算符，用于比较用户属性、列或属性以及值，以确定用户可以看到哪些数据。细粒度策略具有灵活性，并支持各种比较和术语。

设计细粒度策略

在创建细粒度策略之前，请考虑以下问题：

  • 您希望如何限制对数据的访问？
  • 您可以在策略中使用哪些用户属性？
  • 您的策略可能依赖于哪些列或属性？
  • 哪些标记会通过管道传播，但在创建资源时可能未被标记？
  • 您将如何授予新用户的权限？

建议

在设计细粒度策略时，我们建议遵循以下指南：

  • 保持简单：在创建细粒度策略之前，先写下您想要实施的策略。确定实现该策略所需的最小条件集。策略越复杂，管理与验证起来就越困难。
  • 确保策略列不为空：策略列中包含空值的行将对所有用户不可见。策略列中的空值也会导致同步和下游流程（如Phonograph同步）出现问题。
  • 利用管道：为了降低那些与当前数据结构或字段不匹配的策略的复杂性，可以使用管道计算出列，以使策略编写更加容易。尽量在管道中处理复杂性，而不是在细粒度策略中。
  • 考虑使用专用的策略列： 对于策略中引用的后备数据集中的列进行的更改可能会破坏策略假设。为了防止这种风险，建议将决定策略的逻辑分离出来，并为策略引用创建专用的列（或多个列）。
  • 使用属性： 基于属性的策略通常是最佳选择。属性可以从SSO中提取或通过用户管理器发布。
  • 使用标记： 在策略的后备数据集中应用一个标记，以确保其保护并确保只有具有访问权限的用户才能看到该标记。由于细粒度策略已经控制了用户可以看到的行，因此可以在资源中停止继承该标记。如果敏感数据已在源处进行了标记，并且标记已正确传播，则在创建资源时不需要新的标记。

用户属性

以下是细粒度策略中支持的用户属性列表：

  • 用户ID：由Foundry为每个用户生成的唯一ID。
  • 用户名：用户在登录时由身份提供者提供的唯一ID。
  • 组ID：所有包含该用户作为成员（直接和继承）的组的ID。
  • 组名称：所有包含该用户作为成员（直接和继承）的组的名称。
  • 授权组ID：这是一个与作用域会话相关的高级概念。如果您计划使用作用域会话进行细粒度策略，请联系您的Palantir管理员。
  • 组织标记ID：所有包含该用户作为成员（主用户和访客）的组织的标记ID。请注意，这些是标记ID，它们是UUID，不是以ri.multipass..organization.开头的组织ID。
  • 标记ID：用户有权查看的所有标记的ID。
  • 自定义属性：您的身份提供者在控制面板中配置的任何自定义属性。

在引用用户、组或组织时，策略要求在策略列和策略定义中都使用唯一标识符（UUID）。不支持使用名称代替ID，以防止重命名相关的问题。

策略比较

细粒度策略支持以下比较类型：

  • 等于： 比较的左右两边必须是单值（不是集合）且类型相同。
  • 相交： 至少有一边是集合，且两边类型相同（或该类型的集合）。
  • 子集： 右边必须是集合，且两边类型相同（或该类型的集合）。
  • 超集： 左边必须是集合，且两边类型相同（或该类型的集合）。
  • 小于： 比较的左右两边必须是单值（不是集合）且类型相同。
  • 小于或等于： 比较的左右两边必须是单值（不是集合）且类型相同。
  • 大于或等于： 比较的左右两边必须是单值（不是集合）且类型相同。
  • 大于: 比较的左右两边必须是单值（不是集合）且类型相同。

对象安全策略不支持小于/大于比较运算符。

策略被定义为模板，用户属性、组成员身份和数据值可以填充到其中。当使用应用程序请求数据时（例如在Contour中），该模板会被转换成一个查询。此查询只返回与用户属性和权限相关的行或对象。

策略限制

一个策略最多可以包含十个比较。细粒度策略会根据策略是比较集合还是常量与字段来计算每个比较的权重：

  • 将常量与字段进行比较的权重为1。
  • 将集合与字段进行比较的权重为1,000。
  • 标记条件的权重为3,000。

  • 策略中所有比较的权重总和必须低于10,000。

例如，在一个基本策略中：

  • 规则（1）的权重为1，因为它是在将常量（用户的ID）与字段进行匹配。
  • 规则（2）的权重为1,000，因为它是在将集合（用户所属的所有组）与特定组进行匹配。
  • 此策略的权重总和为1,001（远低于10,000的限制）。

当前的策略构建限制旨在对特定策略设计施加最小的约束。它对防止超出权重限制的策略提供的保护很小。

如果您在构建策略时收到权重限制错误，请联系Palantir支持团队以获得帮助。

策略管理

当您管理细粒度策略时，请考虑两个目标：

  • 透明性：负责管理数据访问的任何人都应了解策略以及这些策略如何与平台中的数据相互作用。
  • Robust change management: Policy management should be an orderly, organized process. Always review and manage changes before applying them.

Managing integrity of data pipelines

Granular policies introduce a set of assumptions about the data that backs them. Therefore, granular policies will only correctly control access to data as long as these assumptions are true. Consider whether the use case requires building out machinery to ensure that those assumptions hold and data stays secured if those assumptions are broken.

One way to solve this problem is by introducing a step in the pipeline directly upstream of the resource that checks the assumptions made about the data. These checks might include:

  • Invariants: Write a list of invariants that will force the downstream build to fail if they are not true. For example, assume that whenever an event_occurred_in_state has the value NY, another column in the dataset called state_name should have value New York. Have the transform check that this is true before surfacing this data to users.
  • Statistics: Define a set of statistics and the range they should always be within. For example, a granular policy may be used to enforce access controls mirroring an organization hierarchy; each user can only see data about individuals below them in the hierarchy. Have the transform assert that if more than 20% of the hierarchy changes from one day's build to the next, something is wrong. At this point, users responsible for policy management should check to ensure everything is correct before surfacing this data to users.
