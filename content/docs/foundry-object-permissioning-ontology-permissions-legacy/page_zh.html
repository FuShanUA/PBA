本体论权限 [旧版]

本体论资源现在通过Compass进行权限管理

本体论资源（对象类型、操作类型、关联类型、共享属性和接口）可以被 常规项目资源，通过Compass文件系统进行管理。。这取代了之前的本体论角色和数据源派生的权限模型。有关更多信息，请参阅 关于本体论权限的更新文档。

本体论资源指的是对象类型、关联类型和操作类型及其元数据（模式）。

您可能会遇到两种适用于您注册的旧版权限模型之一：

  1. 本体论角色 是授权本体论资源的默认方法。本体论角色允许将 角色 直接应用于每个本体论资源，而不受其支撑数据源的影响。
  • 例如，用户只需要在对象类型上拥有 Ontology Editor 角色（角色），而无需在支撑数据源上拥有任何权限即可编辑本体论中的对象类型。
  • Ontology Editor 角色（角色）仅允许编辑本体论资源及其元数据，并不会在数据或数据源本身授予任何权限。对对象数据（非元数据）的访问仍由在支撑数据源上授予的权限来管理。
  1. 数据源派生权限 是授权本体论资源的旧解决方案。数据源派生权限依赖于每个对象类型在支撑数据源上定义的权限，从而在本体论中的对象类型和支撑数据源之间创建直接的 1:1 依赖关系。因此，具有数据源派生权限的对象类型需要一个支撑数据集。
  • 例如，用户必须拥有 Editor 访问权限到支撑数据源，并且是 Ontology Administrators 组（组）的成员（在本体论级别）才能编辑本体论中的对象类型。

本体论角色

本体论角色定义如下：

  • Ontology Owner：可以编辑本体论资源，并对其安全性和共享具有完全控制权
  • Ontology Editor：可以编辑本体论资源
  • Ontology Viewer：可以查看本体论资源，但不能编辑它们
  • Ontology Discoverer：只能看到本体论资源的名称和元数据，不包括模式

除了直接在本体论资源上授予上述角色外，您还可以通过导航到 本体论管理器 应用程序中某个本体论的 本体论配置 选项卡，在本体论级别授予这些角色。只有在本体论级别授予的 Ontology owner 角色会继承到该本体论的所有资源；Ontology editor 角色仅适用于本体论级别的权限。

作为最佳实践，我们强烈建议定义一个可信的用户组，负责整个本体论（也称为本体论治理委员会），并为整个本体论授予该用户组 Ontology Owner 角色。

可以自定义默认本体论角色中包含的操作，或者根据不同用户组的具体需求配置其他自定义角色。有关角色及其自定义方式的更多信息，请参阅 关于角色的文档。

使用本体论角色创建新资源

在本体论中，资源创建仅限于具有 Ontology Owner 或 Ontology Editor 角色的用户。使用角色创建的新对象类型、关联类型、共享属性和操作类型将显示创建者为该资源的 Ontology Owner，而所有其他用户则默认显示为 Ontology Viewer。资源创建后，创建者可以进一步对该资源应用其他角色。

默认情况下，每个用户都会在本体论级别被授予 Ontology Editor 角色，并可以为其工作流程创建新的本体论资源。要自定义允许添加新本体论资源的用户组，Ontology Owner 可以导航到本体论管理器中的 本体论配置 选项卡，并调整本体论级别的角色授权。

使用本体论角色进行类型特定的编辑权限

编辑对象类型及其属性的权限

要对对象类型及其属性进行更改，用户必须在该对象类型上拥有 Ontology Editor 权限。如果用户希望将数据源/列映射到对象类型属性，则还需要对正在映射的数据源拥有 Viewer 权限。

共享属性的权限

要对< a href="/docs/foundry/object-link-types/shared-property-overview/">共享属性进行更改，用户必须在该共享属性上具有Ontology Editor权限。用户还必须在想要添加该共享属性的任何对象类型上具有Ontology Editor权限。

编辑链接类型的权限

要对链接类型（创建、删除、更新等）进行更改，用户必须拥有以下权限：

  • 在链接类型的两侧引用的对象类型上具有Ontology viewer权限。
  • 在链接类型本身上具有Ontology editor权限。

如果链接类型使用了连接表，并且修改涉及对连接表的更改，则还需要对支持该链接类型的连接表数据源具有Viewer权限。

编辑操作类型的权限

要对操作类型（创建、删除、更新等）进行更改，用户必须拥有以下权限：

  • 至少Editor该操作类型的权限，可以直接获得或通过从本体论级别继承获得
  • 在执行期间该操作类型可以生成编辑的所有对象类型的Ontology Editor。

该操作类型可以生成编辑的对象类型包括以下内容：

  • 在创建、修改和删除对象规则中引用的对象类型。
  • 在创建和删除链接规则中引用的链接类型连接的对象类型。
  • 在函数支持的操作中的函数所编辑的对象类型。
  • 操作日志对象类型（如果已配置的话）。

只读视图

当用户没有权限编辑对象类型、关联类型、共享属性或操作类型时，编辑视图将被禁用，并且会显示一个横幅信息，向用户说明其拥有的和不拥有的权限。

查看权限

对支持对象类型或链接类型的源数据具有Viewer权限的用户可以查看与该特定源数据相关的对象类型或链接类型。

默认情况下，操作类型对所有有权访问本体论（Ontology）的用户可见。所有用户都可以看到所有操作类型的标题、描述和规则，使用的是基于数据源的权限模型。

类型特定编辑权限

要在Ontology Manager中进行任何更改，用户必须是Ontology Administrators用户组的成员。有关< a href="/docs/foundry/platform-security-management/manage-groups/">组和平台安全的更多信息。

当使用基于数据源的权限时，用户可能需要额外的类型特定权限才能成功在Foundry Ontology中进行更改。

编辑对象类型及其属性的权限

要对对象类型及其属性进行任何更改，用户必须拥有Editor权限以访问支持该对象类型的数据库。

共享属性的权限

要创建或编辑一个共享属性或将共享属性添加到对象类型中，用户必须是Ontology Administrators组的成员。

编辑链接类型的权限

要对链接类型进行任何更改，用户必须拥有Editor权限以访问支持该链接类型的数据库，并且在链接类型中引用的两个对象类型的数据库上拥有Viewer权限。

编辑操作类型的权限

  • 所有可以访问本体论（Ontology）的用户都可以查看完整的操作类型（可编辑属性、名称或用户权限等）。
  • 要在本体论（Ontology）中对操作类型进行更改（创建、删除、更新等），用户必须是Ontology Administrators组的成员。
  • 要运行操作，用户必须在所有已编辑对象类型上具有Viewer。
  • 如果用户创建了一个修改或添加到对象类型的的操作，该对象类型必须启用Edits选项。

有关操作类型权限的更多信息，请参阅 文档。

只读视图

当用户没有编辑对象类型、关联类型或操作类型的权限时，编辑视图将被禁用，并会有一个横幅向用户说明他们具有哪些权限和不具有哪些权限。

删除支撑数据集

如果一个带有数据源派生权限的对象类型的数据集已被永久从垃圾箱中删除，该对象类型将被视为孤立的。由于权限是从支撑数据集中派生的，而该数据集现在无法再访问，用户将无法再修改该对象类型，因为所有编辑权限都已丢失。本体论会自动删除孤立的对象类型。

警告

对于数据源派生权限，所有对象类型都必须拥有一个支撑数据集。为了防止不可编辑的本体类型积累，具有数据源派生权限但没有支撑数据集的对象类型将在24小时后被删除。
