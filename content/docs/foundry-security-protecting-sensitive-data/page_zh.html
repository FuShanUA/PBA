保护敏感数据

正如你在《巩固数据基础》中所见，各个项目内的数据访问权限可由项目数据主人管理。这就是所谓的自主访问控制 ↗。然而，面对敏感数据时，我们需要一种更强有力、更集中的访问控制模型。

在我们的示例中，有一个包含乘客出生日期 (DOBs) 的模拟数据集，这属于个人身份信息 (PII)。我们可能希望严格控制此类 PII，并确保只有接受过 PII 培训的人员才能访问这些数据。我们可以通过标记来解决这一问题。

标记是 Foundry 对强制访问控制 ↗ 的实现。标记代表一种特定类型的数据（如 PII），只有指定列表中的用户或用户组才能访问。对数据集应用标记后，未拥有该标记访问权限的用户绝对无法访问该数据，即使项目数据主人试图与其共享也无济于事。重要的是，这种限制会传播至平台任何位置基于该数据集派生出的任何数据。

此功能对数据治理至关重要，它使数据保护官能够集中管理并精确审计特定类别数据的访问者。

在向最终用户推出应用之前，我们要确保已对敏感数据加以保护。再次强调，在我们的示例中，我们希望锁定数据管道中的乘客出生日期 (DOBs)。

创建标记

我们需要创建一个标记类别，作为一组标记的名称。在本例中，请创建一个名为“信息”的标记类别，因为未来可能还需要其他与信息相关的标记（例如个人健康信息，即 PHI）。创建类别后，你可以创建 PII 标记。接着，你可以将有权限查看 PII 数据的人员添加为该标记的成员，并将管理员团队添加为该标记的管理者。

模拟标记变更

标记具有沿数据血缘关系传播的强大特性。因此，对现有管道应用新标记可能会导致下游用户意外失去访问权限。最佳实践是始终先对管道应用标记进行模拟，以观察其传播范围。为此，请打开管道的数据血缘视图并开启模拟模式。你可以编辑 raw/passengers 数据集上的标记，并应用你创建的 PII 标记。随后，你会看到应用 PII 标记后将受影响的所有下游数据集。

我们不希望所有数据消费者都必须拥有 PII 访问权限，因此需要在管道的某个环节移除敏感的 DOB 列。为此，我们应依次查看管道，评估在何处移除 PII 标记最为合适。通常的做法是，选中数据集后在底部打开预览视图，以便查看数据和列。

在我们的示例中，可以看到 PII 标记一直传播到了本体数据集，这会锁定所有不具备 PII 访问权限的最终用户。因此，我们决定最好让敏感数据在管道中保留尽可能长的时间，但移除乘客数据在本体版本（即 /Sky Industries/Customer Metrics [Ontology]/passengers）中的 “dob” 列。

移除即将继承的标记

在数据血缘视图中，点击 Customer Metrics [Ontology]/passengers 数据集，点击代码，然后点击在代码仓库中查看。这将打开用于创建该模拟数据集的代码仓库。在代码仓库中，你需要：1) 创建分支，2) 移除敏感列（即删除 dob 列），3) 从输入数据集中移除即将继承的 PII 标记，然后 4) 创建拉取请求。我们建议查阅有关如何移除继承的标记和组织的文档。

同事批准了停止传播 PII 标记的拉取请求后，我们需要构建此数据集及其下游的所有内容，确保最新数据集事务全部“停止传播”即将添加的标记。此外，对于 APPEND 或 UPDATE 事务类型，还需特别注意一些事项。不过在我们的示例中，所有内容均以 SNAPSHOT 方式构建，这也是 Foundry 中的默认事务类型。

应用标记

应用标记前，我们要再次确认它只会传播到预期的数据集，而不会影响其他数据集。为此，请再次打开管道的数据血缘关系视图，开启模拟模式，在 raw/passenger 数据集上应用该标记，并确认 ontology/passenger 数据集未受影响。这表明上一节中的 stop_propagating 逻辑已正确应用。

现在可以应用标记了，如下方示例数据截图所示。为此，请转到 raw/passenger 数据集，打开安全助手并应用标记。点击保存后，PII 标记会立即应用并向下传播。通过查看数据血缘关系，可以在现已带有标记的数据集上看到标记徽章，这表明您已成功保护了敏感的 PII 数据。
