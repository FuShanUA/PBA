接口操作类型约束 [Beta]

Beta

接口操作类型约束处于 beta 开发阶段，可能在您的注册中不可用。在积极开发期间，功能和平台支持可能会发生变化。

接口操作类型约束定义了实现接口的对象类型之间常见的预期操作能力。该约束描述了操作的用户面向含义、API 名称和参数结构。然后，每个实现的对象类型可以将该约束映射到该对象类型的实际操作类型。

例如，Ticket 接口定义了一个 Create ticket 操作类型约束。 Bug 对象类型通过 Create bug 操作类型满足该约束，而 Feature request 对象类型通过 Create feature request 操作类型满足该约束。两个操作类型都实现了相同接口级别的功能，尽管具体的操作逻辑是针对每个对象类型的。

Interface 操作类型约束不定义操作执行逻辑、提交条件、副作用或表单布局。要配置这些行为，请在满足约束的具体操作类型上进行配置。有关关于对界面对象起作用的操作规则的更多信息，请参阅 界面操作。

目前，操作类型约束在描述满足该约束的操作类型必须如何行为方面表达能力有限。使用约束描述来告知应用构建者满足该约束的操作类型应该做什么，包括预期的编辑、副作用、提交行为或其他约束模型未强制执行的语义要求。更具有表达力的行为约束正在积极开发中。

操作类型约束

操作类型约束定义了接口级操作功能的参数。这些参数包括以下内容：

  • 显示名称：操作类型约束的用户界面名称。
  • 描述：描述满足该约束的操作类型应执行的功能的用户可读说明。
  • API 名称：用于在代码和平台配置中引用该约束的 API 名称。
  • 参数约束： 一个满足的操作类型必须公开的参数。
  • 操作类型约束是否作为对象类型实现的一部分是必需的。

如果操作类型约束是必需的，那么每个实现该接口的对象类型都必须将该约束映射到具体的操作类型。可选的操作类型约束可以在实现对象类型支持该功能时进行映射，但当该功能不适用时可以跳过。

参数约束

参数约束定义了满足操作类型必须提供的操作参数的结构。每个参数约束包括：

  • 类型： 参数类型，例如字符串、时间戳、对象引用、接口引用、对象集、附件、媒体引用或结构体。
  • 单值或列表值： 参数接受一个值还是多个值。
  • 显示名称： 参数约束的用户界面名称。
  • API 名称： 用于引用参数约束的 API 名称。

当您创建一个参数约束时，可以手动定义一个新参数，或者从现有的接口属性中添加参数。从现有接口属性中添加参数会根据所选属性的显示名称、类型和必需状态创建参数约束。参数的API名称是从显示名称生成的，可以进行编辑。

必需的参数约束必须映射到具体操作类型上的必需参数。在有用的情况下，您可以映射可选的参数约束。请注意，同一具体操作参数不能满足同一操作类型约束下的多个参数约束。

对于对象引用、接口引用、对象集和结构参数，接口操作类型约束描述的是通用参数类型，而不是每个具体的实现细节。例如，一个对象引用参数约束要求满足操作类型上具有兼容的对象引用参数，但该约束本身并不绑定到特定的对象类型。

实现操作类型约束

当一个对象类型实现了具有操作类型约束的接口时，该对象类型必须选择满足任何所需约束的具体操作类型。当具体操作类型具有与约束所需的参数兼容的参数时，它就满足该约束。

在Ontology Manager中添加接口实现后，对象类型的接口选项卡中的操作子选项卡会列出接口的操作类型约束。为每个所需约束在实现该接口的对象类型上选择一个具体的操作类型。您可以跳过或映射可选约束。

要配置参数映射，请选择已映射操作类型约束的设置图标以打开配置参数对话框。将每个所需参数约束映射到所选操作类型上的兼容所需参数。如果应用程序需要通过接口操作功能传递或预填这些值，您也可以映射可选参数约束。

本体论管理器在您保存本体论更改时会验证操作类型约束映射。如果出现以下情况，您将无法保存：

  • 一个必需的操作类型约束未被实现对象类型映射。
  • 一个必需的参数约束未被映射。
  • 一个已映射的参数具有不兼容的参数类型。
  • 同一个具体操作参数被映射到多个参数约束。
  • 已映射的操作类型约束或参数约束不再存在。

接口扩展

操作类型约束通过接口扩展继承。如果子接口扩展了带有操作类型约束的基接口，那么实现子接口的对象类型必须满足继承的所需操作类型约束以及子接口上直接声明的任何约束。

当某个功能适用于一个广泛接口并通过更具体的接口可用时，请使用继承的操作类型约束。当该功能仅适用于子接口时，在子接口上定义新的操作类型约束。

选择必需与可选约束

当每个实现接口的对象类型都必须支持该功能时，才能使用必需的操作类型约束，以确保基于接口的应用程序正常工作。

当该功能有用但不普遍时，使用可选操作类型约束。在迭代一个测试版接口模型时，可选约束也很有用，因为它们允许对象类型采用该功能，而不会阻止那些尚未准备好支持它的实现。

对于参数约束，只有当每个满足条件的操作类型都必须将该参数作为必需的操作输入时，才将参数标记为必需。当实现可以内部计算值、使用默认值或根本不需要该值时，保持参数约束为可选。

限制

Interface action type constraints are currently configured and mapped in Ontology Manager. Pipeline Builder does not currently support action type constraint mapping when implementing an interface. If your interface contains required action type constraints, implement and map the interface through Ontology Manager. Additional platform integrations are under active development.

End users cannot currently discover or invoke interface action type constraints from Foundry applications. Users can run the satisfying concrete action types directly through any application that supports them.

Interface action type constraints are not currently supported in the Ontology SDK. You cannot currently use an interface action type constraint in OSDK to discover or invoke the concrete action types that satisfy it.

Interface action type constraints define the expected shape of a concrete action type. They do not make object-type-specific action logic uniform. Review each satisfying action type's rules, submission criteria, permissions, and side effects to ensure they match the semantics described by the interface constraint.
