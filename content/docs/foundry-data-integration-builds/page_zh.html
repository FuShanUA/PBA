构建

构建是用于在 Foundry 中计算数据集新版本的机制。构建提供了计算的编排与协调功能，确保读取正确的输入数据，并将输出数据写入到合适的位置。

构建由作业组成，每个作业是一个工作单元，由共享逻辑定义，用于计算一个或多个输出数据集。需要注意的是，如果一个作业定义了多个输出数据集，它们将始终一起更新；如果不运行完整的作业，则无法仅构建部分数据集。作业规范，即 JobSpec，定义了作业的构建方式。当在 Foundry 中对数据转换逻辑进行更改时，例如数据工程师在代码库中提交新的转换代码时，就会发布 JobSpec。

运行构建会对一组输出数据集进行一次性计算。为了保持数据在系统中的持续流转，可以通过调度来定期运行构建。

你可以使用构建应用程序在 Foundry 中探索构建。

作业与 JobSpec

作业封装了从一组输入数据集的数据中计算一个或多个输出数据集新版本的过程。JobSpec 通过详细说明输入数据集依赖关系以及作为作业一部分执行的逻辑，来定义如何构建作业。

输入数据集依赖关系声明为一组 InputSpecs，每个 InputSpec 指定一个特定的输入数据集。InputSpecs 根据数据集的视图指定要读取的数据子集。

在 Foundry 中，有多种类型的逻辑可以表示为作业，包括但不限于：

  • 数据连接同步定义了如何从外部数据源读取数据。
  • 在代码库中编写的转换允许你编写代码来转换数据集。
  • 健康检查定义为在数据集上生成的作业，用于验证数据集的特征。
  • 分析应用程序支持定义转换数据集的逻辑。
  • 导出定义了如何将输入数据发送到 Foundry 外部。

作业状态

在任何时候，作业始终处于以下状态之一：

  • WAITING：作业的初始状态；作业正在等待其依赖的作业完成，尚未被调用。
  • RUN_PENDING：作业等待运行，但其执行环境尚未确认状态。
  • RUNNING：作业已被调用，正在计算中。
  • ABORT_PENDING：作业已中止，但其执行环境尚未确认中止状态。
  • ABORTED：作业因用户请求或依赖作业失败而中止。
  • FAILED：作业已被调用，但计算失败。
  • COMPLETED：作业已被调用，且计算成功完成。

构建生命周期

运行构建时，会执行几个步骤来验证提交的构建、确保数据一致性，并且仅运行生成新输出所需的必要作业。

构建解析

作为第一步，构建会：

  • 检测指定输入数据集中的循环依赖，如果存在循环，则构建失败。
  • 验证所有输入数据集是否存在，并识别每个输入数据集的相应模式。
  • 在每个输出数据集上打开新事务，以确保只有活动的构建才能写入输出数据集。这被称为构建锁定。
  • 检测是否有其他正在进行的构建会更改此构建的输入。如果是，该构建可能会排队，等待其他构建完成。

作业执行

完成上述步骤后，构建中的作业将开始执行。互不依赖的作业会并行运行。随着作业经历各种作业状态，整体构建的状态也会相应更新：

  • 如果构建中的某个作业失败，该构建内所有直接依赖该作业的作业以及输出数据集上的事务都将终止。您也可以选择将构建配置为同时中止所有非依赖性作业。
  • 如果构建中的所有作业均已完成，则该构建即被视为已完成。

请注意，如果构建中的某个作业失败，先前已完成的作业可能已将数据写入其输出数据集。

陈旧度

如果构建解析步骤确定输入数据集和 JobSpec 中指定的逻辑自上次构建输出数据集以来未发生更改，则该输出数据集被视为最新。如果输出数据集处于最新状态，则在后续构建中不会重新计算。

若要覆盖构建系统默认的陈旧度行为，您可以执行强制构建，这会将所有数据集作为构建的一部分重新计算，无论它们是否已经是最新的。

分支

Foundry 中的构建通过实施分支来支持数据管道上的协作工作流。如需了解有关分支的更多信息：

  • 请参阅《构建中的分支》部分，了解分支在构建中的运作方式详情。

实时日志

实时日志提供了对正在运行作业的实时可见性，允许您监控作业的进度，并检查流或计算模块等长时间运行的任务。

您可以通过构建应用程序访问实时日志。在查看作业时，选择日志查看器右上角的查看实时按钮即可开始生成。

实时日志的一项关键功能是内置的按日志级别进行颜色编码，这让识别警告和错误并确定其优先级变得更加容易：

  • 信息：蓝色
  • 致命/错误：红色
  • 警告：橙色
  • 调试/跟踪：灰色

此外，安全参数和参数以 JSON 块形式显示，为您的数据提供了结构化且易读的格式。

您可以随时通过从界面右上角选择暂停来停止实时日志流，并从同一位置恢复。

请注意，时间范围选择不适用于实时日志，因为它们是从作业实时流式传输的。

启用后，可能会有十秒的延迟，实时日志才会显示在界面中。
