分支

软件开发人员通常使用版本控制系统来协调代码库中的工作。这使得多名工程师能够安全地参与同一份代码的编写。

在 Foundry 中，我们对待数据的方式与软件开发人员对待代码的方式如出一辙：你需要一种机制，允许许多人对相同的数据进行更改和交互，同时互不干扰。我们汲取了软件开发领域的最佳实践，将其应用于数据管道的编写中，利用了版本控制系统中一项名为分支的常见特性。

从宏观层面来看，分支允许你另辟蹊径，在自己的分支上处理数据。随后，一旦对更改感到满意，你就可以将这些更改从你的分支推回主干道。

分支工作流

如何在 Foundry 中使用分支工作流来更改数据管道代码：

  1. 创建分支。在 Foundry 中，master 分支指的是主数据管道。当你想要处理自己的更改时，你可以创建自己的分支，这为你提供了一个试验和验证想法的环境，而无须担心影响 master 分支。
  2. 创建提交。在你的分支内，你可以对数据转换进行更改。这些更改（包括新增和删除）被称为提交。你的提交会被持续跟踪，从而留下一条清晰的记录，涵盖你在分支上执行的所有更改。
  3. 创建拉取请求。一旦你在独立环境中完成工作，你会希望将更改推回主分支。要启动此过程，你需要创建一个拉取请求。拉取请求会向团队发出信号，表明你已做出更改，并希望他们为 master 数据管道进行审查和验证。
  4. 审查代码。在你创建拉取请求后，你的团队将有机会审查你的提交。每个组织在审查拉取请求时都会有不同的流程或团队配置。

Foundry 的分支机制实现了业界标准的类 Git 版本控制范式。因此，代码仓库应用的设计初衷是，在单个文件上为每个独立分支分配一名活跃开发者。如果其他用户正在你的personal 分支上工作，你的更改可能会被覆盖。为了避免这种情况，我们强烈建议让每个人都在独立的分支中工作。

本页面的剩余部分将描述分支在 Foundry 中运作的技术细节。如果你有兴趣了解如何在实践中使用分支，请按照教程在《管道构建器》或《代码仓库》中创建一个简单的批处理管道。

分支技术细节

在 Foundry 中，数据的分支机制是通过两个不同层面的特性来实现的：在单个数据集内进行分支，以及在构建数据集时进行分支。

数据集分支

正如有关数据集的页面所述，数据集具有以一系列事务形式存在的版本历史记录。从概念上讲，数据集分支类似于 Git 中的分支——数据集、分支和事务分别类似于 Git 中的仓库、分支和提交。

与 Git 中的分支类似，数据集分支仅仅是一个指向该分支上最新事务的指针。因此，一个分支可以看作是按提交时间排序的线性事务序列。当数据集在某分支上因提交事务而发生更改时，所有其他分支上的事务和视图均保持不变。与 Git 不同的是，此处不支持数据集分支的合并。

每个数据集分支都恰好有一个父分支，除非它是根分支。在实践中，Foundry 中的大多数数据集都有一个名为 master 的单一根分支，子分支则是从这个单一根分支创建而来的。

以下是所有受支持的数据集分支相关操作：

  • 创建新分支。可以在没有父分支和没有事务的情况下创建根分支。
  • 创建子分支。可以从另一个分支或从任何事务创建子分支。新分支指向与父分支相同的事务。随后，可以在父分支和子分支上同时启动事务，并且它们的事务指针将独立移动。
  • 在分支上启动新事务。启动事务并将分支指针移动到新事务。
  • 删除分支。删除分支指针，但不删除该分支上的任何事务。请参阅下文保证中关于父分支的说明。

数据集分支保证

Foundry 对数据集分支提供以下保证：

  • 每个分支仅允许一个开启的事务。每个分支最多只能有一个开启（即已开启且既未提交也未中止）的事务，并且该事务始终是分支上的最新事务。因此，只有一个分支的数据集最多只能有一个开启的事务。如果分支指向一个开启的事务，则可以基于该分支创建新分支，但在该事务关闭之前无法启动进一步的事务。
  • 在每个分支上，事务按启动和提交时间排序。此保证是上一条约束的结果。但请注意，对于不同分支上事务的时间先后顺序，不作任何保证。
  • 每个非根分支有且仅有一个父分支。如果删除“中间”分支（即带有子分支的分支），子分支将重新挂载到被删除分支的父分支下（如果被删除的是根分支，则没有父分支）。请注意，在此过程中不会重新排列任何事务；重新设置父分支仅改变分支的祖先记录。

构建中的分支

数据集分支为数据版本控制提供基础语义。Foundry 将数据集分支与 Git 分支相结合，支持同时对逻辑和数据进行分支，使数据工程师在将更改引入生产环境之前，能够安全地对数据管道的更改进行试验。

将 Git 中的逻辑分支与数据集中的数据分支关联起来，是通过 Foundry 的构建系统完成的。每次构建都在用户指定的分支上运行，构建中的作业仅修改该分支上的数据集。因此，构建中的分支提供了一种将不同用户的更改彼此隔离的方法。

Foundry 构建执行涉及分支的两个主要功能，如下所述：

  • 首先，通过从相应分支收集作业规范（或称 JobSpecs），来编译构建图。
  • 其次，根据用户定义的构建分支和一系列回退分支，解析作业输入和输出。

作业图编译

当您在 Foundry 的代码仓库分支中编写数据转换时，提交代码会将 JobSpecs 发布到构建系统的相应分支。在该分支上运行构建时，构建系统会遍历您所在分支上的 JobSpecs 及其依赖项，以确定应执行哪些逻辑。

构建通常会指定分支回退规则，用于规定在构建分支上未设置 JobSpec 时从哪个分支检索 JobSpec。例如，如果您在 develop 分支上运行构建，且回退链为 develop --> master，如果给定输出数据集没有已发布的 JobSpec，则会改为从 master 分支读取 JobSpec。

数据集图标颜色提供了有关 JobSpecs 和分支的信息。如果数据集图标为灰色，表示 master 分支上不存在 JobSpec。如果数据集图标为蓝色，则表示 master 分支上定义了 JobSpec。

输入和输出解析

对于构建中由 JobSpecs 指定为输出的所有数据集，会在构建分支上开启一个事务。如果数据集上不存在该分支，则会从回退链中的第一个分支创建该分支；如果不存在回退分支，则将其创建为根分支。作业输入会尽可能从构建分支读取，否则使用回退链中第一个存在的分支。

示例：在分支上构建

为了理解分支在构建中的工作原理，让我们逐步来看一个示例工作流：

  1. 假设数据集 A、B 和 C 存在于 master 分支上。
  2. 数据工程师在其代码仓库中创建一个名为 feature 的分支。这会在底层 Git 仓库中创建一个分支。
  3. 数据工程师修改生成数据集 B 和 C 的代码。代码提交时，代码库会向 feature 分支上的这两个数据集发布 JobSpecs。
  4. 数据工程师在 feature 分支上发起构建。由于 feature 分支是从 Git 代码库的 master 分支创建的，因此本次构建的回退链为 feature --> master。系统会读取步骤 (3) 中发布的 JobSpecs，并启动与用户代码对应的两个作业。在 feature 分支上，数据集 B 和 C 均开启事务。两个作业按如下方式串行执行：
    1. 作业 1：由于分支回退机制，使用数据集 A 的 master 分支作为输入。计算产生新数据，并将其写入当前开启事务中的数据集 B。随后提交事务。
    2. 作业 2：接着，该作业开始执行。使用数据集 B 的 feature 分支作为输入。计算产生新数据，并将其写入当前开启事务中的数据集 C。随后提交事务。
  5. 最终，由于构建中的所有作业均执行成功，本次构建成功。

修改代码、提交并运行构建后，数据工程师在数据集 B 和 C 的 feature 分支上生成了新数据。需要注意的是，数据集 A 完全未受此过程影响，且数据集 B 和 C 的 master 分支也保持不变。

构建分支保障机制

Foundry 中的构建为分支提供以下保障：

  • 仅当指定的分支回退序列与相关数据集的分支祖迹兼容时，构建解析才会成功。
  • 构建绝不会修改构建分支以外的任何数据集分支。
  • 构建绝不会在输入数据集上创建分支。
