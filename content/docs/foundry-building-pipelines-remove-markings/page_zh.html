移除标记的指导

平台资源的访问权限由 标记（Marking） 控制。标记以一种 全有或全无（all-or-nothing） 的方式限制访问：要访问一个资源，用户必须是该资源上所有标记的成员。此外，标记会沿文件层次结构和直接依赖关系 继承（inherited）。如果您拥有正确的权限，可以直接从资源以及 直接依赖关系 中移除标记。

标记被频繁使用，因为它们在整个平台上易于理解，并且沿着直接依赖关系传播，从而保护敏感数据。在某些情况下，标记可能在管道早期被应用，之后需要在管道中后期移除。本页提供了有关根据您的管道结构如何移除标记的更多信息。

  • 如果您刚开始在管道中应用标记，我们建议阅读 场景（scenarios）
    • 确保您不会意外地将任何最终用户锁定。
  • 此外，我们建议在应用和移除标记时遵循最佳实践，并查阅如何移除继承的标记和组织的文档以获取更多高级信息。

您也可以通过Pipeline Builder 移除标记或组织。

场景

以下是三个与在管道早期应用标记并在管道后期移除它的相关场景。

场景 1：通过移除标记来替代断开

此场景适用于以下情况：

  • 已经应用了标记，
  • 通过断开操作移除了标记，并且
  • 项目级别的传播视图要求权限未开启。

因此，您正在从使用分离（已弃用的功能）迁移到移除继承的标记。

如果启用了传播视图要求，请阅读下面的场景2。

场景2：应用标记后，再进行标记移除以禁用“传播视图要求”

此场景涉及在管道中的数据集中应用标记，以禁用项目级别的传播视图要求设置。

新项目默认禁用了传播视图要求选项，如上所示。对于这些新项目，不会对下游派生数据集强制执行视图要求。具体来说，这意味着在单独的项目中访问数据下游版本的用户不需要同时访问此配置被禁用的项目中的上游数据。

提醒

标记始终会传播。如果新项目中的数据有标记，无论“传播视图要求”设置如何，该标记仍会传播到所有下游数据集。

如果您有启用了传播视图要求选项的项目，如上图所示，则视图要求已传播到这些项目的数据集。这意味着在单独的项目中访问数据下游版本的用户还需要访问启用了此配置的上游项目。

我们强烈建议禁用视图要求传播，转而使用 标记。

在禁用视图需求传播并引入标记到您的管道之前，值得考虑启用传播视图需求的原始目的：

  1. 可能没有充分的理由来保持“传播视图需求”处于启用状态。在这种情况下，您可能只需禁用此设置，然后确保用户仅被授予他们明确需要访问的数据集的权限。用户将不再需要访问以前传播视图需求的上游数据集。
  2. 项目中有一些敏感数据集。在这种情况下，最好将敏感数据集与非敏感数据集隔离，并按照以下步骤操作。这些步骤展示了如何用标记代替视图需求传播，标记是用于这些场景的适当安全原语。
  3. 如果以下步骤无法满足您的需求，请联系您的Palantir代表。

在下面的示例中，我们的目标是在数据源上禁用“传播视图需求” 项目。在按照上述步骤操作后，我们了解到在项目上启用“传播视图需求”的原因是保护 raw_dataset_1 数据集，因为它包含敏感数据。

在旧状态中，查看数据集 A 的内容需要在 数据源 项目和 下游 项目上至少具有“查看者”权限。随后，使用断开功能来移除对数据集 B 的视图需求传播。

在新状态中，查看数据集 A 的内容需要在 下游 项目上至少具有“查看者”权限，并且需要访问标记。请注意，当“传播视图需求”被禁用时，仅需在 下游 项目上具有“查看者”权限即可访问数据集 C 和 D。

此更改允许通过使用安全标记来禁用“传播视图需求”。在下面的解决方案中，我们将在 raw_dataset_1 上应用一个标记，然后该标记将立即传播到所有下游数据集，这些数据集以 raw_dataset_1 作为输入且未被断开连接。假设 断开已处于生效状态，并且断开仅被标记移除所替代。如果在您的情况下不成立，请参见 场景 3，我们在其中详细讨论了在数据集上应用标记的影响。

为了引入此更改，建议按照以下步骤操作，以便在禁用“传播查看需求”后向数据源项目中添加标记时，用户不会失去对数据集B的访问权限：

  1. 创建一个新的标记，并为相关用户授予对该标记的访问权限。
  2. 将该标记应用于 raw_dataset_1。
  3. 现在，一旦标记被应用，您可以安全地在数据源项目中禁用“传播查看需求”。
  4. 在转换中用标记移除替换断开，并确保断开和标记移除同时发生。
  5. 确保管道中的所有数据集都已构建。
  6. 从不再需要访问数据源项目的数据源项目中移除用户。

场景3：在数据集级别应用新的标记后，再移除该标记

这个潜在复杂的场景涉及在现有管道早期引入一个新的标记，而不会不小心在管道后期将用户锁定。

需要注意的是，对数据集 A 引入的标记会立即传播到该数据集下游的所有资源。用户需要该标记才能访问从该标记数据集派生出的任何内容。

为了更好地理解这一点，我们可以通过以下方式扩展上面的例子，添加一个管道：数据集 A → 数据集 B → 下游 数据集：

  • 数据集 A 是一个原始数据集（即将被标记）。
  • 数据集 B 是从数据集 A 派生出来的，但去除了敏感数据（因此可以移除标记）。
  • 下游 数据集是从数据集 B 派生出来的数据集，用户需要访问这些数据集。

我们的目标是确保 下游 数据集不会继承数据集 A 的标记。

我们需要理解的第一件事是，对数据集 A（或包含数据集 A 的文件夹）进行标记，实际上会标记数据集 A 整个历史中的所有事务。因此，数据集 B 和 下游 数据集会立即继承该标记。

如果我们执行以下步骤：

  • 在数据集 A → 数据集 B 的转换中添加标记移除,
  • 更新数据集 B 的代码，并且
  • 重新构建数据集 B ...

... 那么数据集 B 最新的快照事务将没有标记，但数据集 B 所有旧的事务仍然会被标记。

请注意，虽然标记一个数据集会标记其所有交易，但移除转换中的标记只会移除新输出交易的标记。现有交易中的标记不会被移除。这种行为是不对称的。

这意味着，从数据集B的旧交易派生出的下游数据集中的任何数据，例如逐步构建的下游数据集，仍将继承该标记。 然而，Foundry在验证数据集视图的权限时，仅检查每个输入的最新交易。对于增量数据集来说，这意味着常规构建足以移除标记。

为了确保每个增量的下游数据集未被标记，当在数据集B上应用了取消标记转换后，必须重新构建在数据集B上的取消标记转换与增量的下游数据集之间的一切内容。必须重新构建。这将确保每个下游数据集仅依赖于数据集B的最新（未标记）交易，而不是早期（已标记）交易。

如果下游 数据集无法手动触发重建，我们建议采取以下步骤：

  1. 创建一个数据集 A′，并确保其内容与数据集 A 完全相同。
  2. 重写数据集 B 的代码，用数据集 A′ 作为输入，替代数据集 A。同时，确保在转换中添加适当的标记移除操作，然后立即构建数据集 B。无需对数据集 B 进行快照构建。

请注意进行此交换的性能影响，因为它可能会触发数据集 B 的 SNAPSHOT 构建。

  1. 标记数据集 A′。
  2. 将数据集 A 移动到垃圾桶。
  3. 此时不需要进一步的重新构建。

如果你正在重要的管道中进行这些更改，或者对其中的任何步骤不确定，请联系你的Palantir代表以获得帮助。

最佳实践

  • 不要在没有绝对必要的情况下将仓库设置为“不需要重新批准”模式。
    • 如果你必须启用此模式，请确保该仓库的编辑者数量尽可能少，并且确保如果需要的话，获得对该设置的确认（sign-off）。
  • 应为哪些资源需要标记制定明确、定义清晰的标准。
  • 应编写标记移除转换逻辑，主动选择不需要标记的数据。
  • 不要编写过滤掉需要标记的数据的标记移除转换逻辑。这是因为上游可能会经常引入需要标记的新数据，你需要防止它意外地通过你的标记移除转换逻辑。
    • 显示“过滤进入”方法的示例（推荐）： 
      Copied!
      1# column-based
2df.select("salary","title","department")
3
4# row-based
5states_to_keep =["OH","CA","DE"]
6df.filter(df.state.isin(states_to_keep))
    • 显示“过滤出去”方法的示例（不推荐）： 
      Copied!
      1# column-based
2df.drop("firstname","lastname")
3
4# row-based
5states_to_drop =["FL","TX","IL"]
6df.filter(~df.state.isin(states_to_drop))
  • 应在实现敏感数据删除逻辑的转换中执行标记删除。
    • 不应在不同的仓库中抽象或隐藏此逻辑。
    • 同样，不应创建一个单独的标记删除仓库（带有身份转换套件）。标记删除逻辑在标记删除的PR审查过程中应被明确批准。
  • 应在项目内尽可能早地（当标记的数据集作为导入添加时）或尽可能晚地（在创建项目导出的最后一步）从数据中删除标记。

常见问题

每次代码中出现任何逻辑更改时，是否需要对标记移除进行审批？

这取决于你的标记移除转换所在的仓库是否设置了 需要重新审批 或 不需要重新审批 的审批模式。 了解更多关于审批模式。

标记移除工作流和“每个项目一个仓库”的建议不太兼容。在标记移除工作流中，每个项目应该设置多少个仓库？

理想情况下，每当标记移除转换的 逻辑 发生变化时，都应该进行安全审批。为了在审批流程中的过度摩擦与良好的安全态势之间取得平衡，我们建议，如果可以的话，将所有包含标记移除逻辑的转换（如数据混淆、删除列等）移动到一个单独的仓库，并将该单独的仓库设置为“需要重新审批”。

我可以在我的转换中的输出上添加标记移除属性（stop_propagating 和 stop_requiring）吗？

不可以，这些是 输入 属性，不能添加在输出上。如果你的目标是从某个输出中移除一个标记，你需要识别所有携带标记的输入，并分别向它们添加 stop_propagating 语句。更多详情，请参阅 输入转换属性 文档。

哪些语言支持标记移除？

以下语言支持标记移除：

  • Python
  • SQL
  • Java

为什么标记移除优于断开？

为什么移除标记比断开连接更受推荐？

解密应谨慎进行，不应随意分布在项目和仓库中。平台中的标记移除功能提供了对权限传播更改的细粒度控制，并确保此类更改得到适当的审查。

我们正在将仓库转换为使用标记移除而不是断开连接。如何为新的转换禁用断开连接？

联系您的Palantir代表以禁止在之前未启用断开连接的数据集上添加断开连接。
