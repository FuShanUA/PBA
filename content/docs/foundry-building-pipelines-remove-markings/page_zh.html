移除标记指南

平台资源的访问要求由标记控制。标记以全有或全无的方式限制访问：用户必须具备应用于某一资源的所有标记的成员身份，才能访问该资源。此外，标记会沿着文件层级和直接依赖关系继承。如果你拥有正确的权限，可以直接从资源中以及沿着直接依赖关系移除标记。

标记被广泛使用，因为它们在整个平台中清晰可见，并会沿着直接依赖关系传播，从而保护敏感数据。在某些情况下，可能会在数据管道的早期应用标记，而需要在后续阶段将其移除。本页提供了有关如何根据你的数据管道结构移除标记的更多信息。

  • 如果你准备开始在数据管道中应用标记，建议先通读下方的场景部分，确保不会意外锁定任何最终用户。
  • 此外，我们建议在应用和移除标记时遵循最佳实践，并查阅《关于如何移除继承的标记与组织的文档》以获取更多进阶信息。

你还可以使用数据管道构建器移除标记或组织。

场景

以下三个场景与在数据管道早期应用标记并在后续阶段将其移除有关。

场景 1：通过移除标记替代切断操作

此场景适用于以下情况的数据管道：

  • 已应用标记，
  • 标记通过切断方式移除，且
  • 项目级别的传播视图要求权限未开启。

因此，你正在从使用切断（一项已弃用的功能）迁移到移除继承的标记。

如果启用了传播视图要求，请阅读下方的场景 2。

场景 2：应用标记随后移除标记以禁用“传播视图要求”

此场景涉及向数据管道中的数据集应用标记，以禁用项目级别的传播视图要求设置。

如上图所示，新项目默认禁用传播视图要求选项。对于这些新项目，下游派生数据集将不强制执行视图要求。具体而言，这意味着用户在单独的项目中访问数据的下游版本时，不需要同时访问禁用此配置的项目中的上游数据。

提醒

标记总是会传播。如果新项目中的数据带有标记，无论“传播视图要求”设置如何，该标记仍将传播至所有下游数据集。

如果你的项目如上图所示启用了传播视图要求选项，则视图要求已对这些项目中的数据集进行了传播。这意味着用户在单独的项目中访问数据的下游版本时，还需要额外访问启用了此配置的上游项目。

我们强烈建议禁用 视图要求传播，转而使用标记。

在禁用视图要求传播并在数据管道中引入标记之前，有必要考虑一下最初启用传播视图要求的目的：

  1. 可能并没有必须启用“传播视图要求”的充分理由。在这种情况下，你可以直接禁用此设置，然后确保仅向用户授予他们明确需要访问的数据集的访问权限。用户将不再需要访问以前传播视图要求的上游数据集。
  2. 项目中存在部分敏感数据集。此时，最好将敏感数据集与非敏感数据集隔离，并遵循以下步骤。这些步骤演示了如何使用标记（Markings）来替代视图权限要求传播，这正是适用于此类场景的安全原语。
  3. 如果以下步骤无法满足需求，请联系您的 Palantir 代表。

在下面的示例中，我们的目标是禁用 数据源 项目上的“传播视图权限要求”。遵循上述步骤后，我们了解到该项目启用“传播视图权限要求”的原因是为了保护 raw_dataset_1 数据集，因为其中包含敏感数据。

在原有状态下，查看数据集 A 的内容至少需要同时拥有 数据源 项目和 下游 项目的“查看者”权限。随后，通过切断操作移除了数据集 B 上的视图权限要求传播。

在全新状态下，查看数据集 A 的内容需要至少拥有 下游 项目的“查看者”权限以及对该标记的访问权限。请注意，在禁用“传播视图权限要求”后，要求访问数据集 C 和 D 只需拥有 下游 项目的“查看者”权限即可。

此项更改允许通过使用安全标记来禁用“传播视图权限要求”。在下方提议的解决方案中，我们将在 raw_dataset_1 上应用标记，该标记随后会立即传播到所有以 raw_dataset_1 的任何未切断事务作为输入的下游数据集。这里有一个前提假设：切断操作已经实施，且切断操作仅由标记移除来替代。如果您的情况并非如此，请参阅 场景 3，其中我们详细讨论了在数据集上应用标记的影响。

建议按照以下步骤引入此项更改，以避免在 数据源 项目禁用“传播视图权限要求”并添加标记后，用户失去对数据集 B 的访问权限：

  1. 创建一个新标记，并授予相关用户对该标记的访问权限。
  2. 在 raw_dataset_1 上应用该标记。
  3. 应用标记后，即可安全地禁用 数据源 项目中的“传播视图权限要求”。
  4. 在转换中使用标记移除替代切断操作，并确保取消切断与移除标记同时进行。
  5. 确保构建管道中的所有数据集。
  6. 将不再需要访问 数据源 项目的用户移除。

场景 3：应用新标记随后在数据集级别移除标记

这种可能较为复杂的场景，涉及在现有管道早期引入新标记，同时避免在管道后期意外锁定用户。

必须注意，在数据集 A 上引入的标记会沿着事务血缘关系立即传播到该数据集下游的所有资源。用户需要具备该标记权限才能访问任何源自该已标记数据集的内容。

为了更好地理解这一点，让我们在上述示例的基础上扩展一个如下管道：数据集 A → 数据集 B → 下游 数据集：

  • 数据集 A 是一个原始数据集（即将被标记）。
  • 数据集 B 源自数据集 A，并已移除敏感数据（因此可以移除标记）。
  • 下游 数据集是源自数据集 B 的数据集，用户需要访问这些数据集。

在本示例中，我们的目标是确保 下游 数据集绝不会继承来自数据集 A 的标记。

我们首先需要明确的是，对数据集 A（或包含数据集 A 的文件夹）进行标记，实际上等同于对数据集 A 整个历史记录中的所有事务进行标记。因此，数据集 B 和 下游 数据集会立即继承该标记。

如果我们执行以下步骤：

  • 在数据集 A → 数据集 B 的转换中添加标记移除操作，
  • 更新数据集 B 的代码，并且
  • 重新构建数据集 B ......

......那么数据集 B 上的最新快照事务将不包含标记，但数据集 B 上的所有旧事务仍将带有标记。

请注意，虽然对数据集进行标记会标记其所有事务，但在转换中移除标记仅会移除新输出事务的标记。现有事务上的标记不会被移除。此行为是 非 对称的。

这意味着，基于数据集 B 早期事务派生出的任何下游数据集（例如增量构建的下游数据集）仍将继承标记。 然而，Foundry 在校验数据集视图权限时，仅会检查每个输入的最新事务。对于增量数据集而言，这意味着常规构建即可去除标记。

为确保每个增量下游数据集都不再带有标记，在数据集 B 应用去除标记的转换后，位于该转换与增量下游数据集之间的所有内容都必须重新构建。这样可以保证每个下游数据集仅依赖于数据集 B 的最新（已去除标记）事务，而非早期（带有标记的）事务。

如果下游数据集数量庞大，手动触发重新构建不切实际，我们建议采取以下步骤：

  1. 创建数据集 A′，并确保其内容与数据集 A 完全一致。
  2. 重写数据集 B 的代码，使用数据集 A′ 替代数据集 A 作为输入。同时，务必在转换中添加适当的标记去除操作，并随即构建数据集 B。此处无需对数据集 B 进行快照构建。

请考虑执行此替换操作对性能的影响，因为这可能会触发数据集 B 的 SNAPSHOT 构建。

  1. 为数据集 A′ 添加标记。
  2. 将数据集 A 移至回收站。
  3. 此时无需进行进一步的重新构建。

如果你正在重要的管道中进行这些更改，或对其中任何步骤存有疑问，请联系你的 Palantir 代表获取协助。

常见问题解答

每次代码逻辑发生变更时，都需要对标记移除进行审批吗？

这取决于包含标记移除 transform 的代码仓库所设置的审批模式是需要重新审批还是无需重新审批。了解有关审批模式的更多信息。

标记移除工作流与“每个项目一个代码仓库”的建议不太兼容。对于标记移除工作流，我们每个项目应该设置多少个代码仓库？

理想情况下，每当标记移除 transform 的逻辑发生变更时，都应经过安全审批。为了在审批流程的过度摩擦与良好的安全态势之间取得平衡，我们建议，如果条件允许，将所有包含标记移除逻辑的 transform（如数据混淆、移除列等）迁移至单独的代码仓库，并将该独立仓库设置为“需要重新审批”。

我可以在 transform 的输出中添加标记移除属性（stop_propagating 和 stop_requiring）吗？

不可以，这些是输入属性，不能添加到输出上。如果您的目标是从特定输出中移除标记，您需要识别出所有携带标记的输入，并分别向它们添加 stop_propagating 语句。有关更多详细信息，请参阅输入 transform 属性文档。

哪些语言支持标记移除？

以下语言支持标记移除：

  • Python
  • SQL
  • Java

为什么优先选择标记移除而不是切断？

降密操作应当谨慎进行，而不应分散在各个项目和代码仓库中。平台中的标记移除功能对权限传播变更提供了细粒度的控制，并确保此类变更得到适当的审查。

我们正在将代码仓库过渡为使用标记移除代替切断。我们如何为新的 transform 禁用切断？

请联系您的 Palantir 代表，以禁止在之前未启用切断的数据集上添加切断功能。
