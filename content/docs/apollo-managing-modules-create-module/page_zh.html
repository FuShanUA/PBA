创建模块

前提条件

变量

您可以在模块的 variables 块中定义任何 变量。对于每个变量，您应定义以下配置项：

  • 名称：变量的名称，例如 myNamespace。
  • type：变量的类型。目前，Apollo 支持两种主要类型的变量：
    • 标准字符串变量，使用 type: string。
    • 预设变量，使用 type: options 和 type: stringPreset 一起。

只有在模块 YAML 文件的 variables 块中定义了变量时，才能使用大括号语法（{{ moduleVariable.myNamespace }}）来引用该变量。如果变量在模块定义中的其他地方仍被引用，则不能从现有模块的 variables 块中删除该变量。

字符串变量

字符串变量允许您在安装模块时输入文本值。字符串变量有以下特定配置选项：

  • description：对变量的描述，用于在安装模块时为用户提供指导。
  • defaultValue（可选）：安装模块时变量的默认值。如果没有默认值，则在安装模块时必须输入一个值。
  • constraint（可选）：变量可接受值的限制。如果指定了默认值，该默认值必须满足约束条件。支持的约束类型包括：
    • regex：值必须与指定的正则表达式匹配。请注意，验证时假设两边都带有锚定。
    • oneOf：值必须与预定义值中的一个匹配。

预设变量

通常，相关变量需要作为一组进行配置。通过预设变量，您可以定义已知的变量配置，并通过一个直观的控件将其暴露出来。常见的用例是资源配置文件，您可以创建“小”、“中”和“大”预设，以自动为目标环境正确配置资源。例如，选择“小”选项会自动将所有相关服务的CPU和内存值设置为较小的数值。

预设变量由两种协同工作的变量类型组成：

  1. options：定义可用选项，例如 [small, medium, large]。
  2. stringPreset：将每个选项映射到特定的字符串值。Apollo会根据所选选项解析变量的值。

当安装一个带有预设变量的模块时，您将选择一个选项，所有相关的预设变量会根据所选选项自动配置。

以下是预设变量的示例：

内容：

预设变量的重要注意事项：

  • 每个 stringPreset 变量必须通过其 options 字段引用一个 options 变量。
  • 每个 stringPreset 映射必须包含所引用的 options 变量中定义的所有选项。
  • 在安装时，用户从 options 变量中选择一个选项，而不是为每个 stringPreset 变量单独选择值。您不能在安装过程中直接覆盖每个 stringPreset 变量。
  • 当您在新模块版本中更新 stringPreset 变量的映射时，新值会自动传播到现有安装中。例如，如果您将内存变量的“small”映射从“1Gi”更新为“1.5Gi”，则所有使用“small”预设的安装在更新到新模块版本时会自动更新为使用“1.5Gi”。

实体

内容： 您可以在模块的 YAML 文件中的 entities 块中为模块的每个实体定义特定的配置。

请注意，当您引用一个 变量 时，该值应使用双大括号包围，并且应在变量名前加上 moduleVariable 前缀。例如，您可以通过编写 {{ moduleVariable.myNamespace }} 来引用 myNamespace 变量。对于每个实体，您应定义以下配置项：

  • type: 实体的产品类型。目前仅支持 Helm 图表。
  • productId: 实体的 产品 ID，即产品的 Maven 坐标 的 groupId:artifactId。
  • entityName（可选）：实体的标识符。默认值为实体的 ProductId 中的 artifactId。
  • k8sNamespace: 实体应安装的 Kubernetes 命名空间。
  • 内容： ignoredDependencies: Apollo 会忽略您在被忽略的依赖项中列出的产品的任何 产品依赖约束。如果没有，请输入一个空列表，如下例所示。
  • configOverrides（可选）：您可以在模块 YAML 文件中定义 实体配置覆盖。
  • installConfigOverrides（可选）：仅在实体首次安装时应用的配置覆盖。创建后，这些键将成为实体所有，并且不会由模块更新。需要 syncConfigOverrides: MERGE。有关详细信息，请参阅下方的 仅安装时的配置覆盖。
  • syncConfigOverrides（可选）：确定 Apollo 在安装模块或更新现有模块安装时如何处理实体的现有配置覆盖。默认为 ENABLED。有关选项，请参阅 使用模块管理配置覆盖。
  • markedForUninstallation（可选）：如果此字段存在，当安装模块时，Apollo 会将实体标记为卸载。如果该实体在环境中不存在，Apollo 在安装模块时会忽略该实体。
    • minVersion（可选）：应卸载的实体的最低版本。如果未指定，Apollo 会卸载该实体的所有版本。
    • unmanageAfterUninstall（可选）：如果设置为 AUTOMATIC，Apollo 在卸载实体后会将其设为非管理状态。如果设置为 MANUAL，Apollo 在卸载实体后会保持其为管理状态。默认值为 AUTOMATIC。
  • 内容： secretRequirements（可选）：您可以指定一组必须存在的密钥名称和相关密钥列表，才能安装或升级模块。模块安装界面将引导用户为具有此字段的实体设置正确的密钥。请参阅下面的完整示例以了解密钥要求的语法。
  • dependencyGroups（可选）：您可以将实体分配到一个或多个依赖组。只有当实体与依赖实体共享至少一个组时，才会认为依赖关系已满足。如果没有指定，实体将默认使用default组。
  • dependencyGroupOverrides（可选）：您可以为特定依赖项覆盖默认的依赖组行为。这允许您指定在解析特定产品的依赖项时要搜索哪些组。使用通配符"*"可允许实体从任何组满足依赖项。

模块 YAML 文件中每个实体的所有非可选字段都必须存在。如果您在创建模块时不知道某个特定值，可以使用 变量 在模块安装时定义该值。

以下是一个实体定义示例：


内容：

已复制！
1实体: 2 - 类型: helmChart 3 helmChart: 4 产品ID: com.palantir.example:backend-service 5 k8s命名空间: "{{ moduleVariable.myNamespace }}" 6 syncConfigOverrides: 合并 7 configOverrides: | 8 2.0.0: 9 覆盖项: 10 线程数: 100

如果已存在同名的实体，Apollo 将覆盖现有实体的设置以匹配模块定义中的设置。

实体安装顺序

您可以通过在产品发布清单中声明 产品依赖项 来控制模块中实体的安装顺序。

当模块安装到环境中并因此声明实体时，Apollo 会将每个实体之间的产品依赖关系作为约束评估的一部分进行计算。这将确定实体的安装顺序。

内容： 例如，假设您有一个包含数据库实体和应用程序实体的模块。为了确保在应用程序之前安装数据库，应用程序产品发布清单必须使用 product-dependencies 扩展 指定对数据库的依赖关系：

1"com.example:app-service" 产品发布的示例清单 2extensions: 3 product-dependencies: 4 - product-group: com.example 5 product-name: database-service 6 最低版本: 1.0.0 7 最高版本: 1.x.x
内容：

了解有关 通过依赖项发布产品版本 以及 安装顺序 的详细规则。

子模块

您可以通过在 submodules 列表中引用其他现有的模块来创建一个 复合模块。这使您可以将复杂的配置分解为更小、可重用的组件进行管理。

以下是将基本子模块包含在模块 YAML 文件中的结构：


内容：

已复制！
1submodules:
2  - name: "submodule-name"
3    version: 0.3.0
内容：

对于每个子模块，您应该定义以下配置项：

  • name：要用作子模块的模块名称。这必须是已存在于Apollo中的模块。
  • version：要作为子模块使用的模块的版本。给定父模块版本的提供版本是不可变的；要更新包含的子模块，必须创建父模块的新版本。
  • key（可选）：当同一模块多次用作子模块时，需要一个唯一的键来区分它们。
  • variableOverrides（可选）：允许覆盖子模块中定义的任何变量。覆盖可以硬编码一个值，或者将子模块的变量重新映射到父模块中定义的变量。
  • entityOverrides（可选）：允许覆盖子模块中定义的实体的配置和密钥。这可以用于在父模块内自定义子模块的行为。

以下是复合模块的一个示例定义：


内容：

已复制！
1名称: control-plane-module 2displayName: 控制平面模块 3description: 包含2个子模块的模块。 4variables: 5 parentNamespace: 6 type: string 7 string: 8 defaultValue: "default" 9 description: "Default Namespace for all Entities" 10子模块: 11 - 名称: "backend-module" 12 版本: 0.3.0 13 变量覆盖: 14 myNamespace: "{{ moduleVariable.parentNamespace }}" 15 - name: "frontend-module" 16 version: 0.2.0 17 variableOverrides: 18 enableExperimentalFeature: "true"

在此示例中，control-plane-module 包含两个子模块：backend-module 和 frontend-module。在 backend-module 中定义的 myNamespace 变量被覆盖，改为使用在父 Module 中定义的 parentNamespace 变量提供的值。在 frontend-module 中定义的 enableExperimentalFeature 变量被硬编码为 true。在安装时不需要为其提供值。

有关子模块的更深入信息，请参阅 模块组合。

Submodule usage restrictions

To prevent parent composite Modules from applying Entity overrides to your submodule, add the following to your Module definition:



Copied!
1restrictions:
2  submoduleUsage:
3    allowEntityOverrides: false

When allowEntityOverrides is set to false:

  • Parent Modules cannot use entityOverrides to modify the configuration for Entities in your Module (for example configOverrides, secretRequirements, ignoredDependencies). Attempts to create a composite Module with entityOverrides will fail validation.
  • Parent Modules can still use variableOverrides to customize your Module's behavior.

For example, a database Module might have configOverrides that are not intended for parent Modules to customize. When this database-module is used as a submodule in a composite Module, the composite Module cannot modify any configOverrides.



Copied!
1name: database-module 2displayName: Database Module 3description: A database module with strict configuration requirements 4restrictions: 5 submoduleUsage: 6 allowEntityOverrides: false # Prevents parent modules from using entityOverrides to override anything 7variables: 8 postgresReplicated: 9 type: string 10 string: 11 defaultValue: 'false' 12 description: ... 13 backupTuningParameter: 14 type: string 15 string: 16 defaultValue: '3' 17 description: ... 18 postgresMaxConnections: 19 type: string 20 string: 21 description: ... 22entities: 23 - type: helmChart 24 helmChart: 25 productId: com.example:database 26 k8sNamespace: databases 27 configOverrides: | 28 1.0.0: 29 overrides: 30 postgresql: 31 name: postgres-shared-cluster 32 feature-flag-1: true 33 feature-flag-2: true 34 postgresql-version: '17' 35 data-storage: 400Gi 36 replicated: "{{moduleVariable.postgresReplicated}}" 37 env: 38

Publish the Module to Apollo

Run the following command in your terminal to publish your Module to Apollo.



Copied!
1apollo-cli module create --module-definition-file /path/to/yaml/file.yml

Learn more about the apollo-cli module create command.

Module versions

Once created, each Module will be automatically assigned a version. Versions starts at 0.1.0 and the minor version is incremented each time you update your Module. Versions are immutable, and once created cannot be edited or deleted.

View your Module

You can confirm that your Module was published in the Modules tab of the Software Catalog.

Edit a Module

You can edit a Module by publishing a new version of the Module YAML file. To download the Module YAML file, navigate to the Module overview page and select Edit module from the Actions dropdown.

This will open a dialog where you can select Download module definition.

You can edit the Module YAML file and publish it to Apollo using the Apollo CLI.

Once the new version of the Module is published you can view it in the version selector next to the Module name.

You can also view the version of each Module installation in the Module installations table.

Note that Module versions are immutable. In addition, when a new version of a Module is published, Apollo will attempt to update existing Module installations to the latest version of the Module by proposing a change request.

When creating a new version of a Module, you should consider how the changes will impact existing installations of the Module. The following sections outline best practices for evolving Modules.

Evolving a Module: adding a new variable

If you introduce a new Module variable, you should provide a defaultValue so that all existing installations of your Module can be seamlessly updated to the new version.

If you do not supply a defaultValue, the Module update will be blocked until users enter a value for this variable.

Evolving a Module: deleting a variable

To delete a variable, you must ensure it is not used anywhere in the definition of the Module. In the example below, the enableFeatureFlag variable cannot be deleted because it is still referenced in config overrides.



Copied!
12.0.0:
2  overrides:
3    my-feature-flag: {{ moduleVariable.enableFeatureFlag }}

One way you can eliminate the reference is by replacing it with a hard-coded value, such as true. When existing installations are updated to the new version of your Module, whatever value had previously been chosen for the enableFeatureFlag variable will be overwritten to true.



Copied!
12.0.0:
2  overrides:
3    my-feature-flag: true

Another way to eliminate this reference is to delete the reference entirely:



Copied!
12.0.0:
2  overrides: {}

Finally, you can replace it by referencing a different variable - as long as this other variable is defined in the Module definition.



Copied!
12.0.0:
2  overrides:
3    my-feature-flag: {{ moduleVariable.enableNewFeatures }}

After you remove all references to the variable, you can delete the variable itself from the Module definition.

Evolving a Module: renaming a variable

There is no way to rename a variable while preserving all the values that were chosen by existing Module installations. This is because Apollo cannot differentiate between a change intended to rename one variable in a Module definition from a change intended to delete one variable and create a new one.

Evolving a Module: uninstalling an Entity

If you want Apollo to uninstall a specific Entity from every installation of your Module, you must explicitly signal this to Apollo using the markedForUninstallation field (see example below).



Copied!
1name: control-plane-module 2displayName: Control Plane Module 3description: Entities required for the application control plane. 4entities: 5 - type: helmChart 6 helmChart: 7 productId: com.palantir.example:deprecated-application 8 k8sNamespace: app-namespace 9 markedForUninstallation: {}

Once Module installations are updated to this version of your Module, Apollo will uninstall the Entity (subject to dependency constraints).

You can choose to uninstall a Product only after it has upgraded to some version, for example, when it has finished a migration or offloaded some responsibility to another service. To do this, you can specify a minVersion at which the Entity will become eligible for uninstallation.



Copied!
1name: control-plane-module 2displayName: Control Plane Module 3description: Entities required for the application control plane. 4entities: 5 - type: helmChart 6 helmChart: 7 productId: com.palantir.example:deprecated-application 8 k8sNamespace: app-namespace 9 markedForUninstallation: 10 minVersion: 1.45.0

If you remove an Entity from a Module definition without specifying the markedForUninstallation field then existing installations of the Entity will remain installed but will no longer be managed by this Module. This can be useful if you want to seamlessly transfer the responsibility for managing a particular Entity from one Module to another. The original Module can stop managing the Entity, allowing the new Module to take over and start managing the Entity.

Evolving a Module: Moving an Entity from one Module to another Module

You can move an Entity from an old Module definition to a new Module definition without disrupting installations of that Entity. Entities can only be managed by one Module at a time. In the example below, we will move the monitoring-service Helm chart from the core-services Module to a new observability Module.

To move an Entity between Modules:

  1. Delete the Entity from the old Module definition: Do not mark it for uninstallation, just delete the Entity from the Module definition. When installations of this old Module are updated, the Entity will remain installed, but no longer managed by this Module.
Copied!
1 name: core-services 2 displayName: Core Services 3 description: Essential services for the platform 4 entities: 5- - type: helmChart 6- helmChart: 7- productId: com.palantir.example:monitoring-service 8- k8sNamespace: monitoring 9 - type: helmChart 10 helmChart: 11 productId: com.palantir.example:auth-service 12 k8sNamespace: auth
  1. Add the Entity to the new Module: Update the new Module definition to include the Entity. You may need to carry over any variables that were referenced by this Entity.
Copied!
1name: observability 2displayName: Observability Module 3description: Monitoring and observability tools 4variables: 5 k8sNamespace: 6 type: string 7 string: 8 description: "k8s namespace to deploy observability helm-charts into" 9entities: 10 - type: helmChart 11 helmChart: 12 productId: com.palantir.example:monitoring-service 13 k8sNamespace: "{{ moduleVariable.k8sNamespace }}"

  1. Update old Module installation: Update installations of the old Module so that it relinquishes control over the Entity.

  2. Update new Module installation: Once the Entity is no longer managed by any Module, your new Module can now take it over. Update the installations of the new Module, which will start managing the Entity. You will need to provide values for any newly added variables, such as k8sNamespace in the example above. There should be zero diff.

The result is that zero Plans are issued for the Entity, it remains installed throughout the entire process.

Evolving a Module: Managing config overrides with Modules

Modules provide a way to manage config overrides for Entities in Apollo across multiple Environments. When you define an Entity in a Module, you can specify config overrides for the Entity in the configOverrides field. You can use variables to add flexibility to the config overrides of an Entity. The example below shows the config overrides for an Entity in a Module that defines a variable enableFeatureFlag:



Copied!
12.0.0:
2  overrides:
3    my-feature-flag: {{ moduleVariable.enableFeatureFlag }}

When installing the Module, you will be prompted to provide a value for the variable. You can also decide to use the default if specified in the Module definition.

The syncConfigOverrides field on an Entity (see Entity configuration options) controls how Apollo handles existing Entity config overrides when installing a Module onto an existing Environment or updating an existing Module installation. With all options, when adding a new Entity, Apollo will add the config overrides specified in the Module definition regardless. The field supports the following values:

  • ENABLED (default): Updates to existing Entities will use the config overrides specified in the Module definition, overwriting any existing config overrides on the Entity. This allows you to reduce config overrides divergence across Environments where the Module is installed. You should use variables to handle cases where specific values in the config overrides are expected to be different across Environments.
  • MERGE: Config keys in the Module definition will always overwrite matching config keys in Entity config overrides. All other config keys in Entity config overrides will be left untouched. Users cannot override any config keys in config overrides that are controlled by a module. You can use the magic value {{ unset }} in a Module definition to remove a config point from Entity config overrides and prevent users from being able to set it in Entity config overrides. Removing a config point from a Module will leave it untouched in existing Environments, but allow it to be edited freely.
  • DISABLED: Updates to existing Entities will always keep the Entity's config overrides instead of using the config overrides specified in the Module definition. New Module installations that start managing an existing Entity will also leave the Entity's config overrides untouched.

Example: ENABLED, MERGE, and DISABLED values

The following example illustrates the behavior of each option. Given this existing Entity config:



Copied!
13.0.0: 2 rolling-restart-config: 3 conf: 4 key1: a 5 key2: b 6 key3: c

And this config in the Module definition:



Copied!
13.0.0: 2 rolling-restart-config: 3 conf: 4 key1: x 5 key3: "{{ unset }}" 6 key4: d

The resulting config with the MERGE strategy would be:



Copied!
13.0.0: 2 rolling-restart-config: 3 conf: 4 key1: x # Value overridden by the Module 5 key2: b # Untouched from the Entity 6 # key3 removed entirely 7 key4: d # Added by the Module

With the ENABLED strategy, the Entity config is replaced entirely by the Module definition (minus any {{ unset }} keys):



Copied!
13.0.0:
2  rolling-restart-config:
3    conf:
4      key1: x
5      key4: d

With the DISABLED strategy, the Entity config is left entirely unchanged:



Copied!
13.0.0: 2 rolling-restart-config: 3 conf: 4 key1: a 5 key2: b 6 key3: c

Example: Removing a key from the Module definition

If you then updated the Module definition example above to remove a key:



Copied!
13.0.0: 2 rolling-restart-config: 3 conf: 4 # key1 removed from the module definition 5 key3: "{{ unset }}" 6 key4: d

With the MERGE strategy, the resulting config would be unchanged, since removing a key from the Module definition will not remove it from the Entity config:



Copied!
13.0.0: 2 rolling-restart-config: 3 conf: 4 key1: x 5 key2: b 6 key4: d

Similarly, with the DISABLED strategy, the Entity config remains unchanged from the original:



Copied!
13.0.0: 2 rolling-restart-config: 3 conf: 4 key1: a 5 key2: b 6 key3: c

With the ENABLED strategy, the resulting config would remove key1 entirely, since config is always synced exactly:



Copied!
13.0.0:
2  rolling-restart-config:
3    conf:
4      key4: d

Install-only config overrides

When using the syncConfigOverrides: MERGE behavior, you can set the installConfigOverrides field to specify config overrides that will only be applied when an Entity is first installed. After creation, these keys become Entity-owned and are never updated by the Module. This is useful for providing defaults for config values that Environments may need to diverge on.

On first install, Apollo applies both configOverrides and installConfigOverrides. On subsequent Module updates, only configOverrides are applied — installConfigOverrides keys are not modified. New keys added to installConfigOverrides in later Module versions only take effect on future first-time installations. For instance, in the example Helm chart definition below, Apollo will always sync the value of feature-flag to all installations of the Module when it gets updated. The resources config will only be synced once when the Entity is first installed by the Module, and after that the resources field will never be updated and users can update the memory and CPU values as needed.



Copied!
1 - type: helmChart 2 helmChart: 3 productId: com.palantir.example:database-service 4 k8sNamespace: "{{ moduleVariable.myNamespace }}" 5 syncConfigOverrides: MERGE 6 configOverrides: | 7 2.0.0: 8 overrides: 9 feature-flag: enabled 10 installConfigOverrides: | 11 2.0.0: 12 overrides: 13 resources: 14 requests: 15 cpu: 8 16 memory: 4Gi

Pausing Module updates

You can temporarily pause a Module installation to stop all Module updates and freely edit Module-owned Entity config overrides. This is intended as a temporary break-glass mechanism for situations where you need to make urgent configuration changes without waiting for a Module update.

When you pause a Module installation:

  • Apollo cancels any open Module update change requests.
  • Apollo prevents new Module update change requests from being created.
  • You can edit config override keys that are normally controlled by the Module.

To pause updates for a Module installation, navigate to the Installations tab on the Module page or the Modules tab on the Environment page and select the Pause Module updates option from the dropdown accessible when you select the ellipsis icon:

To resume normal Module updates, re-enable updates for the Module installation. Apollo will open a change request to update the Module to the latest version, which reconciles any configuration divergence introduced while Module updates were paused. Until this change request merges, the Module installation will show as Resuming Updates.

Delete a Module

You can only delete a Module if there are no remaining installations of the Module. You must unlink the Module from all installations before you can delete it.

To delete a Module, you can navigate to the Module page and select Delete module from the Actions dropdown.

This will remove the Module from the Software Catalog.
