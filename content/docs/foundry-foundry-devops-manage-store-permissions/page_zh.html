管理商店权限

Marketplace 商店可以是 Foundry 租户本地的，也可以是远程的。本地 Marketplace 商店位于项目或文件夹中，并继承其所在项目或文件夹的权限。远程商店在一个 Foundry 租户上创建，随后提供给其他租户使用。远程商店的权限需在控制面板中配置。

查看权限

要在 DevOps 或 Marketplace 中查看本地 Marketplace 商店，您需要具备 marketplace:read-local-marketplace 操作权限，该权限通常通过 Viewer 角色授予。远程商店的 Viewer 权限需在控制面板中配置。

安装产品权限

要从本地或远程商店安装产品，您必须能够查看该商店，并具备 marketplace:install-from-local-marketplace 操作权限，该权限通常通过 Viewer 角色授予。

对于选作此次安装输入项的每个资源，您必须具备 marketplace:use-resource-as-input 操作权限，该权限通常也通过 Viewer 角色授予。

此外，允许安装的目标位置（通常为 Space 和 Ontology）要求具备 marketplace:install-in 操作权限，该权限通常通过 Editor 角色授予。

每次安装时，Marketplace 会在选定的 Space 中创建一个新项目，或将其安装到现有项目中。为此，您需要对 Space、所选项目或文件夹具备 marketplace:install-in 操作权限。此权限通常通过 Editor 角色授予。

您还必须拥有对商店上存在的至少一个组织标记的访问权限。不过，由于访问包含该商店的项目本身就已经要求具备对其中一个组织标记的访问权限，因此这一条件会自然满足。

创建商店权限

要创建本地商店，您必须在项目或文件夹中具备 marketplace:create-local-marketplace 操作权限，该权限通常通过 Editor 角色授予。

目前，远程商店只能由 Palantir 创建。

编辑产品权限

要在本地商店中创建或编辑产品，您必须具备 marketplace:create-block、marketplace:edit-block-set 和 marketplace:upload-attachment 操作权限，这些权限通常授予 Editor 角色。

无法在 DevOps 中编辑远程商店。

导出产品权限

要从本地商店导出产品，用户必须具备 marketplace:export-block-set 操作权限，该权限通常授予 数据主人 角色。目前，用户无法从远程商店导出产品。

导入产品权限

要向本地商店导入产品，用户必须具备 marketplace:import-blockset-with-provenance 操作权限，该权限通常授予 数据主人 角色。目前，用户无法向远程商店导入产品。

编辑商店标签权限

要编辑本地商店上的标签，用户必须具备 marketplace:edit-local-marketplace 操作权限，该权限通常授予 Editor 角色。目前，用户无法编辑远程商店上的标签。

应用于产品创建的组织标记

打包在 Marketplace 产品中的所有资源都会带有一个或多个组织标记，这些标记通常继承自存储资源的项目。 同样，Marketplace 商店也会带有其所在项目的组织标记。

如果产品资源与 Marketplace 商店的组织标记不一致，您必须为这些组织标记获取扩展访问权限。 例如，假设某个 Workshop 应用属于组织 A，而该商店同时属于组织 A和组织 B。要在 Marketplace 商店中成功打包此 Workshop 应用，需要针对组织 A的扩展访问权限，因为您正在将内容从组织 A扩展到组织 B。

如果产品资源拥有 Marketplace 商店所不具备的组织标记，您必须获取这些组织标记的移除权限。 例如，假设某个 Workshop 应用同时属于组织 A和组织 B，而该商店仅属于组织 A。要在 Marketplace 商店中成功打包此 Workshop 应用，需要针对组织 B的移除权限，因为您正在移除（或取消标记）组织 B的标记。

作为用户，您可能无权查看资源所属的所有组织。

应用于产品安装的组织标记

通常，Marketplace 商店必须包含您要安装内容的空间所需的所有相关组织标记。这意味着本地 Marketplace 商店必须位于一个项目中，且该项目需具备您要安装内容的空间所需的所有相关组织标记。

例如，如果 Marketplace 商店仅具有组织 A的标记，而您希望将该商店的产品安装到同时包含组织 A和组织 B的空间中，则在安装时必须获取组织 A的扩展访问权限，因为您正在将内容从组织 A扩展到组织 B。

在安装过程中，您也可以选择仅将组织 A的标记应用于您的产品安装；这样可以免去扩展访问权限的需要。

或者，您可以将组织 B 的标记添加到商店中，这将允许更多用户从该商店安装产品。例如，如果 Marketplace 商店同时具有组织 A 和 B 的标记，而您希望将该商店的产品安装到仅包含组织 A 的空间中，则无需任何额外权限。但是，仅拥有组织 B 访问权限的用户也将能够从该商店安装产品。

要求批准新产品版本

可将 Marketplace 商店配置为在发布新产品版本前需要获得批准。启用此功能后，审批者必须先审查并批准每个草稿，然后才能最终确定新版本。

审批用户必须与草稿作者不同，并且必须对该商店具有marketplace:finalize-block-set操作权限。此操作默认授予商店数据主人和编辑者。要自定义允许的审批者列表，请使用自定义角色集。

产品创建和安装权限

多组织场景

在 Foundry 中，要在一个项目与另一个项目之间移动资源，您必须是该资源的数据主人并具备足够的权限；但是，如果移动操作会扩大移动后可访问该资源的组织范围，您还需要对所涉及的标记拥有额外权限。

例如，假设某个资源位于属于组织 A的项目 A中，而您希望将该资源移动到同时属于组织 A和组织 B的项目 B中。您必须拥有针对组织 A标记的扩展访问权限。扩展访问权限是一项提升的权限，允许您将资源（在此例中属于组织 A）的访问范围扩展到其他组织（如组织 B）。

如果资源带有指示个人身份信息或其他敏感数据的标记，您必须拥有移除标记权限才能移除该特定标记。

如果 项目 A 中的资源被打包为产品并存储在 项目 B 中，随后该产品又被安装到 项目 C 中，您必须拥有足够的权限，既能 扩展 组织标记，也能在存在额外标记时将其 移除，因为这些标记会在产品移动过程中被移除。

权限结构指南

为了避免在产品创建、安装及后续环节中出现阻碍，我们建议遵循以下原则：

  • 资源最初应放置在属于单一组织（专用于打包资源）的项目中，且不带任何额外标记。
    • 负责将资源打包为产品的用户必须对该单一组织拥有上文列出的充分权限，这些权限可授予给更广泛的用户群体。
  • 任何计划从商店安装产品的组织，都应有权访问市场商店项目所在的 Space。您可以为此目的创建一个专属 Space，并将需要访问商店的各个组织纳入其中。
    • 安装产品的用户无需对组织标记拥有额外权限即可执行安装操作。

这种权限结构能够将所有潜在的阻碍转移至打包阶段，从而让您在安装流程开始前完成必要的调整。
