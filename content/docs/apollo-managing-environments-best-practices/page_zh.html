将软件部署到新环境的最佳实践

本节概述了将软件部署到新环境的最佳实践。这些建议在以下几种情况下很有帮助，例如：

  • 在完全运行的测试环境部署后首次在生产环境中安装
  • 在 spoke 环境之间迁移软件
  • 部署到新的 hub 或网络

Apollo 模块

模块 是 Apollo 的一种资源，允许您定义产品和配置覆盖的分组。您可以将模块部署到多个环境，导出 到新的中心，或管理您环境中现有的产品。当您的平台包含多个需要安装的 Helm 图表时，或者您管理一个环境群，并希望确保整个群中的安装和配置保持一致时，这会很有帮助。此外，您还可以在不同版本之间设置 产品依赖项，以确保在新环境中按正确顺序安装产品。

我们建议创建一个 用于模块的 Git 仓库，用于存储模块的定义。这使您可以对模块定义进行版本控制，并在您的 CI 管道中自动将新定义发布到 Apollo。

当您将模块部署到一个空的环境时，该模块将触发定义中产品的安装计划。如果您将模块部署到一个 已有的环境，该环境中实体将自动由该模块进行管理。这两种操作都会创建一个变更请求，您可以查看该模块对环境提出的更改内容，无论是产品安装、配置更改，还是对现有实体不执行任何操作。

当您对模块进行 更改 时，这些更改会传播到已经订阅该模块的每个环境。这在您需要在整个平台中进行配置更改或希望在舰队中安装新产品时非常有用。这些更改会产生一个变更请求，您可以审查以确认该模块在环境中想要做出的更改。

变量

以下值不应在应用程序配置中硬编码，而应在安装期间选择使用模块变量或创建密钥：

  • 镜像拉取密钥名称
  • 命名空间
  • 域名别名
  • 注册表
  • 实例组
  • CAs
  • 云资源区域/端点

虽然这些值在不同环境之间可能保持不变，但我们建议将这些值抽象为变量，以确保您能够灵活地执行如更改命名空间名称或密钥值等操作。要使用模块管理密钥，请参阅模块文档中的 密钥要求。

一次性设置

在将应用部署到新环境时，您应确保在向该环境部署实体之前配置了以下资源：

  • 命名空间
  • 实例组
  • 工件注册表
  • 入口和出口策略

除了为您的新环境准备一个模块进行部署外，您还应制定一个计划，在部署前必须完成这些一次性更改。

发布管道

您可以利用 产品评估和发布 来确定哪些产品将被交付到不同的环境。在模块定义中，您可以指定一个发布通道变量来指示哪些发布版本将被安装到该环境。这使您能够确保部署到特定环境的模块遵循该环境的正确发布通道，例如确保生产环境遵循所有生产产品安装的要求。

评估标准的一些示例包括

  • Pod健康要求，例如，发布版本在预发布环境中至少保持健康24小时。
  • 安全扫描结果，例如，通过 security-scan-outcome 标签确保漏洞扫描已通过。
