配置环境设置

环境的设置选项卡允许您查看和编辑环境及其管理实体的设置。

编辑环境设置将创建一个更改请求。环境编辑者可以批准或拒绝该更改请求。

常规设置使您可以：

  • 安装实体
  • 卸载实体
  • 恢复实体
  • 编辑外部依赖项

工件注册表

工件注册表设置允许您将一个工件注册表分配给一个环境。

了解有关分配工件注册表的更多信息。

维护与抑制窗口

维护和抑制 选项卡允许您查看和管理环境及其实体的维护和抑制窗口。

选择 操作 以编辑维护窗口、创建维护窗口覆盖项，并创建抑制窗口。

维护覆盖 表显示了环境及其实体的维护窗口覆盖项。该表可以过滤为在环境或单个实体上的覆盖项，以及当前或历史覆盖项。

了解更多关于定义维护窗口覆盖项的信息。

抑制 表显示了环境及其实体的活动抑制窗口。该表可以过滤为在环境或单个实体上的抑制窗口，以及当前或历史抑制窗口。

了解更多关于配置抑制窗口的信息。

显示元数据

在 显示元数据 选项卡中，您可以更新环境的显示名称、描述和 README。

显示名称将替换环境页面标题中的环境名称，以及 Apollo 中任何引用该环境的地方，例如下拉菜单和列表。

描述应包含有关环境的高级信息，并将在环境页面标题下的标题下方显示。

README 字段应包含有关环境的详细信息，例如迁移计划或文档。 README 支持 Markdown 渲染，并将渲染外部链接、表格、列表等。您可以在环境 概览 选项卡中查看 README。

环境和实体维护窗口

环境和实体维护窗口是安排好的重复时间段，在此期间 Apollo 可以执行维护操作，例如升级服务、应用配置更改以及在特定环境范围内安装新实体。

您可以提交更改请求，为环境和实体创建临时维护窗口覆盖，以创建非重复性的维护窗口。

定义维护窗口

环境编辑器可以在环境的设置文件中通过 global-maintenance-windows 字段指定环境的维护窗口。

实体解析维护窗口

实体解析维护窗口是将用于 维护窗口约束 评估的窗口。Apollo 将根据实体声明的维护窗口和 产品维护窗口 计算每个实体的 解析后的维护窗口。

维护窗口：无停机时间与停机时间

Apollo 提供两种类型的维护窗口：

  • 停机时间维护窗口：允许进行更改，其中该实体上的所有节点可能会离线
  • 无停机时间维护窗口：允许进行不会对终端用户造成可见停机时间的更改

不需要停机时间的更改仍可以在停机时间维护窗口中执行，但反过来则不行；需要停机时间的更改不能在停机时间维护窗口之外执行。

（可选）覆盖产品维护窗口

用于禁用产品维护窗口的可选标志。当启用时，Apollo 将忽略产品的维护窗口，仅考虑环境的维护窗口。默认值为 false。示例：

override-product-maintenance-windows: true

（可选）要求无停机时间的已召回版本回滚使用维护窗口

Apollo 的默认行为是将不需要停机时间的已召回版本在非维护窗口期间进行回滚。您可以使用以下可选标志来使环境上的所有产品退出 Apollo 的默认召回行为。默认为 false。示例：

require-maintenance-windows-for-no-downtime-recalled-release-roll-offs: true

强烈建议不要覆盖此标志以确保有问题的版本召回不会出现延迟。只有在有充分理由的情况下才应将此标志设置为 true。

依赖组

环境编辑器可以创建依赖组，将实体组织成逻辑分组。

当配置了依赖组时：

  • 一个实体只有在与依赖实体共享至少一个组的情况下，才会认为 产品发布依赖 已满足
  • 一个实体只有在与目标实体共享至少一个组的情况下，才会认为 产品发布不兼容 是激活的

示例用例

考虑这样一个场景：你有一个依赖关系 A -> B，表示 A 依赖于 B。你希望同时运行这两个产品的两个独立实例：A1 和 A2，其中 A1 依赖于 B1，而 A2 依赖于 B2。

最初，你安装并运行了 A1 和 B1。现在你想安装 A2，但需要它依赖于 B2 而不是现有的 B1。

通过使用依赖组，你可以通过以下方式实现这种隔离：

    依赖组

    环境编辑器可以创建依赖组，将实体组织成逻辑分组。

    当配置了依赖组时：

    • 一个实体只有在与依赖实体共享至少一个组的情况下，才会认为 产品发布依赖 已满足
    • 一个实体只有在与目标实体共享至少一个组的情况下，才会认为 产品发布不兼容 是激活的

    示例用例

    考虑这样一个场景：你有一个依赖关系 A -> B，表示 A 依赖于 B。你希望同时运行这两个产品的两个独立实例：A1 和 A2，其中 A1 依赖于 B1，而 A2 依赖于 B2。

    最初，你安装并运行了 A1 和 B1。现在你想安装 A2，但需要它依赖于 B2 而不是现有的 B1。

    通过使用依赖组，你可以通过以下方式实现这种隔离：

    • 将 A1 和 B1 放在同一个依赖组（例如 production）
    • 将 A2 放在单独的依赖组（例如 staging）

    此配置确保当您安装 A2 时，其对 B 的依赖不会由现有的 B1 实例满足。相反，A2 将等待在同一个依赖组中安装 B2 后继续执行。

    配置依赖组

    要为一个实体配置依赖组，请在 entity-locator 块下方添加一个 dependency-groups 字段。您可以为一个实体指定多个组。

    您可以使用 dependency-group-overrides 覆盖特定依赖项的默认依赖组行为。这允许您指定在解析特定产品的依赖项时要搜索的组。

    通配符覆盖

    使用通配符 "*" 允许一个实体从任何组中满足依赖关系：

    

    已复制！
    1托管的helm图表:
2  - 实体定位器:
3      名称: 共享服务
4    k8s命名空间: shared
5    产品: com.example:共享服务
6    依赖项组:
7      - 共享服务
8    依赖项组覆盖:
9      com.example:监控:
10        - "*"

    默认行为

    • 未指定分组： 没有显式 dependency-groups 的实体被分配到 default 分组。
    • 空分组： 一个空的 dependency-groups 列表被视为 ["default"]。
    • 无覆盖： 当没有指定覆盖时，依赖项在实体自身的分组内解析。
