Advanced single-node compute options

The basic principle of the following integrations is that you can access tabular Foundry datasets in multiple formats, such as a pandas DataFrame, Arrow Table, Polars DataFrame, and even as raw Parquet or CSV files. This is also shown in the transforms overview. When saving tables from memory to Foundry, we can pass them in any of the formats in which they have been read.

Working with Apache DataFusion

Sometimes, it is easier to cut out the data deserialization step and directly pass the raw underlying dataset files to our compute engine. We can get the path to the files on disk, which get downloaded on-demand by calling my_input.path(). When it comes to writing raw files back to Foundry, we have two limitations to keep in mind:

  • Only Parquet files can be stored in Foundry datasets through this API.
  • Files must be placed in the folder located at the value of my_output.path_for_write_table.

When both criteria are met, we can call write_table with the path to this folder, like so:

Copied!
1my_output.write_table(my_output.path_for_write_table)

To see this in action, consider the following example demonstrating how to use DataFusion ↗ in the platform.

Copied!
1import datafusion
2from datafusion import lit
3from datafusion.functions import col, starts_with
4from transforms.api import transform, Output, Input
5
6
7@transform.using(my_input=Input('my-input'), my_output=Output('my-output'))
8def my_datafusion_transform(my_input, my_output):
9    ctx = datafusion.SessionContext()
10    table = ctx.read_parquet(my_input.path())
11    my_output.write_table(
12        table
13        .filter(starts_with(col("name"), lit("John")))
14        .to_arrow_table()
15    )

Using cuDF

You can also achieve integration through the use of pandas.DataFrame. The following snippet shows an instance using cuDF ↗ in a lightweight transform. This will essentially run pandas code in a highly parallelized manner on the GPU where possible.

Copied!
1@transform.using(
2    my_input=Input('my-input'),
3    my_output=Output('my-output')
4).with_resources(
5    cpu_cores=4,
6    memory_gb=32,
7    gpu_type='NVIDIA_T4'
8)
9def my_cudf_transform(my_input, my_output):
10    import cudf  # Only import CUDF at runtime, not during CI
11    df = cudf.from_pandas(my_input.pandas()[['name']])
12    filtered_df = df[df['name'].str.startswith('John')]
13    sorted_df = filtered_df.sort_values(by='name')
14    my_output.write_table(sorted_df)

The above snippet assumes that your Foundry enrollment is equipped with an NVIDIA T4 GPU and it is available to your project through a resource queue.

GPU-accelerated Polars with cuDF

Your Foundry enrollment must be equipped with a supported NVIDIA GPU, and it must be available to your project through a resource queue to use this feature. See GPU provisioning for the list of available GPU types.

You can accelerate Polars lazy API queries on GPUs using cudf-polars ↗, a GPU-backed execution engine for Polars. This allows you to run existing Polars lazy API queries on GPUs with minimal code changes by passing pl.GPUEngine() to the collect() call. For details on supported operations and performance characteristics, see the Polars GPU support documentation ↗.

GPU-accelerated collection currently only supports collect(), which fully materializes results in memory. Streaming operations such as sink_parquet() are not yet supported. Ensure that your output data fits in available GPU and system memory.

Setup

To use GPU acceleration for the Polars lazy API, add the cudf-polars package to the run requirements in your conda_recipe/meta.yaml file. For details on adding packages, refer to the Python libraries documentation.

Copied!
1requirements:
2  run:
3    - python
4    - transforms {{ PYTHON_TRANSFORMS_VERSION }}
5    - cudf-polars

Next, request a GPU in your transform by chaining .with_resources() onto your decorator:

Copied!
1from transforms.api import transform, Output, Input
2
3@transform.using(
4    output=Output("/Users/jsmith/output"),
5    my_input=Input("/Users/jsmith/input"),
6).with_resources(
7    gpu_type="NVIDIA_T4",
8    memory_gb=32,
9)
10def compute(output, my_input):
11    ...

Writing with the Polars GPU engine

Build your query using the standard Polars lazy API, then pass engine="gpu" to write_table() to collect and write the result using the GPU:

Copied!
1import polars as pl
2from transforms.api import transform, Output, Input
3
4@transform.using(
5    output=Output("/Users/jsmith/output"),
6    my_input=Input("/Users/jsmith/input"),
7).with_resources(
8    gpu_type="NVIDIA_T4",
9    memory_gb=32,
10)
11def compute(output, my_input):
12    lf = my_input.polars(lazy=True)
13
14    result = (
15        lf.with_columns(
16            (pl.col("a") * pl.col("b")).alias("product"),
17            (pl.col("a").pow(2) + pl.col("b").pow(2)).sqrt().alias("norm"),
18        )
19        .group_by("category")
20        .agg(
21            pl.col("product").sum().alias("total_product"),
22            pl.col("norm").mean().alias("mean_norm"),
23            pl.len().alias("n"),
24        )
25        .sort("total_product", descending=True)
26    )
27
28    output.write_table(result, engine="gpu")

The GPU engine only supports a subset of Polars operations. For the full list, see the Polars GPU support documentation ↗. By default, unsupported queries silently fall back to the CPU engine. To raise an error instead, collect manually with raise_on_fail=True and pass the resulting DataFrame to write_table():

Copied!
1df = lf.collect(engine=pl.GPUEngine(raise_on_fail=True))
2output.write_table(df)
