读写数据系统

查询面板支持对数据源进行查询。根据数据源类型的不同，您可以在查询面板中编写不同的查询语句。下文将概述并示例说明如何编写各类查询、在各类型查询中使用 Handlebars 时的安全考量，以及查询片段和条件查询的介绍。

查询安全概述

Foundry 查询采用 Foundry 同步器，该组件对所有同步表强制实施只读权限；此外，对单个表的访问权限严格遵循在 Foundry 数据集级别所授予的访问控制。

对于 Foundry 外部的数据源，在查询中使用 Handlebars 会引发安全隐患，恶意用户可能会用恶意代码替换模板内容，从而实施注入攻击。因此，此类查询在使用 Handlebars 时需遵循额外的安全规则——下文将在SQL 查询和HTTP JSON 查询部分对此规则进行深入说明。

此外，任何引用用户变量（例如 {{user.firstName}}）的模板，其值均从服务器端获取，而非接收登录时传递至浏览器的值。

SQL 查询

安全注意事项

SQL 查询中的所有 Handlebars 模板（使用 Foundry 数据源的模板除外）必须使用 SQL 安全助手或 Handlebars 内置助手 ↗进行封装。有关如何以及何时使用各个 SQL 安全助手的完整详细信息，请参阅《SQL 助手》。

共有五种助手，分别是 schema、table、column、alias 和 param。

  • schema, table：schema 和 table 辅助函数的工作原理非常相似。给定一个名称和允许的名称列表，辅助函数会检查该名称是否存在于允许的名称列表以及相应的信息模式表中。例如，table 辅助函数会检查表名是否存在于允许的名称列表以及 information_schema.tables 或数据库中相应的模式表内。指定允许的名称列表可以防止查询访问任何不应访问的模式或表。不能对允许的名称使用模板化处理，否则会失去验证的意义。
Copied!
1SELECT column1 FROM {{schema someSchemaName 'allowedSchemaName1' 'allowedSchemaName2'}}.{{table someTableName 'allowedTableName1'}};
  • column：column 辅助函数会检查该名称是否存在于 information_schema.columns 或数据库中相应的模式表内。
Copied!
1SELECT {{column someColumnName}} FROM table1;
  • alias：当需要为带有别名的模式、表或列名设定模板时，需使用 alias 辅助函数。由于带别名的名称并不存在于信息模式中，必须通过 alias 辅助函数在 Slate 中对其进行注册，否则无法完成校验。alias 辅助函数仅限配合常量字符串使用，不可用于引用，即 {{alias 'someConstantString'}} 写法允许使用，而 {{alias someReference}} 则不允许。若将其模板化，会使 schema、table 或 column 中的校验失去意义，因为它们可能指向同一对象。
SELECT column1 as {{alias 'aliasedColumnName'}} FROM table1 ORDER BY {{column someColumnName}} where someColumnName is 'aliasedColumnName' and 'aliasedColumnName' is not a valid column name in the database's schema.
  • param：param 助手会将模板替换为 ‘?’，以便后续使用 preparedStatement 设置值。PreparedStatement 是防范 SQL 注入最安全的方式之一。请注意，来自前端的值均为数字或字符串，因此若要在查询中使用非数字或字符串类型的值，必须将该值转换为相应的类型。
    • 已复制！
    1SELECT column1 FROM table1 WHERE column1 > {{param value1}} and dateColumn1 < {{param value2}}::date

    我该在何时使用哪种助手？

    • 若要对模式/表/列名进行模板化，应使用对应的 schema、table 或 column 助手。
    • 若要对带别名的表/列进行模板化，应使用 alias 助手注册别名。
    • 若要对参数值进行模板化，例如 where 子句中比较操作的值，应使用 param 助手。

    编写 SQL 查询

    查询 SQL 数据源时，编辑器支持任意 SQL 命令。通常情况下，你会执行 SELECT 语句。例如：

    Copied!
    1SELECT name,diameter,period FROM allNamed;

    Slate 会将结果行解析为 JSON，每一列对应一个键，从而可以通过 handlebars 进行访问。

    
```json
{
    "name": ["Undina", "Hekate"],
    "diameter": [126.42, 88.66],
    "period": [5.68801089633658, 5.42957878301233]
}
```
你可以通过使用 SQL 内置函数来执行数据转换，例如基本的字符串和数学运算。

HTTP JSON 查询

    安全注意事项
    

    所有 HTTP JSON 查询必须遵循以下规则：
    

      

    • 所有 Handlebars 模板必须包裹在 jsonStringify 助手函数中。jsonStringify 助手函数可确保模板的值不会脱离其当前作用域。例如，它无法闭合当前区块并向请求中添加额外属性。
      
以下是使用它为属性设置模板的示例：
      • 

    


    Copied!
    1{
2    "path": "path/to/api",
3    "method": "POST",
4    "bodyJson": {
5        "filter": {{jsonStringify w1.text}}
6    },
7    "extractors": {
8        "result": "$"
9    },
10    "headers": {
11        "Custom_Header": "my custom header value"
12    }
13}
    

将其作为属性的一部分进行模板化使用的示例：

```json
{
    "path": "path/to/api",
    "method": "POST",
    "bodyJson": {
        "filter": {{#jsonStringify}}some text plus {{w1.text}}{{/jsonStringify}}
    },
    "extractors": {
        "result": "$"
    },
    "headers": {
        "Custom_Header": "my custom header value"
    }
}
```


      

    • 路径中不允许使用 ..。这确保了查询路径不会索引到任何父级作用域，也不会访问不应被访问的信息。
      • 

    

    编写 HTTP JSON 查询
    

    HTTP JSON 数据源的查询是一个包含以下属性的对象：path、method、bodyJson、extractors。
    

      

    • path：数据源的 URL 路径
      • 

    • queryParams：（可选）构建请求时附加到 URL 的键值对映射（例如，“query”: “something” 会在 path 后附加 ?query=something）。请注意，当此映射不为空时，查询参数不应在 path 中指定。
      • 

    • method：发起请求所使用的 HTTP 方法。支持的方法包括 GET、POST、DELETE 和 PUT。
      • 

    • bodyJson：（可选）作为数据发送到 API 端点的 JSON（例如，如何格式化和聚合数据）。如果您的数据源端点不需要 JSON，则无需填写此字段。
      • 

    • extractors：查询返回的结果。使用 JSONPath ↗ 确定要提取的内容。例如，要查看完整结果，请使用 "result": "$"。如需编写 JSONPath 的帮助，请参阅测试工具 ↗。有关 JSONPath 的更多信息，请参阅 JSONPath 示例 ↗。
      • 

    • headers：（可选）要在请求上设置的标头映射。如果存在身份验证标头，则会添加到此列表之上。
      
例如：
      • 

    

```json
{
    "path": "astronomy/_comets",
    "queryParams": {
        "limit": 5,
        "text": "searchabc"
    },
    "method": "GET",
    "bodyJson": {
        "fields": ["name", "type", "date"],
        "query": {
            "type": "dust"
        }
    },
    "extractors": {
        "name": "$.results[*].fields.name"
    },
    "headers": {
        "Custom_Header": "my custom header value"
    }
}
```

    查询片段
    

    查询片段允许你编写可在文档中的多个查询里复用的查询代码。要创建片段，请点击查询面板中的 + 新建片段 按钮。
    

    你可以通过编写 {{>partialName}} 将片段插入到查询中。例如，假设你有一个名为 columnFilter 的片段，其内容为 WHERE column={{param w8.selectedValue}}。你可以使用 SELECT * from table {{>columnFilter}} 代码创建另一个查询。这最终会渲染为查询语句 SELECT * from table WHERE column={{param w8.selectedValue}}。
    

    你还可以向片段传递参数，语法为 {{>partialName arg1=value1 arg2=value2 arg3=value3}}。片段上下文中的参数值将替换为你在特定查询中提供的值。值可以是静态值（如字符串或数字），也可以是 Handlebars 引用（如 w8.selectedValue）。在上面的例子中，如果你有两个完全相同的查询，只是通过两个不同的选定值进行过滤，你可以将 columnFilter 重新定义为 WHERE column={{param columnValue}}，并将查询设为 SELECT * from table {{>columnFilter columnValue=w8.selectedValue}}，如前所述，这会渲染为 SELECT * from table WHERE column={{param w8.selectedValue}}。
    

    你还可以嵌套片段，从而在代码复用中实现代码复用。
    

    片段是 Handlebars 的一个概念，Slate 的实现使用了 Handlebars 语法。请参阅《Handlebars 片段文档》了解更多信息。
    

    条件查询
    

    

触发器与交互 选项卡允许你控制查询运行的条件。有两种条件执行查询的方式：你可以选择 所有依赖项均不为 null，这意味着查询中的每个 Handlebars 引用都必须不为 null 才能运行；你也可以选择 Handlebar 输入返回 true，借此指定一个 Handlebars 条件。该条件可以是对函数、组件属性或任何你希望用来控制查询何时运行的逻辑的引用。只有当该 Handlebars 引用的求值结果为 true 时，查询才会运行——否则，查询将不会执行。
