读取和写入数据系统

查询面板允许您查询您的数据源。根据数据源的类型，您可以在查询面板中编写不同的查询。下面提供了有关如何编写不同类型的查询、在每种查询类型中使用Handlebars时的安全注意事项，以及查询片段和条件查询的介绍。

查询安全概述

Foundry 查询 使用 Foundry 同步器，该同步器对所有同步表施加只读权限；此外，对单个表的访问尊重在 Foundry 中数据集级别的访问权限。

对于Foundry之外的数据源，使用Handlebars进行查询会引发安全问题，因为恶意用户可以通过替换模板内容来执行注入攻击。因此，这些查询需要额外的安全规则来限制Handlebars的使用——详细说明请参见下面的 SQL查询 和 HTTP JSON查询 部分。

此外，任何引用用户变量（例如 {{user.firstName}}）的模板都会从服务器获取其值，而不是接受登录时传递给浏览器的值。

SQL 查询

安全注意事项

所有 SQL 查询中的 handlebars 模板（使用 Foundry 数据源的除外）必须由 SQL 安全辅助函数或 Handlebars 内置辅助函数 ↗ 包裹。您可以在 SQL 辅助函数 中找到每个 SQL 安全辅助函数的使用方法和使用时机的详细信息。

有五个辅助函数，分别是 schema（模式）、table（表）、column（列）、alias（别名） 和 param（参数）。

  • schema, table：schema 和 table 辅助函数工作方式非常相似。给定一个名称和一组允许的名称列表，这些辅助函数会检查该名称是否存在于允许的名称列表中以及对应的模式表中。例如，表辅助函数会检查表名是否存在于允许的名称列表中以及 information_schema.tables 或您数据库中的相应模式表中。指定允许名称列表可以防止查询访问它不应该访问的任何模式/表。您不能对允许的名称进行模板化，因为这将违背验证的目的。
Copied!
1SELECT column1 FROM {{schema someSchemaName 'allowedSchemaName1' 'allowedSchemaName2'}}.{{table someTableName 'allowedTableName1'}};
  • column：column 辅助函数会检查该名称是否存在于 information_schema.columns 或您数据库中的相应模式表中。
Copied!
1SELECT {{column someColumnName}} FROM table1;
  • alias：当您想要对别名的模式、表或列名进行模板化时，会使用 alias 辅助工具。因为别名不在信息模式中，您必须使用 alias 辅助工具将其注册到 Slate；否则，该名称无法被验证。您只能将 alias 辅助工具与常量字符串一起使用，不能与引用一起使用，即 {{alias 'someConstantString'}} 是允许的，而 {{alias someReference}} 不是。对其进行模板化会违背在 schema、table 或 column 中验证它的目的，因为它们可能引用同一事物。
Copied!
1SELECT
2column1 as {{alias 'aliasedColumnName'}}
3FROM table1
4ORDER BY {{column someColumnName}}
5
6where someColumnName is 'aliasedColumnName' and 'aliasedColumnName' is not a valid column name in the database's schema.
  • param：param 辅助工具会用 '？' 替换模板，这样可以稍后使用 preparedStatement 设置值。PreparedStatement 是保护免受 SQL 注入攻击最安全的方法之一。请注意，所有从前端传来的值都是数字或字符串，因此要在查询中使用除数字或字符串以外类型的值，您必须将该值转换为该类型。
Copied!
1SELECT column1 FROM table1 WHERE column1 > {{param value1}} and dateColumn1 < {{param value2}}::date

我应该何时使用哪个辅助工具？

  • 当你想要对模式/表/列名进行模板化时，应该使用对应的schema、table或column帮助函数。
  • 当你想要对别名表/列进行模板化时，应该使用alias帮助函数来注册别名。
  • 当你想要对参数值进行模板化时，即在where子句中的比较值，应该使用param帮助函数。

HTTP JSON 查询

安全注意事项

所有 HTTP JSON 查询必须符合以下要求：

  • 所有 Handlebars 模板必须用 jsonStringify 辅助函数包裹。jsonStringify 辅助函数确保模板的值不会超出其当前作用域。例如，它不能关闭块并给请求添加额外属性。
    使用它的示例来对属性进行模板化：
Copied!
1{
2    "path": "path/to/api",
3    "method": "POST",
4    "bodyJson": {
5        "filter": {{jsonStringify w1.text}}
6    },
7    "extractors": {
8        "result": "$"
9    },
10    "headers": {
11        "Custom_Header": "my custom header value"
12    }
13}
An example to use it to template as part of a property:

Copied!
1{
2    "path": "path/to/api",
3    "method": "POST",
4    "bodyJson": {
5        "filter": {{#jsonStringify}}some text plus {{w1.text}}{{/jsonStringify}}
6    },
7    "extractors": {
8        "result": "$"
9    },
10    "headers": {
11        "Custom_Header": "my custom header value"
12    }
13}
  • 路径中不允许使用 ..。这确保了查询路径不会指向任何父作用域，也不会访问不应访问的信息。

编写 HTTP JSON 查询

HTTP JSON 数据源的查询是一个包含以下属性的对象：path（代码仓库）, method（代码工作区）, bodyJson（数据集）, extractors（租户）。

  • path：数据源的URL路径
  • queryParams：（可选）构建请求时要附加到URL的键值对映射（例如。“query”：“something”会在path后追加?query=something）。请注意，当此映射不为空时，不应在path中指定查询参数。
  • method：用于发起请求的HTTP方法。支持的方法包括GET、POST、DELETE和PUT。
  • bodyJson：（可选）发送到API端点的JSON数据（例如，如何格式化和聚合数据）。如果数据源端点不需要JSON，则此字段不是必需的。
  • extractors：查询返回的结果。使用JSONPath ↗来确定提取的内容。例如，要查看整个结果，请使用"result": "$"。有关编写JSONPath的帮助，请参阅以下测试器 ↗。有关JSONPath的更多信息，请参见JSONPath示例 ↗。
  • headers：（可选）要设置在请求上的头部映射。如果有认证头部，将会添加到此列表之上。
    例如：
Copied!
1{
2    "path": "astronomy/_comets",
3    "queryParams": {
4        "limit": 5,
5        "text": "searchabc"
6    },
7    "method": "GET",
8    "bodyJson": {
9        "fields": ["name", "type", "date"],
10        "query": {
11            "type": "dust"
12        }
13    },
14    "extractors": {
15        "name": "$.results[*].fields.name"
16    },
17    "headers": {
18        "Custom_Header": "my custom header value"
19    }
20}

查询片段

查询片段允许您编写可在文档中的多个查询中重复使用的查询代码。要创建一个片段，请点击查询面板中的 + 新建片段 按钮。

您可以通过编写 {{>partialName}} 将片段插入到查询中。例如，假设您有一个名为 columnFilter 的片段，其内容为 WHERE column={{param w8.selectedValue}}。您可以使用代码 SELECT * from table {{>columnFilter}} 创建另一个查询。这将生成查询 SELECT * from table WHERE column={{param w8.selectedValue}}。

你可以通过语法 {{>partialName arg1=value1 arg2=value2 arg3=value3}} 向部分模板传递参数。在部分模板的上下文中，这些参数的值将被你在特定查询中提供的值替换。值可以是静态值（如字符串或数字），也可以是 Handlebars 引用（如 w8.selectedValue）。在上面的例子中，如果你有两个完全相同的查询，只是由两个不同的选定值进行过滤，那么你可以将 columnFilter 重新定义为 WHERE column={{param columnValue}}，并将查询重新定义为 SELECT * from table {{>columnFilter columnValue=w8.selectedValue}}，这会渲染成 SELECT * from table WHERE column={{param w8.selectedValue}}，和之前一样。

你还可以嵌套部分模板，从而在代码复用中实现代码复用。

部分模板是 Handlebars 的概念，Slate 实现使用了 Handlebars 语法。请参阅 Handlebars 部分模板文档 ↗ 以了解更多内容。

条件查询

触发器 & 交互 标签将允许您控制查询运行的条件。有两种选择可以有条件地运行查询，您可以选择 All dependencies are not null，这意味着查询中的每一个 Handlebars 引用都必须不为 null 才能运行，或者您可以选择 The handlebar input returns true，这将允许您指定一个 Handlebars 条件。此条件可以是对一个函数、小部件属性或任何您想要控制查询运行逻辑的内容的引用。只有当这个 Handlebars 引用评估为 true 时，查询才会运行；否则，查询不会运行。

Postgres中的默认表名将包含文件路径和混合大小写的数据集名称。为了处理特殊字符/、大写字母和空格，Postgres会将表名视为带引号的标识符。这意味着在查询中引用该表时，必须包含双引号，否则Postgres将抛出语法错误。我们建议在设置中包含一个Postgresql table name，采用下划线命名法、小写字母和_，以避免使用双引号。
