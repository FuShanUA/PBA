JupyterLab®

代码工作区（Code Workspaces）使您能够在Foundry中使用JupyterLab® ↗。在代码工作区中的JupyterLab®支持：

  • 读取和写入表格数据集，Iceberg表，以及虚拟表。
  • 从非结构化数据集下载或上传文件。
  • 从conda或PyPI导入Python库。
  • 预览Dash、Streamlit或TensorBoard™应用程序以促进迭代。
  • 发布Dash或Streamlit应用程序。
  • 与本体论（Ontology）进行交互。

代码工作区目前支持Dash ↗和Streamlit ↗用于Python应用程序。用户可以直接在代码工作区中创建应用程序，并内置了Foundry的版本控制、分支和数据治理功能。

代码工作区应用程序支持分支。如果您 创建一个新的工作区分支 , 发布新应用程序或同步更改将在该分支上发布应用程序的新版本。这使您可以在将应用程序暴露给用户之前预览它。要发布到主分支，请简单地将您的分支合并到主分支中。

Dash 应用程序

已发布的 Dash 应用程序有 30 秒的超时限制，这意味着 Dash 服务器必须在执行您的 Dash 应用程序 Python 文件后 30 秒内启动。否则，您的 Dash 应用程序将无法启动。

要创建一个 Dash ↗ 应用程序，请按照以下说明操作：

  1. 在 JupyterLab® 工作区中打开 应用程序 选项卡。
  2. 选择 发布应用程序，选择您 文件和项目 中的位置
    3. 为您的新应用程序提供模块名称和变量名称。将根据提供的模块名称在对应的路径下自动生成一个文件。
  3. 可选：配置高级设置。默认情况下，应用程序设置将与 Jupyter® 工作区的设置相匹配。
  4. 选择发布并同步以注册您的新 Dash 应用程序并将代码同步到支撑代码仓库。CI 检查和发布完成后，您可以选择“应用”面板中的链接来查看已发布的应用。

Dash 应用程序的模块名称应作为相对于仓库根目录的点分路径输入（例如app，或者如果您的模块位于一个文件夹中，则为folder.app）。变量名称是模块文件中声明为 Dash 应用服务器的 Python 变量，通常称为server。

要针对现有 Dash 应用程序进行本地开发：

  1. 打开终端窗口并执行您新创建的文件，例如python app.py。
  2. 选择预览 在应用程序选项卡的Dash部分中，预览应用程序。

您可以使用Jupyter® Notebook中的管理conda/PyPI环境，确保您的JupyterLab®工作区中的conda/PyPI环境将在发布的Dash应用中自动恢复。

Streamlit 应用程序

已发布的Streamlit应用程序会受到30秒的超时限制，这意味着Streamlit服务器必须在执行您的Streamlit应用程序Python文件后30秒内启动。否则，您的已发布Streamlit应用程序将无法启动。

要创建一个 Streamlit ↗ 应用程序，请按照以下说明操作：

  1. 选择 发布应用程序，在您的 文件和项目 中选择一个位置 为您的新应用程序，输入相对于仓库根目录的 Python 文件名称，例如 app.py。
  2. 可选：配置高级设置。默认情况下，应用程序设置将与 JupyterLab® 工作区的设置相匹配。
  3. 选择发布并同步以注册您的新 Streamlit 应用程序并将代码同步到支撑代码仓库。在 CI 检查和发布完成后，您可以选择应用程序面板中的链接以查看已发布的应用程序。

要在现有 Streamlit 应用程序上进行本地开发：

  1. 打开终端窗口并使用 streamlit run app-name.py 运行代码。
  2. 在 JupyterLab® 代码工作区中打开 应用程序 选项卡，并在 Streamlit 部分选择 预览 以预览您正在本地开发的应用程序。

您可以使用 Jupyter® 笔记本中的托管 conda/PyPI 环境 来确保 JupyterLab® 工作区中的 conda/PyPI 环境会在已发布的 Streamlit 应用程序中自动恢复。

TensorBoard™

您可以直接从代码工作区预览 TensorBoard™ ↗。要做到这一点，请在终端中运行 tensorboard --logdir=<$PATH_TO_LOGS_DIRECTORY>，并填写包含 TensorFlow 日志的目录路径。然后，在 Jupyter® 工作区中打开 Applications（应用程序）选项卡，并在 TensorBoard™ 部分选择 预览。

Jupyter® Notebook 转换

Jupyter Notebook 转换目前不支持对外部服务进行 API 调用，即使在代码工作区中添加了网络策略也是如此。为了运行可以对外部系统进行 API 调用的可重复管道，您应该使用 外部转换。

代码工作区可以将 Jupyter® 笔记本发布为数据转换管道，并为笔记本生成的输出数据集注册转换 JobSpec。

请按照以下步骤为输出数据集注册转换：

  1. 将您的 Jupyter® 笔记本保存为工作区中的文件。

  2. 在数据面板中选择数据集旁边的蓝色构建图标。

  1. 按照界面配置您的转换。代码工作区将推断出用于生成输出数据集的 Jupyter® 笔记本以及添加的输入。如果推断错误，您可以手动编辑输入和输出。
  1. 选择同步并运行以保存转换配置，并触发 CI 检查，以在输出数据集上发布转换和 JobSpec。

一旦CI检查完成，您的转换就准备好进行构建了。您可以使用其他Foundry 数据集成工具来管理您的转换以及它所属的 数据管道。

该转换不会自动使用代码工作区的环境。您应该在脚本文件的顶部安装运行时所需的所有库。安装库的代码片段可以在 库 面板中找到。 例如，要在您的转换中使用 scikit-learn 和 scikit-image，请将此代码片段添加到文件的顶部：

Copied!
1!mamba install -y -q scikit-learn scikit-image

在Jupyter® 笔记本中的托管conda/PyPI环境

Jupyter® conda/PyPI 环境可以使用由 Maestro 支持的托管环境功能在 Jupyter® 工作区、应用程序、转换和模型输出中一致地恢复。Maestro 是一个 Foundry 库管理工具，允许您为 Jupyter 工作区安装和管理库。它与 conda 和 pip 集成，使库的安装更简单，并有助于保持一致的环境。要将命令转换为 Maestro，请参考以下有关 conda 或 PyPI 的说明。

托管环境在代码工作区工作流中保持一致，消除了每次重启工作区、启动应用或运行 Jupyter® 笔记本转换时都需要重新运行相同的 mamba 和 pip 命令的需求。有了托管环境，您工作环境中安装的库会通过 /home/user/repo 文件夹中的文件进行跟踪，从而可以在不同工作流之间将环境恢复到原始状态。

以下两个文件用于跟踪您的托管环境：

  1. /home/user/repo/.envs/maestro/meta.yaml：称为环境 清单，它包含您环境的 请求 库及其版本。 了解有关 meta.yaml 文件的更多信息。
  2. /home/user/repo/.envs/maestro/hawk.lock - 称为环境 锁定文件，它包含你的环境的解析库和版本集，这些是从清单中指定的约束条件求解得到的。hawk.lock 文件取代了conda.lock，后者已被弃用。了解有关hawk.lock 文件的更多信息。

我们不建议修改hawk.lock 锁定文件。相反，请依靠安装命令来代表你更新这些文件。除非存在 git 冲突，否则也不需要修改meta.yaml 文件；当库被交互式安装时，它会自动更新。

你的托管环境安装在你的/home/user/envs/default 文件夹中。此环境在以下阶段之前恢复：

  • 启动你的 Jupyter® 工作区
  • 启动从你的工作区生成的应用程序
  • 运行从你的工作区生成的 Jupyter® 笔记本转换
  • 运行模型

使用方法

1. 启用托管环境

要启用此功能，请在工作区设置中切换Managed Conda environments高级功能，该功能位于左侧侧边栏。然后重新启动工作区：

一旦启用了此功能，您将可以使用名为default的托管环境：

2. 更新您的托管环境

按照以下步骤更新您的托管环境并管理库。

通过Maestro管理conda库

  • 使用库面板安装： 搜索库名称（例如，numpy）并安装。

  • 使用终端进行安装：要安装库的最新版本，请在终端中运行以下命令：

    Copied!
    1maestro env conda install <library_name>

    要安装特定版本，请运行以下命令：

    Copied!
    1maestro env conda install <library_name>==<library_version>

  • 使用终端进行卸载：要卸载库，请在终端中运行以下命令：

    Copied!
    1maestro env uninstall --conda-dependencies <library_names>
使用 Maestro 管理 PyPI 库

  • 使用库面板进行安装：搜索库名称（例如，numpy）并安装。

  • 使用终端进行安装：要安装库的最新版本，请在终端中运行以下命令：

    Copied!
    1maestro env pip install <library_name>

    要安装特定版本，请运行以下命令：

    Copied!
    1maestro env pip install <library_name>==<library_version>

  • 使用终端进行卸载：要卸载库，请在终端中运行以下命令：

    Copied!
    1maestro env uninstall --pip-dependencies <library_names>

通过 库 选项卡安装库时，安装命令是在 Jupyter® 终端中直接执行的，而不是在您的笔记本或应用程序代码中执行。每次运行安装命令时，meta.yaml 清单文件将更新以包含您请求的库，并且 hawk.lock 文件将反映当前环境中的已安装库。

3. 查看您管理环境中当前安装的库

已安装的库会在 库 选项卡中显示：

4. 在 JupyterLab® 界面中查看环境文件

在工作区中导航到 视图 下拉菜单并选择 显示隐藏文件。

Then, navigate to the .envs/maestro folder to view the meta.yaml and hawk.lock files.

5. Synchronize your managed environment with changes to the hawk.lock file

After checking out a new branch or pulling new git changes, the contents of your hawk.lock file may change.

To synchronize your managed environment, execute the following command in your workspace terminal:

Copied!
1maestro env install

maestro is the command line interface that updates managed environments. The above command re-resolves the lock file from the manifest and installs the environment into the /home/user/envs/default folder.

6. Resolve git conflicts to your environment files

To resolve git conflicts in your managed environment, first delete the hawk.lock file. Navigate to the meta.yaml file to resolve any conflicts, then merge your branch.

After merging, run the maestro env install command in your terminal to regenerate the hawk.lock file and reinstall your managed environment.

7. Upgrade your managed environment

To update your managed environment, first remove the hawk.lock file. Then, delete your /home/user/envs/default folder (rm -rf /home/user/envs/default). Finally, run the maestro env install command in your terminal to regenerate your hawk.lock file. Your environment will be re-installed with upgraded library versions.

Example: meta.yaml file

The meta.yaml file represents a specification of requested libraries. This file conforms to the Conda metadata specification ↗, and can contain specifications for both conda and PyPI dependencies. Only libraries specified in the requirements.run and requirements.pip section are used for Jupyter® managed environments.

Below is an example of a meta.yaml file:

Copied!
1package:
2  name: '{{ PACKAGE_NAME }}'
3  version: '{{ PACKAGE_VERSION }}'
4requirements:
5  build: []
6  run:
7    - ipykernel
8    - pip
9    - foundry-transforms-lib-python
10    - pandas
11    - polars
12    - dash
13    - streamlit
14    - tensorboard
15  pip:
16    - Flask-Testing
17  run_constrained: []
18source:
19  path: ../src
20build: null
21test: null
22about: null
