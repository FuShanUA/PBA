检查类型

本页概述了 Data Health 中可用的各类检查，包括作业级检查、构建级检查、调度级检查和时效性检查。

作业级检查与调度级检查

本节介绍作业状态、调度状态和调度时长检查。

以下定义阐明了在 Foundry 中作业和调度的具体含义：

  • 作业：由单个转换中的逻辑定义的 Spark 计算。换言之，作业就是一个生成单个数据集的单个转换（如果使用多输出转换，则生成多个数据集）。作业会被拆分为一系列阶段。
  • 调度：具有已定义目标数据集并配置为定期运行的一组作业集合。调度可用于表示协同构建的流水线子集。

我们使用以下 Data Health 检查来确保作业和调度成功运行：

  • 作业状态：当安装了此检查的数据集被刷新，或作为任何构建的一部分被创建时，便会触发该检查。只要目标数据集成功构建，作业状态检查就会通过，即使其所属的构建在下游发生失败也是如此。但请注意，如果构建在目标数据集的上游失败，目标数据集将记录为构建已取消，且不会对该目标数据集评估作业状态。
  • 调度时长与调度状态：通过这些检查，您可以监控调度构建的状态，包括所有中间环节。请注意，只有当检查配置的调度运行时，才会触发此项检查。
    • 通常建议所有调度都配置调度状态检查。如果您已经有了调度状态检查，则不建议在由同一调度构建的其他数据集上安装作业状态检查，因为调度中任何一个作业失败都会触发调度状态检查。

在确定何时何地配置作业或调度状态检查时，请参阅我们关于应用何种健康检查的指南。

关于检查本身的更多细节和进一步说明，请参阅调度状态和作业状态的检查参考文档。

时效性检查

本节介绍同步时效性、数据时效性和距上次更新的时间检查。

这三种检查都与“时效性”（即数据的某些方面有多新）有关，但它们评估时效性的方法各不相同：

  • 距上次更新的时间：评估数据集的时效性。此检查计算当前时间与最后一次提交事务之间经过了多长时间，即使该事务为空也是如此；空事务不会更改数据集中的数据。
  • 数据时效性：通过计算最后一次提交事务与时间戳列最大值之间经过了多长时间，来评估数据集中数据的时效性。此检查仅在提交事务时运行。
  • 同步时效性：通过计算数据集最近一次同步时间与日期时间列最大值之间经过了多长时间，来评估同步数据集中数据的时效性。

对于数据时效性和同步时效性，如果列中的时间戳代表该行在源系统中被添加的时间，那是比较理想的状态。

在确定何时何地配置时效性检查时，请参阅我们关于应用何种健康检查的指南。

关于检查本身的更多细节，请参阅距上次更新的时间、数据时效性和同步时效性的检查参考文档。

健康检查失败时可以中止构建吗？

大多数标准健康检查依赖于作业完成才能进行计算。如果您的数据集是在 Code Repository 中创建的，您可以使用数据期望来定义在构建期间运行的检查。这样您就可以在出错时中止构建，并使用 Data Health 监控这些检查。
