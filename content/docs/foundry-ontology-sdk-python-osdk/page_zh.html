加载示例餐厅分页数据


加载指定分页大小的对象列表，若提供了分页标识，则从该标识处开始加载。


请注意，该端点采用了该对象类型所使用的底层对象同步技术。如果示例餐厅由 Object Storage V2 支撑，则没有请求限制。如果由 Phonograph 支撑，则结果数限制为 10,000 条——当请求的示例餐厅数量超过 10,000 条时，将抛出 ObjectsExceededLimit 错误。


参数：


    

  • pageSize integer（可选）：请求的分页大小，最大不超过 10,000。若未提供，最多加载 10,000 个示例餐厅。首页的 pageSize 将用于后续的所有分页。
    • 

  • pageToken string（可选）：若提供此参数，将请求一个大小小于或等于首次请求页面 pageSize 的分页。
    • 



查询示例：




已复制！
1result = client.ontology.objects.ExampleRestaurant.page(page_size=30, page_token=None)

3data = result.data

加载有序结果


通过为特定属性指定排序方向，加载有序的示例餐厅列表。通过 API 调用时，排序条件通过 fields 数组指定；通过 SDK 调用时，可以将多个 orderBy 调用串联在一起。字符串的排序规则区分大小写，即数字排在пасп大写字母之前，大写字母排在小写字母之前。例如，Cat 会排在 bat 之前。


参数：


    


  • field string：指定想要排序的属性。使用 SDK 时，可通过 sortBy 接口提供该参数。
    • 

  • direction asc| desc ：指定排序方向，升序或降序。使用 SDK 时，可通过 sortBy 接口上的 asc() 和 desc() 函数提供该参数。
    •

过滤


你可以执行的过滤类型，取决于给定对象类型上的属性类型。这些过滤器还可以通过布尔表达式组合起来，构建更复杂的过滤条件。




请注意，该端点采用了对象类型所使用的底层对象同步技术。如果“示例餐厅”后端使用的是 Object Storage V2，则没有请求限制。如果后端使用的是 Object Storage V1（Phonograph），则结果数量限制为 10,000 个——当请求的“示例餐厅”数量超过 10,000 个时，系统将抛出 ObjectsExceededLimit 错误。


参数：


    

  • where SearchQuery（可选）：对特定属性进行过滤。可执行的操作取决于该属性的类型。
    • 

  • orderBy OrderByQuery（可选）：根据特定属性对结果进行排序。若使用 SDK，可通过链式调用 .where 和 orderBy 达到相同效果。
    • 

  • pageSize integer（可选）：请求的页面大小，最大不超过 10,000。若未提供，将加载最多 10,000 个“示例餐厅”。首页的 pageSize 将用于后续所有页面。若使用 SDK，请将 .where 调用与 .page 方法进行链式调用。
    • 

  • pageToken string（可选）：若提供该参数，将请求一个大小小于或等于首次请求页面 pageSize 的页面。若使用 SDK，请将 .where 调用与 .page 方法进行链式调用。
    • 



查询示例：


Copied!
1from ontology_sdk.ontology.objects import ExampleRestaurant
2
3page = client.ontology.objects.ExampleRestaurant.where(ExampleRestaurant.object_type.restaurant_name.is_null()).iterate()

  • value string：用于与餐厅名称进行前缀匹配的值。例如，“foo” 将匹配 “foobar”，但不匹配 “barfoo”。
