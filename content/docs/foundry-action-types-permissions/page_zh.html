权限

权限以以下方式适用于操作类型：

  • 谁可以查看特定的操作类型？
  • 谁可以编辑特定的操作类型？
  • 谁可以使用特定参数集应用操作类型？

应用操作

应用操作类型的权限取决于它所编辑的对象类型和链接类型的配置。在所有情况下，提交操作的用户必须能够查看被编辑的对象类型和链接类型及其数据源，并通过 提交标准 如果对象类型只能通过操作进行编辑，用户可以对其可查看的所有对象进行编辑。对于允许通过操作以外的方式进行编辑的对象类型和关联类型，如果该对象类型或关联类型由数据集支持，则用户还需要在写回数据集上具有编辑权限。如果对象类型或关联类型由 受限视图 支撑，则用户需要通过编辑策略。

使用侧边栏中的 检查访问 面板来检查用户对 Workshop 模块的访问权限，包括对相关操作类型的访问权限及其提交标准。更多信息，请参阅 检查访问面板文档。

提交标准

操作提交条件允许对谁可以运行一个操作进行细粒度控制。简单的提交条件可以要求特定的用户ID或组ID，并且可以与参数中的信息结合使用。有关更多信息，请参阅 提交条件文档。

对象编辑权限

对象编辑可以被锁定，使得只能通过操作进行编辑，或者重新开放，使得可以通过操作、Foundry 表单、直接的对象资源管理器编辑和 API 调用进行编辑。为了在多个工作流程中实施一致的安全范式，默认情况下，新的对象类型仅允许通过操作进行编辑。其他形式的编辑不推荐用于新用途。

对于仅允许通过操作进行编辑的对象类型，提交操作的用户只需要在正在编辑的对象上具有 Read 权限。这意味着用户可以创建他们无法查看的对象。

相比之下，当一个由数据集支持的对象类型可以通过操作、Foundry 表单、直接对象资源管理器编辑和 API 调用进行编辑时，提交该操作的用户必须在所有被编辑对象的写回数据集上拥有 Edit 权限。具有 Edit 权限的用户将能够查看写回数据集中的所有数据。

因此，不建议将对象类型设置为通过操作、Foundry 表单、直接对象资源管理器编辑和 API 调用进行编辑，因为仅仅为了对象编辑而授予 Edit 权限可能会使用户接触到比完成本体论编辑工作流所需的更多数据。

无论采用哪种写回设置，操作类型的配置都不会显示对受影响的基础对象类型的权限设置；配置操作类型的人员必须确保这些权限是正确的。

将对象类型的编辑权限更新为“仅允许通过操作进行编辑”不会删除历史上的非操作性编辑，但会阻止进一步通过 Foundry 表单、直接对象资源管理器编辑和 API 调用进行的编辑。

了解有关写回权限的更多信息。

副作用权限

任何可以设置操作的用户都可以配置副作用。

  • Webhook 副作用默认未启用。在 Data Connection 应用中配置 Webhook 插件之前，需要额外的权限才能在操作设置页面中使用它。如对在您的 Foundry 实例上使用 Webhook 有任何疑问，请联系您的 Palantir 代表。

提交标准必须正常通过；如果操作提交标准失败，则不会触发副作用。

接收者必须有权访问通知中包含的任何对象数据。

  • 如果用户没有访问通知内容中的所有数据的权限，通知将不会发送给他们。
  • 如果有多个接收者，并且某些人缺少通知中包含的正确权限数据，只有具有足够权限的用户才会收到通知。
  • 如果由于任何原因通知无法发送，编辑仍可能成功。

执行操作的用户必须能够查看将接收通知的用户和/或组。
