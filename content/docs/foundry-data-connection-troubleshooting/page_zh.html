故障排除

当数据源没有按照预期工作时，使用此页面来诊断和解决该问题。它描述了在数据连接中可用的故障排除工具，当连接无法工作时首先应检查的地方，以及如何解决常见的连接和证书问题。

对于同步或代理特有的问题，请参见 功能特定的问题。

从哪里开始

当数据源或连接无法正常工作时，可使用此列表进行初步排查。

  1. 确认数据源可以访问。 在数据源页面上，选择 预览 以确认连接已建立。如果失败，请打开 数据源终端 并针对数据源的主机名和端口运行 dig、curl 和 openssl s_client。
  2. 检查出站配置。 对于 Foundry worker（Foundry 工作人员） 源，确认已附加正确的 出口策略（egress policies），并且它们允许的地址、端口和协议与源连接到的匹配。查看 网络出口日志（network egress logs）。
  3. 检查凭据（Check credentials）。 如果您怀疑凭据已过期，请重新输入，尤其是在迁移后。
  4. 检查证书（Check certificates）。 PKIX 或 SSLHandshakeException 错误表示源未配置正确的证书。参见 证书和 TLS（Certificates and TLS）。
  5. 检查 OAuth 授权（Check OAuth authorization）。 如果您的源使用了一个 出站应用程序（outbound application），HTTP 401: Unauthorized 和类似错误通常表示调用用户尚未完成交互式授权流程，或者出站应用程序被禁用，或者配置的作用域不足。参见 OAuth 和出站应用程序（OAuth and outbound applications）。
  6. 转交 AI FDE 处理（Hand off to AI FDE）。 对于 Foundry worker 源，使用 通过 AI FDE 进行调试（Debug with AI FDE），让 AI FDE 执行这些检查并显示具体的修复方法。

如果您的源使用了一个 代理工作程序（agent worker），请确认代理显示为 在线（Online） 在数据连接中进行检查，并查阅代理故障排除参考以获取特定状态的指导。

故障排除工具

数据连接提供了多种内置工具，帮助您在不离开平台的情况下调查连接性、身份验证和数据移动问题。

使用AI FDE进行调试

对于Foundry worker源，您可以将调试任务转交给 AI FDE，即Palantir的由人工智能驱动的前线部署工程师。AI FDE会将您的源及其相关的 出站策略 加载到上下文中，并利用本页描述的相同出站日志和源调试工具来代表您调查连接失败的问题。

从源开始AI FDE调试会话的步骤如下：

  1. 在Data Connection应用程序中打开您想要调试的源。
  2. 在连接详情面板中，选择调试旁边的下拉箭头，然后选择使用AI FDE进行调试。
  3. AI FDE会在新标签页中打开，并预先加载源及其出站策略作为上下文，提出一个调试计划，包括检查源配置、审查每个出站策略、检索网络出站日志以及运行实时连接测试。

AI FDE将总结哪些部分正常工作，哪些部分失败，并建议具体的修复方法（例如：缺少出站策略、地址或端口错误、TLS配置问题或凭证问题）。

使用AI FDE进行调试仅适用于使用Foundry worker运行时的源。AI FDE需要在您的注册中启用AIP。有关AI FDE的工作原理、功能以及如何管理上下文和权限的更多信息，请参阅AI FDE概述。

源终端

一个终端，使用与您的源相同的连接设置，可用于帮助您通过命令如 dig、curl、netcat 和 openssl 测试与外部系统的网络连接。

要打开终端，请在源页面的 连接详细信息 下的 网络连接 面板中选择 调试。

dig

dig 命令测试 DNS 解析，可用于检查您的源的主机名是否正确解析。例如，要测试 DNS 是否解析 palantir.com，请运行 dig palantir.com。如果您在响应中看到 ANSWER SECTION，则表示 DNS 解析成功。

以下是一个成功的 dig 命令的示例：

appuser@localhost:/root$ dig palantir.com

; <<>> DiG 9.18.30-0ubuntu0.20.04.2-Ubuntu <<>> palantir.com
;; global options: +cmd
;; Got answer:
;; ->>HEADER<<- opcode: QUERY, status: NOERROR, id: 13561
;; flags: qr rd ra; QUERY: 1, ANSWER: 4, AUTHORITY: 0, ADDITIONAL: 0

;; QUESTION SECTION:
;palantir.com.                  IN      A

;; ANSWER SECTION:
palantir.com.           300     IN      A       151.101.129.170
palantir.com.           300     IN      A       151.101.1.170
palantir.com.           300     IN      A       151.101.193.170
palantir.com.           300     IN      A       151.101.65.170

;; Query time: 99 msec
;; SERVER: 10.100.0.53#53(10.100.0.53) (UDP)
;; WHEN: Fri Oct 31 14:56:58 UTC 2025
;; MSG SIZE  rcvd: 142

curl

curl 命令测试与外部系统上特定主机和端口的 HTTP 连接性。如果未指定端口，将根据协议推断（例如，HTTP 的 80 或 HTTPS 的 443）。例如，要测试是否可以连接到 https://palantir.com，运行 curl https://palantir.com。

以下是一个成功的 curl 命令的示例：

appuser@localhost:/root$ curl https://palantir.com
<HTML><HEAD><meta http-equiv="content-type" content="text/html;charset=utf-8">
<TITLE>301 Moved</TITLE></HEAD><BODY>
<H1>301 Moved</H1>
The document has moved
<A HREF="https://www.palantir.com/">here</A>.
</BODY></HTML>

如果 curl 命令挂起，请使用 --connect-timeout 标志设置超时时间（以秒为单位）：

Copied!
1curl --connect-timeout 10 https://palantir.com

openssl

openssl 命令测试 SSL/TLS 连接并验证 HTTPS 连接的证书。例如，要测试是否可以建立到 palantir.com 端口 443 的 SSL/TLS 连接，运行 openssl s_client -connect palantir.com:443 -servername palantir.com。

以下是一个成功的 openssl 命令的示例：

appuser@localhost:/root$ openssl s_client -connect palantir.com:443 -servername palantir.com

verify return:1
depth=1 C = BE, O = GlobalSign nv-sa, CN = GlobalSign Atlas R3 DV TLS CA 2024 Q2
verify return:1
depth=0 CN = *.palantir.com
verify return:1
---
Certificate chain

   i:C = BE, O = GlobalSign nv-sa, CN = GlobalSign Atlas R3 DV TLS CA 2024 Q2
   a:PKEY: rsaEncryption, 2048 (bit); sigalg: RSA-SHA256
   v:NotBefore: Jun 17 16:50:15 2024 GMT; NotAfter: Jul 19 16:50:14 2025 GMT
---

---

如果连接成功，您将看到证书详细信息和表明SSL握手已完成的消息。如果命令卡住或显示证书验证错误，可能是网络或证书配置问题。

在UI中配置的证书不会自动包含在终端环境中。要在终端中使用它们，请手动添加，可以内联添加或使用 nano/vim:

Copied!
1echo "-----BEGIN CERTIFICATE-----
2MIIGXjCCBUagAwIBAgIQASByQ6gv8Z6X7wEqsyBb1DANBgkqhkiG9w0BAQsFADBY
3...
49lc=
5-----END CERTIFICATE-----" > server-cert.pem

要使用自定义证书进行测试，例如自签名证书或mTLS，请指定证书文件：

Copied!
1openssl s_client -connect my-server.example.com:443 \
2  -servername my-server.example.com \
3  -CAfile server-cert.pem \
4  -cert client-cert.pem \
5  -key client-key.pem

如果这些命令成功但您的连接仍然无法工作，可能是一些证书或源凭据配置不正确。

网络出站日志和指标

对于 Foundry worker 源，网络出站日志和指标可提供关于您的源实际发起的网络调用的可见性。这些工具可以帮助识别仅通过终端调试可能无法察觉的网络问题。

  • 构建日志：从构建页面中，选择 日志 并应用 the 网络出站日志 过滤器用于查看该构建的网络出站活动。
  • 控制面板： 导航到 控制面板 > 网络出站，选择一个策略，并打开 可观测性 选项卡以查看每个来源的聚合日志和策略的使用指标。

源预览与探索

在深入调试之前，请确认源是否基本可访问且配置正确：

  • 预览： 在源页面上，在右侧面板中选择 预览 以确认连接是否已成功建立。
  • 探索： 使用 源探索 视图来检查来自源的表、文件和预览。如果探索成功但特定同步失败，则问题更可能是与同步配置有关，而不是与源本身有关。

作业跟踪器

要调查正在运行或最近完成的同步状态，请打开作业跟踪器应用程序并选择同步。从那里，您可以检查状态、阶段和日志，包括JDBC同步的_data-ingestion.log。

连接性和出口

为了使连接成功，Foundry必须被允许通过网络访问您的外部系统。Foundry worker源通过出口策略控制这一点，这些策略列出了源被允许连接到的具体主机、端口和协议。在认证之前失败的连接通常是出口问题。

要调试疑似出口问题：

  1. 确认正确的出口策略已附加到源，并且它们允许的主机、端口和协议与您要连接的系统匹配。
  2. 使用 源终端 要测试主机是否可达，使用 dig、curl 和 openssl s_client。
  3. 查看 网络出站日志 以了解源的实际出站调用并识别哪些被阻止。

要创建或请求出站策略，请参阅 配置出站策略。

证书和TLS

本节假设Foundry作为客户端与作为服务器的外部系统建立连接。有关底层概念的背景信息，请参见 TLS ↗、mTLS ↗ 和 SSL ↗。

PKIX和SSL异常

PKIX 异常和其他 SSLHandshakeException 在 Foundry 没有与源进行身份验证所需的证书时发生。安装正确的证书：

  • Foundry worker（工作进程） 源：在源的设置中配置证书。参见 添加证书。
  • Agent worker（代理工作进程） 源：参见 数据连接和证书。

对于仅支持已弃用的 TLSv1.0 或 TLSv1.1 协议的源，请参见 使用不安全的 TLSv1.0 和 TLSv1.1 协议连接到数据源。Palantir 不建议使用已弃用的 TLS 版本。

服务器和客户端证书

了解服务器证书和客户端证书之间的区别非常重要。

服务器证书（TLS）： 当Foundry客户端与外部系统建立TLS连接时，外部系统会出示一个证书以证明其身份，并提供建立加密连接所需的公钥。然后，Foundry客户端通过验证证书链至受信任的根证书颁发机构（CA）来验证外部系统的身份。大多数系统默认信任一个知名公共CA列表。只有在外部系统出示的证书不是由这些公共CA之一签名时（例如自签名证书或由私有或内部CA颁发的证书），您才需要将服务器证书添加到您的源配置中（Foundry的truststore）。服务器证书仅以证书形式提供，从不包括私钥。

客户端证书（mTLS）： 客户端证书用于Foundry客户端向外部系统证明其自身身份。在mTLS中，外部系统和Foundry各自验证对方的身份。客户端证书以证书和私钥对的形式提供。

要配置证书，请参阅 添加证书。

Common certificate errors

[SSL: CERTIFICATE_VERIFY_FAILED] certificate verify failed: self signed certificate in certificate chain

This error indicates that server certificates are missing from the source configuration. In Python environments, the entire certificate chain ↗ is validated by the underlying OpenSSL library when establishing secure connections.

To retrieve the entire certificate chain for an external system, run the following command:

Copied!
1openssl s_client -connect <external_systems_hostname>:<desired_port> -showcerts < /dev/null

This command must be run from a host with network access to the external system. For Foundry worker sources, run it from the source terminal. For agent worker sources, run it from the agent host or another system with access to the agent's network.

OAuth and outbound applications

When a source is configured with an outbound application , authentication is delegated to an external OAuth 2.0 provider on behalf of the calling user. The user must complete the interactive authorization flow at least once before the source can be used from non-interactive contexts, such as scheduled functions or an automation.

Common OAuth errors

HTTP 401: Unauthorized

If a function or workflow returns HTTP 401: Unauthorized despite the source being configured with an outbound application, verify that:

  • The user invoking the function has completed the interactive authorization flow at least once.
  • The outbound application is Enabled in Control Panel.
  • The scopes configured on the outbound application include the permissions required to access the requested API resource.

Credentials expired and no refresh handler provided

This error is surfaced when calling a source from a Python or TypeScript function. The cached refresh token has expired or was revoked by the external OAuth 2.0 server. The user must invoke the function (or another workflow backed by the same outbound application) from an interactive interface to complete the authorization flow again.

Resolved source credentials are not present on the Source

This error is surfaced when calling a source from a Python or TypeScript function. It indicates that the source does not have an outbound application configured, or that the calling identity could not be resolved to an authorized user. Confirm that the source's domain is configured with OAuth 2.0 authentication and that the function is invoked on behalf of an authenticated end user rather than a service account.
