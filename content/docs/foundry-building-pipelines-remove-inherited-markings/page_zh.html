移除继承的标记和组织

标记和组织根据用户的资格限制对资源的访问。

在派生依赖资源时，如果受限内容已被移除或混淆，用户可能希望移除该派生资源上的标记和/或组织。这种移除继承的标记和组织的过程，可以通过使用stop_propagating和stop_requiring输入转换属性来完成。

  • stop_propagating用于移除继承的标记（例如 PII）。
  • stop_requiring用于移除继承的组织（例如 Palantir）。

术语

  • 组织是应用于项目的访问要求，确保用户群体与资源之间实现严格的隔离。为了满足访问要求，用户必须是应用于项目的至少一个组织的成员或访客成员。
  • 标记是应用于资源的访问要求，以全有或全无的方式限制访问。为了满足访问要求，用户必须是应用于资源的所有标记的成员。
  • 角色是权限的集合，定义了用户可以对给定资源执行的特定工作流（例如查看者、编辑者等）。

重要须知

  • stop propagating和stop requiring关键字仅适用于组织和标记，而不适用于角色。
  • 代码库必须至少拥有一个受保护分支（例如 main）。该分支还必须强制至少一名必需的审批者。
  • 只能在受保护分支上移除组织和标记。提及非受保护分支（例如 on_branches=[..., "not-protected-branch"]）会导致构建失败。
  • 支持在 Python、Java 和 SQL 中移除组织和标记。
  • 审批移除继承的组织和标记需要特殊的用户权限。用户需要具备 Remove marking 权限才能移除标记，并具备 Expand access 权限才能移除组织。

步骤

  1. 从受保护分支（例如 main）创建一个新分支。

  2. 将 stop_propagating 和 stop_requiring 属性中的一个或两者添加到输入转换中。例如：

  1. 创建拉取请求以将此代码合并到受保护分支。

  2. 拥有标记的 Remove marking 权限或组织的 Expand access 权限的用户可以批准或拒绝提议的更改。如果添加了多名审查者，任何审查者的拒绝都将导致整个拉取请求被拒绝。

  3. 如果获批，代码编辑器将合并拉取请求并构建输出数据集。输出数据集构建完成后，将不再带有传播的标记和/或组织。

在内部，组织以一种稍有不同的标记类型来表示，因此跟随 stop_requiring 之后的转换关键字被称为 OrgMarkings。

输入转换属性

要移除继承的标记（例如 PII），请使用 stop_propagating 关键字。

要移除继承的组织（例如 Palantir），请使用 stop_requiring 关键字。

这些关键字必须在需要移除标记或组织的每个输入上指定。对于每次移除，还必须指定移除应应用到的受保护分支。标记 ID、组织 ID 和分支应始终以带引号的字符串形式指定。

您需要提供至少一个上游组织，因为用户只需满足至少一个组织即可。列出的每个组织都需要进行审批。下方的详细工作流提供了一个说明此情况的示例。

Python

在 Python 中，标记移除是在输入构造函数中指定的。

@transform(
  input_1=Input("<input_id>",
      stop_propagating=Markings([markingId1, ...], [branch1, ...]),
      stop_requiring=OrgMarkings([orgMarking1, ...], [branch2, ...])),
  output=Output("<output_id>")
  )

Markings 类接收一个标记 ID 列表和一个受保护分支列表，在这些分支上应用标记移除。可以在 Settings 页面的 Markings 列表中找到标记 ID。

OrgMarking 类接收一个组织 ID 列表和一个受保护分支列表，在这些分支上应用标记移除。可以在 Settings 页面的 Organizations 列表中找到组织 ID。

Java

Java 自动注册

在 Java 中，标记移除是通过对自动注册的转换的输入使用注解来指定的。

语法：



Copied!
1@Compute
2public void myComputation(
3  @StopPropagating(markings = {markingId1, ...}, onBranches = {branch1, ...})
4  @StopRequiring(orgMarkings = {orgId1, ...}, onBranches = {branch2, ...})
5  @Input("<input_id>")
6  FoundryInput input,
7  @Output("<output_id>")
8  FoundryOutput output)

@StopPropagating 和 @StopRequiring 注解接收一组标记 ID 和一组受保护的分支，在这些分支上应用移除标记的操作。如果只指定了一个标记或分支，则无需将其用 {} 括起来（例如 @StopPropagating(markings = marking1, onBranches = "my-branch")）。

SQL

在 SQL 中，通过使用 SparkSQL hint 语句来指定移除标记：

Copied!
1CREATE TABLE <output_id> AS
2  SELECT /*+ foundry_stop_propagating(markingId1, ...) foundry_stop_requiring(orgMarkingId1, ...) foundry_on_branches(branch1, ...) */ *
3  FROM <input_id>

在 SQL 中，移除标记和组织的操作可以添加到任何 SELECT 语句中。例如：

Copied!
1CREATE TABLE <output_id> AS SELECT * FROM <input_1_id>
2  CROSS JOIN (SELECT /*+ foundry_stop_propagating(markingId1) foundry_on_branches("my-branch") */ * FROM <input_2_id>)

移除权限

为了能够查看代码并一般性地批准拉取请求，审批者必须通过项目和代码库本身上的所有组织和标记，同时拥有包含基础 Stemma 查看代码库 工作流的角色（默认包含在 Viewer 角色中）。用户还必须拥有对每个组织和标记的权限，才能设置审批模式或批准移除这些组织和标记的拉取请求。用户不一定是该组织或标记的成员。

对于标记审批，执行批准的用户需要拥有该标记的 移除标记 角色。

对于组织审批，审批用户需要在该标记上拥有 Expand access 角色。

审批模式

对于每个代码库以及每个组织和标记，数据治理用户可以定义应使用哪种模式来触发新的审批：

  • 需要重新审批：这是每个组织和标记的默认模式。对于针对移除了该组织和标记的分支提交的任何拉取请求，代码库始终需要进行安全审批。此模式可防范逻辑变更，从而确保组织和标记被安全地移除。
  • 无需重新审批：当在给定输入的转换上首次移除此组织和/或标记时，需要审批。后续的逻辑变更不会因安全审批而受阻。

详细工作流

在此示例场景中，一位代码编辑者希望使用来自敏感上游项目的两个数据集，移除特定信息，并允许更广泛的受众访问生成的数据集。这两个数据集各有两个标记，已作为引用添加到下游项目中。代码编辑者希望停止传播四个标记中的三个，使其不出现在输出数据集上。此外，上游项目仅限 OrgA 或 OrgB 的用户访问，而目的是将下游数据分发给 OrgC 的用户。

之前：代码编辑者分支上的输出数据集继承了全部四个标记，并且仍然仅限 OrgA 或 OrgB 的用户访问。

之后：输出数据集一旦合并到受保护分支（例如 main）中，现在仅继承一个标记，并且不再要求用户是 OrgA 或 OrgB 的成员。

步骤

  1. 代码编辑者在下游项目的 feature/clean-data 分支上编写新的转换。

  1. 由于所有标记更改都是针对 master 分支请求的，因此在 feature/clean-data 上工作无需任何批准。换言之，当输出数据集在 feature/clean-data 分支上构建时，仍将继承所有上游标记。

  2. 代码编辑者向 main 分支创建拉取请求，并请求管理受 lemon、apple 和 cherry 标记限制数据的数据治理用户批准。代码编辑者还请求来自 OrgA 的 扩大访问范围 组织管理员批准，当 OrgA 数据需要与其他组织共享时，该管理员可以予以批准。

通过移除继承的组织来扩大访问范围，其效果是移除所有继承的组织，但只需要在转换中列出的组织具有相应权限的用户批准。如果希望要求所有组织都批准，则需要在 `stop_requiring` 组件中列出所有 ID。在本示例中，OrgA 主要负责上游项目中的数据，因此编辑者选择 OrgA 进行跨组织批准流程。如此一来，编辑者只需 OrgA 管理员批准即可移除继承的组织。根据编辑者希望请求哪些组织批准，编辑者可以选择 `stop_requiring`： (1) OrgA（需 OrgA 管理员批准）， (2) OrgB（需 OrgB 管理员批准），或 (3) OrgA 和 OrgB 两者（需两个组织的管理员共同批准）。

最终结果是，输出数据集将不会从输入中继承任何组织，而是仅遵循其所在项目的组织限制。

  1. 数据治理用户和组织管理员会收到 Foundry 通知，提示已请求他们的批准。

  2. 假设拉取请求获得批准，代码编辑者将其合并并构建输出数据集，如上方的之后图像所示。

  3. 下周，另一位代码编辑者对不同的代码文件进行更改，并打开拉取请求以合并到 master。

如果所有标记都不需要重新批准，则无需经过安全审查即可批准该拉取请求。如果任何标记确实需要批准，则此新拉取请求将需要由管理该标记的数据治理或组织管理员进行安全审查。
