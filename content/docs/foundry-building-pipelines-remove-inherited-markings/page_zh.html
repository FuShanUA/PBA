移除继承的标记和组织

标记（Markings） 和 组织（Organizations） 根据用户是否符合条件来限制对资源的访问。

当在派生依赖资源时，受限内容被删除或模糊处理，用户可能希望从该派生资源中移除标记（Marking）和/或组织（Organization）。可以通过使用 stop_propagating 和 stop_requiring 输入转换属性来完成此过程。

  • stop_propagating 用于移除继承的 标记（Markings）（例如，个人身份信息（PII））。
  • stop_requiring 用于移除继承的 组织（Organizations）（如 Palantir）。

术语

  • 组织（Organizations） 是应用于项目的访问要求，它保证了用户组和资源之间的严格隔离。为了满足访问要求，用户必须是至少一个应用于项目的组织的成员或客座成员。
  • 标记是应用于资源的访问要求，它以全有或全无的方式限制访问。为了满足访问要求，用户必须是应用在该资源上的所有标记的成员。
  • 角色是一组权限的集合，用于定义用户可以在给定资源上执行的具体工作流程（例如：查看者、编辑者等）。

重要须知

  • stop propagating 和 stop requiring 关键词仅适用于组织和标记，不适用于角色。
  • 仓库需要至少有一个受保护的分支（例如：main）。此分支还必须 强制至少一个必需的审批人。
  • 你只能在受保护的分支上删除组织和标记。提及一个未受保护的分支（例如 on_branches=[..., "not-protected-branch"]）会导致构建失败。
  • 在 Python、Java 和 SQL 中支持删除组织和标记。
  • 需要特殊用户权限才能批准删除继承的组织和标记。用户需要 Remove marking 权限来删除标记，需要 Expand access 权限来删除组织。

    • 基本流程

    项目C中的灰色数据集框突出了这样一个事实：必须为目标项目的所有输入添加项目引用。

    步骤

    1. 在受保护分支（例如 main）上创建一个新分支。

    2. 将 stop_propagating 和 stop_requiring 属性之一或两者添加到输入转换中。例如：

    1. 创建一个拉取请求，将此代码合并到受保护分支。

    2. 拥有 Remove marking 权限的用户（用于标记）或 Expand access 权限的用户（用于组织）可以批准或拒绝建议的更改。如果添加了多个审阅者，任何审阅者的拒绝都将导致整个拉取请求被拒绝。

    3. 如果获得批准，代码编辑器将合并 PR 并构建输出数据集。输出数据集构建完成后，将不再具有传播的标记和/或组织。

    在内部，组织被表示为一种略有不同的标记类型，因此在 stop_requiring 之后使用的转换关键字被称为 OrgMarkings。

    输入转换属性

    要移除继承的标记（例如 PII），请使用 stop_propagating 关键字。

    要移除继承的组织（例如 Palantir），请使用 stop_requiring 关键字。

    这些关键字必须在需要移除标记或组织的每个输入中指定。对于每次移除操作，还必须指定应应用该移除的受保护分支。标记 ID、组织 ID 和分支应始终作为带引号的字符串指定。

    您需要提供至少一个上游组织，因为用户只需要满足至少一个组织即可。每个列出的组织都需要审批。下面的 详细流程 提供了一个说明这一点的例子。

    Python

    在 Python 中，标记移除是在输入构造函数中指定的。

    Copied!
    1@transform(
2  input_1=Input("<input_id>",
3      stop_propagating=Markings([markingId1, ...], [branch1, ...]),
4      stop_requiring=OrgMarkings([orgMarking1, ...], [branch2, ...])),
5  output=Output("<output_id>")
6  )

    Markings 类接收一组标记 ID 和一组要应用标记删除的受保护分支。标记 ID 可以在 Markings 列表中找到，该列表位于 Settings 页面上。

    OrgMarking 类接收一组组织 ID 和一组要应用标记删除的受保护分支。组织 ID 可以在 Organizations 列表中找到，该列表位于 Settings 页面上。

    移除权限
    

要查看代码并批准一般性的拉取请求，审批者必须通过项目的组织和标记，以及仓库本身，同时还必须拥有包含基本Stemma View Repository工作流的 角色（默认情况下，包含在查看者角色中）。用户还必须对每个组织和标记拥有权限，才能设置审批模式或批准移除这些组织和标记的拉取请求。用户不一定需要是该组织或标记的成员。
    对于标记的审批，审批用户需要在该标记上拥有 Remove marking 角色。
    对于组织的审批，审批用户需要在该标记上拥有 Expand access 角色。
    

审批模式
    对于每个代码仓库以及每个租户和标记，数据治理用户可以定义应使用哪种模式来触发新的审批：
    • 要求重新审批：这是每个租户和标记的默认模式。当此租户和标记被移除时，任何针对该分支的拉取请求都将始终需要安全审批。此模式防止对逻辑的更改，从而确保租户和标记被安全地移除。
    • 不要求重新审批：当此租户和/或标记首次在某个输入上被移除时，需要审批。后续对逻辑的更改将不会因安全审批而被阻止。

    详细流程
    在这个示例场景中，一个代码编辑器想要使用来自敏感上游项目的两个数据集，删除某些信息，并允许更广泛的受众访问生成的数据集。这两个数据集各有两个标记，已经在下游项目中作为引用添加。代码编辑器希望阻止四个标记中的三个继续传播，这样它们就不会出现在输出数据集中。此外，上游项目限制为OrgA或OrgB的用户，目的是将下游数据分发给OrgC的用户。
    之前：代码编辑器分支上的输出数据集继承了所有四个标记，仍然仅限于OrgA或OrgB的用户。
    之后：代码编辑器分支上的输出数据集不再包含被阻止的标记，现在可以被OrgC的用户访问。
    

之后：当 输出 数据集合并到受保护的分支（例如，main）后，现在只继承了一个 Marking（标记），不需要用户成为 OrgA 或 OrgB 的成员。
    步骤
    1. 代码编辑器在下游项目的仓库的 feature/clean-data 分支上编写一个新的转换（转换）。
    1. 由于所有的标记（标记）更改都是针对 master 分支提出的，因此在 feature/clean-data 上工作不需要任何批准。换句话说，当输出数据集在 feature/clean-data 分支上构建时，所有上游的标记（标记）仍将被继承。
    2. 

代码编辑器会向主分支创建一个拉取请求，并请求数据治理用户批准，这些用户管理受lemon、apple和cherry标记限制的数据。代码编辑器还请求来自OrgA的Expand access组织管理员的批准，当OrgA的数据需要与其他组织共享时，该管理员可以进行批准。
    

通过移除一个继承的组织，会起到移除所有继承的组织的效果，但只需从转换中列出的组织上具有适当权限的用户进行批准。如果您希望从所有组织获得批准，那么需要在 `stop_requiring` 组件中列出所有ID。在这个例子中，OrgA主要负责上游项目的数据，因此编辑者选择了OrgA来进行跨组织的批准流程。因此，编辑者只需要获得OrgA管理员的批准即可移除继承的组织。根据编辑者想要请求的组织批准，编辑者可以选择 `stop_requiring` 以下内容之一：
(1) OrgA（由OrgA管理员批准），
(2) OrgB（由OrgB管理员批准），或
(3) OrgA和OrgB（由两个组织的管理员批准）。
    最终结果是，输出数据集将不会从输入中继承任何组织，并且只会尊重其所在项目的组织。
